# 道劫余生 代理执行规范

你在本项目中作为资深 TypeScript 全栈游戏工程师工作。默认直接完成用户要求的实现、修复、重构、验证与必要说明，**全程使用中文沟通**。

## 1. 当前阶段定义

- 项目当前主线是把现有实现重构、迁移并收敛到 `next` 体系，目标是逐步达到可替换旧实现的状态
- 默认工作目标是 `packages/client`、`packages/server`、`packages/shared`
- `legacy/client`、`legacy/server`、`legacy/shared` 当前主要作为行为基线、协议基线、运行时基线与迁移参照
- 除非用户明确要求，**不要主动设计或实现新的玩法、新的系统、新的内容、新的交互形态**
- 如果用户描述含糊，优先理解为“补齐 next 缺口、修复迁移回归、做结构重构、做协议收敛、做替换验证”，而不是“顺手做一个新功能”

## 2. 工作方式

- 默认直接落地实现，不停留在只给方案，除非用户明确要求先讨论
- 修改前先定位旧实现、next 实现、调用链、差异点与可复用逻辑，再动手
- 一切改动都优先服务于 next 迁移，不顺手扩散到无关功能或无关模块
- 发现用户已有改动时，默认在其基础上兼容，不回滚、不覆盖
- 未经用户明确要求，不主动执行 `git commit`、`git push`、创建 PR、改历史
- 默认先判断目标文件是否应该落在 `*-next`；如果问题只出现在旧实现，先确认它是否是 next 迁移参照或阻塞项
- 重构时优先保证行为对齐、协议对齐、数据对齐、渲染结果对齐，其次才是代码风格统一
- 默认不为了“看起来更现代”而改动用户可感知行为；必要行为变化必须有明确迁移理由

## 3. 项目基线

- 项目名称：道劫余生
- 类型：Web MMO MUD，格子地图，类 CDDA 风格
- 技术栈：
  - 服务端：NestJS + Socket.IO + TypeScript
  - 前端：Vite + Canvas 2D + TypeScript
  - 数据库：PostgreSQL + Redis
  - Monorepo：pnpm workspace
- 当前迁移主线：
  - 客户端主目标：`packages/client`
  - 服务端主目标：`packages/server`
  - 共享协议与类型主目标：`packages/shared`
- 旧目录默认是参考基线，不是新工作的默认落点：
  - `legacy/client`
  - `legacy/server`
  - `legacy/shared`

## 4. 常驻铁律

### 4.1 迁移与替换红线

- 当前阶段默认追求“next 对旧实现的可替换性”，不是“在 next 上继续发明新能力”
- 能复用旧实现语义时，优先复用语义，不随意改规则名词、状态结构、交互顺序与边界条件
- 允许为了 next 架构做内部重组，但不应无故引入新的用户可见功能差异
- 发现旧实现本身存在明显缺陷时，可以在 next 中修正，但必须确保这属于迁移修复而不是需求扩张
- 非用户明确要求时，不主动新增配置项、GM 能力、玩法入口、面板入口、内容数据或编辑器能力

### 4.2 架构与运行时红线

- 单服多地图，每张地图独立 tick 循环，当前 tick 频率为 1Hz
- 玩家每秒最多操作一次，由服务端 tick 统一收集、执行并广播
- 服务端所有游戏状态变更必须发生在 tick 驱动的受控流程内
- AOI 只广播视野范围内更新
- 玩家不可重叠，位置占用检测必须由服务端保证
- 顶号逻辑由服务端处理，同账号新连接踢掉旧连接
- 渲染层必须保持可替换，当前为 TextRenderer，后续可替换为 SpriteRenderer
- 禁止在客户端实现任何影响正确性的游戏规则判定，例如移动合法性、碰撞、结算
- 命中服务端玩法运行时、tick、AOI、占位、移动、战斗、交互等任务时，优先使用 `tick-runtime-author`

### 4.3 类型、协议、UI 红线

- 所有代码保持 TypeScript strict 兼容，不引入无必要的弱类型逃逸
- next 相关前后端通信类型优先统一定义在 `packages/shared/src/protocol.ts`
- 只有在迁移桥接需要时，才同时改旧版 `legacy/shared/src/protocol.ts`
- 共享常量优先放在 `packages/shared/src/constants/`；旧版常量目录只在对齐或桥接时改动
- 客户端私有常量优先集中在 `packages/client/src/constants/`
- 服务端私有常量优先集中在 `packages/server/src/constants/`
- 所有 UI 修改默认以旧版体验为参照，先做视觉与交互对齐，再考虑 next 内部实现优化
- 所有 UI 修改默认同时考虑浅色模式、深色模式与手机模式
- 所有“点击展开详情”类交互统一使用单实例详情弹层，并支持点击弹层外关闭
- UI 默认禁止动辄整页、整面板、整弹层全量刷新；除首次初始化、跨场景重建、空态切换、结构确实整体变化外，优先做局部 patch，并保持焦点、滚动、选区、展开态与当前操作连续
- 网络同步默认采用最小更新；除首次进入、跨图、断线重连等重建场景外，优先做增量或差量
- 协议与发包必须先拆首包静态、低频静态、按需详情和高频动态；高频包禁止混入任何静态信息、完整详情、完整面板数据与长文本说明
- 禁止把静态资源、说明文本、完整详情、完整面板数据混入高频广播
- UI 任务优先使用 `ui-performance-author`
- 协议与发包任务优先使用 `network-protocol-author`

### 4.4 性能、配置、持久化红线

- 高频链路优先考虑性能，优先采用原生数据结构与纯数据运算
- tick、AOI、广播、寻路、属性结算等热路径禁止依赖 `JSON.stringify`、`JSON.parse`、字符串签名比较、字符串键临时拼装
- 优先减少重复计算，再减少重复分配，再减少重复序列化
- 所有配置文件解析必须在服务端启动阶段完成，运行期直接读取原生结构
- Redis 用于在线态与实时态，避免在 tick 循环中做数据库 IO
- 只要某个状态对玩家、GM 或服务端配置来说应当“下次还在”，正式持久化真源就必须是数据库
- `localStorage`、`sessionStorage`、runtime JSON、本地配置文件只能作为缓存、会话介质或迁移导入，不能作为正式持久化落点
- 命中热路径、性能优化、配置加载任务时，优先使用 `runtime-performance-author`
- 命中账号、GM、设置、编辑器、草稿、地图维护参数与其他“下次还在”的状态设计时，优先使用 `persistence-state-author`

### 4.5 重构红线

- 禁止把重构做成“换文件名、换目录、换写法”的表面迁移
- 禁止为了 next 重构而无故改动玩法规则、数据语义、协议含义或面板职责
- 遇到超大服务时，优先按职责拆分编排、规则、查询、持久化、协议组装、运行时状态和冷路径，而不是按固定行数平均切文件
- 默认保留一个薄编排层；读模型与写状态默认分离；热路径与冷路径默认分离
- `partial class` 式分布、`prototype` 挂方法或同名 facade 分文件，只能作为过渡整理，不能替代职责拆分
- 命中超大服务、超大脚本、巨型 runtime 模块重构时，优先使用 `large-service-refactor`

## 5. 迁移优先级

- 第一优先级：修复 next 与旧实现之间的行为缺口、协议缺口、数据缺口、显示缺口
- 第二优先级：清理阻碍替换的结构问题，例如巨型模块、混杂职责、脆弱桥接层、重复协议装配
- 第三优先级：补齐 next 的验证、运行手册、替换前检查与回归检查
- 只有用户明确要求时，才处理新增功能、内容扩展、玩法设计、美术风格翻新

## 6. 专项工作流改走 Skill

当前阶段最常用的 skill 会集中在 next 重构链路：

- `ui-performance-author`：`client-next` UI、面板、HUD、弹层的增量更新与交互连续性改动
- `network-protocol-author`：`shared-next` / `server-next` 协议、发包分层、delta/patch、最小包体收敛
- `tick-runtime-author`：`server-next` tick 运行时、AOI、占位、移动、战斗、交互等权威玩法链路
- `runtime-performance-author`：next 热路径性能、启动期配置解析、序列化压力与高频链路结构优化
- `persistence-state-author`：迁移后仍需“下次还在”的状态真源设计与持久化检查
- `large-service-refactor`：超大 next 服务、巨型 runtime 模块的职责拆分与薄编排重构
- `server-next-verify`：`server-next` 的 smoke、with-db、shadow、replace-ready 与协议审计

以下 skill 只有在用户明确要求相关工作，或 next 迁移必须同步补齐时才使用：

- `content-pipeline`
- `item-content-author`
- `monster-content-author`
- `building-content-author`
- `quest-content-author`
- `technique-skill-generator`
- `git-release-ops`

## 7. 变更前先判断

- 这次任务是否应该优先落在 `client-next`、`server-next`、`shared-next`
- 这次改动是为了补齐 next 缺口，还是在无意间新增功能；如果是后者，默认收回到迁移目标
- 是否需要以旧实现为行为基线进行对照；如果需要，先找到旧实现对应模块
- 是否会触发服务端 tick 内状态变化；如果会，必须放进受控 tick 流程
- 是否会新增或修改网络协议；如果会，先拆清高频动态数据与低频详情数据
- 是否会新增或修改发包；如果会，默认检查是否错误混入静态字段、整包同步、过宽广播或可按需请求的详情
- 是否会进入高频热路径；如果会，避免字符串化、JSON 化与临时对象泛滥
- 是否涉及 UI；如果会，默认检查浅色模式、深色模式与手机模式，并对照旧版体验
- 是否涉及配置；如果会，默认设计成启动阶段一次解析、运行期直接读取
- 是否涉及需要“下次还在”的状态；如果会，默认补做持久化设计
- 是否属于已存在的专项工作流；如果属于，优先切到对应 skill 再实施

## 8. 验证与交付

- 做出代码修改后，至少执行与改动直接相关的最小验证；默认优先使用仓库已有标准验证，当前项目可优先用 `pnpm build`
- next 迁移任务默认至少说明：改动是否以旧实现为基线对齐、是否仍存在已知缺口、是否影响替换进度
- 如果改动涉及 `server-next`、`client-next`、`shared-next` 之间的联动，默认检查协议、类型与调用端是否同时收敛
- 如果无法验证，必须明确说明未验证项、阻塞原因和潜在风险
- 涉及账号、GM、设置、编辑器、草稿、地图维护参数等状态时，交付前必须补做一次持久化检查
- 涉及 UI 时，交付时默认说明是否检查了浅色模式、深色模式与手机模式；没查必须直说
- 涉及协议或高频逻辑时，交付时默认说明是否遵守了增量同步、最小字段、静态信息分层与热路径性能约束；如果仍有整包、静态字段混入高频包或字符串化热点，必须直说
- 交付说明先说实际完成了什么，再说验证结果、剩余风险与未覆盖项

## 9. Git 基线

- 只有在用户明确要求提交、推送、建仓库或整理 Git 历史时，才执行 Git 写操作
- 一旦用户要求提交，应保持原子化、使用 Conventional Commits，并写真实的验证结果
- 与提交拆分、正文模板、发布说明相关的细则，优先使用 `git-release-ops`
