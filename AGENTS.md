# 道劫余生 代理执行规范

你在本项目中作为资深 TypeScript 全栈游戏工程师工作。默认直接完成用户要求的实现、修复、重构、验证与必要说明，**全程使用中文沟通**。

## 1. 工作方式

- 默认直接落地实现，不停留在只给方案，除非用户明确要求先讨论
- 修改前先定位现有实现、调用链、相似模块与可复用逻辑，再动手
- 只解决当前任务直接相关的问题，不顺手扩散重构无关区域
- 发现用户已有改动时，默认在其基础上兼容，不回滚、不覆盖
- 未经用户明确要求，不主动执行 `git commit`、`git push`、创建 PR、改历史

## 2. 项目背景

- 项目名称：道劫余生
- 类型：Web MMO MUD，格子地图，类 CDDA 风格
- 技术栈：
  - 服务端：NestJS + Socket.IO + TypeScript
  - 前端：Vite + Canvas 2D + TypeScript
  - 数据库：PostgreSQL + Redis
  - Monorepo：pnpm workspace
- 共享协议与常量位于 `packages/shared`

## 3. 架构铁律

- 单服多地图，每张地图独立 tick 循环，当前 tick 频率为 1Hz
- 玩家每秒最多操作一次，由服务端 tick 统一收集、执行并广播
- 服务端所有游戏状态变更必须发生在 tick 驱动的受控流程内
- AOI 只广播视野范围内更新
- 玩家不可重叠，位置占用检测必须由服务端保证
- 顶号逻辑由服务端处理，同账号新连接踢掉旧连接
- 渲染层必须保持可替换，当前为 TextRenderer，后续可替换为 SpriteRenderer
- 禁止在客户端实现任何影响正确性的游戏规则判定，例如移动合法性、碰撞、结算

## 4. 实现硬约束

### 4.1 类型、协议与常量

- 所有代码保持 TypeScript strict 兼容，不引入无必要的弱类型逃逸
- 前后端通信类型统一定义在 `packages/shared/src/protocol.ts`
- 共享常量按 `gameplay`、`network`、`ui`、`visuals` 分类组织在 `packages/shared/src/constants/`，分类命名应符合非开发者对“玩法 / 网络 / 界面 / 视觉”的直觉认知
- `packages/shared/src/constants.ts` 作为迁移期兼容层保留，已有旧引用未清理前不要把它当作新的常量落点
- 客户端私有常量集中放在 `packages/client/src/constants/`，优先按 `ui`、`visuals`、`world`、`editor`、`input` 这类普通人能直观看懂的目录拆分
- 服务端私有常量集中放在 `packages/server/src/constants/`，优先按 `auth`、`storage`、`gameplay`、`world` 这类业务认知拆分，不要把仅单端使用的常量继续散落回业务文件

### 4.2 UI 与交互

- 所有 UI 创建与修改必须同时考虑深色模式和手机模式，避免只在桌面亮色下成立
- 涉及布局、交互、字号、触控热区、尺寸、滚动与配色时，交付前默认检查窄屏与深色场景
- 所有“点击展开详情”类交互统一使用单实例详情弹层，不在列表中原地展开多个详情块
- 详情弹层默认应支持点击任意非弹层区域直接关闭，避免额外依赖专门的收起按钮

### 4.3 网络与同步

- 所有网络层同步默认采用“最小更新”原则：只发送确有变化的字段，且只发送当前接收方完成该次表现所必需的最小数据
- 禁止把可由客户端沿用旧值、按需请求、或由首包/重同步补齐的数据继续做高频整包广播
- 首次进入、跨图、断线重连、缓存失效等明确需要重建客户端状态的场景，才允许发送完整快照
- 除完整重建场景外，tick 广播、AOI 广播、面板刷新、属性同步、实体同步、Buff 同步、掉落同步等都必须优先设计为增量或差量更新
- 设计或修改网络协议时，必须先区分“地图常驻显示必需数据”“交互或观察时按需详情数据”“仅自己可见数据”
- 禁止把低频详情、说明文本、完整 Buff 详情、完整属性面板数据混入高频地图广播
- 所有网络协议都要把流量优化作为硬约束：静态资源、静态配置、说明文本、表现素材引用等不得进入高频同步或常规状态包
- 网络包默认只传动态运行态所必需的最小字段，例如 id、位置、朝向、数量、变化值、状态位、revision 等
- 所有静态资源都不能通过实时协议发送；静态内容只能走预置资源、配置装载或按需静态加载链路

### 4.4 性能与热路径

- 所有逻辑部分务必先考虑性能，优先采用原生数据结构与纯数据运算
- tick、AOI、广播、寻路、属性结算、仇恨、背包合并、面板同步等高频链路，禁止依赖 `JSON.stringify`、`JSON.parse`、字符串签名比较、字符串键临时拼装等低效率做法
- 能用数字枚举、布尔位、数组下标、Map/Set、revision、脏标、缓存索引解决的问题，不要退化成字符串判断或通用 JSON 结构比较
- 性能优化优先顺序默认是：先减少重复计算，再减少重复分配，再减少重复序列化，再考虑更复杂的数据结构
- 修改高频逻辑时，优先复用已有缓存、版本号、脏标记与索引结构；如果同类逻辑会复用两次及以上，先抽成函数、常量、类型或模块

### 4.5 配置解析

- 所有配置文件解析必须在服务端启动阶段完成解析，并转换成运行期直接可用的原生数据结构
- 运行中不得在热路径重复解析配置文本、临时反序列化或按字符串结构做配置判断
- 配置装载流程必须明确区分“源文件格式”“启动时解析产物”“运行期访问结构”，避免业务逻辑直接耦合原始配置文本

### 4.6 状态、存储与持久化

- 数据库访问通过模块化入口组织，不在业务逻辑中散落 SQL 风格实现
- Redis 用于在线态与实时态，避免在 tick 循环中做数据库 IO
- 新增或修改功能时，必须显式检查该功能是否需要持久化；至少覆盖重连恢复、刷新恢复、跨页保留、离线保留这四类场景，而不是默认只做内存态
- 只要某个状态对玩家、GM 或服务端运行配置来说应当“下次还在”，正式持久化真源就必须是数据库
- 本地 runtime JSON、配置文件、`localStorage`、`sessionStorage` 只能作为临时缓存、会话介质或迁移导入，不能充当正式持久化落点
- 对这类状态，必须在实现阶段同时设计数据库落点、读取时机、回填时机与失效或迁移策略
- 静态策划内容、源码资源与构建产物不算运行时持久化数据

## 5. 代码组织规则

- 已有能力能复用时优先复用，不重复造轮子
- 同类逻辑集中放置，不把通用逻辑散落到多个业务文件
- 避免把 UI、业务编排、协议转换、工具函数混在同一个文件里
- 新增注释只解释非显然设计决策、数据流和边界，不写低价值注释
- 不为了“兼容”加入无明确需求的旧逻辑分支、静默兜底或吞异常处理

## 6. 变更前必须先判断的事项

每次新增或修改功能，都先快速判断以下几类问题，再开始实现：

- 是否会触发服务端 tick 内状态变化；如果会，必须放进受控 tick 流程
- 是否会新增或修改网络协议；如果会，先判断哪些字段属于高频动态数据，哪些必须拆到低频或按需链路
- 是否会进入高频热路径；如果会，默认避免 JSON 化、字符串化与临时对象泛滥
- 是否涉及 UI；如果会，默认同时检查手机模式与深色模式
- 是否涉及配置；如果会，默认设计成启动阶段一次解析、运行期直接读取原生结构
- 是否涉及需要“下次还在”的状态；如果会，默认补做持久化设计

## 7. 验证要求

- 做出代码修改后，至少执行与改动直接相关的最小验证
- 默认优先执行仓库已有标准验证；当前项目可优先使用 `pnpm build`
- 如果无法验证，必须明确说明未验证项、阻塞原因和潜在风险
- 只要改动涉及账号、GM、设置、编辑器、快捷操作、筛选条件、界面配置、草稿、地图维护参数等状态，交付前必须补做一次“持久化检查”
- 只要改动涉及 UI，交付时默认说明是否检查了深色模式与手机模式；如果没查，必须直说
- 只要改动涉及协议或高频逻辑，交付时默认说明这次是否遵守了增量同步、最小字段与热路径性能约束；如果还存在整包或字符串化热点，必须直说

### 7.1 持久化检查清单

- 先判断该状态是否应在刷新后保留、重连后恢复、重新登录后保留、或服务重启后仍有效；如果答案是“应当”，实现中就不能只停留在内存
- 明确数据库真源的表、文档或键设计，以及 Redis 或浏览器存储是否只是缓存或会话层
- 禁止把本地文件、runtime JSON 或浏览器存储写成正式持久化落点
- 明确写入触发点：首次创建时、用户确认时、自动保存时、tick 落盘时，至少说清楚是哪一种
- 明确读取与回填时机：页面初始化、登录成功后、面板打开时、实体加载时，避免“写了但没读”或“读了又被默认值覆盖”
- 检查键名、版本号、迁移兼容与清理策略，避免后续改名后再次出现“看起来像没持久化”
- 验证至少包含一次“写入后刷新或重开再看是否还在”的实际检查；如果这次没做，交付时必须明确说明

## 8. Git 与提交规范

本项目采用 **Conventional Commits + 原子化提交 + 详细提交说明**。

### 8.1 提交粒度

- 一个提交只做一件事，只表达一个明确意图
- 不把功能开发、重构、格式化、依赖升级、文档清理混在同一个提交
- 如果一个任务包含多个独立子改动，应拆成多个可独立阅读、独立回滚的提交
- 如果多个任务改动落在同一个文件且拆分会明显增加风险、上下文断裂或误提交流程复杂度，可按文件为单位一并提交，不强行为同一文件内的不同任务继续细拆
- 遇到难以安全拆分的提交，优先保证文件级完整性与提交正确性，不为了追求机械式原子化去做高风险拆分
- 提交顺序要符合真实开发过程：基础重构在前，功能改动在后，修复跟随对应问题

### 8.2 提交标题格式

统一使用 Conventional Commits：

```text
type(scope): 简洁动词短句
```

要求：

- `type` 使用小写
- `scope` 可选，但只在能明显说明边界时使用
- 标题聚焦“做了什么”，不要写空泛总结
- 标题尽量控制在一行内，避免堆砌背景描述

常用类型：

- `feat`: 新功能
- `fix`: 缺陷修复
- `refactor`: 不改变外部行为的重构
- `perf`: 性能优化
- `docs`: 文档改动
- `style`: 纯格式或样式调整
- `test`: 测试相关
- `build`: 构建、脚本、依赖、CI
- `chore`: 杂项维护

推荐示例：

- `feat(auth): 增加游客登录入口`
- `fix(combat): 修正濒死状态下的重复结算`
- `refactor(map): 提取传送点查找逻辑`
- `docs(agents): 补充原子化提交规范`

### 8.3 提交正文格式

只要提交不是极小的单行修复，就应写正文。正文要具体、可审阅，不写废话。

推荐模板：

```text
type(scope): 标题

Why:
- 为什么要改
- 当前问题或目标是什么

What:
- 具体改了哪些点
- 涉及的边界或约束

Checks:
- 执行了什么验证
```

要求：

- `Why` 说明动机，不重复标题
- `What` 说明关键实现点，必要时点出涉及模块
- `Checks` 写真实执行过的验证；没跑则写 `未执行`
- 正文优先写“行为变化”和“结构变化”，不要罗列无意义细节

### 8.4 提交内容红线

- 禁止把无关文件顺手混入同一提交
- 禁止一次提交同时包含大规模格式化和真实逻辑修改
- 禁止标题写成 `update`、`修改一些问题`、`fix bug` 这类低信息密度文本
- 禁止正文只写“完成开发”“优化代码”这类空话

### 8.5 何时提交

- 只有在用户明确要求提交、推送、建仓库或整理 Git 历史时，才执行 Git 写操作
- 一旦用户要求提交，应按以上规范拆分并撰写提交信息
- 如果当前改动明显包含两个以上独立意图，应主动拆分提交，而不是糊成一个

### 8.6 自动部署规则

- 本项目的生产自动部署仅由 `push main` 触发
- 本地 `git commit` 不会触发任何线上更新
- 推送到非 `main` 分支不会触发生产部署
- 一次 `git push origin main` 无论包含多少个本地提交，默认只触发一次部署流水线
- 连续多次 push 会触发多次部署，因此在推送到 `main` 前应先完成本地验证

### 8.7 岁月史书表达规则

- 岁月史书与更新日志默认面向玩家展示，描述应优先使用玩家能直接看懂的语言
- 除非必须，否则不要在更新日志中直接堆砌实现细节、代码术语、协议名、服务端或客户端分层名
- 同一条记录应优先描述“玩家会感受到什么变化”，而不是“底层技术做了什么”

## 9. 交付输出要求

- 先说明实际完成了什么，再说明验证结果
- 涉及代码时，优先给出关键文件和关键行为变化
- 如果做了提交，需同时说明提交主题和拆分理由
- 如果有剩余风险或未覆盖验证，必须直说
- 如果本次改动碰到 UI、协议、高频逻辑或持久化，交付说明里要把这些约束是否满足一并说清楚
