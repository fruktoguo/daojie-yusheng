# 服务端 Tick 运行时

这个 skill 用于正式修改服务端玩法运行时。核心目标只有两个：所有正确性由服务端掌握，所有状态变化走受控 tick 流程。

适用场景：

- 修改移动、战斗、交互、采集、技能释放、行动队列
- 修改地图 tick、玩家指令收集、行动执行、广播时机
- 修改 AOI、占位检测、顶号处理、地图内实体运行时
- 新增会改变世界状态的玩法行为

## 先看哪里

优先找当前任务对应的：

- 地图 tick 主循环
- 玩家输入收集与消费链路
- occupancy/position 相关实现
- AOI 广播与实体可见性实现
- 当前行为的相似玩法逻辑

先沿调用链确认“动作入队 -> tick 执行 -> 状态变更 -> 广播”的现状，再动手。

## 强制流程

1. 先判断这次改动是否会改变世界状态。
2. 只要会改变世界状态，就把真正的状态写入放进 tick 驱动的受控流程内。
3. 玩家输入先收集，再由 tick 统一执行；不要在 socket 回调里直接改世界状态。
4. 涉及移动、传送、刷怪、死亡、拾取、建造、交互占用时，先确认服务端唯一地维护位置与占用。
5. 涉及广播时，先判断是玩家自有结果、AOI 广播，还是更大范围通知；默认只发视野内需要知道的对象。
6. 涉及客户端表现时，只把客户端当渲染和交互层，不把移动合法性、碰撞、结算、可见性判断下沉到客户端。
7. 如果同账号多连接会影响本链路，沿用服务端顶号逻辑，不在客户端做并发正确性兜底。

## 硬规则

- 服务端所有游戏状态变更必须发生在 tick 驱动的受控流程内。
- 玩家每秒最多操作一次；改相关逻辑时必须保证节拍约束仍成立。
- 玩家不可重叠，位置占用检测必须由服务端保证。
- AOI 只广播视野范围内更新，不为了省事改成全图或全服广播。
- 客户端禁止承担任何影响正确性的规则判定。
- 渲染层必须保持可替换；不要把玩法正确性和 `TextRenderer` 或具体 UI 绑定。

## 实现偏好

- 区分“玩家请求意图”和“tick 中实际执行结果”。
- 区分“运行时状态写入”和“给客户端看的投影/表现”。
- 先做服务端校验，再更新运行时状态，再生成广播 payload。
- 广播只带这次 tick 需要被感知的变化，不把整块世界态重发。
- 涉及占用、碰撞、路径结果时，优先复用现有运行时索引和原生结构，不临时拼字符串键。

## 交付时必须说明

- 这次改动的状态变化具体发生在哪个 tick 流程
- 是否仍保持服务端权威和玩家每秒一次操作约束
- 是否影响 AOI 广播范围、占位检测或客户端职责边界
- 如果改了动作收集或执行顺序，新的顺序是什么
