# 服务端权威运行时

用于修改服务端游戏运行时。目标是把所有影响正确性的游戏状态变化收敛到服务端受控流程内，并在热路径上保持可证明的性能。

## 商业级 MMO 口径

- 服务端运行时必须能支撑长时间在线、多玩家并发、多地图实例和断线重连后的状态一致。
- 正确性优先于表现流畅；客户端预测只能优化体验，不能改变权威结果。
- 每个权威状态变化都要能回答来源、执行时机、是否覆盖同类意图、广播范围、持久化影响和失败恢复。
- 性能设计要面向可扩容热路径，避免每新增玩家、实体或地图都线性放大无谓成本。

## 适用场景

- tick、地图实例、世界状态、玩家意图收集、覆盖、排队和执行。
- 移动、占位、碰撞、交互、战斗、属性结算、buff。
- AOI 计算、运行时广播输入、实体可见性。
- 热路径结构优化，必要时评估 Rust 加速边界。

## 强制流程

1. 先确认逻辑是否必须在 tick 内执行；影响游戏正确性的状态变化默认必须进入 tick。
2. 明确状态真源：内存运行态、Redis 在线态、数据库持久态分别承担什么。
3. 玩家输入可以在一息内多次提交；服务端按领域收集意图，不把 tick 频率误写成输入频率限制。
4. 同类可覆盖意图以最后一次为准，例如寻路目标、目标选择、移动方向；不可覆盖意图必须有明确排队、幂等、去重、冷却或拒绝规则。
5. 玩家意图再由 tick 或领域受控流程统一裁定和执行；不要在 socket handler 中直接改权威世界态。
6. 移动、占位、战斗、交互由服务端裁定；客户端输入只作为意图。
7. AOI 广播只发送视野范围内必要变化，出包细节交给网络层。
8. 热路径先减少重复计算，再减少重复分配，再减少重复序列化。
9. 只有基准或明确热点证明 TypeScript 不够时，才考虑 Rust/N-API/WASM；Rust 边界只处理纯计算或稳定数据结构。

## 热路径红线

- tick、AOI、寻路、占位、战斗、属性结算、广播组装中禁止依赖 `JSON.stringify`、`JSON.parse`、字符串签名比较或临时字符串键拼装。
- tick 中避免数据库 IO；需要持久化时走异步落盘、快照或受控队列。
- 配置解析必须在启动阶段完成，运行期直接读取预解析结构。
- 不以全图扫描替代已有索引、占位表、AOI 结构或增量状态。

## Rust 加速口径

- 只在稳定且可测的纯计算热点上使用，例如寻路批处理、AOI 几何、视野裁剪、批量属性结算。
- Rust 模块不能直接持有数据库真源、socket 连接或复杂业务生命周期。
- 必须提供 TypeScript fallback 或明确构建失败时的门禁行为。
- 必须补最小基准或回归验证，证明行为一致。

## 交付说明

- 哪些状态变化进入 tick 或领域受控流程，哪些只是输入或表现。
- 一息内多次输入如何覆盖、排队、去重或拒绝。
- 是否满足商业级 MMO 的权威一致性、并发、恢复和热路径要求。
- 是否影响 AOI、占位、战斗或持久化真源。
- 热路径是否仍有 JSON/字符串热点残留。
- 是否使用或评估 Rust，加速边界和验证结果是什么。
