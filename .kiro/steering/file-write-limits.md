# Kiro 专属规则：单次文件写入避免截断

> 适用范围：`kiro-cli chat` / Kiro 默认 agent。本文件仅由 Kiro 通过 `.kiro/steering/**/*.md` 加载，与通用的 `AGENTS.md` 并行生效。

## 规则核心

- 本规则的目的是 **避免单次写入被工具或上下文截断而导致代码不完整**，不是给写入设硬性行数上限。
- **判断标准是写入是否完整成功**：只要写入工具返回成功且内容未被截断，就允许一次写入。
- 行数本身不是判罚依据；超过任何具体阈值不会被自动判违规。

## 受约束的写入入口

- `write` 工具的 `create` 命令（`content` 参数）
- `write` 工具的 `insert` 命令（`content` 参数）
- `write` 工具的 `strReplace` 命令（`newStr` 参数）
- 通过 `shell` 等工具执行的 heredoc / `cat > file` / `tee` / `printf` 等等价写入

## 推荐做法

- **优先一次写完**：内容能在一次写入内完整提交、未被截断时，直接一次写完更可读、更易回顾。
- **接近上下文极限或感觉可能被截断时再拆**：超出 ~300 行的写入开始变得有风险，过长（例如 800+ 行）显著增加截断概率，此时主动拆分。
- **拆分时仍按自然边界**：函数、类、章节、顶层语句之间切，避免破坏语法或语义连续性。
- **写入失败或检测到被截断**：必须按拆分策略重写，不可凑合提交不完整内容。

## 不允许的规避方式

- 不得为了压缩行数而把多行代码人为合并到同一行（影响可读性、diff 与回归定位）。
- 不得提交被截断、被吞尾、被吞中段的不完整文件，无论触发原因。

## 例外

- "完整删除文件"或"将文件清空"这类不携带新内容的操作不受本规则约束。
- 自动格式化、`format` 工具等不直接由模型构造内容的操作不计入本规则。
