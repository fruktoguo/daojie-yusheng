# 重构守则

本次是重构（不引入用户可感知的玩法 / 协议 / 持久化语义变化）。按以下守则执行。

## 必须

- 按职责拆分大模块：编排、规则、查询、持久化、协议装配、运行时状态、冷路径工具、运维验证。
- 保留薄编排层；不要把所有职责再卷回一个 facade。
- 新抽象必须 **减少真实复杂度 / 降低重复 / 稳定边界 / 匹配现有架构** 中至少一项；不能为"更现代 / 更优雅"而重构。
- 同步维护 smoke / proof / audit / bench / 最小验证；缺一即视为只搬了代码。
- TypeScript 规范：不允许新增 `// @ts-nocheck`、`// @ts-ignore`、`// @ts-expect-error`（除非有不可避免的单行注释说明）；逐步把假 TS 文件迁移成规范 TS。

## 禁止

- 单纯换目录、换文件名、换写法。
- 顺手扩散到无关模块。
- 改变玩法规则、协议语义、持久化语义、面板职责。
- 引入用户可感知行为变化，除非交付说明里能给出明确工程理由 + 验证证据。
- 把 socket handler、协议拼装、持久化 IO、冷路径脚本重新卷回巨型服务。

## 执行步骤

1. **先告诉我目标拆分前后的职责图**：哪个文件 / 类 / 函数原属于哪一层，要去哪一层；薄编排和领域规则的边界画在哪。
2. 列出会被触碰的 smoke / proof / audit / bench 入口，以及预计被搬动的协议 / 持久化 / 运行时入口。
3. 等我确认后再动；中途若发现需要改协议 / 持久化语义，**停下来问我**，不要顺手改。
4. 全程不要 `--amend` 已推送 commit、不 force push、不 reset --hard。
5. 拆完再用 `verify-pick` 决定门禁。

## 必跑门禁

- 按改动落点选最小门禁（参考 `verify-pick`）。
- 多层重构：`pnpm verify:standard`。
- 涉及协议：`pnpm audit:protocol`。
- 涉及运行时边界：`pnpm audit:boundaries`、`pnpm proof:server-runtime-boundaries`。
- 涉及客户端协议消费：`pnpm proof:client-s2c-consumption`。

## 输出要求

交付说明里写：拆了什么、保留了哪些薄编排、行为是否完全等价、跑了哪些门禁、是否动了 smoke / proof / audit / bench、是否有未覆盖的回归风险。
