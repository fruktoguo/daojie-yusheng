# 数值设计文档

## 1. 目标

本文用于固定《道劫余生》的数值类型体系，重点解决三件事：

- 固定基础六维
- 一次性定稿具体属性主表，避免后续频繁改协议和存档结构
- 固定非线性数值的换算方式、显示方式和境界驱动方式

本文先定“类型”和“字段”，不在本阶段锁死最终系数表。

## 2. 总原则

- 六维是基础成长输入，不直接参与最终战斗结算
- 战斗、资源、光环、冷却等系统统一读取“具体属性”
- 具体属性优先保持稳定；后续做新玩法时尽量复用已有属性，而不是新增顶级字段
- 所有需要递减收益的属性，统一采用 `x / (x + 基数)` 的 `RatioValue` 思路
- UI 默认显示原始数值；悬停时显示对应的实际百分比或实际效果
- 当前项目的成长主轴不是等级，而是境界；动态数值系统中的基准值和换算基数由境界驱动
- 实现层必须优先保证服务端热路径性能，不使用每个属性一个对象的响应式数值容器

## 3. 高性能实现原则

运行时实现不直接照搬 `YuoValue.cs` 的“每属性一个对象 + 多字典 + 每次修改立即重算”模式，而采用更适合 Web MMO 服务端的平面结构。

固定原则如下：

- 使用平面数值结构保存最终具体属性，不为每个属性创建类实例
- 使用固定键列表和直接属性访问，不在热路径里使用 `Object.keys()`、动态反射或递归合并
- 属性修改先收集，再批量重算，避免一次装备切换触发几十次重复重算
- `RatioValue` 只在最终结算或显示时换算，不做级联响应式更新
- 热路径上的 Buff、装备、境界叠加统一走“可变目标对象 + 原地累加”方式，减少临时对象和 GC
- 客户端只需要最终值和显示所需换算结果，不需要完整 modifier 图

设计层允许按分类分组展示，但运行时内部实现允许拍平成固定字段对象，以换取更低的常数开销。

当前共享实现入口约定为：

- `packages/shared/src/numeric.ts`：平面数值类型、固定键表、原地累加函数、`RatioValue` 工具、灵力输出速率计算函数

## 5. 具体属性类型

具体属性不再简单分“整数”和“百分比”，而是固定分为 4 种数值类型。

### 5.1 固定值属性 `flat`

直接参与结算的普通数值，不经过百分比换算。

适用场景：

- 生命值
- 灵力值
- 物理攻击
- 法术攻击
- 物理防御
- 法术防御

### 5.2 感知比率属性 `ratio_value`

属性本体以原始数值存储，结算时通过 `RatioValue` 换算成实际百分比或实际效果。

基础公式：

```text
实际比例 = x / (x + 基数)
```

负值减益时：

```text
实际比例 = -x / 基数
```

适用场景：

- 冷却速度
- 闪避
- 暴击
- 五行减伤
- 破招
- 化解

说明：

- 这类属性在面板中默认显示原始值
- 悬停时显示“按当前境界换算后的实际百分比”
- 换算基数不是固定死值，而是允许按境界配置

### 5.3 直接比例属性 `rate_bp`

直接以万分比表示的比例值，不走 `RatioValue` 递减曲线。

存储约定：

- `10000 = 100%`
- `2500 = 25%`

适用场景：

- 暴击伤害
- 五行增伤
- 自动回复百分比
- 光环消耗减免
- 光环效果增强

### 5.4 限流属性 `throughput`

不直接转成百分比，而是作为每 tick 的资源吞吐上限，超出上限时触发额外代价。

适用场景：

- 灵力输出速率

## 7. 非线性换算规则

参考 `参考/万化修仙/万界轮回者/YuoValue数值系统文档.md` 中的 `RatioValue` 思路，当前项目统一采用以下规则。

### 7.1 统一公式

对于所有 `ratio_value` 属性：

```text
actualRatio(value, divisor) = value / (value + divisor)
```

如果属性被压成负值：

```text
actualRatio(value, divisor) = -value / divisor
```

### 7.2 适用属性

当前固定使用 `RatioValue` 的属性有：

- `dodge`
- `crit`
- `elementDamageReduce.*`
- `breakPower`
- `resolvePower`
- `cooldownSpeed`
- `moveSpeed`

### 7.3 UI 显示规则

客户端显示分两层：

- 面板主文本：显示原始属性值
- 悬停说明：显示当前境界下换算后的实际百分比

示例：

```text
冷却速度：120
悬停：当前境界下实际冷却缩减 54.5%
```

```text
火行减伤：80
悬停：当前境界下实际火行减伤 44.4%
```

### 7.4 境界基数

`RatioValue` 的基数不写死在属性本身，而是按“属性类型 × 境界”配置。

也就是说：

- 同样是 `80` 点闪避，在凡俗境和筑基境的实际效果可以不同
- 同样是 `120` 点冷却速度，在高境界下不应该过早逼近上限

设计意图：

- 用境界替代“等级”作为动态数值的缩放轴
- 低境界数值更敏感，高境界数值更耐堆叠

## 8. 灵力输出速率专用规则

`maxQiOutputPerTick` 不参与普通百分比换算，而是用来计算“本 tick 的实际灵力消耗”。

### 7.1 定义

- 记 `L` 为 `maxQiOutputPerTick`
- 记 `C` 为本 tick 内计划消耗的总灵力
- 记 `overflow = max(C - L, 0)`

### 7.2 基本规则

- `C <= L` 时，按正常消耗
- `C > L` 时，未超出部分按原价结算
- 只有超出部分会触发额外消耗惩罚

### 7.3 分段惩罚

把超出部分按 `L × 20%` 为一个分段。

第 1 段超出量：

- 额外增加 `100%` 消耗
- 即该段按 `2x` 结算

第 2 段超出量：

- 额外增加 `200%` 消耗
- 即该段按 `3x` 结算

第 3 段超出量：

- 额外增加 `300%` 消耗
- 即该段按 `4x` 结算

以此类推。

### 7.4 结算公式

```text
segment = L × 0.2
```

```text
实际消耗 = 未超出部分
        + Σ(第 n 段超出量 × (n + 1))
```

其中：

- `未超出部分 = min(C, L)`
- `n` 从 `1` 开始计数

### 7.5 设计意图

- 允许玩家短时间爆发，不是硬禁止
- 超限爆发会快速变贵，防止低成本无限堆灵力瞬发
- 对持续耗蓝类系统也成立，如阵法、护体、光环

## 9. 境界驱动的动态数值系统

参考项目里是“等级驱动”，本项目改为“境界驱动”。

### 8.1 境界提供两类基准

每个境界阶段都应提供：

1. 核心属性基准值
   - 例如最大生命、最大灵力、攻防底座、灵力输出速率底座
2. 非线性属性换算基数
   - 例如闪避基数、暴击基数、冷却速度基数、五行减伤基数

### 8.2 完整计算顺序

当前版本已实装并定稿的顺序如下：

1. 境界基准模板
2. 六维派生贡献
3. 装备、功法、Buff、Debuff 修正
4. 得到未乘境界倍率前的 `finalAttrs`
5. 对 `finalAttrs` 施加 `realmLv` 全属性乘算倍率
6. 由乘算后的六维推导具体属性，并叠加具体属性修正
7. 对斗法与灵力相关具体属性再施加同一套 `realmLv` 乘算倍率
8. 对 `ratio_value` 属性再换算为实际百分比
9. 把实际百分比带入战斗公式或资源公式

### 8.2.1 境界乘算公式

当前版本统一采用以下规则：

- `realmLv` 每提升 1 级，全属性乘算 `1.1`
- 全属性包括：
  - 六维：体魄、神识、身法、根骨、悟性、气运
  - 斗法/灵力相关具体属性：生命、灵力、物攻、法攻、物防、法防、命中、闪避、暴击、暴伤、破招、化解、灵力输出、生命回复、灵力回复、冷却速度，以及五行增伤、五行减伤
- 通用公式：

```text
全属性倍率 = 1.1 ^ (realmLv - 1)
```

示例：

- `realmLv 1`：倍率 `1.0`
- `realmLv 2`：倍率 `1.1`
- `realmLv 3`：倍率 `1.21`
- `realmLv 4`：倍率 `1.331`

这意味着当前实现不是“只给固定境界面板加一点底座”，而是把 `realmLv` 直接作为整体成长轴。

### 8.2.2 境界差伤害公式

当前版本战斗结算额外引入“境界压制”：

- 攻击方每高防守方 1 个 `realmLv`，最终伤害乘算 `1.2`
- 攻击方每低防守方 1 个 `realmLv`，最终伤害乘算 `0.8`
- 同境界不额外修正

通用公式：

```text
若境界差 = attackerRealmLv - defenderRealmLv

境界差 > 0:
最终伤害乘区 = 1.2 ^ 境界差

境界差 = 0:
最终伤害乘区 = 1

境界差 < 0:
最终伤害乘区 = 0.8 ^ abs(境界差)
```

示例：

- 高 1 级：`1.2x` 伤害，也就是 `20%` 增伤
- 高 2 级：`1.44x` 伤害，也就是 `44%` 增伤
- 低 1 级：`0.8x` 伤害，也就是等价于对方承受侧 `20%` 压制
- 低 2 级：`0.64x` 伤害，也就是等价于对方承受侧 `36%` 压制

说明：

- 玩家对怪、怪对玩家、玩家对玩家统一走同一套境界差乘区。
- 怪物当前使用自身 `level` 作为对位的 `realmLv`。

### 8.3 为什么要按境界缩放

原因很直接：

- 这个项目没有传统等级长线
- 同境界内可以堆属性，但跨境界时基数也会变
- 这样既能保证低境界有成长体感，也能防止高境界过早堆满闪避、冷却、减伤

## 10. 实现约束

- `packages/shared` 里必须把六维和具体属性拆成两套明确结构
- 五行增伤和五行减伤固定为五子键对象，不要展开成临时散字段
- `breakPower` / `resolvePower` 是稳定代码键，后续改文案不改协议
- 客户端属性面板默认显示原始值，带 `ratio_value` 的属性必须提供悬停后的实际百分比
- 战斗结算只读取具体属性，不直接读取六维
- 灵力输出速率必须单独走专用消耗函数，不能混进普通 `rate_bp` 减免计算
- 冷却速度作用于所有拥有冷却时间的内容，包括技能、法器、阵法、光环切换等
- 运行时实现必须使用固定字段结构和批量重算，不得把每个属性做成带多张修改器字典的独立对象
- 服务端热路径优先使用原地修改和复用对象，避免频繁创建临时属性对象

## 11. 大数值边界策略

### 11.1 当前决策

针对中后期指数型成长，当前版本固定采用以下策略：

- 服务端战斗热路径继续使用 TypeScript `number`
- 运行时整数上限固定锁定为 `Number.MAX_SAFE_INTEGER`，即 `9,007,199,254,740,991`
- 超过该上限的最终战斗值统一按上限饱和处理，不继续增长
- 在角色仍处于前中期、实际数值尚未逼近 32 位边界前，暂不提前把整套协议和数据库切到 64 位

这样做的原因是：

- 当前预估的中后期最终攻击量级约为几十万亿，仍明显低于 `number` 的安全整数上限
- `number` 在 Node/NestJS 热路径中的加减乘除、四舍五入和比例结算成本最低
- 如果过早把整套战斗运行时切到 `BigInt`，会显著增加公式实现复杂度和维护成本

### 11.2 为什么当前不直接改成 BigInt

- `BigInt` 不能和 `number` 混算，现有公式链需要整体重写
- `Math.round`、`Math.floor`、`Math.ceil` 等逻辑不能直接复用
- 当前战斗系统中存在大量倍率、减伤、暴击、`RatioValue` 与百分比乘区，更适合继续使用 `number`
- 当前目标并不是冲击 `uint64` 全区间，而是在 `number` 安全整数范围内稳定运行

结论：

- 当前阶段不使用 `BigInt` 作为战斗热路径的默认数值类型
- 只有在未来设计上明确允许战斗核心数值超过 `Number.MAX_SAFE_INTEGER` 时，才进入 `BigInt` / 定点数 / 大数对象方案评估

### 11.3 当前真正的硬边界

虽然运行时 `number` 还能支撑更大的安全整数，但当前项目的首个硬边界并不是 JS，而是协议与持久化层中仍存在的 32 位整数字段。

当前已确认的风险点：

- 高频 Protobuf 同步消息里，多个生命、灵力、属性字段仍为 `sint32/uint32`
- 玩家存档里的 `hp`、`maxHp`、`qi` 仍为 PostgreSQL `int`

这意味着：

- 一旦这些字段实际值超过 `2,147,483,647` 或 `4,294,967,295`
- 即使服务端运行时还能继续算，协议编解码或数据库落盘也会先出现回卷、截断或越界风险

因此当前不改 64 位，并不表示后续不改；只是把迁移时机明确放到“接近边界时再做”，而不是在前几十级就过早升级整条链路。

### 11.4 延后迁移的规则

当前采用“先收口、后迁移”的策略。

要求如下：

- 运行时统一以 `Number.MAX_SAFE_INTEGER` 作为战斗整数极限
- 在关键结算、发包、落盘前保留统一的上限钳制与风险日志
- 当常规角色成长开始逼近 32 位协议或数据库上限时，再启动 64 位迁移

64 位迁移启动后，优先顺序如下：

1. Protobuf 中与战斗、资源、属性同步直接相关的 `sint32/uint32` 字段改为 `sint64/uint64`
2. PostgreSQL 中需要承载大数值的角色字段改为 `bigint`
3. Node 侧统一处理 `bigint` / 64 位字段的读写转换
4. 客户端统一补齐大数字显示格式，例如 `万 / 亿 / 万亿`

说明：

- PostgreSQL 没有原生 `uint64`，数据库侧统一使用 `bigint`
- 因为运行时最终仍锁在 `Number.MAX_SAFE_INTEGER` 以内，所以数据库 `bigint` 的容量已经足够
- 后续若真的出现设计上要突破 `number` 安全整数极限的需求，再单独升级战斗核心数值表示

## 12. 当前定稿结论

本轮定稿的核心结论如下：

1. 基础六维固定为：体魄、神识、身法、根骨、悟性、气运
2. 具体属性主表固定包含：
   - 生命值、灵力值
   - 物理攻击力、法术攻击力
   - 物理防御力、法术防御力
   - 命中、闪避、暴击、暴击伤害
   - 五行增伤、五行减伤
   - 破招、化解
   - 最大灵力输出速率
   - 生命自动回复、灵力自动回复
   - 冷却速度
   - 光环消耗减免、光环效果增强
3. 递减收益型属性统一采用 `RatioValue`
4. 动态数值系统的缩放轴从“等级”改为“境界”
5. 灵力输出速率使用单独的分段超限惩罚公式
6. `realmLv` 每级提供 `1.1` 的全属性乘算倍率
7. 跨境界战斗额外叠加境界差伤害乘区，规则为高一级 `1.2x`、低一级 `0.8x`
8. 战斗热路径当前继续使用 `number`，最终整数上限锁定为 `Number.MAX_SAFE_INTEGER`
9. 在实际数值逼近 32 位协议 / 存档边界前，暂不提前迁移到 64 位协议与数据库字段

下一步如果继续细化，应按这个顺序推进：

1. 六维到具体属性的派生权重表
2. 命中 / 闪避 / 暴击 / 破招 / 化解的正式战斗公式
3. 五行伤害与五行减伤的结算顺序
4. 装备词条格式
5. 协议与前后端数据结构改造
