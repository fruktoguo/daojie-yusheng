# 地图系统架构文档

更新时间：2026-04-18

这份文档回答三件事：

1. `next` 当前地图系统实际上是怎么工作的
2. 为了支持后期上千地图实例、主地图分线、公共随机秘境、宗门独立地图、洞府独立地图，以及“跨图寻路只能在主地图中进行”这组目标，终局架构应该长成什么样
3. 为了兼容后期分布式服务器架构，地图系统现在就必须提前固定哪些边界

这不是单点功能说明，而是服务端 `runtime + persistence + routing + protocol + cluster-compat` 的统一设计口径。

文中如果出现“实例真源”“玩家位置真源”“overlay 真源”，指的是地图系统的运行时与持久化真源；这不替代 `docs/next-plan/02-pin-next-sources-and-protocol.md` 中“`packages/server/data/maps/*` 是静态地图模板唯一真源”的口径。

## 0. 执行摘要

这份文档的核心结论只有 5 条：

- 模板不是世界，实例才是世界
- 玩家正式落点必须是 `instanceId`
- 主地图导航图是全局跨图唯一导航层
- 非主地图实例只做局部寻路，不进入全局导航图
- 动态地图能力必须走 `template + overlay`，不能靠复制整张模板扩容

如果只用一句话概括终局目标，就是：

“把当前以公共地图为中心的设计，升级成以实例目录、实例路由、实例锚点和活跃实例调度为中心的设计。”

### 0.1 当前状态一句话

当前 `next` 已经有地图实例运行时，但整体仍是“少量公共实例”架构，而不是“大量实例”架构。

### 0.2 终局状态一句话

终局应当是：

- `MapTemplate` 负责静态底图
- `InstanceCatalog` 负责实例真源
- `InstanceRuntime` 负责实例内 tick
- `InstanceAnchor` 负责入口
- `MainWorldNavigationGraph` 负责跨图导航
- `Overlay` 负责实例级动态改造

如果你只想快速抓结论，建议按这个顺序读：

## 0.4 术语速查

### 0.4.1 模板、实例、锚点

| 术语 | 含义 | 是否长期真源 |
| --- | --- | --- |
| `MapTemplate` | 静态地图底图和基础配置 | 是 |
| `InstanceCatalog` | 实例目录和元信息真源 | 是 |
| `InstanceRuntime` | 实例在内存中的运行对象 | 否 |
| `InstanceAnchor` | 把实例挂到某个宿主实例坐标上的入口对象 | 是 |
| `Overlay` | 实例对模板的运行期动态覆盖层 | 是 |

### 0.4.2 导航层级

| 层级 | 作用 | 是否允许跨图 |
| --- | --- | --- |
| 局部实例寻路 | 当前实例内走到目标格或本地入口 | 否 |
| 主地图导航图 | 跨图全局路由，只在主地图图层中工作 | 是 |

### 0.4.3 实例类型

| 类型 | 含义 | 是否进入全局导航图 |
| --- | --- | --- |
| `main_line` | 主地图分线实例 | 是 |
| `realm_public` | 公共随机秘境实例 | 否 |
| `sect_private` | 宗门独立实例 | 否 |
| `dwelling_private` | 洞府独立实例 | 否 |

## 1. 目标与约束

### 1.1 终局目标

- 支持上千地图实例
- 主地图支持多分线
- 公共秘境支持随机生成地形
- 每个宗门拥有独立地图
- 每个洞府拥有独立地图
- 宗门、洞府、秘境都必须在主地图留下传送点
- 洞府允许同时在主地图和所属宗门留下入口锚点
- 跨图寻路只能在主地图中进行
- 地图实例可以按需持久化，支持重连、重启后恢复

### 1.2 强制约束

- 地图模板不是运行时真源，实例才是运行时真源
- 玩家正式落点必须是 `instanceId`，不能只靠 `templateId`
- 非主地图实例不参与全局跨图导航图
- 洞府允许双入口，但只有主地图入口参与全局跨图导航
- 所有会改变世界状态的行为必须仍然走服务端受控 tick 流程
- 动态地形不能通过复制整张模板解决，必须走 `template + overlay`

### 1.3 明确不采用的方向

为了避免后期返工，以下方向直接排除：

- 不把所有实例拼成一个全局寻路图
- 不把宗门图、洞府图、秘境图当成全局导航层的一部分
- 不把玩家落点只绑定到 `templateId`
- 不把实例真源继续留在单进程内存
- 不把动态地图做成“每个实例复制一整张完整地图 JSON”

## 2. 当前实现概览

### 2.1 当前主链

当前 `next` 已经有“地图实例”概念，但整体仍偏向“少量公共实例”模式：

- 服务启动时为每张地图模板创建一个公共实例
- 公共实例 ID 统一为 `public:${templateId}`
- 玩家接入地图时默认进入公共实例
- 玩家跨图、复生、GM 查询主链都默认按公共实例处理

相关实现：

- `packages/server/src/runtime/map/map-template.repository.ts`
- `packages/server/src/runtime/instance/map-instance.runtime.ts`
- `packages/server/src/runtime/world/world-runtime.service.ts`
- `packages/server/src/network/world-session-bootstrap.service.js`

### 2.2 当前已具备的能力

- 地图模板加载与运行时实例分离
- 每个实例独立维护：
  - 玩家占位
  - 妖兽、NPC、容器、地面掉落
  - AOI 视图
  - 实例 tick 与 world revision
- 协议层已经能下发 `instanceId`
- 地图持久化已经按 `instanceId` 存快照
- 实例元数据已经预留：
  - `ownerPlayerId`
  - `ownerSectId`
  - `partyId`

### 2.3 当前实现的边界判断

可以把当前状态理解成：

- `MapInstanceRuntime` 已经是合格的“实例内世界容器”
- `WorldRuntimeService` 仍然是面向“少量公共实例”的总编排
- `next` 还没有正式实例目录、正式实例路由、正式实例锚点系统

也就是说，当前短板不是“没有实例”，而是“实例还没有成为系统主语”

### 2.4 当前主要缺口

- 实例目录没有正式数据库真源，实例主要仍靠内存维护
- 玩家位置持久化只存 `templateId`，不存 `instanceId`
- `routeDomain` 还没有进入实例路由主链
- 传送门主要仍按 `targetMapId` 工作，不按 `targetInstanceId` 工作
- tick 调度当前每帧线性扫描全部实例
- GM / runtime 工具链仍默认 `public:${mapId}`
- 动态地形还没有完整迁移到 `next`

### 2.5 当前与终局的差异总表

| 维度 | 当前 `next` | 终局目标 |
| --- | --- | --- |
| 世界主语 | 公共地图实例为主 | 一切运行时都以实例为主 |
| 玩家落点 | 以 `templateId` 为主 | 以 `instanceId` 为主 |
| 跨图入口 | 主要按 `targetMapId` | 通过锚点解析目标实例 |
| 全局导航 | 尚未正式分层 | 只在主地图导航图中进行 |
| 实例真源 | 以内存和局部快照为主 | 以实例目录数据库真源为主 |
| 调度模型 | 全实例扫描 | 活跃实例调度 |
| 动态地形 | 能力未完整迁移 | `template + overlay` |
| 分布式兼容 | 单进程友好 | 接入层、执行层、目录层可拆分 |

## 3. 核心概念

### 3.1 MapTemplate

静态地图模板。

职责：

- 地块底图
- 基础门点
- 基础 NPC / landmark / container 布局
- 地图宽高
- 路网域元数据
- 作为实例运行时的基础底盘

模板不是玩家真正进入的世界，也不是长期实例状态真源。

### 3.2 InstanceCatalog

实例目录真源。

职责：

- 记录“有哪些实例存在”
- 记录实例类型、归属、入口策略、生命周期和恢复所需元信息
- 为实例恢复、GM 查询、玩家重登、跨图路由提供正式查询面

没有 `InstanceCatalog`，就无法在重启后稳定恢复大量宗门图、洞府图和随机秘境。

### 3.3 InstanceRuntime

实例运行时对象。

职责：

- 承载实例内 tick
- 处理占位、移动、战斗、交互、怪物、掉落、AOI
- 提供实例内只读视图与增量投影来源

这层对应当前的 `MapInstanceRuntime`，但后续不能再把实例目录、实例生命周期和实例恢复继续堆在 `WorldRuntimeService` 里。

### 3.4 InstanceAnchor

实例入口锚点。

职责：

- 把“实例”挂接到“某个宿主实例上的某个坐标”
- 定义入口可见性、权限和目标实例解析规则

锚点是终局跨图系统的关键，不能继续只靠模板里的 portal 直接表达所有实例跳转。

### 3.5 MainWorldNavigationGraph

主地图导航图。

职责：

- 作为全局跨图寻路的唯一导航图层
- 只包含主地图实例中的跨图锚点和主地图道路关系
- 不扫描宗门图、洞府图、秘境图内部寻路状态

### 3.6 Overlay

实例动态覆盖层。

职责：

- 表达该实例对模板的运行时改造
- 包括动态地形、额外标记、可持久化环境变化

这是支持宗门挖洞、洞府装修、秘境地形运行期变化的正式载体。

## 4. 终局实例模型

### 4.1 实例类型

建议终局固定 4 类实例：

- `main_line`
  - 主地图分线实例
  - 属于主世界 cluster
  - 参与全局导航图
- `realm_public`
  - 公共秘境实例
  - 可按 seed 生成
  - 不参与全局导航图，只在主地图上暴露入口
- `sect_private`
  - 宗门独立实例
  - 归属某个宗门
  - 不参与全局导航图，只在主地图上暴露入口
- `dwelling_private`
  - 洞府独立实例
  - 归属玩家或宗门
  - 允许有主地图入口和宗门本地入口

### 4.2 实例 ID 规则

建议一开始就固定实例 ID 语义，避免后续迁移：

- `main:{clusterId}:{lineId}`
- `realm:{realmPoolId}:{instanceNo}`
- `sect:{sectId}`
- `dwelling:{ownerType}:{ownerId}:{slotId}`

实例 ID 必须稳定、可持久化、可恢复、可跨节点路由。

### 4.3 实例目录建议字段

建议正式真源至少包含：

- `instanceId`
- `instanceKind`
- `templateId`
- `generatorId`
- `generatorVersion`
- `generatorSeed`
- `ownerType`
- `ownerId`
- `routeDomain`
- `persistent`
- `status`
- `createdAt`
- `lastActiveAt`
- `tickPolicy`
- `shardKey`

说明：

- `generator*` 主要给随机秘境
- `owner*` 主要给宗门图和洞府图
- `tickPolicy` 用于高频 / 低频 / 休眠调度
- `shardKey` 为未来多执行器或多节点预留

### 4.4 设计判断

终局不应再存在“默认公共实例模型”与“少量特例实例模型”两套思路。

统一模型应是：

- 所有地图运行时都先是实例
- 只是不同实例类型有不同的 owner、入口策略、持久化策略和调度策略

这样后续新增玩法场景时，只是扩展实例类型和路由规则，而不是重写地图系统。

## 5. 锚点系统

### 5.1 锚点角色

建议锚点显式区分角色：

- `global_entry`
- `sect_entry`
- `instance_exit`
- `return_entry`

### 5.2 锚点宿主规则

- `global_entry`
  - 宿主实例必须是主地图分线实例
  - 进入全局跨图导航图
- `sect_entry`
  - 宿主实例必须是某个宗门实例
  - 不进入全局跨图导航图
- `instance_exit`
  - 宿主实例是当前实例自身
  - 用于离开当前实例
- `return_entry`
  - 用于离开副本后回归主世界或回归宗门

### 5.3 各实例允许的外部入口

- `main_line`
  - 不需要外部入口
- `realm_public`
  - 仅允许主地图 `global_entry`
- `sect_private`
  - 仅允许主地图 `global_entry`
- `dwelling_private`
  - 允许主地图 `global_entry`
  - 可选允许所属宗门的 `sect_entry`

### 5.4 洞府双入口规则

洞府是唯一允许多外部入口的实例类型。

但要区分：

- 主地图洞府入口：
  - 进入全局跨图导航图
  - 全世界都只能通过它作为全局跨图目标
- 宗门洞府入口：
  - 仅在玩家已经身处所属宗门实例时可用
  - 不进入全局跨图导航图

这条规则保证：

- 玩家体验上洞府可以在主世界和宗门中留下入口
- 架构上仍保持“跨图寻路只能在主地图中进行”

### 5.5 锚点设计原则

锚点设计要满足三条：

- 锚点表达“从哪里进”，实例表达“进到哪里”
- 锚点可以多于实例，但不能反过来让实例依赖某个固定 portal tile
- 锚点必须带权限和可见性，而不是只带目标坐标

## 6. 导航与跨图规则

### 6.1 总原则

跨图导航永远拆成两层：

1. 当前实例内局部寻路
2. 主地图导航图上的全局跨图路由

绝不允许：

- 直接把所有实例拼成一个全局 A*
- 让宗门图、秘境图、洞府图内部锚点参与全局路网图

### 6.2 跨图导航流程

统一流程：

1. 解析目标实例
2. 如果当前已在目标实例内
3. 则只做当前实例内局部寻路
4. 如果当前不在目标实例内
5. 则先解析目标实例的 `global_entry`
6. 若当前不在主地图
7. 则先在当前实例内寻路到出口锚点回主地图
8. 回到主地图后
9. 再在主地图导航图上寻路到目标实例 `global_entry`
10. 到达锚点后传入目标实例

### 6.3 洞府例外语义

如果玩家已经在所属宗门实例中：

- 允许直接寻路到该洞府在宗门内的 `sect_entry`
- 这属于“当前实例内局部入口导航”
- 不属于“全局跨图导航”

如果玩家不在所属宗门实例中：

- 仍然只能通过主地图的 `global_entry` 去洞府

### 6.4 routeDomain 的正式职责

当前 `routeDomain` 已经存在于 shared/map 模型，但后续应正式进入运行时实例路由：

- `system`
  - 主世界 / 系统公共图
- `sect`
  - 宗门域实例
- `personal`
  - 洞府或个人域实例
- `dynamic`
  - 动态秘境、活动场景、随机实例

`routeDomain` 不应只停留在编辑器元数据层，必须成为 `InstanceRoutingService` 的输入条件之一。

### 6.5 总结

跨图系统最终只做两件事：

- 决定“该去哪个实例”
- 决定“该从主地图哪个全局入口进去”

而不会做：

- 直接跨实例寻路
- 直接在宗门图或秘境图里做全局图路由

## 7. 持久化设计

### 7.1 真源分层

地图系统持久化至少分成 4 类真源：

1. 模板真源
   - 地图模板 JSON
2. 实例目录真源
   - 哪些实例存在
3. 实例快照真源
   - 实例环境与运行期可恢复状态
4. 玩家位置真源
   - 玩家当前落在哪个实例

### 7.2 玩家位置真源

玩家正式落点必须升级为：

- `instanceId`
- `templateId`
- `x`
- `y`
- `facing`

`templateId` 只做冗余索引和恢复 fallback，`instanceId` 才是正式落点。

### 7.3 实例快照

实例快照建议拆成两层：

- `InstanceSnapshot`
  - 低频整体快照
- `InstanceOverlay`
  - 高价值长期运行改造层

其中：

- 公共主地图和宗门图建议长期持久化
- 洞府建议长期持久化
- 公共秘境可按活动策略决定持久化周期

### 7.4 动态地形

动态地形不能只靠临时 `tileDamage` 记录。

正式设计应改成：

- 模板底图
- 覆盖层地形状态
- 覆盖层决定：
  - tile type
  - walkable
  - blocksSight
  - durability
  - transformed state

后续移动、视野、寻路、同步、持久化都基于“模板 + overlay”结果。

### 7.5 持久化分工

建议明确这四类状态的职责边界：

| 状态 | 真源 | 说明 |
| --- | --- | --- |
| 模板定义 | 文件 / 内容仓库 | 静态底图与基础配置 |
| 实例目录 | 数据库 | 哪些实例存在、归谁、在哪里恢复 |
| 实例快照 | 数据库 | 环境和低频整体恢复态 |
| 动态覆盖层 | 数据库 | 地形改造、实例级长期变化 |

## 8. 秘境生成

### 8.1 原则

随机秘境的正式真源不应是整张生成后的地图 JSON。

建议真源为：

- `generatorId`
- `generatorVersion`
- `seed`
- `params`

实例启动时：

1. 根据 `generator + seed` 重建基础地形
2. 再加载该实例的 overlay 和快照

### 8.2 好处

- 减少数据库体积
- 支持大量公共秘境实例
- 可随生成器版本切换迁移策略
- 支持“同 seed 可重复恢复”

### 8.3 与宗门 / 洞府的差异

随机秘境与宗门 / 洞府的核心差异是：

- 宗门 / 洞府更强调长期稳定 owner 和长期 overlay
- 秘境更强调生成器、seed、活动窗口和弹性生命周期

因此它们虽然都属于实例，但持久化重点不同，不应该被塞进同一种粗糙快照模型里。

## 9. Tick 与实例调度

### 9.1 当前问题

当前世界帧推进会线性遍历全部实例：

- 规划实例步数
- 推进 tick
- 处理 transfer / monster actions / player advance

这在少量公共实例时可以接受，但不适用于上千实例。

### 9.2 终局调度层级

实例调度应分 4 档：

- `hot`
  - 有玩家、战斗、倒计时任务
  - 正常 tick
- `warm`
  - 短时空闲
  - 低频 tick
- `sleeping`
  - 无活动
  - 不进主循环
- `cold_unloaded`
  - 已从内存卸载
  - 只保留目录和快照真源

### 9.3 唤醒条件

- 玩家进入
- 传送目标命中
- GM 查询或观测
- 持久化恢复
- 倒计时任务到期
- 活动秘境刷新

### 9.4 设计要求

- tick 主循环应只围绕活跃实例集合推进
- 不能继续按“全目录扫描”做计划
- 实例生命周期必须与 runtime 调度解耦

### 9.5 设计判断

从“少量实例”到“上千实例”的关键不是继续优化 `tickOnce()` 本身，而是先把“谁该 tick”这件事从线性扫描改成实例调度系统。

---

## 10. 协议与工具链要求

### 10.1 协议

后续协议必须统一实例视角：

- `MapEnter` 已有 `iid`
- 后续：
  - 玩家位置类数据应稳定包含 `instanceId`
  - 导航结果应能表达“目标实例入口”
  - GM 查询应支持按 `instanceId` 定位

### 10.2 GM / 运维

GM 与运维链路不能继续把 `public:${mapId}` 当默认实例。

需要新增：

- 实例列表查询
- 按类型 / owner / status / routeDomain 过滤
- 指定 `instanceId` 的运行态查看
- 指定实例的持久化状态查看
- 指定实例的激活 / 休眠 / 卸载操作

### 10.3 协议与工具链的统一口径

后续任何面向地图系统的管理接口，都应优先围绕：

- `instanceId`
- `instanceKind`
- `owner`
- `runtimeStatus`

而不是继续围绕：

- `mapId`
- `public:${mapId}`

否则工具链会把旧架构假设重新带回主链。

## 11. 面向分布式架构的兼容要求

### 11.1 总原则

如果后期目标包含分布式服务器架构，那么地图系统现在就必须遵守下面这条硬约束：

- 任何长期真源都不能只存在单进程内存里
- 任何“玩家在哪个实例”“实例在哪个执行器”这类路由信息都必须可外部查询
- 会话接入层、实例执行层、实例目录层必须是可拆分的

当前仓库里很多实现仍然偏单进程世界服思路，这对当前阶段可接受，但终局设计不能继续沿这个方向扩张。

### 11.2 终局分层

为了兼容未来分布式，建议把服务端拓扑在设计上拆成 5 层：

1. `Gateway / Session Access Layer`
   - 负责 HTTP / Socket 接入
   - 不持有世界实例真源
   - 只负责玩家鉴权、会话绑定、请求转发、下行回推

2. `Instance Router`
   - 负责回答：
     - 某玩家当前在哪个实例
     - 某实例当前在哪个执行器 / 分片
     - 某入口锚点应解析到哪个实例

3. `Instance Executor`
   - 真正跑实例 tick
   - 负责实例内状态变化、AOI、怪物、掉落、交互和增量同步源
   - 一个执行器可以承载很多实例

4. `Catalog / Persistence Layer`
   - 实例目录真源
   - 玩家位置真源
   - 实例快照真源
   - 锚点真源

5. `Cluster Coordination Layer`
   - 实例调度、分片分配、迁移、故障接管、租约和心跳

### 11.3 现在就要固定的边界

为了避免后期返工，建议现在就固定以下边界：

- `WorldRuntimeService` 不能既是实例执行器，又是实例目录真源
- `WorldSessionService` 不能继续被默认视为长期可扩展的最终会话真源
- 玩家正式落点不能只写入运行时对象，必须进入数据库
- 协议层不能默认“接入的那台机器一定同时持有目标实例”

### 11.4 分布式下的实例所有权

每个实例都应有明确的“当前执行所有者”概念。

建议实例目录至少再补这些运行期字段：

- `assignedNodeId`
- `leaseToken`
- `leaseExpireAt`
- `runtimeStatus`
- `recoveringFromSnapshot`

含义：

- `assignedNodeId`
  - 当前由哪个执行节点承载该实例
- `leaseToken`
  - 防止旧节点脑裂继续写实例
- `leaseExpireAt`
  - 节点失联后允许其他节点接管
- `runtimeStatus`
  - `active / sleeping / migrating / recovering / offline`
- `recoveringFromSnapshot`
  - 当前是否在恢复流程中

### 11.5 会话路由

分布式下，玩家连接和实例执行通常不会在同一进程。

所以会话系统要升级成两段：

- `playerSession`
  - 玩家当前接在哪个 gateway
- `playerLocation`
  - 玩家当前在哪个 `instanceId`

最少要能回答：

- `playerId -> gatewayNodeId`
- `playerId -> instanceId`
- `instanceId -> executorNodeId`

这样：

- 玩家发移动请求时，gateway 才知道转发给哪个实例执行器
- 实例产生 AOI / notice / kick 时，执行器才知道通过哪个 gateway 推回客户端

### 11.6 协议要求

为了兼容分布式，下行和上行协议最好都具备实例上下文语义。

建议长期约束：

- 进入地图时稳定携带 `instanceId`
- 任何需要服务端执行的实例内行为，最终都落到“某个实例执行器”上
- 不能假定 socket 所在进程一定能直接拿到实例内存对象

这不代表协议必须把 `instanceId` 暴露给所有上行包，但服务端内部转发模型必须按实例路由。

### 11.7 跨节点传送

跨图传送在分布式下应拆成显式阶段：

1. 当前实例执行器确认传送成立
2. 写入玩家目标落点真源
3. 通知目标实例执行器预接入或唤醒目标实例
4. 更新 `playerLocation`
5. 通知 gateway 切换玩家当前实例上下文
6. 下发新的 `MapEnter / Bootstrap`

这里的关键是：

- 传送不是“直接改内存引用”
- 而是“显式迁移玩家实例归属”

### 11.8 主地图分线与分布式

主地图分线天然适合做分布式切分。

建议：

- 每条 `main_line` 都是独立实例
- `clusterId` 只表示它们属于同一逻辑主世界
- 分线可以被分配到不同执行器
- 全局跨图导航图只需要逻辑共享，不要求所有线共用一份运行时对象

### 11.9 随机秘境与分布式

公共随机秘境最适合做弹性实例。

建议：

- `realm_public` 实例按负载或活动窗口动态创建
- 生成真源只保留 `generator + seed + params`
- 实例闲置后可休眠或卸载
- 活动结束后可归档或销毁

这样大量秘境不会把长期内存占满。

### 11.10 宗门 / 洞府与分布式

宗门图和洞府图属于“长期存在但低活跃”的实例。

建议：

- 默认允许休眠
- 玩家进入前唤醒
- 唤醒时从 catalog + snapshot + overlay 恢复
- 洞府与宗门都允许长期分配在非主世界执行器上

这样主世界执行器不会被大量低活跃私有实例拖垮。

### 11.11 事件总线要求

一旦进入分布式，至少需要内部事件总线或消息总线承担这些事件：

- 实例唤醒
- 实例迁移
- 玩家切换实例
- 玩家断线
- gateway 路由更新
- AOI / notice / mail / kick 下发请求

不要求现在就引入消息队列产品，但设计上不能假定“函数调用就是跨层通信方式”。

### 11.12 当前设计里最容易卡住分布式的点

当前最需要提前规避的风险有：

- 线性扫描全部实例做 tick 计划
- 单进程内存持有全部实例真源
- `public:${mapId}` 被视为唯一主路径
- 玩家位置不存 `instanceId`
- 工具链和 GM 查询默认只看公共实例
- 实例路由信息只存在运行时对象，不可外部查询

这些如果现在不切边界，后面上分布式时会演变成系统级返工。

### 11.13 设计结论

为了兼容未来分布式服务器架构，地图系统现在就应被设计成：

- `实例目录可外查`
- `实例执行可迁移`
- `玩家位置可恢复`
- `接入层与执行层可拆分`
- `跨图迁移走显式状态流转`
- `主地图导航图是逻辑层，不依赖单机共享内存`

只要这套边界先立住，后面从单进程扩到多执行器 / 多节点时，主要会是部署和协调层演进，而不是地图系统重写。

### 11.14 当前阶段的现实落点

当前阶段不要求马上把系统拆成多节点，但要求现在的接口和真源设计不能把未来多节点直接堵死。

也就是说：

- 可以先单进程实现
- 但不能再写成“只有单进程才能工作”的最终口径

## 12. 推荐模块拆分

为了避免后期再次大改，建议现在就按终局边界拆模块：

- `InstanceCatalogService`
- `InstanceRoutingService`
- `InstanceLifecycleService`
- `InstanceSchedulerService`
- `InstanceSnapshotService`
- `DynamicTerrainOverlayService`
- `MainWorldNavigationGraphService`
- `InstanceAnchorService`

当前 `WorldRuntimeService` 最终应退化为薄编排层，而不是继续承担：

- 实例目录真源
- 实例创建
- 实例恢复
- 全量实例调度
- 跨图实例路由
- 持久化组装

### 12.1 推荐职责归位

建议最终职责收口如下：

| 模块 | 最终职责 |
| --- | --- |
| `InstanceCatalogService` | 实例目录真源 |
| `InstanceRoutingService` | 从目标语义解析实例 |
| `InstanceAnchorService` | 管理入口锚点 |
| `InstanceLifecycleService` | 创建、激活、休眠、销毁 |
| `InstanceSchedulerService` | 活跃实例调度 |
| `InstanceSnapshotService` | 实例快照组装与恢复 |
| `DynamicTerrainOverlayService` | 实例级动态地图覆盖层 |
| `MainWorldNavigationGraphService` | 全局跨图导航逻辑 |

## 13. 迁移顺序建议

### 阶段 1：实例真源收口

- 新增实例目录真源
- 玩家位置正式改存 `instanceId`
- GM / runtime 查询改成实例视角

### 阶段 2：实例路由收口

- 引入 `InstanceRoutingService`
- `routeDomain` 正式进入运行时路由
- 传送和进入链改为“先解析实例，再落点”

### 阶段 3：锚点系统收口

- 将实例入口抽象为 `InstanceAnchor`
- 主地图 `global_entry` 进入全局导航图
- 宗门 / 洞府本地锚点不进入全局导航图

### 阶段 4：调度系统收口

- 从全实例扫描改为活跃实例调度
- 增加实例生命周期状态机

### 阶段 5：动态地形迁移

- 把 `legacy` 动态地块能力迁到 `next`
- 改成 `template + overlay`
- 接入移动、视野、寻路和持久化

### 阶段 6：随机秘境生成

- 接入 `generator + seed` 真源
- 补实例生成与恢复策略

### 阶段 7：分布式兼容边界收口

- 补 `instance -> node` 路由元信息
- 拆分 gateway 路由与实例执行边界
- 为跨节点传送预留显式迁移流程

---

## 14. 当前正式结论

当前 `next` 已经有足够的实例运行时底子，能够作为终局架构的基础，但还不能直接支撑：

- 上千实例调度
- 宗门 / 洞府 / 秘境的正式实例真源
- 以主地图为唯一全局跨图导航层的完整模型
- 基于 overlay 的长期动态地形

如果要避免后期再次大改，应立即把设计中心从“地图模板 + 公共实例”切换到：

- `模板是底图`
- `实例是世界`
- `锚点是入口`
- `主地图导航图是全局跨图唯一导航层`
- `overlay 是动态改造真源`

一句话版本：

地图系统后续不应围绕“地图文件怎么加载”演进，而应围绕“实例怎么被创建、路由、调度、恢复和迁移”演进。

这套设计能一次性覆盖：

- 主地图分线
- 公共随机秘境
- 宗门独立地图
- 洞府独立地图
- 洞府双入口
- 上千实例的调度与恢复
