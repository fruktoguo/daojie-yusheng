# 战斗链路商业化重构计划

> 本文用于规划当前生产主线战斗系统从“多条分散链路”收敛到“统一服务端权威战斗主链路”的实施方案。
>
> 本文是开发计划，不表示当前主线已经完成。目标设计详见 `docs/开发/链路/战斗链路.md`。

---

## 目标结论

战斗系统最终应收敛为一条主链路：

```text
战斗意图
-> 统一 CombatAction
-> 权威快照
-> 锚点/范围/目标解析
-> 统一合法性判断
-> 资源/冷却/吟唱
-> 统一命中与效果结算
-> 按目标类型应用结果
-> 表现/通知/审计/诊断
```

普攻、技能、怪物普攻、怪物技能、吟唱完成都只是这条链路的不同输入形态。玩家、怪物、地块、阵法、容器这些目标可以在“结果应用”阶段分支，但不能在“能不能打、怎么命中、有没有通知”阶段拆成多套系统。

核心收益：

- 普攻和技能共用目标解析、敌我关系、范围、视线、命中、通知。
- 玩家和怪物只是发起者类型不同，不再各自维护一套攻击结算。
- 怪物 AI 只生成战斗意图，不直接结算伤害。
- 地块、阵法、容器保留各自真源写入，但接入统一合法性和结果事件。
- 任何 action 被拒绝都有结构化原因，不允许无原因静默 `return`。
- 进入结算后，闪避、免疫、未造成伤害也必须产生结果事件。

---

## 当前问题

- [ ] 玩家普攻、玩家技能、怪物普攻、怪物技能仍是多条分支链路。
- [ ] 普攻没有被建模为内置 `CombatAction`，而是拥有独立服务和目标分支。
- [ ] 玩家技能有自己的范围目标收集，怪物技能有另一套目标收集。
- [ ] 怪物 action 应用阶段存在多个静默返回点，排障时难以判断 action 停在哪一步。
- [ ] 敌我关系核心有复用，但包装和调用分散在普攻、技能、自动战斗等多个位置。
- [ ] 命中伤害底层公式有复用，但进入公式之前的合法性和目标收集不统一。
- [ ] 地块、阵法、容器和生命体目标的链路分支过早，难以保证通知和诊断口径一致。
- [ ] 吟唱开始、吟唱完成、即时技能之间缺少统一 action 生命周期。
- [ ] 当前验证更多覆盖单点 smoke，缺少统一战斗链路的矩阵回归。

## 分阶段计划

### 阶段 0：现状审计和行为冻结

- [ ] 列出玩家普攻入口、玩家技能入口、怪物普攻 action、怪物技能 action 的完整调用点。
- [ ] 列出所有直接调用 `resolveCombatHit` 的位置。
- [ ] 列出所有调用 `resolveCombatRelation` / `isHostileCombatRelationResolution` 的位置。
- [ ] 列出玩家技能和怪物技能各自的目标收集逻辑。
- [ ] 列出怪物 action 应用里的所有静默 `return`。
- [ ] 列出现有普攻、技能、地块、阵法、容器相关 smoke。
- [ ] 固化当前可接受玩法行为：伤害公式、闪避提示、死亡、掉落、地块破坏、阵法扣灵力、容器消耗。
- [ ] 给“预警格命中但没有战斗信息”补一个失败复现 smoke 或最小诊断 case。

验收：

- [ ] 现状审计文档或注释能指向每条旧链路。
- [ ] 迁移前后必须保持不变的玩法行为有明确清单。
- [ ] 至少有一个 smoke 能覆盖怪物吟唱技能的最终 action 产出和结算路径。

### 阶段 1：定义统一战斗数据结构

- [ ] 新增 `CombatActor` 类型，覆盖玩家、怪物、系统等发起者。
- [ ] 新增 `CombatAction` 类型，覆盖普攻、技能、怪物技能、系统动作。
- [ ] 新增 `CombatTarget` 类型，覆盖玩家、怪物、地块、阵法、容器、自身、空地。
- [ ] 新增 `CombatOutcome` 类型，表达命中、闪避、伤害、免疫、拒绝原因、应用结果。
- [ ] 新增 `CombatRejectReason` 枚举或常量，所有拒绝原因机器可读。
- [ ] 定义 `CombatActionPhase`：即时、吟唱开始、吟唱完成、取消。
- [ ] 定义 `CombatActionSource`：玩家输入、自动战斗、怪物 AI、系统触发。
- [ ] 明确类型所在边界：服务端运行时内部优先，不先扩散到协议层。

验收：

- [ ] 类型可覆盖当前所有旧链路目标。
- [ ] 普攻可以被表达为内置 `CombatAction`。
- [ ] 怪物技能 action 可以被表达为同一类 `CombatAction`。
- [ ] 拒绝原因不依赖中文文本判断。

### 阶段 2：引入战斗主链路编排器

- [ ] 新增薄编排服务，例如 `WorldRuntimeCombatActionService`。
- [ ] 编排器只推进主链路，不直接持有复杂状态真源。
- [ ] 主链路阶段拆清：动作解析、快照读取、目标解析、合法性、时序、结算、应用、事件。
- [ ] 编排器返回结构化结果：成功目标、拒绝目标、表现事件、通知、脏域、诊断。
- [ ] 编排器支持 dry-run 或 explain 模式，用于 smoke 和诊断。
- [ ] 编排器接入统一日志上下文：actor、action、instance、phase、target count。

验收：

- [ ] 编排器可以在测试中处理一个最小普攻 action。
- [ ] 编排器可以返回拒绝原因而不是抛散落异常或静默返回。
- [ ] 编排器不直接访问数据库。
- [ ] 编排器不直接组装网络包。

### 阶段 3：统一动作定义解析

- [ ] 新增普攻内置 action definition。
- [ ] 玩家技能 id 解析为 action definition。
- [ ] 怪物技能 id 解析为 action definition。
- [ ] 统一表达范围、形状、最大目标数、效果列表、资源、冷却、吟唱。
- [ ] 保留当前数值和公式，不在本阶段改玩法。
- [ ] 将普攻范围、伤害类型、基础伤害来源显式写入 action definition。

验收：

- [ ] 玩家普攻可解析为 `basic_attack` action definition。
- [ ] 玩家技能可解析为 `skill` action definition。
- [ ] 怪物技能可解析为同形态 action definition。
- [ ] 解析失败有结构化拒绝原因。

### 阶段 4：统一目标解析和范围收集

- [ ] 抽出统一锚点解析：目标 id、地块坐标、自身、pending cast 锚点。
- [ ] 抽出统一技能几何计算：单体、线形、范围、棋盘等。
- [ ] 抽出统一候选格生成。
- [ ] 从地图实例按格子占位收集玩家目标。
- [ ] 从地图实例怪物索引收集怪物目标。
- [ ] 从地图实例收集地块和容器目标。
- [ ] 从阵法服务收集阵法和边界目标。
- [ ] 保留主目标优先，但最终输出统一 `CombatTarget[]`。
- [ ] 目标收集阶段不扣血、不发通知、不发奖励。

验收：

- [ ] 玩家技能和怪物技能共用同一套候选格计算。
- [ ] 有预警格的吟唱技能完成时能按预警格收集当前目标。
- [ ] 目标收集为空时返回结构化 `no_targets`，不静默。
- [ ] 单体普攻也能通过目标解析输出一个 `CombatTarget`。

### 阶段 5：统一合法性判断

- [ ] 抽出同实例判断。
- [ ] 抽出敌我关系判断，统一调用现有关系规则。
- [ ] 抽出范围判断。
- [ ] 抽出视线判断。
- [ ] 抽出地图能力判断：PVP、地形伤害、阵法、容器等。
- [ ] 抽出目标类型允许判断：动作是否能影响玩家、怪物、地块、阵法、容器。
- [ ] 抽出资源和冷却判断。
- [ ] 对每个目标输出 `allowed | rejected(reason)`。
- [ ] 玩家主动操作的拒绝原因可转为玩家提示；怪物 AI 的拒绝原因只进入诊断。

验收：

- [ ] 玩家普攻、玩家技能、怪物普攻、怪物技能使用同一套同实例判断。
- [ ] 玩家普攻、玩家技能、地块/阵法/容器目标使用同一套敌我关系入口。
- [ ] 所有合法性拒绝都有 `CombatRejectReason`。
- [ ] 不再新增无原因 `return`。

### 阶段 6：统一命中和效果结算

- [ ] 保留现有 `resolveCombatHit` 作为核心命中伤害公式。
- [ ] 将普攻伤害效果包装为标准 damage effect。
- [ ] 将玩家技能 damage/buff/heal/cleanse 等效果接入统一 effect resolver。
- [ ] 将怪物技能效果接入同一 effect resolver。
- [ ] 多目标技能统一处理 `targetCount` 和资源/冷却跳过策略。
- [ ] 输出统一 `CombatOutcome`。
- [ ] 闪避、免疫、抵抗、未造成伤害都作为合法 outcome。

验收：

- [ ] 普攻和技能都能产出 `CombatOutcome`。
- [ ] 怪物攻击玩家和玩家攻击怪物共用命中解析入口。
- [ ] 进入结算后不会没有 outcome。
- [ ] 旧伤害数值 smoke 保持一致。

### 阶段 7：按目标类型应用结果

- [ ] 新增玩家目标结果应用适配器。
- [ ] 新增怪物目标结果应用适配器。
- [ ] 新增地块目标结果应用适配器。
- [ ] 新增阵法目标结果应用适配器。
- [ ] 新增容器目标结果应用适配器。
- [ ] 玩家适配器负责扣血、buff、死亡、自动反击、活动记录、脏域。
- [ ] 怪物适配器负责扣血、buff、仇恨、死亡、掉落、奖励、脏域。
- [ ] 地块适配器负责耐久、破坏、恢复脏域。
- [ ] 阵法适配器负责灵力、阵眼、边界状态和脏域。
- [ ] 容器适配器负责次数、掉落、刷新状态和脏域。

验收：

- [ ] 目标类型分支只存在于结果应用阶段。
- [ ] 结果应用不重新定义命中、范围、敌我关系。
- [ ] 旧玩家死亡、怪物击杀、地块破坏、阵法扣灵力、容器消耗行为保持一致。

### 阶段 8：统一表现、通知、审计和诊断

- [ ] 新增统一战斗事件构建器。
- [ ] 进入结算的每个 outcome 都生成结果事件。
- [ ] 命中、闪避、免疫、抵抗、未造成伤害都生成表现事件。
- [ ] 玩家通知从 outcome 统一生成。
- [ ] AOI 表现事件从 outcome 统一生成。
- [ ] 诊断事件覆盖 action 拒绝、无目标、实例不一致、状态真源不一致、pending cast 过期。
- [ ] 怪物 action 应用失败必须记录机器可读原因。

验收：

- [ ] 闪避时必有战斗通知。
- [ ] 预警格命中但没有通知时，服务端能指出停在哪一步。
- [ ] 怪物技能 action 不再无原因静默消失。
- [ ] 高频客户端包不混入诊断和审计长文本。

### 阶段 9：吟唱生命周期接入主链路

- [ ] 定义统一 pending cast 数据结构。
- [ ] 玩家吟唱开始保存 `CombatAction` 恢复所需字段。
- [ ] 怪物吟唱开始保存同形态字段。
- [ ] pending cast 保存动作 id、来源、锚点、预警格、目标引用、资源状态、冷却状态、配置 revision。
- [ ] 吟唱完成从 pending cast 恢复 `CombatAction`。
- [ ] 吟唱完成重新读取权威状态和当前目标。
- [ ] 吟唱完成接入同一目标解析、合法性、结算、应用、事件链路。
- [ ] 吟唱取消记录取消原因，并按策略处理资源/冷却。

验收：

- [ ] 玩家吟唱技能和怪物吟唱技能完成时走同一主链路。
- [ ] 有预警格技能按完成时预警格内当前目标结算。
- [ ] 合法落空和异常落空能区分。
- [ ] 唤灵真人残魄掌等吟唱技能有覆盖 smoke。

## 验证矩阵

### 正确性 smoke

- [ ] 玩家普攻怪物：命中、闪避、击杀、掉落。
- [ ] 玩家普攻玩家：命中、闪避、死亡、自动反击。
- [ ] 玩家普攻地块：扣耐久、破坏、恢复。
- [ ] 玩家普攻阵法：扣灵力、边界状态。
- [ ] 玩家普攻容器：扣次数、掉落、刷新。
- [ ] 玩家即时技能：单体、多目标、buff、治疗、无目标。
- [ ] 玩家吟唱技能：预警、完成、落空、被打断。
- [ ] 怪物普攻玩家：命中、闪避、死亡。
- [ ] 怪物即时技能：命中、闪避、buff。
- [ ] 怪物吟唱技能：预警格命中、合法落空、异常诊断。
- [ ] 唤灵真人技能：技能选择、吟唱、残魄掌、其他阶段技能。

### 边界 smoke

- [ ] 目标离开实例。
- [ ] 发起者死亡。
- [ ] 目标死亡。
- [ ] 视线阻挡。
- [ ] 超出范围。
- [ ] 非敌对目标。
- [ ] 地图不允许 PVP。
- [ ] 地图不允许地形伤害。
- [ ] 资源不足。
- [ ] 冷却未完成。
- [ ] pending cast 过期或取消。

### 门禁

- [ ] `pnpm --filter @mud/server compile`
- [ ] 战斗专项 smoke runner。
- [ ] 怪物技能专项 smoke。
- [ ] 玩家技能专项 smoke。
- [ ] 地块/阵法/容器专项 smoke。
- [ ] `pnpm build:shared`，仅在 shared 类型或协议变更时。
- [ ] `pnpm build:server`
- [ ] `pnpm audit:protocol`，仅在协议 payload 变化时。
- [ ] with-db smoke，仅在战斗结果影响持久化真源或恢复链时。

## 风险和控制

| 风险 | 控制 |
|---|---|
| 一次性替换战斗链路导致玩法大面积漂移 | 分阶段迁移，每阶段保留旧行为 smoke |
| 普攻作为内置动作后数值变化 | 第一阶段只封装动作定义，不改公式 |
| 怪物技能迁移后 AI 行为变化 | AI 仍只负责选动作，技能优先级单独保持 |
| 地块、阵法、容器被强行套生命体逻辑 | 只在目标解析和合法性统一，结果应用保留适配器 |
| 统一事件导致高频包变大 | 诊断和审计不下发普通客户端，AOI 只发必要表现 |
| pending cast 保存过多状态 | 保存恢复 action 所需最小字段，配置 revision 用于兼容 |
| 拒绝原因过多影响热路径 | 使用枚举/数字码，不在 tick 热路径拼长字符串 |
| 持久化误入 tick 热路径 | 战斗只标脏域和事件，落库通过 flush/outbox/快照 |

## 不做范围

- [ ] 不在本轮重做数值公式。
- [ ] 不在本轮重做怪物 AI 策略。
- [ ] 不在本轮改变唤灵真人技能设计。
- [ ] 不在本轮改变玩家可见战斗文案风格。
- [ ] 不在本轮把诊断和审计事件发给普通客户端。
- [ ] 不在本轮引入 Rust 加速；只有确认战斗结算成为热点后再评估。

## 最终完成标准

- [ ] 普攻和技能共用统一 `CombatAction` 主链路。
- [ ] 玩家和怪物共用统一 `CombatActor` 快照和结算入口。
- [ ] 目标类型只在结果应用阶段分支。
- [ ] 玩家技能和怪物技能共用目标解析、合法性、命中结算。
- [ ] 吟唱开始和吟唱完成是主链路的两个阶段。
- [ ] 怪物 AI 不直接扣血、不直接发通知、不直接处理玩家死亡。
- [ ] 所有拒绝都有结构化原因。
- [ ] 所有进入结算的目标都有 `CombatOutcome`。
- [ ] 闪避、免疫、未造成伤害都有表现或通知。
- [ ] 怪物技能不存在无原因静默 return。
- [ ] 高频同步没有混入静态配置、长文本诊断或审计内容。
- [ ] 所有影响玩家资产、死亡、掉落、地图持久状态的结果都有脏域或持久化转出策略。
