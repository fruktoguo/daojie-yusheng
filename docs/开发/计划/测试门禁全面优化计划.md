# 测试门禁全面优化计划

目标：把当前生产主线的测试门禁从“能证明”优化成“更快、更分层、更少重复、更容易定位失败”。本文只规划测试体系优化，不改变现有游戏行为。

## 当前问题

当前门禁已经能覆盖生产主线关键风险，但执行成本偏高：

- `verify:release:local*` 分层清楚，但 `full` 会复跑 `with-db` 已覆盖的一部分内容。
- server smoke-suite 每个 isolated case 都会启动独立服务端，定位稳定，但启动成本高。
- 很多包级 `smoke:*` 命令自带 `pnpm compile`，组合执行时容易重复编译。
- `build / verify:release:local / with-db / acceptance / full` 的定位边界清楚，但缺少一个面向日常开发的短门禁。
- 纯静态 proof、纯内存 smoke、报告类 smoke、带库 destructive proof 混在同一 suite 认知里，后续维护容易把“必须跑”和“按改动面补跑”混读。
- 当前耗时只有日志中的零散结果，缺少统一计时产物，优化慢项时证据不够集中。

## 不优化掉的底线

- 不用单个 `build` 冒充协议、持久化、shadow 或 GM 数据库证明。
- 不把 `shadow / acceptance / full / shadow-destructive` 混成一个结论。
- 不取消有持久化写入的自动清理链。
- 不把维护窗口 destructive proof 常态化。
- 不为了提速牺牲服务端权威、协议包体、数据库真源、恢复和审计验证。

## 目标门禁分层

### `doctor`

用途：1 秒级环境检查。

保留现状：

- 检查 DB、shadow URL、GM 密码、destructive 开关。
- 只回答环境是否齐备，不回答代码正确性。

优化点：

- 输出当前推荐下一步命令。
- 输出当前 shadow 缺失原因时带具体变量名。

### `quick`

用途：日常开发最快反馈。

建议新增命令：

- `pnpm verify:quick`

建议覆盖：

- shared build / server compile / client typecheck 中与改动相关的最短链路。
- `proof:release-gates`
- `proof:production-boundaries`
- `smoke:readiness-gate`
- `smoke:runtime`
- `smoke:session`

不覆盖：

- DB 真源回读。
- shadow 实物环境。
- GM 数据库备份/恢复。
- destructive proof。

适用：

- 普通 TS 改动。
- 小范围 runtime / network / UI 改动的首轮反馈。
- 开始长门禁前的预检。

### `standard`

用途：提交前默认门禁。

建议将当前 `pnpm verify:release:local` 视为 `standard`，可新增别名：

- `pnpm verify:standard`

保留覆盖：

- `build:client`
- server `verify:release:local`
- `audit:protocol`

优化点：

- 不自动升级为 `with-db`。如果 shell 有 DB，也应由显式 `with-db` 命令承担。
- 输出每个 case 耗时和总耗时到 `.runtime/verification-timings/`。

### `with-db`

用途：持久化真源、恢复、回读和清理链。

保留命令：

- `pnpm verify:release:with-db`

建议覆盖：

- 当前 server `run-stable-smoke-suite.js --include-persistence`
- `audit:protocol`

优化点：

- `with-db` 内已经覆盖 `gm-database` 时，后续 `full` 不再无条件复跑同一个 destructive-local proof。
- DB case 串行执行，纯内存/报告类 case 允许并行。
- 每个 DB case 前后保留自动清理摘要。

### `release`

用途：发布前强门禁。

建议新增命令：

- `pnpm verify:release`

建议组合：

- `doctor`
- `standard`
- `with-db`
- `shadow`
- `gm`

说明：

- `release` 是日常发布门禁，不包括 `shadow-destructive`。
- 如果 shadow 环境缺失，失败归类为环境阻塞，不归类为代码失败。

### `full`

用途：最严格自动化门禁。

保留命令：

- `pnpm verify:release:full`

建议调整：

- `full` 继续回答“with-db + shadow + GM 数据库运营链是否全绿”。
- 避免重复跑 `with-db` 已经确认过的 `gm-database`，除非通过参数显式要求 `--rerun-gm-database`。
- `gm-database-backup-persistence` 仍作为 `full` 独立保留，因为它证明备份记录跨重启可见。

### `shadow-destructive`

用途：维护窗口 destructive proof。

保留命令：

- `pnpm verify:release:shadow:destructive:preflight`
- `pnpm verify:release:shadow:destructive`

优化点：

- 默认不纳入 `release` 或 `full`。
- preflight 输出必须包含 target URL、maintenance-active 状态和 destructive 开关来源。
- destructive 执行完成后写入独立证据文件，不能只靠终端输出。

## Case 分组优化

### 静态 proof

可并行：

- `proof:release-gates`
- `proof:production-boundaries`
- `proof:runtime-network-source`
- `proof:client-s2c-consumption`
- `proof:shared-types-source`
- `proof:protobuf-drift`
- `proof:protocol-source`

要求：

- 不访问 DB。
- 不启动 server。
- 不写持久化夹具。

### 独立服务端 smoke

默认串行：

- `readiness-gate`
- `session`
- `runtime`
- `progression`
- `combat`
- `loot`
- `auth-bootstrap`
- `gm`
- `redeem-code`
- `monster-*`
- `player-recovery`
- `player-respawn`

优化方向：

- 复用编译产物。
- 继续每 case 独立端口和独立进程，优先保证失败隔离。
- 只有确认无共享环境、无 DB、无固定端口、无全局 runtime 污染后，才允许并行。

### 纯内存 / 报告类 smoke

优先并行：

- `snapshot-retirement`
- `map-snapshot-retirement`
- `multi-worker-flush-stability`
- `strong-persistence-lease`
- `player-columnar-schema`
- `player-dirty-domain-coverage`
- `world-runtime-instance-capability-guard`
- `world-runtime-player-session-no-auto-instance`

要求：

- 不依赖真实 server 端口。
- 不创建 DB 对象。
- 并行后输出仍要按 case 汇总。

### DB 真源 smoke

默认串行：

- `persistence`
- `player-domain-persistence`
- `player-domain-recovery`
- `durable-operation`
- `gm-database`
- `gm-map-config-persistence`
- `gm-database-backup-persistence`

原因：

- 会创建持久化对象。
- 需要清理链。
- 失败时必须能直接定位首个污染源。

优化方向：

- 串行保留，但减少重复编译和重复备份。
- case 结束后强制输出清理摘要。
- 失败时保留最小诊断文件，不保留大体量临时备份。

## 按改动类型选择门禁

| 改动类型 | 必跑 | 视情况补跑 |
| --- | --- | --- |
| 文档 | `proof:release-gates` 或 markdown 链接检查 | 无 |
| shared 类型 / 常量 | `build:shared`, `verify:quick` | `audit:protocol` |
| 协议 / 发包 / delta | `verify:standard`, `audit:protocol` | `with-db` |
| server runtime | `verify:quick`, 对应 `smoke:*` | `standard`, `with-db` |
| combat / loot / monster | 对应专项 smoke | `standard` |
| persistence / recovery | `with-db` | `full` |
| GM / admin | `smoke:gm` | `shadow`, `full` |
| client UI | `build:client` | 手动浅色/深色/手机检查 |
| config-editor | `build:config-editor` | 内容引用审计 |
| 发布前 | `doctor`, `standard`, `with-db`, `shadow` | `full` |
| 维护窗口 | `shadow-destructive:preflight` | `shadow-destructive` |

## 耗时目标

当前已有本轮实测基线：

- `pnpm verify:quick`：约 31 秒；边界 proof 复用已编译产物后约 1.1 秒。
- `pnpm verify:standard`：约 2 分 44 秒，覆盖 `build:client -> server release local -> audit:protocol`。
- `pnpm verify:release:doctor`：1 秒内，当前缺失项集中在 `SERVER_SHADOW_URL/SERVER_URL`。

| 门禁 | 目标耗时 | 说明 |
| --- | --- | --- |
| `doctor` | 1-3 秒 | 只查环境 |
| `quick` | 30-90 秒 | 日常首轮反馈 |
| `standard` | 2-4 分钟 | 提交前默认 |
| `with-db` | 3-6 分钟 | DB 真源与恢复 |
| `release` | 5-9 分钟 | 发布前自动化 |
| `full` | 6-10 分钟 | 最严格自动化 |
| `shadow-destructive` | 不设固定目标 | 取决于远端备份大小和维护窗口 |

目标不是盲目压到最短，而是让短门禁足够快、长门禁足够可靠。

## 计时与诊断产物

新增统一输出目录：

- `.runtime/verification-timings/latest.json`
- `.runtime/verification-timings/history.jsonl`

建议字段：

- `command`
- `startedAt`
- `finishedAt`
- `durationMs`
- `gate`
- `caseDurations`
- `failedCase`
- `environment`
- `dbEnabled`
- `shadowEnabled`
- `commit`
- `dirty`

用途：

- 追踪慢 case。
- 判断优化是否有效。
- 失败时快速区分编译慢、启动慢、DB 慢、shadow 慢。
