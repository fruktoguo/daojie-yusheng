# 历史遗留入口退役与命名收口计划

## 1. 文档定位

本文用于收口当前仓库里剩余的历史命名、旧数据迁移工具、compat smoke、备份恢复 scope 和文档口径。

这不是恢复旧线、重建 `legacy/` 目录或继续维护双线兼容的计划。当前生产主线仍然只落在 `packages/*`：

- `packages/client`
- `packages/shared`
- `packages/server`
- `packages/config-editor`

## 2. 当前结论

`legacy/` 实体目录已经不存在，也没有 tracked 文件。后续工作不是“删 legacy 文件夹”，而是处理仓库里仍带历史语义的入口：

| 类型 | 例子 | 当前处理原则 |
| --- | --- | --- |
| 已完成切换期入口 | `cutover-auto*`、`proof:cutover*`、`10-cutover-*` | 删除，不保留文档入口 |
| 当前验证入口 | `release*`、`production-boundaries`、`persistence-retirement-audit` | 已改为当前职责命名，旧 alias 不再保留 |
| 离线迁移或历史导入工具 | `import-legacy-persistence-once`、旧 snapshot / `persistent_documents` 导入 | 先证明不再需要，再退役 |
| compat 证明链 | `auth-bootstrap-legacy-import`、legacy snapshot rejection proof | 改成“旧链路被拒绝/不可回读”证明后再删旧命名 |
| 数据备份恢复兼容 scope | `legacy_persistent_documents`、`server_persistence` | 先确认历史备份导入策略，再收窄或改名 |

## 3. 不做事项

- 不恢复 `legacy/`、旧 Dockerfile、旧 compose 或旧运行时目录。
- 不因为字符串里出现 `legacy / compat / mainline / migration` 就机械删除。
- 不在未确认历史备份恢复策略前删除旧 JSON 导入能力。
- 不把验证入口只改名不改 proof 合同；改名必须同步 package scripts、文档、doctor、smoke suite 和 CI。
- 不让正式 runtime 重新读取旧 snapshot、旧 map JSON、旧 mailbox JSON 或通用 `persistent_documents` 作为真源。

## 4. 目标状态

完成后仓库应满足：

- 根级和包内脚本名只表达当前职责，例如 build、boundary、persistence-retirement、runtime smoke、backup restore。
- 旧切换期入口全部消失，不再有 cutover runbook 或自动切换脚本。
- 旧数据迁移工具只在明确维护窗口、离线归档或历史备份导入策略里出现。
- 所有 compat smoke 的目标从“兼容仍能跑”改为“旧路径被拒绝、不可作为 runtime fallback、迁移后不可回读”。
- `release` 是正式发布验证命令族，不再作为旧命名 alias。
- `persistence-retirement-audit` 回答的是“旧真源是否回潮”，不是项目阶段仍在 hard cut。

## 5. 阶段 0：基线扫描与分类

### 5.1 扫描命令

```bash
rg -n "legacy|compat|migration|mainline|cutover|hard-cut|release gate|server-next|start-next" \
  package.json packages scripts docs .github \
  --glob '!packages/**/dist/**' \
  --glob '!node_modules/**'

git ls-files | rg "legacy|compat|migration|mainline|cutover|hard-cut|replace|server-next|start-next"
```

### 5.2 分类规则

- 正式 runtime 依赖旧路径：P0，必须删除或改成硬失败。
- 冷路径迁移/审计工具：P1，先补退役条件和 owner，再决定保留或删除。
- 验证入口旧命名：P1，按命名计划统一，不单点改。
- 文档历史记录：P2，归档或改写为当前口径，不影响运行时。
- 测试数据、trace、`.runtime` 残留：P0/P1，tracked 的直接删，ignored 的按环境清理策略处理。

### 5.3 输出物

- 一张保留 / 改名 / 删除清单。
- 每个保留项必须写明“谁还用、回答什么、不回答什么、何时退役”。
- 每个删除项必须写明替代验证或确认无引用。

## 6. 阶段 1：验证入口命名统一

### 6.1 根级脚本

当前保留：

- `verify:release:local`
- `verify:release:doctor`
- `verify:release:with-db`
- `verify:release:shadow`
- `verify:release:acceptance`
- `verify:release:full`
- `proof:release-gates`

目标命名建议：

| 当前入口 | 目标入口 | 说明 |
| --- | --- | --- |
| `verify:release:local` | `verify:release:local` | 本地生产主线 gate |
| `verify:release:doctor` | `verify:release:doctor` | 环境 readiness 检查 |
| `verify:release:with-db` | `verify:release:with-db` | 数据库真源与回读 |
| `verify:release:shadow` | `verify:release:shadow` | 已部署 shadow 只读验收 |
| `verify:release:acceptance` | `verify:release:acceptance` | local + shadow + GM |
| `verify:release:full` | `verify:release:full` | with-db + GM database + shadow + GM |
| `proof:release-gates` | `proof:release-gates` | 脚本和文档合同 proof |

迁移策略：

1. 新增目标入口并删除旧入口 alias。
2. 更新 `scripts/check-release-gates.js`，只校验当前入口。
3. 更新 README、`packages/server/TESTING.md`、`packages/server/RUNBOOK.md`、`docs/next-plan/09-verification-and-acceptance.md`。
4. 跑通后确认旧 alias 不再出现在 package scripts。

验收：

```bash
pnpm proof:release-gates
pnpm verify:release:doctor
pnpm verify:release:local
pnpm build
```

## 7. 阶段 2：production / persistence-retirement 审计命名收口

### 7.1 当前入口

- `packages/client/scripts/check-client-production-boundaries.js`
- `packages/server/src/tools/check-production-boundaries.ts`
- `packages/server/src/tools/audit/production-boundary-audit.ts`
- `packages/server/src/tools/prove-runtime-network-source.ts`
- `packages/server/package.json` 内 `proof:production-boundaries`
- `packages/server/package.json` 内 `audit:persistence-retirement`

### 7.2 目标命名

| 当前名 | 目标名 | 说明 |
| --- | --- | --- |
| `production-boundaries` | `production-boundaries` | 表达当前生产边界，不表达历史阶段 |
| `prove-runtime-network-source` | `prove-runtime-network-source` | 证明 runtime/network 真源，不带阶段名 |
| `persistence-retirement-audit` | `persistence-retirement-audit` | 证明旧真源退役，不表达仍在 hard cut |

### 7.3 执行要求

- 脚本文件、package script、输出文案、文档引用要同轮更新。
- 不能只改 package script，不改 `dist/tools` 编译后的调用路径。
- 如果 CI 或外部脚本还调用旧名，应更新调用方，不继续保留 package alias。

验收：

```bash
pnpm build:client
pnpm build:server
pnpm --filter @mud/server audit:boundaries
pnpm --filter @mud/server audit:persistence-retirement
pnpm proof:server-runtime-boundaries
```

目标入口已落地，后续只允许新增当前职责命名。

## 8. 阶段 3：迁移工具退役判定

### 8.1 待判定对象

- `packages/server/src/tools/import-legacy-persistence-once.ts`
- `packages/server/package.json` 的 `smoke:import-legacy-persistence`
- `deploy-database-preflight` 对迁移状态的引用
- `map-dynamic-persistence-audit`
- `instance-domain-migration-smoke`
- `auth-bootstrap-legacy-import`

### 8.2 保留条件

只有满足以下任一条件才允许继续保留：

- 仍需导入历史生产备份。
- 仍需在维护窗口把旧 snapshot / `persistent_documents` 转为结构化表。
- 仍被 `deploy-database-preflight` 用来阻断危险部署。
- 仍被 GM 数据库恢复链作为历史备份导入路径。
- 仍有未完成的旧数据离线归档验收。

### 8.3 删除条件

全部满足后必须删除或改名为 archive/import 工具：

- 正式库确认不再存在需要转换的旧 snapshot / 旧 scope。
- 历史备份已经转存为当前 `server_persistence` 或新命名的 production backup 格式。
- `verify:release:with-db` 不再需要 migration case 才能证明当前真源。
- `deploy-database-preflight` 不再依赖一次性迁移脚本判断。
- `persistence-retirement-audit` 能证明正式 runtime 没有旧读写入口。

### 8.4 退役方式

优先顺序：

1. 如果仍要支持历史备份导入，保留 `import-legacy-persistence-once`；如果只面向备份文件转换，再改成 `convert-legacy-persistence-dump`。
2. 如果只剩审计，把迁移脚本拆成只读 report，不保留 `--apply`。
3. 如果完全不用，删除脚本、package entry、smoke case 和文档引用。

验收：

```bash
pnpm build:server
pnpm --filter @mud/server audit:persistence-retirement
pnpm verify:release:with-db
pnpm verify:release:full
```

## 9. 阶段 4：auth / bootstrap compat 证明链改造

### 9.1 当前问题

`auth-bootstrap-smoke.ts` 里仍有大量 `legacy / compat / migration / mainline` 文案和 case。它们不一定是运行时兼容路径，有一部分是在证明旧路径不会被误用。

不能直接整块删除，必须先拆语义。

### 9.2 拆分目标

| 当前语义 | 目标语义 |
| --- | --- |
| `auth-bootstrap-native` | `auth-bootstrap-native` |
| `auth-bootstrap-legacy-import` | `auth-bootstrap-legacy-import-rejection` 或 `auth-bootstrap-legacy-import` |
| `legacy_runtime` | 如果仅测试输入，改成 `legacy_import_fixture` |
| `compat snapshot ignored` | 保留为 `legacy snapshot rejected` proof |
| `mainline_invalid` | 改成 `native_invalid` 或 `current_invalid` |

### 9.3 执行步骤

1. 先把 smoke suite case 名字抽出映射表，避免字符串散落。
2. 新增新 case 名，删除旧 case alias。
3. trace 文件名统一为 `auth-bootstrap-trace-*`。
4. 把测试 fixture 的 `legacy_*` 命名改为 `legacy_import_fixture_*`，避免误读为 runtime 入口。
5. 把“兼容成功”类断言改为“旧路径被拒绝 / 旧 snapshot 不可作为 fallback / 缺新真源必须失败”。
6. 删除旧 alias。

验收：

```bash
pnpm --filter @mud/server smoke:auth-bootstrap
pnpm --filter @mud/server smoke:auth-bootstrap:native
pnpm --filter @mud/server smoke:auth-bootstrap:legacy-import
pnpm verify:release:proof:with-db
pnpm verify:release:with-db
```

目标命名落地后，把上面的旧入口替换为新入口。

## 10. 阶段 5：备份恢复 scope 收口

### 10.1 当前对象

- `GmDatabaseBackupFormat = 'postgres_custom_dump' | 'legacy_json_snapshot'`
- `scope = 'legacy_persistent_documents' | 'server_persistence'`
- `restoreMode = 'replace_server_persistence'`
- `NativeGmAdminService` 对历史 JSON 的导入和懒初始化逻辑

### 10.2 目标

- 正式备份 scope 命名改为 `server_persistence`，历史 JSON 只显示为 `legacy_json_snapshot`。
- `legacy_persistent_documents` 只允许作为历史导入格式，不作为新备份默认 scope。
- 恢复链必须清楚区分：
  - PostgreSQL dump 全库恢复
  - 当前结构化业务备份恢复
  - 历史 JSON 备份导入

### 10.3 执行要求

- shared API 类型先加新 enum 值，服务端兼容旧值一轮。
- GM UI 显示用新名字，不再向操作者暴露 mainline / persistent_documents 旧口径。
- 恢复 destructive proof 必须覆盖 PostgreSQL dump 分支。
- 历史 JSON 备份只允许列表展示和下载，不允许作为正式恢复输入。

验收：

```bash
pnpm build
pnpm --filter @mud/server smoke:gm-database:backup-persistence
pnpm verify:release:shadow:destructive:preflight
SERVER_SHADOW_ALLOW_DESTRUCTIVE=1 pnpm verify:release:shadow:destructive
```

## 11. 阶段 6：文档归档与当前口径统一

### 11.1 文档处理规则

- `docs/开发/计划/**`：保留仍待执行的计划，删除已完成的历史执行清单。
- `docs/开发/设计/**`：保留目标架构和当前状态，不保留“当时怎么切”的流水账。
- `docs/next-plan/**`：只保留当前验证入口和后续专题，不恢复 cutover 任务。
- README / RUNBOOK / TESTING：只写当前生产运维入口，不写迁移期故事。

### 11.2 必改措辞

| 旧措辞 | 新措辞 |
| --- | --- |
| next 主线 | 当前生产主线 |
| next 主链 | 当前主链 |
| hard cut | 旧真源退役 / 旧链路退役 |
| cutover | 发布 / 维护窗口 / 验证窗口 |
| mainline backup | production persistence backup |
| legacy compat | history import / rejected legacy input |

### 11.3 不必改的词

以下词如果描述玩法或业务动作，不属于历史遗留：

- 世界迁移
- 玩家迁移
- 地图实例迁移
- 数据库 migration / schema migration
- mail archive / retention archive

## 13. 完成定义

- [x] 没有实体 `legacy/` 目录，也没有 tracked `legacy/**` 文件。
- [x] 根级不再存在 cutover 自动脚本或 cutover proof。
- [x] 当前验证入口使用 `release*` 新命名，旧 `replace-ready*` alias 已删除。
- [x] `production-boundaries` 类入口完成命名收口，并作为当前边界 proof 名。
- [x] `import-legacy-persistence-once` 已作为历史导入工具保留，旧 `migrate-next-mainline-once` 名称退役。
- [x] `auth-bootstrap-legacy-import` 已作为历史导入证明链保留，旧 `auth-bootstrap-migration` 名称退役。
- [x] 备份恢复 scope 区分当前 `server_persistence` 和历史 `legacy_json_snapshot`。
- [x] `persistence-retirement-audit` 明确解释为旧真源退役审计。
- [x] 文档不再把项目描述为切换期。
- [x] `pnpm build` 通过。
- [x] 带库、shadow、acceptance、full 的未跑项都有明确环境原因，不混读结论。

## 14. 推荐执行顺序

1. 先改验证入口命名，保证日常 build / doctor / with-db 不漂。
2. 再改 boundary / persistence-retirement audit 命名，保持架构 proof 可读。
3. 再审 `import-legacy-persistence-once` 和 `auth-bootstrap-legacy-import`，决定保留或继续退役内部 fixture 旧名。
4. 再收口 GM 数据库备份恢复 scope。
5. 最后做文档归档和全仓关键词复扫。

这套顺序的理由是：验证入口是保护网，先稳住它；迁移和备份恢复涉及玩家资产与运营回滚，必须在证明链清楚之后再删。
