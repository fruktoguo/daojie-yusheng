# 商业级数据落盘改造计划

> 基于 [mmo商业级数据落盘方案.md](../mmo商业级数据落盘方案.md) 的终局设计，结合当前 next 主线代码现状制定。

---

## 2026-04-28 调整：数据层一次性商用级硬切

本计划从今天起不再按“兼容迁移、旧链路兜底、逐步弱化”的口径推进。用户目标已经调整为：

> 数据部分一次性做到商业级；不为了兼容旧存档、旧文档、旧 fallback 继续保留运行时分叉。

因此本文后续任务按“同一次数据硬切版本”验收。可以并行拆任务包，但不能拆成多个兼容上线阶段；只要旧链路仍参与正式运行时读写，就不能宣称数据层商用级完成。

### 硬切原则

- PostgreSQL 结构化表是唯一正式真源；Redis、内存、Socket 投影、本地 JSON、`persistent_documents` 都不能作为“下次还在”的业务真源。
- `server_player_snapshot`、旧地图快照、旧邮箱 JSON、市场兼容文档等不再作为恢复、checkpoint、fallback 或双写镜像；最多在停服迁移窗口作为一次性输入文件读取。
- 所有资产、货币、背包、邮件附件、市场成交、任务奖励、兑换码奖励、GM 发放都必须走 durable transaction；durable 条件不满足时返回硬错误，不允许 direct runtime fallback。
- 地图高频域只能走 worker + ledger + row-level delta；`MapPersistenceFlushService` 不再常规扫描并直接刷 `monster_runtime / tile_resource / tile_damage / ground_item / container_state / overlay`。
- 任何“分域缺失 -> 回退旧快照 / starter snapshot / fallback 出生点并继续进入游戏”的路径，在正式硬切后都视为数据损坏或恢复失败，必须阻断启动或阻断该玩家登录。
- 旧数据如果不能确定性转换为新表，迁移失败并回滚；不做静默丢字段、不做运行时兼容补洞。
- GM、兑换码、建议、宗门、市场订单、爱发电、备份任务等低频状态也要专表化；`persistent_documents` 只允许作为离线导入/导出报告格式，不允许被服务端 runtime service 读写。

### 一次性硬切执行窗口

```text
停止所有游戏节点和 worker
-> 备份当前数据库和旧 dump 文件
-> 执行完整 DDL：玩家、实例、宗门、市场、GM、兑换码、建议、爱发电、运维任务全量专表化
-> 离线转换旧 snapshot / persistent_documents / dump 到结构化表
-> 校验 row count、watermark、资产总账、邮件附件、市场托管仓和实例域行数
-> 删除或隔离旧 scope / 旧快照表；运行时 DB 连接不得再看到旧业务真源
-> 部署无兼容读写路径的新代码
-> 跑 with-db、shadow、acceptance、full、重启恢复、备份恢复、压测和故障注入
-> 只有全部通过才开放玩家入口
```

这不是“边运行边迁移”。如果需要保留旧数据，只能在维护窗口前导出归档；正式运行库里不保留旧业务真源参与服务。

### 必删运行时旧链路

| 类别 | 一次性硬切要求 |
|---|---|
| 玩家大快照 | 删除 `server_player_snapshot / server_next_player_snapshot` 的恢复、checkpoint、GM 读取和 flush 写入；玩家恢复只从 `player_*` 分域装配 |
| 地图大快照 | 删除 `MapPersistenceService.loadMapSnapshot/saveMapSnapshot` 在 runtime、flush、restore、GM 普通操作中的使用；旧地图 JSON 只允许离线 converter 读取 |
| `persistent_documents` | 迁走所有业务 scope；runtime service 禁止注入通用文档持久化作为正式真源 |
| 资产 fallback | 兑换码、战斗掉落、PvP 奖励、NPC 任务奖励、GM 钱包等 direct runtime fallback 全部改成 durable 失败即失败 |
| worker/interval 双链路 | 高频实例域只由 domain worker claim 后写入；周期服务只处理低频 checkpoint、shutdown force 和诊断，不得常规刷高频域 |
| 整域 replace | `monster_runtime / tile_resource / tile_damage / ground_item / container_state / overlay / temporary_tile` 普通刷盘全部 row-level 或 key-level delta |
| 旧兼容 smoke | 以旧快照一致性为目标的 smoke 改成“旧链路不存在/被拒绝/迁移后不可回读”为目标 |

### 一次性任务包

这些任务可以多人并行，但必须在同一个硬切版本一起合入和验收：

1. 真源表补齐：为宗门、GM auth/config、兑换码、建议、市场订单/历史、爱发电配置/订单、数据库任务元数据补专表和 repository。
2. 玩家恢复硬切：`PlayerRuntimeService` 直接从 `player_*` domain 装配运行态；删除旧 snapshot loader、投影回组装旧快照、presence-only fallback。
3. 资产入口硬切：统一 durable command intent；删除所有 durable 条件不满足时的运行态直写分支。
4. 实例域硬切：高频域 row-level delta；容器、overlay、temporary tile、formation 纳入统一 ledger 或强事务直写边界。
5. 调度硬切：worker 是高频域唯一主路径；周期 flush 改名并降级为 checkpoint/shutdown/diagnostic。
6. 离线转换脚本：只作为停服迁移工具，把旧 snapshot、旧 `persistent_documents`、旧 dump 转到新表；转换完成后旧数据移出运行库。
7. 运维真源硬切：GM 备份/恢复、数据库任务、审计、死信、outbox、归档表全部覆盖新表；恢复后不重建任何旧文档 scope。
8. 验证硬切：新增“旧链路不可用”proof，失败条件包括任何 runtime 代码仍读取旧 snapshot、旧 map JSON、旧 mailbox JSON 或旧 business `persistent_documents`。

### 商用级完成标准

- `rg` 级静态门禁确认 runtime、network、persistence、http 普通服务不再读取旧业务 snapshot 或 `persistent_documents` 作为真源。
- 所有“下次还在”的状态都有专表、主键、版本/水位、恢复路径、备份/恢复路径和清理策略。
- 所有资产变化都有 `durable_operation_log + outbox_event + asset_audit_log` 或等价强事务审计链。
- `verify:replace-ready:with-db / shadow / acceptance / full` 全部通过；单独的 `with-db` 不能代表完成。
- 停服迁移 dry-run、真实迁移、回滚演练、备份恢复演练都通过。
- kill 进程、worker 重启、节点 lease 丢失、重复 operation、延迟 outbox、死信重试都不会造成资产重复或丢失。
- 1000 玩家、1000 地图实例基准下，row-level flush 不退化成整表/整实例 delete+insert，p95/p99 和 backlog 有报告。

## 2026-04-28 本轮硬切落地结果

本轮已经完成“旧业务真源退出正式 runtime”的代码硬切，覆盖：

## 实施优先级与依赖关系

历史阶段只保留为工作拆分索引。新的执行顺序是“一次性硬切版本”，不再允许某阶段先上线、靠旧链路兜底。

```text
硬切准备：schema freeze + 数据盘点 + 迁移 dry-run + 回滚脚本
  -> 并行任务包：玩家恢复硬切 / 资产 durable 硬切 / 实例 row-delta 硬切 / 低频业务专表化 / 运维备份恢复硬切
  -> 集成门禁：旧链路静态门禁 + 单域 smoke + with-db
  -> 停服迁移：备份 -> DDL -> 离线转换 -> 校验 -> 隔离旧表/旧 scope
  -> 正式部署：无兼容 runtime 代码
  -> 验收门禁：shadow -> acceptance -> full -> 压测 -> 故障注入 -> 备份恢复演练
```

- **资产 durable 硬切是阻塞项**：任何发钱、发物、扣费、领取、交易、奖励入口仍有 direct runtime fallback，整体验收失败。
- **恢复硬切是阻塞项**：玩家或实例恢复仍能回退旧 snapshot / 旧 JSON / starter fallback 继续进入正式服，整体验收失败。
- **高频实例域 row-delta 是阻塞项**：普通刷盘仍出现整实例 delete+insert 或 interval 常规刷 `monster_runtime`，整体验收失败。
- **低频业务专表化是阻塞项**：宗门、GM、兑换码、建议、市场、爱发电、数据库任务等仍把 `persistent_documents` 当真源，整体验收失败。
- **门禁必须一起过**：`with-db` 只是数据库主链证明，不能替代 `shadow / acceptance / full` 和停服迁移演练。
