# 商业级数据落盘改造计划

> 基于 [mmo商业级数据落盘方案.md](../mmo商业级数据落盘方案.md) 的终局设计，结合当前 next 主线代码现状制定。

---

## 现状诊断

### 当前持久化架构

| 数据域 | 当前存储 | 问题 |
|---|---|---|
| **玩家全量** | `server_next_player_snapshot` 单表，`payload` JSONB 整档写入 | 写放大极高，改一格背包重写整档 |
| **玩家身份** | `server_next_player_identity` 专表 | ✅ 已拆分 |
| **地图实例** | `persistent_documents` 通用表，`scope=server_next_map_aura_v1`，整包 JSON | 写放大高，无法按域恢复 |
| **邮件** | `persistent_documents`，`scope=server_next_mailboxes_v1`，整箱 JSON | 无法单封操作 |
| **坊市** | `persistent_documents`，三个 scope，已有事务 | 缺幂等，缺审计 |
| **兑换码/建议** | `persistent_documents` | 低频，暂可保留 |

### 当前 Flush 架构

| 组件 | 参数 | 问题 |
|---|---|---|
| `PlayerPersistenceFlushService` | 5s 间隔，batch 24，并行 4 | 整档写，无分域 dirty |
| `MapPersistenceFlushService` | 5s 间隔，batch 16，并行 3 | 整包写，无分域 dirty |

### 当前 Dirty 追踪

- 玩家：`persistentRevision > persistedRevision`，全局单一 revision，无域级标记
- 地图：`instance.isPersistentDirty()` + container dirty set，无域级标记

### 关键缺失

- ❌ 玩家数据分域专表（wallet/inventory/equipment/location/vitals 全在一个 JSON）
- ❌ 实例目录真源（`instance_catalog`）
- ❌ 实例状态分域（tile resource/ground item/container/overlay 全在一个 JSON）
- ❌ 强持久化命令链（`durable_operation_log` / `outbox_event` / `operation_id`）
- ❌ 域级 dirty 追踪
- ❌ 多 flush worker
- ❌ 恢复水位（`player_recovery_watermark` / `instance_recovery_watermark`）
- ❌ 实例 lease / fencing
- ❌ 资产审计日志
- ❌ 分布式节点注册与发现（多节点部署的前提）
- ❌ 玩家会话路由（哪个节点拥有哪个玩家）
- ❌ 跨节点玩家转移协议
- ❌ 数据库连接池管理（当前每个 service 各自 `new Pool()`，多节点下连接数爆炸）
- ❌ Outbox 消费者架构（多节点下谁消费 outbox）
- ❌ 邮件分域拆表（当前整箱 JSON，无法单封操作，影响附件领取的强持久化）
- ❌ 容量规划与压测基准

- [ ] 玩家域默认使用“列式主字段 + 局部 `JSONB` 细节字段”口径，不再设计“每张表一个大 JSON”

## 阶段 1.5：强资产命令链

> 目标：货币、背包、装备等资产操作不再依赖异步 flush，走即时事务提交。

### 1.5.1 命令链基础设施

- [ ] 设计并创建 `durable_operation_log` 表（`operation_id, operation_type, aggregate_type, aggregate_id, player_id, request_id, payload_jsonb, status, error_code, created_at, committed_at`）
- [ ] 设计并创建 `outbox_event` 表（`event_id, operation_id, topic, partition_key, payload_jsonb, status, attempt_count, next_retry_at, created_at, delivered_at`）
- [ ] 设计并创建 `asset_audit_log` 表（`log_id, operation_id, player_id, asset_type, asset_ref_id, action, delta_jsonb, before_jsonb, after_jsonb, created_at`）
- [ ] 设计并创建 `player_recovery_watermark` 表（`player_id, identity_version, presence_version, anchor_version, position_checkpoint_version, vitals_version, progression_version, attr_version, wallet_version, inventory_version, market_storage_version, equipment_version, technique_version, body_training_version, buff_version, quest_version, map_unlock_version, combat_pref_version, profession_version, active_job_version, logbook_version, updated_at`）
- [ ] 建立幂等唯一键索引

### 1.5.2 强持久化事务路径

- [ ] 实现 `DurableOperationService`，提供 `executeAssetMutation(operationId, mutation)` 
- [ ] 货币增减 → 在同一事务中写 `durable_operation_log` + 修改 `player_wallet` + 更新 `player_recovery_watermark`
- [ ] 背包变更 → 在同一事务中写 `durable_operation_log` + 修改 `player_inventory_item / player_market_storage_item` + 更新 watermark
- [ ] 装备穿卸 → 在同一事务中写 `durable_operation_log` + 修改 `player_equipment_slot` + `player_inventory_item` + 更新 watermark
- [ ] 技艺作业创建/取消/完成 → 在同一事务中写 `durable_operation_log` + 修改 `player_active_job` + 必要时修改 `player_wallet / player_inventory_item / player_enhancement_record` + 更新 watermark
- [ ] 事务提交成功后才返回成功结果

### 1.5.3 运行时集成

- [ ] `WorldRuntimeService` 中资产操作流程改为：tick 内判定 → 产出 mutation intent → 生成 `operation_id` → 调用 `DurableOperationService` → 成功后更新内存态
- [ ] 邮件附件领取走强持久化路径
- [ ] 坊市成交走强持久化路径

### 1.5.4 验证

- [ ] 验证资产操作幂等性（重复 operation_id 不重复生效）
- [ ] 验证事务失败不影响运行时状态
- [ ] 验证 watermark 正确推进

## 阶段 2.5：分布式基础设施 — Lease、路由与跨节点协调

> 目标：为多节点部署建立完整的所有权、路由和协调基础。这是从单服走向分布式的关键阶段。

### 2.5.1 节点注册与发现

- [ ] 设计并创建 `node_registry` 表（node_id, address, port, status, heartbeat_at, started_at, capacity_weight）
- [ ] 实现节点启动自注册
- [ ] 实现节点心跳（10-30s 周期）
- [ ] 实现节点故障检测（心跳超时 → 标记 suspect → 超时 → 标记 dead）
- [ ] 实现节点优雅下线（drain players → release leases → deregister）

### 2.5.2 实例 Lease 机制

- [ ] 实现 CAS（compare-and-swap）认领逻辑：`UPDATE instance_catalog SET assigned_node_id=$1, lease_token=$2, ownership_epoch=ownership_epoch+1 WHERE instance_id=$3 AND (assigned_node_id IS NULL OR lease_expire_at < now())`
- [ ] 实现周期性续约（lease TTL 建议 30-60s，续约周期 10-20s）
- [ ] 续约失败则停止写入并下线实例
- [ ] 节点故障后，其他节点可通过 lease 过期自动接管

### 2.5.3 Fencing

- [ ] 所有实例状态写入必须附带 `ownership_epoch`
- [ ] 数据库侧校验：`WHERE ownership_epoch = $expected_epoch`，旧 epoch 写入自动失败
- [ ] 运行时 fencing guard：写入前校验本节点 lease 是否仍有效

### 2.5.4 玩家会话路由

- [ ] 设计并创建 `player_session_route` 表或 Redis key（player_id → node_id 映射）
- [ ] 玩家登录时注册路由：`player:{playerId}:node → node_id`
- [ ] 玩家下线时清除路由
- [ ] Gateway 层根据路由将 socket 连接导向正确节点
- [ ] 路由不存在时分配到负载最低的节点

### 2.5.5 跨节点玩家转移

- [ ] 定义跨节点转移协议：
  1. 源节点 flush 玩家所有 dirty 域
  2. 源节点写 `player_position_checkpoint` 到目标实例，并按需要更新 `player_world_anchor`
  3. 源节点释放玩家会话路由
  4. 目标节点从分域表恢复玩家
  5. 目标节点注册新路由
- [ ] 实现转移超时与失败回滚
- [ ] 实现转移期间消息缓冲（避免丢包）

### 2.5.6 Outbox 消费架构

- [ ] 设计 Outbox 消费策略：每个节点只消费自己产出的 outbox 事件
- [ ] 或：独立 outbox-worker 进程轮询 `outbox_event` 表
- [ ] 消费成功后标记 `status=delivered`
- [ ] 消费失败进入 retry → dead-letter
- [ ] 多节点下保证幂等消费（通过 `event_id` 唯一约束）

### 2.5.7 Split-brain 防护

- [ ] 旧节点 lease 过期后，必须立即停止所有该实例的内存写入
- [ ] 定期扫描本节点持有的实例 lease，与数据库 catalog 对账
- [ ] 发现 lease 被其他节点抢占时，立即放弃本地实例
- [ ] 所有强持久化事务在提交时二次校验 lease 有效性

### 2.5.8 数据库连接池统一管理

- [ ] 抽取统一的 `DatabasePoolProvider`，替代当前各 service 各自 `new Pool()` 的模式
- [ ] 配置连接池参数：单节点建议 `max=20-30`，多节点通过 PgBouncer 收敛
- [ ] 引入 PgBouncer 或 Supavisor 做连接复用（4 节点 × 30 连接 = 120，需要 pooler 控制总连接数）

### 2.5.9 验证

- [ ] 验证 lease 认领与续约流程
- [ ] 验证 fencing 阻止过期写入
- [ ] 验证节点故障后实例自动接管
- [ ] 验证跨节点玩家转移正确性
- [ ] 验证 split-brain 场景下不会出现双写

## 实施优先级与依赖关系

```
阶段 1（止血）──→ 阶段 1.5（强资产）──→ 阶段 5（清理旧快照+邮件分域）
     │                    │
     └───→ 阶段 2（目录化）──→ 阶段 2.5（分布式基础设施）──→ 阶段 3（实例分域）──→ 阶段 4（Worker化）──→ 阶段 4.5（物理层）
                                                                                                              │
                                                                                      阶段 6（观测运维） ←──────┘
                                                                                              │
                                                                                      阶段 7（容量规划压测）
```

- **阶段 1 是最高优先级**：直接解决"几百玩家整档刷盘堵几分钟"
- **阶段 1.5 紧随其后**：资产安全是商业级底线
- **阶段 2/2.5** 可与阶段 1.5 并行设计，串行实施
- **阶段 2.5 是分布式部署的前提**：没有 lease/fencing/路由/节点注册，就不能多节点
- **阶段 3** 依赖阶段 2 的 catalog 基础
- **阶段 4/4.5** 依赖阶段 1+3 的分域基础
- **阶段 5** 在阶段 1+1.5 完成后即可开始
- **阶段 6** 可渐进补齐，每个阶段完成时补对应指标
- **阶段 7** 贯穿始终，但正式压测在阶段 4 之后
