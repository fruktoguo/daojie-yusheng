# 稀疏坐标地图底层重构计划

> 本文用于设计 next 主线地图底层从“矩形二维数组 + `tileIndex = y * width + x`”重构为“稀疏坐标地块平面 + 高性能 cell handle”的实施方案。
>
> 本文仅为开发计划，不表示当前主线已经完成。主工作区固定为 `packages/*`；旧行为审计输入默认看 `main` 同名 `packages/*` 或 `参考/main-packages-ref`。

---

## 目标结论

地图运行时不应继续把“地图存在性”绑定到矩形 `width * height` 数组。最终底层要支持：

- 任意坐标查询：传入 `x / y`，返回稳定的运行时格子引用。
- 非矩形地图：不存在的坐标就是虚空，不再强制补齐外接矩形。
- 动态扩展：只新增尚不存在的格子，已有区域不重复生成、不覆盖、不迁移。
- 扩展内容可配置：扩展时每个新格子生成什么，由传入的 generator 决定。
- 热路径低 GC：寻路、占用、攻击地块、阵法、AOI 不在每次查询中创建对象，不拼字符串 key，不走 JSON。
- 坐标语义不变：玩法规则、持久化、GM 审计仍以 `instanceId + x + y` 表达地图位置。

第一阶段目标不是一次性重写所有地图系统，而是先引入统一底层抽象：

```text
静态地图模板 / 动态宗门扩展 / 后续洞府秘境
-> 启动期或 tick 内受控写入 RuntimeTilePlane
-> 热路径只通过 CellHandle 读写地块、占用、资源和动态状态
```

## 当前代码事实

### 当前矩形依赖

当前 `MapTemplateRepository` 在启动期把地图模板编译成固定矩形结构：

- `width * height`
- `walkableMask`
- `blocksSightMask`
- `portalIndexByTile`
- `safeZoneMask`
- `baseAuraByTile`
- `baseTileResourceEntries`

当前 `MapInstanceRuntime` 也沿用矩形数组：

- `occupancy = new Uint32Array(width * height)`
- `auraByTile = new Int32Array(baseAuraByTile)`
- `toTileIndex(x, y) = y * width + x`
- `isInBounds(x, y) = x >= 0 && y >= 0 && x < width && y < height`
- 地块伤害、地面物品、容器、NPC、怪物、可见性等大量结构都以 `tileIndex` 作为运行时索引。

这套结构对静态矩形地图很快，但对宗门这类动态扩展地图会产生两个问题：

- 扩展一侧时必须重建完整外接矩形，新增大量玩家不可见、暂时不会用到的格子。
- 坐标存在性被矩形边界代替，无法表达“不规则地图边缘”和局部生成。

### 不能破坏的主链

重构后必须保持这些主链行为不变：

| 主链 | 必须保持 |
|---|---|
| 寻路 | 不存在格子视为不可走；已存在格子的 walkable、占用、NPC/怪物阻挡语义不变 |
| 移动 | 玩家输入仍由服务端收集，tick 内执行，占位仍由服务端唯一维护 |
| 攻击地块 | `getTileCombatState(x,y)` 与 `damageTile(x,y,damage)` 语义不变 |
| 地块恢复 | 普通阵法固脉和宗门内置固脉仍能阻止恢复 |
| 阵法 | 范围、减伤、边界阻挡按坐标判断，不依赖矩形数组 |
| AOI / 视野 | 仍按玩家视野范围投影；不存在格子返回空态或虚空态 |
| 传送点 | 动态 portal 仍按坐标查找、投影和持久化 |
| 持久化 | 下次还在的状态必须落 PostgreSQL 分域表或明确 checkpoint |

## 设计原则

### 地图真源分层

必须继续区分三层真源：

| 层 | 说明 |
|---|---|
| 静态模板真源 | `packages/server/data/maps/*`，描述初始地图模板 |
| 运行时地块真源 | `RuntimeTilePlane`，描述实例内当前真实存在的地块集合 |
| 持久化动态真源 | 结构化表或 checkpoint，描述跨重启仍要存在的动态地块和状态 |

静态模板不再等同于运行时完整地图。模板只是初始输入；实例运行后，真实地块集合以 `RuntimeTilePlane` 为准。

### 坐标是规则真源，handle 是热路径真源

- `x / y` 是玩法、协议、持久化、GM、审计使用的稳定语义。
- `CellHandle` 是进程内热路径引用，不作为长期持久化真源。
- 任何状态保存时必须能回到 `x / y`，不能只保存 handle。

### 扩展策略和格子生成分离

动态扩展必须拆成两层：

- `ExpansionShape`：决定本次尝试生成哪些坐标。
- `TileGenerator`：决定某个新增坐标生成什么格子。

底层只提供高性能插入与查询，不内置“宗门一定 5x5”这类业务规则。

## 核心类型设计

### CellHandle

```ts
type CellHandle = number;
```

约定：

- `0` 表示坐标不存在。
- 非 `0` 表示运行时格子引用。
- handle 可以编码为 `(chunkIndex + 1) << LOCAL_BITS | localIndex`。
- handle 只在当前进程和当前实例 runtime 内稳定，不跨重启持久化。

### RuntimeTilePlane

```ts
interface RuntimeTilePlane {
  has(x: number, y: number): boolean;
  getHandle(x: number, y: number): CellHandle;

  getX(handle: CellHandle): number;
  getY(handle: CellHandle): number;

  getTileType(handle: CellHandle): TileType;
  setTileType(handle: CellHandle, tileType: TileType): void;

  getFlags(handle: CellHandle): number;
  setFlags(handle: CellHandle, flags: number): void;

  getOccupancy(handle: CellHandle): number;
  setOccupancy(handle: CellHandle, occupancyHandle: number): void;

  getPortalIndex(handle: CellHandle): number;
  setPortalIndex(handle: CellHandle, portalIndex: number): void;

  getResource(resourceKeyId: number, handle: CellHandle): number;
  setResource(resourceKeyId: number, handle: CellHandle, value: number): void;

  activateCell(x: number, y: number, seed: MutableTileSeed): CellHandle;
  deactivateCell(handle: CellHandle): boolean;
}
```

热路径调用要求：

- `getHandle()` 不分配对象。
- `getTileType()` / `getFlags()` / `getOccupancy()` 只做 typed array 读取。
- 不在热路径拼接 `x,y` 字符串。
- 不在热路径返回 `{ x, y, tileType }` 这种短命对象。

### MutableTileSeed

扩展生成使用可复用 seed，避免每格生成对象。

```ts
interface MutableTileSeed {
  tileType: TileType;
  flags: number;
  aura: number;
  portalIndex: number;
  safeZone: number;
  resourceKeyId: number;
  resourceValue: number;
}
```

实际实现可用类或结构化参数对象，但调用方必须复用同一个 seed 实例：

```ts
resetTileSeed(out);
generator.generate(x, y, ctx, out);
plane.activateCell(x, y, out);
```

### ExpansionShape

```ts
interface ExpansionShape {
  collect(originX: number, originY: number, writer: CoordinateWriter): void;
}

interface CoordinateWriter {
  push(x: number, y: number): void;
}
```

形状策略只负责提交候选坐标，不负责判断是否已存在。底层执行扩展时统一跳过已存在坐标：

```ts
if (plane.getHandle(x, y) !== 0) {
  continue;
}
```

可选策略：

- 方形半径：`radius = 1 / 2 / 3`
- 菱形半径
- 单方向扇形
- 噪声边缘
- GM 指定坐标列表
- 由地脉、宗门等级或地图 seed 推导的形状

### TileGenerator

```ts
interface TileGenerator<TContext> {
  generate(x: number, y: number, context: TContext, out: MutableTileSeed): void;
}
```

生成器只对不存在的坐标执行，必须满足：

- 不覆盖已有格子。
- 不直接改玩家、怪物、阵法等非地块状态。
- 不在生成器里做数据库 IO。
- 不返回对象，写入复用 `out`。
- 生成结果立即写入 plane，并标记对应 dirty domain。

宗门默认生成器：

```text
核心附近已经存在的空地不重复处理
新增坐标按宗门规则生成石头或地板
石头耐久仍由坐标圈层 resolver 计算
后续可按宗门等级、方向、地脉 seed 生成矿、灵脉、特殊墙体
```

## 底层存储方案

### Chunk 只是存储页，不是玩法规则

内部建议使用固定小 chunk 作为存储页，以减少 hash 表条目数量。chunk 大小是性能参数，不是业务规则。

```ts
const CHUNK_SIZE = 8; // 可 benchmark 后确定
const CHUNK_AREA = CHUNK_SIZE * CHUNK_SIZE;
```

选择 `8x8` 而不是把业务上的 `5x5` 写入底层，是为了：

- handle 编码更简单。
- typed array 对齐更稳定。
- 扩展形状可以任意大小，不和存储页绑定。
- 后续 benchmark 可调整为 `4x4 / 8x8 / 16x16`。

### Chunk 索引

热路径不使用 `Map<string, number>`。建议实现 primitive open addressing：

```ts
class ChunkIndex {
  private keyX: Int32Array;
  private keyY: Int32Array;
  private value: Int32Array; // chunkIndex + 1；0 表示空槽
  private mask: number;

  find(chunkX: number, chunkY: number): number;
  getOrInsert(chunkX: number, chunkY: number): number;
}
```

注意：

- hash 输入是两个 int，不拼字符串。
- 扩容发生在冷路径或扩展动作中，不能在寻路邻居检查里频繁 rehash。
- 宗门扩展属于受控世界状态变更，必须在 tick 执行链或服务端明确动作链中完成。

### Cell 数据列

基础列：

| 列 | 类型 | 说明 |
|---|---|---|
| `activeMask` | `Uint32Array` 或 `BigUint64Array` | chunk 内真实存在格子 |
| `tileType` | `Uint16Array` | 地块类型 |
| `flags` | `Uint16Array` | walkable、blocksSight、damageable、safeZone 等 |
| `occupancy` | `Uint32Array` | 玩家占用 handle |
| `portalIndex` | `Int32Array` | 静态或动态 portal 索引，`-1` 表示无 |
| `aura` | `Int32Array` | 默认灵气资源 |

动态列：

| 列 | 建议 |
|---|---|
| 多资源桶 | 按 `resourceKeyId -> Int32Array` 延迟分配 |
| 地块伤害 | `CellHandle -> TileDamageState` 热路径可用 typed columns，第一阶段可保留 Map 过渡 |
| 地面物品 | `CellHandle -> GroundPile` 第一阶段可保留 Map 过渡 |
| 容器 / NPC / landmark | 先保持 ID 索引，tile 侧只存 ID 或索引 |

### 坐标边界缓存

plane 可维护以下冷路径统计：

- `minX`
- `maxX`
- `minY`
- `maxY`
- `activeCellCount`
- `chunkCount`

这些字段用于 GM、保存、遍历和粗略视口裁剪，不能作为“格子是否存在”的判断依据。真实存在性必须问 `getHandle(x,y)`。

## 对现有系统的兼容设计

### 寻路

路径计算必须从矩形边界判断切到 plane 查询：

```ts
function canEnter(x: number, y: number): boolean {
  const handle = plane.getHandle(x, y);
  if (handle === 0) return false;
  if ((plane.getFlags(handle) & TILE_FLAG_WALKABLE) === 0) return false;
  if (plane.getOccupancy(handle) !== 0) return false;
  if (monsterIndex.has(handle) || npcIndex.has(handle)) return false;
  return true;
}
```

要求：

- 邻居枚举仍可用四方向或八方向坐标。
- 不存在格子自然不可走。
- A* open / closed 不再依赖 `width * height` 数组。
- 第一阶段可用数字坐标 key 过渡，但不能拼字符串；建议使用 packed coordinate key 或 handle。
- 对不存在但未来可扩展的坐标，寻路不提前穿越。

### 移动与占位

移动校验只关心目标坐标：

```text
getHandle(nextX,nextY) === 0 -> 不可进入
非 walkable -> 不可进入
occupancy 非空 -> 不可进入
怪物/NPC 阻挡 -> 不可进入
```

玩家位置仍保存 `x / y`。占位写入从：

```ts
occupancy[toTileIndex(x,y)] = handle;
```

改为：

```ts
plane.setOccupancy(cellHandle, playerHandle);
```

### 攻击地块

外部 API 保持：

```ts
getTileCombatState(x: number, y: number)
damageTile(x: number, y: number, damage: number)
```

内部流程：

```text
getHandle(x,y)
-> 不存在返回 null
-> 读取 tileType / flags
-> 不可伤害返回 null
-> 读取或创建 tile damage state
-> 写入伤害
-> 标记 tile_damage dirty
```

地块耐久 resolver 必须改成坐标语义：

```ts
resolveTileDurability(instanceRules, x, y, tileType)
```

宗门石头耐久继续按“相对核心坐标圈层”计算，不依赖矩形中心。

### 阵法

阵法范围必须继续按坐标判断：

```ts
if (!plane.has(x, y)) return false;
return formation.contains(x, y);
```

影响面：

- `isTerrainStabilized(instanceId,x,y)`
- `mitigateTerrainDamage(instanceId,x,y,damage)`
- 护宗大阵边界阻挡
- 阵法投影
- 阵眼坐标

不存在的虚空格：

- 不参与地块恢复。
- 不参与地块减伤。
- 不可被攻击。
- 不阻挡视线，除非后续规则明确虚空阻挡。

### 视野与 AOI

服务端视野仍按玩家视野半径扫描坐标窗口：

```text
for y = selfY - radius .. selfY + radius
for x = selfX - radius .. selfX + radius
```

每个坐标：

```text
handle = plane.getHandle(x,y)
handle === 0 -> tile = null 或 void
handle !== 0 -> 构建 tile sync state
```

第一阶段协议可保持矩阵形状，只是矩阵内允许空格子。后续优化包体时再改为稀疏 patch：

```ts
interface SparseTilePatch {
  x: number;
  y: number;
  tile: TileSyncState | null;
}
```

### 传送点、NPC、容器、地标

这些仍然按坐标挂载，但内部索引要从 `tileIndex` 改成 `CellHandle`：

| 当前 | 目标 |
|---|---|
| `portalIndexByTile[tileIndex]` | `plane.getPortalIndex(handle)` |
| `npcIdByTile: Map<number,string>` | `npcIdByCell: Map<CellHandle,string>` 或 typed id index |
| `containerIdByTile` | `containerIdByCell` |
| `landmarkIdByTile` | `landmarkIdByCell` |

持久化仍写 `x / y`。恢复时先 `getHandle(x,y)`，不存在则按数据一致性策略：

- 静态模板数据：启动期创建对应 cell。
- 动态 overlay：如果 overlay 要求存在地块，则先恢复地块；否则记录诊断并跳过。
- 旧脏数据：进入迁移告警，不默默生成错误地块。

## 动态扩展设计

### 扩展入口

统一入口建议放在实例运行时或独立动态地形服务：

```ts
interface DynamicTerrainExpansionService {
  expand(
    instance: MapInstanceRuntime,
    originX: number,
    originY: number,
    shape: ExpansionShape,
    generator: TileGenerator<unknown>,
    context: unknown,
  ): TerrainExpansionResult;
}
```

返回结果避免 per-cell 对象数组，使用统计和可选复用 buffer：

```ts
interface TerrainExpansionResult {
  attempted: number;
  created: number;
  skippedExisting: number;
  rejected: number;
  worldRevisionChanged: boolean;
}
```

### 重复扩展规则

动态扩展必须满足：

```text
候选坐标存在 -> 跳过
候选坐标不存在 -> 调 generator -> activateCell
```

不能因为二次扩展覆盖：

- 玩家挖开的通道
- 被破坏地块状态
- 容器库存
- 地面物品
- 手工 GM 修改
- 后续建筑/装修状态

### 宗门默认扩展

宗门当前默认可以使用：

```text
破坏边界或特殊触发点
-> 以触发坐标为 origin
-> 选择一个策略形状，例如半径 2 方形、菱形或按地脉方向的形状
-> 对不存在坐标调用宗门生成器
```

宗门生成器初版规则：

- 靠近核心的既有区域不覆盖。
- 新增地块默认生成石头。
- 可配置内圈或指定规则生成地板。
- 地块耐久按相对核心坐标计算。
- 可根据宗门等级、方向、地脉 seed 后续生成矿脉、灵脉、阵基槽、特殊墙。

### 生成器可修改能力

为后续支持修改，生成上下文必须带规则来源：

```ts
interface SectTileGenerationContext {
  sectId: string;
  coreX: number;
  coreY: number;
  sectLevel: number;
  seed: number;
  ruleVersion: number;
  directionMask: number;
}
```

后续可由宗门管理、GM、地图编辑器或配置文件修改：

- 扩展形状。
- 新增格子的默认地形。
- 特殊资源概率。
- 特殊不可破坏结构。
- 是否生成装饰或可交互物。

但修改规则只影响未来新增格子；已有格子除非走明确编辑/建筑/破坏动作，否则不被 generator 回写。

## 持久化设计

### 持久化真源

新增或扩展实例地块分域表，目标字段：

| 字段 | 说明 |
|---|---|
| `instance_id` | 地图实例 |
| `x` / `y` | 坐标真源 |
| `tile_type` | 当前基础地块 |
| `flags` | 当前地块 flags |
| `base_aura` | 基础灵气 |
| `rule_source` | `template / generated / gm / building / migration` |
| `rule_version` | 生成规则版本 |
| `created_at` / `updated_at` | 审计 |

可以命名为：

```text
instance_tile_cell
```

已有分域继续独立：

- `instance_tile_damage_state`
- `instance_tile_resource_state`
- `instance_ground_item`
- `instance_container_state`
- `instance_overlay_chunk`

### 保存策略

动态扩展时：

```text
activateCell
-> 标记 tile_cell dirty
-> 如有资源，标记 tile_resource dirty
-> 如有 portal / overlay，标记 overlay dirty
-> 提升 worldRevision
-> 提升 persistentRevision
```

不要每次扩展保存整张地图。只 upsert 新增或变更 cell。

### 恢复策略

恢复顺序：

1. 从静态模板构建初始 `RuntimeTilePlane`。
2. 从 `instance_tile_cell` 加载动态新增或覆盖 cell。
3. 加载 tile damage。
4. 加载 tile resource。
5. 加载 ground item / container / monster / overlay。
6. 重建 portal、NPC、container、monster、occupancy 等运行时索引。

如果动态状态引用不存在 cell：

- 能明确从同批 `tile_cell` 恢复的，先恢复 cell。
- 不能恢复的，记录诊断并跳过该动态状态。
- 不在热恢复链里用 generator 猜测补格子。

### 旧数据迁移

迁移期间要支持旧 `tileIndex`：

```text
旧 tileIndex + 旧 width -> x/y
```

迁移目标：

- 旧矩形静态图启动期进入 plane。
- 旧 instance 分域中保存的 tileIndex 在读入时转换为 x/y。
- 新写入只写 x/y，不再新增 tileIndex 真源。

## 协议与客户端兼容

### 第一阶段协议不强制大改

客户端仍可以接收玩家视野窗口矩阵：

- 视野窗口仍是矩形。
- 地图本体可以非矩形。
- 不存在的格子投影为 `null`、`void` 或不可见 tile。

这能减少前端改动范围，先把服务端正确性收口。

### 后续稀疏协议

后续可增加稀疏 tile patch：

```ts
interface VisibleSparseTileSync {
  x: number;
  y: number;
  tile: TileSyncState | null;
}
```

使用场景：

- 宗门边界空洞多。
- 秘境或洞府非矩形明显。
- 视野窗口中 `null` 占比高，需要降低包体。

协议原则：

- 高频包不发送静态长文本。
- 地块静态和动态分层。
- 动态扩展只广播 AOI 内新增/变化格子。
- 不因扩展全量重发整张地图。

## 风险与边界

### 高风险点

| 风险 | 处理 |
|---|---|
| 一次性改太多 tileIndex 调用 | 分阶段迁移，每阶段保持兼容 API |
| handle 被误当持久化真源 | 类型命名和持久化接口强制用 x/y |
| 稀疏地图破坏客户端矩阵渲染 | 第一阶段协议仍输出视野矩阵，空格为空态 |
| 寻路性能下降 | primitive hash + handle，补 benchmark |
| 恢复链乱生成地块 | 恢复只读持久化 cell，不用 generator 猜测 |
| 动态扩展绕过 tick | 扩展动作必须进入服务端受控状态变更链 |

### 非目标

- 不在第一阶段改客户端为完全稀疏渲染。
- 不把所有静态地图 JSON 立刻改成新格式。
- 不把宗门扩展规则写死成固定 `5x5`。
- 不让 generator 覆盖已有格子。
- 不让客户端承担移动、碰撞、攻击地块或阵法正确性判断。

## 推荐执行顺序

优先顺序：

1. 先做 `RuntimeTilePlane` 和静态模板加载。
2. 再迁移占用、移动、寻路这条正确性主链。
3. 再迁移攻击地块、地块恢复、阵法。
4. 再接动态扩展服务和宗门扩展。
5. 最后收口持久化分域和协议稀疏优化。

不要先改宗门扩展外观。底层没有收口前继续在矩形模板上做扩展，只会把后续迁移成本继续放大。
