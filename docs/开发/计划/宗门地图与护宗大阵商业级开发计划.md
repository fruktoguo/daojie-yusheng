# 宗门地图与护宗大阵商业级开发计划

> 本文面向 next 主线实现“建宗令、宗门地图、宗门入口、宗门管理核心、护宗大阵”的开发落地。
>
> 主工作区固定为 `packages/*`。旧行为审计输入默认看 `main` 同名 `packages/*` 或 `参考/main-packages-ref`，不作为新功能落点。

---

## 目标结论

宗门不是一个普通地图模板，也不是单个传送点。宗门应作为一个持久化领域对象落地：

- 玩家使用 `建宗令` 后，在当前主世界位置创建一个可交互宗门入口。
- 服务端创建独立宗门地图实例，初始为 `3x3` 全石头地图。
- 宗门地图中心生成 `宗门管理核心`，后续宗门操作都由核心承接。
- 宗门地图可按圈扩展，逻辑上无上限；当前运行时以“有限激活矩形 + 按圈扩容”实现。
- 宗门石头耐久按圈翻倍，第一圈基础血量 `100000`。
- 宗门内部以核心为中心自带半径 `8` 的 `17x17` 固脉效果，只阻止地块复生和临时地块自然消散，不提供地块减伤。
- 护宗大阵位于主世界入口处，阵眼在宗门核心内；本宗门玩家可自由通行，外宗或无宗门玩家按大阵阻挡规则处理。

第一版验收闭环：

```text
建宗令 -> 创建宗门真源 -> 创建宗门地图 -> 创建主世界入口 -> 可进入宗门 -> 可返回主世界
       -> 宗门核心可交互 -> 护宗大阵生效 -> 重启后入口、地图、核心、大阵仍存在
```

### 必须补齐的缺口

- 缺正式宗门真源：宗门 ID、名称、宗主、入口、宗门地图、核心、扩展半径都还没有统一持久化模型。
- 缺宗门成员真源：仅有玩家 `sectId` 字段不够支撑权限、通行、退出、邀请和宗主转让。
- 缺 `建宗令` 的道具配置和服务端专用使用分支。
- 缺动态传送点的完整创建、投影、持久化和回读链。
- 缺宗门地图生成器和按圈扩展机制。
- 缺宗门石头耐久的按圈计算逻辑。
- 缺宗门内置半径 `8` 的 `17x17` 固脉规则，不能直接复用固脉阵减伤效果。
- 缺宗门管理核心实体、交互动作和面板。
- 缺宗门大阵和宗门核心之间的生命周期绑定。

## 领域模型

### 宗门主记录

建议新增 `sect_state` 或等价结构化真源表：

| 字段 | 说明 |
|---|---|
| `sect_id` | 宗门唯一 ID |
| `name` | 宗门名，第一版可默认生成，后续允许改名 |
| `founder_player_id` | 创建者玩家 ID |
| `leader_player_id` | 当前宗主玩家 ID |
| `status` | `active / dissolved / locked` |
| `entrance_instance_id` | 主世界入口所在实例 |
| `entrance_x` / `entrance_y` | 主世界入口坐标 |
| `sect_instance_id` | 宗门地图实例 ID |
| `core_x` / `core_y` | 宗门核心坐标 |
| `expansion_radius` | 当前宗门地图半径，初始 `1` |
| `created_at` / `updated_at` | 创建与更新时间 |

### 宗门成员记录

建议新增 `sect_member`：

| 字段 | 说明 |
|---|---|
| `sect_id` | 宗门 ID |
| `player_id` | 玩家 ID |
| `role` | `leader / elder / member` |
| `joined_at` | 入宗时间 |
| `status` | `active / left / expelled` |

第一版只需要落 `leader` 一人，但表结构必须为后续成员系统预留。

### 宗门地图实例

宗门地图实例建议使用稳定 ID：

```text
sect:<sectId>:main
```

实例元信息：

- `kind`: `sect`
- `persistent`: `true`
- `ownerSectId`: `sectId`
- `displayName`: 宗门名或“某某宗山门”
- `routeDomain`: `sect:<sectId>`
- `defaultEntry`: 核心附近可站立点

### 宗门入口传送点

主世界入口必须是持久化 overlay，不应只写进静态地图模板。

建议模型：

```ts
interface SectEntranceOverlay {
  kind: 'sect_entrance';
  sectId: string;
  entranceId: string;
  instanceId: string;
  x: number;
  y: number;
  targetInstanceId: string;
  targetX: number;
  targetY: number;
  char: string;
  color: string;
  name: string;
}
```

宗门内返回点也走同一套传送点模型，但目标指回主世界入口附近。

## 建宗令规则

### 道具定义

建议新增消耗品：

```json
{
  "itemId": "sect_founding_token",
  "name": "建宗令",
  "type": "consumable",
  "quality": "rare",
  "description": "使用后在当前位置开辟宗门入口，并创建独立宗门地图。",
  "useBehavior": "create_sect"
}
```

字段名可按当前内容仓库解析风格调整，但必须让运行时能可靠识别，不要靠道具名字符串判断。

### 使用校验

使用 `建宗令` 时必须由服务端校验：

- 玩家在线且在世界中。
- 玩家当前位置有有效 `instanceId / x / y`。
- 玩家当前不属于任何宗门。
- 当前实例允许建宗，默认只允许主世界公共实例。
- 当前坐标不能已有宗门入口、普通传送点、不可覆盖 landmark 或其他独占交互物。
- 当前坐标不能处于不允许建造的安全限制或副本临时实例。
- 背包中确实存在 `建宗令`，且本次创建成功后才消耗。
- 数据库宗门创建、玩家入宗、入口 overlay、宗门地图、护宗大阵创建必须尽量在同一个强一致操作链内完成；失败不得出现“扣了令牌但宗门没创建”的半成品。

### 创建结果

成功后：

- 消耗 1 个 `建宗令`。
- 创建宗门记录。
- 创建宗门成员记录，创建者为宗主。
- 更新玩家运行态与持久化 `sectId`。
- 创建宗门地图实例。
- 创建主世界入口传送点。
- 创建宗门内返回点。
- 创建宗门管理核心。
- 创建护宗大阵，并绑定阵眼到宗门核心。
- 给玩家发送成功提示。
- 推送 AOI 增量，让附近玩家看到新入口。

## 宗门地图设计

### 坐标与尺寸

初始地图：

```text
半径 r = 1
尺寸 = 2r + 1 = 3
范围 = [-1, 1] x [-1, 1]
中心 = (0, 0)
```

如果当前地图运行时必须使用非负数组坐标，则使用内部偏移：

```text
runtimeX = logicalX + radius
runtimeY = logicalY + radius
```

对外规则文案、扩展计算和石头圈数都使用逻辑坐标，避免扩容时语义漂移。

### 地形

- 初始 `3x3` 全部为 `Stone`。
- 中心格放置 `宗门管理核心`。
- 核心所在格是否可站立需要明确：
  - 推荐核心作为无碰撞交互实体，中心地块仍可站立。
  - 如果核心占格，则默认入口落到核心南侧或最近可站立点。

### 按圈扩展

当前 runtime 不应直接改成真正无限大地图。第一版采用“按圈扩容”：

```text
当前半径 r
扩展后半径 r + 1
新增一圈石头地块
重新构建或扩容地图数组
迁移旧地块伤害、临时地块、实体、入口、核心、玩家位置
保存 expansion_radius
```

后续如果需要大规模宗门建设，再单独改成 chunked map runtime。

### 石头血量

宗门石头血量按中心圈数计算：

```text
ring = max(abs(logicalX), abs(logicalY))
hp = 100000 * 2 ** max(0, ring - 1)
```

示例：

| 圈数 | 覆盖范围 | 石头血量 |
|---:|---|---:|
| 1 | 初始 `3x3` 外圈 | 100000 |
| 2 | 扩展到 `5x5` 的新增外圈 | 200000 |
| 3 | 扩展到 `7x7` 的新增外圈 | 400000 |
| 4 | 扩展到 `9x9` 的新增外圈 | 800000 |

注意：

- 核心实体不使用普通石头血量。
- 宗门地图外圈石头耐久不能写死在地图模板里，应由实例规则按坐标计算。
- 破坏状态仍走现有 tile damage 持久化链。

## 宗门内置固脉规则

宗门地图自带一个以核心为中心、半径为 `8` 的 `17x17` 固脉域：

```text
abs(logicalX - coreX) <= 8
abs(logicalY - coreY) <= 8
```

只包含两类效果：

- 可攻击地块被摧毁后不会自动复生。
- 玩家技能生成的临时地块不会因时间到而自然消散。

明确不包含：

- 不提供地块减伤。
- 不使用固脉阵的 `effectAura / (effectAura + 100000)` 减伤公式。
- 不显示为普通阵法范围，除非后续单独做宗门核心可视化。

实现建议：

- 在实例元信息或宗门地图规则中标记 `innateStabilizer`。
- `advanceTileRecovery()` 判断时同时询问普通阵法固脉和宗门内置固脉。
- 临时地块过期判断也接同一个“是否被稳定”查询。
- 地块受击结算仍只读取普通地块耐久，不读取固脉减伤。

## 宗门管理核心

### 定位

宗门管理核心是宗门地图的权威交互实体，不是普通怪物、普通容器或普通阵法。

第一版职责：

- 展示宗门基础信息。
- 控制护宗大阵开启/关闭。
- 查看护宗大阵剩余灵力。
- 给护宗大阵补充灵力。
- 查看主世界入口位置。
- 触发宗门地图扩展。

后续职责：

- 宗门改名。
- 成员管理。
- 权限管理。
- 宗门建筑。
- 宗门仓库。
- 宗门资源和维护成本。

### 交互入口

服务端 context action 应返回核心操作：

- `sect:core:open`
- `sect:guardian:toggle`
- `sect:guardian:refill`
- `sect:map:expand`
- `sect:exit`

客户端应使用单实例详情弹层或现有上下文动作面板，不做整页刷新。

### 权限

第一版权限：

- 宗主可操作全部核心功能。
- 本宗门成员可查看、进入和离开。
- 非本宗门玩家默认不能操作核心。

护宗通行：

- 本宗门玩家可穿过主世界入口处护宗大阵边界。
- 非本宗门玩家按大阵阻挡。

## 护宗大阵设计

### 与普通阵法的区别

护宗大阵不通过阵盘布置，不走普通布阵面板。

它由宗门生命周期创建和管理：

- 阵眼：宗门地图核心。
- 生效位置：主世界入口。
- 默认范围：`1`。
- 通行规则：本宗门玩家可通过，其他玩家阻挡。
- 受击规则：复用现有边界阵法受击扣灵力逻辑。
- 维护规则：通过宗门核心补充灵力。
- 自然损耗：开启状态按当前剩余灵力 3 天半衰期衰减，关闭状态损耗速率为开启状态的 `1/10`。

### 创建参数

创建宗门时调用护宗大阵创建/更新逻辑：

```ts
upsertSectGuardianFormation({
  formationId: 'sect_guardian_barrier',
  ownerSectId: sectId,
  instanceId: entranceInstanceId,
  x: entranceX,
  y: entranceY,
  eyeInstanceId: sectInstanceId,
  eyeX: coreX,
  eyeY: coreY,
  radius: 1
})
```

### 生命周期

- 宗门创建时自动创建。
- 宗门入口迁移时重新绑定入口坐标。
- 宗门解散时关闭或删除。
- 宗门核心被破坏的玩法如果后续开放，应同步处理大阵失效。

### 可视规则

沿用阵法配置：

- 阵眼在宗门内部显示为宗门核心，不额外显示普通阵眼。
- 主世界入口处显示大阵边界。
- 大阵是否无需感气可见由配置决定。
- 关闭状态下只保留入口和核心，不显示边界阻挡效果。

## 协议与投影

### 服务端权威原则

- 创建宗门、消耗建宗令、生成入口、创建地图、大阵生效全部在服务端完成。
- 客户端只发送“使用道具”或“确认创建宗门”的意图。
- 客户端不能传回宗门 ID、地图实例 ID、传送目标、大阵参数等权威字段。

### 推荐协议

第一版可以复用现有 `UseItem`，如果需要宗门名确认，再新增专用请求：

```ts
interface UseSectFoundingTokenPayload {
  slotIndex: number;
  sectName?: string;
}
```

服务端响应通过现有世界增量和玩家状态更新承接：

- 玩家 `sectId` 变化。
- 当前 AOI 新增入口实体。
- 宗门创建成功 notice。

### 投影字段

入口实体投影至少需要：

- `id`
- `kind: 'sect_entrance'`
- `x / y`
- `char`
- `color`
- `name`
- `targetInstanceId` 不应直接暴露给普通客户端，除非现有传送协议需要。
- `sectId` 可只对 GM 或同宗玩家展示。

宗门核心实体投影至少需要：

- `id`
- `kind: 'sect_core'`
- `x / y`
- `char`
- `color`
- `name`
- `ownerSectId`

## 持久化与恢复

### 必须保证重启后仍存在

- 宗门记录。
- 宗门成员关系。
- 玩家 `sectId`。
- 宗门地图实例目录。
- 宗门地图扩展半径。
- 宗门地图地块破坏状态。
- 主世界入口传送点。
- 宗门内返回传送点。
- 宗门核心实体。
- 护宗大阵实例和剩余灵力。

### 建议恢复顺序

1. 加载宗门记录。
2. 加载宗门成员记录。
3. 恢复宗门地图实例目录。
4. 按宗门 `expansion_radius` 重建宗门地图基础地形。
5. 回放实例地块伤害和 overlay。
6. 恢复宗门核心与传送点。
7. 恢复护宗大阵。
8. 校验入口、核心、大阵三者引用一致性。

### 一致性修复

启动期应有轻量 repair：

- 有宗门记录但无宗门地图实例：重建实例并记录警告。
- 有宗门记录但入口 overlay 缺失：按宗门入口坐标补回。
- 有入口但宗门记录缺失：禁用入口并记录 GM 诊断。
- 有护宗大阵但无宗门记录：删除或禁用大阵。
- 玩家 `sectId` 指向不存在宗门：清空或标记异常，不能让护宗通行误判。

## UI 与交互

### 背包使用

- `建宗令` 在背包中显示为可使用物品。
- 点击使用后，如果不需要宗门名，直接进入服务端创建流程。
- 如果需要宗门名，弹出轻量确认框。
- 创建中禁用重复提交。
- 创建失败必须返回明确原因。

### 地图表现

- 主世界入口必须是可见可交互实体。
- 宗门核心必须是可见可交互实体。
- 护宗大阵边界按阵法现有显示规则渲染。
- 感气态可显示更详细阵法范围和宗门气机信息。

### 宗门核心面板

第一版面板包含：

- 宗门名。
- 宗主。
- 当前扩展半径和地图尺寸。
- 护宗大阵状态。
- 护宗大阵剩余灵力。
- 操作按钮：开启/关闭、补充、扩展、返回入口。

UI 要求：

- 不做整页刷新。
- 面板操作后局部 patch。
- 保留当前面板滚动、焦点和选中态。
- 浅色、深色、手机模式都要可用。

## 推荐开发顺序

1. 先落宗门真源和建宗令最小创建链。
2. 再落宗门地图实例和入口传送。
3. 再接宗门核心交互。
4. 再接护宗大阵绑定。
5. 再做扩圈和石头血量。
6. 最后补 UI 细节、GM 可视化和完整 with-db 验证。

不要先做复杂宗门管理 UI。没有持久化真源和恢复链之前，UI 做得越多，返工越大。
