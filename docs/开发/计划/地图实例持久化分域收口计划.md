# 地图实例持久化分域硬切计划

> 本文用于收口 next 主线地图实例持久化：从“整包 JSONB 快照兜底”一次性硬切到“热状态结构化真源 + worker row-level delta + 冷 checkpoint”。正式运行时不再保留旧 JSON 兼容恢复或兼容写入。
>
> 主工作区固定为 `packages/*`。`legacy/*` 仅可作为旧行为审计输入，不作为新实现落点。
>

## 目标结论

商业级地图持久化不应该把每息变化、地块资源、地块损坏、药草容器、地面物品、动态入口和冷启动 checkpoint 混在一个 JSON 里频繁覆盖。

2026-04-28 起，地图实例持久化按一次性硬切口径执行：

- 高频热状态写结构化表，按行、按域、按版本局部 upsert。
- JSONB 只保留在低频扩展 payload、复杂物品 raw payload、冷 checkpoint 和运营备份里；旧地图 JSON 不再是 runtime 兼容层。
- `MapPersistenceService.loadMapSnapshot()/saveMapSnapshot()` 从正式运行时、恢复、flush、GM 普通操作中删除；只允许离线转换脚本读取旧 dump。
- `MapPersistenceFlushService` 改名或降级为低频 checkpoint / shutdown / diagnostic，不能参与每轮普通 dirty flush。
- 地图 tick 只作为运行时节拍，不因每息推进而标脏；需要跨重启保留的计时状态要用明确的 deadline / remaining / checkpoint 策略。
- 所有“下次还在”的地图状态必须以 PostgreSQL 结构化表或明确 checkpoint 为真源，不能只存在运行时、Redis 或前端缓存。
- 迁移失败即阻断启动，不允许回退旧地图快照继续服务。

硬切验收闭环：

```text
破坏地块 / 改变灵气 / 采集药草 / 掉落物品
-> 只标记对应实例域 dirty
-> 对应 worker 只写对应分域表
-> 服务重启后从分域表恢复
-> 旧 persistent_documents 地图 JSON 不存在 runtime 读写路径
-> 空地图 tick 不触发地图刷盘
```

## 2026-04-28 补充专项：高频域行级增量落盘

### 结论

当前方向已经从“全地图整包 JSON”推进到“按 dirty instance + dirty domain 刷盘”，但高频 domain 内部仍有明显写放大。硬切版本不再保留“整域 replace 兜底上线”的余地，也不把所有域都复杂化成事件溯源；正确口径是：

```text
外层：dirty instance + dirty domain 限流
内层：高频多行域改成 dirty key + tombstone 的 row-level delta
低频小域：继续整行/整域 upsert
强一致资产：继续事务内直接写 + audit/outbox
旧兼容：只允许离线转换，不能进入 runtime
```

### 域级目标矩阵

| domain | 当前写法 | 目标写法 | 是否第一批 |
| --- | --- | --- | --- |
| `monster_runtime` | 按实例构造当前高价值怪物集合，`incoming` 覆盖并删除未出现行 | `dirtyMonsterRuntimeIds + deletedMonsterRuntimeIds`，只 upsert/delete 变化怪物 | 是 |
| `tile_damage` | `DELETE WHERE instance_id` 后插入当前损坏集合 | `dirtyTileDamageIndices + deletedTileDamageIndices`，只 upsert 受损格；恢复基线 delete 单格 | 是 |
| `tile_resource` | 名为 diff，但实际 `DELETE WHERE instance_id` 后插入当前资源集合 | `dirtyTileResourceKeys + deletedTileResourceKeys`，只 upsert 变化资源点；回模板基线 delete 单点 | 是 |
| `ground_item` | `DELETE WHERE instance_id` 后插入当前地面物品 | `dirtyGroundItemIds + deletedGroundItemIds`，按物品 ID insert/update/delete | 是 |
| `container_state` | 删除本实例全部 state/entry/timer 后重插 | 先改成 `dirtyContainerIds` 容器级 replace；再拆 entry/timer delta | 第二批 |
| `overlay` | 删除本实例全部 overlay chunk 后重插 | `dirtyOverlayChunkKeys + deletedOverlayChunkKeys`，按 `(patchKind, chunkKey)` upsert/delete | 第二批 |
| `temporary_tile` | 当前整域 replace | 按 `tileIndex` upsert/delete | 第二批 |
| `time` | 低频 checkpoint | 保持低频整行 upsert | 不改 |
| `instance_checkpoint` | 冷字段 fallback | 保持低频整包 JSON | 不改 |
| `instance_formation_state` | 阵法服务直接写结构化表 | 后续纳入 ledger domain；单阵法行 upsert/delete | 第三批 |

### 状态分类规则

必须按状态性质选择写法：

| 状态性质 | 写法 |
| --- | --- |
| 单行小状态 | 整行 upsert，例如 checkpoint、单个阵法行、单个容器 timer |
| 低频小集合 | 可整域 replace，但必须证明数量上界和触发频率 |
| 高频多行集合 | dirty key delta，不允许继续本实例整域 delete+insert |
| 可回模板基线的状态 | 用 tombstone/delete 代表“删除专表覆盖，恢复模板默认” |
| 资产/邮件/市场交易 | 事务内直接写，进入 durable operation / audit / outbox，不走地图最终一致 flush |

### 运行时 dirty key 设计

`MapInstanceRuntime` 需要在现有 `dirtyDomains` 外增加 domain 内 dirty key 集合。第一阶段只做实例域，不碰玩家域。

建议结构：

```ts
type InstanceDirtyKeyDomain =
  | 'tile_resource'
  | 'tile_damage'
  | 'ground_item'
  | 'container_state'
  | 'overlay'
  | 'monster_runtime'
  | 'temporary_tile';

interface InstanceDomainDirtyKeys {
  domain: InstanceDirtyKeyDomain;
  upsertKeys: readonly string[];
  deleteKeys: readonly string[];
  version: number;
  dirtySinceAt: number;
}
```

第一批最小实现：

- `monster_runtime`
  - `dirtyMonsterRuntimeIds: Set<string>`
  - `deletedMonsterRuntimeIds: Set<string>`
  - 怪物 HP、位置、死亡、复生、仇恨、buff 或贡献状态变化时标记对应 runtimeId。
  - 普通低价值怪物仍不落库；从高价值降级或被移除时写 delete tombstone。
- `tile_damage`
  - `dirtyTileDamageIndices: Set<number>`
  - `deletedTileDamageIndices: Set<number>`
  - 受损、摧毁、复生剩余策略变化、复生完成时只标记对应 tile。
  - 回到模板满血未摧毁状态时进入 delete set。
- `tile_resource`
  - `dirtyTileResourceKeys: Set<`${resourceKey}:${tileIndex}`>`
  - `deletedTileResourceKeys: Set<`${resourceKey}:${tileIndex}`>`
  - 资源变为模板基线或 0 时删除对应覆盖行。
- `ground_item`
  - `dirtyGroundItemIds: Set<string>`
  - `deletedGroundItemIds: Set<string>`
  - 地面物品必须有稳定 ID；禁止继续用 `tileIndex:index` 作为长期真源 ID。

成功 flush 后只能清本次已提交的 key。失败时保留 key，等待下一轮 worker 重试。

### Persistence Service API 目标

在 `InstanceDomainPersistenceService` 中新增行级接口。旧 replace 接口不再作为 worker 兜底；硬切完成时要么删除，要么移动到离线迁移/repair 工具专用模块，runtime 代码不得调用。

第一批 API：

```ts
upsertMonsterRuntimeStateRows(instanceId, rows)
deleteMonsterRuntimeStateRows(instanceId, monsterRuntimeIds)

upsertTileDamageRows(instanceId, rows)
deleteTileDamageRows(instanceId, tileIndices)

upsertTileResourceRows(instanceId, rows)
deleteTileResourceRows(instanceId, keys)

upsertGroundItemRows(instanceId, rows)
deleteGroundItemRows(instanceId, groundItemIds)
```

SQL 形态：

- upsert 使用 `jsonb_to_recordset($2::jsonb)` 或 `unnest` 做批量写入。
- delete 使用 `ANY($2::...)` 批量删除。
- 不允许在 row-level delta worker 内先 `DELETE WHERE instance_id = $1`。
- 每个 API 必须在事务内持有 `acquireInstanceDomainLock(client, instanceId)`，避免与旧 replace / 迁移 / GM 恢复互相踩踏。

### 调度收口目标

worker 成为高频域主路径：

```text
runtime marks domain + dirty keys
-> FlushLedgerService.upsertInstanceFlushLedger(instanceId, domain, latestVersion)
-> FlushWakeupService.signalInstanceFlush(instanceId)
-> Instance*FlushWorker claim(domain)
-> export only dirty keys
-> row-level upsert/delete
-> mark ledger flushed
-> clear only flushed keys
```

`MapPersistenceFlushService` 只保留：

- `time` checkpoint 低频兜底。
- shutdown 强刷。
- worker 长时间未 claim 的诊断/补救。

禁止继续把 `monster_runtime` 这类高频域当作普通 5 秒周期的一部分常规刷。

## 商业级持久化原则

### 热状态直接结构化

这些状态必须直接建列或拆子表，不用整包 JSON 作为主真源：

- 地块资源：灵气、矿物、特殊地块资源。
- 地块损坏：HP、最大 HP、是否摧毁、复生剩余或复生 deadline。
- 地面物品：位置、物品 ID、数量、过期时间、实例 ID。
- 药草/容器：容器 ID、刷新状态、库存、activeSearch。
- 高价值怪物状态：位置、血量、复生、仇恨核心字段。
- 动态 overlay：宗门入口、动态传送点、动态 NPC/容器、临时规则。
- 恢复水位：每域版本、刷盘版本、恢复 checkpoint。

### JSONB 允许但限定用途

允许保留 JSONB 的场景：

- 物品随机词条、装备原始 payload 等低频复杂结构。
- 容器或事件的扩展字段，但主键、索引、倒计时、状态必须直接建列。
- 低频 `instance_checkpoint` 冷启动加速。
- GM 备份、审计报告、一次性迁移输入。
- 离线转换脚本读取的旧地图 dump 文件。旧 dump 不能作为 runtime 表或 runtime service 输入。

禁止场景：

- 每息变化写整包 JSON。
- 高频 dirty flush 里 `JSON.stringify()` 整张地图。
- 把可查询、可并发校验、可局部更新的字段只藏在 JSONB。

## 目标数据模型

### 现有表继续使用

| 表 | 目标职责 | 调整 |
|---|---|---|
| `instance_tile_resource_state` | 地块资源 diff 真源 | 保留 `(instance_id, resource_key, tile_index)` 主键；写入 0 或回到模板基线时要删除对应行 |
| `instance_ground_item` | 地面物品真源 | 保留 `item_instance_payload`，但 `item_id / count / quality / bind` 等热字段逐步拆列 |
| `instance_container_state` | 容器状态入口 | 只保留容器低频元数据，条目和计时已拆到专表 |
| `instance_container_entry` | 容器条目 | 按 `instance_id + container_id + entry_index` 存储物品载荷、创建 tick、可见性 |
| `instance_container_timer` | 容器计时 | 按 `instance_id + container_id` 存储生成 tick、刷新 tick、activeSearch |
| `instance_monster_runtime_state` | 高价值怪物运行态 | 只存需要跨重启保留的怪物，不存普通模板怪 |
| `instance_event_state` | 实例事件状态 | 低频，可保留 payload，但事件 ID、kind、key、resolved_at 直接建列 |
| `instance_overlay_chunk` | 动态 overlay | 保留 chunk 级 payload，但 patch_kind/chunk_key/version 直接建列 |
| `instance_formation_state` | 阵法状态 | 一阵一行，灵力余量、开关、阵眼、生效中心、阵盘和投入量均结构化保存 |
| `instance_checkpoint` | 冷启动 checkpoint | 低频生成，不参与普通热状态每轮刷盘 |
| `instance_recovery_watermark` | 恢复水位 | 记录各域版本、水位、最后成功刷盘时间 |

## 运行时 dirty 设计

### 目标接口

`MapInstanceRuntime` 需要从“实例是否脏”升级为“哪些域脏、版本是多少、能导出哪些变更”。

建议接口：

```ts
type InstanceDirtyDomain =
  | 'tile_resource'
  | 'tile_damage'
  | 'ground_item'
  | 'container_state'
  | 'container_entry'
  | 'overlay'
  | 'monster_runtime'
  | 'checkpoint';

interface InstanceDomainDirtyState {
  domain: InstanceDirtyDomain;
  version: number;
  dirtySinceAt: number;
}
```

需要补齐：

- [ ] `MapInstanceRuntime.getDirtyDomainStates(): InstanceDomainDirtyState[]`
- [ ] `MapInstanceRuntime.markDomainPersisted(domain, version)`
- [ ] `MapInstanceRuntime.buildTileResourceDiffPayload()`
- [ ] `MapInstanceRuntime.buildTileDamagePayload()`
- [ ] `MapInstanceRuntime.buildGroundItemPayload()`
- [ ] `WorldRuntimeLootContainerService.buildContainerDomainPayload(instanceId)`
- [ ] `WorldRuntimeFormationService` 如需挂到实例域，提供独立 formation 域 payload；否则保持自己的持久化 scope，但不得污染地图整包快照

### 关键约束

- 普通 `tick` 不属于 dirty domain。
- 怪物普通游荡不默认持久化；只有高价值怪物、战斗中状态、死亡复生状态进入 `monster_runtime`。
- 药草倒计时 UI 可以每息投影，但倒计时减少本身不标脏；只有库存变化、刷新生成、采集开始/完成、计时策略切换时标脏。
- 地块复生倒计时减少可以不每息刷盘；摧毁、复生完成、被固脉冻结、复生被单位阻挡后更新重试 deadline 时标脏。
