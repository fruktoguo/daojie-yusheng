# 地图实例持久化分域收口计划

> 本文用于收口 next 主线地图实例持久化：从“整包 JSONB 快照兜底”推进到“热状态结构化真源 + 冷 checkpoint + 旧 JSON 迁移兼容”。
>
> 主工作区固定为 `packages/*`。`legacy/*` 仅可作为旧行为审计输入，不作为新实现落点。

---

## 目标结论

商业级地图持久化不应该把每息变化、地块资源、地块损坏、药草容器、地面物品、动态入口和冷启动 checkpoint 混在一个 JSON 里频繁覆盖。

最终口径：

- 高频热状态写结构化表，按行、按域、按版本局部 upsert。
- JSONB 只保留在低频扩展 payload、复杂物品 raw payload、冷 checkpoint、迁移兼容和运营备份里。
- `MapPersistenceService.saveMapSnapshot()` 不再作为普通地图刷盘出口。
- `MapPersistenceFlushService` 降级为低频 checkpoint / shutdown / GM 手动兜底，不能参与每轮普通 dirty flush。
- 地图 tick 只作为运行时节拍，不因每息推进而标脏；需要跨重启保留的计时状态要用明确的 deadline / remaining / checkpoint 策略。
- 所有“下次还在”的地图状态必须以 PostgreSQL 结构化表或明确 checkpoint 为真源，不能只存在运行时、Redis 或前端缓存。

第一版验收闭环：

```text
破坏地块 / 改变灵气 / 采集药草 / 掉落物品
-> 只标记对应实例域 dirty
-> 对应 worker 只写对应分域表
-> 服务重启后从分域表恢复
-> 旧 persistent_documents 地图 JSON 不参与普通恢复
-> 空地图 tick 不触发地图刷盘
```

## 商业级持久化原则

### 热状态直接结构化

这些状态必须直接建列或拆子表，不用整包 JSON 作为主真源：

- 地块资源：灵气、矿物、特殊地块资源。
- 地块损坏：HP、最大 HP、是否摧毁、复生剩余或复生 deadline。
- 地面物品：位置、物品 ID、数量、过期时间、实例 ID。
- 药草/容器：容器 ID、刷新状态、库存、activeSearch。
- 高价值怪物状态：位置、血量、复生、仇恨核心字段。
- 动态 overlay：宗门入口、动态传送点、动态 NPC/容器、临时规则。
- 恢复水位：每域版本、刷盘版本、恢复 checkpoint。

### JSONB 允许但限定用途

允许保留 JSONB 的场景：

- 物品随机词条、装备原始 payload 等低频复杂结构。
- 容器或事件的扩展字段，但主键、索引、倒计时、状态必须直接建列。
- 低频 `instance_checkpoint` 冷启动加速。
- `persistent_documents` 旧地图快照迁移兼容。
- GM 备份、审计报告、一次性迁移输入。

禁止场景：

- 每息变化写整包 JSON。
- 高频 dirty flush 里 `JSON.stringify()` 整张地图。
- 把可查询、可并发校验、可局部更新的字段只藏在 JSONB。

## 目标数据模型

### 现有表继续使用

| 表 | 目标职责 | 调整 |
|---|---|---|
| `instance_tile_resource_state` | 地块资源 diff 真源 | 保留 `(instance_id, resource_key, tile_index)` 主键；写入 0 或回到模板基线时要删除对应行 |
| `instance_ground_item` | 地面物品真源 | 保留 `item_instance_payload`，但 `item_id / count / quality / bind` 等热字段逐步拆列 |
| `instance_container_state` | 容器状态入口 | 只保留容器低频元数据，条目和计时已拆到专表 |
| `instance_container_entry` | 容器条目 | 按 `instance_id + container_id + entry_index` 存储物品载荷、创建 tick、可见性 |
| `instance_container_timer` | 容器计时 | 按 `instance_id + container_id` 存储生成 tick、刷新 tick、activeSearch |
| `instance_monster_runtime_state` | 高价值怪物运行态 | 只存需要跨重启保留的怪物，不存普通模板怪 |
| `instance_event_state` | 实例事件状态 | 低频，可保留 payload，但事件 ID、kind、key、resolved_at 直接建列 |
| `instance_overlay_chunk` | 动态 overlay | 保留 chunk 级 payload，但 patch_kind/chunk_key/version 直接建列 |
| `instance_formation_state` | 阵法状态 | 一阵一行，灵力余量、开关、阵眼、生效中心、阵盘和投入量均结构化保存 |
| `instance_checkpoint` | 冷启动 checkpoint | 低频生成，不参与普通热状态每轮刷盘 |
| `instance_recovery_watermark` | 恢复水位 | 记录各域版本、水位、最后成功刷盘时间 |

### 必须新增或补强的表

#### `instance_tile_damage_state`

用于替代整包快照里的 `tileDamageEntries`。

| 字段 | 说明 |
|---|---|
| `instance_id` | 地图实例 ID |
| `tile_index` | 地块索引 |
| `hp` | 当前 HP |
| `max_hp` | 最大 HP |
| `destroyed` | 是否已摧毁 |
| `respawn_left_ticks` | 暂保留 tick 口径，兼容现有逻辑 |
| `respawn_at_ms` | 后续用于 wall-clock/offline progress 口径 |
| `modified_at` | 最近变更时间 |
| `version` | 单地块状态版本 |
| `updated_at` | 落库时间 |

主键：

```text
(instance_id, tile_index)
```

#### `instance_container_entry`

用于从 `instance_container_state.state_payload.entries` 中拆出药草/容器库存。

| 字段 | 说明 |
|---|---|
| `instance_id` | 地图实例 ID |
| `container_id` | 容器 ID |
| `entry_index` | 容器内条目顺序 |
| `item_payload` | 物品复杂 payload |
| `created_tick` | 条目创建 tick |
| `visible` | 是否可见 |
| `updated_at` | 更新时间 |

主键：`(instance_id, container_id, entry_index)`。

#### `instance_container_timer`

用于承接药草刷新、翻找、采集等计时，不再靠每息写地图 tick。

| 字段 | 说明 |
|---|---|
| `instance_id` | 地图实例 ID |
| `container_id` | 容器 ID |
| `generated_at_tick` | 容器生成 tick |
| `refresh_at_tick` | 下次刷新 tick |
| `active_search_payload` | 当前翻找/采集计时上下文 |
| `updated_at` | 更新时间 |

主键：`(instance_id, container_id)`。

药草刷新建议采用离线推进式；战斗、采集中动作建议采用暂停式。

#### `instance_domain_revision`

可合并进 `instance_recovery_watermark`，也可单独建表。目标是让每个域有独立版本，不再使用一个 `persistentRevision` 清空所有域。

| 字段 | 说明 |
|---|---|
| `instance_id` | 地图实例 ID |
| `domain` | `tile_resource / tile_damage / ground_item / container_state / container_entry / overlay / monster_runtime / checkpoint` |
| `latest_version` | 运行时最新版本 |
| `flushed_version` | 已刷盘版本 |
| `dirty_since_at` | 变脏时间 |
| `updated_at` | 更新时间 |

## 运行时 dirty 设计

### 目标接口

`MapInstanceRuntime` 需要从“实例是否脏”升级为“哪些域脏、版本是多少、能导出哪些变更”。

建议接口：

```ts
type InstanceDirtyDomain =
  | 'tile_resource'
  | 'tile_damage'
  | 'ground_item'
  | 'container_state'
  | 'container_entry'
  | 'overlay'
  | 'monster_runtime'
  | 'checkpoint';

interface InstanceDomainDirtyState {
  domain: InstanceDirtyDomain;
  version: number;
  dirtySinceAt: number;
}
```

需要补齐：

- [ ] `MapInstanceRuntime.getDirtyDomainStates(): InstanceDomainDirtyState[]`
- [ ] `MapInstanceRuntime.markDomainPersisted(domain, version)`
- [ ] `MapInstanceRuntime.buildTileResourceDiffPayload()`
- [ ] `MapInstanceRuntime.buildTileDamagePayload()`
- [ ] `MapInstanceRuntime.buildGroundItemPayload()`
- [ ] `WorldRuntimeLootContainerService.buildContainerDomainPayload(instanceId)`
- [ ] `WorldRuntimeFormationService` 如需挂到实例域，提供独立 formation 域 payload；否则保持自己的持久化 scope，但不得污染地图整包快照

### 关键约束

- 普通 `tick` 不属于 dirty domain。
- 怪物普通游荡不默认持久化；只有高价值怪物、战斗中状态、死亡复生状态进入 `monster_runtime`。
- 药草倒计时 UI 可以每息投影，但倒计时减少本身不标脏；只有库存变化、刷新生成、采集开始/完成、计时策略切换时标脏。
- 地块复生倒计时减少可以不每息刷盘；摧毁、复生完成、被固脉冻结、复生被单位阻挡后更新重试 deadline 时标脏。

## 计时语义收口

### 地图 tick

- [x] 纯 tick 不再标记地图持久化 dirty。
- [ ] `snapshot.tick` 不再作为普通热刷盘字段。
- [ ] checkpoint 可低频保存 `tick`，仅用于冷启动近似恢复或诊断。

### 药草刷新

当前 `refreshAtTick` 需要改成明确语义：

- [ ] 第一阶段保留 `refreshAtTick`，但只在刷新状态创建/变化时写入。
- [ ] 第二阶段增加 `deadlineAtMs`，药草刷新按 wall-clock 离线推进。
- [ ] UI 倒计时从当前 runtime 派生，不因每息减少而写库。
- [ ] 重启后药草是否补刷新必须由 `deadlineAtMs` 决定，不靠地图 tick 恢复精度。

### 地块复生

- [ ] 被摧毁时写 `destroyed=true + respawn_left_ticks + respawn_at_ms`。
- [ ] 固脉阵或宗门内置固脉冻结时不递减复生状态，也不每息写库。
- [ ] 复生点有单位阻挡时，更新一次重试 deadline，不每息写库。
- [ ] 真正复生完成时删除 `instance_tile_damage_state` 行。

### 阵法和护宗大阵

- [x] 阵法自己的灵力消耗属于阵法域，不属于地图整包；当前已落 `instance_formation_state.remaining_aura_budget`。
- [x] 阵法视觉范围不需要每息持久化；恢复时由阵法模板、中心点和半径重新派生。
- [ ] 阵眼、边界、护宗入口这些可交互/阻挡实体必须有 overlay 或 formation 真源。
