# 房间识别与风水直观化重做设计

> 本文用于设计当前生产主线的房间判断、房间类型自动判断和风水判断重做方案。目标是替换“玩家手动指定房间用途 + 隐藏规则加减分”的不直观模型，改成服务端权威、配置驱动、玩家可反推、可验证、可长期扩展的房间与风水系统。

## 设计边界

本方案只使用当前项目已经存在或已有明确承接点的数据：

- 房间拓扑：封闭、面积、边界、门、窗、屋顶覆盖、半室外。
- 建筑配置：五行向量、trait、舒适、稳定、聚气、泄气、煞气产生、煞气抵消、耐久权重。
- 地图环境：房间格子的灵气 `auraByTile` / `qiRaw`。
- 损坏状态：建筑或地块损坏产生完整度惩罚。
- 房间用途枚举：`generic`、`courtyard`、`meditation`、`alchemy`、`artifact`、`storage`、`bedroom`、`sect_hall`、`formation_core`。
- 低频协议：房间 summary、风水 overlay、风水 detail。
- 持久化分域：building、room、fengshui snapshot 可保存、回读、恢复和重算。

本方案当前不引入这些不存在或暂未成体系的系统：

- 怪物巢、尸体、污秽、道路冲煞。
- 山水形势、罗盘方位、时辰季节、玩家命格。
- 门派总风水、跨房间风水网络、全局风水收益。
- 每 tick 风水仿真。
- 客户端权威房间判断或客户端权威风水判断。

## 核心原则

- 房间是否成立由服务端拓扑判断，不由玩家命名决定。
- 房间类型由服务端根据房间内建筑、trait 和空间特征自动推断，不允许玩家直接指定。
- 风水计算只读取 `RoomInstance + RoomAggregate + 编译后配置`，不遍历临时 UI 状态，不执行任意脚本。
- 玩家改变房间类型的方式是改变房间内容，例如放丹炉、蒲团、床、储物架，而不是点下拉框。
- 风水总分必须能拆成可理解分项，并给出具体扣分原因和改造建议。
- 高频 `world delta` 不携带完整房间和风水详情，详情只走按需查询。

## 当前问题

当前实现中，房间检测是自动的，但用途默认是 `generic`，并存在 `RoomSetRole` 让客户端手动设置用途。这会产生三个问题：

- 空房间可以被手动设成丹房、静室等用途，造成玩法解释和收益来源不可信。
- 风水规则依赖 `room.role`，但 `room.role` 不是由房间内容推导出来的。
- 玩家看不懂为什么一个房间是某类型，也看不懂应该怎么改。

当前风水评分是 `500 + 命中规则加减分 - 完整度惩罚`。规则本身可以配置，但玩家视角不直观：

- `qiLeak` 已有配置承接，但当前实际公式没有形成清晰扣分。
- `shaRaw` 已有聚合，但当前规则主要只有影壁加分，缺少明确煞气暴露扣分。
- 用途、五行、聚气、泄气、舒适、损坏之间没有统一解释模型。

## 总体流程

```text
建筑/地块变化
-> 服务端重算受影响房间拓扑
-> 生成 RoomInstance
-> 聚合 RoomAggregate
-> 自动推断 RoomRole
-> 计算 FengShuiSnapshot
-> 持久化 room/fengshui snapshot
-> 按需同步 overlay/detail
```

房间类型推断发生在风水计算之前，因为风水用途五行和用途设施要求依赖房间类型。

## 第一版简化落地方案

第一版目标不是一次做完完整风水系统，而是先解决两个最影响体验的问题：

- 房间类型不能靠玩家手动指定。
- 风水分数要比当前隐藏规则更直观。

第一版只做：

- 自动判断房间类型。
- 简易风水总分。
- 简易风水原因展示。
- 保持现有房间检测、overlay、detail、持久化分域基本不变。

第一版不做：

- 不做完整 `RoomShellKind`。
- 不做复杂 `anchor` 节点系统。
- 不做复杂门对门、门对窗直泄路径。
- 不做家具朝向、寻路可达性、空间拥挤算法。
- 不做门派总风水。
- 不新增当前游戏没有的环境系统。
- 不重做高频协议。

### 第一版输入

直接复用现有数据：

```text
RoomInstance:
area / enclosed / doorCount / windowCount / roofCoverageRatio

RoomAggregate:
elementVector / traitCounts / comfort / stability / qiRaw / shaRaw / integrityPenalty

Building fengShui config:
elementVector / traits / comfort / stability / qiAffinity / qiLeak / shaEmit / shaReduce
```

第一版只需要补一件事：让 `RoomAggregate` 明确记录 `qiLeak`，并让 `shaRaw` 代表 `max(0, shaEmit - shaReduce)` 后的煞气暴露。

### 第一版房间类型判断

第一版不引入复杂配置文件，先基于现有 trait 做用途证据分。后续再把权重抽到 `room-roles.json`。

```text
alchemy:
  facility.alchemy.heat_source +100

meditation:
  facility.meditation +90

bedroom:
  comfort.rest +90

storage:
  storage.shelf +70

courtyard:
  semi_outdoor.corridor +60
  roofCoverageRatio < 80 且 area >= 16 +30
```

判定规则：

```text
最高分 < 60 -> generic
最高分 - 第二名 < 30 -> generic
否则 -> 最高分对应房间类型
```

冲突示例：

```text
丹炉 +100，床 +90
差值 10，小于 30
结果：generic
原因：用途混杂，无法形成明确丹房或卧房
```

第一版不允许普通玩家手动改 `room.role`。如果需要保留 `RoomSetRole`，只允许 GM 调试使用，普通玩家请求直接拒绝或不暴露入口。

### 第一版简易风水公式

第一版保留 `0..1000` 和当前等级阈值：

```text
score = clamp(
  500
+ shellScore
+ roleScore
+ elementScore
+ qiScore
+ comfortScore
+ shaScore
- integrityPenalty,
  0,
  1000
)
```

第一版增加一个普通房限制：

```text
role = generic 且 score > 520 -> score = 520
```

原因：没有丹炉、蒲团、床、储物架等明确用途设施的房间，只能说明“房间壳子不错”，不能天然进入小吉或大吉。玩家必须通过摆放用途核心设施让房间自动识别为丹房、静室、卧房或仓库，才允许突破普通房上限。

### 第一版房间形制分

```text
enclosed = true  -> +80
enclosed = false -> -120
doorCount <= 0 且 enclosed = true -> -80
area 在 6..64 -> +40
roofCoverageRatio >= 80 -> +30
```

第一版不按不同房间类型设置不同面积区间，先统一 `6..64`，避免过早复杂化。

### 第一版房间类型匹配分

```text
role = generic -> 0
role = alchemy 且有 facility.alchemy.heat_source -> +60
role = meditation 且有 facility.meditation -> +50
role = bedroom 且有 comfort.rest -> +50
role = storage 且有 storage.shelf -> +45
role = courtyard 且有 semi_outdoor.corridor -> +30
```

因为房间类型已经由 trait 推断，第一版只奖励“明确成型”的用途，不额外惩罚缺失用途设施。

### 第一版五行分

先沿用当前用途五行映射：

```text
meditation -> water
alchemy -> fire
storage -> earth
bedroom -> wood
courtyard -> wood
sect_hall -> earth
artifact -> metal
generic -> neutral
```

计算房间主五行：

```text
primaryElement = elementVector 中数值最高的元素
```

加减分：

```text
functionElement = neutral -> 0
primaryElement 生 functionElement -> +45
primaryElement = functionElement -> +25
primaryElement 克 functionElement -> -60
functionElement 克 primaryElement -> -40
```

第一版不计算复杂五行比例，只看主五行，保证玩家容易理解。

### 第一版灵气与泄气分

```text
qiDensity = qiRaw / area

qiDensity >= 80 -> +40
qiDensity < 20 -> -30
qiLeak > 0 -> -min(80, qiLeak * 10)
qiAffinity > 0 -> +min(60, qiAffinity * 10)
```

如果当前聚合还没有单独保留 `qiAffinity` 和 `qiLeak`，第一版至少要把 `qiLeak` 从 `fengShuiContrib[3]` 接进聚合，并在 reason 里展示“泄气”。

### 第一版舒适稳定分

```text
comfort >= 12 -> +30
comfort <= -6 -> -30
stability >= 12 -> +20
stability <= 0 -> -20
```

第一版只用聚合数值，不做空间布局和家具朝向。

### 第一版煞气分

```text
shaExposure = max(0, shaEmit - shaReduce)

shaExposure = 0 且 shaReduce > 0 -> +20
shaExposure 1..5 -> -20
shaExposure 6..15 -> -50
shaExposure > 15 -> -90
有 sha.screen trait -> +20
```

第一版必须把 `shaEmit` 和 `shaReduce` 作为真实扣分/抵消来源，而不是只给影壁加分。

### 第一版完整度惩罚

沿用当前 `integrityPenalty`：

```text
score -= max(0, integrityPenalty)
```

展示原因：

```text
完整度 -30：房间结构存在损坏
```

第一版不要求定位到具体哪面墙，后续 detail 再补。

### 第一版原因展示

服务端继续输出 reason code，客户端本地化。

### 第一版验收

第一版只要求这些 smoke：

## 一、房间判断

### 输入

房间判断只读取拓扑和地块层信息：

- 可通行地块。
- 建筑边界。
- 门窗开口。
- 屋顶覆盖。
- 室内子图入口。
- 半室外连接标记。
- 地块或建筑是否已损坏到破坏拓扑。

### 房间成立条件

一个区域必须满足以下条件才成为正式房间：

- 存在可识别的边界或半室外边界。
- 面积达到全局最小值。
- 未超过单房间最大扫描上限，超出则进入 deferred 队列。
- 区域不是无边界的纯室外泛洪。
- 至少有一个有效入口，除非配置声明该类型允许无入口。

建议基础阈值：

| 项 | 默认值 | 说明 |
|---|---:|---|
| 最小面积 | 4 | 小于 4 格不形成正式房间 |
| 普通房最大面积 | 256 | 超过后通常视为大空间或庭院候选 |
| 单次检测最大格数 | 512 | 保护运行时成本 |
| 封闭屋顶阈值 | 80 | 屋顶覆盖达到 80 视为完整室内 |
| 半室外屋顶阈值 | 40..79 | 可进入庭院、回廊候选 |

### 房间状态

房间判断不只输出是否有房间，还应输出房间状态：

```ts
type RoomShellKind =
  | 'indoor'
  | 'semi_outdoor'
  | 'courtyard'
  | 'open'
  | 'invalid';
```

状态含义：

- `indoor`：边界完整，屋顶覆盖充足。
- `semi_outdoor`：边界较明确，但屋顶覆盖不足，适合回廊或庭院候选。
- `courtyard`：边界完整或半完整，大面积、低屋顶，适合庭院。
- `open`：连通外界，通常不作为正式室内风水房。
- `invalid`：面积过小、扫描溢出或拓扑不可判定。

### 房间 ID 稳定性

房间 ID 不能只依赖检测顺序长期稳定。建议后续演进为：

```text
roomId = instanceId + roomHash + stableOrdinal
```

`roomHash` 来自房间边界盒、入口位置、核心 cell 采样和拓扑 revision。短期可以继续使用现有 ID，但持久化恢复和局部重算应尽量依赖 `roomHash` 对齐旧 snapshot。

## 二、房间类型自动判断

### 基本规则

房间类型不允许玩家手动指定。服务端根据房间内的用途证据自动推断：

```text
房间类型 = 分数最高且置信度足够的候选类型
```

如果最高分不足或多个类型冲突，则回退为 `generic`。

```text
最高分 < 60 -> generic
最高分 - 第二名 < 30 -> generic
最高分 >= 60 且明显领先 -> 最高分对应 role
```

可配置为：

```json
{
  "minConfidence": 60,
  "minLead": 30,
  "mixedFallbackRole": "generic"
}
```

### 证据来源

证据分为核心证据和辅助证据。

核心证据能决定房间用途：

| 建筑或 trait | 候选类型 | 建议分 |
|---|---|---:|
| `facility.alchemy.heat_source` | `alchemy` | 100 |
| `facility.meditation` | `meditation` | 90 |
| `comfort.rest` | `bedroom` | 90 |
| `storage.shelf` | `storage` | 80 |
| `facility.artifact.core` | `artifact` | 100 |
| `facility.sect_hall.core` | `sect_hall` | 100 |
| `facility.formation_core` | `formation_core` | 100 |
| `semi_outdoor.corridor` | `courtyard` | 50 |

辅助证据只能增强判断，不能单独把房间定型：

| 指标 | 候选类型 | 建议分 |
|---|---|---:|
| 主五行为火 | `alchemy` | 15 |
| 主五行为水 | `meditation` | 15 |
| 主五行为木 | `bedroom` / `courtyard` | 10 |
| 主五行为土 | `storage` / `sect_hall` | 10 |
| 主五行为金 | `artifact` | 15 |
| 舒适度高 | `bedroom` / `meditation` | 10 |
| 稳定度高 | `storage` / `sect_hall` | 10 |
| 面积大于 64 | `sect_hall` / `courtyard` | 15 |
| 半室外 | `courtyard` | 30 |

### 配置结构

建筑配置增加用途提示，不按 `defId` 写死运行时逻辑：

```json
{
  "id": "alchemy_furnace",
  "fengShui": {
    "elementVector": { "fire": 20, "earth": 6 },
    "traits": ["facility.alchemy.heat_source"],
    "roleHints": [
      { "role": "alchemy", "score": 100, "reasonCode": "role.alchemy.heat_source" }
    ],
    "anchor": {
      "kind": "alchemy_core",
      "element": "fire",
      "weight": 1.5
    }
  }
}
```

房间用途配置声明用途五行、面积舒适区和必需证据：

```json
{
  "role": "alchemy",
  "label": "丹房",
  "functionElement": "fire",
  "areaComfort": { "min": 6, "max": 100 },
  "roofCoverage": { "min": 80 },
  "requiredTraits": ["facility.alchemy.heat_source"],
  "preferredAnchorKinds": ["alchemy_core"]
}
```

### 推断结果

`RoomInstance` 或风水 detail 应包含可解释的用途推断信息：

```ts
interface RoomRoleInference {
  role: RoomRole;
  confidence: number;
  secondRole?: RoomRole;
  secondConfidence?: number;
  reasons: Array<{
    role: RoomRole;
    score: number;
    reasonCode: string;
  }>;
}
```

玩家看到：

```text
识别为：丹房
依据：丹炉火源 +100，火行环境 +15，屋顶完整 +10
```

如果冲突：

```text
识别为：普通房间
原因：丹炉与床同时存在，丹房 100，卧房 90，用途冲突
建议：移除床，或拆分为两个房间
```

## 三、房间聚合

`RoomAggregate` 是房间类型判断和风水判断的共同输入。它应从现有聚合扩展为语义化结构。

```ts
interface RoomAggregate {
  roomId: string;
  shell: {
    area: number;
    perimeter: number;
    enclosed: boolean;
    shellKind: RoomShellKind;
    doorCount: number;
    windowCount: number;
    roofCoverage: number;
    externalOpeningCount: number;
    directLeakPaths: number;
  };
  elements: {
    vector: Int32Array;
    primary: FiveElement;
    balance: number;
  };
  traits: Map<number, number>;
  anchors: FengShuiAnchorAggregate[];
  qi: {
    raw: number;
    density: number;
    gather: number;
    leak: number;
    net: number;
  };
  sha: {
    emit: number;
    reduce: number;
    exposure: number;
  };
  comfort: number;
  stability: number;
  integrityPenalty: number;
  topologyRevision: number;
  aggregateRevision: number;
}
```

### 聚合规则

- 房间格子累计 `auraByTile` 到 `qi.raw`。
- 建筑五行累计到 `elements.vector`。
- 建筑 trait 计数累计到 `traits`。
- `qiAffinity` 累计到 `qi.gather`。
- `qiLeak` 累计到 `qi.leak`。
- `qi.net = qi.raw + qi.gather - qi.leak`。
- `shaEmit` 累计到 `sha.emit`。
- `shaReduce` 累计到 `sha.reduce`。
- `sha.exposure = max(0, sha.emit - sha.reduce)`。
- 建筑损坏和拓扑损坏累计到 `integrityPenalty`。

## 四、风水判断

### 总公式

保留 `0..1000` 总分：

```text
score = clamp(
  500
+ shellScore
+ qiFlowScore
+ anchorScore
+ roleFitScore
+ comfortScore
+ shaScore
- integrityPenalty,
  0,
  1000
)
```

分项含义：

| 分项 | 范围建议 | 含义 |
|---|---:|---|
| `shellScore` | -220..180 | 房间是否成形、封闭、面积、屋顶、门窗基础格局 |
| `qiFlowScore` | -160..140 | 聚气、泄气、灵气密度 |
| `anchorScore` | -180..180 | 核心风水建筑和周围五行关系 |
| `roleFitScore` | -140..140 | 自动识别类型和实际设施是否匹配 |
| `comfortScore` | -80..80 | 舒适、稳定、空间适宜度 |
| `shaScore` | -140..60 | 煞气暴露和屏蔽 |
| `integrityPenalty` | 0..240 | 损坏惩罚，直接扣总分 |

### 等级

```text
850..1000 洞天
700..849  大吉
550..699  小吉
350..549  平
200..349  小凶
80..199   凶
0..79     大凶
```


### 藏风聚气分

```text
qiDensity 在用途舒适区 +40

qiDensity 过高且无稳定承载 -30
qi.gather 每档加分，封顶 +60

外门数量超过用途允许值，每个 -30
直泄路径 directLeakPaths 每条 -40
```

这里必须把当前已有但没有清晰使用的 `qiLeak` 接入公式。

### 核心建筑五行分

核心建筑来自 `anchor` 配置。没有 `anchor` 的普通墙、门、地板只影响环境，不直接作为核心风水建筑。

单个 anchor 的五行亲和：

```text
anchorAffinity =
  sameElementWeight
+ generatesAnchorWeight * 2
- controlsAnchorWeight * 2
- anchorControlsWeight
```

简化解释：

- 同元素：稳定加分。
- 环境生核心建筑：强加分。
- 环境克核心建筑：强扣分。
- 核心建筑克环境：轻扣分，表示消耗外局。

映射：

```text
anchorScore = clamp(anchorAffinity * anchor.weight, -180, 180)
```

玩家显示：

```text
丹炉：吉
原因：木气生火，丹炉火势得助
```

### 用途匹配分

用途匹配取自动推断出的 `room.role`，不允许玩家手动指定。

```text
识别置信度足够 +20
存在该用途 requiredTraits +60
缺少 requiredTraits -80
主五行生用途五行 +50
主五行等于用途五行 +30
主五行克用途五行 -70
用途五行克主五行 -40
用途冲突 mixed -60
```

示例：

- 丹房有丹炉，火源匹配，加分。
- 丹房主五行被水压制，扣分。
- 房间同时有丹炉和床，且分数接近，判 `generic` 或 `mixed`，扣用途冲突。

### 舒适稳定分

当前阶段只用已有 `comfort` 和 `stability`，不引入复杂家具朝向或寻路判断。

```text
comfort >= 12 +30
comfort <= -6 -30
stability >= 12 +20
stability <= 0 -20
用途需要稳定且 stability 不足 -30
```

### 煞气分

必须把当前已有 `shaEmit` 和 `shaReduce` 纳入公式。

```text
sha.exposure = max(0, sha.emit - sha.reduce)
sha.exposure == 0 且 sha.reduce > 0 +20
sha.exposure 1..5 -20
sha.exposure 6..15 -50
sha.exposure >15 -90
有 sha.screen trait +20
```

煞气解释必须具体：

```text
丹炉产生火煞 -20
影壁化煞 +20
当前煞气已被抵消
```

### 完整度惩罚

沿用现有损坏惩罚思路，但展示为独立分项：

```text
轻微损坏 -1..-30
明显损坏 -31..-90
关键边界损坏 -91..-160
严重破损 -161..-240
```

完整度不应藏在总分里，玩家必须看到：

```text
完整度 -45：北侧石墙受损
```

## 五、玩家可见解释

风水详情必须按分项展示，不直接暴露配置字段。

```text
风水：大吉 724

识别依据：丹炉火源 +100，火行环境 +15

房间形制 +150

丹炉属火，房间木气生火。

用途匹配 +60
丹房已有丹炉火源。

煞气 -20
丹炉产生少量火煞，暂无影壁完全化解。

完整度 0
房间结构完好。

减少多余开口以降低泄气。
```

建议只来自当前已有系统能做到的动作：

- 放置或移除已有建筑。
- 修复损坏建筑。
- 增加或减少门窗。
- 增加屋顶或边界。
- 放置影壁类 `sha.screen` 建筑。
- 放置用途核心设施，例如丹炉、蒲团、床、储物架。

## 六、协议设计

### 高频同步

高频 `world delta` 不带房间类型推断详情、风水分项、原因文本、建议文本。

### Overlay

风水 overlay 只在玩家打开风水视图时按需单播：

```ts
interface FengShuiOverlayCellView {
  x: number;
  y: number;
  roomId: string;
  score: number;
  grade: FengShuiGrade;
  revision: number;
}
```

### Detail

风水 detail 按需返回：

```ts
interface FengShuiDetailView {
  room: RoomSummaryView;
  roleInference: RoomRoleInference;
  fengShui: {
    score: number;
    grade: FengShuiGrade;
    dimensions: Array<{
      key: FengShuiDimensionKey;
      score: number;
      grade: FengShuiGrade;
      reasonCodes: string[];
    }>;
    anchors: Array<{
      buildingId: string;
      kind: string;
      element: FiveElement;
      score: number;
      grade: FengShuiGrade;
      reasonCodes: string[];
    }>;
    suggestions: Array<{
      code: string;
      priority: number;
    }>;
    revision: number;
  };
}
```

长文本不从服务端高频同步。服务端只发 `reasonCode` 和 `suggestionCode`，客户端用本地文案表展示。

## 七、持久化与恢复

建筑实例仍是真源。房间和风水 snapshot 是可恢复派生状态。

- `building_state` 保存建筑真源。
- `room_state` 保存房间检测快照、用途推断结果和 revision。
- `fengshui_state` 保存风水分项、总分、原因 code 和 revision。
- 冷启动时加载 snapshot 后校验 `roomHash`、`topologyRevision`、`contentRevision`。
- 校验不一致时，从建筑真源和地图拓扑重算。
- GM 修复接口应能重算房间、用途和风水，并输出修复数量。

## 八、配置文件规划

建议拆分为：

```text
packages/server/data/content/building-runtime/buildings.json
packages/server/data/content/building-runtime/room-roles.json
packages/server/data/content/building-runtime/fengshui-formula.json
packages/server/data/content/building-runtime/fengshui-reasons.json
```

职责：

- `buildings.json`：建筑五行、trait、anchor、聚气、泄气、煞气、舒适、稳定。
- `room-roles.json`：房间用途、用途五行、面积舒适区、屋顶要求、必需 trait、用途推断阈值。
- `fengshui-formula.json`：各分项权重、阈值、封顶、等级。
- `fengshui-reasons.json`：原因文案、建议文案、严重级别、UI 分组。

服务端启动期编译这些配置，运行时只读预解析结构。

## 九、实施步骤

### P1：文档和类型预备

- 明确 `RoomRoleInference`、`FengShuiDimension`、`FengShuiAnchorAggregate` 类型。
- 保留旧 `RoomSetRole`，但标记为待废弃或仅 GM 调试可用。
- 新增配置 schema 草案。

### P2：自动房间类型推断

- 在 `RoomAggregate` 后增加 `inferRoomRole`。
- 用现有 trait 先覆盖丹房、静室、卧房、仓库、庭院。
- 房间 summary/detail 展示自动推断结果和依据。
- 普通玩家不能手动改用途。

### P3：风水分项计算

- 将当前单规则加减分改成分项计算。
- 接入 `qiLeak`、`shaEmit`、`shaReduce`。
- 输出 `dimensions`、`anchors`、`suggestions`。
- 保持总分 `0..1000` 和现有等级兼容。

### P4：UI 解释重做

- 风水详情按分项展示。
- 房间类型展示“识别依据”。
- 建议只展示当前游戏能执行的改造动作。
- 浅色、深色、手机端都要可读。

### P5：废弃普通玩家手动设定

- 普通客户端移除房间用途手动设置入口。
- 协议层拒绝普通玩家 `RoomSetRole`。
- GM 可保留调试接口，但不能影响正式玩家规则，或必须标记为 GM override。

## 十、验证计划

### smoke

必须覆盖：

- 封闭房间能成立。
- 开放区域不成为正式室内房。
- 无门封闭房成立但扣分。
- 丹炉房自动识别为 `alchemy`。
- 蒲团房自动识别为 `meditation`。
- 床房自动识别为 `bedroom`。
- 储物架房自动识别为 `storage`。
- 丹炉和床冲突时回退 `generic` 或输出冲突。
- `qiLeak` 会降低藏风聚气分。
- `shaEmit` 会扣煞气分，`shaReduce` 能抵消。
- 损坏会扣完整度。

### bench

必须覆盖：

- 单房间家具变化只重算受影响房间。
- 建墙/拆墙只触发必要房间拓扑重算。
- 风水重算不扫描全实例所有建筑。
- 100 个房间规模下重算 p95 不超过现有目标。

### protocol audit

必须证明：

- 高频 `world delta` 不带风水详情。
- overlay 只发 `roomId/score/grade/revision/cells`。
- detail 按需单播，原因用 code，不发完整规则表。

### persistence

必须证明：

- 建筑为真源。
- room/fengshui snapshot 可落库、回读。
- 配置 revision 或拓扑 hash 不一致时能重算。
- smoke/proof 夹具自动清理。

### UI

必须检查：

- 浅色模式可读。
- 深色模式可读。
- 手机端弹层高度、滚动、按钮和地图 overlay 不互相遮挡。
- 风水 detail 更新不全量刷新地图和面板，不打断滚动。

## 十一、完成定义

- 普通玩家不能通过手动下拉框改变房间用途。
- 房间用途由服务端根据房间内容自动判断。
- 空房间不会被判成丹房、静室、卧房或仓库。
- 风水总分能拆成清晰分项。
- 每个主要扣分都有玩家可执行建议。
- 新增家具和结构只需要改配置，不需要改房间识别和风水算法。
- `qiLeak`、`shaEmit`、`shaReduce`、损坏惩罚都进入正式公式。
- 高频协议不膨胀。
- 持久化和恢复仍以建筑真源为准。
