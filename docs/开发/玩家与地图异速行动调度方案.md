# 玩家与地图异速行动调度方案

本文只描述目标方案，不代表当前主线已经支持玩家行动流速独立于地图流速。

当前主线是全局世界帧驱动、地图实例按 `tickSpeed` 累积逻辑 tick、玩家在所在实例 tick 中被推进。若要支持“玩家行动也可以和地图不一样快”，不能简单把 `advanceSinglePlayerTick()` 多跑几次，也不能让客户端自行提前移动或结算。正确方向是把“请求输入”“行动准备”“行动到期”“地图落地裁定”拆成服务端权威的调度系统。

## 1. 目标

- 支持同一地图内不同玩家拥有不同个人行动流速。
- 支持玩家行动流速与地图 tick 流速不同。
- 地图暂停、减速、加速时，能明确玩家行动是否跟随地图暂停或继续准备。
- 继续保证移动、碰撞、占位、战斗命中、采集消耗、传送、掉落、地图对象变化全部由服务端裁定。
- 继续保持 AOI 最小广播，不因为个人高速玩家存在而全图广播。
- 支持持久化回读，玩家重连、顶号、服务重启后不会丢失行动时钟和待执行动作。
- 支持 GM 调试：能查看和调整地图速度、玩家行动速度、玩家行动队列、冲突裁定结果。

## 2. 非目标

- 不把客户端变成移动合法性或战斗结算裁判。
- 不允许玩家个人 tick 直接改写地图占位、怪物、地块、容器或地面物品。
- 不做跨服分布式时间同步方案。多节点只沿用当前实例 lease 和玩家 session fencing。
- 不把高频包改成完整状态同步。
- 不把所有 buff、修炼、炼丹、采集、战斗一起硬塞进一个巨型 tick 函数。

## 3. 现状基线

当前相关主链如下：

- `packages/shared/src/constants/gameplay/core.ts`
  - `WORLD_TICK_INTERVAL_MS = 100`
  - `TICK_INTERVAL = 1000`
- `packages/server/src/runtime/tick/world-tick.service.ts`
  - 每个世界帧调用 `WorldRuntimeService.advanceFrame(WORLD_TICK_INTERVAL_MS, getMapTickSpeed)`
  - `tickInFlight` 防止异步帧重入
  - `advanceFrame` 完成后才 flush 同步和 GM 状态
- `packages/server/src/runtime/map/runtime-map-config.service.ts`
  - 当前已有地图级 `tickSpeed`、暂停、`timeConfig`
- `packages/server/src/runtime/world/world-runtime-instance-tick-orchestration.service.ts`
  - 按地图速度累计实例逻辑 tick
  - 实例 tick 中执行怪物、转移、玩家推进、craft job、容器搜索等
- `packages/server/src/runtime/player/player-runtime.service.ts`
  - `advanceTickForPlayerIds()` 每实例 tick 推进玩家寿元、buff、恢复、修炼等
- `packages/server/src/runtime/world/world-runtime-pending-command.service.ts`
  - 当前待执行命令按 `playerId -> command` 覆盖式保存
  - 不适合支持同一玩家多个未来到期动作、不同准备时长或同 tick 冲突排序

这个模型可以支持地图之间不同速，但不支持玩家行动独立不同速。

## 4. 核心模型

### 4.1 三层时间

目标架构拆成三层：

| 层 | 责任 | 是否能改地图状态 |
| --- | --- | --- |
| 世界帧 `frame` | 服务器调度节拍，默认 100ms | 否，只有驱动作用 |
| 地图逻辑 tick `mapTick` | 地图物理时间、怪物、占位、AOI、地块、容器、掉落 | 是 |
| 行动时钟 `actorClock` | 玩家或 NPC 的行动准备、冷却、个人速度累计 | 否，只能产出待裁定动作 |

地图状态只能在地图逻辑 tick 的受控流程中改变。玩家行动时钟只能把动作推进到“到期可裁定”，不能绕过地图 tick 直接写世界。

### 4.2 行动调度器

新增服务建议名：

- `WorldRuntimeActorClockService`
- `WorldRuntimeActionScheduleService`
- `WorldRuntimeActionResolutionService`

职责拆分：

- `ActorClockService`
  - 管理 actor 的行动速度、暂停策略、累计进度、个人行动序号。
  - actor V1 先只包含玩家，后续可扩展到怪物、召唤物、机关。
- `ActionScheduleService`
  - 把玩家输入、自动战斗、路径移动、持续采集、技能读条转换成 `ScheduledAction`。
  - 维护每个实例的到期动作队列。
- `ActionResolutionService`
  - 在地图 tick 中取出到期动作。
  - 统一做排序、冲突裁定、占位检查、资源校验、执行与广播 intent 生成。

### 4.3 数据结构草案

```ts
type RuntimeActorKind = 'player' | 'monster' | 'summon' | 'mechanism';

interface RuntimeActorClockState {
  actorKind: RuntimeActorKind;
  actorId: string;
  instanceId: string | null;
  actionSpeed: number;
  actionProgress: number;
  actorTick: number;
  paused: boolean;
  pausePolicy: ActorPausePolicy;
  updatedAtWorldTick: number;
}

type ActorPausePolicy =
  | 'follow_map'
  | 'continue_when_map_paused'
  | 'freeze_when_offline'
  | 'gm_frozen';

interface ScheduledAction {
  actionId: string;
  actorKind: RuntimeActorKind;
  actorId: string;
  instanceId: string;
  actionKind: ScheduledActionKind;
  requestSeq: number;
  actorReadyTick: number;
  earliestMapTick: number;
  deadlineMapTick?: number;
  priority: number;
  target: ScheduledActionTarget;
  payload: unknown;
  createdAtWorldTick: number;
  source: 'manual' | 'auto_combat' | 'path' | 'sustained_activity' | 'system' | 'gm';
  cancelPolicy: ScheduledActionCancelPolicy;
}

type ScheduledActionKind =
  | 'move'
  | 'portal'
  | 'basic_attack'
  | 'cast_skill'
  | 'start_gather'
  | 'tick_gather'
  | 'take_ground'
  | 'npc_interaction'
  | 'start_craft'
  | 'tick_craft'
  | 'toggle_setting'
  | 'use_item'
  | 'respawn';
```

字段含义：

- `actionSpeed` 是玩家行动流速，不等于地图 `tickSpeed`。
- `actionProgress` 是小数累计器，支持 `0.5x`、`1.25x`、`3x`。
- `actorTick` 是玩家个人行动 tick 计数，只用于个人动作准备和冷却排序。
- `earliestMapTick` 保证动作最终落地仍受地图 tick 控制。
- `requestSeq` 防止同一玩家同一 actorTick 内输入覆盖和乱序。
- `priority` 只解决同一地图 tick 内动作排序，不允许绕过合法性校验。

## 5. 行动生命周期

### 5.1 输入阶段

Socket 或 HTTP 入口只接收玩家意图：

1. 校验玩家 session、顶号、死亡、当前实例。
2. 生成 `PlayerActionIntent`。
3. 写入 actor 的待编排输入槽或输入队列。
4. 不直接移动、不直接攻击、不直接扣物品、不直接改地图。

当前覆盖式 `pendingCommands.set(playerId, command)` 需要升级为：

- 同一玩家手动输入可覆盖同类未开始动作，例如连续点击移动。
- 已进入读条或已到期等待裁定的动作，不允许被无规则覆盖，必须显式取消或被中断规则取消。
- 自动战斗派生动作和手动动作分 lane，手动优先级高于自动。

### 5.2 编排阶段

世界帧进入实例处理前，对玩家行动时钟做轻量推进：

```ts
effectiveDelta = frameDurationMs / 1000
actor.actionProgress += actionSpeed * effectiveDelta
readySteps = Math.floor(actor.actionProgress)
actor.actionProgress -= readySteps
```

每产生一个 `readyStep`：

1. 从该玩家输入 lane 选择一个意图。
2. 根据动作类型计算准备成本。
3. 如果动作需要多步准备，继续保留为未到期。
4. 如果动作到期，生成 `ScheduledAction`，放入当前实例的 `readyActions`。

不要在这个阶段执行世界写入。

### 5.3 地图 tick 裁定阶段

每个实例逻辑 tick 内，顺序建议改为：

1. lease 可写校验。
2. 地图 tick 前收集到期动作。
3. 怪物/NPC 行动时钟推进并生成到期动作。
4. 合并玩家、怪物、系统到期动作。
5. 按排序规则裁定动作。
6. 执行地图状态变更。
7. 推进玩家个人结算、持续 job、buff、恢复、修炼。
8. 生成 AOI delta、self delta、panel delta。
9. 标记持久化 dirty domain。

关键点：同一地图 tick 内，所有动作先收集再裁定，不能玩家 A 执行后立刻让玩家 B 的输入看到半帧状态并插队。

## 6. 排序与冲突裁定

### 6.1 排序规则

建议排序 key：

1. `earliestMapTick` 小的先。
2. `priority` 高的先。
3. `actorReadyTick` 小的先。
4. `actionKind` 固定顺序。
5. `actorId` 稳定字典序。
6. `requestSeq` 小的先。

固定 `actionKind` 顺序建议：

1. `respawn`
2. `portal`
3. `move`
4. `cast_skill`
5. `basic_attack`
6. `use_item`
7. `take_ground`
8. `start_gather`
9. `tick_gather`
10. `npc_interaction`
11. `start_craft`
12. `tick_craft`
13. `toggle_setting`

该顺序必须进入 shared 或 server 常量，禁止散落在多个服务里。

### 6.2 移动冲突

同一 tick 多个玩家进入同一目标格：

- 先按排序规则选胜者。
- 失败者动作失败，不做客户端补偿移动。
- 若失败者有路径自动导航，可在下一次 actor ready 时重新规划。
- 若两个玩家互换位置：
  - V1 默认不允许互换，按目标格占用失败处理。
  - 后续如需要互换，必须作为独立动作类型 `swap_move`，由服务端显式校验双方意图。

占位写入必须继续由 `MapInstanceRuntime` 或其拆分后的 occupancy owner 完成。

### 6.3 战斗冲突

攻击与移动同 tick 冲突时：

- 攻击目标位置以裁定时的当前状态为准。
- 技能若依赖射程和视线，必须在裁定时二次校验。
- 高速玩家不能因为个人行动到期更早而打到已经离开 AOI 或已死亡的目标。
- 技能读条到期只表示“可尝试释放”，不表示必定命中。

### 6.4 采集与容器冲突

多个玩家采同一容器同一物品：

- 到期动作按排序规则裁定。
- 容器库存扣减在地图 tick 内原子完成。
- 后续玩家若库存不足，动作失败或自动转向下一个可采项。
- 采集持续进度可以由玩家行动时钟加速，但最终扣库存和发物必须在地图 tick 内提交 durable 后返回。

### 6.5 地图暂停

地图暂停时分两种：

| 策略 | 行动时钟 | 地图写入 |
| --- | --- | --- |
| `follow_map` | 停止累计 | 不写 |
| `continue_when_map_paused` | 继续累计准备 | 不写，动作积压到地图恢复后裁定 |

V1 建议玩家默认 `follow_map`，GM 或特殊玩法才允许 `continue_when_map_paused`。否则地图暂停后玩家攒满大量动作，恢复时容易出现瞬间爆发和同步压力。

## 7. 玩家个人结算与行动结算的关系

玩家“行动流速”不应无条件等于“成长/寿元/恢复流速”。

建议拆成：

- `actionSpeed`：移动、攻击、交互、采集动作准备速度。
- `personalTimeScale`：寿元、buff 持续、恢复、修炼、炼丹、强化等个人持续结算速度。
- `cooldownScale`：技能冷却恢复速度，可默认跟随 `personalTimeScale`。

V1 可以提供统一配置：

```ts
interface PlayerTimeRuntimeState {
  actionSpeed: number;
  personalTimeScale: number;
  cooldownScale: number;
  pausePolicy: ActorPausePolicy;
  actionProgress: number;
  personalProgress: number;
  cooldownProgress: number;
  actorTick: number;
}
```

但内部必须拆字段，避免后续“加速行动但不加速寿元”“加速修炼但不加速移动”时重构。

## 8. 持久化设计

只要玩家流速、暂停策略、未完成读条、持续采集、技能冷却要求重连后仍在，就必须进数据库真源。

### 8.1 玩家时间状态

建议新增玩家域表或并入玩家设置/运行域：

`player_time_runtime_state`

| 字段 | 说明 |
| --- | --- |
| `player_id` | 主键 |
| `action_speed` | 行动流速 |
| `personal_time_scale` | 个人结算流速 |
| `cooldown_scale` | 冷却流速 |
| `pause_policy` | 暂停策略 |
| `action_progress` | 行动累计小数 |
| `personal_progress` | 个人结算累计小数 |
| `cooldown_progress` | 冷却累计小数 |
| `actor_tick` | 个人行动 tick |
| `updated_at` | 更新时间 |

如果只把它存在运行时内存，顶号、重启、迁移后会发生行动节奏回退。

### 8.2 待执行动作

待执行动作是否持久化按类型区分：

- 必须持久化：
  - 技能读条
  - 持续采集
  - 炼丹/强化/其他 active job
  - 传送读条
- 可以不持久化：
  - 单步移动意图
  - 自动战斗下一次普通攻击意图
  - 可由当前状态重新派生的导航下一步

建议新增：

`player_scheduled_action`

| 字段 | 说明 |
| --- | --- |
| `action_id` | 主键 |
| `player_id` | 玩家 |
| `instance_id` | 当前实例 |
| `action_kind` | 动作类型 |
| `status` | `pending / ready / resolving / cancelled / completed / failed` |
| `actor_ready_tick` | 个人行动到期 tick |
| `earliest_map_tick` | 最早可裁定地图 tick |
| `request_seq` | 输入序号 |
| `priority` | 裁定优先级 |
| `target_payload` | 目标 |
| `payload` | 动作差异字段 |
| `cancel_policy` | 取消策略 |
| `created_at` | 创建时间 |
| `updated_at` | 更新时间 |

玩家在线运行时可以只缓存近期动作；数据库是真源，恢复时按 `player_id + status` 回读必要动作。

### 8.3 分域 dirty

新增 dirty domain：

- `time_runtime`
- `scheduled_action`

落库要求：

- 调整玩家速度、暂停策略，标记 `time_runtime`。
- 创建、取消、完成持久化动作，标记 `scheduled_action`。
- 动作完成导致背包、钱包、职业、buff、progression 改动时，同时标记对应域，不能只标 `scheduled_action`。

## 9. 协议设计

### 9.1 高频同步原则

不要在 `WorldDelta` 中每帧广播玩家完整时间状态。

字段分层：

| 字段 | 层 |
| --- | --- |
| 自己的 `actionSpeed / personalTimeScale / cooldownScale` | `SelfDelta` 或 `ActionsUpdate` |
| 自己当前读条/准备进度 | `SelfDelta`，只在变化或阶段切换时发 |
| 其他玩家是否在读条/移动/释放技能 | AOI `WorldDelta`，只发表现所需最小字段 |
| GM 查看某玩家行动队列 | GM detail API |
| 地图 tickSpeed/timeConfig | GM map runtime payload，不进普通玩家高频包 |

### 9.2 共享类型

建议新增 shared 类型：

```ts
interface PlayerTimeRuntimeView {
  actionSpeed: number;
  personalTimeScale: number;
  cooldownScale: number;
  paused: boolean;
  actorTick: number;
}

interface PlayerActionCastView {
  actionId: string;
  actionKind: ScheduledActionKind;
  startedAtActorTick: number;
  readyAtActorTick: number;
  progressPercent: number;
  targetRef?: string;
}
```

普通客户端只需要知道自己的进度和 AOI 内其他人的表现态，不需要知道完整队列。

### 9.3 Protobuf/压缩

如果进入 protobuf tick 包：

- `actionSpeed` 这类低频字段只在变化时发。
- 进度字段用短名/数值，避免 JSON 字符串。
- 其他玩家表现只发 `id / kind / progress / target`，不发模板说明和技能长文本。

## 10. 客户端职责

客户端只做：

- 展示自己的行动速度、读条、冷却、失败提示。
- 用服务端下发的 `tickDurationMs` 或动作进度做动画插值。
- 展示 AOI 内其他人的读条或动作表现。
- 发起输入意图和取消意图。

客户端不能做：

- 本地提前改坐标。
- 本地判断占位成功。
- 本地判断攻击命中。
- 本地扣容器库存或背包。
- 本地把个人速度应用为真实游戏规则。

## 11. GM 与运营工具

GM 需要新增可观测项：

- 地图 `tickSpeed / paused / timeConfig`。
- 玩家 `actionSpeed / personalTimeScale / cooldownScale / pausePolicy`。
- 玩家当前 `actionProgress / personalProgress / actorTick`。
- 玩家待执行动作列表。
- 最近 N 次动作裁定结果。
- 同 tick 冲突统计：移动冲突、目标失效、资源不足、lease 失败。

GM 写入口：

- `PUT /gm/players/:playerId/time`
- `PUT /gm/players/:playerId/actions/:actionId/cancel`
- `GET /gm/players/:playerId/action-schedule`

这些入口只改玩家时间状态或取消意图，不直接改世界地图状态。

## 12. 与当前系统的具体改造点

### 12.1 `WorldRuntimeInstanceTickOrchestrationService`

当前逻辑按实例 tick 直接推进玩家。目标改造：

- 在实例 tick 前收集 actor ready actions。
- 在实例 tick 中统一裁定 actions。
- 玩家个人结算从“每地图 tick 推一次”改为“按 `personalTimeScale` 累计后推 N 次或按 delta 推进”。
- craft/gather 等持续行为不要散落在多个服务里重复扣 `remainingTicks`，统一接入个人结算或行动调度。

### 12.2 `WorldRuntimePendingCommandService`

当前 `Map<playerId, command>` 不够用。目标改造：

- 拆成输入 lane：
  - `manual`
  - `autoCombat`
  - `navigation`
  - `sustainedActivity`
  - `system`
- 支持同一玩家多个动作状态。
- 支持动作取消、中断、过期、失败原因。
- 支持查询和 GM 调试。

### 12.3 `PlayerRuntimeService`

目标改造：

- 增加 `PlayerTimeRuntimeState`。
- `advanceSinglePlayerTick()` 拆成：
  - `advancePlayerChronology()`
  - `advancePlayerBuffs()`
  - `advancePlayerVitals()`
  - `advancePlayerCultivation()`
  - `advancePlayerCooldowns()`
- 每个子推进函数接收逻辑步数或 delta，不再隐含“地图 tick 一次等于玩家推进一次”。

### 12.4 移动与导航

目标改造：

- `enqueueMove()` 只生成移动意图。
- `materializeNavigationCommands()` 只根据 actor ready 生成可裁定移动动作。
- `dispatchInstanceCommand()` 仍在地图 tick 内执行占位裁定。
- 路径自动导航遇到冲突时，下一次 actor ready 重算，不在当前 tick 无限重试。

### 12.5 自动战斗

目标改造：

- 自动战斗目标选择可以在 actor ready 时计算。
- 技能冷却用 `cooldownScale` 推进。
- 释放动作到期后，在地图 tick 中二次校验目标、距离、视线、蓝量、仇恨合法性。
- 技能失败可以在同一 actor ready 周期内按明确规则 fallback 到其他技能或普攻，但不能无限循环。

### 12.6 采集、炼丹、强化

目标改造：

- 持续作业的准备/进度跟 `personalTimeScale` 或专用 `activitySpeed`。
- 真正扣资源、发物、写 durable operation 仍在地图 tick 或对应玩家命令 await 链中完成。
- 已有 active job 的 `job_run_id + job_version` CAS 语义继续保留。

## 13. 边界规则

### 13.1 玩家跨地图

玩家从 A 地图迁移到 B 地图：

- 未落地的移动/战斗/采集动作默认取消。
- 可迁移的读条类动作必须显式声明 `portable=true`。
- `actorClock` 保留，但 `instanceId` 更新。
- `actionProgress` 是否保留由动作类型决定。

### 13.2 玩家离线

默认策略：

- 手动移动、战斗动作离线取消。
- active job 按现有离线策略决定是否继续。
- `freeze_when_offline` 玩家行动时钟停止。

不能默认离线继续行动，否则会变成挂机自动操作系统，需要单独设计。

### 13.3 顶号

顶号不应复制旧连接的输入队列：

- session fencing 后旧连接不能再追加 intent。
- 已持久化的读条/active job 保留。
- 未持久化的瞬时输入清理。

### 13.4 服务卡顿与补帧

若某次 `frameDurationMs` 很大：

- actor ready steps 必须有上限，例如每帧每 actor 最多 3 步。
- 超出部分保留进度，不一次性爆发。
- 地图 tick 同样不能因为卡顿无限追帧。

这和当前“不做卡顿后 catch-up replay”的口径一致。

## 14. 性能约束

- ready actions 按实例分桶，避免全服扫描。
- actor clock 只推进在线或实例内活跃 actor。
- `ScheduledAction` 使用结构化字段，不用字符串签名比较。
- hot path 不做 `JSON.stringify / JSON.parse`。
- 目标格、actorId、actionId 使用稳定结构，不在循环中拼大量临时字符串。
- 每实例每 tick 设置最大动作裁定预算，超出进入下一 tick，避免单个高速玩家拖垮地图。

## 15. 落地阶段

### 阶段 0：文档与 smoke 骨架

- 固化本文方案。
- 增加 focused smoke 入口，但先不改主链。
- 覆盖排序、冲突、暂停、跨图、顶号、恢复的纯函数测试。

### 阶段 1：只引入 actor clock，不改变行为

- 新增 `PlayerTimeRuntimeState`，默认速度全为 `1`。
- 当前地图 tick 仍每 tick 推玩家一次。
- smoke 证明默认行为与现状一致。

### 阶段 2：改造 pending command 为输入 lane

- 保持默认 `1x`，先替换覆盖式 `pendingCommands`。
- 自动战斗、导航、手动输入都进入 lane。
- smoke 证明当前移动/战斗/采集行为不变。

### 阶段 3：引入 action schedule 和裁定队列

- `move / basic_attack / cast_skill / take_ground` 先接入。
- 所有动作仍默认 `1x`。
- smoke 证明同 tick 冲突裁定稳定。

### 阶段 4：开启玩家行动异速

- 允许 GM 调整玩家 `actionSpeed`。
- 普通玩家入口暂不暴露 UI。
- 覆盖同图两个玩家 `0.5x / 2x` 的移动和战斗冲突。

### 阶段 5：个人结算异速

- 拆 `personalTimeScale / cooldownScale`。
- 接入寿元、buff、恢复、修炼、技能冷却。
- 再接采集、炼丹、强化。

### 阶段 6：持久化闭环

- 新增玩家时间状态表。
- 持久化必要 `ScheduledAction`。
- 补恢复、重连、顶号、服务重启验证。

### 阶段 7：协议和 GM 完整化

- 普通客户端只收最小 self/action view。
- GM 增加 action schedule 查看与取消。
- 协议审计确认高频包不带静态和完整队列。

## 16. 验证计划

最小 smoke：

- `world-runtime-actor-clock-smoke`
  - `0.5x / 1x / 2x` 累计器正确。
  - 卡顿帧有步数上限。
- `world-runtime-action-schedule-smoke`
  - 同玩家多 lane 输入排序正确。
  - 手动输入覆盖自动战斗。
- `world-runtime-action-resolution-smoke`
  - 同 tick 移动抢格稳定。
  - 互换位置默认失败。
  - 目标死亡或离开射程后技能失败。
- `world-runtime-player-time-persistence-smoke`
  - 玩家时间状态落库回读。
  - 持久化读条动作重启后恢复。
  - 顶号后旧连接不能继续追加 intent。
- `world-sync-action-time-smoke`
  - 普通玩家只收到自己的速度/进度。
  - AOI 只收到表现所需字段。
  - GM detail 能看到完整队列。

门禁：

- `pnpm build`
- `pnpm verify:release:doctor`
- 相关 `smoke:world-runtime-*`
- 涉及 DB 后补 `pnpm verify:release:with-db`
- 协议字段稳定后补协议审计

所有会创建玩家、实例、动作或持久化 active job 的 smoke，都必须带成功、失败、中断后的清理路径。

## 17. 主要风险

- 高速玩家导致单实例动作爆发，需要预算和限速。
- 地图暂停但玩家继续准备会产生恢复瞬间积压，V1 默认不开放给普通玩家。
- 自动战斗 fallback 如果没有上限，可能在同一 tick 内循环尝试过多技能。
- 采集/掉落/任务奖励如果没有 commit-gated durable，可能出现运行态成功但落库失败。
- 客户端动画如果自行预测坐标，会和服务端裁定冲突。
- 协议若把完整动作队列塞进高频包，会放大 tick 同步压力。

## 18. 推荐结论

可以支持“玩家行动也和地图不同速”，但它不是简单改一个 `tickSpeed` 字段，而是一次运行时调度重构。

推荐实施顺序：

1. 先引入 actor clock 和输入 lane，但默认行为保持 `1x`。
2. 再把移动、战斗、采集等动作统一改成 scheduled action。
3. 最后开放玩家 `actionSpeed`，并把持久化、协议、GM 观测补齐。

在这套模型下，玩家可以更快或更慢地准备动作，但最终所有地图状态仍只在地图 tick 内统一裁定。这是保持 MMO 服务端权威、不可重叠占位、AOI 最小同步和持久化一致性的核心边界。
