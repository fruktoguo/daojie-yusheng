# 全战斗数值量化设计

本文定义战斗相关内容的长期配置口径：凡是影响战斗强度、承压、资源、行动效率、控制窗口、掉落收益或战斗表现可见数值的效果，都应能从境界等级、品阶、层级、百分比、倍率、公式变量和少量机制参数推导，不再要求策划为每个内容手填一组固定实际数值。

当前状态核对：2026-05-15。

## 目标

全战斗数值量化的目标不是把所有内容变成同一个数值模板，而是消灭不可比较的固定数值来源。

长期配置应回答这些问题：

- 这个内容处于哪个境界等级或装备等级。
- 它是什么品阶、类型、槽位或怪物层级。
- 它的定位偏向哪些属性、战斗数值、元素或机制。
- 它的强度吃多少份同等级基准。
- 它的冷却、消耗、范围、持续、触发条件是否会改变等效价值。

运行时实际数值必须由共享公式和启动期内容加载生成。战斗 tick、技能结算、怪物 AI 和 AOI 发包路径只读取已经展开好的运行时结构，不解析原始比例模板。

## 术语

| 名称 | 含义 |
|---|---|
| 基准值 | 某个境界等级或装备等级的一份通用量化价值。 |
| 占比 | 内容吃到几份基准值，`100` 表示一份。 |
| 倾向 | 多个字段之间的强度分配，例如怪物 `attrTendency/statTendency`。 |
| 固定值 | 直接写入实际战斗字段或价值字段，例如 `attrs`、`valueStats`、`equipValueStats`、逐层 `attrs`。 |
| 等效价值 | 把伤害、增益、控制、治疗、范围、冷却、消耗等折算到同一基准池后的价值。 |
| 机制参数 | 不能简单等价为属性点的规则参数，例如目标数量、位移格数、清除层数、生成地块数量。 |

## 当前已量化范围

### 怪物基础数值

怪物基础六维和基础战斗属性已经基本完成量化。

长期主线字段：

- `level` 或等价境界等级。
- `grade`。
- `tier`。
- `attrTendency`。
- `statTendency`。
- 少量怪物机制字段：`skills`、`aggroRange`、`leashRange`、`wanderRadius`、`attackRange`、掉落、刷新数量等。

公式来源：

- `packages/shared/src/monster.ts` 的 `resolveMonsterTemplateRecord()`。
- `packages/server/data/content/realm-attr-baselines.json` 的 `singleAttr` 和 `singleBaseStatValue`。
- `packages/shared/src/constants/gameplay/monster.ts` 的怪物全局、品阶和层级倍率。
- `docs/design/balance/怪物当前属性计算总览.md`。

长期规则：

- 新怪物不得新增固定 `attrs`、`valueStats`、`numericStats` 或 `stats` 作为基础属性。
- 怪物偏科只写百分比倾向；未写字段按 `100%`。
- 怪物初始 Buff 可以作为机制差异，但不能用来绕过基础数值倾向体系。
- 地图刷新点只配置模板引用、数量、位置和必要等级覆盖，不直接配置实际战斗数值。

### 内功六维与经验

内功功法已经具备量化主线。

长期主线字段：

- `grade`。
- `realmLv`。
- `category: "internal"`。
- `attrRatio`。
- `attrFloat`。
- `maxLayer`。
- `expDifficulty`。
- 稀疏 `layers` 只保留 `qiProjection` 等非主干数值 overlay。

公式来源：

- `packages/shared/src/technique.ts` 的 `expandTechniqueAttrRatio()`。
- `packages/shared/src/technique.ts` 的 `expandTechniqueExpCurve()`。
- `docs/design/systems/AI功法生成方案.md`。
- `docs/design/systems/功法加成设计.md`。

长期规则：

- 内功不得逐层手填完整 `attrs`。
- 满层六维总量由 `grade + realmLv + attrFloat` 决定。
- 六维分配由 `attrRatio` 决定。
- 经验曲线由 `grade + realmLv + maxLayer + expDifficulty + category` 决定。
- 若需要特殊气机吸收、灵气投影或里程碑效果，使用稀疏 overlay，不改变主干六维和经验曲线。

### 装备基础属性

装备基础属性已经有量化路径，但内容尚未完全迁移。

长期主线字段：

- `grade`。
- `level`。
- `equipSlot`。
- `equipBaselinePercents`。
- 必要的特殊属性字段，例如工具效率、材料属性、上下文动作。

公式来源：

- `packages/shared/src/value.ts` 的 `compileEquipmentBaselinePercentsToActualStats()`。
- `docs/design/balance/装备基准值占比改造预览.md`。

长期规则：

- 新装备基础战斗属性应写 `equipBaselinePercents`，不写 `equipValueStats`。
- `equipAttrs` 六维不再作为装备主线属性；如确实需要，必须先定义折算规则或转为特殊机制。
- `viewRange`、`extraRange`、`extraArea`、`actionsPerTurn` 不进入普通装备基础池，应作为特殊机制预算。
- 工具装备的炼丹、炼器、强化、采矿等效率属于技艺效果池，不和战斗基础属性混算。

## 仍需量化的范围

### 术法技能伤害

当前技能伤害已经使用结构化 `formula`，但公式强度还没有完全纳入统一预算。长期需要把每个技能的 damage effect 从“作者凭感觉写公式倍率”改成“按基准预算生成公式”。

需要量化的字段：

- `formula`。
- `damageKind`。
- `element`。
- `targeting.range`。
- `targeting.shape`。
- `targeting.radius`、`width`、`height`、`maxTargets`。
- `cooldown`。
- `cost` 或耗灵倍率。
- `playerCast.windupTicks`。

建议配置口径：

```json
{
  "type": "damage",
  "budgetPercent": 120,
  "scaling": {
    "primary": "caster.stat.spellAtk",
    "primaryPercent": 100,
    "techLevelPercent": 20
  },
  "damageKind": "spell",
  "element": "fire"
}
```

冷路径展开为当前运行时需要的 `formula`：

```json
{
  "op": "add",
  "args": [
    { "var": "caster.stat.spellAtk", "scale": 1 },
    { "var": "techLevel", "scale": 20 }
  ]
}
```

等效价值修正：

| 因素 | 方向 |
|---|---|
| 冷却越长 | 单次预算可提高，但 DPS 预算不应失控。 |
| 耗灵越高 | 单次预算可提高，但要受 `maxQiOutputPerTick` 和持续战斗约束。 |
| 范围越大 | 单目标伤害预算下降。 |
| 目标数越多 | 单目标预算下降。 |
| 需要吟唱或预警 | 单次预算可提高。 |
| 近身技能 | 可比远程技能略高。 |
| 条件增伤 | 基础预算降低，触发上限另算。 |

### 技能 Buff 和 Debuff

当前 `technique-buffs`、术法技能效果和怪物技能里仍有不少 `valueStats`。这是全战斗数值量化的最大残留之一。

需要量化的字段：

- `attrs`。
- `stats`。
- `valueStats`。
- `mainCombatStatsPercent`。
- `statMode`。
- `duration`。
- `maxStacks`。
- `qiProjection`。
- 触发条件、目标类型、可驱散性、死亡保留、回出生点保留。

长期字段建议：

```json
{
  "type": "buff",
  "buffId": "buff.example",
  "target": "self",
  "duration": 6,
  "maxStacks": 3,
  "budgetPercent": 80,
  "statProfile": {
    "physDef": 60,
    "spellDef": 40
  },
  "mode": "percent"
}
```

展开规则：

```text
单层等效预算 = 基准值(level) * budgetPercent / 100
字段实际占比 = statProfile[field] / sum(statProfile)
持续修正 = duration / 标准持续
叠层修正 = 预期层数 / maxStacks 上限
最终 stats/valueStats = 冷路径编译结果
```

特殊规则：

- 防御 Buff、攻击 Buff、命中/闪避 Buff、冷却 Buff 不应共用同一个上限；需要按字段族设置预算上限。
- `mainCombatStatsPercent` 只能作为临时兼容简写，长期应转成明确 `statProfile`。
- 负面 Debuff 的预算应按命中难度、持续、可驱散性和目标范围折算，不能简单等于正面 Buff。
- 控制型 Debuff 必须单独预算，不应同时附带满额伤害和满额属性削弱。

### 治疗、护盾、吸血和资源回复

这些效果当前可能散在技能效果、消耗品、Buff 或装备效果里。长期应统一为资源类效果预算。

需要量化的效果：

- 直接治疗。
- 持续回血。
- 护盾或护体。
- 生命吸取。
- 灵力回复。
- 灵力消耗降低。
- 元气投影或吸收效率。
- 复活、免死、回出生点保留类效果。

建议字段：

```json
{
  "type": "resource",
  "resource": "hp",
  "mode": "restore",
  "budgetPercent": 100,
  "delivery": "instant",
  "target": "ally"
}
```

折算原则：

- 直接治疗按 `maxHp` 基准或同等级伤害预算折算。
- 持续治疗按总量折算，并根据可打断、持续时间和叠层修正。
- 护盾按有效生命折算，但需要考虑是否吃防御、是否可被驱散、是否跨战保留。
- 灵力回复按技能循环价值折算，不直接和生命等价。
- 免死和复活属于高价值机制，不进入普通数值池，必须有独立上限和冷却。

### 控制、位移和行动效率

控制效果不是固定属性，但会强烈影响战斗价值，因此必须纳入量化。

需要量化的效果：

- 定身、眩晕、沉默、封身、封灵。
- 减速、加速。
- 强制位移、拉拽、击退。
- 禁疗、减疗。
- 冷却延长或冷却缩短。
- 额外行动次数。
- 技能射程、范围、目标数变化。

建议字段：

```json
{
  "type": "control",
  "controlKind": "root",
  "duration": 2,
  "budgetPercent": 70,
  "breakOnDamage": false
}
```

折算原则：

- 硬控按 `持续 tick * 目标数量 * 命中难度` 计价值。
- 软控按影响比例计价值，例如减速 30% 持续 5 tick。
- 冷却缩短和额外行动次数属于高危字段，应设独立上限，不进入普通 Buff 池。
- 位移既是控制也是空间收益，需同时按距离、方向、是否可抵抗、地形影响计价。

### 临时地块、召唤物和阵法

临时石头、警戒区、阵法削灵、召唤物等目前是机制配置，未来也需要数值量化。

需要量化的效果：

- 临时地块 HP。
- 临时地块持续时间。
- 临时地块数量和形状。
- 阵法承伤和灵力削减。
- 召唤物基础属性、持续时间、技能和数量。
- 警戒区延迟和命中窗口。

建议字段：

```json
{
  "type": "temporary_tile",
  "tileType": "stone",
  "hpBudgetPercent": 80,
  "durationTicks": 8,
  "maxCells": 6
}
```

折算原则：

- 临时地块 HP 按同等级单体伤害预算或地块耐久基准折算。
- 多格地形按可封锁面积和持续时间折价。
- 召唤物直接使用怪物量化体系，配置 `level/grade/tier/attrTendency/statTendency`，不写固定属性。
- 阵法和召唤物必须明确资产、持久化和清理边界，不能在战斗 tick 中临时解析配置。

### 装备效果、套装和技艺工具

装备基础属性之外，装备效果仍需要分池量化。

需要量化的效果：

- 常驻基础战斗属性。
- 装备触发 Buff。
- 套装效果。
- 技艺工具效果：炼丹、炼器、强化、采矿、建造等。
- 掉落、稀有掉落、修炼效率、功法效率。
- 元素增伤/减伤。

长期规则：

- 常驻基础战斗属性走 `equipBaselinePercents`。
- 战斗触发效果走 Buff/Debuff/伤害/资源效果预算。
- 技艺工具走技艺效果池，不占战斗基础属性预算。
- 掉落、修炼、功法效率属于运营收益池，不和战斗属性混算。
- 元素增伤/减伤作为元素池预算，不直接塞进普通 `valueStats`。

### 消耗品和战斗道具

消耗品可以有固定功能，但战斗效果仍要量化。

需要量化的效果：

- 直接回血、回灵。
- 临时 Buff。
- 临时 Debuff。
- 解控、净化、驱散。
- 复活、传送、回城、绑定点。
- 战斗中可使用次数、冷却和叠加限制。

长期规则：

- 战斗药品按同等级资源预算生成数值。
- 解控和净化按机制预算，不用基础属性价值替代。
- 消耗品要有战斗内冷却、背包使用频率和经济成本联动，否则数值预算会失效。

### 怪物专用技能和 Boss 阶段效果

怪物基础属性已经量化，但怪物专用技能仍存在固定 `valueStats` 或手写公式。

长期规则：

- 怪物技能复用玩家技能公式预算，但允许怪物侧有 AI 和阶段修正。
- Boss 阶段 Buff 用 `phaseBudgetPercent` 或 `mainCombatStatsProfile`，不得直接写固定实际值。
- Boss 特殊机制应拆成明确机制参数：预警范围、持续、召唤数量、地形改变、阶段阈值。
- Boss 低血量削弱或狂暴都应配置为百分比和阶段条件，不写一组新实际属性。

## 推荐字段体系

### 通用预算字段

所有可量化效果都应支持这些冷路径字段中的一部分：

| 字段 | 含义 |
|---|---|
| `budgetPercent` | 该效果吃多少份同级基准。 |
| `profile` | 多字段分配比例。 |
| `grade` | 品阶倍率来源。 |
| `level` 或 `realmLv` | 等级基准来源。 |
| `category` | 决定使用战斗、资源、控制、运营还是技艺池。 |
| `durationTicks` | 持续时间。 |
| `maxStacks` | 最大层数。 |
| `expectedStacks` | 平衡时按几层计算。 |
| `cooldownTicks` | 冷却。 |
| `costMultiplier` | 消耗倍率。 |
| `targetBudget` | 单体、范围、多目标预算修正。 |
| `conditionMultiplier` | 条件触发上限修正。 |

### 冷路径展开结果

配置导入或服务端启动期应把量化字段展开为现有运行时字段：

| 来源字段 | 展开目标 |
|---|---|
| `attrRatio` | `layers[].attrs`。 |
| `equipBaselinePercents` | `equipStats`。 |
| `statProfile + budgetPercent` | `stats` 或 `valueStats`。 |
| `damageBudget + scaling` | `formula`。 |
| `controlBudget` | 控制 effect 参数。 |
| `temporaryTileBudget` | `hpFormula` 或固定展开后的 HP。 |
| `summonTemplate` | 怪物模板引用和量化覆盖。 |

运行时只消费展开结果；原始预算字段可保留在编辑器和审计数据中。

## 迁移顺序

### 第一阶段：盘点和门禁

- 增加内容审计脚本，统计 `attrs/valueStats/equipValueStats/equipStats/layers[].attrs/mainCombatStatsPercent` 等固定值字段。
- 输出按内容类型分组的残留列表：怪物、内功、术法、妖兽专用术法、技能 Buff、装备、消耗品。
- 新增规则：怪物新配置禁止固定基础属性；内功新配置必须走 `attrRatio`。

### 第二阶段：装备基础属性收口

- 把剩余 `equipValueStats` 转为 `equipBaselinePercents`。
- 处理残留 `equipAttrs`，要么删除，要么按明确规则折算。
- 将装备 effects 中的战斗属性 Buff 拆到 Buff 量化池。
- 保留技艺工具效果，不和战斗基础属性混算。

### 第三阶段：术法和怪物技能伤害预算化

- 为 damage effect 增加预算源字段。
- 先保留冷路径展开出的 `formula`，运行时不变。
- 对现有 `formula` 反推预算，生成对照表。
- 针对范围、冷却、消耗、吟唱、多目标建立修正系数。

### 第四阶段：Buff/Debuff 固定值迁移

- 把 `technique-buffs/**/*.json` 的 `valueStats` 迁移到 `budgetPercent + statProfile`。
- 把技能内嵌 buff 的 `valueStats` 迁移到同一口径。
- 将 `mainCombatStatsPercent` 从兼容简写迁移为明确主要战斗属性 profile。
- 对控制、净化、驱散、免死等机制效果建立独立预算，不强行折成普通属性。

### 第五阶段：机制效果预算化

- 临时地块 HP、持续、数量预算化。
- 召唤物使用怪物模板量化口径。
- 阵法承伤、削灵和护宗大阵效果预算化。
- 消耗品战斗效果预算化，并和经济成本、战斗冷却绑定。

### 第六阶段：删除兼容固定值入口

- 怪物 `valueStats/attrs` 兼容入口转为迁移工具专用，不再允许生产内容引用。
- 装备 `equipValueStats` 不再作为新内容字段。
- Buff 的裸 `valueStats` 改为生成产物或迁移输入，不作为主配置字段。
- 内容审计纳入发布门禁。

## 验证要求

### 内容审计

应提供一个稳定审计入口，至少回答：

- 当前还有多少固定基础属性配置。
- 每个固定配置属于哪类内容、哪个文件、哪个字段。
- 是否有新增怪物固定属性。
- 是否有新增内功逐层固定六维。
- 是否有新增装备 `equipValueStats`。
- 是否有新增技能/Buff 裸 `valueStats`。

### 数值回归

每类迁移都要有迁移前后对照：

- 原始等效价值。
- 新预算字段。
- 展开后的运行时值。
- 差异百分比。
- 是否超过允许误差。

怪物和装备应继续使用现有生成快照或 smoke 验证：

- 怪物：`content-monster-spawn-smoke` 和生成的 `monster-runtime-stats.json`。
- 装备：装备面板 payload smoke、客户端 tooltip 检查。
- 功法：内功展开 smoke 和功法面板检查。
- 技能：combat matrix、技能公式 parity smoke。

### 运行时边界

量化字段不得进入战斗热路径解析：

- 技能施放时不解析预算字段，只执行已展开 `formula/effects`。
- 怪物 tick 不解析 `attrTendency/statTendency`，只读模板展开后的属性。
- 装备穿戴不重新解析原始 JSON，只应用 item stack 上已编译字段。
- 战斗通知只展示结构化结果，不承担数值反算。

## 不应量化为普通战斗属性的内容

这些内容也要有规则，但不应塞进基础属性池：

- 地图解锁、传送、绑定点。
- 任务触发、剧情状态。
- GM、运营、审计字段。
- 纯表现字段：颜色、特效、图标、文案。
- 材料来源、采集来源、合成关系。
- 经济价格和交易税，除非明确做战斗消耗品经济平衡。

这些字段可以有自己的配置规范，但不参与全战斗数值量化。

## 最终完成标准

全战斗数值量化完成时，应满足：

- 新怪物只需配置等级、品阶、层级、倾向和技能机制即可得到基础战斗数值。
- 新内功只需配置品阶、境界、层数、六维分配和浮动即可得到逐层六维与经验。
- 新装备只需配置等级、品阶、槽位和属性占比即可得到基础装备数值。
- 新技能只需配置预算、缩放变量、冷却、消耗、范围和机制参数即可得到运行时公式。
- 新 Buff/Debuff 只需配置预算、字段 profile、持续、叠层和目标即可得到运行时效果。
- 新消耗品、召唤、临时地块、阵法、控制和资源效果都有独立预算池。
- 内容审计能阻止新增固定战斗数值配置。
- 发布前验证能证明展开结果、运行时消费、客户端展示和协议字段没有漂移。
