# 分身 / 宠物 / 机器人 共用 Actor 系统设计

> 状态：**草案 v0.1（设计阶段，未实施）**

提供统一的 Actor 抽象，让 Bot（压测机器人）、分身（玩法）、宠物（玩法）共用占位/视野/战斗/AOI 逻辑，差异仅在行为驱动方式和持久化策略。Bot 通过真实 socket.io 协议与服务端交互，属性快照克隆自指定玩家，运营数据零污染。

---

## 2. 设计目标 / 非目标

### 2.1 目标

| # | 目标 |
|---|---|
| G1 | bot 模块作为独立进程 / 独立 npm 包存在；通过 socket.io 客户端协议与 server 通信，与真实玩家走同一套链路 |
| G2 | 服务端能根据"模板玩家 ID"签发一个 ephemeral shadow actor 凭证（包含装备/功法/境界等运行态克隆数据），bot 客户端用此凭证登录后属性与模板玩家相同 |
| G3 | bot 标识在服务端全链路被识别为 "ephemeral / non-persistent"：不入数据库真源（player_runtime_state / inventory / equipment 等）、不进邮件、不进 leaderboard、不进 market、不影响市场风控、不写 player_counters；运营数据零污染 |
| G4 | 抽象出 ActorBlueprint / EphemeralActorIdentity / BehaviorDriver / CombatantProjection / ActorPresence 等共用模块，分身、宠物、bot 都基于这些模块构造 |
| G5 | 200~5000 个 bot 单进程或多进程并发，能稳定 30 分钟以上运行，覆盖移动、攻击、用药、面板订阅、切图、断线重连等场景 |
| G6 | 分身 / 宠物玩法系统能复用 G2 / G4 中的核心抽象，差异点（行为驱动来源、持久化策略、可见性）通过策略接口注入 |

### 2.2 非目标

| # | 非目标 |
|---|---|
| N1 | 不模拟客户端 UI 渲染、Canvas 帧、客户端 JS heap 增长——这些通过真实玩家观察 |
| N2 | 不模拟真实公网延迟 / 丢包；如果需要，靠 Linux tc / docker network emulator 单独叠加 |
| N3 | 不为 bot 设计经济玩法、奖励、邮件、社交关系；bot 不参与运营态 |
| N4 | 不重写战斗系统；bot 走与真实玩家完全一致的 CombatAction 管线 |
| N5 | 分身 / 宠物的具体玩法规则（养成、契约、消耗、绑定）不在本文件展开，仅约定它们如何复用 actor 抽象 |

## 3. 概念模型与角色谱系

### 3.1 核心概念

```
┌────────────────────────────────────────────────────────────┐
│                       Actor（运行时实体）                   │
│  ─ playerId（统一标识；前缀区分类型）                       │
│  ─ identity（名字/称号/外观）                              │
│  ─ presence（占位/位置/朝向/视野）                          │
│  ─ combatant（属性/装备/功法/buff/血量灵力/技能 cd）         │
│  ─ behavior driver（谁在驱动这个 actor 的意图）              │
│  ─ persistence policy（落库与否、落到哪里）                  │
└────────────────────────────────────────────────────────────┘
```

任何"在世界格子上占位、参与战斗、被 AOI 看见"的实体都应该实现这个抽象。当前的"真实玩家"是其中的一个特例，它的：

- identity 来自 `native_player_account`
- presence 由 socket 发起的 Move / Hello 驱动
- combatant 由数据库真源 + runtime tick 推进
- behavior driver = 远程客户端 socket
- persistence policy = 全字段持久化

### 3.2 三种新 Actor

| Actor | 行为驱动 | identity 来源 | combatant 来源 | persistence |
|---|---|---|---|---|
| **Bot（机器人）** | 远程 socket（独立进程；模拟真实客户端） | ephemeral，无账号 | 模板玩家快照（去掉 inventory） | 全链路跳过 |
| **分身（玩法）** | 主玩家指令 + 服务端 AI 协同 | 派生自主玩家身份（如 `主玩家ID:clone:1`） | 主玩家快照或主玩家共享引用（玩法决定） | 派生持久化（落主玩家分身表，不入主线 player_*） |
| **宠物（玩法）** | 服务端 AI（行为树） | 玩家持有的宠物记录 | 独立模板（基于宠物配置成长） | 落到玩家持久化下属（player_pet 表） |

| Actor | 数量级 | 时间尺度 | 与目标玩家关系 |
|---|---|---|---|
| Bot | 200~5000 同时 | 测试期间（分钟~小时） | 一次性快照，独立演化 |
| 分身 | 1~3 个/玩家 | 长期 | 持续依赖主玩家 |
| 宠物 | 1~5 个/玩家 | 长期 | 独立成长 |

### 3.3 共用 vs 独立模块矩阵

| 模块 | Bot | 分身 | 宠物 | 玩家 |
|---|---|---|---|---|
| ActorBlueprint（可克隆 combatant 快照） | ✅ 用 | ✅ 用 | 部分用 | ✅ 写 |
| EphemeralActorIdentity（短期身份） | ✅ | ✅（派生） | — | — |
| ActorPresence（占位/视野/AOI） | ✅ 共用 | ✅ 共用 | ✅ 共用 | ✅ 共用 |
| CombatantProjection（战斗管线接口） | ✅ 共用 | ✅ 共用 | ✅ 共用 | ✅ 共用 |
| BehaviorDriver | RemoteSocketDriver | OwnerCommandDriver + AICompanionDriver | AICompanionDriver | RemoteSocketDriver |
| PersistencePolicy | NoPersistencePolicy | DerivedFromOwnerPolicy | OwnerSubResourcePolicy | FullPolicy |
| Bot 客户端进程 | ✅ 独有 | — | — | — |
| 玩法绑定 / 契约 / 培养 | — | ✅ 独有 | ✅ 独有 | — |

**关键洞察**：四种 actor 真正的差异是 **行为驱动 + 持久化策略**两件事，其余 90% 的运行时表达可以共用。

### 4.3 与现有运行时的关系

- **PlayerRuntimeService 不再硬编码"全字段持久化"**，写入路径检查 `player.persistencePolicy` 决定 dirty domain 是否落库
- **WorldGateway / Native HTTP** 接受 bot ephemeral token 后构造 player runtime entry，但走 `EphemeralActorIdentityService` 而非 `NativePlayerAuthStoreService`
- **CombatPipeline / AOI / tick / 寻路** 路径无须感知 actor 类型，全部通过 player runtime entry 操作；唯一区别是 bot/分身/宠物在 dirty domain 写入时被 PersistencePolicy 拦截

## 5. 关键模块设计

### 5.1 ActorBlueprint（可克隆 actor 状态）

`packages/shared/src/actor/actor-blueprint.ts`（新增）

```ts
/**
 * 可克隆的 actor 战斗运行态快照。
 * 不含 inventory、wallet、market、邮件、quest、accountId、社交关系。
 */
export interface ActorBlueprint {
  /** 仅作为 blueprint 元信息，运行时 actor 自己生成 playerId */
  blueprintId: string;
  sourcePlayerId: string | null;     // 克隆来源；分身/宠物可为 null
  createdAtMs: number;
  templateMapId: string;             // 推荐生成位置
  templateX: number;
  templateY: number;
  identity: {
    name: string;
    displayName: string;
    sectId: string | null;
  };
  combatant: {
    attrs: AttrStateSnapshot;        // baseAttrs / finalAttrs / numericStats / ratioDivisors / specialStats
    realm: RealmStateSnapshot;       // stage / realmLv / progress / breakthroughReady
    bodyTraining: BodyTrainingSnapshot;
    heavenGate: HeavenGateSnapshot | null;
    spiritualRoots: SpiritualRootsSnapshot | null;
    runtimeBonuses: RuntimeBonusSnapshot[];
    equipment: EquipmentSlotSnapshot[];      // 装备本身（ref-only，不入 inventory）
    techniques: TechniqueEntrySnapshot[];    // 已学功法 + 等级 + skills
    activeSkills: ActiveSkillEntry[];        // 当前可用技能（含 cooldown）
    buffs: TemporaryBuffSnapshot[];
    combatPrefs: CombatPreferenceSnapshot;   // autoBattle / autoRetaliate / 自动战斗技能等
  };
  vitalsPolicy: 'full_hp_qi' | 'inherit';    // bot 推荐 full；分身可 inherit
}
```

**生成路径（服务端）**：`ActorBlueprintService.fromPlayer(sourcePlayerId, options): ActorBlueprint`

- 从 `PlayerRuntimeService.getPlayer(sourcePlayerId)` 读取运行态（不读持久化，避免锁）
- 对每个子结构走 freeze / 深拷贝（这部分复用 projector-clone 现有的 clone* 函数族）
- 显式忽略：`inventory`、`wallet`、`marketStorage`、`pendingLogbookMessages`、`alchemyJob` / `forgingJob` / `gatherJob` / `buildingJob` / `enhancementJob`、`mail`、`quests`、`unlockedMapIds`、`heavenGate.requirementsAccessLog`
- 输出 frozen 数据，TS 标 `Readonly<ActorBlueprint>`

**消费路径**：

- bot 客户端登录后 server 拿 blueprintId 还原；或 bot 客户端在登录时直接 inline 携带（小尺寸）
- ActorBlueprint Registry 持有 LRU（容量例如 1000，TTL 30 分钟），过期自动 GC
- 分身系统创建分身时也调用 `ActorBlueprintService.fromPlayer(ownerId, ...)`
- 宠物系统不需要从玩家克隆，但需要相同的"应用 blueprint 到 runtime player"管线

### 5.2 EphemeralActorIdentity（无持久化身份）

`packages/server/src/runtime/actor/ephemeral-actor-identity.service.ts`（新增）

```ts
export type EphemeralActorKind = 'bot' | 'clone' | 'pet';

export interface EphemeralActorIdentity {
  playerId: string;             // 前缀化（bot_ / clone_ / pet_），全链路据此识别
  kind: EphemeralActorKind;
  ownerPlayerId: string | null; // 分身/宠物指向主玩家；bot 为 null
  blueprintId: string | null;
  issuedAtMs: number;
  expiresAtMs: number;          // bot 推荐 2 小时，分身/宠物按玩法长期
}
```

**职责**：

1. 签发 / 注销 ephemeral identity
2. 提供 `isEphemeralPlayerId(playerId): boolean` 与 `getEphemeralKind(playerId)` 两个全链路识别接口
3. 替换现有 `isNativeGmBotPlayerId`：现 GM bot 视为 `kind: 'bot'` 的特例（迁移期保留兼容）

**playerId 前缀**：

- `bot_<token>` — RemoteSocketDriver 驱动的远程客户端 bot
- `clone_<owner>_<seq>` — 主玩家分身
- `pet_<owner>_<petId>` — 主玩家宠物

`isEphemeralPlayerId` 通过前缀直接判断，零开销，可在所有写持久化的服务中嵌入门控。

### 5.3 ActorPresence（占位 / 视野 / AOI 表达）

不新建模块，**复用现有 `MapInstanceRuntime` 的 player 表**：

- 一个 ephemeral actor 在世界上的存在与真实玩家完全一致：进入实例、占位、被 AOI 看见、tick 推进
- 关键改造：`MapInstanceRuntime.connectPlayer` 接受可选的 `actorKind` 字段，仅用于：
  - 在 `playersById` 上同步打 actor kind 标签（供 GM 面板、广播 patch、leaderboard 过滤使用）
  - 不影响占位、移动合法性、战斗结算

> 这是本设计的核心：ephemeral actor **就是 player runtime entry**，不另起一套实体。共用 100% 的占位 / 视野 / 战斗 / 寻路 / AOI 逻辑。

### 5.4 BehaviorDriver（行为驱动接口）

`packages/server/src/runtime/actor/behavior-driver.ts`（新增；接口）

```ts
export interface BehaviorDriver {
  readonly kind: 'remote_socket' | 'owner_command' | 'server_ai_companion' | 'server_ai_pet';
  /** actor 接入实例后调用一次，绑定 driver */
  attach(actorPlayerId: string, deps: BehaviorDriverDeps): void;
  /** actor 被销毁时调用一次 */
  detach(actorPlayerId: string): void;
  /** 每个 tick 调用一次（仅 server-driven 类型实现），让 driver 把意图入队 */
  drive?(actorPlayerId: string, currentTick: number): void;
}
```

| Driver 实现 | 用于 | 入队的意图来自 |
|---|---|---|
| RemoteSocketDriver | 真实玩家 / Bot | 远程 socket 上的 C2S 消息 |
| OwnerCommandDriver | 分身（主玩家明确发指令） | 主玩家通过特定 C2S 消息（`CloneIssueOrder`）下发 |
| ServerAICompanionDriver | 分身（自主跟随/补刀） | 服务端行为树 + 主玩家上下文 |
| ServerAIPetDriver | 宠物 | 服务端行为树 + 玩家位置 |

**真实玩家与 bot 在服务端是同一个 driver 实现**（都是 RemoteSocketDriver），驱动来源都是 socket 消息——这是用户需求"和真实玩家一模一样"的关键支点。

### 5.5 PersistencePolicy（落库策略）

`packages/server/src/persistence/actor-persistence-policy.service.ts`（新增）

```ts
export type ActorPersistencePolicy =
  | { kind: 'full' }                        // 真实玩家：全字段持久化
  | { kind: 'none' }                        // bot：完全不持久化
  | { kind: 'derived_from_owner';           // 分身：派生持久化
      ownerPlayerId: string;
      derivedSubresource: 'clone';
    }
  | { kind: 'owner_sub_resource';           // 宠物：作为玩家子资源
      ownerPlayerId: string;
      derivedSubresource: 'pet';
    };
```

**接入位置**（按现有持久化链路）：

| 服务 | 现状 | 改造 |
|---|---|---|
| `PlayerPersistenceFlushService` | 默认所有 player flush | 入口先查 `PersistencePolicy`：`none` 直接 noop；`full` 走原路径；其余走分身/宠物子表 |
| `PlayerCountersPersistenceService` | 已 `isNativeGmBotPlayerId` 跳过 | 改用 `EphemeralActorIdentity.isEphemeralPlayerId`，覆盖 bot/clone/pet |
| `MailRuntimeService` | 已部分跳过 bot | 同上 |
| `LeaderboardRuntimeService` | 已部分跳过 | 同上 |
| `MarketPersistenceService` | 未对 bot 做处理 | 新增 ephemeral 短路 |
| `OutboxDispatcherRuntimeService` | 落事件 | bot 玩家事件不入 outbox |
| `DurableOperationService` | lease 检查 | 不变 |
| `PlayerDomainPersistenceService` | flush domain | ephemeral 跳过 |
| `WorldRuntimeService.markPersistenceDirtyDomains` | 标 dirty | bot 标记直接丢弃 |

> 实施时**抽出统一的 `withPersistencePolicy(playerId, op)` 装饰器**，所有需要"按 policy 路由"的服务调用它，避免散落在 N 处。

## 6. 协议扩展

### 6.1 Bot 凭证签发

GM 触发或自动化脚本触发：

```http
POST /api/gm/bot/issue-blueprint
Authorization: Bearer <gm token>
Content-Type: application/json

{
  "sourcePlayerId": "p_xxx",
  "count": 200,                    // 一次签发的 bot 数量
  "spawnMapId": "yunlai_town",     // 推荐 spawn 地图（可省略，默认源玩家所在地图）
  "spawnAnchor": { "x": 32, "y": 5 },
  "behaviorProfile": "wander_attack",  // 行为脚本预设
  "ttlMinutes": 120
}
```

响应：

```json
{
  "ok": true,
  "blueprintId": "bp_xxx",
  "issuedTokens": [
    {
      "playerId": "bot_xxx_1",
      "loginToken": "v1.eyJ...",
      "expiresAtMs": 1778950000000,
      "spawnMapId": "yunlai_town",
      "x": 32,
      "y": 5
    },
    ...
  ]
}
```

### 6.2 Bot 客户端登录

bot-runner 拿到 token 后通过现有 native player auth 流程登录：

```
GET  /api/auth/bot/connect-token   → 换发 short-lived connect token
WS   socket.io connect(?token=...) → 建立 socket
emit C2S.Hello { playerId, mapId, preferredX, preferredY, sessionToken }
```

**关键约束**：bot 走的是**与真实玩家完全相同的 Hello / WorldDelta / SelfDelta / PanelDelta** 流量，唯一差异是 server 端识别 playerId 前缀后查 `EphemeralActorIdentityService` 拿到 blueprint 并应用，而不是从 `native_player_account` 读账号。

### 6.3 Bot 释放

```http
POST /api/gm/bot/release
{
  "playerIds": ["bot_xxx_1", ...],   // 或 "all": true
}
```

服务端：

1. 让 socket disconnect
2. 从 PlayerRuntimeService 移除 player entry
3. 从 ActorBlueprint Registry 删除 blueprint
4. 注销 EphemeralActorIdentity

### 6.4 与现有 GM bot 协议的关系

- `GmSpawnBots` / `GmRemoveBots` 协议保留兼容期，但内部实现改为：先调用 `BotIssueBlueprintService` 然后由 server 拉起 in-process loopback bot client（仅作为兼容路径）
- 长期 deprecate 直接 in-process spawn 路径

## 7. Bot Runner（独立进程）设计

### 7.1 Workspace 布局

```
packages/bot-runner/
  package.json
  src/
    cli.ts                  # 入口，parse argv
    config.ts               # 配置 schema（zod）
    bot.ts                  # 单个 bot 状态机
    bot-pool.ts             # 同进程多 bot 编排
    behavior/
      idle.ts
      wander.ts
      autobattle.ts
      mixed.ts
      record-replay.ts      # 录制真实玩家然后回放
    metrics.ts              # RSS / connect 数 / 帧延迟 / 协议错误统计
    transport/
      socket-client.ts      # socket.io-client 封装
      http-client.ts        # native HTTP 调用
  bin/
    bot-runner              # node dist/cli.js
  Dockerfile
```

### 7.2 启动流程

```
$ bot-runner \
    --server-url http://127.0.0.1:3000 \
    --gm-token <gm-token> \
    --source-player-id p_xxx \
    --count 200 \
    --behavior wander_attack \
    --spawn-map yunlai_town \
    --duration 30m \
    --report .runtime/bot-report.json
```

1. 通过 `/api/gm/bot/issue-blueprint` 一次性领取 N 个 bot 的登录 token
2. 启动 N 个 bot socket（同进程内事件循环 + 多 socket.io-client 连接）
3. 每个 bot 进入对应行为状态机
4. 周期性采集：socket connect 数、消息往返延迟、收到的 panel revision 频率、本地内存等
5. duration 到期或收到 SIGINT 时调用 `/api/gm/bot/release` 主动释放

### 7.3 行为脚本

| Profile | 行为 |
|---|---|
| `idle` | 静止挂机 |
| `wander` | 周期性随机走动（每 N tick 随机 1 步） |
| `autobattle` | 进入后开 autoBattle，遇怪自动战斗 |
| `wander_attack` | wander + 看到怪物自动 engage |
| `panel_churn` | 周期性打开/关闭背包/装备/功法面板（触发 panel revision，bumps 持久化测试） |
| `mixed` | 上述行为按权重随机切换 |
| `record_replay` | 回放预先录制的真实玩家 socket 消息流（最高保真） |

行为脚本通过策略模式注入，单个 bot 可叠加多个 profile（例如 `wander + panel_churn`）。

### 7.4 配置规模 / 部署

| 规模 | 部署方式 |
|---|---|
| 50~500 bot | 单进程 single bot-runner |
| 500~2000 bot | 单机多进程（n × bot-runner，每进程 200~500） |
| 2000~5000 bot | 多机 / docker compose（k 个 bot-runner 容器） |

> 单 socket 进程内存估 5 MB（含 socket.io 内部缓冲），200 bot ≈ 1 GB；本机或同 docker network 内可承载。

### 7.5 录制回放（高保真模式预留）

- 真实玩家 socket 经过 server 时，开关打开后落 `.runtime/socket-tape/<playerId>.jsonl`
- bot-runner `--behavior record_replay --tape <file>` 重放该流
- 解决"真实玩家行为多样性 bot 模拟不到"的核心问题；用于上线前回归

## 8. 共用模块在分身 / 宠物上的复用

### 8.1 分身

- identity 用 `clone_<owner>_<seq>`，由 `EphemeralActorIdentityService` 派发
- combatant 来自 `ActorBlueprintService.fromPlayer(ownerId)`，可选 `inherit` vitals 模式
- behavior driver = `OwnerCommandDriver`（玩家通过 `CloneIssueOrder` 下指令） + `ServerAICompanionDriver`（自动跟随/补刀）混合
- persistence policy = `derived_from_owner`，落 `player_clone_state`（新表）
- 与玩家共用：占位 / 视野 / 战斗 / 寻路 / AOI / tick

**额外 vs Bot**：分身有等级、绑定关系、消耗、活跃时长配额、不掉装备等玩法字段；这些进 `player_clone_state`。

### 8.2 宠物

- identity 用 `pet_<owner>_<petId>`
- combatant 来自宠物模板（不是从玩家克隆），但仍走 `ActorBlueprint`（玩家克隆是 source_player_id 模式，宠物是 source_template_id 模式，shape 一致）
- behavior driver = `ServerAIPetDriver`（行为树）
- persistence policy = `owner_sub_resource`，落 `player_pet`
- 与玩家共用：占位 / 视野 / 战斗 / 寻路 / AOI / tick

### 8.3 Bot

- identity 用 `bot_<token>`
- combatant 来自玩家 blueprint
- behavior driver = `RemoteSocketDriver`（与真实玩家同实现）
- persistence policy = `none`

| 复用点 | 玩家 | Bot | 分身 | 宠物 |
|---|---|---|---|---|
| ActorBlueprint | 写源 | 读 | 读（owner 派生） | 读（template 派生） |
| EphemeralActorIdentity | — | bot | clone | pet |
| ActorPresence | ✅ | ✅ | ✅ | ✅ |
| CombatantProjection | ✅ | ✅ | ✅ | ✅ |
| BehaviorDriver | RemoteSocket | RemoteSocket | OwnerCommand+AI | AI |
| PersistencePolicy | full | none | derived | owner-sub |

## 9. 安全与防滥用

| 风险 | 缓解 |
|---|---|
| 生产环境误开 bot 登录 → 任意人可用 GM 创建无限 bot 占资源 | bot login 路由由 `SERVER_BOT_LOGIN_ENABLED` 环境变量门控；默认 prod=0；GM 控制台仅在该开关开启时可见 |
| bot token 泄露 / 被复用 | token 一次签发即绑定 bot playerId，不能跨 actor 复用；TTL 默认 2 小时；服务端记录 issuedTokens 已使用集合 |
| bot 数量爆炸打满 socket | 每 GM token 同时持有的 bot 上限（默认 1000），全局上限（默认 5000） |
| bot 进受限副本（通天塔、师门等）扰乱玩家体验 | `actorKind === 'bot'` 时被实例 entry-policy 主动拒绝；只允许进入"可压测白名单地图" |
| bot 写入污染运营态（市场/邮件/leaderboard/审计） | PersistencePolicy 路由 + ephemeral identity 全链路检测，所有写持久化 / 写运营态的服务必须先过 `withPersistencePolicy` |
| bot 影响真实玩家战斗体验 | 默认创建为"低优先级 actor"，PVE 不主动攻击真实玩家；禁用 PVP；战斗 fx 广播带 `isBotSource` 标签由客户端可选 |
| 分身 / 宠物的克隆能力被滥用（GM 给玩家造无限分身） | 限购 / 限活 / 玩法约束（与本文件无关，由玩法系统管） |

## 10. 可观测性

bot 测试期间需要的指标：

| 指标 | 来源 | 用途 |
|---|---|---|
| 连接数 / 在线 bot 数 | bot-runner | 是否连满 |
| 协议错误率 | bot-runner | 是否触发 server 拒绝 |
| 单 bot 平均消息往返延迟 | bot-runner | server tick 是否积压 |
| 服务端 RSS / heap 曲线 | server `/api/gm/diagnostics/memory` | 内存基线 |
| 服务端 GC pause 直方图 | server（trace-gc） | GC 压力 |
| tick latency 99 分位 | server tick metrics | 是否退化 |
| AOI / projector cache 命中率 | server | 内存优化是否生效 |
| 数据库 flush 队列长度 | server | 是否堆积 |

bot-runner 输出 JSON 报告，与 server 侧指标在 `.runtime/bot-report-<timestamp>.json` 合并，便于前后对比。

## 13. 风险与限制

| 风险 | 评级 | 处置 |
|---|---|---|
| `withPersistencePolicy` 装饰器接入面广，漏写一处就泄露 bot 数据到运营态 | 高 | 走"白名单 + 审计 smoke"：列出所有写持久化的服务清单，逐个接入；新增 `actor-persistence-policy-coverage-smoke` 静态扫描，CI 强制 |
| bot 与真实玩家共用 socket 和命令链路，bot 异常行为可能撑爆 server tick 队列 | 中 | bot 行为脚本带"每 tick 最多 1 个意图"上限；GM 后台可一键 kill 所有 bot |
| ActorBlueprint 内嵌 equipment / techniques 结构在玩家更新装备时与 bot 偏离 | 中 | 设计上 bot 是"快照后独立演化"，源玩家变化不再回灌；分身玩法另行决定是否主动同步 |
| 单进程多 socket 受 ulimit / FD 限制 | 低 | docker-compose 拆多容器；明确单进程上限 1000 socket |
| 分身玩法引入后 PVP 平衡问题 | 中 | 不在本文件范围；玩法系统层面解决 |
| ActorBlueprint TTL 过期导致 bot 长跑后 reload 失败 | 低 | bot-runner 本地缓存 blueprint；server 端续期接口 |
| 录制回放（record_replay）涉及隐私 / 合规 | 中 | 仅在 dev/staging 启用；记录前匿名化 playerId、屏蔽聊天/邮件内容 |
