## 0. 一句话定性

整套服务端可以用一句话概括：

> **"反正只有 1 个 Node.js event loop，那就什么都串行做；反正功能能 work 就行，规模问题以后再说。"**
>
> 表现为：所有实例的 tick、所有玩家的属性结算、所有 socket 广播扫全表、所有持久化 DELETE 全 INSERT 全、所有缓存按玩家全量复制；单体 setInterval 全表轮询 outbox、内存 Map 当数据库用、按用户 ID 全表 SELECT 后 JS 端 filter/sort、每帧重建已收敛资源、协议层有"未启用"的 protobuf 死代码、worker 入口写了一堆但生产没部署、GM 操作完全绕过 audit 链。

这套思路在内测 30 玩家时不会暴露任何问题，5000 玩家 / 10000 实例 / 8 核 / 30Mbps 的商业级目标与这套架构是两个世界。需要改的不是某几个函数，而是体系性结构。

下面 N1–N51 是已经在生产代码里坐实、且**设计层即不成立**的反人类问题。前 14 项（N1–N14）来自首轮按服务端核心链路的扫描，后 37 项（N15–N51）来自多 agent 按领域并行深扫。

## 一、首轮 · 服务端核心链路（N1–N14）

### N1. ✅ 已修复（本轮） · socket.io 没用 room，所有"广播"都是 O(在线人数) 全表扫

**现场**：`packages/server/src` 全文 grep 不到任何 `socket.join` / `io.to(room)` / `socket.to(room)` / `server.to(room)`，**socket.io 的 room 子系统整个没在用**。

**本轮修复**：`WorldSessionService` 增加实例 room 归属索引，room 名统一为 `world:instance:{instanceId}`；`WorldGateway.handleConnection` 把 Socket.IO server 注入会话层；`WorldSyncService` 在初始同步、增量同步和全员 flush 的每个玩家个性化同步前按当前 `view.instance.instanceId` 校准 room；`WorldClientEventService.broadcastChat` 改走 `emitToInstance(instanceId, S2C.Notice, ...)`，不再扫全局 binding 后逐个比较实例。断线、顶号替换、玩家迁图时会同步 leave 旧实例 room 并维护 `instanceId -> playerId` fallback 索引。

**边界说明**：文档原先把 `flushConnectedPlayers` 也归到“广播”下，但该路径发的是每个玩家自己的 `WorldDelta/SelfDelta/PanelDelta/Notice` 个性化 envelope，不能用同一个 room 包群发；本轮只把真正同实例可共享的聊天广播接到 room。后续战斗/宗门/队伍等域广播应继续复用会话层 room API，而不是恢复全局 binding 扫描。

**反人类点**：socket.io 的 room 子系统就是为"按地图/实例广播"而生的。用 room 之后：
- "同实例广播" = O(1) 调用 `io.to(instanceId).emit(...)`，由 socket.io 内部按 room 索引直接发包给所有 join 的 socket
- 玩家进出实例 = `socket.join(instanceId)` / `socket.leave(instanceId)`，O(1)

实际实现绕开 room，**每条广播都从全局 binding 表里扫一遍**：

`network/world-client-event.service.ts:264 broadcastChat`：
```ts
for (const binding of this.worldSessionService.listBindings()) {
    const target = this.playerRuntimeService.getPlayer(binding.playerId);
    if (!target || target.instanceId !== player.instanceId) continue;
    const socket = this.worldSessionService.getSocketByPlayerId(binding.playerId);
    if (socket) this.emitChatMessage(socket, chatMsg);
}
```

`network/world-sync.service.ts:55 flushConnectedPlayers`：
```ts
for (const binding of this.worldSessionService.listBindings()) {
    const socket = this.worldSessionService.getSocketByPlayerId(binding.playerId);
    const view = this.worldRuntimeService.getPlayerView(binding.playerId);
    if (!socket || !view) continue;
    this.syncDeltaForPlayer(binding.playerId, binding.sessionId, socket, view);
}
```

**量化**（5000 在线 / 10000 实例 / 1Hz）：
- 同地图 5 个人聊一句话 → 5000 次 binding 扫描 + 5000 次 hash get + 5000 次实例 ID 字符串比较 + 5000 次 socket 查找，最后只发给 5 个人
- 每帧 `flushConnectedPlayers` → 5000 次 binding 扫描 + 5000 次 view 查找 + 5000 次 syncDeltaForPlayer
- 任何"同实例广播"都退化成 O(总在线)，与"同实例只有 N 个玩家"的预期成本严重不匹配

**应有设计**：按 `instanceId` 分 room、按战斗 `combatId` 分 room、按 `sectId` 分 room。所有"按域广播"统一走 `io.to(room).emit(...)`。binding 索引保留只用于"按 playerId 单播"。

### N2. 一次"建议点赞"会重写整张 SQL 表 + 全服推送全部建议

**现场**：

`runtime/suggestion/suggestion-runtime.service.ts:83 vote`：
```ts
suggestion.upvotes = [...suggestion.upvotes, playerId];
await this.persist();
```

`persistence/suggestion-persistence.service.ts:129 saveSuggestions`：
```ts
await client.query('BEGIN');
await client.query(`INSERT INTO ${SUGGESTION_STATE_TABLE} ... ON CONFLICT ...`);
await client.query(`DELETE FROM ${SUGGESTION_TABLE}`);                 // 整表清
for (const suggestion of normalized.suggestions) {                       // 串行 await
    await client.query(`INSERT INTO ${SUGGESTION_TABLE}(...) VALUES (...)`);
}
await client.query('COMMIT');
```

`network/world-client-event.service.ts:528 broadcastSuggestionUpdate`：
```ts
const suggestions = this.suggestionRuntimeService.getAll();   // 全部 cloneSuggestion + sort
for (const binding of this.worldSessionService.listBindings()) {
    const socket = this.worldSessionService.getSocketByPlayerId(binding.playerId);
    if (socket) this.emitSuggestionUpdate(socket, suggestions);
}
```

**反人类点**：一次点赞动作的副作用：
1. 整张 `server_suggestion` 表 DELETE + N 行串行 INSERT 重写
2. `getAll()` 把每条建议 `cloneSuggestion` 一遍，再 sort 一遍，得到完整列表
3. 完整列表广播给**所有在线玩家**（也是按 N1 全表扫 binding）
4. 每个玩家本地 UI 收到完整列表，按现状要做整片重渲染

**量化**：1000 条建议、5000 玩家、人均点 5 次 / 天：
- SQL 写：25000 次 × 1000 行 INSERT = 2500 万行 INSERT/天
- 网络出包：5000 玩家 × 25000 次 × 1000 条建议 payload = 1.25 亿条 payload/天
- 客户端：每个玩家 25000 次全列表重渲染

实际"点赞"语义只是 `+1 / 状态翻转`。理论代价应该是：
- SQL：1 次 `UPDATE server_suggestion SET upvotes = upvotes || $1 WHERE id = $2`
- 网络：N 个订阅者各收一条 `{ id, vote: 'up', playerId }` 的小 patch
- 客户端：单条 patch 应用

**应有设计**：
- 持久化层走 row-level UPDATE / `INSERT ... ON CONFLICT DO UPDATE`，禁止"DELETE 整表 + 全 INSERT"
- 协议层走 revision + patch（add / remove / vote-change），不再发完整列表
- 客户端按 patch 局部更新

### N3. `PlayerAttributesService.recalculate` 是重型全量结算函数，被挂在 1Hz tick 热路径上

**现场**：`runtime/player/player-runtime.service.ts:2906 advanceSinglePlayerTick`，每个在线玩家每 tick 调用：
- `tickTemporaryBuffs` → 若变化则 `recalculate(player)`
- `recoverPlayerVitals` → 标 vitals dirty
- `shouldResumeIdleCultivation` → 触发 `recalculate(player)`
- `advanceCultivation` → 修改技能后 `applyProgressionResult` → 再可能 `recalculate`

`runtime/player/player-attributes.service.ts:28 recalculate` 一次干这些事：
- `player.techniques.techniques.map(toTechniqueState)` 全量重算
- `equipment.slots` 全部循环、每件装备 `applyEquipmentAttributeEffectivenessToItemStack` 派生新对象
- `getActiveBuffs(player.buffs.buffs)` 至少两轮循环（attr + numericStats）
- `cloneAttributes` / `cloneNumericStats` / `createAttributePercentBonusAccumulator` 各 N 次
- 重新组装 `numericStats / ratioDivisors / specialStats`

**反人类点**：修仙挂机 MUD 的常态是"长时间挂机修炼"。这种期间 buff 没变、装备没变、功法没变，理应**直接复用上一帧快照**或按 dirty 标记增量更新。但当前是每秒整体重算。

**量化**：5000 玩家在线，假设 70% 处于 buff/cultivation 持续生效状态：
- 每秒 3500 次完整 `recalculate`
- 单次涉及 N=20 buff × 8 装备 × 5 技能 × 6 属性键 的全表遍历 + 多次浅克隆
- CPU 直接被吃满

**应有设计**：
- 把"会影响最终属性"的来源拆成槽：技能 / 装备 / buff / 根骨 / 功法
- 任何来源变化时只标对应槽 dirty
- recalculate 时只重算 dirty 槽 + 合成结果
- 没有 dirty 槽就直接返回，零成本

### N4. tick 主循环是单线程串行 await，10000 实例目标在结构上不成立

**现场**：`runtime/world/world-runtime-instance-tick-orchestration.service.ts:21 advanceFrame`：

```ts
for (const { instance, steps, speed } of instanceStepPlans) {        // 实例间串行
    for (let index = 0; index < steps; index += 1) {                 // 步数串行
        ...
        instance.tickOnce();
        instance.advanceTileResourceFlow();
        deps.worldRuntimeFormationService.advanceInstanceFormations(instance, ...);
        instance.advanceTemporaryTiles(...);
        instance.advanceTileRecovery(...);
        for (const action of result.monsterActions) deps.applyMonsterAction(action);
        ...
        if (typeof deps.worldRuntimePlayerSkillDispatchService?.resolvePendingPlayerSkillCast === 'function') {
            for (const playerId of currentPlayerIds) {
                await deps.worldRuntimePlayerSkillDispatchService.resolvePendingPlayerSkillCast(playerId, deps);
            }
        }
        await deps.worldRuntimeCraftTickService.advanceCraftJobs(currentPlayerIds, deps);
    }
}
```

**反人类点**：
- 整个世界 tick = 一个大 for 循环
- AGENTS.md §3 目标 10000 个 map 实例，**全部跑在 1 个 Node.js event loop 的同一个 await 链里**
- `await resolvePendingPlayerSkillCast` 在循环最深层：任意玩家技能解析慢一拍，**后面 9999 个实例排队等**
- `await dispatchPendingCommands()` 一次性把所有玩家 pending command 串行处理完
- `worker_threads` / `cluster` / 按地图分片这些 MMO 标配，整个生产代码 grep 不到使用

**量化**：按 8 核单进程估算：
- 单实例每 tick 平均成本（怪物推进 + 阵法 + 风水 + 资源流 + 临时地块）保守 0.1ms
- 10000 实例 × 0.1ms = 1000ms = 整整 1 秒，已经 100% 占用一个核
- 加上 await 网络/DB（pending command、craft tick）的等待，单帧远超 1s
- 1Hz tick 直接破窗，会被 N9 的 setInterval 静默吃帧

**应有设计**：
- 实例分片到 worker_threads（按 instanceId 哈希分桶，每桶一个 worker）
- 主线程只做编排（pending command 投递 / 玩家路由 / AOI 出包）
- await IO 必须移出 tick 内层循环，要么排队进 outbox，要么用专用 worker
- 至少要把"所有 await"挪出实例-步数双层循环

### N5. ✅ 已修复（55590049 + 本轮） · 玩家面板缓存"双层全克隆"

**现场**：

层 1：`runtime/instance/map-instance.runtime.ts:2089 buildPlayerView` → `playerViewCacheByPlayerId`
- 缓存内容：`view`（含 `visiblePlayers / localMonsters / localNpcs / localPortals / localLandmarks / localSafeZones / localContainers / localGroundPiles / localBuildings`）
- 用户已经发现这一层

层 2：`network/world-projector.service.ts:48 cacheByPlayerId` → `captureProjectorState(view, player) = combineProjectorState(captureWorldState(view), capturePlayerState(player))`

`world-projector.helpers.ts:518 capturePlayerState`：
```ts
return {
    selfRevision: player.selfRevision,
    self: { ..., wallet: cloneWalletState(player.wallet) },
    panel: {
        inventory: captureInventoryPanelSlice(player),    // items.map(entry => ({ ...entry }))
        equipment: captureEquipmentPanelSlice(player),    // slots.map(...) + item ? { ...item } : null
        technique: captureTechniquePanelSlice(player),    // techniques.map(cloneTechniqueEntry)
        attr: captureAttrPanelSlice(player),              // bonuses + finalAttrs + numericStats + specialStats 全 clone
        action: captureActionPanelSlice(player),          // actions.map(...) + autoUsePills + combatTargetingRules clone
        buff: captureBuffPanelSlice(player),              // buffs.revision + projectVisiblePlayerBuffs
    },
};
```

**反人类点**：同一个玩家的状态，在两个独立缓存层里**各克隆一份完整副本**，互不复用。包括：

- 视野信息（怪物/地形/玩家/portal/建筑）层 1 一份、层 2 一份
- 玩家自身面板（背包/装备/功法/属性 bonuses/autoUsePills/buff）每帧给每个在线玩家深克隆

`createDeltaEnvelope` 每帧每玩家执行一次：
- `captureProjectorState` 重新构造一份新的 `previous`（即使 `bonusesChanged === false` 也会先 `nextBonuses = buildAttrBonuses(player)`，已被 `内存问题OOM现场报告` 4.1 记录）
- 老的 ProjectorState 等到下一次 GC 才回收

**量化**：5000 玩家在线、每玩家 panel 平均 50KB（含 bonuses + 装备 attrs + numericStats）：
- 单层缓存稳态 250MB
- 双层 ≈ 500MB
- 加上每帧的临时克隆 + GC 不及时晋升老年代，老年代爆破

**应有设计**：
- 层 1（buildPlayerView）只输出 view，不缓存视野结构本体；视野按"差量条目"缓存
- 层 2（projector）只缓存 revision + 上一次"已发出"的字段哈希，不缓存完整 panel；diff 时按字段读 player 真源
- 玩家自身字段不需要克隆隔离 → 直接读 `player.attr.finalAttrs` 等真源对比 revision 即可

**本轮修复**：`ProjectorState` 删除常驻完整 `panel`，改为 `panelCursor`（各 slice revision + attr/action/buff signature）；no-op delta 只复用 cursor，不再保留第二份背包/装备/功法/属性/动作/buff 克隆。面板 slice 发生变化时，projector 从玩家真源构造一次性 full slice delta 发送给客户端，但不写回完整 panel cache。`WorldSyncPlayerStateService` 的旧 projector panel cache 复用路径也已移除，bootstrap 状态直接从玩家真源构造，避免留下旧兼容缓存入口。

### N6. ✅ 部分修复（本轮） · "DELETE 全实例 + INSERT 全实例" 的写法蔓延整个持久化层

**现场**：

`persistence/instance-domain-persistence.service.ts:180`：
```ts
await client.query(`DELETE FROM ${INSTANCE_BUILDING_CELL_TABLE} WHERE instance_id = $1`, [normalizedInstanceId]);
await client.query(`DELETE FROM ${INSTANCE_BUILDING_STATE_TABLE} WHERE instance_id = $1`, [normalizedInstanceId]);
await client.query(`DELETE FROM ${INSTANCE_ROOM_CELL_TABLE} WHERE instance_id = $1`, [normalizedInstanceId]);
await client.query(`DELETE FROM ${INSTANCE_ROOM_STATE_TABLE} WHERE instance_id = $1`, [normalizedInstanceId]);
await client.query(`DELETE FROM ${INSTANCE_FENGSHUI_STATE_TABLE} WHERE instance_id = $1`, [normalizedInstanceId]);
// 然后 jsonb_to_recordset 全量 INSERT
```

类似模式：
- `mail-persistence.service.ts:231 DELETE FROM player_mail_attachment WHERE player_id=$1` + 全量 INSERT
- `market-persistence.service.ts:357 DELETE FROM player_market_storage_item WHERE player_id IN(...)` + 全量 INSERT
- `durable-operation.service.ts:3316 DELETE FROM player_inventory_item WHERE player_id=$1` + 全量 INSERT
- `durable-operation.service.ts:3450 DELETE FROM player_equipment_slot WHERE player_id=$1` + 全量 INSERT

**反人类点**：任何一个实例 / 玩家只要某域 dirty 一行，就把该域的**所有行先全删后全插**，不区分哪一行真正变了。这是把 PostgreSQL 当 KV 文件用。

**量化**：10000 实例，平均每实例 50 个 building / 200 个 building cell / 30 个 room / 800 个 room cell。某玩家造一面墙：
- 5 张表整体清+重写
- 单次写约 1100 行 = 1100 个 dead tuple
- VACUUM 压力按 dirty 频率 × 1100 累积
- bloat 几小时就把表撑爆，然后 autovacuum 抢锁阻塞写入路径

**应有设计**：
- 单行变更走 `INSERT ... ON CONFLICT DO UPDATE`
- 批量变更走 `UPSERT FROM jsonb_to_recordset` + 单独的 `DELETE WHERE id NOT IN (...)`，不能整域 DELETE
- "DELETE 整域 + INSERT 整域"应作为禁用模式列入 `audit:boundaries` 黑名单

### N41. `OutboxDispatcherRuntimeService` 在主进程 `setInterval(250 ms)` 轮询 `outbox_event`，无 pg_notify、无 Redis Stream
