## 1. 当前结论

- 当前最该优先处理的问题，不是“老审计 73 项平均推进”，而是三条主线：
  - 安全与鉴权里仍有一批确定未处理的高优先级项。
  - 服务端 tick 广播链路仍有一批已命中实时热点的性能债。
  - 客户端运行时、共享编解码层和本地缓存层还有一批会持续放大成本的偷懒实现。
- 当前项目最重的性能债，不是缺少某个复杂算法，而是热路径里长期混用了：
  - `JSON.stringify` 深比较
  - `JSON.parse(JSON.stringify(...))` 深拷贝
  - 每 tick / 每观察者全量重建
  - O(n²) 线性查找 + 重复签名计算
  - Protobuf 外壳里继续塞 JSON 字符串

### 1.1 数据影响判定口径

- `否`
  - 修复只改运行时逻辑、校验、缓存、广播或客户端行为，不改现有 PostgreSQL 真数据结构，也不要求回写旧存档。
- `仅在线态`
  - 修复只影响 Redis、在线缓存、断线保留或当前连接行为，不影响 PostgreSQL 正式存档。
- `需迁移`
  - 修复本身大概率需要改数据库 schema、认证真源或历史数据兼容策略，落地前应单独做 migration 方案。
- `需重审`
  - 当前信息不足，不能直接判断是否会碰现有数据，需要按最终实现方案再确认。

## 4. 当前服务端性能热点

### 4.1 已知热点样本

以下样本来自当前保留下来的服务端 CPU 观察结论，重点不是具体数值永远不变，而是热点顺序已经比较稳定：

| 分项 | 占比特点 | 当前判断 |
| --- | --- | --- |
| 动作重建: 场景动作收集 / 核心构建 | 高频 + 累计占比高 | 仍是第一优先级 |
| 状态同步: 功法面板 / 属性面板 | 高频 + 结构复制重 | 属于增量同步缓存层问题 |
| 实体 Patch: 扫描比较 / 快照复制 | 高频 + 观察者乘法放大 | 仍未版本化 |
| 地块 Patch: 客户端视图转换 / 扫描比较 | 地块链路残余热点 | 已优化一部分，但未收尾 |
| 广播: Socket 发送 | 总耗时高 | 与重复构建、编码统计耦合相关 |
| 网络编解码与收发 | 稳定前列 | Protobuf 外壳内塞 JSON 是直接原因之一 |

### 4.2 当前已经收敛、可作为前提的实现事实

- 普通地图地块 Patch 已接入 `tilePatchRevision + dirtyTileKeys`，不再每 tick 全视野 diff。
- 地块 Patch 快照复制已经不再整片重建缓存，改成只更新变化格子。
- 动作重建第二轮补跑已经缩小到真正受影响玩家。
- 修炼初始化校正已从“每 tick 无条件执行”收敛为运行时一次性初始化。
- Bot 已从玩家动作/状态主链路里剥离，不再污染真人热点。

### 4.3 当前剩余的大头

1. 真人玩家 `syncActions` 仍然接近每 tick 全量重建。
2. 实体 Patch 仍靠结构比较和快照复制。
3. AOI 可见性仍缺少稳定缓存或增量化。
4. Socket 发送、编码统计和真实发包仍耦合在一起。

## 5. 当前全项目性能债总表

### 5.1 P0：已命中实时热点的性能债

| 项目 | 当前状态 | 数据影响 | 问题本质 | 下一步 |
| --- | --- | --- | --- | --- |
| 动作重建仍按真人玩家每 tick 全量执行 | 未处理 | 否 | 本该脏标驱动的逻辑仍在全量轮询 | 为动作系统拆 `actionsDirty / contextDirty / cooldownDirty` |
| 增量同步缓存层大量依赖 JSON 深比较 / 深拷贝 | 未处理 | 否 | 通用 diff/clone 工具被用进了每 tick 热路径 | 改成 revision / signature / 轻量快照 |
| 地块链路仍先做整片客户端视图转换 | 未处理 | 否 | 最终未必发包，仍先整片 `map -> map` | 延后转换时机，优先只转脏格 |
| 广播链路按“观察者 x 目标”重复构建实体 | 未处理 | 否 | 同一目标在同 tick 被重复构建多次 | 拆公共快照和观察者相关快照 |
| Protobuf 内部继续塞 JSON 字符串 | 未处理 | 否 | 复杂字段的序列化成本并没有真正下降 | 先把高频字段改成真正的 message |

### 5.2 P1：规模一上来会炸的服务端问题

| 项目 | 当前状态 | 数据影响 | 问题本质 | 下一步 |
| --- | --- | --- | --- | --- |
| 怪物仇恨刷新是“怪物数 x 玩家数”扫描 | 未处理 | 否 | 缺少附近玩家索引 | 先做地图分桶 / AOI 邻近索引 |
| 寻路调度器每次挑任务都过滤并排序全队列 | 未处理 | 否 | 当前是简单队列，不是高频调度结构 | 换按地图分组的优先队列 |
| 物品堆叠签名在多处线性查找里被重复计算 | 未处理 | 否 | O(n² * 序列化) | 运行时缓存 stack key，改 `Map` 合并 |
| 装备被动效果同步里有 JSON 比较和重复时间状态构建 | 未处理 | 否 | 条件判断和 bonus 比较都偏重 | 按装备 revision / 时间 phase 缓存 |
| Socket 统计逻辑直接污染发送热路径 | 未处理 | 否 | 同步编码、估包长、发包耦合 | 拆成独立统计指标 |

### 5.3 P2：客户端运行时和本地缓存层性能债

| 项目 | 当前状态 | 数据影响 | 问题本质 | 下一步 |
| --- | --- | --- | --- | --- |
| `MapStore` tick 合并大量用 JSON 深拷贝 | 未处理 | 否 | 服务端做增量，客户端又把成本花回去 | 改成按 patch 原位合并或轻量复制 |
| `main.ts` 内属性 / 技能 / 动作状态同步仍大量 clone | 未处理 | 否 | 单线程 UI 主链路不断做结构复制 | 按字段拆同步，减少整包 clone |
| `map-memory.ts` / `map-static-cache.ts` 整包持久化 | 未处理 | 否 | 本地缓存更新成本随规模线性上升 | 改成增量持久化和更轻量结构 |
| 客户端结构签名 / 本地缓存仍偏 JSON 化 | 未处理 | 否 | 持续吞帧预算和交互响应 | 用更窄的签名结构代替通用 JSON |

## 6. 当前建议执行顺序

1. 先完成真人玩家 `syncActions` 的真正脏标驱动。
2. 再做实体 Patch / 面板同步缓存的 revision 化。
3. 然后处理地块客户端视图转换和广播构建复用。
4. 再处理 Protobuf 内部 JSON 字段和网络发送链路拆分。
5. 最后再看怪物仇恨、寻路调度、客户端本地缓存这些第二梯队问题。

## 7. 观察与验证口径

- 每轮优化后先清空旧 CPU 统计，再看新样本。
- 固定看三种排序：
  - 总耗时
  - 次数
  - 单次
- 每次只做一类优化，避免多个收益来源缠在一起无法判断。
- 至少固定观察两个场景：
  - 单人站桩挂机
  - 多人同图移动与战斗
