# 半即时回合制与 AI 技能长期规划

状态：草案 v1

用途：

- 这份文档用于明确当前项目在“半即时回合制 MMO”方向上的长期演进路径
- 重点回答三个问题：
  - `参考/jiuzhou` 哪些设计可以吸收
  - 哪些设计不应照抄
  - 后续如何支持大量技能、AI 生成技能与随机词缀装备，而不把运行时压垮

## 1. 当前判断

当前项目的核心约束与 `jiuzhou` 不同：

- 当前项目是单服多地图、服务端权威、每张地图独立 tick 的半即时回合制 MMO
- `jiuzhou` 更偏角色面板驱动、房间/回合/事件驱动，而不是 AOI 世界态持续广播

这意味着：

- 不能直接照抄 `jiuzhou` 的整体实时架构
- 但可以大量吸收它在“内容建模、实例化、缓存、持久化颗粒度、异步生成管线”上的成熟做法

长期目标不是把本项目改造成另一个 `jiuzhou`，而是：

- 保留当前项目的地图/AOI/tick 权威模型
- 把“内容数量”与“运行时成本”彻底分离
- 让大量技能、大量装备变成内容问题，而不是热路径问题

## 2. 可以吸收什么

### 2.1 模板与实例分离

`jiuzhou` 最值得吸收的做法，不是玩法外观，而是数据分层：

- 装备：静态模板 + 实例 affix
- 技能：静态定义/生成定义 + 角色装配
- 邮件：正文与附件投影分离，列表分页，详情按需
- 角色属性：基础数据 + 现算 + 缓存，而不是持久化整份最终态

吸收到本项目后的对应原则：

- 大量内容只以“定义”存在，不以“运行时对象”存在
- 角色身上只保留“当前学习/装备/启用”的少量实例或引用
- 热路径只消费已经编译好的纯数据结构

### 2.2 生成走冷路径，不进 tick

`jiuzhou` 的 AI 生成功法采用 `job -> draft -> publish` 管线，这个思路应直接吸收：

- 生成请求进入异步任务
- 生成结果先落草稿
- 草稿通过校验后再发布
- 发布后才进入玩家可学习/可装配范围

对本项目来说，所有 AI 生成技能、AI 生成功法、未来词条池扩展，都必须遵守：

- 不在地图 tick 中生成
- 不在高频广播包中首次分发完整定义
- 不在运行时临时解析大 JSON

### 2.3 小表真源与窄缓存

当前项目已经开始把玩家真源从宽表拆向：

- `player_presence`
- `player_settings`
- `player_collections`

这个方向应继续推进，并进一步对齐 `jiuzhou` 的两个核心思想：

- 谁变了写谁，不做整档刷盘
- Redis 只保存实时态/投影态，不保存整份玩家镜像真源

## 3. 哪些不能照抄

### 3.1 不能照抄角色单播驱动的整体模型

本项目后续仍然要维护：

- 地图内占位
- AOI 可见性
- 地块变化
- 世界实体变化
- 服务端统一结算

所以不能把世界同步退化成“角色面板推送 + 局部房间事件”的模式。

### 3.2 不能让生成内容直接污染高频协议

`jiuzhou` 里很多内容更偏面板读取；而本项目如果把完整技能定义、完整装备词条详情直接塞进：

- `Tick`
- `TechniqueUpdate`
- `ActionsUpdate`
- 背包/装备高频同步

会立刻把 CPU 和带宽重新打爆。

所以本项目必须坚持：

- 高频层只发 id、状态位、cd、顺序、少量动态数字段
- 说明文本、完整效果、词条描述、长数组都走低频详情或首包静态

## 4. 长期架构目标

### 4.1 技能系统目标

目标不是“角色永久携带几十万个技能对象”，而是：

- 全服可以有很多技能定义
- 单个角色同时可用技能数量保持很小
- 战斗热路径只处理当前动作栏/自动战斗配置中的技能

建议最终分成四层：

1. `skill_def`
   静态技能真源

2. `generated_skill_def`
   AI 生成技能真源

3. `compiled_skill_def`
   编译后的运行时结构

4. `player_skill_loadout`
   玩家当前装配、优先级、开关、自动战斗策略

设计原则：

- `skill_def/generated_skill_def` 负责内容表达
- `compiled_skill_def` 负责战斗执行
- `player_skill_loadout` 负责角色投影

### 4.2 功法系统目标

功法不应再承担“直接把完整技能数组常驻同步给客户端”的职责，而应拆成：

- 功法定义
- 层级与解锁规则
- 当前层数
- 已解锁技能 id 集合
- 当前装配技能槽

这样功法可以很多，技能也可以很多，但角色运行时只关心：

- 已装备功法
- 当前层数
- 解锁到的技能 id
- 当前装配槽位

### 4.3 装备系统目标

如果后续引入洗词缀/随机词缀装备，建议目标结构为：

- `item_def`
  装备静态模板

- `affix_pool_def`
  词条池静态模板

- `item_instance`
  实例字段只保留：
  - `itemDefId`
  - `seed`
  - `quality`
  - `affixes`
  - `enhance/refine/socket`
  - `bind/owner/location`

要求：

- 词条重洗只更新实例行
- 装备属性由“模板 + 实例 + 强化/精炼/镶嵌”现算
- 不把最终展开后的大对象作为玩家长期真源缓存到各层

## 5. AI 技能生成规划

### 5.1 目标

未来允许系统生成大量技能，但要满足：

- 服务端能校验
- 客户端能展示
- 热路径能高效执行
- 内容生成失败或下线时不污染正式真源

### 5.2 生成管线

建议采用下面的状态机：

1. `pending`
   玩家发起生成请求，扣材料/研究点，创建任务

2. `generated_draft`
   模型返回结构化候选，写入草稿表

3. `validated`
   服务端完成结构校验、平衡校验、危险字段校验

4. `compiled`
   编译为运行时结构，写入 `compiled_skill_def`

5. `published`
   正式发布，进入玩家可学习/可掉落/可装配集合

6. `failed/refunded`
   生成失败或超时退款

### 5.3 生成结果约束

AI 返回的技能定义必须是结构化数据，而不是自由文本脚本。

建议延续当前项目已存在的公式思路：

- 使用结构化公式 AST
- 使用有限白名单的 `effect` 类型
- 使用有限白名单的 `targetMode`、`range`、`cooldown`、`cost`
- 禁止生成任意服务端脚本

必须校验的维度：

- 技能 id 唯一性
- 资源消耗上下界
- 冷却上下界
- 目标类型合法性
- effect 白名单合法性
- 数值预算不越界
- 与境界/功法分类的适配关系

### 5.4 发布后的使用方式

发布后的技能不直接塞进玩家角色大对象，而是：

- 玩家学习时，只记录 `skillId`
- 玩家装配时，只记录 `slot -> skillId`
- 战斗时，从 `compiled_skill_def` 读取执行结构

## 6. 大量技能的可支持性判断

结论：可以支持很多技能，但前提是运行时模型必须收窄。

### 6.1 可以支持很多技能的原因

本项目当前已有一个很好的基础：

- 技能公式是结构化 AST
- 不是运行时拼字符串脚本
- 不是 `eval`

这意味着：

- 技能定义可以继续增长
- 生成技能可以落到同一套结构
- 编译后可以稳定进入战斗链路

### 6.2 真正的上限不在“技能数量”，而在这些点

- 单角色当前可用技能数是否过大
- 高频包是否传完整技能对象
- tick 是否在反复深拷贝、深比较、临时解析定义
- 客户端是否反复重建整面板

只要把这四个点压住，全服技能数上万甚至更多都不是第一瓶颈。

## 7. 协议规划

### 7.1 高频层

高频层只允许出现：

- `skillId`
- `cooldownLeft`
- `enabled`
- `order`
- 少量资源变化
- 本 tick 实际触发的战斗表现

禁止混入：

- 完整技能说明
- 完整效果数组
- 完整功法层级数据
- 完整装备词缀描述

### 7.2 低频层

低频层负责：

- 技能详情
- 功法详情
- 装备详情
- 词缀详情
- 邮件正文与附件详情

建议补出明确分层：

- `Bootstrap`
- `MapStatic`
- `WorldDelta`
- `SelfDelta`
- `PanelDelta`
- `Detail`

### 7.3 编码层目标

当前协议里仍有把复杂对象 `JSON.stringify` 后塞进 protobuf 的热点，后续应逐步改为：

- 高频结构做真正 protobuf message
- 低频详情保留 JSON/按需请求
- 高频链路不再二次 JSON 化

## 8. 持久化规划

### 8.1 技能相关真源

建议长期拆分到这些表或等价结构：

- `player_skill_learned`
- `player_skill_loadout`
- `player_technique`
- `generated_skill_def`
- `compiled_skill_def`
- `skill_generation_job`

### 8.2 Redis 角色

Redis 只承担：

- 在线态
- 资源实时态
- 计算缓存
- 窄投影

不承担：

- 玩家完整正式真源
- 大量可恢复的长期配置

### 8.3 邮件目标

邮件后续应继续保持：

- 一行一封
- 分页列表
- 未读/未领计数缓存
- 正文按需详情
- 一键领取按批次/逐封推进，避免大事务长锁

## 9. 运行时性能规划

### 9.1 热路径红线

只要落在以下链路，就默认视为热路径：

- 地图 tick
- AOI 计算
- 世界广播
- 属性结算
- 战斗执行

这些链路禁止继续依赖：

- `JSON.stringify`
- `JSON.parse`
- 字符串签名比较
- 临时字符串坐标键
- 大对象深拷贝后再比较

### 9.2 未来优化方向

优先级从高到低：

1. 把 AOI 坐标键从字符串改为稳定整数 key/位图索引
2. 把高频协议里的 JSON 字符串字段彻底移出
3. 把世界同步和面板同步继续拆窄
4. 把角色计算缓存和资源缓存从整档缓存改成窄缓存
5. 需要时把重离线计算、重生成任务、重批处理移入 worker

## 10. 分阶段路线图

### 阶段一：先止血

目标：

- 降低当前 tick CPU
- 降低高频包体
- 降低玩家整档写库与整份 Redis 镜像

重点工作：

- 继续推进 `player_presence / player_settings / player_collections`
- 删减高频协议中的静态与长字段
- 逐步移除高频 protobuf 里的 JSON 字段
- 压缩 AOI 与广播链路中的深拷贝/深比较

### 阶段二：建立内容分层

目标：

- 把“技能定义很多”和“角色当前可用很少”彻底拆开

重点工作：

- 新增技能学习表、技能装配表
- 功法改为只投影解锁关系与装配关系
- 技能详情改按需请求

### 阶段三：接入生成管线

目标：

- 支持 AI 生成技能/功法草稿
- 发布前可校验、可退款、可审计

重点工作：

- 新增生成任务表
- 新增草稿/发布状态流转
- 新增编译表或编译缓存层
- 把生成任务放到冷路径异步执行

### 阶段四：扩展实例化装备

目标：

- 支持大量不同装备实例
- 支持洗词缀、锁词条、强化、精炼、镶嵌

重点工作：

- 装备模板/词条池/实例结构稳定化
- 实例属性改现算
- 背包/装备同步改为 id + 必要动态字段
- 装备详情改按需拉取

## 11. 成功标准

达到以下状态时，说明这条路线基本跑通：

- 全服技能定义数量增长，不明显拉高地图 tick CPU
- 单角色切换功法/技能装配，不触发整面板整包同步
- AI 生成技能失败不会污染正式技能真源
- 装备词缀洗炼只改实例，不触发整档刷盘
- 邮件可承载高数量存量，列表读取和一键操作仍保持分页/批量策略

## 12. 本轮之后的建议顺序

建议按这个顺序推进，而不是同时摊大饼：

1. 先继续压缩现有热路径与高频协议
2. 再把技能系统改成“定义/装配”两层
3. 再接 AI 生成草稿管线
4. 最后再做大规模随机词缀装备

原因：

- 当前最大的瓶颈仍是世界热路径与同步链路
- 若先上 AI 生成和大规模随机实例，只会把现有瓶颈放大
