# 战斗链路

本文同时描述“商业化目标链路”和当前 `packages/*` 生产主线的过渡态。

当前实现仍处于过渡态，但权威链路已经完成关键收敛：玩家普攻、玩家技能、怪物普攻、怪物技能的目标裁定和结果应用已经接入统一 `CombatAction` / `CombatOutcome` / 结果适配器；进入结算的成功目标由 `applyCombatOutcome(record: true)` 在应用权威状态后统一记录 outcome。仍保留旧服务编排、旧命中公式调用、旧 `WorldDelta.fx` / `S2C.Notice` 发包真源，避免一次性替换造成玩法漂移；战斗服务侧的表现和通知调用已先收敛到统一 presentation helper。

目标只有一个：所有战斗行为都进入同一条服务端权威主链路。玩家普攻、玩家技能、玩家吟唱、怪物普攻、怪物技能、怪物吟唱，都只是这条主链路的不同输入形态。玩家、怪物、地块、阵法、容器这些目标可以有不同的结果应用方式，但不能在命中结算之前拆成多套互不一致的战斗系统。

## 1. 核心原则

### 1.0 当前实现快照

当前已经落地的部分：

1. `packages/server/src/runtime/world/combat-action.types.ts` 定义了服务端内部 `CombatActor`、`CombatAction`、`CombatTarget`、`CombatOutcome`、`CombatRejectReason` 和 `CombatActionPhase`。
2. `WorldRuntimeCombatActionService` 已覆盖动作定义解析、目标收集、合法性校验、资源和冷却 dry-run、结构化拒绝、统一 outcome 记录、事件构建和内部运维查询接入点。
3. 玩家普攻和玩家技能入口已先包装为 `CombatAction`，生产目标裁定消费统一 action plan，结果应用经 `applyCombatOutcome()` 和目标适配器落到玩家、怪物、地块、阵法、容器等权威状态。
4. 怪物普攻和怪物技能已经把关键拒绝路径改成结构化诊断，怪物吟唱完成按预警格重新收集当前玩家目标，不再只依赖旧目标玩家 id；命中、闪避、0 伤害等成功结算目标会经统一结果应用入口记录 outcome。
5. 玩家和怪物 pending cast 已使用同形态内存结构保存 actor、action、anchor、warningCells、tick、资源和冷却快照；尚未写入 Redis 或数据库。
6. `world_delta_fx`、`notice`、`audit_internal`、`diagnostic_internal` 四层事件边界已经定义，内部审计和诊断不会进入普通客户端 S2C payload。
7. `packages/server/src/runtime/combat/combat-event-query.ts` 提供内部事件数组查询：最近审计、诊断聚合、怪物技能失败原因、战斗热力点。它不是数据库持久化查询。
8. `emitCombatPresentation()` 已包装玩家普攻、玩家技能、怪物普攻、怪物技能、吟唱开始、吟唱完成 label、预警格、临时地块技能攻击线和玩家战斗通知；底层仍进入现有 `CombatEffect` 队列和玩家 notice queue。
9. `packages/server/src/persistence/combat-audit-outbox.service.ts` 的 `CombatAuditOutboxService` 已把成功 `CombatAuditEvent` 接到异步持久化转出：同步链路只入内存队列，异步 flush 到 `outbox_event(topic=combat.audit.recorded)` 和 `asset_audit_log(asset_type=combat)`。

当前还没有完成的部分：

1. 旧生产服务仍承担命中公式调用、死亡后续编排和部分文案决策，尚未收敛成一个薄编排器。
2. 目标类型分支在生产服务中仍存在，用于调用击杀、扩张等旧后续副作用；权威结果写入已经收敛到结果适配器，表现/通知调用已经通过 presentation helper 包装。
3. 审计事件已有 outbox/asset audit 基础落库，但击杀、掉落、经验、死亡、资产变更等语义化审计分类和数据库复合查询还没完成。
4. pending cast 还没有 Redis/DB 恢复链，断线重连、服务重启、实例迁移策略仍待实现；当前生产策略是内存态和重启清空。
5. AOI/通知事件 sink 还没有替代旧 `WorldDelta.fx` / `S2C.Notice` 生产入口。
6. 已有 100 玩家普攻核心批处理、50 怪物技能核心批处理和 100 内部事件构建基准；真实 AOI 序列化广播和完整地图实例 tick 基准仍未完成。

### 1.1 一条主链路

所有战斗动作统一走：

1. 接收战斗意图。
2. 规范化为统一战斗动作。
3. 读取服务端权威快照。
4. 解析动作定义。
5. 解析锚点、范围和目标。
6. 做统一合法性判断。
7. 处理资源、冷却和吟唱。
8. 进入统一命中与效果结算。
9. 按目标类型应用结果。
10. 生成表现、通知、审计和诊断。

这条链路必须覆盖：

- 玩家普攻怪物。
- 玩家普攻玩家。
- 玩家普攻地块、阵法、容器。
- 玩家即时技能。
- 玩家吟唱技能。
- 怪物普攻玩家。
- 怪物即时技能。
- 怪物吟唱技能。

### 1.2 普攻是默认战斗动作

普攻不应拥有独立于技能之外的完整战斗系统。

普攻应被视为一个内置动作：

- 动作类型：`basic_attack`。
- 动作来源：系统内置，不来自技能书。
- 范围：默认近战范围，允许被武器、怪物配置或状态修正。
- 效果：一次基础伤害，允许带战斗经验、武器、属性、buff、普攻加成。
- 资源：通常不消耗灵气，也可以由配置定义固定消耗。
- 冷却：使用普攻冷却或动作间隔。

普攻和技能应共用：

- 目标解析。
- 同实例判断。
- 敌我关系判断。
- 范围判断。
- 视线判断。
- 命中、闪避、暴击、减伤。
- 伤害和 buff 效果结算。
- 战斗通知和表现口径。
- 诊断和审计口径。

### 1.3 分支只能晚分支

战斗链路允许分支，但分支位置必须清晰。

允许早期分支：

- 玩家输入、自动战斗、怪物 AI 如何生成战斗意图。
- 普攻、技能、道具、陷阱如何解析为动作定义。
- 即时动作和吟唱动作的时序不同。

不允许早期分支：

- 普攻打怪物一套合法性，技能打怪物另一套合法性。
- 玩家打玩家一套命中，怪物打玩家另一套命中。
- 怪物技能在进入结算前静默丢弃，而其他攻击有通知。
- 地块、阵法、容器在目标解析前就绕出战斗主链路。

必须晚期分支：

- 玩家扣血写玩家运行态。
- 怪物扣血写地图实例怪物运行态。
- 地块扣耐久写地图实例地块状态。
- 阵法扣灵力写阵法运行态。
- 容器扣次数或刷新状态写容器运行态。

也就是说，分支应该发生在“结果应用”阶段，而不是“能不能打、怎么命中、有没有通知”阶段。

## 2. 统一模型

### 2.1 战斗发起者

发起者统一抽象为 `CombatActor`。

字段含义：

- 发起者类型：玩家、怪物、机关、地形、系统。
- 发起者 id：玩家 id 或怪物 runtime id。
- 所在实例：`instanceId`。
- 当前坐标：用于范围、视线、朝向和表现。
- 战斗属性：物攻、法攻、命中、闪避、暴击、减伤、元素等。
- 成长属性：等级、境界、战斗经验、技能等级。
- 资源状态：血量、灵气、冷却、临时 buff。
- 权限状态：是否死亡、是否可攻击、是否被控制、是否可行动。

玩家和怪物的原始状态来源可以不同，但进入战斗结算前必须转为同一类战斗快照。

### 2.2 战斗动作

所有攻击、技能和特殊战斗行为统一抽象为 `CombatAction`。

字段含义：

- 动作 id：普攻是内置 id，技能是技能 id。
- 动作类型：普攻、技能、道具、地形触发、系统触发。
- 动作来源：玩家输入、自动战斗、怪物 AI、陷阱、buff 触发。
- 目标引用：玩家 id、怪物 id、地块坐标、阵法 id、容器 id 或空目标。
- 锚点：范围技能从哪个格子展开。
- 形状：单体、圆形、方形、线形、棋盘形等。
- 最大目标数：本次最多影响多少目标。
- 效果列表：伤害、buff、治疗、净化、击退、地形变化、资源扣减等。
- 时序：即时、吟唱开始、吟唱完成、持续引导、延迟触发。
- 资源和冷却策略：开始时扣、完成时扣、命中后扣、取消返还、不返还。

普攻和技能的区别只在动作定义，不在主链路。

### 2.3 战斗目标

目标统一抽象为 `CombatTarget`。

目标类型：

- 玩家。
- 怪物。
- 地块。
- 阵法。
- 容器。
- 自身。
- 空地。

目标字段：

- 目标类型。
- 目标 id 或坐标。
- 所在实例。
- 当前坐标。
- 是否可被该动作影响。
- 用于结算的防御属性或耐久属性。
- 用于应用结果的状态写入适配器。

目标可以来自明确选择，也可以来自范围扫描。最终进入结算前都必须变成统一目标列表。

### 2.4 战斗结果

每个目标都会产生 `CombatOutcome`。

结果内容：

- 本目标是否通过合法性判断。
- 如果未通过，原因是什么。
- 是否命中。
- 是否闪避、格挡、抵抗、免疫。
- 原始伤害。
- 实际伤害。
- 元素和伤害类型。
- 应用的 buff、治疗、击退、地形变化等效果。
- 是否死亡或被破坏。
- 是否需要触发反击、仇恨、掉落、奖励、任务进度。
- 应发给哪些玩家的通知。
- 应广播到哪些 AOI 的表现事件。
- 需要标记哪些运行态脏域。

只要进入命中/效果结算，就必须产出结果。结果可以是闪避、免疫、未造成伤害，但不能静默消失。

## 3. 完整主链路

### 3.1 接收战斗意图

战斗意图来自多个入口：

- 玩家点击普攻。
- 玩家释放技能。
- 玩家自动战斗。
- 怪物 AI 决定普攻。
- 怪物 AI 决定释放技能。
- buff、陷阱、地形或系统触发战斗动作。

入口只负责收集意图，不直接扣血、不直接发奖励、不直接推断最终伤害。

入口输出统一的战斗意图：

- 谁发起。
- 想执行什么动作。
- 指向哪个目标或锚点。
- 来源是什么。
- 是否由自动逻辑产生。

### 3.2 规范化为战斗动作

服务端把意图规范化为 `CombatAction`。

处理内容：

1. 把普攻解析为内置动作定义。
2. 把技能 id 解析为技能动作定义。
3. 把怪物技能解析为和玩家技能同形态的动作定义。
4. 把目标 id 或地块坐标解析为动作目标引用。
5. 补齐范围、形状、效果、资源、冷却、吟唱等字段。
6. 标记动作来源：玩家、怪物、自动战斗、系统。

这一阶段只做动作定义归一，不做伤害结算。

### 3.3 读取权威快照

服务端读取发起者和目标相关的权威状态。

必须读取：

- 发起者是否仍存在。
- 发起者是否仍在该实例。
- 发起者是否还活着。
- 发起者当前坐标。
- 发起者当前属性、buff、资源、冷却。
- 地图实例状态。
- 目标引用对应的当前对象。

状态真源分工：

- 玩家血量、属性、buff、资源来自玩家运行态。
- 怪物血量、坐标、buff、资源来自地图实例怪物运行态。
- 地图格子、占位、地形、容器来自地图实例。
- 阵法状态来自阵法运行态。
- 玩家在哪个实例来自玩家位置服务。

同一类问题只能有一个裁定真源：

- 玩家在哪个实例：位置服务。
- 玩家站在哪个格子：地图实例。
- 玩家有多少血和属性：玩家运行态。
- 怪物站在哪个格子和有多少血：地图实例怪物运行态。

### 3.4 解析锚点和影响范围

服务端决定动作从哪里展开。

锚点来源：

- 明确目标的当前坐标。
- 客户端选择的地块坐标。
- 吟唱开始时保存的锚点。
- 自身坐标。
- 怪物 AI 选定的目标坐标。

范围解析：

1. 根据动作定义读取范围、形状、半径、宽度、高度、最大目标数。
2. 根据发起者位置和锚点计算影响格。
3. 对线形、范围形、棋盘形等统一走同一个几何计算。
4. 输出本次动作的候选格子列表。

客户端可以显示预警，但最终影响格必须由服务端计算。

### 3.5 收集候选目标

服务端从候选格子和目标引用中收集目标。

收集规则：

- 明确目标优先进入候选列表，但仍需通过合法性判断。
- 范围技能从地图实例格子占位读取当前站在格子里的玩家。
- 怪物从地图实例怪物索引读取。
- 地块从地图实例读取。
- 阵法从阵法运行态读取。
- 容器从地图实例或容器运行态读取。
- 自身目标直接引用发起者。

收集结果是统一的 `CombatTarget[]`，不允许在这里就直接扣血或直接发通知。

### 3.6 统一合法性判断

每个候选目标都要走同一套合法性管线。

判断内容：

1. 发起者是否仍可行动。
2. 动作是否允许当前发起者使用。
3. 发起者和目标是否在同一实例。
4. 目标是否存在。
5. 目标是否仍可被攻击或影响。
6. 目标是否和发起者满足敌我关系。
7. 目标是否在动作范围内。
8. 目标是否满足视线或阻挡规则。
9. 地图是否允许这种战斗行为。
10. 动作是否允许影响该目标类型。
11. 资源和冷却是否满足当前阶段要求。

判断结果不能静默返回。

每个被拒绝目标都必须有机器可读原因，例如：

- 发起者不存在。
- 发起者死亡。
- 目标不存在。
- 目标死亡。
- 不在同一实例。
- 超出范围。
- 视线阻挡。
- 非敌对目标。
- 地图不允许 PVP。
- 地图不允许地形伤害。
- 动作不能影响该目标类型。
- 资源不足。
- 冷却未完成。
- 吟唱已经取消。

玩家不一定看到所有拒绝原因，但服务端诊断必须能看到。

### 3.7 资源、冷却和吟唱

动作时序统一处理。

即时动作：

1. 通过合法性判断。
2. 按动作策略扣资源。
3. 启动冷却。
4. 立即进入效果结算。

吟唱开始：

1. 通过开始阶段合法性判断。
2. 按动作策略预扣或冻结资源。
3. 按动作策略启动或预留冷却。
4. 保存 pending cast。
5. 保存发起者、动作、锚点、预警格、目标引用、开始 tick、剩余 tick。
6. 推送吟唱动作和预警表现。
7. 不造成伤害。

吟唱完成：

1. 从 pending cast 恢复统一战斗动作。
2. 使用保存的锚点和预警格进入同一条主链路。
3. 重新读取发起者和目标权威状态。
4. 重新收集当前有效目标。
5. 按完成阶段合法性判断过滤目标。
6. 不重复扣已经在开始阶段扣过的资源。
7. 不重复启动已经在开始阶段启动过的冷却。
8. 进入效果结算。

吟唱取消：

- 必须记录取消原因。
- 根据动作策略决定是否返还资源、是否保留冷却。
- 必须清理 pending cast。
- 必须产生诊断事件。

### 3.8 统一命中和效果结算

所有能造成战斗结果的动作都进入统一结算。

输入：

- 发起者战斗快照。
- 目标战斗快照。
- 动作定义。
- 目标数量。
- 距离。
- 当前 tick。
- 地图和环境修正。

结算内容：

1. 计算基础伤害或基础效果值。
2. 应用发起者属性、境界、等级、战斗经验、技能等级。
3. 应用目标防御、抗性、闪避、格挡、免疫、减伤。
4. 应用元素、伤害类型、暴击、增伤、易伤。
5. 解析命中、闪避、抵抗、免疫、未造成伤害。
6. 解析 buff、治疗、净化、击退、地形变化等非伤害效果。
7. 生成每个目标的 `CombatOutcome`。

普攻只是动作效果比较简单，不应绕过这一步。

怪物攻击玩家也不应有单独的命中公式。怪物只是在发起者快照里提供怪物属性。

### 3.9 按目标类型应用结果

只有到这里才允许按目标类型分支。

玩家目标：

1. 对玩家运行态扣血、回血或应用 buff。
2. 记录活动，打断修炼或生产。
3. 激活自动反击或仇恨。
4. 如果死亡，进入玩家死亡、复活、离线日志等流程。
5. 标记玩家状态脏域。

怪物目标：

1. 对地图实例怪物运行态扣血、回血或应用 buff。
2. 更新怪物仇恨。
3. 如果死亡，进入击杀、掉落、奖励、任务进度流程。
4. 标记怪物运行态脏域。

地块目标：

1. 对地图实例地块耐久或临时地块状态应用结果。
2. 处理地块破坏、恢复、掉落或阻挡变化。
3. 标记地图地块脏域。

阵法目标：

1. 对阵法灵力、阵眼状态或边界状态应用结果。
2. 处理阵法失效、地形稳定、保护范围变化。
3. 标记阵法和地图相关脏域。

容器目标：

1. 对容器次数、耐久或刷新状态应用结果。
2. 处理采集掉落、空容器、刷新倒计时。
3. 标记容器和地图相关脏域。

自身目标：

1. 对发起者自己应用治疗、buff、净化或状态变化。
2. 标记发起者对应脏域。

这一阶段可以分支，因为不同目标的状态真源不同；但分支只负责“写哪里、触发什么副作用”，不负责重新定义命中规则。

### 3.10 通知、表现、审计和诊断

所有动作最终都产出统一事件。

表现事件：

- 动作文字。
- 攻击轨迹。
- 预警格。
- 伤害浮字。
- 闪避、格挡、免疫、抵抗浮字。
- 死亡或破坏表现。

玩家通知：

- 攻击者通知。
- 被攻击者通知。
- 旁观或队友通知，按需求控制。

审计事件：

- 发起者。
- 动作。
- 目标。
- 结算结果。
- 资源消耗。
- 死亡、掉落、奖励。
- 异常拒绝原因。

诊断事件：

- action 没产出。
- action 被拒绝。
- 无有效目标。
- 状态真源不一致。
- 视线阻挡。
- 实例不一致。
- pending cast 丢失或过期。

强制要求：

- 进入结算后，即使命中结果是闪避、免疫、抵抗、未造成伤害，也必须有统一结果事件。
- 没有玩家战斗通知，只能发生在动作未进入结算时，并且服务端必须有机器可读原因。
- 不允许无原因 `return` 吞掉战斗动作。

## 4. 吟唱动作的商业级规则

吟唱技能是主链路的时序分支，不是另一套技能系统。

### 4.1 pending cast 保存内容

pending cast 必须保存足够恢复为统一战斗动作的信息：

- 发起者类型和 id。
- 实例 id。
- 动作 id。
- 动作来源。
- 开始 tick。
- 剩余 tick。
- 锚点。
- 预警格。
- 原始目标引用。
- 资源消耗状态。
- 冷却状态。
- 取消策略。
- 动作定义版本或配置 revision。

这样吟唱完成时，不需要猜测当初要做什么，只需要把 pending cast 转回 `CombatAction` 并继续走主链路。

### 4.2 开始阶段和完成阶段分开判断

开始阶段判断：

- 发起者能否开始施法。
- 技能是否存在。
- 资源是否够。
- 冷却是否好。
- 初始锚点是否在范围内。
- 地图是否允许该动作。

完成阶段判断：

- 发起者是否还活着。
- 发起者是否还在同一实例。
- pending cast 是否仍有效。
- 锚点和预警格是否仍可结算。
- 预警格内当前有哪些有效目标。
- 每个目标是否仍满足同实例、可被影响、视线等完成阶段条件。

完成阶段不应重新判断“是否拥有技能”这类会导致历史动作无法解释的冷路径状态，除非配置明确要求。配置版本变化应通过 action definition revision 处理。

### 4.3 合法落空和异常落空

合法落空：

- 目标离开预警格。
- 目标离开实例。
- 发起者死亡。
- 发起者被打断。
- 视线在完成时被阻挡。
- 目标变为不可攻击。

异常落空：

- 预警格显示玩家在格子里，但地图实例占位查不到玩家。
- 地图实例认为玩家在格子里，位置服务却认为玩家不在实例。
- 玩家运行态坐标和地图实例坐标不同。
- pending cast 完成时 action 无原因丢失。
- 技能配置能显示预警，却无法恢复结算动作。

合法落空可以没有受击通知；异常落空必须有服务端诊断。

## 5. 怪物 AI 与主链路边界

怪物 AI 不负责战斗结算。

怪物 AI 只负责：

- 找目标。
- 决定移动还是攻击。
- 选择普攻或技能。
- 选择锚点。
- 生成战斗意图。
- 创建或推进 pending cast。

怪物 AI 不负责：

- 扣玩家血。
- 判断最终命中。
- 生成玩家战斗通知。
- 处理玩家死亡。
- 应用 buff 到玩家。
- 发放奖励。

怪物发起的普攻和技能都必须转成统一 `CombatAction`，然后进入主链路。

这样怪物普攻、怪物技能、玩家普攻、玩家技能使用同一套合法性、命中、通知、诊断规则。

## 6. 目标类型分支边界

目标类型分支必须只处理状态写入和副作用。

### 6.1 玩家目标

玩家目标适配器负责：

- 写玩家血量、灵气、buff。
- 写玩家活动时间。
- 处理修炼、生产、自动反击。
- 处理死亡和复活。
- 标记玩家同步脏域。

不负责：

- 决定动作是否命中。
- 决定敌我关系。
- 决定技能范围。
- 决定是否发战斗结果事件。

### 6.2 怪物目标

怪物目标适配器负责：

- 写怪物血量、灵气、buff。
- 更新怪物仇恨。
- 处理怪物死亡、刷新、掉落、奖励。
- 标记地图实例怪物脏域。

不负责重新定义命中公式。

### 6.3 地块、阵法、容器目标

这些目标适配器负责各自真源写入：

- 地块：耐久、破坏、临时地形、恢复。
- 阵法：灵力、阵眼、边界、稳定效果。
- 容器：次数、掉落、空态、刷新。

它们可以有不同的“可被影响规则”，但这些规则必须注册到统一合法性管线中，而不是绕出主链路。

## 7. 失败处理口径

战斗失败必须分层。

### 7.1 意图拒绝

发生在动作还没进入主链路前。

例子：

- 玩家未登录。
- 玩家不存在。
- 客户端参数格式错误。
- 技能 id 非法。

处理方式：

- 返回操作错误。
- 不生成战斗表现。
- 记录必要诊断。

### 7.2 动作拒绝

发生在主链路合法性判断阶段。

例子：

- 目标不在同一实例。
- 超出范围。
- 非敌对。
- 冷却未好。
- 资源不足。
- 地图不允许该动作。

处理方式：

- 对玩家主动操作，返回明确错误或提示。
- 对怪物 AI 操作，记录机器可读原因，允许 AI 下次重新选择。
- 不进入伤害结算。

### 7.3 合法未命中

发生在主链路结算阶段。

例子：

- 闪避。
- 格挡。
- 抵抗。
- 免疫。
- 伤害被减为 0。

处理方式：

- 必须生成战斗结果。
- 必须有浮字或表现。
- 对相关玩家必须有战斗通知。

### 7.4 异常不一致

发生在权威状态之间互相矛盾时。

例子：

- 地图占位和玩家位置服务不同步。
- pending cast 指向不存在的实例。
- 怪物 action 指向不存在但未清理的怪物。
- 技能定义 revision 不匹配。

处理方式：

- 不静默吞掉。
- 记录诊断事件。
- 按安全策略拒绝结算。
- 必要时触发运行态修复或重新同步。

## 8. 服务端权威和网络边界

客户端只负责提交意图和展示结果。

客户端可以提交：

- 想普攻哪个目标。
- 想释放哪个技能。
- 想选择哪个地块锚点。
- 想开启或停止自动战斗。

客户端不能裁定：

- 是否真的能打。
- 目标是否真的在范围内。
- 伤害是多少。
- 是否闪避。
- 是否死亡。
- 掉落和奖励。
- 地块、阵法、容器最终状态。

服务端向客户端发送：

- 战斗动作表现。
- 预警区域。
- 伤害、闪避、免疫等浮字。
- 玩家通知。
- 状态增量。
- 必要的详情刷新。

高频同步要求：

- 战斗表现走 AOI。
- 玩家通知只发相关玩家。
- 状态变化走增量。
- 配置静态信息不混入高频包。
- 诊断和审计不发给普通客户端。

## 9. 持久化和恢复口径

战斗主链路本身运行在内存权威态中，tick 热路径不直接做数据库 IO。

需要持久化的结果：

- 玩家血量、死亡、复活位置。
- 玩家资源、buff、冷却。
- 玩家奖励、掉落、任务进度。
- 地图实例中需要持久化的怪物状态。
- 持久地图的地块、阵法、容器变化。
- 重要战斗审计。

处理方式：

- 战斗链路只标记脏域和产生事件。
- 持久化由 flush、outbox、快照或受控队列异步处理。
- 崩溃恢复时从持久化真源和运行时快照恢复。
- pending cast 如果需要跨重启保留，必须进入可恢复快照；否则重启时必须有明确清理策略。

## 10. 可观测性要求

商业化链路必须能回答：

- 谁发起了动作。
- 动作来自玩家输入、自动战斗还是怪物 AI。
- 动作定义是什么。
- 锚点和影响格是什么。
- 候选目标有哪些。
- 每个候选目标为什么通过或被拒绝。
- 每个目标的命中和伤害结果是什么。
- 结果写入了哪个真源。
- 给哪些玩家发了通知。
- 广播了哪些表现。
- 哪些状态被标记为脏。

最低要求：

- 所有拒绝原因机器可读。
- 所有异常落空有诊断。
- 所有进入结算的目标有结果事件。
- 怪物 action 不允许无原因静默 return。
- 预警格命中但没有通知时，必须能定位停在主链路哪一步。

## 11. 推荐模块边界

目标模块可以按职责拆分，不要做一个巨型战斗服务。

### 战斗编排

负责把一次 `CombatAction` 按主链路推进。

不直接写玩家、怪物、地块、阵法、容器状态，只调用后段适配器。

### 动作定义解析

负责把普攻、技能、怪物技能、系统动作解析成统一动作定义。

普攻定义应该和技能定义同形态，只是来源为系统内置。

### 目标解析

负责锚点、影响格、目标收集、最大目标数。

目标解析不扣血、不发奖励。

### 合法性策略

负责同实例、敌我关系、范围、视线、地图能力、资源、冷却等判断。

所有拒绝都返回结构化原因。

### 吟唱调度

负责 pending cast 的创建、推进、取消和完成。

完成时重新进入主链路。

### 效果结算

负责命中、闪避、伤害、buff、治疗、净化等计算。

不直接写最终真源。

### 结果应用适配器

负责按目标类型写状态真源。

- 玩家适配器。
- 怪物适配器。
- 地块适配器。
- 阵法适配器。
- 容器适配器。

### 事件构建

负责把结果转成：

- 玩家通知。
- AOI 表现。
- 状态同步脏域。
- 审计事件。
- 诊断事件。

## 12. 迁移策略

当前链路不应一次性硬切。

建议分阶段：

1. 先定义统一数据结构和拒绝原因，不改变玩法行为。
2. 先让玩家普攻走统一动作定义，但结果仍调用旧写入逻辑。
3. 再让玩家技能复用统一目标解析和合法性判断。
4. 再让怪物普攻转成统一 `CombatAction`。
5. 再让怪物技能和吟唱完成转成统一 `CombatAction`。
6. 最后收敛通知、表现、诊断和审计。
7. 每阶段都保留对照 smoke，确保伤害、闪避、死亡、掉落、预警、合法落空行为一致。

迁移期间的硬要求：

- 不改变数值结果，除非明确修 bug。
- 不改变玩家可见规则，除非文档和测试同步更新。
- 每个旧分支迁入主链路时，必须证明“进入结算必有结果事件”。
- 怪物吟唱技能必须优先接入结构化拒绝原因，避免继续静默吞 action。

## 13. 最终验收标准

商业化战斗链路完成后，应满足：

1. 普攻和技能共用同一条主链路。
2. 玩家和怪物只是发起者类型不同。
3. 怪物 AI 不直接结算伤害。
4. 地块、阵法、容器只在结果应用阶段分支。
5. 命中、闪避、减伤、免疫、伤害统一结算。
6. 敌我关系、范围、视线、地图能力统一判断。
7. 吟唱开始和完成是同一条主链路的两个阶段。
8. 所有拒绝都有机器可读原因。
9. 所有进入结算的目标都有通知或结果事件。
10. 所有状态写入都有明确真源。
11. 高频同步只发必要表现和增量状态。
12. 持久化通过脏域、快照或 outbox 转出，不在 tick 热路径直接 IO。
