# 关闭链路编排

本文档定义服务端从 GM restart / SIGTERM / SIGINT 到进程退出的关闭链路编排。关闭链路的目标不是“尽快退出”，而是在固定时间预算内完成流量隔离、玩家状态收口、最终落盘、租约释放和节点下线，避免重启后出现双写、丢玩家位置、在线态残留或实例租约悬挂。

当前代码已有 `WorldShutdownDrainService` 执行基础 drain，但还没有被完整收进 `ServerLifecycleCoordinatorService`。本文档先定义目标合同，后续代码按本文迁移。

## 1. 关闭编排目标

商业级 MMO 关闭必须回答：

- 新连接和新玩家意图什么时候停止进入运行时。
- 已连接玩家如何断开、detach、写 presence、清 session route。
- tick、flush、outbox、worker 的关闭顺序是什么。
- 哪些状态必须 final flush，哪些可以延迟由恢复链补偿。
- 实例租约什么时候释放，释放失败如何处理。
- 关闭总预算、单阶段预算、失败重试和强制退出策略是什么。
- health / GM / 日志如何呈现 draining 进度和失败对象。

核心目标：

1. 关闭开始后立即关闭 `trafficOpen`，socket bootstrap 和玩家命令不得继续进入 runtime。
2. 停止 tick 前必须冻结新意图；停止 tick 后不得再产生新的 gameplay dirty。
3. final flush 必须发生在释放实例 lease 之前。
4. release lease 必须发生在玩家 detach、presence 写入、final flush 之后。
5. 关闭失败只能进入可恢复状态，不能在 lease 已释放后继续写玩家或地图状态。
6. 关闭过程必须幂等；SIGTERM、SIGINT、Nest `beforeApplicationShutdown` 重入时只执行一次。

## 2. 现状问题边界

当前真实关闭入口：

1. GM restart 或运维信号触发 `SIGTERM` / `SIGINT`。
2. `main.ts` 信号 handler 先取 `WorldShutdownDrainService` 调用 `drain(signal)`。
3. `main.ts` 随后执行 `app.close()`。
4. 15 秒超时后强制 `process.exit(1)`。
5. Nest `enableShutdownHooks()` 会触发各 service 的 `onModuleDestroy` / `beforeApplicationShutdown`。

当前 `WorldShutdownDrainService.drain()` 顺序：

1. `worldGateway.setDraining(true)`。
2. `worldGateway.disconnectAllForShutdown('server_shutdown')`。
3. 逐玩家 `drainDetachedBinding()`：
   - clear disconnected player state。
   - persist offline presence。
   - flush single player。
4. `worldRuntimeService.releaseLocalInstanceLeasesForShutdown()`。
5. `worldTickService.stopForShutdown()`。
6. `marketRuntimeService.drainForShutdown()`。
7. `playerPersistenceFlushService.flushAllNow()`。
8. `mapPersistenceFlushService.flushAllNow()`。
9. `tongtianTowerPersistenceService.flushAllProgress()`。

主要差距：

- 关闭状态没有由统一生命周期编排器持有，`main.ts` 直接调用 `WorldShutdownDrainService`。
- release instance lease 发生在 stop tick 和 final flush 之前，目标合同应改为 final flush 后释放 lease。
- worker / outbox / flush consumer 还没有统一 close gate，只依赖各自 `onModuleDestroy`。
- 关闭 phase、耗时、失败对象没有统一暴露到 health / GM。
- 单阶段预算不清晰；只有总 15 秒硬超时。
- release lease 失败、flush 失败、玩家 detach 失败只打日志，缺少可查询的 drain result。

## 3. 编排器职责边界

目标新增或扩展：

- `ServerLifecycleCoordinator.drain(reason)`：进程级关闭总入口。
- `ShutdownStatusService` 或复用 `StartupStatusService` 的 `draining` phase：记录关闭阶段、耗时、错误和结果。
- `StartupBarrierService.closeForDrain()`：关闭 traffic、tick、flush、outbox、worker、instance write、player attach。
- `WorldShutdownDrainService`：降级为关闭链路的领域执行器，不直接拥有全局 phase 顺序。

职责分配：

| 模块 | 职责 |
|------|------|
| `main.ts` | 只捕获信号、调用生命周期编排器、设置总硬超时 |
| 生命周期编排器 | 唯一拥有关闭 phase、预算、闸门关闭和失败策略 |
| `WorldGateway` | 停接入、断开 socket、逐玩家 detach |
| `WorldSessionService` | session binding detach、expire、route 清理输入 |
| `WorldRuntimeService` | 冻结实例写入、释放本节点实例 lease |
| `WorldTickService` | 停止调度并等待 in-flight tick |
| `Player/Map/Tongtian Flush` | final flush 和失败返回 |
| `MarketRuntimeService` | 等待交易 mutation 队列收尾 |
| `Worker/Outbox/Flush Runtime` | 停止消费，避免 drain 期间新任务写入半关闭状态 |
| `NodeRegistryRuntimeService` | 最后注销节点 |

## 4. 关闭状态机

关闭状态必须暴露给 health / GM / 日志：

| 状态 | 含义 |
|------|------|
| `drain_requested` | 收到关闭请求，准备关闭闸门 |
| `traffic_closed` | 新连接、新 bootstrap、新玩家命令已拒绝 |
| `sessions_draining` | 已连接 socket 正在断开和 detach |
| `runtime_frozen` | tick、玩家 attach、实例写入已冻结 |
| `workers_stopping` | worker、outbox、flush consumer 正在停止消费 |
| `final_flushing` | 玩家、地图、通天塔、市场等最终落盘中 |
| `leases_releasing` | 本节点实例 lease 正在释放 |
| `node_deregistering` | 节点心跳停止并注销 |
| `drain_completed` | 关闭 drain 完成，可进入 `app.close()` |
| `drain_failed` | 存在失败对象，按策略继续关闭或强制退出 |
| `force_exit` | 超过硬预算，进程强制退出 |

每个阶段记录：

- `phase`
- `startedAt`
- `finishedAt`
- `durationMs`
- `attempt`
- `errorCode`
- `errorMessage`
- `affectedPlayers`
- `affectedInstances`
- `metrics`

## 5. 关闭链路总顺序

目标顺序如下。所有阶段必须幂等，重复调用不得重复断开、重复释放 lease 或重复清 dirty。

### Phase 0：接收关闭请求

触发源：

- GM restart。
- `SIGTERM`。
- `SIGINT`。
- 容器编排停止。
- 运维维护窗口。

动作：

1. 生成 `shutdownRunId`。
2. CAS 设置 `draining=true`，后续重入直接等待同一 promise。
3. 记录 `reason`、signal、pid、nodeId、role、当前 runtime summary。
4. 设置总预算，例如 15 秒硬超时，目标软预算 12 秒。

失败策略：

- 无法创建 drain 状态不阻断关闭，但必须输出 fatal 日志。
- 重入不得并发执行第二条 drain。

### Phase 1：关闭外部流量

目标：立即阻断新请求进入运行时。

动作：

1. `StartupBarrierService.closeForDrain()`：
   - `trafficOpen=false`
   - `tickOpen=false`
   - `flushOpen=false`
   - `outboxOpen=false`
   - `workerOpen=false`
   - `instanceWriteOpen=false`
   - `instanceAttachOpen=false`
2. `WorldGateway.setDraining(true)`。
3. health readiness 返回 not ready / draining。
4. 新 socket connect 返回 `SERVER_SHUTTING_DOWN`。
5. GM/HTTP 写操作返回维护态或 draining 错误。

硬规则：

- 关闭 traffic 必须是第一阶段。
- 关闭 traffic 后不得再接受玩家 gameplay intent。
- GM 只允许只读 drain 状态查询和必要强制退出命令。

### Phase 2：冻结运行时输入

目标：让已经进入服务器的异步命令不再制造新状态变化。

动作：

1. 停止 player attach。
2. 停止 command intake，把 pending player command 标记为 shutdown rejected 或保留到重连后重发。
3. 停止转图、进副本、市场新写操作、GM runtime mutation。
4. 冻结实例写入闸门，之后 gameplay write 必须拒绝。

失败策略：

- 已进入 tick 的操作允许在当前 tick 内完成。
- 未执行 pending command 不做半执行，不写玩家资产。

### Phase 3：断开 socket 与玩家 detach

目标：把在线 socket 变成可恢复的离线状态。

动作：

1. `WorldGateway.disconnectAllForShutdown('server_shutdown')`。
2. 对每个 detached binding：
   - 清 disconnected player state。
   - persist offline presence。
   - flush single player。
   - clear local session route。
   - clear detached player caches。
3. 对失败玩家记录 `player_detach_failed`，进入 drain result。

硬规则：

- 玩家 detach 不得创建新 runtime player。
- flush single player 失败时不清 dirty，不覆盖空快照。
- presence 写入失败时不得假装玩家已彻底离线；恢复链应能识别残留在线态。

### Phase 4：停止 tick 并等待 in-flight tick

目标：停止世界时间推进，避免 final flush 后又产生新 dirty。

动作：

1. 调用 `WorldTickService.stopForShutdown()`。
2. 停止调度 timer。
3. 等待 in-flight tick，当前实现预算 5 秒。
4. 若超时，记录 `tick_stop_timeout`，继续后续 drain，但实例 lease 释放需要更保守。

硬规则：

- final map/player flush 必须在 tick 停止后执行。
- 如果 tick 超时仍未完成，后续 release lease 必须按失败策略决定是否跳过。

### Phase 5：停止 worker / outbox / flush consumer

目标：让后台消费者不再认领新任务或写出新结果。

动作：

1. 停止 background worker orchestrator timer。
2. 停止 outbox dispatcher 调度。
3. 停止 flush task runtime consumer 和 staging collector。
4. 等待已 in-flight 的 worker task 完成或到达单阶段预算。

硬规则：

- drain 期间不允许 worker 认领新 outbox / flush task。
- 已认领但未完成任务必须按 retry/claim_until 机制交给下次启动恢复。

### Phase 6：领域队列收尾

目标：等待内存中的串行 mutation 队列完成，避免 final flush 捕获半状态。

动作：

1. `MarketRuntimeService.drainForShutdown()` 等待市场 mutation queue。
2. 邮件、交易、GM runtime mutation 若存在队列，按同样方式 drain。
3. 对超时队列记录 `domain_queue_timeout`。

失败策略：

- 资产类队列超时：相关域应进入 degraded drain result，后续启动恢复和审计必须可查。
- 非资产类队列超时：可跳过，但记录。

### Phase 7：最终落盘

目标：在租约仍归本节点所有时，完成最后一次玩家和地图持久化。

固定顺序：

1. `PlayerPersistenceFlushService.flushAllNow()`。
2. `MapPersistenceFlushService.flushAllNow()`。
3. `TongtianTowerPersistenceService.flushAllProgress()`。
4. 其他实例 domain flush。
5. outbox / audit 关键事件按预算 flush 或保留 ready 状态给下次 worker。

硬规则：

- final flush 必须早于 release lease。
- final flush 失败不得清 dirty。
- lease 已丢失的实例不得写 map/domain 快照，只能记录 skipped。
- 玩家资产、位置、市场、邮件等关键域失败必须进入 drain result。

### Phase 8：释放实例租约

目标：只有在本节点停止写入并完成 final flush 后，才释放本节点持有的实例租约。

动作：

1. 遍历本节点持有的 active instance lease。
2. 确认没有 connected players。
3. 调用 `InstanceCatalogService.releaseInstanceLease()`。
4. 清 runtime meta 中的 `assignedNodeId / leaseToken / leaseExpireAt`。
5. 输出 `released / skipped / failed`。

硬规则：

- release lease 不能早于 final flush。
- 如果仍有在线玩家或 detach 未完成，跳过释放并记录。
- 如果 tick stop 超时或 final flush 失败，默认跳过释放相关实例，避免其他节点接管未稳定状态。
- release lease 失败不重试无上限；最多短重试后交给 lease TTL / recovery 处理。

### Phase 9：节点注销

目标：本节点不再被调度新实例、worker 或玩家路由。

动作：

1. 停止 node heartbeat。
2. `NodeRegistryService.deregisterNode()`。
3. 清本地 session route 残留。
4. 记录节点下线时间和 drain result。

失败策略：

- deregister 失败时记录，依赖 node heartbeat 过期推进 suspect/dead。
- 不因 deregister 失败阻塞进程无限等待。

### Phase 10：关闭 Nest app 与进程退出

目标：释放 DI、数据库连接、Redis 连接和 timer。

动作：

1. `app.close()`。
2. Nest 触发 `onModuleDestroy` / `beforeApplicationShutdown`。
3. database pool close。
4. 关闭完成后清 `bootstrapApp=null`。
5. 超过硬预算强制 `process.exit(1)`。

硬规则：

- `beforeApplicationShutdown` 不能再次并发执行完整 drain，只能 await 已有 drain promise。
- 强制退出前必须输出最后 drain status。

## 6. 关闭闸门合同

| 闸门 | 关闭阶段 | 关闭后行为 |
|------|----------|------------|
| `trafficOpen` | Phase 1 | 拒绝新 connect / bootstrap / 玩家 HTTP 写入口 |
| `tickOpen` | Phase 1/4 | tick 不再调度；in-flight tick 等待完成 |
| `flushOpen` | Phase 5 | 周期 flush / staging 停止，Phase 7 手动 final flush 仍可执行 |
| `outboxOpen` | Phase 5 | outbox 不再认领新事件 |
| `workerOpen` | Phase 5 | background worker 不再认领新任务 |
| `instanceWritable` | Phase 2 | gameplay write 拒绝，final flush 只读快照 |
| `instanceAttachAllowed` | Phase 1 | 玩家不再 attach 到实例 |

注意：`flushOpen=false` 不代表禁止 final flush。周期消费者关闭后，编排器仍需要显式调用 final flush 方法。

## 7. 失败隔离策略

| 失败类型 | 策略 |
|----------|------|
| 重入 drain | 返回同一 drain promise |
| socket disconnect 失败 | 记录 socket/player，继续 drain |
| presence 写入失败 | 保留恢复线索，不清本地 dirty，记录玩家 |
| single player flush 失败 | 不清 dirty，继续其他玩家 |
| tick stop 超时 | 继续 drain，但相关实例默认不释放 lease |
| worker in-flight 超时 | 停止认领，交给 claim TTL / retry |
| market queue 超时 | 标记资产域 drain_degraded |
| final player flush 失败 | 记录玩家，禁止清 dirty |
| final map flush 失败 | 记录实例，默认跳过释放对应 lease |
| release lease 失败 | 记录实例，依赖 TTL/recovery，继续关闭 |
| node deregister 失败 | 记录，依赖 heartbeat stale 推进 |
| 总预算超时 | 输出 drain status 后强制退出 |

## 8. 与启动链路的恢复关系

关闭链路必须给启动链路留下可恢复状态：

1. 如果 final flush 成功且 lease 释放成功，下次启动可正常 claim/hydrate。
2. 如果 final flush 成功但 release lease 失败，下次启动等待 TTL 或按策略 force reclaim。
3. 如果 final flush 失败，不应释放对应实例 lease；下次启动通过 dirty/domain watermark 恢复。
4. 如果 presence 写入失败，启动链路需要清理 stale online presence 或恢复离线挂机。
5. 如果 worker/outbox 未完成，claim_until / retry 队列必须允许下次 worker 接续。

关闭链路的目标不是消灭所有失败，而是让每种失败都能被启动链路、诊断接口和 GM 工具识别并恢复。

## 9. 推荐接口草案

```ts
interface ServerLifecycleCoordinator {
  start(): Promise<void>;
  drain(reason: ShutdownReason): Promise<ShutdownResult>;
  getStatus(): LifecycleStatusSnapshot;
}

type ShutdownReason =
  | { type: 'SIGTERM'; signal: 'SIGTERM' }
  | { type: 'SIGINT'; signal: 'SIGINT' }
  | { type: 'gm_restart'; requestedBy: string }
  | { type: 'maintenance'; requestedBy: string };

interface ShutdownResult {
  shutdownRunId: string;
  completed: boolean;
  forced: boolean;
  durationMs: number;
  phases: ShutdownPhaseSnapshot[];
  players: {
    detached: number;
    flushFailed: string[];
    presenceFailed: string[];
  };
  instances: {
    flushed: number;
    flushFailed: string[];
    leaseReleased: number;
    leaseReleaseSkipped: string[];
    leaseReleaseFailed: string[];
  };
}
```

## 10. 迁移路线

### Step A：状态与闸门

- `ServerLifecycleCoordinatorService` 增加 `drain()`。
- 关闭时调用 `StartupBarrierService.closeForDrain()`。
- health readiness 暴露 draining 状态。

### Step B：main.ts 收口

- `main.ts` 不再直接取 `WorldShutdownDrainService`。
- signal handler 调用 `ServerLifecycleCoordinatorService.drain()`。
- `beforeApplicationShutdown` 只 await 已有 drain promise，避免二次 drain。

### Step C：重排 drain 顺序

- 把 release lease 移到 final flush 后。
- stop tick 和 stop worker/outbox/flush consumer 放到 final flush 前。
- final flush 失败影响 lease release 策略。

### Step D：补阶段预算和结果对象

- 每个 phase 有 soft timeout。
- drain 返回结构化 `ShutdownResult`。
- GM/health 可以查询最近一次 shutdown status。

### Step E：补证明链

至少补以下 smoke/proof：

- `shutdown-coordinator-order-smoke`：证明 traffic close → detach → stop tick/worker → final flush → release lease。
- `shutdown-final-flush-before-lease-release-smoke`：证明 lease 释放晚于 final flush。
- `shutdown-drain-idempotent-smoke`：证明 SIGTERM / beforeApplicationShutdown 重入只执行一次。
- `shutdown-failed-flush-keeps-lease-smoke`：证明实例 flush 失败时跳过 release lease。
- `shutdown-readiness-draining-smoke`：证明 draining 时 socket bootstrap 被拒绝。

## 11. 完成标准

关闭链路重整完成必须同时满足：

1. 关闭请求进入后，新 socket bootstrap 和玩家命令立即拒绝。
2. tick 不会在 final flush 后继续产生 dirty。
3. worker/outbox/flush consumer 不会在 drain 期间认领新任务。
4. 玩家 detach、presence、session route、cache 清理有明确结果。
5. final flush 发生在 release lease 之前。
6. final flush 失败的实例不会释放 lease。
7. release lease 后本节点不会继续写该实例。
8. node deregister 在 lease release 后执行。
9. 关闭链路重入是幂等的。
10. health / GM 能看到 drain phase、耗时、失败玩家、失败实例和 release 结果。
