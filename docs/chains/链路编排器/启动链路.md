# 启动链路编排

本文档定义服务端从进程启动到 socket 玩家流量开放的启动链路编排。

## 1. 启动编排目标

商业级 MMO 启动不能依赖 NestJS 生命周期的自然执行顺序。启动必须有一个显式编排器回答：

- 哪些依赖先完成，哪些服务可以并行准备。
- 哪些阶段失败必须 fail-fast，哪些可以降级启动。
- tick、flush、worker、socket 流量什么时候允许打开。
- 地图实例什么时候算“可写”。
- 玩家什么时候允许 attach 到地图实例。
- 实例租约失败、持久化恢复失败、玩家恢复失败时如何隔离。
- readiness、GM、日志和指标如何呈现启动进度。

核心目标：

1. 启动期只有一个权威编排入口，不让各 service 在 `onModuleInit` / `onApplicationBootstrap` 中自行抢跑。
2. DB 持久化真源、实例租约、运行时内存态、Redis 在线态、网络流量按固定阶段闭环。
3. 任何玩家 attach 前，目标实例必须完成模板校验、租约确认、持久化 hydrate 和可写闸门。
4. tick、flush、outbox、background worker 不得消费半恢复状态。
5. readiness 只在启动链路完成后开放，socket bootstrap 只在 readiness 开放后进入。

## 2. 现状问题边界

当前服务启动顺序主要由 Nest 生命周期自然拼接：

1. `main.ts` 执行 `bootstrapLoadDbConfig()`，解析 runtime role，创建 Nest app/context。
2. 大量 persistence、runtime、worker service 在 `onModuleInit()` 中初始化表、启动定时器或构造运行态。
3. `WorldTickService.onModuleInit()` 直接启动权威 tick。
4. `WorldRuntimeService.onModuleInit()` 先 bootstrap 公共实例。
5. `WorldRuntimeService.onApplicationBootstrap()` 再执行持久化世界恢复、实例租约同步定时器。
6. Socket gateway 通过 readiness 拦截玩家流量，但 readiness 当前更多是依赖汇总，不是启动阶段编排状态机。

这导致启动期存在天然竞态：

- tick 可能早于世界恢复完成开始推进。
- flush / outbox / worker 可能早于完整恢复开始消费。
- 实例创建、租约接管、持久化 hydrate、玩家恢复分散在多个 service 内。
- 通天塔等特殊实例路径可能绕过通用 catalog claim，先生成 runtime/cache，再补租约。
- 玩家恢复使用 `resolveTargetInstance()` 时可能触发特殊实例创建，导致玩家 attach 早于实例完成统一租约准入。

本文档的目标不是立刻改代码，而是把后续重整启动链路的编排合同先定下来。

## 3. 编排器职责边界

建议新增统一启动编排器：

- `ServerLifecycleCoordinator`：进程级启动/关闭阶段总控。
- `WorldRuntimeStartupCoordinator`：世界运行时启动子链路，负责实例、租约、hydrate、玩家恢复、tick 开闸。
- `StartupBarrierService`：集中暴露启动闸门，供 tick、flush、worker、gateway、world runtime 查询。
- `StartupStatusService`：记录阶段状态，输出 health / GM / 日志可观测信息。

职责分配：

| 模块 | 职责 |
|------|------|
| `main.ts` | 只负责进程级 env 预加载、role 解析、Nest app 创建、signal 入口 |
| `AppModule` | 只负责 DI 装配，不承载启动顺序 |
| 生命周期编排器 | 唯一拥有 phase 顺序、失败策略、闸门开闭 |
| `WorldRuntimeService` | 提供实例构造、恢复、查询、tick 执行能力，不自行决定启动顺序 |
| `WorldTickService` | 只在编排器允许后启动调度；tick 前检查闸门 |
| `Flush/Outbox/Worker` | 只在对应 role 和闸门打开后消费任务 |
| `Gateway` | readiness 未开放前拒绝玩家 bootstrap |
| `Persistence` | 提供真源初始化、schema 检查、恢复读取、事务写入，不驱动 runtime 顺序 |

## 4. 启动状态机

启动状态必须持久暴露给 health / GM / 日志：

| 状态 | 含义 |
|------|------|
| `created` | 进程已启动，Nest app 尚未完成依赖准备 |
| `preparing` | 静态配置、内容、持久化依赖准备中 |
| `recovering_world` | 地图实例、租约、持久化 domain hydrate 中 |
| `recovering_players` | 离线挂机和在线态残留玩家恢复/隔离中 |
| `starting_loops` | tick、flush、worker、outbox 按 role 开闸 |
| `ready` | socket 玩家流量已开放 |
| `degraded` | 非关键域失败，核心玩家流量可开放但带降级原因 |
| `failed` | 关键域失败，进程不得对玩家开放 |
| `draining` | 已进入关闭排空，启动闸门全部关闭 |

每个阶段记录：

- `phase`
- `status`
- `startedAt`
- `finishedAt`
- `durationMs`
- `errorCode`
- `errorMessage`
- `blocking`
- `metrics`

## 5. 启动链路总顺序

目标启动顺序如下。所有阶段必须幂等；重试只能从阶段边界重试，不允许半阶段内部重复写出不可逆结果。

### Phase 0：进程与角色预启动

目标：只完成进程级输入，不启动 runtime 循环。

动作：

1. 安装日志捕获与 fatal handler。
2. 从 DB 预加载可用于启动的配置到 env；DB 不可用时按环境策略决定 fail-fast 或 fallback。
3. 解析 runtime role：`all` / `api` / `worker`。
4. 创建 Nest app/context，但所有 runtime loop 默认关闭。
5. 初始化 `StartupBarrierService`：
   - `trafficOpen=false`
   - `tickOpen=false`
   - `flushOpen=false`
   - `workerOpen=false`
   - `instanceAttachOpen=false`

失败策略：

- role 解析失败：fail-fast。
- 生产环境关键 DB config 加载失败：fail-fast。
- 开发/测试可按配置降级，但必须记录 `degraded` 原因。

### Phase 1：静态配置与内容校验

目标：运行时热路径只读预解析内容，不在 tick 或玩家 bootstrap 时再发现模板错误。

动作：

1. 校验环境变量、端口、CORS、JWT/GM secret、DB/Redis 配置。
2. 加载共享 schema、服务端内容模板、地图模板、怪物/物品/技能/buff/npc/quest/drop/landmark/tile registry。
3. 校验地图实例模板引用、通天塔模板、宗门模板、公共地图默认入口。
4. 冻结模板 registry，建立运行期只读索引。

失败策略：

- 模板缺失、schema 错误、关键内容引用断裂：fail-fast。
- 非关键展示字段缺失可降级，但不得影响运行时权威计算。

### Phase 2：持久化真源启动

目标：数据库真源和 Redis 在线态边界明确可用，后续恢复只走正式仓储。

动作：

1. 初始化 PostgreSQL 连接池。
2. 检查 migration/schema/table/index 是否满足当前代码版本。
3. 初始化玩家域、地图实例域、邮件、市场、GM 审计、outbox、flush ledger、node registry。
4. 初始化 Redis 在线态、session route、短期索引。
5. 清理明显过期的本节点残留锁或会话路由，但不得清理其他活跃节点租约。

失败策略：

- 玩家、实例、租约、flush ledger 真源不可用：fail-fast。
- 邮件/建议/排行等非关键域可降级，但必须阻断对应功能写入并暴露 readiness 详情。

### Phase 3：流量与运行时冻结

目标：进入恢复前关闭所有会消费或写出半状态的入口。

动作：

1. Gateway 维持 `trafficOpen=false`，所有玩家 socket bootstrap 返回 not ready。
2. Tick 调度器保持停止。
3. Player flush、map flush、domain flush、outbox dispatcher、background worker 保持停止或只允许只读探测。
4. 运行时允许构造内存对象，但默认 `instanceWritable=false`、`playerAttachAllowed=false`。
5. 所有 `dirty` 标记在恢复提交前进入 staging，不直接进入 flush 队列。

硬规则：

- 不允许“先恢复玩家，后补实例租约”。
- 不允许“tick 先跑，恢复慢慢补”。
- 不允许 worker 在启动恢复未完成时消费玩家资产、地图实例或市场任务。

### Phase 4：节点注册与集群身份

目标：本进程获得稳定 node identity，用于实例租约、session route、审计和 worker ownership。

动作：

1. 注册或刷新 `nodeId`、role、clusterId、启动时间、版本号。
2. 标记节点状态为 `starting`。
3. 为后续实例租约准备 node-scoped lease token。
4. 记录启动批次 `startupRunId`，贯穿日志、GM、health、审计。

失败策略：

- 需要持有权威运行态的 role 无法注册 node：fail-fast。
- 纯 API role 可降级，但不得持有 runtime lease。

### Phase 5：世界恢复计划构建

目标：先生成恢复计划，不直接 attach 玩家或启动实例写入。

输入真源：

- `instance_catalog`
- 地图实例 domain snapshot
- 玩家位置 / presence / session route
- 通天塔、宗门、公共地图等专属持久化域

动作：

1. 扫描 `instance_catalog`。
2. 将实例分类：
   - public long-lived
   - sect
   - tower / dungeon / secret realm
   - player-owned / party-owned
   - destroyed / stopped
   - template_missing
3. 为每个可恢复实例生成 `InstanceStartupPlan`：
   - `instanceId`
   - `templateId`
   - `kind`
   - `persistentPolicy`
   - `owner`
   - `routeDomain`
   - `leaseRequired`
   - `hydrateRequired`
   - `playerAttachAllowed=false`
4. 对缺模板、重复 instanceId、非法状态、孤儿 owner 生成 quarantine 计划。

硬规则：

- 通天塔实例必须进入同一份恢复计划，不得由专属 cache 路径绕过 catalog/lease 计划。
- 计划阶段不得创建可写 runtime 实例。
- 计划阶段不得恢复玩家。

### Phase 6：实例租约获取

目标：所有可写实例先完成租约准入，再允许 hydrate 后进入 runtime。

动作：

1. 对 `leaseRequired=true` 的实例执行 claim / renew / force-reclaim 策略。
2. force-reclaim 只能按环境和时间窗配置使用，生产默认保守。
3. 租约成功后写入：
   - `assignedNodeId`
   - `leaseToken`
   - `leaseExpireAt`
   - `ownershipEpoch`
4. 租约失败的实例进入 `lease_quarantine`：
   - 不创建可写 runtime。
   - 不 attach 玩家。
   - 不 flush。
   - health / GM 暴露实例和阻断原因。

硬规则：

- 任何 `syncInstanceLease()` 失败不得在已有在线玩家的情况下继续写入。
- `fenced` 实例只能只读诊断或排空，不能接受 gameplay write。
- 通天塔层实例与普通 catalog 实例使用相同租约门禁。

### Phase 7：实例构造与持久化 hydrate

目标：构造 runtime aggregate，并从 DB hydrate domain 状态，但仍不开放玩家 attach。

动作：

1. 只对租约成功或无需租约的计划创建 runtime instance。
2. 从模板 registry 创建运行态实例，不复制模板大字段进实例 own field。
3. hydrate 地图 domain、怪物/容器/掉落、建筑/房间、宗门、通天塔 dungeon state。
4. 恢复完成前实例状态为：
   - `runtimeStatus=hydrating`
   - `writable=false`
   - `playerAttachAllowed=false`
5. hydrate 成功后标记：
   - `runtimeStatus=leased`
   - `writable=true`
   - `hydrateComplete=true`

失败策略：

- 单实例 hydrate 失败：实例 quarantine；若是默认出生点或核心公共地图，整体 fail-fast。
- 非核心副本实例失败：降级启动，但相关玩家恢复进入 quarantine。

### Phase 8：运行时不变量检查

目标：启动玩家恢复前，先证明实例集合内部一致。

检查项：

1. 每个 runtime 实例有唯一 `instanceId`。
2. 可写实例必须有本节点有效 lease。
3. `runtimeStatus=fenced` 或 `lease_quarantine` 实例不可写。
4. occupancy index 为空或与 hydrate 后实体一致。
5. 怪物、容器、阵法、建筑索引引用存在。
6. `destroyed/stopped` 实例没有被恢复为可写。
7. 通天塔 layer 与 `tower:tongtian:layer:*` / `tongtian_tower_layer_*` 映射一致。

失败策略：

- 索引不变量破坏：fail-fast，避免玩家进入错乱世界。
- 可隔离的单实例问题：quarantine，但核心地图失败仍 fail-fast。

### Phase 9：玩家恢复计划构建

目标：从持久化真源生成玩家恢复计划，不直接连接玩家。

动作：

1. 读取离线挂机玩家位置和 presence。
2. 过期超时离线玩家，按规则标记彻底离线。
3. 读取可能的 session route 残留，执行 session fencing 计划。
4. 为每个玩家生成 `PlayerStartupPlan`：
   - `playerId`
   - `targetInstanceId`
   - `targetTemplateId`
   - `x/y`
   - `presenceState`
   - `requiresRuntimeAttach`
   - `attachBlockedReason`
5. 如果目标实例未 ready，玩家进入 `player_recovery_quarantine`，不得触发 fallback 随机出生或空快照覆盖。

硬规则：

- 玩家恢复计划不能反向创建未租约实例。
- 玩家位置实例不存在时，只能进入隔离或按明确迁移规则迁移，不能静默落到默认地图。

### Phase 10：玩家 attach / 恢复

目标：只把玩家恢复到已经可写、已 hydrate、已通过不变量检查的实例。

准入函数：

```ts
instanceReadyForPlayerAttach(instanceId) =
  runtimeExists
  && hydrateComplete
  && localLeaseWritable
  && runtimeStatus !== 'fenced'
  && instanceAttachOpenFor(instanceId)
```

动作：

1. 恢复玩家 runtime snapshot。
2. 校验目标实例 `instanceReadyForPlayerAttach(instanceId)`。
3. 通过占位与落点修正把玩家加入实例 occupancy。
4. 写入本节点在线/挂机恢复态，但 socket session 仍为空。
5. 失败玩家进入 quarantine，不覆盖玩家持久化资产和位置。

通天塔专项要求：

- `ensureLayerInstanceForRestore()` 不得在玩家 attach 路径中创建绕过租约的可写实例。
- `prepareRestoredLayerInstance()` 不得清空租约后立即暴露为 `running` 可写。
- 通天塔玩家恢复必须先通过同一 `instanceReadyForPlayerAttach()`。

### Phase 11：二级领域恢复

目标：恢复不直接决定地图 attach 的领域服务，但在流量开放前建立一致只读视图。

领域：

- market / auction
- mail
- redeem
- suggestion
- leaderboard
- GM runtime flags
- runtime diagnostics ring
- background task metadata

策略：

- 会影响玩家资产或交易的市场、邮件写入域属于关键域，失败时阻断对应写入口。
- 只影响展示的排行、建议可降级。
- 所有降级项写入 readiness details。

### Phase 12：开启权威 tick

目标：世界与玩家恢复完成后，才允许时间推进。

动作：

1. 设置 `tickOpen=true`。
2. `WorldTickService` 启动自调度。
3. 每次 tick 前检查 `StartupBarrierService.tickOpen` 和 `draining=false`。
4. 首次 tick 记录 `firstTickAt`、实例数、玩家数、quarantine 数。

硬规则：

- tick 不负责补启动恢复缺口。
- tick 遇到 lease_fenced 实例必须跳过 gameplay write，并触发隔离指标。

### Phase 13：开启 flush / outbox / worker

目标：只有在世界状态完整后，才允许异步写出和后台消费。

动作：

1. `flushOpen=true` 后启动玩家、地图、domain flush staging 和 consumer。
2. `outboxOpen=true` 后启动 outbox dispatcher。
3. `workerOpen=true` 后按 role 启动 background worker。
4. worker 启动前必须读取 `role` 和 `startupStatus`。

失败策略：

- flush consumer 不能启动：玩家流量不得开放。
- 非关键 worker 不能启动：可降级，但对应 job 标记 paused。
- outbox dispatcher 失败：交易/邮件/审计类功能按影响范围降级。

### Phase 14：开放 readiness 与 socket bootstrap

目标：对玩家开放前，所有权威闸门已经就绪。

动作：

1. 节点状态从 `starting` 更新为 `ready`。
2. `trafficOpen=true`。
3. health readiness 返回 ok 或 degraded-ok。
4. Socket gateway 开始接受 connect / hello / bootstrap。
5. 玩家在线 bootstrap 只允许进入已 ready 实例；如果目标实例隔离，返回明确重试/维护错误。

硬规则：

- readiness 不是“依赖看起来都活着”，而是启动状态机完成。
- socket bootstrap 不得触发地图实例租约抢跑。

### Phase 15：启动后审计

目标：启动成功后立即给运维可观测证据。

指标：

- 启动总耗时、各 phase 耗时。
- 恢复实例数、quarantine 实例数、租约失败数。
- 恢复玩家数、跳过玩家数、quarantine 玩家数。
- 首次 tick 时间、tick 健康状态。
- flush/outbox/worker 启动状态。
- gateway traffic open 时间。
- 通天塔层实例恢复数、租约成功数、被隔离数。

日志要求：

- 每个 phase 只打一条开始和一条完成日志。
- 大量实例/玩家失败输出采样 + GM/诊断接口详情，不刷屏。
- 所有隔离项带 `startupRunId`、`instanceId` / `playerId`、`reason`。

## 6. 启动闸门合同

所有服务只读闸门，不自行推断启动状态。

| 闸门 | 默认 | 打开阶段 | 消费方 |
|------|------|----------|--------|
| `trafficOpen` | false | Phase 14 | Gateway、HTTP 玩家入口 |
| `tickOpen` | false | Phase 12 | WorldTickService |
| `flushOpen` | false | Phase 13 | Player/Map/Domain flush |
| `outboxOpen` | false | Phase 13 | Outbox dispatcher |
| `workerOpen` | false | Phase 13 | Background workers |
| `instanceWritable(instanceId)` | false | Phase 7 单实例 hydrate 成功后 | gameplay write、flush snapshot |
| `instanceAttachAllowed(instanceId)` | false | Phase 10 前按实例打开 | player session attach |

闸门关闭时：

- gameplay write 必须拒绝或排队，不得静默执行。
- flush snapshot 必须跳过不可写实例。
- player attach 必须返回可诊断错误。
- worker 必须暂停消费，而不是忙循环重试。

## 7. 失败隔离策略

启动失败不能靠“继续启动后日志报警”兜底。必须按影响面分类。

| 失败类型 | 策略 |
|----------|------|
| DB 玩家真源不可用 | fail-fast |
| instance catalog 不可用 | 权威 runtime role fail-fast |
| 核心公共地图模板缺失 | fail-fast |
| 非核心副本 hydrate 失败 | 隔离实例，降级启动 |
| 实例租约失败 | 隔离实例，禁止 attach / write / flush |
| 玩家恢复失败 | 隔离玩家，不覆盖持久化快照 |
| flush consumer 失败 | fail-fast 或 maintenance，不开放玩家流量 |
| 非关键 worker 失败 | degraded，暂停对应 job |
| socket readiness 未完成 | 拒绝 bootstrap，不进入 runtime |

隔离对象必须能被 GM/诊断接口查询，并支持后续人工处理或自动重试。

## 8. 与当前通天塔租约问题的关系

日志中的问题：

```text
租约隔离命中但仍有在线玩家，已停止写入：lease_sync_failed players=...
```

在目标编排下，这类状态不应在正常启动后出现。正确链路是：

1. 通天塔 layer 进入 Phase 5 的统一实例恢复计划。
2. Phase 6 先获取该 layer 的实例租约。
3. Phase 7 再构造并 hydrate 通天塔 runtime。
4. Phase 10 玩家恢复前调用 `instanceReadyForPlayerAttach(layerInstanceId)`。
5. 若租约失败，该 layer 进入 `lease_quarantine`，玩家进入 `player_recovery_quarantine`。
6. 因为玩家没有 attach，后续不会出现“租约隔离命中但仍有在线玩家”。

必须消除的旧路径：

- 通天塔从 `claimRecoverableCatalogInstances()` 特判跳过。
- 玩家恢复时通过 `resolveTargetInstance()` 触发 `ensureLayerInstanceForRestore()` 创建/暴露 layer。
- `prepareRestoredLayerInstance()` 清空 lease 字段后标记 `runtimeStatus=running`。
- `restoreOfflineHangingPlayers()` 只在实例已创建后补 `syncInstanceLease()`，而不是在实例计划阶段先过 lease gate。

## 10. 迁移路线

### Step A：引入状态与闸门，不搬迁行为

- 新增 `StartupStatusService` 和 `StartupBarrierService`。
- 先让 health / GM 能看到启动状态。
- tick、flush、worker、gateway 增加只读闸门检查，但初始保持兼容默认。

### Step B：接管 loop 启动

- `WorldTickService.onModuleInit()` 不再直接 `scheduleNextTick()`。
- flush/outbox/background worker 不再在 `onModuleInit()` 直接启动定时器。
- 统一由编排器在 Phase 12/13 打开。

### Step C：接管世界恢复

- `WorldRuntimeService.onModuleInit()` 不再直接 bootstrap public instances。
- `WorldRuntimeService.onApplicationBootstrap()` 不再直接 `rebuildPersistentRuntimeAfterRestore()`。
- 编排器调用显式方法完成 public/catalog/sect/tower 恢复。

### Step D：统一实例计划和租约门禁

- 删除通天塔绕过 `claimRecoverableCatalogInstances()` 的启动特判。
- 所有持久实例先进入 `InstanceStartupPlan`。
- lease 成功才允许 runtime hydrate。

### Step E：统一玩家恢复准入

- `restoreOfflineHangingPlayers()` 改为消费 `PlayerStartupPlan`。
- 玩家 attach 前必须通过 `instanceReadyForPlayerAttach()`。
- 失败玩家进入 quarantine，不 fallback 覆盖位置。

### Step F：补证明链

至少补以下 smoke/proof：

- `startup-coordinator-order-smoke`：证明 phase 顺序和闸门默认关闭。
- `tower-lease-before-attach-smoke`：证明通天塔租约失败时玩家不会 attach。
- `offline-hanging-recovery-gate-smoke`：证明离线挂机恢复必须通过实例 ready gate。
- `readiness-blocks-before-runtime-ready-smoke`：证明 runtime ready 前 socket bootstrap 被拒绝。
- `flush-worker-wait-for-startup-smoke`：证明 flush/worker 不消费半恢复状态。

## 11. 完成标准

启动链路重整完成必须同时满足：

1. 没有任何玩家能在目标实例 lease + hydrate 前 attach。
2. tick 永远不会在世界恢复完成前推进。
3. flush/outbox/worker 永远不会消费半恢复 runtime。
4. socket bootstrap 永远不会绕过 readiness 进入 runtime。
5. 通天塔、宗门、公共地图、玩家拥有实例都走同一恢复计划和租约门禁。
6. 租约失败实例只会进入 quarantine，不会继续写入。
7. 玩家恢复失败不会覆盖持久化资产、位置或快照。
8. health / GM 能看到启动 phase、耗时、隔离对象和降级原因。
9. 相关 smoke/proof 可以复现并证明上述行为。
