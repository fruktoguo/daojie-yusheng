  1.1 完整目标判定流程

  目标引用协议：player:xxx / monster:xxx / tile:x:y / self / formation:xxx

  resolveAttackableTargetRef：

  1. 按前缀解析类型（player/monster/tile）
  2. PvP 目标必须满足 instance.meta.supportsPvp === true
  3. 调用 resolveCombatRelation 判定敌对关系
  4. 校验距离（切比雪夫） ≤ maxDistance（视野半径或技能射程）
  5. 校验目标 hp > 0、同实例

  PvP 关系判定（resolveCombatRelation）：

  - hostile / friendly / neutral / blocked 四态
  - 规则键：monster / all_players / demonized_players / retaliators / non_hostile_players / party / sect / terrain
  - 煞气系统：持有 PVP_SHA_INFUSION_BUFF_ID 且叠层 > PVP_SHA_DEMONIZED_STACK_THRESHOLD 且剩余息数 > 0 → 进入
  passivelyHostile（入魔），被动可攻击
  - 反击者标记：attacker.combat.retaliatePlayerTargetId === target.playerId → retaliator 命中
  - AoE 全域敌对：allowAoePlayerHit === true 时自动并入 all_players

  1.2 手动战斗流程

  普攻路径（dispatchBasicAttack）：

  1. 记录玩家活动（打断修炼）
  2. 打断制作活动（interruptCraftForReason）
  3. ensureAttackAllowed 检查（死亡/被禁锢）
  4. 判定伤害类型：spellAtk > physAtk 则 spell，否则 physical
  5. 基础伤害 = max(1, round(对应攻击力))
  6. resolvePlayerBasicAttackActionPlan 解析目标
  7. 分支派发：玩家 / 怪物 / 地块 / 阵法 / 容器
  8. resolveCombatHitForAction 结算
  9. emitCombatPresentation 推送表现（攻击轨迹、浮字、观察通知）
  10. applyMonsterCombatOutcome 或 applyPlayerDamage

  技能路径（castSkill / castSelfSkill / castSkillToMonster）：

  1. resolvePlayerSkill：在功法列表里查找 skill，校验解锁等级
  2. normalizeResolvedPlayerSkillCooldown：修正冷却（越界即清零）
  3. executeResolvedSkillCast：
    - 校验 HP（攻击者/目标都不能死）
    - 校验范围：distance > range 拒绝
    - 校验冷却：currentTick < readyTick 拒绝
    - 计算 qiCost = round(calcQiCostWithOutputLimit(plannedCost, maxQiOutputPerTick))，不足拒绝
    - setCooldownReadyTick：currentTick + resolveSkillCooldownTicks(cooldown)（受 cooldownSpeed 影响）

  4. 遍历技能 effects：
    - damage 类型：evaluateSkillFormula 求基础伤害 → resolveDamage
    - buff 类型：toTemporaryBuff → 施加 self/target buff

  5. 写回：扣元气、冷却、累加 totalDamage / totalRawDamage / damageRolls
  6. setRetaliatePlayerTarget：把自己设为目标的反击对象
  7. applyDamage：对目标应用总伤害

  1.3 吟唱系统（延迟技能）

  1.4 伤害计算完整公式（resolveCombatHit）

  按固定顺序：

  1) 破甲判定
     if breakPower > resolvePower:
       breakChance = ratioValue(breakPower, resolvePower + DEFAULT_RATIO_DIVISOR)
       broken = rng() < breakChance
     破甲后：hit ×= 2, crit ×= 2

  2) 闪避判定（战斗经验优势参与）
     attackerBonus/defenderBonus = resolveCombatExperienceAdvantage(...)
     effectiveHit = hit × (broken?2:1) × (1 + attackerBonus)
     effectiveDodge = dodge × (1 + defenderBonus)
     dodgeChance = ratioValue(effectiveDodge, effectiveHit + DEFAULT_RATIO_DIVISOR)
     if dodged: 返回 {hit:false, damage:0}

  3) 化解判定
     if resolvePower > breakPower:
       resolveChance = ratioValue(resolvePower, breakPower + DEFAULT_RATIO_DIVISOR)
       resolved = rng() < resolveChance
     化解后：defense ×= 2

  4) 暴击判定
     critChance = ratioValue(crit × (broken?2:1), antiCrit + DEFAULT_RATIO_DIVISOR)
     crit = rng() < critChance

  5) 元素加成
     if element: damage ×= percentModifierToMultiplier(elementDamageBonus[element])

  6) 防御减伤
     defense = (damageKind=='physical' ? physDef : spellDef) × (resolved?2:1)
     reduction = ratioValue(defense, max(1, attackBasis×0.1 + 100))
     if element:
       elementReduce = elementDamageReduce[element]
       reduction = 1 - (1-reduction)×(1-elementReduce)
     damage = max(1, round(damage × (1-reduction)))

  7) 暴击倍率
     if crit:
       critMultiplier = (200 + max(0,critDamage)/10) / 100
       damage ×= critMultiplier
       rawDamage ×= critMultiplier

  8) 境界差倍率
     realmGapMultiplier = getRealmGapDamageMultiplier(attackerRealmLv, targetRealmLv)
     damage ×= realmGapMultiplier

  9) 技能伤害倍率（damageMultiplier）
     damage ×= damageMultiplier（来自技能配置）

  战斗经验优势（resolveCombatExperienceBonus）：

  - baseline = COMBAT_EXPERIENCE_ADVANTAGE_BASELINE
  - 只有更多经验一方获得加成，最大 100%
  - 加成随 ratio 从 1 到 COMBAT_EXPERIENCE_ADVANTAGE_THRESHOLD 线性

  随机源：crypto.randomInt(0, 2^31)/2^31；smoke 测试可通过 setCombatRngForTesting 注入确定性 RNG。

  1.5 结果应用（combat-outcome-apply-adapters）

  按目标类型分发：

  Player 目标：

  1. setRetaliatePlayerTarget：目标被攻击后自动把攻击者标为反击对象
  2. applyPlayerDamage：扣血、入魔 buff 计算
  3. applyPlayerBuff：施加 debuff
  4. recordActivity(interruptCultivation:true)：打断修炼
  5. activateAutoRetaliate：目标开启自动反击
  6. 若 defeated：handlePlayerDefeat（掉落、复活、通知）

  Monster 目标：

  1. applyMonsterDamage：扣血，累加 damageContributors[playerId]
  2. aggroTargetPlayerId = attackerPlayerId（切换仇恨）
  3. applyMonsterBuff
  4. 若击败：handlePlayerMonsterKill（给攻击者加经验、掉落判定、任务进度、宗门贡献）

  Tile 目标：

  1. damageTile：扣 HP，记录 tileDamageByTile
  2. 若破坏：标记 destroyed，触发宗门扩圈 expandSectForDestroyedTile
  3. 按 shouldRecalculateRoomsForTileMutation 决定重算房间/风水

  Formation 目标：

  1. applyDamageToBoundaryBarrier（边界屏障）或 applyDamageToFormation（阵眼）
  2. 护宗大阵、阵法等经过同一分支

  Container 目标：

  1. 可攻击容器（宝箱、灵草）：damageAttackableContainerAtTile / damageHerbContainerAtTile
  2. 耗尽 → consumed=true；记录重生时间 respawnRemainingTicks

  1.6 自动战斗完整流程

  入口：materializeAutoCombatCommands（每帧 tick 编排里调用）

  跳过条件：

  - 已有 pendingCommand 或 navigationIntent → 跳过
  - HP ≤ 0 → 跳过
  - 有 pendingSkillCast → 跳过（正在吟唱）
  - autoBattle / autoRetaliate / manualEngagePending 都为假 → 跳过
  - 在安全区（isSafeZoneTile）→ 关闭 autoBattle，清空目标，发提示

  目标选择（selectAutoCombatTarget）：

  1. 反击抢占：resolveRetaliatePlayerOverrideTarget — 被玩家攻击时临时抢占为反击目标（不改写锁定）
  2. 锁定目标跟踪：resolveTrackedAutoCombatTarget — 锁定 combatTargetId 且仍可达
  3. 候选收集：
    - 可见怪物（instance.buildPlayerView(radius)）：过滤仇恨我 / 同关系规则命中
    - 可见玩家：过滤敌对关系（PvP）

  4. 评分公式：

     score = (aggroRank × 100000)       # 反击我的排最前
           + (priority × 10000)          # retaliators=3, demonized=1
           + (1000 / (distance + 1))    # 距离近加分
           + (1 - hpRatio) × 100         # 残血加分
           × targetingModePreferenceMultiplier(5×)

  5. 偏好乘区：nearest / low_hp / full_hp / boss / player 五种模式，命中加 5×
  6. 平分破除：距离近 → 字典序 tieBreaker 稳定

  行动选择（buildAutoCombatCommand）：

  1. 距离 ≤ 1 且无可施放技能 → 普攻
  2. 有可用技能（resolveAutoBattleSkillChoice）→ castSkill
  3. 距离 > 1 且技能射程不足 → 寻路到 desiredRange 内：
    - findPathToTargetWithinRangeOnMap
    - 连续移动 maxSteps = resolveInitialRunLength

  4. autoBattleStationary === true → 不移动，只原地战斗
  5. 单息行动预算：actionsPerTurn 控制每 tick 可执行的 basicAttack/castSkill 次数；超过则跳过

  2.1 关键常量

  MAX_ENHANCE_LEVEL = 20
  ENHANCEMENT_BASE_JOB_TICKS = 5
  ENHANCEMENT_JOB_TICKS_PER_ITEM_LEVEL = 1
  ENHANCEMENT_LOWER_LEVEL_SUCCESS_PENALTY = 0.1
  ENHANCEMENT_EXTRA_SUCCESS_RATE_PER_LEVEL = 0.002
  ENHANCEMENT_EXTRA_SPEED_RATE_PER_LEVEL = 0.02
  ENHANCEMENT_INTERRUPT_PAUSE_TICKS = 10

  ENHANCEMENT_TARGET_SUCCESS_RATE_BY_LEVEL =
    [0.5, 0.45, 0.45, 0.4, 0.4, 0.4,
     0.35, 0.35, 0.35, 0.35,
     0.3, 0.3, ..., 0.3]  // 按目标等级索引

  2.2 发起流程

  dispatchStartEnhancement：

  1. 取玩家快照
  2. 判断是否启用 durable（runtimeOwnerId 非空、sessionEpoch > 0）
  3. 启用 → dispatchStartEnhancementDurably（commit-gated）
  4. 不启用 → craftPanelRuntimeService.startTechniqueActivity 直接走内存态

  startEnhancement 详细校验：

  1. resolveEnhancementTarget：从装备槽或背包槽解析
  2. item.type !== 'equipment' → 拒绝
  3. enhanceLevel >= MAX_ENHANCE_LEVEL → 拒绝
  4. hasAnyActiveTechniqueActivity → 转入队列（enqueueCraftQueueItem）
  5. getEnhancementRequirements：按目标等级拉材料清单
  6. resolveEnhancementProtection：校验保护物
  7. getEnhancementSpiritStoneCost(itemLevel, hasMaterials)：灵石成本
  8. hasEnoughEnhancementResources：综合检查灵石 / 材料 / 保护物
  9. 把装备从背包取出（extractInventoryItemAt）或从装备栏引用
  10. 扣材料（灵石不扣，成功时扣）
  11. 构造 player.enhancementJob：

      {jobRunId, jobType:'enhancement', target, item,
       targetItemId/Name/Level, currentLevel, targetLevel, desiredTargetLevel,
       spiritStoneCost, materials, protectionUsed/ItemId/Name/Signature, protectionStartLevel,
       phase:'enhancing', pausedTicks:0,
       successRate, totalTicks, remainingTicks,
       roleEnhancementLevel, totalSpeedRate, jobVersion:1, queuedJobs}

  12. touchEnhancementRecord：写历史记录（in_progress 状态）
  13. finalizeMutation：标记 dirty domains: active_job + enhancement_record

  2.3 成功率 / 速率公式

  baseSuccessRate = ENHANCEMENT_TARGET_SUCCESS_RATE_BY_LEVEL[targetLevel-1]
  equipmentLevelModifier =
    itemLevel × ENHANCEMENT_EXTRA_SUCCESS_RATE_PER_LEVEL    # 高级装备加成
  hammerBonus = 强化锤.enhancementSuccessRate (可选)
  roleBonus = 强化技艺等级修正
  successRate = clamp(baseSuccessRate + equipmentLevelModifier + hammerBonus + roleBonus, 0, 1)

  totalSpeedRate =
    1 + itemLevel × ENHANCEMENT_EXTRA_SPEED_RATE_PER_LEVEL
      + 强化锤.enhancementSpeedRate
      + roleEnhancementLevel 修正
  jobTicks = (ENHANCEMENT_BASE_JOB_TICKS + itemLevel × ENHANCEMENT_JOB_TICKS_PER_ITEM_LEVEL) / totalSpeedRate

  2.4 tick 推进（tickEnhancement）

  每息：

  1. remainingTicks -= 1
  2. phase === 'paused' → advanceTechniqueActivityPause 尝试恢复 enhancing
  3. remainingTicks > 0 → 持续等待
  4. remainingTicks === 0 → 结算：
    - success = Math.random() < job.successRate
    - 成功：debitWallet(spirit_stone, spiritStoneCost)（扣灵石）
    - 失败且保护生效：消耗保护物 → resultingLevel = currentLevel - 1
    - 失败且无保护：resultingLevel = 0（归零！）
    - touchEnhancementLevelRecord：写单级结果
    - 经验结算：resolveEnhancementSkillExpGain → applyCraftSkillExp

  5. resultingLevel < desiredTargetLevel → advanceEnhancementJob 继续下一级（自动连打）
  6. 达到 desiredTargetLevel 或中止 → finishEnhancementJob，从 queuedJobs 拉下一个任务

  2.5 保护机制

  - shouldUseProtectionForStep(targetLevel, protectionStartLevel)：只在 targetLevel >= protectionStartLevel 时生效
  - 失败且保护生效 → 只降 1 级（不归零）
  - 失败且保护生效 → consumeProtectionItemForFailure：消耗保护物
  - 保护物不足 → finishEnhancementJob('stopped') 终止

  2.6 队列机制

  queuedJobs：按 enqueueCraftQueueItem 入队

  - queueMode: 'append'：追加到队尾
  - queueMode: 'replace'：清空队列，替换
  - queueMode: 'preserve'：追加到当前任务之后

  任务完成后 startNextQueuedCraftJob 递归启动下一项。

  2.7 中断规则

  interruptEnhancement(player, reason)：

  - 移动 / 被攻击 / 使用物品等触发
  - applyTechniqueActivityInterrupt(job, ENHANCEMENT_INTERRUPT_PAUSE_TICKS) → phase='paused', pausedTicks += 10
  - 推进时 advanceTechniqueActivityPause 倒计时后恢复 enhancing
  - 已扣材料不退还

  2.8 取消规则

  cancelEnhancement：

  - 立即 finishEnhancementJob(currentLevel, 'cancelled')
  - 本阶材料不退、灵石本阶未结算就不扣、保护物仅失败时扣

  2.9 Durable Operation 三阶段

  启动 startActiveJobWithAssets：

  - operationId = op:${playerId}:enhancement-start:${jobRunId}
  - 期望值校验：runtimeOwnerId / sessionEpoch / instanceId / assignedNodeId / ownershipEpoch
  - 下一份快照：inventory + wallet + activeJob + enhancementRecords

  更新 updateActiveJobState(action:'update')：每 tick 推进时

  完成 completeActiveJobWithAssets：

  - 下一份快照：inventory + wallet + equipmentSlots + enhancementRecords
  - expectedJobVersion = resolveCompletedEnhancementJobVersion

  取消 cancelActiveJobWithAssets：回滚到 captureEnhancementRollbackState 快照

  幂等性：durable_operation_log 表按 operationId 主键；重复调用直接返回 alreadyCommitted:true

  回滚：try 失败 → restoreEnhancementRollbackState 恢复内存态

  3. 炼丹 / 炼器链路

  核心文件：同强化，加 world/world-runtime-alchemy.service.ts、craft-panel-alchemy-query.service.ts

  3.1 关键常量

  ALCHEMY_PREPARATION_TICKS = 10     # 开炉前准备息数
  ALCHEMY_INTERRUPT_PAUSE_TICKS = 10 # 中断暂停息数
  ALCHEMY_FURNACE_OUTPUT_COUNT       # 单炉输出数（shared 常量，通常是 3）

  3.2 配方系统

  - alchemyCatalog / forgingCatalog：从内容包加载，每条 {recipeId, outputItemId, outputName, outputLevel, outputCount,
  ingredients[{itemId, options[], count}], category, baseBrewTicks}
  - category === 'buff'：buff 类丹药，单炉只出 1 个
  - 正常丹药：单炉出 ALCHEMY_FURNACE_OUTPUT_COUNT 个

  3.3 发起流程（startAlchemy）

  1. ensureCraftSkills：确保有技艺状态
  2. 查找配方 → 不存在 → 拒绝
  3. validateAlchemySelection：材料选择合法性
  4. normalizeQuantity(1, 99)：炉次 1~99
  5. hasAnyActiveTechniqueActivity → 入队列
  6. 材料数量检查：countInventoryItem(player, ingredient.itemId) >= ingredient.count × quantity
  7. spiritStoneCost = getAlchemySpiritStoneCost(outputLevel, isBuff) × quantity
  8. 钱包检查：canAffordWallet(playerId, 'spirit_stone', spiritStoneCost)
  9. 扣材料：consumeInventoryItemByItemId 每 ingredient × quantity
  10. 扣灵石：debitWallet(spirit_stone, spiritStoneCost)（一次性全扣）
  11. furnaceOutputCount = forging/buff 时 1，否则 ALCHEMY_FURNACE_OUTPUT_COUNT
  12. 成功率：

      baseSuccessRate = computeAlchemySuccessRate(recipe, ingredients)
      adjustedSuccessRate = computeAlchemyAdjustedSuccessRate(
        baseSuccessRate, outputLevel, alchemySkillLevel, toolSuccessRate)

  13. 单炉息数：

      batchBrewTicks = computeAlchemyAdjustedBrewTicks(
        baseBrewTicks, recipe, ingredients, outputLevel,
        alchemySkillLevel, toolSpeedRate, furnaceOutputCount)

  14. 总息数：

      totalTicks = computeAlchemyTotalJobTicks(
        batchBrewTicks, quantity, ALCHEMY_PREPARATION_TICKS)

  15. 单炉产量（乘算炉输出数）：

      batchOutputCount = computeAlchemyBatchOutputCountWithSize(
        outputCount, furnaceOutputCount)

  16. exactRecipe = isExactAlchemyRecipe(recipe, ingredients)：材料是否精确匹配
  17. 构造 player.alchemyJob：

      {jobRunId, jobType, recipeId, outputItemId, outputCount, quantity,
       completedCount:0, successCount:0, failureCount:0,
       ingredients, phase:'preparing', preparationTicks, batchBrewTicks,
       currentBatchRemainingTicks, pausedTicks:0,
       spiritStoneCost, totalTicks, remainingTicks,
       successRate, jobVersion:1, exactRecipe, outputLevel, baseBrewTicks,
       startedAt, queuedJobs}

  3.4 tick 推进（tickAlchemy）

  phase 状态机：preparing → brewing → paused（可从任何 phase 触发）

  1. remainingTicks -= 1
  2. paused → advanceTechniqueActivityPause 恢复 brewing 或 preparing
  3. preparing → 计算 brewTicksRemaining = (quantity - completedCount) × batchBrewTicks；若 remainingTicks <=
  brewTicksRemaining → 切到 brewing，重置 currentBatchRemainingTicks = batchBrewTicks
  4. brewing → currentBatchRemainingTicks -= 1
  5. currentBatchRemainingTicks === 0 && remainingTicks === 0（或 batch 耗尽）→ 结算：

     successCount = resolveAlchemyBatchSuccess(batchOutputCount, successRate)
     failureCount = batchOutputCount - successCount
     completedCount += 1
     successCount / failureCount 累加到 job

  6. 背包空间不足 → 掉落到地面（groundDrops）
  7. 技艺经验：

     skillGain = resolveAlchemySkillExpGain(catalog, job, skill, success, failure)
     applyCraftSkillExp(skill, skillGain, resolveCraftSkillExpToNextByLevel)

  8. jobCompleted = completedCount >= quantity || remainingTicks <= 0
  9. 完成 → 从 queuedJobs 启动下一项；否则 currentBatchRemainingTicks = batchBrewTicks 继续

  3.5 取消（cancelAlchemy）

  按剩余炉次按比例退款：

  refundableBatchCount = max(0,
    quantity - completedCount - (phase === 'brewing' ? 1 : 0))

  - 当前正在 brewing 的炉不退
  - 剩余未开炉：逐 ingredient refundCount = count × refundableBatchCount 退回背包；装不下掉地上
  - 灵石按比例退：spiritStoneCost × (refundableBatchCount / quantity)

  3.6 中断（interruptAlchemy）

  - 暂停 ALCHEMY_INTERRUPT_PAUSE_TICKS = 10 息
  - phase 切到 paused
  - 原因来自：移动 / 被攻击 / GM 强制等

  3.7 炼器差异

  - startForging = startAlchemy({kind:'forging'}) 复用整条流水
  - forging 和 buff 类：单炉固定 1 件
  - 催化剂、成功率、速率工具（强化锤可兼作）机制相同

  3.8 预设系统

  - saveAlchemyPreset：保存配方 + 材料选择的预设，写到 player.alchemyPresets
  - deleteAlchemyPreset：按 presetId 删除
  - 立即 persistAlchemyPresets 落库（dirtyDomains: ['alchemy_preset']）

  4.1 协议入口

  - C2S.NavigateQuest：按任务 ID 导航

  4.2 导航意图系统

  数据结构：navigationIntents: Map<playerId, intent>

  Intent 类型：

  - {kind: 'point', mapId, x, y, allowNearestReachable, clientPathHint}
  - {kind: 'quest', questId}

  覆盖语义：同玩家一息内多次 setNavigationIntent → 只保留最后一次

  清理时机：

  - 到达目标 → kind:'done'
  - 手动 enqueueMove → clearNavigationIntent
  - 跨图传送 → handleTransfer 按需清理
  - 玩家下线 / HP=0 → 删除

  4.3 物化流程（materializeNavigationCommands）

  tick 编排最开始执行：

  for (playerId, intent) of navigationIntents:
    if hasPendingCommand(playerId): skip
    if !player || !instanceId || hp<=0: delete & skip
    step = resolveNavigationStep(playerId, intent)
    switch step.kind:
      'done'   → delete & skip
      'portal' → enqueuePendingCommand({kind:'portal'})
      'move'   → enqueuePendingCommand({
                   kind:'move', direction,
                   continuous:true, maxSteps, path, resetBudget:false})

  4.4 路径规划

  resolveNavigationStep：

  1. 解析 destination = {mapId, goals}
  2. 如果 destination.mapId !== player.templateId → 跨图寻路：
    - findMapRoute(fromMapId, toMapId)：BFS 整张地图拓扑
    - 取 route[1] 作为下一跳地图
    - selectNearestPortal(portals, nextMapId, x, y)：曼哈顿距离选最近界门
    - 玩家已站在界门上 → kind:'portal'
    - 否则 findOptimalPathOnMap → portal

  3. 同图：
    - 目标已到 → kind:'done'
    - resolvePreferredClientPathHint：优先复用客户端提示路径
    - 否则 findOptimalPathOnMap → destination.goals
    - directionFromStep(currentX, currentY, nextX, nextY) 解析方向
    - 返回 {kind:'move', direction, maxSteps, path}

  4.5 A* 算法细节

  findOptimalPathOnMap(instance, playerId, startX, startY, goals, allowOccupiedGoals=true)：

  1. 过滤无效 goal（出界）
  2. buildPathingBlockMask(instance, playerId, goals, allowOccupiedGoals)：
    - instance.forEachPathingBlocker 收集所有阻挡（其他玩家、怪物、建筑、阻挡地块）
    - 若 allowOccupiedGoals → 把目标格从 blocked 移除

  3. 使用 Float64Array(size) 存 bestCost，Int32Array(size) 存 previous
  4. 最小堆 heap（用 pushPathNode / popPathNode）
  5. 节点优先级：priority = cost + chebyshev(x, y, goals)（启发函数）
  6. PATH_DIRECTION_STEPS：4 方向（N/S/E/W）
  7. 每步代价：instance.getTileTraversalCost(x, y, playerId)（地块基础代价）
  8. 命中任一 goalIndex → reconstructPathPoints 回溯路径
  9. 堆尽空无解 → 返回 null

  findPathToTargetWithinRangeOnMap（自动战斗接近用）：

  - isStopNode(x,y) = chebyshev(x, y, target) <= stopDistance（不含目标本格）
  - 到达停止距离内任一格即返回

  buildGoalPoints(instance, targetX, targetY, allowNearestReachable, playerId)：

  - 目标本格可走 → 返回 [target]
  - 否则 allowNearestReachable=true → 在半径 1~8 内搜索可走格，按曼哈顿距离排序

  客户端路径提示复用（resolvePreferredClientPathHint）：

  1. clientPathHint.startX/Y === current 或 hint.points 里有 current → 从该位置后截取
  2. 最后一点必须匹配某 goal
  3. 每步必须单格移动、可走、不阻挡、代价有限
  4. 满足则直接返回（省掉 A*）

  4.6 移动预算

  resolveInitialRunLength(path, startX, startY, direction)：

  - 从 startX/Y 出发，沿着给定 direction 找连续同方向的步数
  - 返回连续段长度（作为 maxSteps）

  MapInstanceRuntime.player.moveSpeed：

  - 由 dispatchMoveCommand 调用 instance.setPlayerMoveSpeed(playerId, attrs.numericStats.moveSpeed) 同步
  - tick 内按 movePoints 逐格推进
  - 占位冲突 → 停在前一格
  - lastMoveBudgetTick：跨 tick 预算累计

  4.7 传送门系统

  分类：

  - trigger: 'manual' → 玩家按交互键触发
  - trigger: 'auto' → 踩到自动触发

  dispatchPortalCommand：

  1. interruptCraftForReason('move') 打断制作
  2. 打断吟唱
  3. tryPortalTransfer('manual_portal')（交互界门在距离 ≤ 2 内）
  4. 失败则 tryPortalTransfer('auto_portal')（当前格自动门）
  5. 还失败 → 实例 enqueuePortalUse

  tryPortalTransfer：

  - manual_portal → getInteractablePortalNear(x, y, SECT_ENTRANCE_INTERACTION_RADIUS=2) 且 portal.trigger === 'manual'
  - auto_portal → getPortalAt(x, y) 且 portal.trigger === 'auto'
  - 命中则 buildTransfer(player, portal, reason) 返回 Transfer 对象

  applyTransfer：

  1. 目标线路偏好（peaceful / real）
  2. getInstanceRuntime(targetInstanceId) 或 getOrCreateDefaultLineInstance
  3. 玩家从旧实例 disconnect → 新实例 connect(preferredX, preferredY)
  4. 清理 pendingCommand / navigationIntent / pendingSkillCast
  5. 推送 queuePlayerNotice('通过界门抵达 XXX')

  4.8 移动对其他系统的影响

  1. instance.setPlayerMoveSpeed(playerId, moveSpeed)

  6. 市场 / 交易 / 拍卖链路

  核心文件：

  6.1 架构

  内存态：

  - openOrders: MarketOrder[]：挂单列表
  - tradeHistory：成交记录
  - storageByPlayerId: Map<playerId, MarketStorage>：坊市仓库（未领取的成交所得）
  - auctionBidsByItemKey: Map<itemKey, AuctionBid[]>：拍卖出价
  - auctionTimingByItemKey: Map<itemKey, {endAtMs}>：拍卖截止

  并发控制：

  - marketOperationQueue = Promise.resolve() 串行链
  - 所有写操作必须通过 runExclusiveMarketMutation(playerId, action)

  6.2 订单数据结构（MarketOrder）

  {id, ownerId, side('buy'|'sell'), item, itemKey,
   unitPrice, quantity, remainingQuantity,
   status('open'|'filled'|'cancelled'),
   createdAt, updatedAt,
   listingMode('market'|'auction'),
   auction?: {mode, startedAt, endAtMs, startingUnitPrice, ...}}

  6.3 挂单流程

  createSellOrder（runExclusiveMarketMutation 包装）：

  1. 校验 listingMode / 数量 / 单价 / 持有量 / 不能挂坊市货币
  2. normalizeUnitPrice / normalizeQuantity
  3. isValidMarketTradeQuantity(unitPrice, quantity)：价格-数量规则（高价物最小量 1）
  4. toOrderItem + buildItemKey
  5. hasConflictingOpenOrder(playerId, itemKey, 'sell')：同玩家不能同时挂售+求购同物品
  6. captureOnlinePlayerState：写 snapshot 到 context
  7. splitInventoryItem(slotIndex, quantity)：把物品从背包切出
  8. 即时撮合：
    - 过滤所有 buy 订单中 ownerId !== playerId && unitPrice >= 挂售价
    - 按 unitPrice DESC 排序
    - planOrderMatches：按顺序匹配直到数量耗尽

  9. 逐 match 成交：
    - deliverItemToPlayer(buyerId, {...item, count:matchQty}) → 买家坊市仓库
    - deliverMarketCurrencyToPlayer(sellerId, totalCost) → 卖家坊市仓库
    - recordTrade：写成交历史
    - sellOrder.remainingQuantity -= matchQty
    - touchAffectedPlayer(result, buyerId) → 推送通知

  10. 剩余未撮合 → openOrders.push(newSellOrder)

  createBuyOrder：

  1. 先扣灵石：consumeMarketCurrencyFromInventory(playerId, totalCost)
  2. 即时撮合对方 sell 单
  3. 剩余挂单

  6.4 一口价（buyNow / sellNow）

  buyNow：

  1. 找到所有 ownerId !== playerId 的卖单
  2. 按 unitPrice ASC 排序
  3. planOrderMatches(sells, quantity) 试算
  4. 数量不足 → 拒绝
  5. 扣灵石 → 逐 match 成交
  6. durable 分支：settleMarketBuyNow commit-gated：
    - 期望值校验
    - nextBuyerInventoryItems / nextBuyerWalletBalances / matches[]
    - 每 match 有 nextSellerInventoryItems
    - commit 后统一 replaceInventoryItems / creditWallet

  7. 卖单剩余归零 → status='filled' + deleteOrder

  sellNow：对称，匹配买单

  6.5 撤单（cancelOrder）

  1. runExclusiveMarketMutation
  2. 找到 order，校验 ownerId === playerId && status === 'open'
  3. 剩余数量返还：
    - 卖单：把 remainingQuantity 件物品放回仓库
    - 买单：把 remainingQuantity × unitPrice 灵石放回仓库

  4. status = 'cancelled' → deleteOrder
  5. durable 分支：settleMarketCancelOrder

  6.6 拍卖行

  挂拍：createSellOrder({listingMode:'auction'}) → openOrders.push(order)；auctionTimingByItemKey.set(itemKey, {endAtMs:now
  + duration})

  出价（placeAuctionBid）：

  1. 校验拍品存在 / 未结束
  2. minBidPrice = getAuctionMinimumBidPrice(currentPrice)（加价最小步长）
  3. quantity = getMarketMinimumTradeQuantity(unitPrice)（最小成交数）
  4. totalCost = calculateMarketTradeTotalCost(quantity, unitPrice)
  5. 最高价冻结（reservedCost）：只有最高价出价者的灵石被冻结；被超过时退还
  6. 冻结差额：debitCost = totalCost - previousReservedCost（如果之前是最高价，只补差价）
  7. consumeMarketCurrencyFromInventory(playerId, debitCost)
  8. 被挤下的前最高价 → deliverMarketCurrencyToPlayer（退还到仓库）
  9. 30 秒延时窗口：extendAuctionIfEndingSoon：
    - 结束前 AUCTION_EXTENSION_WINDOW_MS = 30s 内出价 → 延长到 now + 30s
    - 累计延长不超过 AUCTION_MAX_EXTENSION_MS = 60min

  10. auctionBidsByItemKey.set(itemKey, sortedBids)
  11. persistAuctionStateToCarrier(itemKey, context)

  一口价（buyoutAuctionLot）：

  1. buyoutUnitPrice = max(sellOrder.unitPrice, highestBid.unitPrice)
  2. 买家如已出过价，buyerReservedCost 可抵扣
  3. 所有非自己的出价者退还 reservedCost
  4. 成交：卖家拿钱，买家拿物品

  到期结算（settleExpiredAuctionLots）：

  - 定期扫描 auctionTimingByItemKey 找出 endAtMs <= now
  - 最高出价者成交
  - 无出价 → 返还卖家（status='cancelled'）

  6.7 坊市仓库

  - 成交所得先进 storageByPlayerId（不直接到背包，避免背包满导致交易失败）
  - claimStorage(playerId)：领取到背包 / 钱包
  - 离线玩家成交 → 直接存仓库，上线后领取

  6.8 启动恢复

  onApplicationBootstrap → reloadFromPersistence：

  - marketPersistenceService.loadOpenOrders → openOrders
  - tradeHistory / storage
  - hydrateAuctionStateFromOpenOrders：从 order.raw_payload 恢复 auctionBidsByItemKey / auctionTimingByItemKey

  8.4 recalculate 与 hp/qi 按比例缩放

  previousMaxHp = max(1, round(player.maxHp))
  previousMaxQi = max(0, round(player.maxQi))

  if !hasAttrStateChanged: return false

  player.maxHp = nextMaxHp
  player.maxQi = nextMaxQi
  player.hp = previousMaxHp > 0
    ? clamp(round(hp / previousMaxHp × nextMaxHp), 0, nextMaxHp)

  player.attrs.revision += 1

  9.2 Tick 编排（advanceFrame）

  每帧流程（advanceFrame(deps, frameDurationMs=1000, getInstanceTickSpeed)）：

  1. worldRuntimeCombatEffectsService.resetFrameEffects
  2. 收集 instanceStepPlans（本帧每实例要执行几步逻辑 tick）：
    - 读 lease：isInstanceLeaseWritable 检查（多节点租约）
    - speed = getInstanceTickSpeed(templateId)（默认 1）
    - accumulated = previousProgress + speed × (frameDurationMs/1000)
    - steps = floor(accumulated)，余数进 progress
    - plannedLogicalTicks += steps

  3. plannedLogicalTicks === 0 → recordIdleFrame，返回 0
  4. 预处理阶段：
    - processPendingRespawns：处理怪物/掉落重生
    - materializeNavigationCommands：导航意图 → 移动指令
    - materializeAutoUsePills：自动吃药
    - materializeAutoCombatCommands：自动战斗指令

  5. 派发阶段：
    - dispatchPendingCommands：执行所有排队命令（移动、普攻、技能、使用物品、装备、坊市、邮件、制作、建造、宗门、传送等）
    - dispatchPendingSystemCommands：系统命令（GM、自动重生等）

  6. 实例 tick 循环：

     for each instance, step {
       deps.tick += 1
       totalLogicalTicks += 1
       isInstanceLeaseWritable check
       isFormationTerrainStabilized = formation 固脉阵检查
       result = instance.tickOnce()
       instance.advanceTileResourceFlow()
       formation.advanceInstanceFormations
       instance.advanceTemporaryTiles
       instance.advanceTileRecovery  # 地块自然修复
       for building of result.completedBuildings: notifyBuildingConstructionCompletion
       for transfer of result.transfers: deps.applyTransfer
       for action of result.monsterActions: deps.applyMonsterAction
       currentPlayerIds = instance.listPlayerIds()
       syncWorldTimeVisionForPlayers  # 更新世界时辰 buff
       cultivationAuraMultiplier = buildCultivationAuraMultiplierByPlayerId
       playerRuntimeService.advanceTickForPlayerIds(playerIds, instance.tick, {...})
       applyTileQiDrainForPlayers  # 修炼消耗灵气
       for playerId: worldRuntimePlayerSkillDispatchService.resolvePendingPlayerSkillCast  # 吟唱结算
       worldRuntimeCraftTickService.advanceCraftJobs(playerIds)  # 制作推进
       worldRuntimeTongtianTowerService.advanceInstance  # 通天塔
     }

  7. 后处理：
    - worldRuntimeTongtianTowerService.cleanupIdleInstances
    - worldRuntimeLootContainerService.advanceContainerSearches
    - 每个玩家 refreshQuestStates
    - worldRuntimeMetricsService.recordFrameResult（pendingCommandsMs / systemCommandsMs / instanceTicksMs /
  playerAdvanceMs）

  10.4 Durable Operation（强一致写）

  用于资产/市场/邮件/制作/商店的关键写链路。DurableOperationService 提供 commit-gated 三阶段 API：

  1. start   —— 写 durable_operation_log 行，状态=pending，携带 expectedSessionEpoch/expectedInstanceId/expectedNodeId 等
  fencing 条件
  2. update  —— 业务 Tx：扣资产、发物品、记录 outbox_event
  3. commit  —— 置 log 状态 = committed；或 rollback 清理

  10.5 幂等保护

  每次请求必须携带 operationId（如 market-buy-now:<playerId>:<ts>:<uuid>）：

  - durable_operation_log 以 operationId 为唯一主键
  - 重放同 operationId → 直接返回上次结果（alreadyCommitted），不再写入
  - 崩溃恢复时，扫描 pending 行重跑或补偿

  10.6 Session Fencing 与 Instance Lease

  - Session Epoch：玩家每次登录 sessionEpoch++，旧 session 请求带旧 epoch 会被拒
  - Instance Lease：每张地图实例由单 assignedNodeId 独占 + ownershipEpoch
  - Durable 请求校验 expectedAssignedNodeId / expectedOwnershipEpoch / expectedRuntimeOwnerId，避免多节点同时写同一玩家/实例

  10.7 Outbox 事件

  outbox_event 表：业务落盘后需通知外部（社交、审计、统计）的事件排队表：

  - outbox-dispatcher.service 定期拉取未发送事件
  - 失败重试 attempt_count++；超限进入 DLQ
  - 消费者通过 outbox-event-consumer-registry 注册

  10.8 审计日志与清理

  - asset_audit_log — 资产变更审计（发放/消耗/交易/邮件附件/GM 操作）
  - asset-audit-log-retention.worker — 按保留期归档到 asset_audit_log_archive
  - mail-expiration-cleanup.worker — 过期邮件清理
  - mail-soft-delete-purge.worker — 软删除邮件彻底清理
  - instance-state-purge.worker — 废弃实例状态清理
  - checkpoint-compaction.worker — 快照压缩
  - instance-ground-item-ttl.worker — 地面掉落 TTL 过期

  10.9 服务启动恢复链

  启动期按顺序：

  1. deploy-database-preflight 预检 schema
  2. 各 *Persistence 服务 onApplicationBootstrap 回填内存态
  3. durable_operation_log 重放 pending 操作
  4. outbox-dispatcher 重启未投递事件
  5. instance-catalog 恢复实例 lease，绑定到当前节点

  12. 建筑/风水/宗门链路

  12.1 建筑系统

  核心文件：runtime/world/world-runtime-building.service.ts · runtime/instance/map-instance.runtime.ts（建筑段） ·
  runtime/building/*

  放置流程

  handleBuildPlaceIntent → 材料/灵石校验 → placeBuildingInstance → 改写地块 → 房间/风水重算

  1. 请求去重：operationKey = place:<playerId>:<requestId>，重复请求返回缓存结果
  2. 材料选择：resolveSelectedBuildingCost(player, compiled, selectedMaterialItemIds) ——
  客户端可选用的材料由建筑定义允许；计算总消耗
  3. 建造强度：buildStrength（1~N），越高施工越慢但上限 HP 越高
  4. 位置校验：placeBuildingInstance 检查地块是否可放置、是否占用、朝向、大小
  5. 初始状态：state=building、hp=0、buildRemainingTicks=buildStrength
  6. 材料消耗：consumeBuildingCost 从背包扣除；失败回滚已放置实体
  7. 拓扑更新：若建筑影响房间边界（墙/门/窗），触发 recalculateRoomsAndFengShuiAfterTopologyChange；仅影响影响圈时局部重算

  施工推进

  1. 玩家到建筑格位 startBuildingConstruction → activeBuilderPlayerId = playerId
  2. 每 tick advanceBuildingConstruction：活跃 builder 在现场则 buildRemainingTicks -= 1
  3. buildRemainingTicks == 0 → state=active、hp=maxHp、触发 completedBuildings 事件
  4. 完成回调：notifyBuildingConstructionCompletion 给建造者发成就/技艺经验
  5. 离开现场 stopBuildingConstruction → 进度保留但暂停推进

  拆除

  deconstructBuildingInstance：

  - 恢复 previousTileType
  - 清理 buildingCellsById / buildingIdByCell
  - 影响房间边界则触发全量重算，否则局部

  12.2 房间检测

  核心文件：runtime/building/room-detection.service.ts · runtime/building/building-topology-index.service.ts

  - 基于 4-连通泛洪填充：封闭区域检测为房间
  - 检测条件：地块是 floor/walkable、边界有墙/门/窗
  - RoomInstance 字段：id / area / perimeter / doorCount / windowCount / roofCoverage / traits
  - 房间变更触发 roomIdByCell 更新和 fengShuiByRoomId 重算

  12.3 风水系统

  核心文件：runtime/building/fengshui-calculator.service.ts

  评分维度

  RoomAggregate：

  - elementVector[5] — 金木水火土含量
  - traitCounts — 家具/装饰/功能设施数量
  - comfort / stability — 舒适/稳定度
  - qiRaw — 房内灵气总量（累加 auraByTile）
  - shaRaw — 煞气总量
  - integrityPenalty — 地块破损折损（按 (1-hp/maxHp)*30 累计）
  - formationScore — 阵法加成

  计算流程

  calculateFengShuiSnapshot(room, aggregate, rules, ctx)：

  1. 基础分 FENGSHUI_BASE_SCORE
  2. 遍历规则：每条 FengShuiRuleDef 判定 when（角色/封闭/特征/指标/元素），匹配则 scoreDelta
  3. capGroup 分组封顶，避免单一规则刷高
  4. 五行生克：主元素与功能元素比对，相生加分 / 相克扣分
  5. 最终分数在 [FENGSHUI_SCORE_MIN, FENGSHUI_SCORE_MAX] 区间
  6. 按 FENGSHUI_GRADE_THRESHOLDS 映射为等级（凶/平/吉/大吉/至吉）

  影响范围

  - 玩家在吉位房间：获得 qi_regen_rate / realm_exp_per_tick 等 buff
  - 宗门大殿风水影响护宗大阵效率

  12.4 宗门系统

  核心文件：runtime/world/world-runtime-sect.service.ts · runtime/world/world-runtime-formation.service.ts

  创建宗门

  使用 create_sect 类型物品（如"建宗令"）：

  1. dispatchCreateSect — 校验物品、距离、玩家状态
  2. 从地图模板目录动态生成 sect:<sectId> 实例
  3. 以当前位置为中心清出 SECT_FOUNDING_CLEAR_RADIUS = 2 格空地
  4. 放置"宗"字核心和"门"字入口
  5. 核心地块注入 SECT_GUARDIAN_INITIAL_AURA = 100000 作为护宗大阵初始气
  6. 玩家成为宗主（leader），playerSectId.set(playerId, sectId)

  角色与权限

  7 个角色（SECT_ROLES）：

  - leader / supreme_elder（不可委任）
  - deputy / elder / inner / outer / labor（可委任）

  3 项权限（SECT_PERMISSIONS）：

  - guardian — 护宗大阵控制
  - member_remove — 踢出成员
  - member_role — 修改职位

  默认权限表 DEFAULT_SECT_ROLE_PERMISSIONS 按角色给出

  护宗大阵

  - 核心地块持续注入灵气到 guardian aura
  - 玩家在核心附近攻击破坏地块时，大阵消耗灵气抵挡
  - 灵气耗尽后，宗门内地块可被自由破坏
  - SECT_INNATE_STABILIZER_RADIUS = 8 内地块不会自然腐烂（advanceTileRecovery 跳过）

  扩圈

  随宗门成员/建筑增多：

  - SECT_EXPAND_CHUNK = 8 格为单位扩张
  - 稀疏范围 SECT_SPARSE_EXPAND_RADIUS = 2 允许地块破坏
  - replaceTemplateForSectExpansion 整个实例切到更大模板，坐标偏移平移

  地块破坏触发扩圈

  战斗链路 createTileOutcomeApplyAdapter → 宗门可控制的"受保护地块"被攻破后 → expandSectForDestroyedTile 注入自动扩张
