# 大型 MMO 架构优化审查与优先级路线

更新时间：2026-04-20

## 1. 文档目的

这份文档回答四件事：

1. 当前 `packages/*` 主线架构对于“大型 MMO”到底处于什么水平。
2. 现阶段真实存在的架构级短板是什么，而不是抽象层面的“最佳实践愿望清单”。
3. 所有值得做的架构优化点有哪些，并按优先级排序。
4. 接下来应该先做什么、后做什么，才能把当前 next 主线推进到更接近大型 MMO 生产架构。

这份文档只基于仓库当前事实，不把未来理想态当作已完成事实。

---

## 2. 审查范围

本轮审查覆盖：

- `packages/server/src/runtime/**`
- `packages/server/src/network/**`
- `packages/server/src/persistence/**`
- `packages/server/src/http/**`
- `packages/shared/src/**`
- `packages/client/src/network/**`
- `packages/client/src/runtime/**`
- `packages/client/src/ui/**`
- `docs/next-plan/**`
- `docs/protocol-audit.md`
- `docs/next-replacement-blocker-board.md`
- `docs/map-system-architecture.md`

本轮不回答：

- 单个玩法数值是否平衡
- 单个 UI 面板的视觉细节优劣
- 部署成本估算
- 云资源、K8s、跨地域容灾的具体选型

## 3. 执行摘要

一句话结论：

**当前架构已经具备“权威单体 MMO 主服 + next 主线 + 强证明链”的骨架，但还没有达到“大型 MMO 可持续生产架构”的完成度。**

当前更准确的定位不是“架构错误”，而是：

- 方向对
- 主链已成型
- 骨架明显优于普通中小项目
- 但关键链路仍偏单进程中心化
- 类型与静态约束明显弱于商业级要求
- 安全与横向扩展能力还没有真正收口

如果按“大型 MMO 生产架构”标准粗评分：

| 维度 | 当前判断 |
| --- | --- |
| 运行时正确性 | 强 |
| 协议分层 | 强 |
| 验证与证明链 | 强 |
| 性能潜力 | 中 |
| 安全硬化 | 中偏弱 |
| 可水平扩展性 | 中偏弱 |
| 持久化体系完整度 | 中 |
| 工程可维护性 | 中偏弱 |
| 大型团队长期演进友好度 | 中偏弱 |

对外表述应保持克制：

- 可以说“当前主线已具备商业级收口意识”
- 不应说“当前已经是大型 MMO 完成态架构”

## 4. 当前事实基线

### 4.1 已经做对的事

当前仓库已经固定了几条重要主线：

- `packages/shared/src/protocol.ts` 是 next 协议真源。
- 客户端已经按 `Bootstrap / InitSession / MapStatic / WorldDelta / SelfDelta / PanelDelta` 这套分层消费。
- 服务端运行时、网络、持久化、HTTP 已经有明确目录边界。
- `replace-ready / with-db / shadow / acceptance / full` 证明链意识已经建立。
- 地图系统已经明确提出实例化终局方向，而不是继续停留在“地图模板即世界”的口径。

这些不是表面目录整理，而是正确的主线方向。

### 4.2 当前最关键的仓库级 blocker

仓库级 blocker 看板已经明确：

- `auth/bootstrap` 真源仍是主阻塞
- `world sync compat/perf` 仍是主阻塞
- `client patch-first` 仍未完全收口
- `shared` 字段级硬门禁仍未完全收口
- `with-db / shadow / acceptance / full` 仍缺最新一轮实环境复证

因此当前不是“已完成架构，需要锦上添花优化”，而是“主架构进入后半程收口，仍有硬阻塞未清”。

### 4.3 当前代码事实

本轮审查看到的几个关键事实：

- `WorldTickService` 当前 tick 间隔是 `100ms`，即 10Hz，而不是文档里常说的 1Hz。
- `WorldGateway` 仍是重量级入口，当前约 `2128` 行。
- `WorldProjectorService` 当前约 `2024` 行。
- `PlayerRuntimeService` 当前约 `3653` 行。
- `MapInstanceRuntime` 当前约 `2524` 行。
- `WorldRuntimeService` 已比过去显著下降，但仍约 `1650` 行，仍属于重编排服务。
- `packages/server/tsconfig.json` 仍是 `allowJs: true`、`checkJs: false`、`strict: false`。
- 当前仓库存在大量 `// @ts-nocheck`，说明“文件迁移为 .ts”不等于“真正获得 TS 约束”。
- `packages/server` 当前依赖里没有 Redis、消息队列或分布式协调层，说明当前仍是“数据库 + 进程内实时态”为主的单体权威服。
- 前端重构文档已明确：多个面板和 modal 仍存在整块重建，patch-first 未全收口。

## 5. 对大型 MMO 的总体判断

### 5.1 当前适合什么

当前架构适合：

- 单区或少区的权威 MMO 主服
- next 主线替换与功能收口
- 中大型内容规模
- 需要较强 proof / smoke / acceptance 的工程推进方式
- 以单体主服为核心、后续再拆在线态和实例调度的演进路线

### 5.2 当前不适合直接宣称什么

当前不适合直接宣称：

- 已具备大型 MMO 横向扩容能力
- 已具备多节点实例调度能力
- 已具备强隔离的高并发在线架构
- 已具备严格安全基线
- 已具备大型团队长期高频协作下的低风险演进能力

## 6. 分维度审查

### 6.1 运行时正确性

判断：**强**

优点：

- 服务端 tick 推进和同步下发仍保持权威链。
- 运行时、协议、持久化的边界意识明确。
- 运行时正在持续按职责拆分，而不是继续堆在一个巨型 service 里。

问题：

- 运行时仍然以单进程中心编排为主。
- 多个重服务仍承担明显过宽的 orchestrator 职责。
- 文档口径和代码事实存在偏移，例如 tick 频率口径。

结论：

- 正确性主链已经足够好。
- 但对大型 MMO 来说，单体权威服的正确性不等于可扩展正确性。

### 6.2 网络协议与同步架构

判断：**强**

优点：

- 协议已经拆成首包、静态、世界增量、自身增量、面板增量、详情。
- 有 `protocol-audit` 这类实际审计链，而不是只靠人工检查。
- 客户端主消费入口围绕 next 协议组织，方向正确。

问题：

- `WorldProjectorService` 仍偏重，capture/diff/payload 拼装耦合度仍高。
- `Bootstrap / MapStatic / PanelDelta` 的进一步去重和裁剪仍是 blocker。
- 高低频链路虽然已分层，但仍没有完全压到“高频链只带最低必要字段”的终局状态。

结论：

- 协议架构比运行时架构更接近大型 MMO 要求。
- 下一阶段重点不是重写协议，而是继续压缩高频链、降低 projector 复杂度。

### 6.3 性能与热路径

判断：**中**

优点：

- 有 protocol audit、bench、smoke，说明团队已有性能意识。
- 有 world projector cache、增量 envelope、实例化地图方向，这些都为后续优化提供空间。

问题：

- 当前核心仍是单 Node 进程里统一推进 tick、统一做 sync flush。
- 没有 Redis/消息层/实例目录服务等在线态基础设施，扩容天花板明显。
- 多个热路径服务仍然较大，后续继续加玩法时容易回卷复杂度。
- 当前没有看到“活跃实例调度、实例休眠、实例所有权迁移、跨节点实例目录”的正式落地链。

结论：

- 当前更像“单体主服可优化性能”，不是“天然大规模可扩展性能架构”。

### 6.4 持久化与状态真源

判断：**中**

优点：

- 已经明确 next 真源和 migration-only 边界。
- 玩家快照、身份、市场、邮件等持久化已成体系，不是纯内存试玩架构。
- 地图系统文档已经清楚提出 `InstanceCatalog / Overlay / Anchor` 等终局要求。

问题：

- 当前实例目录真源还没有完整落地为“正式实例目录数据库主链”。
- 在线态与正式真源之间仍偏“数据库 + 进程内状态”模式。
- 大型 MMO 所需的异步作业、增量事件、延迟写回、恢复编排层还不够完整。

结论：

- 真源意识正确。
- 但持久化架构还没有进化到“大量实例、强恢复、强调度”的阶段。

### 6.5 安全性

判断：**中偏弱**

优点：

- 有 CORS 白名单策略。
- 有登录限流服务。
- sessionId 有基本合法性校验。

问题：

- GM 默认密码这类安全硬伤已开始收口，但 GM 高危操作审计、细粒度权限和正式 secret 轮换仍未完成。
- 安全更多体现在“工程控制”和“输入边界”层面，而不是正式攻防级体系。
- 当前没有看到更完整的生产安全基线，例如密钥轮换、细粒度审计、连接滥用隔离、分层权限域、敏感操作二次校验等。

结论：

- 对本地和 shadow 线够用。
- 对大型 MMO 正式生产环境还不够。

### 6.6 可维护性与类型安全

判断：**中偏弱**

优点：

- 目录边界比过去清晰很多。
- 服务拆分方向基本正确。
- server `.js` 真源已经清零，这是重要进步。

问题：

- `strict: false`
- `allowJs: true`
- `checkJs: false`
- 大量 `@ts-nocheck`
- 若干 `.ts` 文件内容仍保留强烈的编译产物风格，不是理想的人类维护源码形态

结论：

- 当前是“文件名 TS 化”完成度高于“真实类型收口”完成度。
- 这对大型 MMO 长线迭代是实质风险，不是风格问题。

### 6.7 客户端架构与 UI 增量消费

判断：**中偏强**

优点：

- 前端网络消费主链已经 next-native。
- 主同步入口围绕几类明确增量包组织。
- 已有单实例详情弹层、selection-preserver、responsive-viewport 等稳定基建。

问题：

- 多个大面板仍有整块重建。
- patch-first 未完全收口。
- 高频状态与面板状态时序错位仍是长期风险。
- 手机 / 浅色 / 深色虽然代码有考虑，但系统手工回归仍未闭环。

结论：

- 客户端主链问题已经不是“有没有 next 协议适配”，而是“能否把 UI 真正收成局部 patch 架构”。

### 6.8 运维与验证链

判断：**强**

优点：

- `verify:replace-ready*`、`with-db`、`shadow`、`acceptance`、`full` 分层非常清楚。
- audit / smoke / bench / proof 较完整。
- 文档口径整体上比很多项目更重视“回答什么、不回答什么”。

问题：

- 最新一轮带库和 shadow 实环境证据仍需刷新。
- 一些文档中的历史基线数字和当前代码事实已出现偏移。

结论：

- 这是当前架构最强的一维。
- 后续不能让“证明链领先、实现约束落后”的差距继续扩大。

## 7. 架构优化点全集

以下列表按领域分组，目标是覆盖所有当前值得进入工程视野的架构优化项。

### 7.1 运行时与调度

1. 把当前“单进程统一 tick + flush”继续推进为“实例级活跃调度模型”。
2. 引入活跃实例、休眠实例、待恢复实例的明确状态机。
3. 为实例调度增加 owner 概念，避免未来集群化时仍由单总服务持有全部实例。
4. 明确实例生命周期真源，不再把实例恢复和实例运行期状态混在单体服务里。
5. 把 `WorldRuntimeService` 继续压到真正的薄 orchestrator，不再承载新的跨域胶水。
6. 把 `MapInstanceRuntime` 再拆，尤其是实例级热路径、观测、落盘、容器、AOI、战斗域边界。
7. 建立正式的 tick phase 契约文档和代码常量，避免文档口径漂移。
8. 为 tick 增加 budget 与超时统计，不只记录耗时，还要有告警阈值。
9. 区分“世界帧推进”和“网络同步帧”概念，避免未来所有重活都卡在同一节拍。
10. 为未来跨节点实例迁移保留 runtime detach / attach 合同。

### 7.2 网络协议与投影

1. 继续压缩 `Bootstrap` 包内容，把能拆去 `MapStatic`、`Detail`、`PanelDelta` 的字段继续拆走。
2. 继续压缩 `PanelDelta`，避免演化成“低频大杂烩”。
3. 把 `WorldProjectorService` 按 capture / diff / encode envelope 再分层。
4. 为各类 delta 建立字段级 hard gate，新增字段必须声明频率层级。
5. 为高频包建立 payload budget，明确上限而不是只做观测。
6. 进一步减少“无变化重复发”的容忍空间。
7. 明确哪些字段只允许单播、哪些允许 AOI 广播、哪些允许全局广播。
8. 为 `MapStatic` 建立 revision/patch 级缓存策略，避免实例静态层重复组装。
9. 对 world sync compat/perf blocker 做专项清账，而不是继续散点修补。
10. 为未来二进制高频包预留正式迁移策略，但不要过早把 protobuf 变成当前主战场。

### 7.3 持久化与实例真源

1. 正式落地 `InstanceCatalog` 数据库真源。
2. 把玩家正式落点完全切到 `instanceId` 主链。
3. 为地图实例建立生命周期字段：创建来源、owner、锚点、过期、恢复策略。
4. 建立实例 overlay 真源，而不是继续依赖“复制模板”思路。
5. 为玩家状态、实例状态、地图状态建立不同的 flush 和恢复策略，不再一锅端。
6. 明确哪些状态是“运行时可重建”，哪些必须下次仍在。
7. 引入持久化写回队列或事件层，避免后期所有持久化都直接压在主线程逻辑里。
8. 对异常恢复路径做正式 runbook，不只依赖 smoke。
9. 给关键真源增加 schema/version 演进策略。
10. 为 restore / backup / replay 建立可审计的数据闭环。

### 7.4 安全与权限

1. 继续维持“默认 GM 密码禁止进入正式环境”的硬门禁，并把历史弱口令轮换纳入运维清单。
2. 把 GM 登录、GM token、GM 操作审计拆成独立安全域。
3. 为 GM 高危操作增加更细粒度权限，不再只按“是否 GM”判断。
4. 增加敏感操作审计事件真源。
5. 建立正式的 secret 管理与轮换口径。
6. 为连接层增加更稳的 abuse 控制，不只停留在 auth rate limit。
7. 为协议入口增加更系统的 payload 校验和异常拒绝统计。
8. 为 shadow/ops 脚本补权限和环境隔离基线。
9. 为 HTTP、Socket、GM、维护态分别定义最小权限集。
10. 建立“开发环境可降级、生产环境强限制”的统一安全开关矩阵。

### 7.5 类型系统与可维护性

1. 把 `packages/server` 的 `strict: false` 收紧成 staged strict 计划。
2. 系统性清理 `@ts-nocheck`，优先热路径、安全链路、持久化链路。
3. 停止继续接受“编译产物风格 TS 源码”扩散到主链。
4. 为巨型 service 建立规模警戒线和 owner。
5. 明确“过渡 seam”与“最终边界”，避免长期停在伪 partial class 状态。
6. 让新的模块必须显式声明自己是 orchestrator / query / runtime / persistence / protocol 之一。
7. 为关键服务建立更细颗粒 smoke，不再让大服务只能整链路验证。
8. 建立更严格的边界检查，防止 runtime / network / persistence 反向耦合回卷。
9. 把 server TS loop 从“换扩展名完成”推进到“类型约束真实生效”。
10. 让 shared 类型的新增字段默认要求三端消费与 proof 对齐。

### 7.6 客户端与 UI

1. 继续清掉仍存在整块重建的主面板和 modal。
2. 让 `PanelDelta` 的消费真正落到局部 patch，而不是大段 DOM 重刷。
3. 收紧空间真源与面板真源的边界，避免多源写状态。
4. 为高频世界态更新建立更明确的本地派生规则，减少 UI 层误写。
5. 把业务 UI 再压成 recipe，而不是继续在每个面板里重复壳体逻辑。
6. 做一轮系统性的浅色 / 深色 / 手机模式手工回归。
7. 为地图、HUD、面板之间的状态桥接写更清楚的 owner 口径。
8. 为客户端协议消费增加回归检查，防止 shared 字段新增后 UI 靠人工补洞。
9. 收口 GM 前端入口的职责，避免 GM 工具链继续膨胀成平行应用。
10. 把 React prototype 与主 UI 主链的关系继续钉死，避免双套 UI 结构长期并存。

### 7.7 运维与验证

1. 补最新一轮 `with-db / shadow / acceptance / full` 实环境复证。
2. 把最新实跑事实回写到文档，清掉旧数字和旧口径。
3. 为性能建立正式门禁，不只测 correctness。
4. 补充大负载、长连接、跨图、掉线重连、恢复、刷盘并发下的压力回归。
5. 为实例恢复、GM 数据库恢复、维护态切换建立更细的自动检查。
6. 增加对 protocol payload budget 的审计。
7. 增加对热路径 `@ts-nocheck` 残留的 proof。
8. 增加对“新增字段未三端消费”的 fail-fast 门禁。
9. 把替换准备度和生产准备度分开评估，避免混读。
10. 为未来分布式拆分预留“单机证明链”和“多节点证明链”两套层级。

### 7.8 面向大型 MMO 的中长期架构能力

1. 实例目录服务化。
2. 活跃实例调度器。
3. 在线态缓存层。
4. 异步事件总线或作业层。
5. 连接层与世界执行层解耦。
6. 跨节点实例定位与路由。
7. 维护态与切流能力增强。
8. 分区、分线、实例 owner 迁移能力。
9. 观测、压测、容量规划的正式化。
10. 高危 GM / restore / shadow 操作的最小化与审计化。

## 8. 优先级排序

### P0：必须先做，否则不应宣称接近大型 MMO 生产架构

1. 清掉 `auth / bootstrap / session` 真源主阻塞。
2. 清掉 `world sync compat/perf` 主阻塞。
3. 补最新一轮 `with-db / shadow / acceptance / full` 实环境复证。
4. 清除默认 GM 密码与相关安全口径。
5. 开始对 server 主链做真实类型收紧：先收热路径、安全链、持久化链。
6. 统一 tick 频率和阶段口径，清掉文档与代码漂移。
7. 明确实例目录与 `instanceId` 持久化主链，不再停留在地图系统终局设计层。

### P1：下一阶段最值得投入，直接决定能否迈向大型 MMO

1. 把 runtime 调度继续推进到活跃实例模型。
2. 继续压薄 `WorldRuntimeService`、`MapInstanceRuntime`、`WorldProjectorService`、`PlayerRuntimeService`。
3. 为高频协议建立字段级 budget 和更严格 gate。
4. 为客户端 patch-first 做第二轮系统收口。
5. 建立实例生命周期数据库真源。
6. 建立持久化写回与恢复分层，不再让运行态直接承受全部长尾语义。
7. 为连接滥用、敏感操作、GM 权限做更正式的安全层。

### P2：中期必须做，否则扩容会反复返工

1. 引入在线态缓存或协调层。
2. 引入异步作业/事件层。
3. 为实例 owner、实例迁移、实例休眠恢复建立正式合同。
4. 将连接层、执行层、目录层进一步解耦。
5. 建立更正式的性能压测门禁。
6. 清理更多 `@ts-nocheck` 和编译产物风格源码。

### P3：大型 MMO 终局能力建设

1. 多节点实例调度。
2. 跨节点玩家定位与转移。
3. 更强的容量治理和扩容策略。
4. 多区、多分线、多实例 owner 迁移体系。
5. 更细颗粒的运营、审计、灰度和切流能力。

## 11. 哪些事情现在不要做

为了避免架构工作失焦，当前不建议优先做：

1. 为了“更现代”而重写通信栈。

## 13. 结论

当前 next 主线最大的价值，不是“已经是大型 MMO 终局架构”，而是：

- 已经摆脱 legacy 并建立了正确主线

- 继续清主阻塞
- 把类型和安全短板补起来

- 把单体主服打磨到足够稳，再去做大型 MMO 所需的分布式能力

这条路线是渐进式升级，不是推倒重来。
