>
> 本清单基于当前 GM 面板最新一轮 CPU 热点分项数据整理，目标不是“泛泛优化”，而是逐项定位当前真实热点，并给出可直接落地的改造顺序。

## 2. 当前结论

- 当前第一梯队热点已经很明确，不再是“大类不清楚”，而是：
  - 地块 Patch 全视野扫描与快照复制
  - 动作重建整链路
  - Socket 发送
  - 实体 Patch 扫描与复制
  - 修炼初始化校正
- 当前最值得优先优化的，不是单次最慢，而是“高频 + 可避开”的重复工作。
- 现阶段最不划算的 CPU 消耗主要来自两类：
  - 每 tick 都做整片视野的比较与深拷贝
  - 每 tick 都重复做本可按脏标驱动的重建

## 3. P0 清单

### [ ] P0-1 地块 Patch 改成脏区驱动，停止每 tick 全视野扫描

- 现状：
  - [packages/server/src/game/tick.service.ts](/home/yuohira/mud-mmo/packages/server/src/game/tick.service.ts#L1373) 中 `buildSparseVisibleTilePatches` 仍会逐格扫描当前可视范围。
  - 即使地图没有变化，也要把整片可见区域逐格 `isStructuredEqual` 比较一遍。
- 直接原因：
  - 当前逻辑是“按视野全量对比”，不是“按地图脏块增量推送”。
- 改造目标：
  - 地图没变时，不进入地块 Patch 比对。
  - 只有动态地块、门、容器、机关、覆盖层发生变化时，才构造 Patch。
- 具体动作：
  - 在 `MapService` 维护每张地图的 `tileRevision` 和 `dirtyTileKeys` / `dirtyRects`。
  - `tickDynamicTiles`、地图交互、门状态变化、掉落阻挡变化等所有会影响地块显示的地方统一上报脏区。
  - `broadcastTicks` 中若 `viewer` 的可见范围未变化，且当前地图 `tileRevision` 未增加，直接跳过 `buildSparseVisibleTilePatches`。
  - 切图、视野半径变化、首次进入视野时仍保留全量首帧。
- 验收口径：
  - `地块 Patch: 扫描比较` 降到当前的 `30%` 以下。
  - 静态站桩场景下，该项应接近 0。

### [ ] P0-2 地块 Patch 快照复制去掉 JSON 深拷贝式缓存

- 现状：
  - [packages/server/src/game/tick.service.ts](/home/yuohira/mud-mmo/packages/server/src/game/tick.service.ts#L1410) 中 `地块 Patch: 快照复制` 主要成本来自 `JSON.parse(JSON.stringify(...))`。
- 直接原因：
  - 即使只变动极少量格子，当前仍会对下一帧缓存做大量深拷贝。
- 改造目标：
  - 快照复制只针对脏格，不针对整片视野。
  - 复制方式从通用 JSON 深拷贝改成受控结构复制。
- 具体动作：
  - 在完成 P0-1 后，仅更新脏格缓存，不再每 tick 重建整张 `nextCache`。
  - 为 `VisibleTile` 增加稳定签名或轻量快照结构，优先缓存 `tileId + overlay + blocker + interactable` 这类固定字段。
  - 对 `VisibleTile` 使用手写字段复制函数，替代 JSON 深拷贝。
- 验收口径：
  - `地块 Patch: 快照复制` 至少下降一半。
  - 该项下降后，总 CPU 榜第一名不应再是地块快照复制。

### [ ] P0-3 动作重建改成脏标驱动，并消除每 tick 双重重建

- 现状：
  - [packages/server/src/game/tick.service.ts](/home/yuohira/mud-mmo/packages/server/src/game/tick.service.ts#L518) 和 [packages/server/src/game/tick.service.ts](/home/yuohira/mud-mmo/packages/server/src/game/tick.service.ts#L537) 中，`syncActions(player)` 每个玩家每 tick 最多会跑两次。
  - 当前热点已拆出：
    - `动作重建: 场景动作收集`
    - `动作重建: 核心构建`
    - `动作重建: 重建后快照`
- 直接原因：
  - 当前是“先跑一轮，再在怪物结算后再跑一轮”的保守做法。
  - 是否重建依赖前后快照比较，而不是动作源头脏标。
- 改造目标：
  - 玩家动作只在影响动作列表的状态发生变化时重建。
  - 同一 tick 内，默认最多只重建一次；只有确实被怪物结算、Buff、状态变化影响时才补第二次。
- 具体动作：
  - 为玩家新增 `actionsDirty` / `actionContextDirty` / `actionCooldownDirty` 一类脏标。
  - 所有会影响动作列表的入口统一打脏标，例如：
    - Buff 变化
    - 冷却变化
    - 修炼状态变化
    - 切图
    - 邻近交互对象变化
    - 战斗目标变化
  - 第一轮重建后清脏；怪物结算后只对受影响玩家补做一次。
  - 后续把前后 `JSON.stringify` 快照比较改成版本号或签名比较。
- 验收口径：
  - `syncActions` 调用次数从当前近似 `玩家数 x 2` 降到接近 `真正脏玩家数`。
  - `动作重建: 核心构建 + 场景动作收集 + 重建后快照` 合计下降 `40%` 以上。

### [ ] P0-4 修炼初始化校正改成一次性初始化或版本校正

- 现状：
  - [packages/server/src/game/technique.service.ts](/home/yuohira/mud-mmo/packages/server/src/game/technique.service.ts#L151) 中 `cultivateTick` 每 tick 都会先执行 `initializePlayerProgression(player)`。
- 直接原因：
  - 初始化校正与周期推进没有分离，导致“保障一致性”的逻辑变成了每 tick 固定成本。
- 改造目标：
  - 初始化校正只在登录、读档、突破、学习功法、装备变化、属性重算后触发。
- 具体动作：
  - 为玩家增加 `progressionInitialized` 或 `progressionRevision`。
  - `cultivateTick` 内只做必要前提校验，不再无条件完整初始化。
  - 所有真正会影响修炼结构的入口统一调用校正。
- 验收口径：
  - `修炼: 初始化校正` 从热点榜前列移出。
  - 角色挂机人数增加时，该项不再线性上升得这么明显。

## 4. P1 清单

### [ ] P1-1 实体 Patch 改成版本化缓存，减少扫描和复制

- 现状：
  - [packages/server/src/game/tick.service.ts](/home/yuohira/mud-mmo/packages/server/src/game/tick.service.ts#L1428) 中实体 Patch 仍按实体逐个比较，再进行快照复制。
- 具体动作：
  - 为 `RenderEntity` 增加稳定版本或签名。
  - Buff、观察信息、任务标记等高成本字段单独维护 revision。
  - 仅当某类字段 revision 变化时才复制该字段。
- 目标：
  - `实体 Patch: 扫描比较 + 快照复制` 合计下降 `30%` 以上。

### [ ] P1-2 AOI 可见性做增量化或缓存化

- 现状：
  - `广播: AOI 可见性` 已进入前十。
- 具体动作：
  - 玩家未移动、视野半径未变、地图遮挡修订号未变时复用上次可见性结果。
  - 移动时优先做边缘增量更新，而不是全量重算。
- 目标：
  - 站桩状态下 AOI 可见性基本不产生稳定热点。

### [ ] P1-3 Socket 发送与编码链路分离统计

- 现状：
  - `广播: Socket 发送` 已达 `8.6%`，但还没区分“编码成本”和“真正发包成本”。
- 具体动作：
  - 继续把发送前准备、序列化、socket.emit 拆成独立分项。
  - 对同视野多人共享的数据包，评估是否可以复用序列化结果。
- 目标：
  - 明确是“包太大”还是“发得太多”，避免误判。

## 5. P2 清单

### [ ] P2-1 怪物时间效果批处理

- 现状：
  - `怪物: 时间效果` 次数极高，但单次极低，说明是“高频小成本”项。
- 处理策略：
  - 当前不是第一优先级。
  - 等 P0、P1 热点压下去后，再看是否需要按地图统一时间状态批处理，避免每只怪单独同步。

### [ ] P2-2 网络编解码继续拆分

- 现状：
  - `网络编解码与收发` 还在前十，但暂时不是主矛盾。
- 处理策略：
  - 后续视 P0 结果决定是否继续拆：
    - decode
    - encode
    - socket flush
    - GM API 轮询

## 6. 建议执行顺序

1. 先做 P0-1 和 P0-2，直接砍掉地块 Patch 双热点。
2. 再做 P0-3，处理动作重建双调用和全量重建。
3. 然后做 P0-4，去掉修炼初始化的常驻成本。
4. 再看 P1-1 和 P1-2，处理实体 Patch 与 AOI。
5. 最后继续拆 Socket 发送和网络编解码。

## 7. 每轮优化后的观察口径

- 重启服务端，清空旧 CPU 统计后再看新样本。
- 优先看三种排序：
  - 总耗时：确认真正的大头
  - 次数：确认是否是高频重复工作
  - 单次：确认是否存在单次尖峰
- 每次只做一类优化，避免多个改动同时落地后无法判断收益来源。
- 每轮至少观察两个场景：
  - 单人站桩挂机
  - 多人同图移动与战斗

## 8. 完成判定

达到以下状态，可以认为当前这轮 CPU 优化告一段落：

- 地块 Patch 不再长期占据前两名。
- 动作重建不再稳定以双倍次数出现。
- 修炼初始化校正退出热点前列。
- 前十热点中，真正不可避免的业务项才开始占主导，例如战斗、AOI、网络发送。
