# 战斗链路运维手册

本文描述当前生产主线可用的战斗诊断和验证入口。它不是完整数据库审计手册；战斗审计 outbox 和持久化查询尚未完成。

## 可观测事件

### 结构化拒绝日志

运行时会输出 `combat_action_rejected` 结构化日志，关键字段包括：

- `reason`：机器可读拒绝原因，例如 `line_of_sight_blocked`、`target_instance_mismatch`、`no_runtime_targets_in_warning_cells`。
- `phase`：动作阶段，例如 `instant`、`chant_start`、`chant_resolve`。
- `actor`：发起者，格式如 `player:<id>` 或 `monster:<runtimeId>`。
- `action`：动作或技能 id。
- `instance`：地图实例 id。
- `target`：目标。
- `target_count`：本阶段目标数量。

这类日志用于定位怪物技能预警命中但无结算、目标跨实例、视线阻挡、目标死亡等问题。

### 内部 layered events

`WorldRuntimeCombatActionService.buildCombatEvents()` 会把 outcome 分为四层：

- AOI 表现：`aoiEvent`，只保留视觉需要的小字段。
- 玩家通知：`notificationEvent`，只进入玩家 Notice 单播层。
- 内部审计：`auditEvent`，包含 actor、action、target、result、application。
- 内部诊断：`diagnosticEvent`，包含 reason、details、severity。

内部审计和诊断当前不属于普通客户端 S2C payload。

## 查询 helper

`packages/server/src/runtime/combat/combat-event-query.ts` 提供纯函数查询入口，只消费内存中的 layered events 或内部事件数组。

可用函数：

- `queryRecentCombatAuditEvents(events, { playerId, monsterRuntimeId, instanceId, since, until, limit })`
- `aggregateCombatDiagnostics(events, { reason, instanceId, since, until })`
- `queryMonsterSkillFailureReasons(events, { monsterRuntimeId, actionId, instanceId, since, until })`
- `buildCombatAuditHeatmap(events, { instanceId, since, until })`

限制：

- 不读数据库。
- 不跨进程。
- 服务重启后不可查询历史。
- 不能代替正式战斗审计表、outbox 或 GM 查询接口。

## 常用诊断流程

### 怪物吟唱预警格命中但没有攻击信息

1. 查 `combat_action_rejected` 日志。
2. 优先看 `phase=chant_resolve`。
3. 若 reason 是 `no_runtime_targets_in_warning_cells`，说明完成时预警格没有权威玩家目标。
4. 若 reason 是 `line_of_sight_blocked`，说明预警格有目标但视线校验拒绝。
5. 若 reason 是 `target_instance_mismatch`，说明玩家位置真源和实例占位不一致。
6. 若没有拒绝日志，跑 `smoke:world-runtime-combat-action-service` 验证诊断链是否被破坏。

### 怪物一直释放同一技能

1. 跑 `pnpm --filter @mud/server smoke:monster-skill`。
2. 确认唤灵真人配置顺序和 fallback smoke 是否通过。
3. 看技能冷却、血量阶段、目标状态 buff 或地形条件是否让高优先级技能不可用。
4. 不要只看客户端表现，运行时选择和配置 smoke 才是权威判断。

### 玩家或怪物攻击完全没有通知

1. 判断是“未命中/闪避”还是“没有进入结算”。
2. 闪避应有 combat notice。
3. 没有 notice 时查 diagnostic reason。
4. 如果是合法落空，应有 `no_targets` 或 `no_runtime_targets_in_warning_cells`。
5. 如果是异常落空，应补结构化拒绝原因或 smoke。

## 验证命令

```bash
pnpm --filter @mud/server smoke:world-runtime-combat-action-service
pnpm --filter @mud/server smoke:combat
pnpm --filter @mud/server smoke:monster-combat
pnpm --filter @mud/server smoke:monster-skill
pnpm --filter @mud/server bench:combat
pnpm audit:protocol
pnpm audit:boundaries
pnpm --filter @mud/server proof:production-boundaries
```

这些命令不证明：

- 战斗审计已经持久化。
- pending cast 断线重连或服务重启后可恢复。
- 100 玩家和 50 怪物并发场景已经达标。
- 所有旧生产分支已删除。
