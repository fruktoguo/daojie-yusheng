# 气机系统设计文档

## 1. 目标

本文用于定义《道劫余生》的通用气机系统，用来在不推翻当前地图灵气系统的前提下，平滑升级出一套可长期扩展的世界气机底层。

本文重点解决以下问题：

- 保留当前“地图灵气值 + 灵气等级 + 感气观察 + 修炼加成”的既有能力
- 让凝练灵气、逸散灵气都能真实参与系统，而不是只做文案设定
- 为五行灵气、魔气、煞气等后续内容提供统一底层
- 让灵根、功法、体质、Buff 真正影响玩家“能看见什么、能吸收什么、能高效利用什么”
- 保持地块动态数据统一存储，不为每一种气机单开数据库或单开运行时文件

本文是系统设计文档，不是一次性全部实现清单。目标是先定模型、边界和迁移方向，再分阶段接入现有代码。

## 2. 总原则

- 不推翻当前灵气系统，而是在其上做抽象升级
- 气机系统是“世界资源生态层”，不是单独服务某一个玩法的私有逻辑
- 地块相关的所有动态资源统一收口到同一份地块运行时存储中
- 地图源点、技能外泄、怪物污染、阵法改造等，统一通过“气机资源变动”表达
- 玩家最终看到的等级，不必等于地块原始总量，而应等于“该玩家当前能利用的有效气机总量”
- 气机谱系只定义“世界真实存在什么”，不直接等于“所有玩家最终看到什么”
- UI 默认只展示玩家当前真正关心的聚合信息，详细拆分只在观察副 tip 中展开
- 服务端仍然是唯一正确性来源；客户端只做展示，不做规则判定

## 3. 为什么不能继续只做“灵气”

如果后续只是在当前灵气系统旁边追加“魔气系统”“煞气系统”“五行灵气系统”，项目会很快出现以下问题：

- 同类逻辑被拆成多套近似实现，维护成本高
- 地图动态数据会越来越散，后续存档和迁移困难
- 玩家感知、吸收、修炼、污染、阵法等逻辑无法共享
- “灵根”只能停留在文案上，无法稳定影响实际数值
- 客户端展示会被迫塞入越来越多特判

因此后续不应采用“每种气机重写一套”的做法，也不建议走传统 OOP 的“继承 Aura 类派生 DemonAura 类”路线。更合理的方式是统一成一个多维气机系统。

## 4. 气机的三维模型

一份具体气机不再只用“灵气”一个名字描述，而是由三个维度共同定义：

- `family`：气机种类
- `form`：气机形态
- `element`：气机属性

### 4.1 气机种类 `family`

用于表达世界观中的大类气机。

首批推荐固定为：

- `aura`：灵气
- `demonic`：魔气
- `sha`：煞气

后续可以继续扩展，但不建议一开始做成无限开放自由文本，优先由共享配置统一定义。

### 4.2 气机形态 `form`

用于表达同一种气机在世界中的存在状态。

首批固定为：

- `refined`：凝练
- `dispersed`：逸散

含义如下：

- 凝练气机：长期稳定存在，通常与地脉、阵法、福地、污染核心等源点绑定
- 逸散气机：短时存在，通常来自技能释放、战斗外泄、死亡残留、临时环境扰动

### 4.3 气机属性 `element`

用于表达气机内部的属性谱系。

首批推荐固定为：

- `neutral`：无属性
- `metal`：金
- `wood`：木
- `water`：水
- `fire`：火
- `earth`：土

说明：

- 当前已有普通灵气可以先视为 `neutral`
- 五行灵气不是“新的大类气机”，而是灵气内部不同属性的分化
- 魔气、煞气未来也可以拥有属性维度，但首阶段可先只使用 `neutral`

### 4.4 资源键 `resourceKey`

三维组合后生成唯一资源键：

```text
family.form.element
```

示例：

- `aura.refined.neutral`
- `aura.dispersed.neutral`
- `aura.refined.fire`
- `aura.refined.water`
- `demonic.refined.neutral`
- `sha.dispersed.neutral`

### 4.5 与当前系统的兼容映射

当前项目中的“灵气”应整体视为：

```text
aura.refined.neutral
```

这样老地图、老玩法、老观察逻辑都还能继续成立。后续新增逸散灵气、五行灵气、魔气、煞气时，只是在同一底层上新增资源键，而不是另起一套系统。

## 5. 为什么采用三维模型而不是平铺一堆资源名

例如“火属性逸散灵气”如果用平铺命名，会变成一个新的平级资源名；久而久之会得到：

- 火灵气
- 水灵气
- 逸散灵气
- 火逸散灵气
- 魔火气
- 煞火气

这会让策划认知、代码认知、UI 认知同时变乱。

三维模型的好处是：

- “灵气 / 魔气 / 煞气”始终是大类
- “凝练 / 逸散”始终是存在形态
- “金木水火土”始终是内部属性
- 玩家面板和观察副 tip 可以按“大类 -> 细分”展示，更符合直觉

## 6. 配置体系

气机系统推荐拆成 4 层配置，而不是所有参数全部堆在一个对象里。

### 6.1 家族配置 `family config`

用于定义某一类气机的基础认知和等级规则。

建议字段：

- `key`
- `label`
- `levelBaseValue`
- `defaultObservePriority`
- `defaultColorStyle`
- `enabledElements`

示例：

- `aura` 的 `label` 为“灵气”
- `demonic` 的 `label` 为“魔气”
- `sha` 的 `label` 为“煞气”

说明：

- 等级基准值应允许按家族独立配置，不强制所有家族都使用同一个 `1000`
- 但首阶段可以让灵气、魔气、煞气都共用 `1000`，降低复杂度

### 6.2 形态配置 `form config`

用于定义某种形态的默认衰变参数。

建议字段：

- `key`
- `label`
- `halfLifeTicks`
- `fixedDecayPerTick`
- `minDecayPerTick`

推荐默认认知：

- `refined`：半衰期长、固定衰减小、最低衰减小
- `dispersed`：半衰期短、固定衰减大、最低衰减大

### 6.3 属性配置 `element config`

用于定义五行或无属性标签。

建议字段：

- `key`
- `label`
- `displayOrder`

### 6.4 资源覆写配置 `resource override`

用于在具体资源键上覆写默认规则。

因为仅靠 `family` 和 `form` 两层配置还不够。例如：

- 普通凝练灵气和火行凝练灵气可能半衰期不同
- 凝练魔气可能比凝练灵气更难消散
- 某些煞气可能固定每息至少消退 3 点

因此需要允许对某个资源键单独配置：

- `halfLifeTicks`
- `fixedDecayPerTick`
- `minDecayPerTick`
- `levelBaseValueOverride`
- `visualOverride`

配置优先级建议为：

```text
resource override > form config > family config default
```

## 7. 地图与地块数据模型

### 7.1 地图源点配置

地图编辑层不应继续只支持 `auras`，而应逐步升级为通用 `qiSources`。

建议形态：

```ts
type MapQiSourceRecord = {
  x: number;
  y: number;
  resourceKey: string;
  value: number;
};
```

含义：

- `value` 表示该地块该资源的天然稳定源值
- 例如福地可配置 `aura.refined.neutral`
- 火山口可配置 `aura.refined.fire`
- 魔域祭坛可配置 `demonic.refined.neutral`

为了兼容旧内容，首阶段保留旧 `auras` 字段，并在加载时映射为：

```text
aura.refined.neutral
```

### 7.2 地块运行时状态

运行时不再维护单独的 `auraStates`，而是统一为：

```ts
resourceStates[mapId][tileKey][resourceKey]
```

单个资源状态建议为：

```ts
type TileQiRuntimeState = {
  value: number;
  sourceValue: number;
  remainder: number;
};
```

说明：

- `value`：当前真实值
- `sourceValue`：长期稳定会收敛到的目标值
- `remainder`：半衰期固定点结算余数

如后续需要更精细统计，可以再加：

- `lastModifiedAt`
- `lastSourceChangedAt`
- `version`

但首阶段不建议增加与玩法无关的元数据噪声。

### 7.3 地块统一存储原则

当前项目已经有“地块运行时总文件”的思路，这个方向应保留并继续扩展，而不是为灵气、魔气、煞气分别新建存储。

原则固定如下：

- 地块生命、地形破坏、地块时间、气机资源，统一落在同一份地块运行时存储中
- 某一地块的所有动态状态应能在同一处读写、迁移和恢复
- 服务重启后，地块资源状态应完整恢复
- 不在 tick 热路径里直接做数据库 IO

## 8. 结算规则

## 8.1 结算目标

每种资源都应根据自己的配置，逐步向 `sourceValue` 收敛，而不是简单地对当前值做固定加减。

这点非常关键。

如果把“固定衰减”和“固定补充”都直接施加到总值上，会出现以下问题：

- 天然地脉无法稳定维持在源值附近
- 固定衰减会把源点也慢慢吃掉
- 需要写越来越多特判去强行拉回

正确做法是对“当前值与源值之间的差值”做收敛。

### 8.2 核心变量

设：

- `current = value`
- `source = sourceValue`
- `diff = current - source`
- `magnitude = abs(diff)`

解释：

- `diff > 0`：当前高于源值，应向下收敛
- `diff < 0`：当前低于源值，应向上收敛
- `diff = 0`：已等于源值，不应被强行推动

### 8.3 每息收敛步长

每种资源在每息的变化量由 3 部分共同决定：

- 半衰期步长
- 固定衰减步长
- 最低衰减步长

推荐计算过程：

```text
halfLifeStep = floor((magnitude * rateScaled + remainder) / scale)
rawStep = halfLifeStep + fixedDecayPerTick
step = min(magnitude, max(minDecayPerTick, rawStep))
```

然后：

```text
if diff > 0:
  current = current - step
else if diff < 0:
  current = current + step
else:
  current = source
```

### 8.4 这套规则的性质

这套规则能够同时满足以下目标：

- 天然源点可稳定维持
- 逸散资源会自然快速消散
- 玩家或技能造成的额外增减不会被硬锁死
- 当前值低于源值时能自然回补
- 当前值高于源值时能自然回落
- 刚好等于源值时不会因为最低步长被强行推走

### 8.5 对用户需求的对应关系

你希望每种气机都能配置：

- 半衰期
- 固定衰减数值
- 最低衰减数值

在本方案中，这 3 个参数都保留，并且都作用在“偏离源值的差值”上，而不是粗暴地直接作用在总值上。

### 8.6 典型示例

#### 示例 1：凝练灵气地脉

配置：

- 资源：`aura.refined.neutral`
- `sourceValue = 4000`
- 半衰期很长
- 固定衰减很小
- 最低衰减很小

结果：

- 正常情况下长期稳定在约 `4000`
- 如果玩家或事件抽空到 `2500`，会缓慢回升
- 如果技能外泄叠到 `7000`，会缓慢回落

#### 示例 2：逸散灵气

配置：

- 资源：`aura.dispersed.neutral`
- `sourceValue = 0`
- 半衰期短
- 固定衰减较大
- 最低衰减较大

结果：

- 技能释放后可瞬间抬高数值
- 随后快速向 `0` 消散
- 不会长期停在地图上

#### 示例 3：火行灵脉

配置：

- 资源：`aura.refined.fire`
- `sourceValue = 3200`

结果：

- 地块长期保持火行灵气
- 具备火灵根或火系功法的玩家收益更高

## 9. 玩家侧：感知、吸收、炼化

要让灵根、功法、Buff 真正参与系统，不能只在玩家身上挂一个布尔值 `senseQiActive`。玩家侧还需要一套气机权限与效率模型。

推荐拆成 3 层：

- 感知资格：玩家是否能看见某类气机
- 吸收资格：玩家是否能把某类气机纳入可用总值
- 炼化效率：玩家把某类气机转成自己收益时的效率

### 9.1 感知资格 `observable`

用于控制观察和感气视角中能不能看见某类气机。

示例：

- 普通修士默认只能感知 `aura.refined.*`
- 学会某魔功后可感知 `demonic.refined.*`
- 服下异丹后可暂时感知 `sha.dispersed.*`

### 9.2 吸收资格 `absorbable`

用于控制玩家能否把该资源计入自己的有效气机总量。

示例：

- 默认只能吸收 `aura.refined.*`
- 某功法允许额外吸收 `aura.dispersed.*`
- 某魔功允许吸收 `demonic.refined.*`
- 某体质能够吞煞，允许吸收 `sha.dispersed.*`

### 9.3 炼化效率 `affinity / efficiency`

玩家即便能吸收，也不一定能高效利用。这里由灵根、功法、Buff、体质共同决定效率。

建议最终用一个倍率表达：

```text
effectiveContribution = rawValue * efficiency
```

其中 `efficiency` 可由以下因素共同计算：

- 五行灵根
- 功法加成
- Buff 修正
- 特殊体质
- 污染或反噬

### 9.4 灵根如何真正进入系统

灵根不再只是一个文案标签，而应变成玩家对不同 `element` 的吸收效率参数。

例如：

```text
金 0.4
木 0.9
水 0.6
火 0.2
土 0.3
```

这表示玩家面对相同数量的五行灵气时，最终可利用贡献不同。

这样可以自然产生以下结果：

- 同一块火灵脉，不同玩家看到的“有效灵气等级”不同
- 灵根不好的玩家不是“完全不能修”，而是效率差
- 特殊功法可以临时修正某一系的吸收效率

### 9.5 为什么还需要玩家投影层

仅有气机谱系还不够。

谱系能回答的是：

- 世界里存在什么气机
- 某格子原始持有什么资源

但谱系不能直接回答：

- 这个玩家能感知什么
- 这个玩家能吸收什么
- 这个玩家对不同资源的吸收效率是多少
- 这个玩家最终看到的家族等级是多少

因此必须在“世界原始资源层”之上，再增加一层“玩家投影层”。

### 9.6 玩家投影层的职责

玩家投影层用于把同一块地的原始气机资源，转换成某个玩家当前视角下的派生结果。

它至少负责 4 件事：

- 判定该玩家能否感知某个 `resourceKey`
- 判定该玩家能否吸收某个 `resourceKey`
- 计算该 `resourceKey` 对该玩家的有效系数
- 产出该玩家最终使用的家族级总值、等级和玩法收益

换句话说：

- 谱系负责“世界里有什么”
- 投影负责“这个玩家能从中得到什么”

### 9.7 推荐使用编译后的玩家投影配置

为了避免每 tick 在热路径里反复从功法、Buff、体质、灵根等零散字段现推导，推荐在玩家运行时维护一份“编译后的投影配置”。

建议形态：

```ts
type CompiledQiProjectionProfile = {
  revision: number;
  observableFamilies: Set<string>;
  absorbableFamilies: Set<string>;
  resourceCoefficientsBp: Record<string, number>;
  familyDisplayMode: Record<string, 'hidden' | 'observable' | 'absorbable'>;
};
```

说明：

- `revision` 用于缓存失效
- `resourceCoefficientsBp` 使用整数基点保存有效系数
- `familyDisplayMode` 用于支持“可见但不可吸收”的展示状态

### 9.8 投影配置的编译时机

编译后的投影配置不应每 tick 重算，而应只在相关状态变化时重建。

推荐触发点：

- 登录或重连回填
- 感气开关改变
- 功法切换
- Buff 增减
- 灵根、体质、境界变化
- 装备改变且该装备影响气机吸收

因此热路径只读编译结果，冷路径才做规则归并。

## 10. 等级与玩家视角

### 10.1 家族总值

某一大类气机的“原始总值”定义为同一家族下、玩家当前可见资源的总和。

例如灵气原始总值可由以下资源求和：

- `aura.refined.neutral`
- `aura.refined.fire`
- `aura.refined.water`
- `aura.dispersed.neutral`

但原始总值不一定直接用于玩家面板等级。

### 10.2 有效总值

玩家当前看到的等级，应默认根据“有效总值”计算。

有效总值定义为：

```text
effectiveFamilyValue = sum(rawValue * playerEfficiency)
```

前提是该资源同时满足：

- 玩家能感知
- 玩家能吸收

这意味着：

- 玩家默认只能从凝练灵气中得到灵气等级
- 学会某功法后，逸散灵气也能计入灵气等级
- 若后续可修魔功，则魔气可单独形成“魔气等级”

### 10.3 为什么等级按有效总值而不是原始总值

因为这能让“灵根、功法、体质、Buff”真正影响角色所处环境的意义。

否则会出现一种假的真实：

- 地图明明有火灵气
- 玩家面板也显示火灵气很强
- 但角色其实根本炼不动

按有效总值计算后，玩家看到的是“自己此刻真正能用上的世界气机”，这更贴近修仙语境。

### 10.4 观察时应显示什么

常规显示：

- 只显示家族级等级，如“灵气 3”“魔气 1”

观察副 tip：

- 当前等级
- 原始总值
- 有效总值
- 各形态拆分
- 各属性拆分
- 不可吸收项可标记为“可见但不可炼化”

## 11. 客户端显示原则

### 11.1 常规状态

不开启感气时：

- 不显示任何灵气、魔气、煞气等相关信息

### 11.2 感气视角

开启感气后：

- 地图渲染层可显示家族级感气信息
- 只显示玩家当前可感知的家族
- 感气颜色仍以“家族级表现”为主，不直接把所有底层资源键裸露给玩家

### 11.3 观察副 tip

在感气状态下使用观察：

- 右侧生成副 tip
- 主面板仍保持地块、物品、生灵等常规内容
- 副 tip 用来展开气机细节

建议展示顺序：

1. 家族汇总
2. 形态拆分
3. 属性拆分
4. 特殊说明

### 11.4 无气机时不显示副 tip

如果目标格子在玩家当前感知与吸收规则下，没有任何可展示气机：

- 不显示气机副 tip

## 12. 协议与广播原则

### 12.1 为什么不能继续只发全局 `tile.aura`

当前系统里，地块广播时可以把 `tile.aura` 统一转成等级发送给所有客户端，因为“灵气等级”对所有人相同。

但在新系统中，这个前提不成立。因为：

- 不同玩家能感知的资源不同
- 不同玩家能吸收的资源不同
- 不同玩家的灵根效率不同

所以未来的家族级等级，是“玩家相关”的派生结果，不再是地块的全局单值。

### 12.2 推荐广播方式

推荐思路如下：

- 地块底层运行时只保存原始气机资源数据
- 普通状态下，不在可见地块里附带任何气机字段
- 玩家开启感气后，服务端在发给该玩家的 AOI 可见地块中，按其个人规则派生出家族级等级
- 客户端只缓存自己当前视角下的派生结果

### 12.3 详细观察

详细资源明细不做常驻广播，而是沿用当前模式：

- 感气状态下使用观察
- 客户端主动请求一次详细数据
- 服务端返回该玩家视角下的详细家族与资源拆分

### 12.4 更新优化原则

后续仍应保持以下优化原则：

- 不开启感气的玩家，不做多余气机广播
- 开启感气的玩家，只同步其当前会看到的家族级等级
- 客户端地块更新只在“对该玩家而言的家族等级发生变化”时才算需要刷新
- 详细数值只在观察时主动拉取，不常驻推送

### 12.5 为什么必须由服务端动态计算

当系统引入灵根、功法、Buff、体质、逸散吸收能力之后，以下结果都不再是地块公共属性：

- 家族等级
- 有效总值
- 修炼增幅
- 观察副 tip 中的“可吸收 / 不可吸收”判定

这些结果都依赖玩家当前状态，因此必须由服务端动态投影后再发给客户端。

客户端只负责展示，不负责推导正确性。

### 12.6 动态计算不等于每 tick 全量暴力扫描

服务端动态计算是必须的，但实现上不能做成：

- 每 tick 为所有玩家扫描全图所有地块
- 每 tick 为所有玩家重新拼装所有气机规则
- 每 tick 对所有详细明细做全量展开

正确思路应是：

- 只对开启感气的玩家做投影
- 只对 AOI 可见地块做投影
- 只对含有气机资源的地块做投影
- 只对玩家可感知的家族做聚合
- 详细明细只在观察时单点展开

### 12.7 推荐的服务端计算链路

推荐按以下固定链路执行：

1. 读取玩家编译后的投影配置
2. 判断玩家当前是否开启感气，未开启则跳过气机投影
3. 获取该玩家 AOI 可见地块
4. 过滤出“含有任意气机资源”的可见地块
5. 对每个地块按家族聚合原始总值与有效总值
6. 按有效总值换算该玩家当前看到的家族等级
7. 只把家族级结果发送给该玩家
8. 详细明细仅在观察请求时展开并返回

### 12.8 推荐缓存层次

为了兼顾性能和可扩展性，建议使用三层缓存。

第一层：地块原始资源缓存

- 保存 `resourceKey -> TileQiRuntimeState`
- 作为世界真实状态的唯一来源
- 与玩家无关

第二层：玩家投影配置缓存

- 保存 `CompiledQiProjectionProfile`
- 只在玩家相关状态变化时重编译
- 不随每 tick 重算

第三层：玩家视角下的家族级地块缓存

- 保存该玩家最近一次看到的家族等级结果
- 只用于增量比较和 patch 发送
- 玩家关闭感气时可直接清空

### 12.9 推荐统一使用整数基点

玩家吸收效率、家族系数、形态系数、属性系数等，建议统一使用整数基点表达。

示例：

```text
10000 = 1.0
8000 = 0.8
12000 = 1.2
```

这样做的好处是：

- 热路径更稳定
- 避免浮点误差积累
- 更利于服务端日志和持久化排查
- 更适合未来协议与配置统一

### 12.10 如何避免热路径字符串和规则拼装开销

热路径里不建议频繁拆 `resourceKey` 字符串，也不建议反复从功法、Buff、灵根原始定义里临时推导。

更合理的方式是：

- 地图资源初始化时，就把每个 `resourceKey` 对应的 `family/form/element` 元信息建好
- 玩家投影配置编译时，就为每个已知 `resourceKey` 生成整数系数
- tick 热路径里只做查表、整数乘法、整数加法、等级换算

慢逻辑前置，快逻辑留在热路径。

### 12.11 脏标记应拆成两类

后续的气机系统不应只维护“地块资源变了没有”，还应维护“玩家投影视图变了没有”。

推荐拆成：

- 地块原始资源脏
- 玩家投影视图脏

地块原始资源脏的触发示例：

- 地图 tick 更新资源值
- GM 修改地块资源
- 技能外泄、阵法、污染、净化等事件改值

玩家投影视图脏的触发示例：

- 感气开关变化
- 功法切换
- Buff 增减
- 灵根、体质、装备改变
- 地图切换

只有把这两类脏分开，才能同时兼顾正确性和增量发送效率。

## 13. 玩法挂钩建议

气机系统本身只负责表达世界资源，不负责把所有玩法硬编码进去。玩法系统通过读取气机系统派生结果接入。

推荐挂钩方向如下：

### 13.1 修炼

- 读取玩家当前位置的有效灵气总值
- 按有效灵气等级提供修炼加成
- 后续可允许特定功法读取魔气、煞气

### 13.2 技能外泄

- 玩家释放技能时，可向地块注入某种逸散气机
- 例如灵技外泄增加 `aura.dispersed.neutral`
- 魔功外泄增加 `demonic.dispersed.neutral`
- 杀招外泄增加 `sha.dispersed.neutral`

### 13.3 地脉与阵法

- 阵法可提高或压制某资源的 `sourceValue`
- 福地、洞天、秘境可长期持有高阶凝练资源

### 13.4 污染与净化

- 魔域可逐步把灵气环境转化为魔气环境
- 净化阵可压低魔气、煞气并恢复灵气源值

### 13.5 灵根与功法

- 灵根决定五行效率
- 功法决定可吸收范围
- Buff 决定临时偏转、压制、放大或污染

## 14. 分阶段迁移方案

本系统不建议一步切完，建议按 4 阶段落地。

### 阶段 1：抽象资源键，兼容旧灵气

目标：

- 把当前 `aura` 内部提升为通用 `resourceKey`
- 旧 `aura` 统一映射为 `aura.refined.neutral`

效果：

- 现有地图、修炼、观察逻辑继续可用
- 底层开始具备扩展空间

### 阶段 2：接入凝练 / 逸散

目标：

- 支持 `refined` 与 `dispersed`
- 技能或事件可向地块注入逸散灵气

效果：

- 逸散气机开始真正参与环境
- 当前灵气系统从“单值”升级为“同族多层”

### 阶段 3：接入五行属性

目标：

- 资源键加入 `element`
- 灵根正式影响吸收效率

效果：

- 五行灵气不再是文案设定，而是底层真实资源

### 阶段 4：接入魔气、煞气与玩家权限

目标：

- 扩展 `family`
- 让功法、Buff、体质控制感知和吸收权限

效果：

- 玩家视角与环境关系开始真正分化
- 世界气机生态系统成型

## 15. 当前推荐定稿

综合当前项目阶段与后续扩展需求，推荐最终定稿如下：

- 气机使用三维模型：`family + form + element`
- 当前灵气整体映射为 `aura.refined.neutral`
- 所有气机统一存入地块运行时总存储
- 每种资源独立配置半衰期、固定衰减、最低衰减
- 收敛逻辑作用于“当前值与源值的差值”
- 玩家看到的家族等级按“自己当前可利用的有效总值”计算
- 灵根决定五行效率，功法和 Buff 决定可感知与可吸收范围
- 常规 UI 只显示家族级结果，详细拆分仅在感气观察副 tip 中展开

## 16. 首轮实现边界建议

为了避免首轮实现过大，建议先明确哪些内容进入第一阶段，哪些暂时只定设计不立刻做。

第一阶段建议进入实现：

- 通用资源键
- 地块统一资源运行时
- `aura.refined.neutral` 兼容迁移
- 凝练 / 逸散两层
- 技能外泄可增加逸散灵气

第一阶段建议暂不一次性做满：

- 五行全量玩法结算
- 魔气与煞气的完整生态
- 玩家侧复杂多重炼化公式
- 多家族并行面板与高级视觉表现

原因：

- 先把底层和迁移路径打稳，比一次塞满全部玩法更重要
- 只要底层模型定对，后续玩法接入是增量工作

### 16.1 当前已落地状态

截至当前代码版本，第一阶段已经落地到以下程度：

- 共享层已经引入通用气机资源键模型，默认灵气映射为 `aura.refined.neutral`
- 玩家侧已经接入服务端动态气机投影，装备、Buff、属性附加可改变“可见 / 可吸收 / 效率”
- 地图运行时资源存储已从单独 `auraStates` 升级为 `resourceStates[mapId][resourceKey][tileKey]`
- 旧 `resources.aura` 与旧灵气 JSON 持久化可自动兼容读取，并在后续落盘时写回通用资源键
- 当前 tick 中真正参与半衰期推进的仍只有默认灵气资源 `aura.refined.neutral`
- 对外兼容层暂时仍保留 `tile.aura`、`getTileAura()`、`setTileAura()` 与观察面板里的旧 `aura` 展示键

这意味着：

- 底层持久化与运行时容器已经完成泛化
- 现有玩法链路仍可继续工作
- 凝练 / 逸散多资源并行推进、五行资源源点、魔气 / 煞气收敛规则还需要继续补完

## 17. 一句话结论

后续应把“地图灵气系统”升级为“世界气机生态系统”。

它不是为灵气单独做补丁，而是让灵气、逸散、五行、魔气、煞气、灵根、功法、Buff、阵法、污染都挂到同一套底层之上，并且在不推翻当前内容的前提下，逐步进入真实数值与真实表现。
