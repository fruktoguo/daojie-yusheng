# ADR-0002: Tick 调度模型

- **状态**: 已采纳
- **日期**: 2025-01-20

## 背景

MMO 需要一个统一的时间推进机制来处理玩家行动、怪物 AI、状态更新、战斗结算等。需要在响应性和服务端负载之间取得平衡。

## 决策驱动因素

- 公平性：所有玩家在同一时间框架内行动
- 性能：服务端能承受多玩家并发
- 确定性：相同输入产生相同结果，便于调试和回放
- 可扩展：支持多地图独立运行

## 考虑的选项

### 选项 A: 事件驱动（无 tick）

- 优点：响应即时，无固定开销
- 缺点：时序难以保证，状态更新分散

### 选项 B: 全局统一 tick

- 优点：简单，时序清晰
- 缺点：单点瓶颈，地图间耦合

### 选项 C: 每地图独立 tick

- 优点：地图间解耦，可并行
- 缺点：跨地图交互需要额外处理

## 决策

采用 **选项 C: 每地图独立 tick**，tick 频率 1Hz。

每张地图独立运行 tick 循环，地图间通过消息队列通信。1Hz 频率在响应性和负载之间取得平衡。

## 后果

### 正面

- 地图可独立扩展，负载分散
- 单地图故障不影响其他地图
- tick 内操作顺序确定，便于调试

### 负面

- 跨地图操作有额外延迟
- 1Hz 对快节奏操作可能感觉迟钝

### 约束

- 玩家意图在 tick 内收集，tick 结束时统一执行
- 同类可覆盖意图以最后一次为准（如寻路目标）
- 不可覆盖意图（如交易）需要排队或拒绝机制
- tick 内禁止直接数据库 IO

## 验证方式

- [ ] 单地图 tick 延迟不超过 100ms
- [ ] 地图间消息延迟不超过 2 tick
- [ ] tick 内无数据库阻塞调用
