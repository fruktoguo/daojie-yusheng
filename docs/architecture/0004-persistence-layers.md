# ADR-0004: 持久化分层策略

- **状态**: 已采纳
- **日期**: 2025-02-01

## 背景

MMO 需要持久化玩家数据、地图状态、交易记录等。需要在数据安全性和性能之间取得平衡，同时支持断线重连和服务器重启恢复。

## 决策驱动因素

- 数据安全：玩家资产不能丢失
- 性能：tick 内不能阻塞于 IO
- 恢复能力：服务器重启后能恢复状态
- 审计需求：关键操作可追溯

## 考虑的选项

### 选项 A: 实时写入数据库

- 优点：数据最安全
- 缺点：性能差，tick 阻塞

### 选项 B: 纯内存 + 定时快照

- 优点：性能好
- 缺点：可能丢失数据

### 选项 C: 分层持久化

- 优点：平衡安全和性能
- 缺点：实现复杂

## 决策

采用 **选项 C: 分层持久化**。

| 层次 | 存储 | 用途 | 写入时机 |
|------|------|------|----------|
| 内存 | 进程内存 | 运行时状态 | 实时 |
| Redis | Redis | 在线态、会话、缓存 | 实时/异步 |
| PostgreSQL | 数据库 | 持久化真源 | 异步批量 |

## 后果

### 正面

- tick 内无数据库阻塞
- 关键数据有持久化保障
- 支持快速恢复

### 负面

- 需要维护多层数据一致性
- 异步写入可能有短暂数据丢失窗口

### 约束

- PostgreSQL 是所有"下次还在"数据的唯一真源
- Redis 只做缓存和在线态，不是真源
- tick 内通过 outbox 队列异步写入数据库
- 玩家下线时必须完成 flush

## 验证方式

- [ ] tick 内无同步数据库调用
- [ ] 玩家下线后数据完整落库
- [ ] 服务器重启后玩家数据完整恢复
