# 架构改进分析

统计日期：2026-05-11

本文基于当前生产主线代码，按架构层逐一分析可改进点。每项标注优先级（P0 紧急 / P1 重要 / P2 改善）、预估工作量和开展方式。

---

## 1. 世界 Tick 与运行时编排

当前 `WorldRuntimeService` 是一个 4000+ 行的巨型编排器，承担了命令分发、战斗调度、任务、NPC、采集、炼丹、强化、传送、自动战斗、寻路等所有职责。

- [ ] **1.1** [P1] 将 `WorldRuntimeService` 拆分为薄编排层 + 领域命令处理器
  - 当前状态：已拆出 `WorldRuntimeCombatActionService`、`WorldRuntimeAlchemyService` 等，但主类仍保留大量 `dispatch*` 方法作为透传入口
  - 开展方式：
    1. 定义 `WorldRuntimeCommandRouter`，按 command kind 分发到对应领域 service
    2. 主类只保留 `advanceFrame()`、实例生命周期管理、玩家位置索引
    3. 每个 `dispatch*` 方法迁移到对应领域 service，主类只做一行转发
    4. 逐个迁移，每迁一个跑 `pnpm verify:quick`

- [ ] **1.2** [P2] 将 tick 内各域推进顺序显式化为 Phase 枚举
  - 当前状态：`advanceFrame` 内按代码顺序调用各域，顺序隐含在代码里
  - 开展方式：
    1. 定义 `TickPhase` 枚举（PendingCommands → MonsterAI → Combat → PlayerTick → CraftTick → ResourceFlow → PersistenceMark）
    2. 每个 phase 对应一个方法，`advanceFrame` 按枚举顺序调用
    3. 便于后续插入新 phase 或调整顺序时有明确语义

- [ ] **1.3** [P2] 抽取自动战斗为独立 System
  - 当前状态：`WorldRuntimeAutoCombatService` 已拆出，但仍由 `WorldRuntimeService.materializeAutoCombatCommands()` 调用
  - 开展方式：让 `advanceFrame` 直接调用 `autoCombatService.materialize()`，移除主类中间方法

## 2. 地图实例运行时

`MapInstanceRuntime` 约 3000+ 行，是地图内所有状态的 Aggregate Root，但职责过多导致难以维护。

- [ ] **2.1** [P1] 将建筑/风水逻辑从 MapInstanceRuntime 中彻底分离
  - 当前状态：建筑放置、拆除、风水重算、房间拓扑等方法仍在 `MapInstanceRuntime` 内
  - 开展方式：
    1. 已有 `world-runtime-building.service.ts` 的外部 helper 函数
    2. 将 `MapInstanceRuntime` 中 `configureBuildingRuntime()`、`placeBuildingInstance()`、`deconstructBuildingInstance()` 等方法改为委托到外部 service
    3. `MapInstanceRuntime` 只暴露 `getBuildingRuntime()` 访问器

- [ ] **2.2** [P1] 将怪物 AI 推进逻辑从 MapInstanceRuntime 中分离
  - 当前状态：`advanceMonsters()`、`resolveMonsterTarget()`、`stepMonsterIdleRoam()` 等全在实例类内
  - 开展方式：
    1. 新建 `MapInstanceMonsterAdvancer`，接收实例引用，负责怪物 tick 推进
    2. `MapInstanceRuntime.advanceMonsters()` 改为一行委托
    3. 怪物状态仍存在实例内（保持 Aggregate 一致性），但行为逻辑外移

- [ ] **2.3** [P2] 将视野/FOV 计算抽为独立模块
  - 当前状态：`castLight()`、`collectVisibleTileVisibility()`、`canSeeTileFrom()` 等 shadowcasting 算法直接写在实例类里
  - 开展方式：抽为纯函数模块 `fov.ts`，接收 `(width, height, isSightBlocked) => Set<number>`

- [ ] **2.4** [P2] 将持久化 delta 构建逻辑从实例类中分离
  - 当前状态：`buildMonsterRuntimePersistenceDelta()`、`buildGroundPersistenceDelta()` 等 20+ 方法在实例类内
  - 开展方式：新建 `MapInstancePersistenceProjector`，从实例 dirty 状态构建 delta，实例类只暴露 dirty 标记和状态访问

## 3. 玩家运行时

`PlayerRuntimeService` 约 5000+ 行，200+ 方法，既管状态又管行为又管持久化标脏。

- [ ] **3.1** [P1] 将玩家状态操作按领域拆分为子 service
  - 当前状态：背包、装备、钱包、buff、修炼、任务、统计、离线收益全在一个 service
  - 开展方式：
    1. `PlayerInventoryService` — 背包增删、排序、容量检查
    2. `PlayerBuffService` — buff 添加、tick、移除、查询
    3. `PlayerProgressionService`（已有）— 修炼、突破、天门
    4. `PlayerVitalService` — 生命/灵力恢复、伤害、死亡
    5. 主 service 保留 `ensurePlayer()`、`advanceTick()`、`snapshot()` 等编排方法
    6. 逐个领域迁移，每迁一个验证 `pnpm verify:quick`

- [ ] **3.2** [P1] 将玩家状态从裸对象改为有行为的领域实体
  - 当前状态：玩家运行时状态是 plain object，所有行为在外部 service
  - 开展方式：
    1. 定义 `RuntimePlayer` class，封装 `canReceiveItem()`、`debitWallet()`、`applyDamage()` 等高频操作
    2. service 层调用实体方法而非直接操作字段
    3. 渐进式：先把最高频的 10 个操作封装，不需要一次全改

- [ ] **3.3** [P2] 离线收益计算逻辑独立为专用 service
  - 当前状态：`buildOfflineGainSnapshot()`、`diffOfflineGainProgress()` 等 50+ 函数在 `player-runtime.service.ts` 内
  - 开展方式：新建 `PlayerOfflineGainService`，把所有 `offlineGain*` / `buildOfflineGain*` 函数迁出

- [ ] **3.4** [P2] 玩家统计（PlayerStatistic）逻辑独立
  - 当前状态：`recordPlayerStatisticTotals()`、`buildPlayerStatisticTotalsView()` 等在主 service
  - 开展方式：新建 `PlayerStatisticService`

## 4. 战斗链路

当前 Pipeline 实现质量高，主要改进点在旧分支收口和统一入口。

- [ ] **4.1** [P1] 完成旧生产分支到统一 CombatAction 主链路的迁移
  - 当前状态：ADR 已采纳，wrapper 和 adapter 已有，但旧分支仍在生产运行
  - 开展方式：按战斗链路重构计划阶段 10-14 执行

- [ ] **4.2** [P2] 将 `PlayerCombatService` 的技能公式解析与结算分离
  - 当前状态：`resolveSkillFormulaVar()`、`evaluateSkillFormula()` 和 `castSkill()` 混在一起
  - 开展方式：公式解析抽为纯函数模块 `skill-formula.ts`，`PlayerCombatService` 只负责调用和组装

- [ ] **4.3** [P2] pending cast 持久化到 Redis 实现断线重连恢复
  - 当前状态：已有 `pending-combat-cast-recovery.helpers.ts` 契约层，未接入真实 Redis
  - 开展方式：按战斗链路计划的持久化阶段执行

## 5. 持久化层

整体设计优秀，改进点在细节。

- [ ] **5.1** [P1] 地图实例分域 flush 完成 row-level delta 硬切
  - 当前状态：`monster_runtime`、`tile_damage`、`ground_item` 已有 delta API，但部分域仍走整域 replace
  - 开展方式：按「地图实例持久化分域收口计划」执行剩余批次

- [ ] **5.2** [P2] 统一 flush worker 和周期兜底的职责边界
  - 当前状态：worker 和 interval flush 可能同时观察同一 dirty domain
  - 开展方式：worker 成为高频域主路径，interval 只做 shutdown 和超时兜底

- [ ] **5.3** [P2] `PlayerDomainPersistenceService` 方法数过多，按域拆分文件
  - 当前状态：单文件 2000+ 行，所有域的 save/load 混在一起
  - 开展方式：按域拆为 partial 文件或子模块（`player-persistence-inventory.ts`、`player-persistence-progression.ts` 等）

## 6. 网络同步层

Delta Projection 实现精细，改进点在可维护性。

- [ ] **6.1** [P1] `world-projector.helpers.ts` 拆分
  - 当前状态：单文件 3000+ 行，包含所有 diff 函数、clone 函数、类型定义
  - 开展方式：
    1. `projector-diff.ts` — 所有 `diff*` 函数
    2. `projector-clone.ts` — 所有 `clone*` 函数
    3. `projector-compare.ts` — 所有 `isSame*` 函数
    4. `projector-types.ts` — 接口和类型定义
    5. 主文件只做 re-export 和顶层编排

- [ ] **6.2** [P2] 将 `WorldSyncAuxStateService` 的缓存比较逻辑统一化
  - 当前状态：每个辅助状态（时间、小地图、威胁箭头、战利品窗口）各自有一套 `isSame*` + `clone*`
  - 开展方式：抽象为 `SyncSlot<T>` 泛型容器，统一 `{ previous: T, isDirty(): boolean, commit(): void }`

- [ ] **6.3** [P2] 协议包体自动化回归门禁
  - 当前状态：`audit:protocol` 已有，但只覆盖部分用例
  - 开展方式：扩展 protocol audit 覆盖所有高频事件，CI 中自动对比包体大小回归

## 7. Socket 网关

- [ ] **7.1** [P1] `world.gateway.ts` 拆分为按领域的 handler 模块
  - 当前状态：单文件 2000+ 行，所有 `@SubscribeMessage` 混在一起
  - 开展方式：
    1. 已有 `WorldGatewayMarketHelper`、`WorldGatewayActionHelper`、`WorldGatewayBootstrapHelper`
    2. 继续拆出 `WorldGatewayCraftHelper`、`WorldGatewayMailHelper`、`WorldGatewayCombatHelper`
    3. 主 gateway 只做路由分发

- [ ] **7.2** [P2] 统一 handler 的错误处理和响应格式
  - 当前状态：部分 handler 用 `emitError`，部分用 `emitProtocolFailure`，部分直接 return
  - 开展方式：定义统一的 `GatewayResult<T>` 类型，所有 handler 返回统一格式，由网关层统一处理错误发送

## 8. 客户端 UI

- [ ] **8.1** [P1] 抽象统一的面板 patch 框架
  - 当前状态：每个面板自己实现 `tryPatch*`、`captureViewState`、`restoreViewState`、`buildStableRenderKey`
  - 开展方式：
    1. 定义 `PatchablePanel<TState>` 基类或 mixin
    2. 提供 `captureState() → TState`、`shouldPatch(prev, next) → boolean`、`applyPatch(root, prev, next)` 生命周期
    3. 新面板用框架，旧面板逐步迁移

- [ ] **8.2** [P1] 将 3000+ 行的巨型面板拆分为子组件
  - 当前状态：`ActionPanel`（5000+ 行）、`MarketPanel`（4000+ 行）、`CraftWorkbenchModal`（4000+ 行）
  - 开展方式：
    1. `ActionPanel` → `SkillManagementSubpanel`、`CombatSettingsSubpanel`、`SectManagementSubpanel`
    2. `MarketPanel` → `MarketBrowseView`、`MarketTradeDialog`、`AuctionView`
    3. `CraftWorkbenchModal` → `AlchemyView`、`EnhancementView`、`CraftQueueView`
    4. 每个子组件独立文件，有自己的 state/render/patch 生命周期

- [ ] **8.3** [P2] 新增面板优先使用 React（react-ui）
  - 当前状态：`react-ui` 目录已存在但使用有限
  - 开展方式：新面板默认用 React 实现，通过 bridge 与现有 runtime store 通信，不迁移旧面板

## 10. 跨层通用问题

- [ ] **10.1** [P1] 建立模块大小门禁
  - 当前状态：无自动化检查，巨型文件持续膨胀
  - 开展方式：
    1. 在 CI 中加入文件行数检查，超过 1500 行的 `.ts` 文件报 warning，超过 3000 行报 error
    2. 新增代码不允许让已超标文件继续膨胀

- [ ] **10.2** [P1] 统一服务拆分模式
  - 当前状态：有的用 helper 函数文件，有的用独立 service class，有的用 partial-like 多文件
  - 开展方式：
    1. 纯计算/无状态 → 独立 `.helpers.ts` 纯函数文件
    2. 有状态/需要依赖注入 → 独立 NestJS service
    3. 编排/协调 → 薄 facade service
    4. 文档化这个约定，新代码遵循

- [ ] **10.3** [P2] 热路径函数标注 `@hot` 注释并建立 bench 回归
  - 当前状态：`bench:combat` 已有，但其他热路径（寻路、AOI 收集、属性重算）缺少基准
  - 开展方式：为 `findPathPointsOnMap`、`collectVisibleTileVisibility`、`recalculateMonsterDerivedState` 等补 bench

## 优先级总览

| 优先级 | 数量 | 核心主题 |
|--------|------|----------|
| P0 | 0 | 无紧急阻塞项 |
| P1 | 12 | 巨型类拆分、旧链路收口、门禁建立 |
| P2 | 13 | 可维护性改善、框架抽象、bench 补充 |

建议执行顺序：
1. 先做 10.1（门禁）防止继续恶化
2. 再做 1.1 + 3.1（最大的两个巨型 service 拆分）
3. 然后 7.1 + 6.1（网关和投影器拆文件，低风险高收益）
4. 最后按各自计划推进 4.1、5.1（战斗收口、持久化硬切）
