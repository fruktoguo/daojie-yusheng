# ADR-0003: 网络同步分层

- **状态**: 已采纳
- **日期**: 2025-02-01

## 背景

MMO 网络流量是主要成本和性能瓶颈。需要设计合理的同步策略，在实时性和带宽之间取得平衡。

## 决策驱动因素

- 带宽成本：移动端流量敏感
- 实时性：战斗等场景需要快速响应
- 可扩展：支持大量玩家同屏
- 断线重连：玩家重连后能快速恢复状态

## 考虑的选项

### 选项 A: 全量同步

- 优点：实现简单，状态一致
- 缺点：带宽爆炸，不可扩展

### 选项 B: 增量同步

- 优点：带宽低
- 缺点：状态可能不一致，重连复杂

### 选项 C: 分层同步

- 优点：按需同步，灵活
- 缺点：实现复杂

## 决策

采用 **选项 C: 分层同步**。

将数据分为多个层次，按不同策略同步：

| 层次 | 内容 | 同步策略 |
|------|------|----------|
| 首包 | 地图静态、玩家完整状态 | 进入场景时全量 |
| 高频 | 位置、气机、战斗状态 | 每 tick 增量 |
| 低频 | 背包、属性、技能 | 变化时增量 |
| 按需 | 详情、历史、排行 | 请求时拉取 |

## 后果

### 正面

- 带宽可控，支持移动端
- 高频数据实时，低频数据按需
- 断线重连只需重发首包

### 负面

- 实现复杂度高
- 需要维护多套同步逻辑

### 约束

- 高频包禁止混入静态数据、长文本、完整详情
- 能发 ID/revision/patch 的不发完整对象
- 能单播就不 AOI，能 AOI 就不全图

## 验证方式

- [ ] 高频包大小不超过 1KB
- [ ] 100 玩家同屏带宽不超过 100KB/s
- [ ] 断线重连恢复时间不超过 3 秒
