# 商业级持久化刷盘管线改造计划

> 目标：把当前单进程 inline flush 从低频兜底升级为可支撑 5000 在线玩家、10000 地图实例的商业级持久化写入管线。核心不是取消处理上限，而是按优先级、按实体聚合、可横向扩展、可自动反压地消费玩家和实例 dirty 状态。

## 当前问题

- 生产 stack 当前 `SERVER_FLUSH_TASK_RUNTIME_MODE=inline`，flush 消费与 server 主进程同进程运行。
- 旧默认每轮只认领 `64` 个玩家 task 和 `64` 个实例 task，且玩家 task 按 domain 行串行处理。
- `attr / progression / technique / vitals` 等同一玩家多个脏域会在 ledger 中表现为多行，裸按 domain 消费会放大排队。
- GM worker 面板只显示 backlog/窗口完成数，缺少 dirty 产生率、backlog 增长率、flush pool waiting、锁等待和循环耗时，容易把容量问题误读成“心跳异常”。
- 10000 地图场景下，`time / monster_runtime / tile_resource / tile_damage / fengshui` 不能依赖单 loop 小批量全局额度。

## 商业级原则

- 服务端运行时仍是权威真源；flush 只负责把已发生的权威状态异步落到数据库。
- 数据库是真正长期真源；Redis、内存和 ledger 只做在线态、调度态或缓存态。
- tick、登录、重连、交易、GM 等实时链路优先，flush 必须受 DB pool 和锁等待反压。
- 按实体聚合消费：玩家以 `playerId` 为单位，实例以 `instanceId + ownershipEpoch` 为单位。
- 按优先级配额消费：关键资产/位置/presence 优先，派生属性和地图自然变化可合并、可延迟。
- 所有 worker/flush 失败必须分类：连接超时、死锁、租约失效、不变量错误和未知错误不能同一种重试策略。
- 验证必须覆盖 ledger 认领、执行、批量确认、失败重试、清理和回读。

## 目标架构

### 主进程

- 收集 runtime dirty domains。
- 写入 `player_flush_ledger` / `instance_flush_ledger`。
- 在 inline 模式下作为兜底消费器运行。
- 不在 tick 热路径直接做数据库写入。

### Flush Worker 服务

- 后续独立部署，可水平扩容。
- 使用 `FOR UPDATE SKIP LOCKED` 和 ownership epoch 保证多 worker 不重复写错。
- 玩家侧按 `playerId` 聚合，一次写掉本轮多个 dirty domain。
- 实例侧按 `instanceId + ownershipEpoch` 聚合，能批量的 domain 走批量 upsert。
- 失败时按分类退避；不变量错误进入诊断/死信，不无限重试。

### 数据库与反压

- flush 使用独立 DB pool。
- 当 flush pool waiting 或 PG lock wait 超阈值时自动降速。
- 批量 mark flushed/retry，避免确认阶段本身写放大。
- 低优先级高频 domain 使用 coalesce window。

### GM 观测

- 展示 dirty 产生率、claim 认领率、flush 完成率、backlog 增长率。
- 展示 flush pool waiting、PG lock wait、P50/P95/P99。
- 展示每个 domain 最老 dirty 年龄，而不是只展示 ledger 总行数。
- 告警区分 backlog high、inactive、dead letter、db backpressure、lock wait。

## 实施阶段

### Phase 1：inline 消费器容量底座

- [x] 玩家/实例独立 claim 预算，默认从全局 `64` 提升为玩家 `256` + 实例 `256`。
- [x] 玩家和实例消费使用受控并发，默认各 `8`。
- [x] 玩家按 `playerId` 聚合多 domain 一次刷盘；失败后降级到逐 domain 隔离。
- [x] 实例按 `instanceId + ownershipEpoch` 聚合多 domain 一次刷盘；失败后降级到逐 domain 隔离。
- [x] ledger 批量 `mark flushed` / `mark retry`，降低确认写放大。
- [x] claim helper 内部上限放宽到 `5000`，让生产环境变量能实际调高。
- [x] GM 面板展示新容量指标。

### Phase 2：优先级调度

- [x] 高优先级：`presence / position_checkpoint / world_anchor / inventory / equipment / market / mail / GM edit`。
- [x] 中优先级：`progression / technique / quest / vitals / buff / profession` 及其他玩家普通域。
- [x] 低优先级：`time / monster_runtime / tile_resource / tile_damage / fengshui`。
- [x] 每级独立预算，低优先级不能挤占高优先级。

### Phase 3：独立 worker 服务化

- [x] 增加 `flush-worker` 服务入口，生产 stack 可配置副本数。
- [x] 主进程保留 inline 兜底；当前生产默认仍为 inline，独立 worker 副本默认 `0`，用于灰度验证。
- [ ] worker 不持有 runtime 权威；只通过正式 persistence service 和 ledger 协调写入。
- [x] 支持水平扩容、灰度切换和安全回滚的部署开关。

> 当前代码已具备独立 worker 入口，但玩家快照和实例 delta 仍依赖主进程权威运行态构造；因此不能把生产默认直接切到无状态 worker。已补 no-op flush 不得 mark flushed 的保护，避免独立 worker 在缺少运行态时误确认账本。真正完全独立 worker 需要 Phase 4 把 dirty payload/staging 写入持久化中间层后再切默认。

### Phase 4：批量写入扩大

- 玩家 domain write plan 合并成更大批量 SQL。
- 实例 `time / monster_runtime / tile_resource / tile_damage` 扩展批量 upsert。
- ledger dirty upsert 支持批量写入，减少收集阶段 SQL 次数。

### Phase 5：容量证明

- 压测 5000 玩家 dirty 产生模型：低/中/高 mutation 率。
- 压测 10000 地图实例：冷态、活跃子集、全活跃 checkpoint。
- 输出吞吐、P95、DB pool waiting、lock wait、WAL 压力和 backlog 增长率。

## 非目标

- 不改变玩家资产、战斗、移动、交易、邮件、GM 修改的权威语义。
- 不把长期真源移到 Redis 或内存。
- 不让 worker 直接持有 socket、runtime 对象或数据库外的第二真源。
- 不用无限并发替代容量设计。

## 验证矩阵

- `pnpm --filter @mud/server smoke:flush-task-runtime`：验证 dirty 收集、ledger claim、实体聚合 flush、批量确认闭环。
- `pnpm --filter @mud/server smoke:flush-pool-backpressure`：验证 flush pool waiting 会触发反压，不挤占 DB。
- `pnpm --filter @mud/server smoke:player-persistence-flush`：验证玩家分域写入、markPersisted 和恢复水位。
- `pnpm --filter @mud/server compile`：验证 TypeScript 和服务端构建。
- 后续独立 worker 化后补 `with-db` 压测和生产 shadow。

## 完成判定

- 玩家/实例 dirty 产生率低于处理率时 backlog 能稳定下降。
- flush DB pool waiting 长期为 0 或低位波动，出现等待时自动降速。
- 同一玩家多域 dirty 不再逐域串行放大。
- 同一实例多域 dirty 不再逐域串行放大。
- GM 面板能解释“为什么积压”，而不是只显示“积压了”。
