# 旧实现残留清理计划

## 背景

项目经历多轮重构（持久化分域、地图分层、通知结构化、战斗重构等），大量旧实现未及时删除，导致：
- 新开发者无法判断哪个是当前主线实现
- IDE 搜索和跳转命中废弃代码
- 旧代码与新代码共存增加维护负担和出错概率
- 部分旧代码仍被注入/导入但实际不执行任何有效逻辑

本计划基于 2025-05-12 全面排查结果，按优先级分阶段清理。

## 原则

- 只删除确认无活跃调用方或已被新实现完全替代的代码
- 有活跃调用方的旧兼容层按迁移计划推进，不强删
- 每阶段清理后执行对应验证门禁
- 不改变任何运行时行为，纯减法操作

## 不动项（设计如此，暂不清理）

| 项目 | 原因 |
|------|------|
| `docker-compose.shadow.yml` | shadow 验证专用，端口和密码隔离有实际意义 |
| `LEGACY_BACKUP_FILE_KIND` 等旧备份常量 | 需要识别和恢复历史备份文件 |
| 玩家属性旧字段兼容 `comprehension→strength` | 运行时仍在使用，等存档全量迁移 |
| `chat-storage.ts` 旧 localStorage 清理逻辑 | 一次性迁移用，对已有玩家仍有价值 |
| `config-editor` 怪物三种 sourceMode | 旧数据只读兼容，等数据全量迁移到 tendency |
