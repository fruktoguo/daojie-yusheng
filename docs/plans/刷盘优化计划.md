# 刷盘优化计划

> 目标：降低玩家刷盘与 flush ledger 的体积膨胀，明确各分域是否全量落盘，逐步把最重的域拆成更薄的写入路径。

## 目标

- [ ] 找出玩家刷盘里最肥的分域和 payload 来源
- [ ] 区分 ledger 中间态和最终真源写入
- [ ] 明确哪些玩家分域是整组重写，哪些是增量 upsert + prune
- [ ] 保持空覆盖、item_instance_id 修复、恢复语义不退化
- [ ] 保持 flush ledger 的回收自动化，不误删 pending/retry/claim

## 现状分区

- [ ] `player_flush_ledger` / `instance_flush_ledger` 只作为刷盘中间账本
- [ ] `flush-ledger-retention` 负责自动回收完成账本
- [ ] `payload_jsonb` 进入 TOAST 是 ledger 体积膨胀主因
- [ ] `inventory` 是最重的玩家分域之一

## 开发任务

- [ ] 统计各玩家分域的实际 payload 大小分布
- [ ] 统计 `player_flush_ledger` 中各 domain 的平均 / 最大 `payload_jsonb` 大小
- [ ] 识别 `inventory`、`buff`、`quest`、`progression` 的主要膨胀来源
- [ ] 评估 `inventory` 是否能从整组重写继续拆成更细粒度写计划
- [ ] 评估 `buff`、`quest`、`progression` 是否能继续拆薄
- [ ] 保留 `replacePlayerInventoryItems()` 的空覆盖和修复语义
- [ ] 保留 `replacePlayerEquipmentSlots()` 的空覆盖和修复语义
- [ ] 保留现有 `prunePlayerRowsBySnapshotKeys()` 的回收语义
- [ ] 保留 `flush-ledger-retention` 的自动回收语义

## 验收

- [ ] 能明确说出哪些分域是全量、哪些不是
- [ ] 能明确说出哪些分域是账本大头
- [ ] `inventory` 的落盘粒度有明确下一步方案
- [ ] flush ledger 回收后能观察到物理空间下降
- [ ] 不引入资产丢失、重复写入、空覆盖回退问题

## 风险

- [ ] 过度拆分会增加写路径复杂度
- [ ] inventory 的 item_instance_id 修复必须保留
- [ ] 不能把 dirty / retry / claim 中的账本误删
- [ ] 不能把 ledger 当成最终真源
