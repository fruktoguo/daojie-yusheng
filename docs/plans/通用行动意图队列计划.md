# 通用行动意图队列计划

## 1. 目标

将挖矿、采集和标准技艺（炼丹/炼器/强化/建造）统一到一个**行动意图系统**中：

- 记录玩家当前持久意图，被动打断后能自动恢复
- 每种 kind 存储不同 payload，但对外暴露统一接口（执行、进度查询、结束判定、恢复条件检查）
- 替代现有 `TechniqueActivityQueueService`，成为所有技艺活动的统一意图容器
- 挖矿（纯战斗流程）和采集（gatherJob tick 流程）都能接入同一套意图管理

## 2. 现状问题

| 问题 | 影响 |
|------|------|
| 挖矿没有持久意图概念 | 被反击切目标后不会自动回到矿脉 |
| 采集中断后 sleepToQueue，但主动移动也 sleep 而非 remove | 玩家走开后又自动恢复采集，语义错误 |
| 现有 `TechniqueActivityQueueService` 只有 sleepToQueue/tickQueue | 没有统一的执行/进度/结束接口 |
| 挖矿无法接入现有队列 | 挖矿走自动战斗循环，不走 job/tick |
| 驱动时机单一 | 现有队列只在"没有活跃 job"时推进，不适合挖矿的恢复场景 |
| 被动打断与主动终止没有统一区分 | 各处散落的清理逻辑不一致 |

## 3. 设计决策

| 决策 | 选择 | 理由 |
|------|------|------|
| 挖矿启动入口 | 沿用锁定目标（engageBattle） | 挖矿本质是对矿脉持续自动战斗，已有完整链路 |
| 挖矿适用范围 | 仅矿脉地块（isOreMinableTileType） | 普通可破坏地块无"持续挖"需求 |
| 关闭自动战斗 | 同时终止挖矿意图 | 挖矿执行依赖自动战斗循环 |
| 主动移动 | remove（不再 sleep） | 主动移动 = 明确意图变更 |
| 持久化 | 不持久化 | 断线后地图状态可能已变，恢复失效意图风险大于收益 |
| 与现有 pipeline 关系 | 意图层在 pipeline 之上 | ActionIntentStrategy.execute 内部调用 pipeline.start |
| 队列 vs 单条 | 队列，按 slot 分组 | 允许"挖矿 + 排队炼丹"并存 |

## 4. 架构总览

```
┌─────────────────────────────────────────────────────────────────────┐
│                      ActionIntentService                             │
│  （统一意图调度器）                                                   │
├─────────────────────────────────────────────────────────────────────┤
│  push(player, intent, slot)     — 入队                               │
│  sleep(player, intentId, reason) — 转休眠                            │
│  wake(player, intentId)          — 唤醒                              │
│  remove(player, intentId)        — 移除                              │
│  removeByKind(player, kind)      — 按 kind 移除                      │
│  removeBySlot(player, slot)      — 按 slot 移除                      │
│  clear(player)                   — 全部清空                          │
│  tick(player, slot, ctx)         — 推进指定 slot                     │
│  getActive(player, slot)         — 获取 slot 当前 active 项          │
│  getSleeping(player, slot)       — 获取 slot 中 sleeping 项          │
│  list(player)                    — 列出所有意图                      │
└──────────────────────────────────┬──────────────────────────────────┘
                                   │ 委托
                                   ▼
┌─────────────────────────────────────────────────────────────────────┐
│              ActionIntentStrategy 接口                                │
│  （每种 kind 实现一个）                                               │
├─────────────────────────────────────────────────────────────────────┤
│  kind: RuntimeTechniqueActivityKind                                  │
│  slot: 'field' | 'craft'                                             │
│                                                                      │
│  execute(player, intent, ctx)       — 执行/恢复                      │
│  queryProgress(player, intent, ctx) — 进度查询                       │
│  isCompleted(player, intent, ctx)   — 任务结束判定                   │
│  canResume(player, intent, ctx)     — 恢复条件检查                   │
│  onSleep(player, intent, reason, ctx) — 转休眠时清理                 │
│  onTerminate(player, intent, ctx)   — 终止时清理                     │
│  conflictsWith(other: ActionIntent) — 互斥判定                       │
└─────────────────────────────────────────────────────────────────────┘
```

### 4.1 Slot 分组

| Slot | 包含 kind | 互斥规则 |
|------|-----------|----------|
| `field` | mining, gather, building | 同时只能有一条 active |
| `craft` | alchemy, forging, enhancement | 同时只能有一条 active |

两个 slot 之间不互斥：玩家可以同时有一条 field active（挖矿）和一条 craft active（炼丹排队）。

同 slot 内允许多条 pending/sleeping 排队，但只有头部一条可以是 active。

### 4.2 意图状态机

```
         push
          │
          ▼
      ┌────────┐
      │pending │ ──── tick/条件满足 ────→ ┌────────┐
      └────────┘                          │ active │
          ▲                               └────────┘
          │                                    │
     wake │                          被动打断  │
          │                                    ▼
      ┌────────┐                          ┌─────────┐
      │sleeping│ ◄────── onSleep ─────────│ active  │
      └────────┘                          └─────────┘
          │
          │ 主动终止 / isCompleted
          ▼
       [removed]
```

## 5. 统一接口详解

### 5.1 ActionIntent 数据结构

```typescript
interface ActionIntent {
  id: string;                              // 唯一标识
  kind: RuntimeTechniqueActivityKind;      // 'mining'|'gather'|'alchemy'|'forging'|'enhancement'|'building'
  slot: 'field' | 'craft';                // 所属槽位
  state: 'pending' | 'active' | 'sleeping';
  sleepReason?: string;                    // 休眠原因（显示用）
  payload: unknown;                        // 各 kind 存不同内容
  label: string;                           // 显示名
  createdAt: number;
}
```

### 5.2 各 kind 的 payload

```typescript
/** 挖矿 */
interface MiningIntentPayload {
  tileX: number;
  tileY: number;
  tileTargetRef: string;       // "tile:x:y"
  tileType: string;            // 矿脉类型
  instanceId: string;
}

/** 采集 */
interface GatherIntentPayload {
  sourceId: string;            // 容器 ID
  containerName: string;
  tileX: number;
  tileY: number;
  instanceId: string;
}

/** 炼丹 */
interface AlchemyIntentPayload {
  recipeId: string;
  recipeName: string;
  batchCount: number;
  presetIndex?: number;
}

/** 锻造 */
interface ForgingIntentPayload {
  recipeId: string;
  recipeName: string;
  batchCount: number;
}

/** 强化 */
interface EnhancementIntentPayload {
  targetSlot: number;
  materialSlots: number[];
  targetName: string;
}

/** 建造 */
interface BuildingIntentPayload {
  buildingId: string;
  buildingName: string;
  tileX: number;
  tileY: number;
  instanceId: string;
}
```

### 5.3 Strategy 接口方法语义

| 方法 | 调用时机 | 职责 |
|------|----------|------|
| `execute` | pending→active 或 sleeping→active | 启动实际工作。mining: 设 combatTargetId；gather: 调 startGather；alchemy: 调 pipeline.start |
| `queryProgress` | 客户端请求或 tick 投影 | 返回进度。mining: 地块 hp/maxHp；gather: remainingTicks/totalTicks；alchemy: job.remainingTicks/totalTicks |
| `isCompleted` | 每次 tick 或恢复前 | 判断目标是否已消失/耗尽。mining: 地块 destroyed；gather: 容器耗尽；alchemy: job 完成且无后续批次 |
| `canResume` | sleeping 项尝试恢复时 | 检查恢复条件。mining: 矿脉未摧毁+视野内+无敌人；gather: 容器存在+1格内；alchemy: 材料充足 |
| `onSleep` | active→sleeping | 清理执行态。mining: 不清 combatTargetId（由反击接管）；gather: 清 gatherJob；alchemy: 暂停 job |
| `onTerminate` | 主动终止或 isCompleted=true | 彻底清理。mining: clearCombatTarget；gather: clearGatherJob+释放容器；alchemy: cancel+退还 |
| `conflictsWith` | push 新意图时 | 判断是否需要终止已有意图 |

### 5.4 IntentProgress 返回结构

```typescript
interface IntentProgress {
  /** 0~1 归一化进度，null 表示不适用 */
  ratio: number | null;
  /** 当前值（如 hp、remainingTicks） */
  current?: number;
  /** 最大值（如 maxHp、totalTicks） */
  max?: number;
  /** 显示用文本 */
  label?: string;
}
```

## 6. 驱动时机

```
每 tick 执行顺序：

1. autoCombat 阶段（WorldRuntimeAutoCombatService.materializeAutoCombatCommands）
   └─ 自动战斗无合法目标时：
      └─ 检查 field slot 中 mining intent（sleeping/pending）
         └─ isCompleted? → remove
         └─ canResume? → execute（设回 combatTargetId）

2. craftTick 阶段（WorldRuntimeCraftTickService.advanceCraftJobs）
   └─ 无活跃 craft job 时：
      └─ 检查 craft slot 中 pending/sleeping intent
         └─ isCompleted? → remove
         └─ canResume? → execute（调 pipeline.start）
   └─ 无活跃 gatherJob 时：
      └─ 检查 field slot 中 gather/building intent（sleeping）
         └─ isCompleted? → remove
         └─ canResume? → execute（调 startGather/startBuilding）
```

## 7. 事件 → 意图动作对照表

### 7.1 启动事件

| 事件 | 动作 |
|------|------|
| 玩家锁定矿脉地块（engageBattle） | push mining intent, state=active |
| 玩家发起采集（startGather） | push gather intent, state=active |
| 玩家开始炼丹（startAlchemy） | push alchemy intent, state=active |
| 玩家开始锻造（startForging） | push forging intent, state=active |
| 玩家开始强化（startEnhancement） | push enhancement intent, state=active |
| 玩家开始建造（startBuilding） | push building intent, state=active |

### 7.2 被动打断（sleep，保留可恢复）

| 事件 | 动作 |
|------|------|
| 被怪物/玩家命中，自动反击接管 | field slot active 项 → sleeping，原因"被攻击" |
| 自动战斗临时锁敌（retaliatePlayerTarget） | mining intent → sleeping，原因"自动战斗中" |
| 采集时被攻击 | gather intent → sleeping，原因"被攻击" |

### 7.3 恢复事件

| 事件 | 动作 |
|------|------|
| 反击/自动战斗无合法敌人 | field slot sleeping mining → canResume → execute |
| 玩家回到采集点 1 格内 | field slot sleeping gather → canResume → execute |
| craft job 完成，队列中有下一条 | craft slot pending → execute |

### 7.4 明确终止（remove）

| 事件 | 动作 |
|------|------|
| 矿脉地块被挖碎（destroyed） | remove mining intent + clearCombatTarget |
| 采集容器耗尽/被摧毁 | remove gather intent + clearGatherJob |
| 玩家主动移动 | remove 所有 field slot intent |
| 玩家主动取消挖矿/采集/炼丹等 | remove 对应 intent |
| 玩家主动清除战斗目标 | remove mining intent |
| 玩家关闭自动战斗 | remove mining intent |
| 玩家被击杀 | remove 所有 intent |
| 玩家复活/遁返/强制回城 | remove 所有 intent |
| 换图/断线/session 重建 | remove 所有 intent |

### 7.5 被动 vs 主动区分原则

- **被动** = 玩家没有做出新意图，外部事件强制打断 → sleep
- **主动** = 玩家做出了"不想继续做这件事"的操作 → remove

## 8. 与现有系统的关系

### 8.1 替代关系

| 现有 | 新系统 |
|------|--------|
| `TechniqueActivityQueueService` | 被 `ActionIntentService` 替代 |
| `player.techniqueActivityQueue` | 改为 `player.actionIntents: ActionIntent[]` |
| `sleepToQueue` 调用 | 改为 `ActionIntentService.sleep(intentId, reason)` |
| `tickQueue` 调用 | 改为 `ActionIntentService.tick(player, slot, ctx)` |

### 8.2 保留关系

| 现有 | 关系 |
|------|------|
| `TechniqueActivityPipelineService` | 保留，作为 job 执行引擎 |
| 各 Strategy（AlchemyStrategy 等） | 保留，被 ActionIntentStrategy.execute 内部调用 |
| `player.alchemyJob / gatherJob / buildingJob` | 保留，intent 只是意图记录，job 是执行态 |
| `combatTargetId / autoBattle` | 保留，mining intent 的 execute 设置它们 |

### 8.3 层次关系

```
ActionIntentService（意图调度：谁在做什么、能否恢复）
  │
  ├─ MiningIntentStrategy.execute()
  │    └─ setCombatTarget + autoBattle（走自动战斗循环）
  │
  ├─ GatherIntentStrategy.execute()
  │    └─ WorldRuntimeLootContainerService.dispatchStartGather()
  │
  └─ AlchemyIntentStrategy.execute()
       └─ TechniqueActivityPipelineService.start()（job 执行引擎）
```

## 10. 边界风险

| 风险 | 处理 |
|------|------|
| 跨图恢复 | 换图时 clear all，不存在跨图恢复 |
| 顶号/session 替换 | session 重建时 clear all |
| 离线时矿脉/容器消失 | 不持久化，登录后无残留 |
| 同 slot 多条 active | push 时检查，同 slot 已有 active 则先 sleep/terminate 旧项 |
| 挖矿和采集同时 | 同属 field slot，互斥，push 新的会终止旧的 |
| 反复 sleeping→active 抖动 | canResume 中加入"无敌人"条件，避免刚恢复又被打断 |
| 炼丹材料不足时排队 | canResume 返回 false，保持 sleeping，不反复尝试 |
| 队列最大长度 | 沿用 TECHNIQUE_ACTIVITY_QUEUE_MAX_LENGTH=20，超出拒绝 push |

## 11. 不做的事

- 不持久化意图队列（断线清空）
- 不在 tick 热路径做数据库 IO
- 不改变现有 job 执行引擎（pipeline）的内部逻辑
- 不改变现有协议事件名（C2S/S2C 保持兼容）
- 不扩展挖矿到非矿脉地块
- 不为意图系统引入新的外部依赖
