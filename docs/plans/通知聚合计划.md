# 通知聚合计划

> 解决问题：运行时事件总线中 `playerNotice` 在战斗高密度场景下按单条通知入队，容易触发 `Dropped N events ... top sources: playerNotice` 告警；部分战斗通知仍走旧文本链路，导致结构化通知无法按类型聚合和稳定渲染。

## 背景

- 当前 `RuntimeEventBusService.queuePlayerNotice()` 对玩家通知采用追加队列，单玩家单 tick 上限为 32。
- 结构化通知已经支持 `key / vars / pills / badges`，客户端可通过语言包渲染中文文本和胶囊样式。
- 日志中出现的 `playerNotice:combat:notice.combat.killed` 表示击杀通知在过载时被裁剪。
- 日志中出现的 `playerNotice:combat:unstructured` 表示仍有旧格式战斗通知进入队列。

## 目标

- 降低单玩家单 tick 的通知条数，优先减少高频战斗和掉落通知对队列上限的占用。
- 保持服务端只发送结构化数据，不在服务端拼接玩家可见中文长句。
- 保持现有 `worldDelta.eventBus.notices[]` 协议形状，避免一次性扩大前后端联动范围。
- 保留 `warn / system / success` 等关键通知的独立性，避免重要操作结果被错误合并。
- 让客户端优先渲染 `structured` 通知，旧文本解析仅作为 fallback。

## 非目标

- 不改变玩家资产、战斗结算、掉落归属、任务进度等权威语义。
- 不把所有通知粗暴合并成一条文本。
- 不新增数据库真源或持久化表；通知聚合仍是 tick 内临时表现层逻辑。
- 不把完整通知历史改造成可回放消息系统。

## 设计原则

- 聚合发生在服务端运行时事件总线入队阶段或 tick drain 前，聚合结果仍是普通 `NoticeQueueEntry`。
- 只聚合同一玩家、同一 tick、同一语义类型的结构化通知。
- 聚合 key 必须稳定，避免用中文文本作为判断依据。
- 聚合后的通知继续使用语言包 key，例如 `notice.combat.killed-batch`。
- 需要数量、目标列表、物品列表时，服务端只放入结构化 `vars`，客户端负责模板渲染。
- 高价值通知优先保留；低价值高频表现尽量走 `combatEffect` 或聚合摘要。

## 风险与约束

- 聚合不能改变权威结果，只能改变表现层通知形态。
- 聚合后的 `vars` 不能无限增长，必须设置列表长度上限。
- 服务端 fallback `text` 只能作为兼容旧客户端和日志的兜底，不能作为客户端结构化渲染真源。
- 若未来 protobuf/envelope 对通知字段做强约束，需要同步更新共享类型、编码和协议审计。
