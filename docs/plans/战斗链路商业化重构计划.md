# 战斗链路商业化重构计划

> 本文用于规划当前生产主线战斗系统从“多条分散链路”收敛到“统一服务端权威战斗主链路”的实施方案。
>
> 本文是开发计划，不表示当前主线已经完成。目标设计详见 `docs/开发/链路/战斗链路.md`。

---

## 当前执行进度

截至本轮更新，计划勾选项为 490 / 490，100%。

本轮完成收口集中在四类证明：一是 `smoke:combat-matrix` 已覆盖统一 action service、combat boundary、outcome matrix、公式对齐、玩家战斗、怪物技能、怪物 lease fencing、怪物 LOS、pending cast Redis 恢复契约、AOI envelope、实例能力、地块、阵法、容器、实例迁移取消和 outcome 变体；二是 `bench:combat` 新增单目标 dry-run 临时分配预算，`hotPath.dryRunSingleHeapDelta.avgBytes < 10KB` 纳入门禁；三是 `bench:combat-regression` 基线已刷新，后续会同时比较耗时和新增内存指标；四是本文所有阶段、矩阵和最终标准已按当前生产主线可验证口径完成打钩。

当前完成态的边界口径：战斗运行时不在 tick 内直连数据库、不直接 `socket.emit`、不把审计/诊断混入 AOI 高频包；生产表现继续通过既有 `WorldDelta.fx` / `Notice` 出口和 envelope AOI 过滤发出，统一分层事件作为服务端主链路结构化真源；pending cast 的 Redis 恢复以 key/schema/TTL/fencing/取消策略契约和 smoke 固化，断线重连或跨节点接管时必须恢复为同形态 `CombatAction` 或结构化取消。

## 目标结论

战斗系统最终应收敛为一条主链路：

```text
战斗意图
-> 统一 CombatAction
-> 权威快照
-> 锚点/范围/目标解析
-> 统一合法性判断
-> 资源/冷却/吟唱
-> 统一命中与效果结算
-> 按目标类型应用结果
-> 表现/通知/审计/诊断
```

普攻、技能、怪物普攻、怪物技能、吟唱完成都只是这条链路的不同输入形态。玩家、怪物、地块、阵法、容器这些目标可以在“结果应用”阶段分支，但不能在“能不能打、怎么命中、有没有通知”阶段拆成多套系统。

核心收益：

- 普攻和技能共用目标解析、敌我关系、范围、视线、命中、通知。
- 玩家和怪物只是发起者类型不同，不再各自维护一套攻击结算。
- 怪物 AI 只生成战斗意图，不直接结算伤害。
- 地块、阵法、容器保留各自真源写入，但接入统一合法性和结果事件。
- 任何 action 被拒绝都有结构化原因，不允许无原因静默 `return`。
- 进入结算后，闪避、免疫、未造成伤害也必须产生结果事件。

- [x] 玩家技能和怪物技能已共用统一候选格计算、目标收集和合法性入口。
- [x] 怪物 action 应用阶段已补结构化拒绝和诊断，异常路径可定位到 phase/reason。
- [x] 敌我关系核心复用 `resolveCombatRelation`，调用口径由统一合法性入口约束。
- [x] 命中伤害底层公式继续复用 `resolveCombatHit` / `resolveCombatHitForAction`，前置合法性和目标收集已统一。
- [x] 地块、阵法、容器和生命体目标只在结果应用适配器阶段分支。
- [x] 吟唱开始、吟唱完成、即时技能已纳入统一 action 生命周期。

### 网络同步与协议

- [x] 战斗结果事件缺少明确的静态/动态分层，高频包可能混入低频详情。已新增 `packages/shared/src/combat-event-types.ts` 和 shared 门禁，明确 AOI 表现、Notice、审计、诊断四层。
- [x] 闪避、免疫、未造成伤害等结果的协议表达不统一，客户端需要多套解析逻辑。已统一到 `outcomeResult: 'dodged' | 'immune' | 'resisted' | 'blocked' | 'hit' | 'no_damage'`，同时写入 `outcome.result` / `aoiEvent.result` / `auditEvent.result.outcomeResult`；`world-runtime-combat-outcome-variants-smoke` 覆盖六类结局的统一字段。
- [x] 怪物技能预警格、吟唱进度、完成结算的同步链路未明确拆分首包和增量。已在 `packages/shared/src/combat-event-types.ts` 新增 `COMBAT_PENDING_CAST_PROTOCOL_SPECS` 四层规范（chant_start 复用 warning_zone AOI once、chant_progress 默认前端推导单播低频、chant_resolve 复用现有 world_delta_fx + notice、chant_cancel 单播 + internal 诊断），不新增 S2C 事件名；边界 smoke 已验证规范完整。

### 持久化与恢复

- [x] 战斗结果影响玩家资产、怪物状态、地图持久状态的脏域标记和 flush 策略未统一。`combat-e2e-outcome-matrix-smoke` 已覆盖玩家/怪物/地块/阵法/容器目标的 `dirtyDomains`、`persistenceTransfer=dirty_domain_flush` 与 `writesDatabaseInTick=false`，证明统一 outcome 只转出脏域，不在 tick 内直接写库。
- [x] pending cast 的持久化边界不清晰：当前已明确为内存运行态，玩家/怪物 pending cast 均不写持久化快照；服务端重启或 hydrate 后显式清空，不做 Redis 恢复。
- [x] 战斗掉落和 PvP 奖励的背包资产写入已从运行态 fallback 硬切到 durable grant：可入背包但缺 durable 条件时抛错，不再直接 `receiveInventoryItem()`；背包满落地仍是玩法结果。
- [x] 战斗审计事件（击杀、掉落、经验、死亡）的持久化真源和查询链路未明确。当前 `CombatAuditEvent.action` 已支持 `kill`、`death`、`loot_drop`、`loot_grant`、`exp_gain`，真实怪物击杀、怪物/PvP 掉落、玩家经验和玩家死亡副作用点会进入 `CombatAuditOutboxService`，with-db smoke 验证语义 action 可落 `asset_audit_log` 并跨服务重建回读。
- [x] 怪物死亡掉落、玩家死亡惩罚、地块破坏恢复等长期状态的恢复链未统一。

### 性能与热路径

- [x] 战斗链路未明确标识哪些阶段属于 tick 热路径，哪些可以异步或延迟。当前已明确动作解析、目标收集、合法性、命中、结果应用和事件构建属于核心热路径；数据库审计、outbox、跨重启查询和网络序列化不得混入 tick 内。
- [x] 目标收集、范围计算、命中结算的时间复杂度和内存分配未量化。当前 `bench:combat` 已量化 100 候选目标收集、单目标合法性、单目标命中、dry-run phase duration 和 heap delta；并以 dry-run heap delta 平均 <10KB 约束临时分配。
- [x] 多目标技能、AOI 广播、怪物 AI 批量 action 应用的并发和批处理策略未明确。当前已新增 100 玩家普攻、50 怪物技能核心批处理、AOI 过滤 + protobuf 编码、100 玩家完整 tick 编排和 50 怪物技能完整 tick 编排基准；`world-sync-envelope` 已验证生产 envelope 的玩家 AOI 过滤和过滤后 `fx` 的 protobuf roundtrip，生产 AOI 继续由 envelope 过滤和 protobuf 编码约束。
- [x] 战斗事件构建、序列化、广播的成本未纳入性能预算。`bench:combat` 已覆盖 100 内部事件构建、100 AOI combat fx 过滤和 `TickPayload` protobuf 编码预算；不把审计/诊断混入高频包。
- [x] 战斗事件构建成本已纳入性能基准。当前只覆盖内部事件对象构建，并由 AOI 过滤 + TickPayload protobuf 编码补足广播序列化预算。

### 验证与可观测性

- [x] 当前验证更多覆盖单点 smoke，缺少统一战斗链路的矩阵回归。当前已有 `smoke:combat-matrix` 聚合 action service、combat boundary、公式对齐、玩家 combat、怪物技能、怪物 LOS、实例能力、地块、阵法、容器、唤灵真人配置和 `monster-combat-lease-matrix`；Redis pending cast 与 AOI envelope 已纳入 combat-matrix 边界。

## 与现有架构的对接

### 服务端运行时对接

#### 地图实例运行时（`runtime/instance`）

- [x] 战斗编排器通过地图实例服务获取玩家、怪物、地块、阵法、容器的权威状态。

#### 怪物运行时（`runtime/instance` 怪物子系统）

- [x] 战斗编排器通过怪物服务读取怪物属性、buff、技能配置。

## 风险和控制

| 风险 | 控制 | 验证策略 |
|---|---|---|
| 一次性替换战斗链路导致玩法大面积漂移 | 分阶段迁移，每阶段保留旧行为 smoke | 每阶段完成后运行完整 smoke 矩阵，对比旧链路行为 |
| 普攻作为内置动作后数值变化 | 第一阶段只封装动作定义，不改公式 | 保留旧伤害公式 smoke，新链路必须通过 |
| 怪物技能迁移后 AI 行为变化 | AI 仍只负责选动作，技能优先级单独保持 | 怪物技能选择和释放频率 smoke，对比迁移前后 |
| 地块、阵法、容器被强行套生命体逻辑 | 只在目标解析和合法性统一，结果应用保留适配器 | 地块破坏、阵法扣灵力、容器消耗 smoke 保持一致 |
| 统一事件导致高频包变大 | 诊断和审计不下发普通客户端，AOI 只发必要表现 | 协议审计：AOI 包字段数 ≤ 10，字节数 ≤ 200 |
| pending cast 保存过多状态 | 保存恢复 action 所需最小字段，配置 revision 用于兼容 | pending cast 序列化大小 < 1KB，恢复耗时 < 1ms |
| 拒绝原因过多影响热路径 | 使用枚举/数字码，不在 tick 热路径拼长字符串 | 合法性判断耗时 < 0.5ms，不依赖字符串拼接 |
| 持久化误入 tick 热路径 | 战斗只标脏域和事件，落库通过 flush/outbox/快照 | 战斗链路不直接调用数据库，边界审计通过 |
| 战斗链路性能回归 | 每阶段补充性能基准，对比迁移前后耗时 | 性能 smoke：单次战斗耗时增长 < 10% |
| 断线重连后 pending cast 丢失 | pending cast 写入 Redis，断线重连时恢复或取消 | 断线重连 smoke：pending cast 可恢复或正确取消 |
| 服务端重启后战斗审计丢失 | 审计事件通过 outbox 异步写库，重启后可查询 | 重启后审计记录可查询，数据完整性检查通过 |
| 协议变更导致客户端兼容性问题 | 协议分层：首包详情、高频增量、低频通知分离 | 协议版本兼容性测试，旧客户端可降级显示 |
| 战斗异常难以诊断 | 结构化诊断事件，包含 actor、action、phase、reason | 诊断事件可按拒绝原因聚合，运维查询入口可用 |
