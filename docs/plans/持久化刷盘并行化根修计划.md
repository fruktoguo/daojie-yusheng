# 持久化刷盘并行化根修计划

更新时间：2026-05-20

## 背景

线上日志出现持续慢刷盘退避：

```text
PlayerPersistenceFlushService 玩家最终一致刷盘触发降级退避：durationMs=461~776 thresholdMs=120 backoffMs=5000
MapPersistenceFlushService 地图最终一致刷盘触发降级退避：durationMs=316~348 thresholdMs=100 backoffMs=5000 dirtyInstanceCount=22 domainCounts=fengshui:2~3,tile_damage:17,tile_resource:6~7
```

这不是单纯“线程不够”。当前代码里的并行化没有落到真实瓶颈层：

- `PersistenceWorkerPool` 只做玩家 snapshot 预序列化，返回值没有接入真正写库路径。
- `MapPersistenceFlushService` 的并行只是 `Promise.allSettled` 并发打 PG，不是 worker_threads，也不是批量 SQL。
- `player:state-worker`、`instance:*worker` 等脚本没有进入生产 `docker-stack.tencent.yml` 编排。
- 这些 worker 脚本即使单独启动，也会 `NestFactory.createApplicationContext(AppModule)` 拉起新的 runtime，拿不到主进程内存 dirty 态。
- 地图单轮 22 个实例会拆成大量小事务，并且每个实例成功后还会额外写一次 recovery watermark。
- `tile_damage`、`tile_resource`、`fengshui` 本身来自运行时持续变化，不能靠简单降频掩盖。

本计划目标是从根上修复刷盘并行化模型，而不是止血关闭开关。

## 当前事实

### 玩家刷盘

- 入口：`packages/server/src/persistence/player-persistence-flush.service.ts`
- 周期：默认 `1500ms`
- 并发：`PLAYER_PERSISTENCE_FLUSH_PARALLELISM = 4`
- 慢阈值：默认 `120ms`
- 当前 worker 逻辑：
  - `flushPlayerDirtyDomains()` 已移除无效的 `persistenceWorkerPool.submit('persistence-build')`。
  - 真实写库仍走 `PlayerDomainPersistenceService.savePlayerSnapshotProjectionDomains()`。
  - `savePlayerSnapshotProjectionDomainsWithClient()` 仍在主线程内做 domain 归一化、row 构造、`JSON.stringify`、事务和 SQL。
  - `PersistenceWorkerPool` 仍保留给后续真正接入写计划/其它 worker 场景使用。

### 地图刷盘

- 入口：`packages/server/src/persistence/map-persistence-flush.service.ts`
- 周期：默认 `1500ms`
- 并发：`MAP_PERSISTENCE_FLUSH_PARALLELISM = 3`
- 慢阈值：默认 `100ms`
- 当前写库链：
  - `MapPersistenceFlushService.runFlushCycle()`
  - `worldRuntimeService.buildDomainDeltaBatch(domain, instanceIds)` / `markDomainBatchPersisted(domain, instanceIds)`
  - `WorldRuntimePersistenceStateService.listDirtyPersistentInstanceDomains()` 提供 domain 元数据
  - `InstanceDomainPersistenceService.saveTileDamageDeltaBatch()` / `saveTileResourceDeltaBatch()` / `saveInstanceRecoveryWatermarkBatch()` 处理批量写入
  - 其余 domain 仍保留 `saveBuildingRoomFengShuiState()` 等实例级写入
  - `saveInstanceRecoveryWatermark()` / batch watermark 合并提交
- 真实瓶颈主要在 PG 事务、advisory lock、`jsonb_to_recordset`、`ON CONFLICT`、连接池排队和小事务数量。

### 生产编排

- `docker-stack.tencent.yml` 只部署：
  - `client`
  - `server`
  - `backup-worker`
  - `postgres`
  - `redis`
- 未部署：
  - `player:state-worker`
  - `player:flush-worker`
  - `instance:flush-worker`
  - `instance:tile-damage-worker`
  - `instance:ground-worker`
  - `instance:container-worker`
  - `instance:overlay-worker`
  - `instance:monster-runtime-worker`
- 所以生产实际靠主进程 interval flush 落库。

### 连接池

- `DatabasePoolProvider` 当前全进程共享单一 PG pool。
- 默认 `SERVER_DATABASE_POOL_MAX = 24`，上限代码限制为 `50`。
- 玩家请求、GM 查询、outbox、durable operation、player flush、map flush、ledger 查询共用一池。
- 增加异步并发会更快把任务堆到同一个 pool 前，不等于吞吐提升。

## 根因判断

### R1. “多线程”没有替换真实写库构造

玩家 `PersistenceWorkerPool` 目前只是额外工作：

- 多一次 snapshot 结构化克隆。
- 多一次 worker `JSON.stringify`。
- 多一次等待 worker response。
- 后续主线程仍重新构造 SQL 参数并写库。

根因不是 worker 数量少，而是 worker 输出没有成为写库输入。

### R2. 地图并发是 DB 并发，不是计算卸载

地图刷盘的 `parallelism=3` 同时跑多个实例事务。它减少不了每个实例的 SQL 数量，只会提高同一时间对 PG 的压力。

在 PG 是瓶颈时，这会表现为：

- 单轮耗时上升。
- pool 等待增加。
- advisory lock 等待增加。
- outbox / GM / 玩家操作一起排队。

### R3. 生产 worker 拓扑不成立

当前 worker 类和脚本证明“代码路径可运行”，但不证明生产在用。

更关键的是，这些脚本启动的是新 AppModule，不共享主进程 runtime dirty map。它们适合处理数据库 ledger 中已经存在的任务，不适合直接读取主进程内存 dirty 态。

如果要做独立进程 worker，必须先把 dirty 事件转成可跨进程消费的持久队列或 Redis Stream，而不是依赖内存读取。

### R4. 地图 flush 写放大过高

以日志中的单轮 `dirtyInstanceCount=22` 为例：

- `tile_damage:17` 可能对应 17 个实例事务。
- `tile_resource:7` 可能对应 7 个实例事务。
- `fengshui:3` 可能对应 3 个建筑/房间/风水事务。
- 每个 persisted instance 还会再写一次 `instance_recovery_watermark`。

实际一轮可能接近 40-50 个事务。并发只是在同一时间打出去，不减少事务总数。

### R5. 高频 dirty 没有合并窗口

`tile_damage` 在破坏和恢复 tick 中持续标 dirty。

`tile_resource` 在资源变动时标 dirty，并触发风水相关重算。

这些 dirty 是自然运行时事件，不是偶发异常。根修必须做 domain 级合并和批量提交。

## 目标

- [ ] 让“并行化”落到真实瓶颈层：减少主线程 CPU、减少 SQL 往返、减少事务数量、减少 pool 排队。
- [ ] 统一刷盘调度模型：主进程 interval 与独立 worker 不再半套并存。
- [ ] 玩家和地图刷盘都支持 domain 批量写入，而不是逐玩家/逐实例小事务。
- [ ] 恢复水位和 domain 写入合并提交，减少额外事务。
- [ ] 高风险玩家资产域保持强一致/可恢复，不因批量化牺牲正确性。
- [ ] GM worker 页能反映真实生产 worker/flush 状态，而不是只读 ledger 静态摘要。
- [ ] 新增 with-db / bench / smoke 证明，能回答吞吐、延迟、正确性和清理链。

## 非目标

- [ ] 不通过单纯调大慢阈值隐藏 WARN。
- [ ] 不通过只加 worker 数量解决。
- [ ] 不把玩家资产、地图状态或交易正确性放到客户端。
- [ ] 不把所有持久化改成即时同步 DB 写。
- [ ] 不在 tick 热路径直接等待数据库。
- [ ] 不一次性重写所有持久化表结构。

## 总体方案

选择一条明确主线：**主进程负责收集权威 dirty，写入可跨进程消费的轻量 flush task；flush 执行层按 domain 批量 claim、批量构造、批量提交。**

短期允许主进程内消费 task，但 task 形态必须和未来独立 worker 共用，避免继续维护两套模型。

核心边界：

| 层 | 职责 |
|----|------|
| runtime | 只标 dirty 和维护最小脏键，不直接写库 |
| flush scheduler | 把 dirty 合并成 domain task，控制合并窗口和优先级 |
| build phase | 把 snapshot/delta 编译成 write plan |
| persistence writer | 批量事务写库，合并 recovery watermark |
| ledger/queue | 跨进程 claim、retry、backlog、dead-letter |
| GM/diagnostics | 展示真实 backlog、吞吐、pool、锁等待和慢 SQL |

## 风险与约束

- 批量写入必须保持玩家资产和地图状态的恢复语义，不能为了吞吐牺牲正确性。
- 地图 instance advisory lock 必须按稳定顺序获取，避免 batch 内死锁。
- 玩家 domain 空覆盖 guard 必须继续区分 runtime dirty flush 和 ordinary snapshot/import。
- 独立 worker 不能直接读主进程内存，必须通过持久 task、Redis Stream 或其他跨进程队列。
- worker 不应启动完整 AppModule，否则会重复启动 tick/outbox/flush timer 并抢连接池。
- shutdown 强刷必须绕过合并窗口。

## 推荐执行顺序

1. Phase 0：补指标，先证明慢在哪里。
2. Phase 1：修掉玩家 `PersistenceWorkerPool` 的无效工作。
3. Phase 2：做地图 `tile_damage/tile_resource` 批量写。
4. Phase 3：给自然 dirty 加合并窗口。
5. Phase 5：拆 PG pool，给 flush 限流。
6. Phase 4 和 Phase 6：统一 worker/task 模型并决定生产拓扑。

原因：先减少已经确认的写放大和无效工作，再决定是否引入独立 worker。否则会把当前错误模型部署成更复杂的错误模型。
