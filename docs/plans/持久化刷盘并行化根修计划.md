# 持久化刷盘并行化根修计划

> 解决问题：线上持续慢刷盘退避（玩家 461-776ms，地图 316-348ms），根因是并行化没有落到真实瓶颈层——写放大过高、无合并窗口、worker 输出未接入写库。

## 目标

- 减少 SQL 往返和事务数量（domain 批量写入）
- 高频自然 dirty 有合并窗口
- PG pool 分级避免 flush 抢占玩家请求
- 统一 flush task 模型（主进程 interval 与独立 worker 不再半套并存）

## 约束

- 批量写入必须保持玩家资产和地图状态的恢复语义
- 地图 instance advisory lock 必须按稳定顺序获取
- 玩家 domain 空覆盖 guard 必须继续区分 runtime dirty flush 和 ordinary snapshot/import
- shutdown 强刷必须绕过合并窗口

## 分阶段进度

### Phase 0：证据补齐 ✅ 已完成
- GM 数据库/worker 页展示 flush 诊断指标（CPU build/pool waiting/SQL/lock）

### Phase 1：修正玩家 PersistenceWorkerPool ✅ 已完成
- 删除无效 `persistenceWorkerPool.submit()`
- 保留同步分域写库路径与 lease/空覆盖守卫语义

### Phase 2：地图 domain 批量写入 ✅ 已完成
- `saveTileDamageDeltaBatch` / `saveTileResourceDeltaBatch` / `saveInstanceRecoveryWatermarkBatch`
- 事务数量从 ~40-50 降到按 domain 个位数级别
- 待验证：`smoke:instance-tile-damage-flush-worker` / `verify:release:with-db`

### Phase 3：高频 dirty 合并窗口 ✅ 已完成
- tile_damage/tile_resource 默认 3-5 秒合并窗口
- 区分 durable priority（玩家主动高优先，自然恢复低优先）

### Phase 5：PG pool 分级和限流 ✅ 已完成（代码层）

### Phase 6：生产编排落地 🔲 待执行
