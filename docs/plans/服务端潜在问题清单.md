## 一、协议 / 网络层

### S1. [P1] WorldGateway.attachRateLimitGuard 全局硬编码 60 包/秒，所有事件共享同一桶

- 文件：`network/world.gateway.ts:188-196`
- 现象：`client.use((packet, next) => { if (!checkRateLimit(client, event, 60, 1000)) return next(new Error('RATE_LIMIT_EXCEEDED')) })`，限流参数 `60/1s` 硬编码且 **所有事件共享同一桶（按 event 名计数）**，没有按事件类别差异化（chat 文本 vs movement 意图 vs panel 拉取）。
- 影响：
  - 攻击者只要让单条包体足够大（合法上限范围内），1 秒钟内能做 60 次大包消耗仍合法，限流形同虚设。
  - 高频玩家操作（一息内多次更新寻路目标 / 频繁打开面板 / 战斗中刷新背包）极易在 1 秒桶里命中 60 上限被全局拒绝，把正常玩法打挂。
  - 60 packets/sec 不是从协议预算反推的，而是经验值；缺少按事件名映射限速表。
- 建议：
  - 在 `network/world-gateway-guard.helper.ts` 里建立按事件名分级限速表（高频 movement/intent 一档、面板查询一档、敏感操作如 craft/market 单独一档）。
  - 大包体单独按 byte/sec 桶限流，与 packet/sec 互独立。
  - 命中限速的事件必须区分"客户端 bug"（kick）vs "竞速暴力"（直接断开）。

### S2. [P1] WorldGateway 单文件聚合 19 个 helper，构造器参数列表无类型注解

- 文件：`network/world.gateway.ts:78-145`
- 现象：单个 gateway 类的构造函数接受 19 个 service / helper，构造体内手动 `new WorldGatewayXxxHelper(this)`，参数列表大量 `: any`、字段批量注释 `/** worldGmSocketService：世界Gm Socket 服务引用。*/`，看起来是从 JS 迁回 TS 时机器化生成的副产物。
- 影响：
  - 这种 facade 是 AGENTS.md §15 明确禁止的"巨型重构 facade"反例。任何 helper 之间的隐性依赖都通过 `this` 传递，无法做单元测试隔离。
  - `marketRuntimeService: any` / `worldRuntimeService: any` 绕过类型检查，未来协议字段变更不会在编译期发现。
- 建议：将 19 个 helper 拆成独立 NestJS 子模块；helper 之间的协作走 service 注入而不是共享 gateway `this`。

### S3. [P1] WorldSessionService 缺 onModuleDestroy，关停时 detached binding 与定时器丢失

- 文件：`network/world-session.service.ts:80-108, 268-288`
- 现象：
  - 类未实现 `OnModuleDestroy`，`expiryTimerByPlayerId` / `expiredBindings` 在容器收到 SIGTERM 后没有强制 flush。
  - `setTimeout(unref())` 配合 `unref` 让计时器不阻塞 event loop，所以容器一旦决定关停，**所有未到期的 detached binding 不会触发 expire 回调，也不会进入 expiredBindings**，最终 reaper 来不及消费就被进程结束。
  - 玩家在 detach 窗口（默认 15s）内重启服务器，期间 detach 后变更（战斗结算尾包、邮件/坊市操作产物）就会在 PlayerPersistenceFlushService.flushPlayer 还没被 reaper 触发的窗口里丢失。
- 影响：每次发布更新都可能丢一批"短窗口断线但已结算"玩家的尾部数据，与 §13 持久化真源原则冲突。
- 建议：在 `onModuleDestroy` / `beforeApplicationShutdown` 里强制把 `bindingByPlayerId` 中所有 connected=false 的 binding 立即推入 `expiredBindings`，并触发同步 reaper 一次（带超时上限）。

### S4. [P1] WorldSessionService.registerSocket 与 reaper.consumeExpiredBindings 之间存在 race

- 文件：`network/world-session.service.ts:118-180` × `network/world-session-reaper.service.ts:90-103`
- 现象：
  - register 路径 line 145：`this.expiredBindings.delete(playerId)` 试图回收过期标记。
  - reaper 路径在 1Hz 间隔里先 `consumeExpiredBindings()`（清空 map），随后 `await flushPlayer(binding.playerId)` 是异步的。
  - 玩家在两者之间完成 detach → consumeExpiredBindings 已取走 → reaper 在 await flushPlayer 期间，玩家发起重连 register。register 时 `expiredBindings.delete` 已经无 entry，但 reaper 的 `flushPlayer(binding.playerId)` 仍在跑，下一步会执行 `clearLocalRoute(binding.playerId, routeSessionEpoch)` 与 `clearDetachedPlayerCaches(binding.playerId)`，把玩家**新会话刚完成的 bootstrap 缓存清掉**。
- 影响：高并发断线重连时，新 bootstrap 写入的 detached 缓存可能被旧 reaper 回收清掉，触发"重连后看到空白世界 → 再过几秒同步恢复"。
- 建议：
  - reaper 在 flush 完成后再读一次 `getBinding(playerId)`，如果当前 binding 是 connected 且 sessionId 不同，跳过 `clearDetachedPlayerCaches` / `unloadIdleDetachedRuntime` 分支。
  - 或在 register 时记录"reset epoch"，reaper 写回时比较 epoch。

### S5. [P2] WorldSessionService.sessionDetachExpireMs 在构造期一次性读取，env 修改不重启不生效

- 文件：`network/world-session.service.ts:79`
- 现象：`private readonly sessionDetachExpireMs = resolveSessionDetachExpireMs()` 在 constructor 阶段固定。如果运维通过 GM 接口或动态 env reload 想缩短/延长 detach 窗口，必须重启进程。
- 建议：改为按需读取或暴露 setter（GM 路径），同时落 audit。

### S6. [P1] WorldTickService.beforeApplicationShutdown 等待 tick 完成无超时

- 文件：`runtime/tick/world-tick.service.ts:115-122`
- 现象：`while (this.tickInFlight) await sleep(25)` 没有最大等待时间。如果 `worldRuntimeService.advanceFrame` 因数据库阻塞、worker 卡死或代码 bug 永久挂起，关停永远不会返回，进程被 docker `stop` 默认 10s 后强制 SIGKILL。
- 影响：
  - 不仅当前 tick 状态完全丢，所有还没 flush 的玩家变更（战斗结算、坊市挂单、邮件领取）也跟着 SIGKILL 一起丢，与 §13 写入策略冲突。
  - 灰度发布、热更新场景每次都有概率触发数据丢失。
- 建议：补 `Promise.race([waitTickIdle(), sleep(SHUTDOWN_TICK_BUDGET_MS)])`，超时后强制 log + 落地 metric，让外层运维知道这次发布跨过了 tick boundary。

### S7. [P1] WorldTickService 使用 setInterval 而非递归 setTimeout，慢 tick 会被覆盖跳帧

- 文件：`runtime/tick/world-tick.service.ts:107-110`
- 现象：`setInterval(() => void this.runTickOnce(), WORLD_TICK_INTERVAL_MS)`。`runTickOnce` 因 `tickInFlight` 锁会立即 return，但 setInterval 的下一拍仍按固定 1s 触发。
- 影响：当 advanceFrame 真实执行 1.5s 时，下一拍触发后被锁直接放弃，那次 tick 就被吞掉，实际 tick 频率从 1Hz 退化到 0.66Hz，且**没有指标暴露**这种跳帧。AOI/buff/战斗结算频率被牵连。
- 建议：改为 `setTimeout` 递归调度："上一次执行结束 → 计算下次到点时间 → 设定 setTimeout"，并把"被跳过的 tick 数"暴露给 GM 性能观测。

### S8. [P1] WorldTickService.runTickOnce 异常仅 logger.error，不影响 readiness

- 文件：`runtime/tick/world-tick.service.ts:91-100`
- 现象：catch 分支只 `logger.error`。tick 持续失败时 readiness 不会降级，外部 LB / docker healthcheck 仍判活，故障无法触发自动剔除。
- 建议：维护一个 `recentTickFailureCount`，连续 N 次失败把 `runtime.ready=false` 注入 `HealthReadinessService`，让 /health 端点正确反映降级。

## 二、持久化层

### S16. [P0] [FIXED 2026-05-10] OutboxDispatcherService.markFailed 死信迁移分两步且第一步不在事务，崩溃可丢消息

- 文件：`persistence/outbox-dispatcher.service.ts:117-180`
- 现象：失败处理流程为：
  ```
  1) 单独 pool.query 把 outbox_event 行 status 改成 'dead_letter'（不在事务内）
  2) 单独 pool.query SELECT 出该行
  3) client.connect → BEGIN → insertDeadLetterEventWithClient → DELETE FROM outbox_event WHERE status='dead_letter' → COMMIT
  ```
- 影响：
  - 步骤 1 完成后 / 步骤 3 BEGIN 之前进程崩溃或数据库瞬断：行卡在 `outbox_event` 表 `status='dead_letter'`。
  - `claimReadyEvents` 的 WHERE 限制 `status IN ('ready', 'claimed')`，**dead_letter 行不会被任何 dispatcher 重新认领**；同时它也没进入 `dead_letter_event` 表，运维查询死信也找不到。
  - 该事件永久消失，运营事件、邮件、坊市结算尾包等会出现"凭空蒸发"。
- 建议：
  - 把整个失败迁移流程合并到一个事务：BEGIN → SELECT FOR UPDATE → INSERT INTO dead_letter_event → DELETE FROM outbox_event → COMMIT。事务回滚天然让行保持在 `attempt_count < max_attempts` 状态，可被下次 markFailed 重试。
  - 或在启动期增加"扫描 outbox_event status='dead_letter' 但 dead_letter_event 不存在的孤儿行"恢复任务。

### S17. [P1] OutboxDispatcherService.claimReadyEvents 不按 partition_key 串行化，同一玩家事件可并发投递

- 文件：`persistence/outbox-dispatcher.service.ts:62-95`
- 现象：`SELECT ... FOR UPDATE SKIP LOCKED ORDER BY created_at ASC LIMIT $3`，没有按 `partition_key`（同一玩家/实例聚合 ID）做 advisory lock 或 GROUP BY 顺序保证。多 worker 同时拉取时，同一 `partition_key` 的多条事件会被分到不同 dispatcher 并发消费。
- 影响：商业级 MMO 中"邮件领取 → 邮件提示 → 邮件清空"等同 partition_key 链路如果乱序消费，会出现 UI 状态错乱、市场撮合结果发反、邮件计数错误。
- 建议：
  - 按 partition_key 哈希到固定 worker（一致性哈希）；或在 SELECT 中加 `pg_try_advisory_xact_lock(hashtext(partition_key))` 串行同 partition 事件。
  - 同 partition_key 内事件一次只允许一个 in-flight。

### S18. [P1] OutboxDispatcherRuntimeService 串行 for-of 处理事件，单个慢消费阻塞整批

- 文件：`persistence/outbox-dispatcher-runtime.service.ts:80-105`
- 现象：`for (const event of events) { await this.consumeEvent(event) }`，128 个事件按串行处理；topic 内任何一个事件 consumer 阻塞（如 combat audit 落库慢、market 撮合 stuck）会拖累整批 latency。
- 影响：高峰 outbox 写入速率撞到串行消费上限（约 128/250ms = 512 events/s/dispatcher），与"runtime 1Hz / 5000 玩家"目标不匹配；一旦某 consumer 慢，整个 outbox 投递速率骤降。
- 建议：按 partition_key 桶化并发执行，桶之间无序、桶内有序。

### S19. [P0] [FIXED 2026-05-10] CombatAuditOutboxService 内存队列上限 5000，溢出 / 进程崩溃丢审计

- 文件：`persistence/combat-audit-outbox.service.ts:34-160`
- 现象：
  - `MAX_QUEUE_SIZE = 5000`，超过时 `this.queue.shift()` 丢最早事件。
  - flush 失败的 batch 又被 `concat(this.queue).slice(0, MAX_QUEUE_SIZE)` 重新拼接，仍按 5000 截断丢最旧。
  - 进程 SIGKILL 时整队列丢光（onModuleDestroy 只 best-effort flushOnce，且 catch 后吞错误）。
- 影响：
  - 战斗审计是 §6 / §13 商业级 MMO 必要链路，运营纠纷、外挂回查、掉落争议都依赖这条流水。一旦丢失，运营无法追溯。
  - 5000 上限对全服 5000 玩家 + 多人战斗远远不够，单次大型副本结算就能溢出。
- 建议：
  - 改为先写本地 SQLite/Postgres staging 表（或 outbox），再异步迁移到正式 audit 表，进程崩溃可恢复。
  - 或直接强制同步写 DB（让上游产生背压而不是吞事件）。
  - flush 失败必须有 disk persistent staging，不能只放回内存。

### S20. [P1] PlayerPersistenceFlushService.markPersisted 全 domain 清除，缺少版本/epoch 校验

- 文件：`persistence/player-persistence-flush.service.ts:99-163`
- 现象：`flushPlayer` 流程：
  ```
  buildPersistenceSnapshot → await DB write → playerRuntimeService.markPersisted(playerId)
  ```
  `markPersisted(playerId)` 是按 playerId 全清 dirty 标记，**不区分 domain**，也不带 expected version。
- 影响：在 await DB write 期间，runtime 内可能新 dirty 一个 domain（玩家收到新邮件、产生战斗结算 / 移动）；await 完成后调用 markPersisted 会把新 dirty 一并清掉，下一 cycle 错过新写入。
- 建议：
  - flushPlayer 开头记录 `dirtySnapshot = listDirtyPlayerDomains().get(playerId)`，写库后用 `markPersisted(playerId, dirtySnapshot)` 仅清当时的 domain set。
  - 或在 runtime 侧加 dirtyVersion，markPersisted 时只 ack <= flushedVersion 的标记。

### S21. [P1] PlayerPersistenceFlushService.flushPlayer / runFlushCycle 双入口对同 playerId 无互斥

- 文件：`persistence/player-persistence-flush.service.ts:99-180, 230-300`
- 现象：
  - reaper 路径直接调 `flushPlayer(playerId)`。
  - 周期路径走 `runFlushCycle` → 内部走 batch + concurrency。
  - 两者在同一进程都可能为同一 playerId 触发 buildPersistenceSnapshot + 落库；只有 `flushPromise` 保证 runFlushCycle 之间串行，**没有 per-player 互斥**。
- 影响：
  - 同一玩家两份 snapshot 并发写 DB，最后一次 ack 才生效，但中间 ack 已经清掉 dirty bit。
  - PlayerDomainPersistenceService 内层的 INSERT/UPSERT 顺序乱掉，一些 domain（quest progress、enhancement）可能被旧 snapshot 覆盖新值。
- 建议：维护 `inflightByPlayerId: Map<playerId, Promise>`，per-player 复用同一 inflight。

### S22. [P1] DurableOperationService 直接 new Pool 不限 max/timeout（已在 H2，但还伴随未关闭锁定）

- 文件：`persistence/durable-operation.service.ts:387-391`
- 现象：H2 已记录 14 处 new Pool；这里附加风险：`acquirePlayerAssetLock` 通过 `pg_advisory_xact_lock` 占用连接。如果 connection pool 满（默认 pg max=10），后续锁请求阻塞 pool 队列直至 EAI_TIMEOUT，玩家操作全部超时。
- 影响：这一点放大了 H2 的连接池缺失影响——不仅是池数量，而是**事务中长时间持锁的连接被穷尽时整个 durable operation 子系统全停**。
- 建议：合并 H2 修复时，给 `durable-operation` pool 单独命名空间，max=8，独占连接预算，避免与轻量查询 pool 互相挤兑。

### S23. [P2] schema-bigint-migration / ensureXxxTable 在 onModuleInit 并发 DDL

- 文件：`persistence/schema-bigint-migration.ts`，多个 service 的 onModuleInit
- 现象：`MailPersistenceService` / `PlayerDomainPersistenceService` / `OutboxDispatcherService` / `DurableOperationService` 等 12+ service 启动期都跑 `ensureXxxTable` + `ensureBigintColumnType`。NestJS bootstrap 时这些并发触发 DDL（`CREATE TABLE IF NOT EXISTS`、`ALTER COLUMN`）。
- 影响：
  - PostgreSQL DDL 会持有 `ACCESS EXCLUSIVE LOCK`，多 service 并发跑同一表 DDL 会互相等。极端情况下一个 ALTER 卡住，bootstrap 链全部卡住，导致 Nest 初始化超时不可用。
  - 生产部署的滚动更新场景：旧实例还在用表，新实例跑 ALTER 会被旧实例的查询持续阻塞。
- 建议：
  - 把所有 schema migration 集中到一个 `DatabaseMigrationService`，在 main.ts bootstrap 之前用 advisory lock 串行执行；service onModuleInit 不再做 DDL。
  - 或参考 `verify:release:doctor`，把 schema 迁移作为部署步骤而不是运行期副作用。

### S24. [P2] CombatAuditOutboxService.flushOnce flush 失败仅 logger.warn，无指标 / 无降级

- 文件：`persistence/combat-audit-outbox.service.ts:118-145`
- 现象：连续 flush 失败时只是 warn log，没有把失败次数注入 readiness、没有触发限流、没有切断上游战斗 enqueue。配合 S19 的 5000 上限，DB 长时间不可用会导致整个高频战斗审计静默丢失。
- 建议：补失败计数 metric，连续失败 N 次降级 readiness、对上游 enqueue 返回 false 触发战斗结算补偿（或拒绝）。

### S25. [P1] PlayerPersistenceFlushService 与 reaper 流程对 playerId 字符串 trim/normalize 不一致

- 文件：`persistence/player-persistence-flush.service.ts` × `network/world-session-reaper.service.ts:90-103`
- 现象：reaper 把 binding.playerId 直接传入 `flushPlayer(binding.playerId)`，而 binding.playerId 在 register 阶段从 token / session 获取，未做强制 normalize；DurableOperationService 内部对 playerId 都做 `normalizeRequiredString(playerId)` 去除空白。如果某条路径忘了 trim，会出现"`'abc'` 与 `'abc '` 是不同 player"的 bug。
- 建议：统一在 register 端 normalize 一次，或在所有 ByPlayerId map 操作前都强制 normalize。

## 三、运行时层（runtime）

### S26. [P0] world-runtime-instance-tick-orchestration.advanceFrame 全部实例顺序串行 await，10000 实例无法 1Hz 收敛

- 文件：`runtime/world/world-runtime-instance-tick-orchestration.service.ts:21-150`
- 现象：advanceFrame 内 instance 推进采用 **完全串行** 结构：
  ```typescript
  for (const { instance, steps } of instanceStepPlans) {
    for (let index = 0; index < steps; index += 1) {
      // ... 串行执行单实例单 step ...
      for (const playerId of currentPlayerIds) {
        await deps.worldRuntimePlayerSkillDispatchService.resolvePendingPlayerSkillCast(playerId, deps);
      }
      await deps.worldRuntimeCraftTickService.advanceCraftJobs(currentPlayerIds, deps);
      for (const playerId of currentPlayerIds) { ... }
    }
  }
  ```
- 影响：
  - AGENTS.md §2 的目标是 10000 实例 + 5000 玩家 + 1Hz tick，单核串行处理 10000 实例完全不可能在 1 秒内完成。
  - 即便每实例 0.1ms，10000 个就要 1s 全占用；tick 必然跨 boundary，触发 S7 跳帧，AOI/战斗/buff 频率退化。
  - `await deps.dispatchPendingCommands()` 也是串行链路，玩家意图按 tick 顺序排队但单点处理。
- 建议：
  - 实例 tick 改成按"实例分桶 + Promise.all 并发"结构；同 instance 内仍按顺序，但跨 instance 并发。
  - 可借助 worker_threads 把单实例的纯计算搬出主线程，主线程只做协议组包。
  - 必须有"tick 实际耗时"指标暴露给 GM 性能观测（已有 perf observer 但没和 tick budget 关联）。

### S27. [P1] runtime 巨型类违反 §15，单文件 100KB+ 难以维护

- 文件清单：
  - `runtime/player/player-runtime.service.ts`（300KB，本仓库最大单文件类）
  - `runtime/instance/map-instance.runtime.ts`（324KB）
  - `persistence/player-domain-persistence.service.ts`（194KB）
  - `persistence/instance-domain-persistence.service.ts`（157KB）
  - `runtime/market/market-runtime.service.ts`（150KB）
  - `http/native/native-gm-admin.service.ts`（131KB）
  - `runtime/craft/craft-panel-runtime.service.ts`（125KB）
  - `content/content-template.repository.ts`（98KB）
  - `runtime/world/world-runtime-sect.service.ts`（87KB）
  - `http/native/native-gm-player.service.ts`（80KB）
- 现象：这些类把"规则、查询、持久化、协议装配、运行时状态、冷路径工具"混在同一个 class 里，违反 AGENTS.md §15 第一条红线。
- 影响：

### S28. [P1] PlayerRuntimeService.buildRuntimeOwnerId 用 Math.random 拼接，可在并发下撞 ID

### S9. [P0] [FIXED 2026-05-10] NativePlayerAuthService.login 候选账号顺序串行 verify，可被放大攻击

- 文件：`http/native/native-player-auth.service.ts:267-291`
- 现象：login 同时按 username 和 roleName 查候选 user，把所有匹配组合到 `candidates: Map<id, user>`，然后 **依次** `await verifyPassword(password, user.passwordHash)`：
  ```typescript
  for (const user of candidates.values()) {
    if (await verifyPassword(password, user.passwordHash)) {
      matchedUsers.push(user);
    }
  }
  ```
- 影响：
  - scrypt 配置 N=16384、keyLength=64，每次 ~50–100ms（即便用 libuv 线程池也是单 hash 单 worker）。如果攻击者用一个常见角色名（多人使用过）触发 N 个候选，登录请求会消耗 N × 100ms CPU。
  - bcryptjs fallback 路径（旧 hash）是 **同步阻塞主线程**，会让所有在线玩家的 socket 事件循环卡顿。
  - 失败请求与成功请求耗时差异（matchedUsers.length 不同）会泄漏"用户名是否存在"和"哪个候选密码正确"，可作为侧信道枚举有效角色名。
- 建议：
  - login 阶段最多匹配 1 个候选，不允许通过 roleName 模糊匹配多个账号；roleName 重名应在注册时阻止或改为按账号显式登录。
  - 把 bcrypt 旧 hash 路径改为异步（用 `bcrypt` native 或 `compareBcryptAsync`），并在恒定时间内返回。
  - 在认证完成之前固定 verifyPassword 调用次数（如统一调一次 dummy hash），杜绝时序差异。

### S11. [P1] WorldPlayerTokenCodecService 没有 audience 校验，刷新 token 无 rotation/jti

### S44. [P1] CORS 默认放行无 origin 请求 + Strict-Transport-Security 强制设置

### S52. [P1] smoke-suite autoCleanupSmokeArtifacts 调用前后的失败路径未保证清理

### S55. [P1] main.ts 端口冲突诊断采样最多 12 次 × 100ms = 1.2s 阻塞 bootstrap

- 文件：`main.ts:14-18, 187-205`

  - 进一步在 bootstrap 内存数据被采集到 stderr 之前可能丢失（异常退出）。
- 建议：诊断采样改成异步 fire-and-forget + 超时上限 500ms。

### S56. [P1] main.ts 缺少 graceful shutdown timeout，依赖各 service 自实现

- 文件：`main.ts:153-220`
- 现象：调用 `app.enableShutdownHooks()` 后依靠 NestJS 默认。SIGTERM 后没有 `Promise.race(shutdown, sleep(MAX_GRACEFUL_MS))` 兜底，若任意 service 的 onModuleDestroy 阻塞（比如 S6 的 tick 等待），整体 shutdown 永远不返回。

### S58. [P2] HealthReadinessService 用 PersistenceServiceLike 鸭子类型 `pool?: unknown`

- 文件：`health/health-readiness.service.ts:14-23`
- 现象：用鸭子接口而不是真正的 service interface。这意味着如果某 service 内部把 `pool` 字段重命名（如 H2 修复时改成 `dbPool`），readiness 检测会静默失败（条件分支假定 `pool` 存在）。

### S59. [P1] native-postgres-backup spawn pg_dump 无 timeout 上限保护备份导出

- 文件：`http/native/native-postgres-backup.ts:42-50, 161-202`

- 影响：
  - GM 触发备份失败后，临时备份文件可能堆积在 backup 目录，长期累积消耗磁盘。
  - 子进程僵尸（kill SIGTERM 后未退出）会持有 PG 连接，间接挤占 connection pool。
- 建议：备份文件输出统一到固定 backup 目录 + 启动期 / 后台扫描清理超龄文件；spawn 用 `signal: AbortSignal` 控制 + force kill 后执行 `wait`。

### S60. [P2] NATIVE_HTTP_PROVIDERS 单数组 17 个 service，无子模块隔离

- 文件：`http/native-http.registry.ts:38-67`
- 现象：所有原生 HTTP 服务（auth、account、GM admin、editor、mail、map、player、worker、world、secret、restore-coordinator）扁平注册到同一 module。
- 影响：
  - GM 接口和玩家 auth 接口共享同一注入域，不利于权限隔离审计。

- 建议：拆分为 `AuthModule` / `GmModule` / `AccountModule`，每个 module 内部独立 providers。

### S61. [P1] WorldGateway 未配置 pingTimeout / pingInterval，断线检测窗口与 detach 窗口不匹配

    1. 0–45s：server 仍认为 socket 在线，`unregisterSocket` 没被调；玩家状态保留 connected=true。

### S64. [P0] 战斗结算 / 强化成功率 / 灵根 / 掉落全用 Math.random，无 seed-based PRNG，无审计回放

- 文件：

  - `runtime/player/player-progression.service.ts:1465-2450`：8 处 Math.random，覆盖灵根分配、掉落权重、品质骰子、保留几率

  - **不可审计**：商业级 MMO 红线 §6 "新增或重构功能必须可验证、可观测、可维护"。强化、炼丹、灵根、掉落是玩家最敏感的概率链路，需要 seed-based PRNG（如 xoshiro256**）+ 把 seed + step 写入 asset_audit_log，运营随时回放。

  - 验收链路：固定 seed 跑 1000 次强化，期望成功率与声明值偏差 < 0.5%。

### S70. [P1] PlayerSessionRouteService.clearLocalRoute epoch <= 条件可能误清同 epoch 新路由

- 文件：`persistence/player-session-route.service.ts:155-180`
- 现象：
  ```sql
  DELETE FROM player_session_route
  WHERE player_id = $1 AND node_id = $2 AND session_epoch <= $3
  ```
  reaper 在 detach 后用 binding 旧 epoch 触发清理。但 `<=` 包括等于：如果 `registerLocalRoute` 用相同 epoch 重新写入（比如同节点 register / unregister / register 序列），第二次 register 的路由会被 reaper 用同 epoch 误删。
- 影响：
  - 玩家"flicker disconnect"（断线 < 1s 内重连）场景：reaper 还在 1Hz 周期里准备清理旧 detached binding，binding epoch=5；同节点新 register 也用 epoch=5（如果 epoch 没自增）。reaper 触发 DELETE WHERE epoch <= 5，清掉刚 register 的路由。
  - 客户端发现 route 丢失被强制 fallback。
- 建议：
  - 条件改为 `session_epoch < $3`（严格小于）。
  - 或在 reaper 触发清理前 re-fetch binding，确认 epoch 仍是检查点的值。

### S71. [P1] RedeemCodeRuntimeService 用整文档 persist + 内存 mutationQueue，多节点写互相覆盖

- 文件：`runtime/redeem/redeem-code-runtime.service.ts:50-376`
- 现象：
  - 单进程内 `mutationQueue = Promise.resolve()` 链式串行所有 mutation。

- 影响：

- 建议：
  - 单码消费走 DurableOperationService（已注入但未用于消费）：每次兑换在事务内 SELECT FOR UPDATE 单条 code → 校验 remainingUses > 0 → UPDATE remainingUses-- → INSERT redeem_record。
  - 文档级 group / code 元信息可以保留整文档，但 **使用次数必须按行 partial update**。

### S72. [P1] Worker runLoop 永久 while(true) 无 SIGTERM hook，关停丢正在认领批次

- 文件清单：14 个 worker 文件（`runtime/world/worker/*.worker.ts`）
- 现象：所有 worker 使用相同模式：
  ```typescript

    while (true) {
      const processed = await this.runOnce(workerId);
  ```

  - 关停期间 5–30s 的玩家持久化暂停。
- 建议：
  - worker 启动入口注册 SIGTERM/SIGINT，把 `shouldStop=true`，runLoop 检查后退出。
  - 退出前调 `playerFlushLedgerService.releaseClaims(workerId)` 主动释放认领。

### S75. [P2] main.ts bootstrap 异常路径用 process.exit / void 抛出，无最后落日志窗口

### S97. [P1] instance-domain-persistence ensureXxxTable 用 pg_advisory_lock（session-level）而非 xact_lock，lock 跟随 connection 复用泄漏

- 文件：`persistence/instance-domain-persistence.service.ts:2880-2906`
- 现象：
  ```typescript
  const client = await pool.connect();
  try {
    await client.query('BEGIN');
    await client.query(`SELECT pg_advisory_lock($1, $2)`, [...]);
    // ... CREATE TABLE / INDEX ...
    await client.query('COMMIT');
  } catch (error) {
    await rollbackQuietly(client);
    throw error;
  } finally {
    await client.query(`SELECT pg_advisory_unlock($1, $2)`, [...]).catch(() => undefined);
    client.release();
  }
  ```
  关键问题：
  - 用的是 `pg_advisory_lock` 而非 `pg_advisory_xact_lock`：lock 是 **session-level**，不随事务 COMMIT/ROLLBACK 自动释放。
  - finally 内 `unlock` 用 `.catch(() => undefined)` 兜底，**unlock 失败被吞**。
  - finally 内 `client.release()` 把 connection 还回 pool，但 advisory session-level lock **不随 release 释放**，直接传给下一个用户该 connection 的请求。
- 影响：
  - 罕见情况下 unlock 失败（PG 短暂故障 / 网络瞬断），lock 持续占用 connection。
  - 该 connection 被 pool 重新分配给其他业务路径时，新业务"莫名其妙"持有了上一个表初始化的 advisory lock，可能影响其他并发的 ensureXxxTable 路径。
  - 多个 ensureXxxTable 并发跑（S23 已记录，启动期并发 DDL）撞到这个泄漏会出现循环阻塞。
- 建议：
  - 改用 `pg_advisory_xact_lock`（事务级，COMMIT/ROLLBACK 自动释放）。
  - 或在 release 前显式 `pg_advisory_unlock_all` 清空 session 持有的所有锁。

### S101. [P2] HTTP body 大小限制无显式配置，依赖 NestJS / express 默认 100kb

- 文件：`main.ts`、各 controller，grep `bodyParser|express\.json|json\(\{ limit` 在生产代码 0 处显式配置
- 现象：NestJS 默认用 express body-parser，json/urlencoded limit 默认 100kb（更新后可能是 1mb）。
- 影响：
  - 玩家 API 不需要大 body，OK。
  - GM 备份恢复接口需要上传备份文件，可能 > 100kb，必须配置 multipart/raw 接收并显式 limit。
  - 没有"按 endpoint 限制"机制，可能让小请求接口被攻击者大 body 占用 socket。
- 建议：
  - main.ts 显式 `app.useGlobalPipes` + body-parser limit。
  - GM 备份上传走专门的 multipart/file-upload 路径，独立 limit。
  - 玩家公开 API 限制 16kb。

### S102. [P1] [FIXED 已有] native-postgres-backup createWriteStream 默认权限 0644，敏感备份可被本地用户读

- 文件：`http/native/native-postgres-backup.ts:44-50, 105-130`
- 现象：grep `chmod|0o6\d\d|backupDir` 在 backup 路径 0 处显式 chmod。`createWriteStream(filePath)` 默认权限 = 0666 & ~umask = 0644（默认 umask 0022）。
- 影响：
  - 备份文件包含玩家密码哈希、GM 密码哈希、邮件内容、坊市订单、宗门数据等敏感信息。
  - 默认 0644 让"同主机任意登录用户" 能读备份文件。
  - 容器内通常只有 root + node 用户，docker 隔离基本 OK；但备份文件如果挂出到宿主机 volume，宿主机其他用户可读。
- 建议：
  - createWriteStream 用 `{ mode: 0o600 }` 显式限定 owner-only。
  - backup 目录启动期 `fs.mkdir({ mode: 0o700 })`。
  - docker-stack volume 配置 owner=node:node, mode=0o700。

### S103. [P2] pg query 不使用 named prepared statement，每次完整 parse + plan

- 文件：所有 persistence service 用 `client.query(`SELECT ... $1 $2`, [...])`，grep `prepared` 0 处
- 现象：pg 库支持 `client.query({ name: 'select-player', text: '...', values: [...] })` 让 PG 缓存 prepared plan，重复 query 跳过 parse + plan。当前代码全用匿名 query。
- 影响：
  - 高频 query（玩家 presence 更新、leaderboard 查询）每次走完整 parse + plan + execute。
  - 5000 玩家 × 多 domain × 5s flush cycle，每秒数千次 query，节省 parse plan 能省 10-30% DB CPU。
- 建议：
  - 关键热查询（savePlayerPresence / loadMailbox / claimReadyPlayers 等）改用命名 prepared statement。
  - 注意 PG 单连接 prepared statement 缓存上限，pool.connect 时清理或用 `pg-prepared` 类工具。

### S104. [P2] pg_dump / pg_restore SIGKILL 后无 await child.exit，可能僵尸

- 文件：`http/native/native-postgres-backup.ts:478-490`
- 现象：
  ```typescript
  const timeoutId = setTimeout(() => {
    child.kill('SIGTERM');
    forceKillTimer = setTimeout(() => {
      child.kill('SIGKILL');
    }, PROCESS_FORCE_KILL_GRACE_MS);
  }, PROCESS_TIMEOUT_MS);
  ```
  SIGKILL 后没有显式 `await new Promise(res => child.on('close', res))`。
- 影响：
  - SIGKILL 一般立即生效，但 PG 子进程可能持有 PG server 端连接没释放，PG 端看到 idle in transaction。
  - 备份/恢复链路里持续超时 + SIGKILL 会累积 PG 端死连接。
- 建议：SIGKILL 后 await child 的 close event，确认子进程退出。如果 close 也不来（罕见），强制 logger.error 上报。

### S105. [P2] NativeGmController 单文件汇集 30+ @Post / @Get 接口（建/删/迁实例、密码、ban、flush、备份、恢复、重启）

- 文件：`http/native/native-gm.controller.ts`（33KB，30+ 路由）
- 现象：单 controller 文件汇集 GM 全部高危接口。与 S60 / S74 同类问题但聚焦 GM 接口面。
- 影响：
  - 难以做按职责的权限隔离（mail/admin 应该有不同 GM 等级，但单 controller 用同一 guard）。
  - rate-limit / audit interceptor 难以按接口分类配置。
- 建议：拆成 `NativeGmServerController`（lifecycle）/ `NativeGmPlayerController`（玩家）/ `NativeGmMailController`（邮件）/ `NativeGmInstanceController`（实例）等。
