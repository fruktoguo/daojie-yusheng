# 性能优化清单

## 待处理
- [ ] `packages/server/src/concurrency/encoding-worker-pool.service.ts`
  - 已完成：维护 `activeWorkerCount`，用常量计数代替热路径上的重复数组扫描。
  - 结果：减少 worker 池提交和生命周期管理中的同步开销，不改变行为。
- [ ] `packages/server/src/concurrency/instance-worker-pool.service.ts`、`packages/server/src/concurrency/persistence-worker-pool.service.ts`
  - 问题：这两处 worker 池与 encoding worker pool 相同，`submit()`、`initialize()`、`ensureWorkersStarted()`、`isEnabled()` 和 `handleWorkerDeath()` 都在重复扫描 worker 数组来统计活跃 worker。
  - 影响：worker 池是高频调度入口，重复扫描会在任务密集或 worker 波动时放大同步开销。
  - 处理建议：同样维护 `activeWorkerCount`，在启动、死亡、关闭时增减，只在真正需要枚举 worker 的地方扫描数组。
  - 判断：可在不改变设计和表现的前提下修复。
- [ ] `packages/server/src/concurrency/workers/encoding.worker.ts`
  - 问题：`handlePathfind()` 里只要 `input.walkable` 和 `input.traversalCost` 存在，就会强制刷新 `cachedGrid`，而 `AsyncPathfindingService` 每次提交 pathfinding 任务时都会携带这两个数组，导致 worker 侧的静态网格缓存几乎失效。
  - 影响：同一张地图/同一 revision 下的重复寻路请求，会反复重建 `PathfindingStaticGrid`，放大 worker 侧 CPU 与内存访问成本，抵消缓存收益。
  - 处理建议：只按 `(mapId, mapRevision)` 判定是否需要重建缓存，去掉对 payload 中 `walkable/traversalCost` 的强制刷新条件。
  - 判断：可在不改变设计和表现的前提下修复。
- [ ] `packages/server/src/runtime/world/combat/world-runtime-auto-combat.service.ts`
  - 问题：`resolveAutoBattleSelfSkillChoice()`、`resolveAutoBattleSkillChoice()`、`resolveAutoBattleDesiredRange()` 在遍历每个技能动作时，都会重复调用 `findPlayerSkill()`，而 `findPlayerSkill()` 会再次线性扫描 `player.techniques.techniques`。
  - 影响：同一轮自动战斗技能筛选中，会出现“动作列表遍历 × 技能列表遍历”的重复成本；当可用动作和已学技能都偏多时，这部分会放大热路径开销。
  - 处理建议：在进入筛选前为当前玩家构建一次技能索引，并在本轮筛选中复用；保持首个匹配技能的语义不变。
  - 判断：可在不改变设计和表现的前提下修复。
