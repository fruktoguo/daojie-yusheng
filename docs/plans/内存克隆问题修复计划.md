## 紧急修复 P0-LEAK（2026-05-17，正式服 RSS 2.76 GB / old_space 2.35 GB 现场反推）

### 现象

正式服在已经包含 P0/P1/P2 全部修复的版本上观察到：

```
RSS 2759 MB · Heap used 2438 MB · old_space used 2.35 GB · old_space size 2.53 GB · 使用率 92.8%
```

old_space 几乎触顶（剩 130 MB 余量），距离 OOM 一步之遥；比修复前的 1.37 GB old_space 还高。

### 根因

P0 一轮新增了 12 个 entry-level cache（map-instance 内 9 个、network 服务内 1 个、world-runtime 内 1 个、envelope 内 1 个 WeakMap），但**只有 2 个写了对应的 lifecycle 清理路径**：

| Cache | 类型 | 修复后是否随实体释放 |
| --- | --- | --- |
| `playerViewCacheByPlayerId` | map-instance 字段 | ✅ removePlayer 调 delete |
| `localMonsterViewCacheByRuntimeId` | map-instance 字段 | ✅ collectLocalMonsters 检查 alive 后 delete |
| `tileProjectionByCoord` | **service 字段（按 `${instanceId}:${x},${y}` 索引）** | ❌ 实例销毁不清理；line/sect/tongtian-tower 等动态实例累积所有 tile entry，正式服跑几天就堆几十万 |
| `npcQuestMarkerViewCacheByPlayerId` | **service 字段（按 playerId 索引）** | ❌ 玩家断线/移除不清理；累积 2200+ 历史玩家 |
| `localPlayerViewCacheByPlayerId` | map-instance 字段 | ❌ removePlayer 不清理 |
| `localBuildingViewCacheById` | map-instance 字段 | ❌ building deconstruct 不清理 |
| `localGroundPileViewCacheBySourceId` | map-instance 字段 | ❌ 拾取空时 groundPilesByTile.delete 但 view cache 不清理（每天数千~数万次掉落 / 拾取） |
| `localNpcViewCacheById` / `localPortalViewCacheById` / `localContainerViewCacheById` / `localLandmarkViewCacheById` / `localSafeZoneViewCacheByKey` | map-instance 字段 | 跟随实例 GC，但因缺少销毁链路统一调用，依赖隐式 |
| `containerRespawnProjectionCache` | WeakMap by entry | ✅ entry GC 后自动清 |
| `overlayProjectionCache` | WeakMap by entry | ✅ |
| `runtimeFormationProjectionCache` | WeakMap by formation | ✅ |
| `visibleBuffProjectionCache` | WeakMap by buff source | ✅ |
| `cultivationBuffProjectionCache` / `buildingBuffProjectionCache` / `darknessBuffProjectionCache` | Map（key 短串） | 容量极小可接受 |
| projector 7 个 entry WeakMap | WeakMap | ✅ |

`tileProjectionByCoord` 与 `npcQuestMarkerViewCacheByPlayerId` 是 **service-level 永久 Map**，跨实例 / 跨玩家累积；其余 5 个 instance 字段在动态实体（玩家、建筑、地面物品堆）上缺 delete 钩子。这就是 old_space 不降反升的根因——修复并未泄漏运行态短命对象，而是把"曾经看过的实体"持续 freeze 到这些 cache 里再也不释放。

### 修复方案（已实施）

按"让 cache 跟随宿主对象生命周期"原则，避免再增加销毁钩子：

1. **`tileProjectionByCoord` 从 service 字段移到 `MapInstanceRuntime` 字段**：实例对象 GC 时整张 Map 跟随释放，无需 lease helper 调用清理钩子（避免 network 包反向依赖 runtime）。
2. **`npcQuestMarkerViewCacheByPlayerId` 从 service 字段移到 player runtime 对象上（`player.npcQuestMarkerCache`）**：`PlayerRuntimeService.removePlayerRuntime` 即 player 引用消失时整张 Map 跟随释放。`createFreshPlayer` / hydrate 路径 / `cloneRuntimePlayerState` 三处都同步初始化字段。
3. **`map-instance.runtime` 在动态实体销毁路径补 delete**：
   - `removePlayer` → `localPlayerViewCacheByPlayerId.delete(playerId)`
   - building deconstruct → `localBuildingViewCacheById.delete(buildingId)`
   - groundPile 拾取空 → `localGroundPileViewCacheBySourceId.delete(buildGroundSourceId(tileIndex))`
   - `hydrateGroundPiles` 重置时 → `localGroundPileViewCacheBySourceId.clear()`
4. **其他 5 个 cache（NPC/portal/container/landmark/safeZone）绑定 template 来源的固定实体**，实例 GC 时跟随释放，无需额外 delete。
5. **retention smoke 增加 `cacheLifecycleProof`**：通过源码扫描验证上述 4 条修复全部落到生产代码：
   - `tileProjectionOnInstance`：`MapInstanceRuntime.tileProjectionByCoord = new Map()` 存在 + service 写 instance 字段 + service 不再持 service-level 字段
   - `npcQuestMarkerCacheOnPlayer`：player 三处初始化都写 `npcQuestMarkerCache: new Map()` + service 改读 `player.npcQuestMarkerCache` + service 不再持 service-level 字段
   - `removePlayerClearsLocalPlayerView` / `buildingDeconstructClearsCache` / `groundPilePickupClearsCache` / `hydrateGroundPilesClearsCache`：4 个生产源码标记都存在
6. retention smoke 旧的 `proveTileProjectionRefsReachPlayerCache` mock 同步更新，提供 `getInstanceRuntime` 返回 fake instance，并新增 `instanceCacheUsed` 字段断言 cache 真的写到了 instance 对象上。

## 总览

| 类别 | 数量 | 典型场景 |
| --- | --- | --- |
| 每人每帧（tick 1Hz） | 9 | projector world/panel 缓存、map static tile 缓存、aux realm/loot 缓存、view collect、视野 envelope spread |
| 每人面板 revision 变化时 | 4 | 背包、装备、功法、属性、buff 全量 clone |
| 每人 + 每实例特效 × 4 | 1 | combat effect 三~四重 clone 链 |
| 每人首次进入/跨图 | 1 | bootstrap self 状态深克隆 |
| 每实例每若干 tick 或低频 | 2 | 持久化 dirty domain flush、market snapshot 深克隆 |

### 估算口径

- 在线玩家 200，离线挂机玩家额外 + ~500（按现有运营预算 5000 玩家的 4% 同时在线 + 10% 离线挂机近似）
- 每玩家视野内 50 怪物 / 10 NPC / 5 portal / 10 容器 / 5 landmark / 3 safeZone / 5 ground pile / 3 building / ~3 formation overlay
- 怪物全服 2000，地图实例 100 张
- tick 频率 1 Hz，玩家面板 revision 变化频率约 0.3~1 次/玩家/秒（属性结算、装备更新、buff 衰减）
- 视野半径 r=10 → 每人 visibleTiles = 21×21 = 441
- 单 plain Object 约 60 B（V8 hidden class + 字段 slot），其中含数组/Map 结构再 +50~200 B
- old_space 当前约 1.37 GB；本次目标降到 600~800 MB（减少 40~60%）

### 估算总内存收益

| 优先级 | 改造完成后预估节省 | 说明 |
| --- | --- | --- |
| **P0 全部完成** | 380~500 MB | 直接消除 visibleTiles/world cache 共 ~3 万实体 ×（200 在线 + 500 离线挂机）份 plain object |
| **P0 + P1 全部完成** | 480~620 MB | 加上面板 revision 缓存共享化、attr bonuses 引用化 |
| **P0 + P1 + P2 全部完成** | 510~650 MB | 加上特效链路、loot/realm 比较前不克隆 |

> 估算偏保守。实际可能更高，因为 V8 在大堆下 GC 不充分会让短命对象晋升 old_space。

### 改造难度分级

- **难度 1**：定位明确，改"先比较再克隆"或"直接复用引用"，不需要改 diff 对比逻辑
- **难度 2**：需要把"per-player 缓存"重构为"per-instance 共享投影 + 引用比较"，但 diff 算法只需把"逐字段对比"换成"prev !== curr"
- **难度 3**：需要重新设计投影 revision/版本约定（实例侧维护 entry version，玩家侧只存 version 数字而非 entry 副本），跨多个文件协同
- **难度 4**：触动持久化或战斗结算正确性，需补 smoke/proof 验证

## 共性架构方案（不可变投影 + 引用缓存）

本节给出适用于 P0/P1 多数条目的共性改造方案，避免每条目重复展开。

### 现状问题模式

```
[运行时真源] -> [每帧 per-player clone 视野/面板快照] -> [缓存到 prev[playerId]]
                              ↓                                    ↓
                       [发包: 全量或 patch]            [下一帧 diff 基线]
```

每个玩家独立持有一份 prev 快照，N 玩家就有 N 份 entry 副本常驻 old_space。即使内容完全一致，每帧 clone 一份新版本作为 base，旧版本失引用等待 GC，但分配速率高于 GC 速率时会晋升 old_space 而非及时回收。

### 推荐架构

```
[运行时真源（实例侧或玩家侧权威态）]
        ↓ entry-level revision/version 维护（自增计数器）
[ImmutableEntryProjection: { version: N, payload: { ... } }]   ← 每个实体共享同一份引用
        ↓ 玩家侧只存 version map: Map<entryId, version>
[Diff: prev.get(entryId).version !== curr.version → 触发 patch]
        ↓
[发包]
```

关键点：

1. **实体投影对象只在运行时数据真正变化时新建一次**（实例 / runtime 真源侧）。视野内同一个怪物/NPC 被 N 个玩家看到时，N 个玩家共享同一个 ProjectedXxxEntry 引用。
2. **玩家侧 prev 缓存只存 `Map<entryId, versionNumber>` 或 `Map<entryId, sharedRef>`**。不再 clone 每个 entry。
3. **diff 算法降级为引用相等性比较**：`prev.get(id) !== curr.get(id)` → 变化。原"逐字段对比生成最小 patch"改为"已知整个 entry 变化，直接 push 整 entry"。极少数需要更细粒度 patch 的字段（HP/QI/位置）单独走"实体内部高频字段子 revision"通道。
4. **静态字段与动态字段分离**：tile 的 type/walkable/blocksSight/terrain/structure 等极少变化的字段属于静态半冷数据，应该走"模板基线 + 玩家首次进入快照 + 仅在改造/破坏时 invalidate"模型，不在 tick 热路径每帧重建。

### 不可变投影约束

- 投影对象创建后，所有字段都不再被原位修改；改变即新建一个 entry。
- 所有 consumer 不得 mutate 投影对象。改造时建议 `Object.freeze` 或 TS 标注 `Readonly<T>`，防回退。
- 玩家侧 prev 缓存只持有 entry 的引用或 version。entry 共享 + version 比较是最低分配代价的 diff 实现。

### diff 算法迁移

| 当前 diff 行为 | 改造后 diff 行为 |
| --- | --- |
| 逐字段比较 prev.x !== curr.x → 生成 { x: curr.x } | prev !== curr → 生成 { x, y, hp, ... } 完整 patch；客户端按字段更新 |
| previous: Map<id, ProjectedEntry>（per-player 独立副本） | previous: Map<id, sharedRef>（引用共享）或 Map<id, versionNumber> |
| 每帧 captureWorldState → 全量 .map → 新 Map | 实例侧维护 sharedWorldProjection；玩家侧只 capture id 集合 |
| panel slice 每次 revision 变化全量 clone | panel slice 直接持引用，diff 用 `prev.inventory !== curr.inventory` |

性能与包体折中：

- "不再生成最小字段 patch，而是发整 entry"会让单 patch 字段数从 1~3 增加到 8~12，但只在 entry 真正变化时才发，并不增加 patch 数量；包体增量在 5000 玩家口径下可控。如果证明字段维度增量必要（典型是怪物高频 hp/x/y 的 1 字段更新），可以保留怪物入口的细粒度 diff，其它实体走整 entry。

## 风险与注意事项

### 引用共享的可见性风险

- 一旦投影 entry 共享 ref，**任何下游消费者绝对不能 mutate 该对象**。需要：
  - TS 标注 `Readonly<T>` / `ReadonlyArray<T>`，编译期阻止误改；
  - 关键 entry 在 dev 与 smoke 环境用 `Object.freeze` 防御；
  - audit 现有消费侧（envelope 拼装、客户端发包前后处理）是否原位改字段。
- 现有 `world-sync-envelope.withContainerRespawnProjection` 就是典型"对消费侧 view 做 spread + 改字段"。改造后这类需求要么走 instance 投影再生（写路径触发），要么走"复制一次 entry 的浅 patch"独立路径。

### diff 颗粒度从字段级降到 entry 级的包体影响

- 当前 `diffMonsterEntries` 等只下发变化的字段（如 monster 移一格只发 `{id, x, y}`）。改成"prev !== curr → 整 entry"后，包体增长 5~10 倍（按字段数）。
- 缓解：在 entry-level cache 之上，仍可保留"字段级 patch"作为协议下发；但服务端比较从"isSameXxx 逐字段"换成"prev.ref === curr.ref ? 0 patch : per-field-diff"。即引用相等优先，不等才走逐字段。
- 重点验证 5000 玩家压测下网络出口带宽不超 30 Mbps 上限。

### worldRevision 颗粒度问题

- 当前 worldRevision 是实例级，任何实体变化即全图 invalidate。共享 ref 化后 worldRevision 仍需保留作为 fast path（worldRevision 不变 → projector cache 直接命中），但**不再当作"必须重建所有 entry"的信号**，因为 entry 已经按各自 lifecycle 维护 ref。
- 转换期建议保留 worldRevision 作为兜底 invalidate（防止有写路径漏触发 entry 级 ref 更新），但日志/指标观察 ref 命中率。

### 持久化语义不变

- buildPersistenceSnapshot 现状每 5s 走完整克隆 → 落库 payload 是 plain JSON，序列化逻辑兼容。改造后若改为"持 frozen ref + 序列化时再读"，要保证序列化期间没有并发写真源。
- 推荐保留 buildPersistenceSnapshot 输出"独立 plain JSON"语义不动，仅按 dirty domain 子集减少克隆量（P2-2 最低风险版）。

### 战斗 / 行为正确性

- 战斗 effect / aoi event 改为 ref 透传后，需要确认实例 flushTick 清空时不影响已经"在玩家 envelope 引用中的对象"（ref 仍可读）。当前 `RuntimeEventBusService.drainInstance` 是"移交引用 + 清空 Map"，已经符合；但要检验 GC 路径上没有别处再"reset 字段"。
- 建议给 CombatEffect 加 freeze 或类型 readonly。

### 测试 / smoke 改造

- `server-memory-retention-smoke` 是本计划主要验证抓手。需要扩展：
  - `instanceProjectionProof`：同实例多玩家共享 entry ref 数。
  - `tileProjectionProof`：90 tick 静态 tile ref 全程不变。
  - `panelSliceRefProof`：inventory/equipment/buff revision 不变时 cache.panel.X.items 全程同 ref。
  - `combatEffectRefProof`：envelope 中 combatEffects/fx 与 instanceQueues 共享 ref。
- bench：在 200 玩家 / 50 怪物 / 视野 10 场景下，每 tick 的 plain Object 分配数量级（用 `--inspect` + heap snapshot diff 对比）。

### dev vs prod 环境行为差异

- dev 模式下 nest 模块缓存抬高基础堆 ~30 MB（OOM 报告 4.4 注解），改造效果在 prod docker 环境会更明显。建议：
  - 改造完成后用本地 docker stack 跑 200 bot 压测 30 分钟，对比 RSS 曲线。
  - 在 GM「内存」面板观察 `v8_heap_used_unattributed` 改造前后的稳态。
