# 服务端与客户端多线程并行化改造计划

> 落点：`packages/server`（主）+ `packages/shared`（协议契约 + Pathfinding 边界）+ `packages/client`（OffscreenCanvas/Web Worker）+ `scripts/`（验证编排并行化）
> 类型：跨 runtime / network / persistence / renderer / tooling 的并发能力基础设施改造
> 优先级：**P1**（不解决商业级 MMO 5000 在线 / 10000 实例的单核 CPU 瓶颈，1Hz tick 与 AOI 广播会先于持久化、协议层成为天花板）
> 配套阅读：`AGENTS.md` §4 商业级 MMO 口径、§8 权威运行时红线、§9 网络同步红线、§12 性能红线

---

## 1. 背景

### 1.1 当前事实（基于全仓扫描）

仓库 `packages/*` 与 `scripts/` 目前**完全没有真正的多线程实现**：

- 没有 `worker_threads`、`cluster`、Web Worker、`SharedArrayBuffer/Atomics`、`OffscreenCanvas`
- `packages/server/src/runtime/world/worker/*.worker.ts` 共 14+ 个文件，全部是 NestJS `@Injectable` + `setInterval` 的"业务定时作业 worker"，跑在主事件循环上，与命名误导无关
- `child_process.spawn / spawnSync` 仅用于：
  - `native-postgres-backup.ts` 的 `pg_dump / pg_restore`（运维冷路径，唯一真子进程）
  - `main.ts` 端口冲突诊断（启动期一次性）
  - `tools/dev-hot.ts`（开发期热重载）
  - `tools/smoke-suite.ts` / `tools/run-stable-*.ts` 系列（CI/验证期，且**串行** spawn）
  - `scripts/release-{full,acceptance,with-db,shadow,...}.js`（**全 `spawnSync` 顺序执行**）

### 1.2 已经预留好横向扩展协议但仍单进程跑的地方

- `packages/server/src/persistence/outbox-dispatcher.service.ts`：`dispatcherId` + `FOR UPDATE SKIP LOCKED` 已经写好，运行时仍是单 timer 单进程
- `packages/server/src/persistence/flush-ledger.service.ts` + `tools/multi-worker-flush-stability-report.ts`：ledger claim 协议已为多 workerId 设计，smoke 已在**同进程**模拟多 workerId 并发抢 ledger，但生产仍是单线程同 Promise 队列
- `packages/shared/src/pathfinding.ts` 的 `findBoundedPath` 已改成 `Uint8Array walkable / Uint16Array traversalCost` 的静态网格形态——**完全契合 Transferable** 跨 worker 传输
- `benchmarks/pathfinding/rust/` 已经存在独立 Rust A\* 实现 + benchmark 二进制，证明寻路热点已被识别且做过 cross-runtime 评估

### 1.3 关键的单线程瓶颈点（按 5000 玩家 / 10000 实例口径）

| 模块 | 关键文件 | 单线程开销特征 |
|---|---|---|
| AOI 广播 envelope 编码 | `packages/server/src/network/world-sync.service.ts:81` `flushConnectedPlayers` 顺序 for-of 5000 玩家 | 每帧串行调用 N 次 `getPlayerView` + `createDeltaEnvelope` + protobuf encode + emit |
| protobuf 编码 / 解码 | `packages/shared/src/network-protobuf*.ts`（10 个文件） | 纯 CPU bound，主线程同步 encode 5000 次/帧 |
| 服务端 A\* 寻路 | `packages/server/src/runtime/world/world-runtime.path-planning.helpers.ts` | 注释明说"在 tick 外执行"，但仍与主事件循环共享 V8 isolate |
| 实例 tick advance | `packages/server/src/runtime/world/world-runtime-instance-tick-orchestration.service.ts:21` `advanceFrame` 顺序 for-of 全实例 | 10000 实例时每帧串行推进，跨实例无依赖却共用 1 个核 |
| FOV / 怪物派生状态 | `packages/server/src/runtime/instance/fov.helpers.ts`、`map-instance-monster-advancer.ts` | 已被列入 `bench-hot-paths.ts` 基准，玩家间纯独立计算 |
| Postgres 持久化 flush | `packages/server/src/runtime/world/worker/*.worker.ts`（14+ 个 ledger worker） | 主线程 SQL 构造 + bigint 转换 + JSON.stringify 抢 tick CPU |
| 客户端 Canvas / 协议 decode / 大 catalog 解析 | `packages/client/src/{game-map,renderer,network,constants}/**` | 全在主 UI 线程，手机端弱 CPU 上撞首屏与帧率 |

### 1.4 问题定性

按 `AGENTS.md` §4：

> 硬件：8 核 CPU / 16GB 内存 / 30Mbps 出口带宽（单服）；并发玩家：5000；地图实例：10000

按 §8：

> 服务端是唯一权威来源；玩家不可重叠；战斗、怪物、掉落、采集、任务、市场、邮件、GM 命令等都必须保持服务端权威

按 §12：

> 高频链路必须按玩家数、实体数、地图数增长后的成本设计

当前架构在"**所有运行时逻辑共享一个 V8 isolate**"前提下，1 个 Node.js 进程只能用满 1 个核，剩下 7 个核完全闲置。5000 玩家 1Hz tick 把"视图组装 + envelope 编码 + emit"压在同一个核上时，会**先于 DB / 网络带宽**成为瓶颈。这是当前架构最确定要打破的天花板。

## 2. 目标与不变量

### 2.1 目标

1. **建立服务端 worker pool 基础设施**：以 `worker_threads` 为载体，提供"**纯计算任务**"派发池（输入：plain payload；输出：plain payload 或 `Transferable`），与 NestJS DI 容器边界清晰
2. **AOI envelope 编码外移**：把 `flushConnectedPlayers` 单帧 5000 次 protobuf encode 卸到 worker pool，主线程只做"收 envelope bytes → emit"
3. **A\* 寻路外移**：服务端 / 客户端共用 `shared/pathfinding` 入口，新增 worker 异步通道；服务端用 worker pool，客户端用 Web Worker
4. **多地图实例 tick 分片**：`MapInstanceRuntime` 跨实例完全独立的部分（资源流动、阵法 tick、怪物 AI、自动战斗预决算）外移到 instance worker；权威写仍在主线程汇总
5. **客户端 OffscreenCanvas 渲染分离**：地图层渲染、tile-sprite-cache 预生成、文字测量缓存外移到 Web Worker
6. **持久化 flush 序列化外移**：14 个 ledger worker 中"序列化 + bigint 转换 + JSON.stringify"部分外移到 worker pool，主线程只发 query
7. **运维 / 验证编排并行化**：`scripts/release-*.js`、`smoke-suite` 把无依赖步骤改为 `Promise.all + spawn`
8. **不引入新进程模型**：单服仍是 1 个 Node 主进程 + 内嵌 worker pool，**不**走 cluster / 多 Node 进程架构（不影响 socket.io 单实例广播、不影响 lease 与 session 语义）

### 2.2 不变量（必须维持）

- **服务端权威不变**：worker 不持有 `MapInstanceRuntime` 真源、不持有 `socket.io` 连接、不直接访问 DB pool；worker 只做"接收 plain 输入 → 纯计算 → 返回 plain 输出"
- **协议字段不变**：envelope / panel / aux / threat / minimap 输出与现网完全一致；多线程化只改"哪个核做编码"，不改"编码什么"
- **持久化真源不变**：`postgres` + `redis` 仍是唯一持久化与在线态出口；worker 不直接 write through
- **1Hz tick 不变**：`WorldTickService` 的 setTimeout 自调度形态保留，`advanceFrame` 内部对独立实例 fan-out 时必须能在 tick interval 内 join 完成
- **断线重连 / lease / 顶号 / fence 语义不变**：所有 session 状态变更仍由主线程串行处理
- **Outbox / DurableOperation / 审计链路不变**：worker 不发 outbox 事件、不写 durable_operation
- **同步组包契约不变**：`worldDelta.eventBus` / `pendingPlayerNotices` 等仍由主线程编排
- **错误恢复不变**：worker 异常必须能 fallback 回主线程同步执行，**不允许**因 worker 死亡导致玩家被踢、tick 卡死
- **5000 玩家 1Hz tick / 10000 实例 / 8C16G / 30Mbps 单服口径不变**

### 2.3 不在范围

- **不**改 1Hz tick 频率、AOI 半径、socket 协议形态
- **不**做 cluster 多进程（Node cluster 与 socket.io 单实例广播不兼容；后续如果做横向扩，应直接走多 Node 实例 + Redis adapter，不在本计划）
- **不**重写 NestJS 启动模型为 worker-first
- **不**在 worker 里直连 PG/Redis 池（保持"主线程持有连接池"的边界）
- **不**做 GPU 加速 / WebGL（Canvas 2D 不变）
- **不**改持久化 schema、不改 outbox / dead_letter / durable_operation 表结构
- **不**引入 Rust/N-API（pathfinding Rust 基准是对照参考，本计划仍走 worker_threads 纯 JS 边界；Rust 路线另起 ADR）

## 3. 总体架构

### 3.1 三类 Worker 形态

```
┌──────────────────────────────────────────────────────────┐
│                Node.js 主线程（NestJS App）                │
│  WorldTickService / WorldRuntimeService / WorldSyncService │
│  socket.io / DatabasePool / RedisClient / Outbox           │
└──────────────────────────────────────────────────────────┘
       │                │                  │
       │ A. CPU pool    │ B. Instance pool │ C. Persistence pool
       ▼                ▼                  ▼
┌────────────────┐  ┌────────────────┐  ┌────────────────┐
│ EncodingWorker │  │ InstanceWorker │  │ FlushWorker    │
│ × N            │  │ × M            │  │ × K            │
│                │  │                │  │                │
│ - protobuf     │  │ - resource     │  │ - SQL build    │
│   encode       │  │   advance      │  │ - bigint conv  │
│ - A* path      │  │ - monster AI   │  │ - JSON.stringify│
│ - FOV          │  │ - formation    │  │   raw_payload  │
│ - fengshui     │  │ - building     │  │                │
└────────────────┘  └────────────────┘  └────────────────┘
       (无状态)         (实例分片态)        (无状态)
```

- **A. CPU pool（`EncodingWorkerPool`）**：无状态，纯函数池。任务粒度小（单玩家 envelope、单次 A\*、单玩家 FOV），并发度 = `min(N_cpu - 2, 6)`，主线程 round-robin 派发
- **B. Instance pool（`InstanceWorkerPool`）**：每个 worker 持有 K 个 `MapInstanceRuntime` 的**只读快照**，按 `instanceId` 哈希分片；只承担"**可在 tick 内完整算完且不需写权威态**"的子阶段（资源流动、阵法 tick 推进、怪物 AI 决策、自动战斗预选目标）；产出"操作意图列表"回主线程，由主线程权威应用
- **C. Persistence pool（`PersistenceWorkerPool`）**：无状态，吃 dirty domain 列表，输出 SQL 参数集合；主线程负责 `pool.query`

### 3.2 主线程 → Worker 协议

所有 worker 任务用统一 envelope：

```typescript
// packages/server/src/concurrency/worker-task.types.ts (新增)
interface WorkerTaskEnvelope<TPayload> {
  taskId: string;          // 主线程分配 UUID v4
  kind: 'envelope-encode' | 'pathfind' | 'fov' | 'instance-advance' | 'persistence-build';
  payload: TPayload;
  deadlineMs: number;      // 主线程要求的最迟完成时间（Date.now() + budget）
  fallbackOnTimeout: boolean; // 超时是否允许主线程自己跑兜底
}

interface WorkerTaskResult<TResult> {
  taskId: string;
  ok: boolean;
  result?: TResult;
  errorMessage?: string;
  durationMs: number;
}
```

**强约束**：
- 入参出参必须是 plain object / `ArrayBuffer` / `Uint8Array` / `TypedArray`，不传 class 实例、不传 `Map/Set`（postMessage 结构化克隆代价高，禁止传大 graph）
- 大 buffer 用 `transferList` 移交，不复制
- 任何 worker 异常 → 主线程 fallback 到原同步路径，**不能**让玩家请求挂死
- 任务必须有 `deadlineMs`：worker 池满或 worker 死锁时，主线程超时降级，记录指标到 `RuntimeGmStateService`

### 3.3 NestJS 集成边界

新增模块 `packages/server/src/concurrency/`：

```
concurrency/
  worker-pool.module.ts          # NestJS Module
  worker-task.types.ts            # 协议契约
  encoding-worker-pool.service.ts # CPU pool（envelope/path/fov）
  instance-worker-pool.service.ts # Instance pool
  persistence-worker-pool.service.ts # Persistence pool
  worker-pool-metrics.service.ts  # 指标采集（任务数、p50/p95、超时数、异常数）
  workers/                        # 实际 worker 入口（编译后是 dist/concurrency/workers/*.js）
    encoding.worker.ts
    instance-advance.worker.ts
    persistence-build.worker.ts
```

worker 入口文件**不**依赖 NestJS 容器，直接 `import` shared 纯函数。这避免了 worker 启动时把整个 NestJS 反射元数据 + decorator 反射加载一遍（实测会让 worker 启动 > 500ms，破坏池化收益）。

## 4. 阶段拆解

### Phase 0 — 基础设施落地（前置，不改业务行为）

**目标**：给后续阶段铺好 worker pool 基础设施，本阶段不影响任何业务行为。

**范围**：
- 新增 `packages/server/src/concurrency/` 模块骨架
- `EncodingWorkerPool` / `InstanceWorkerPool` / `PersistenceWorkerPool` 三个 service，初始**空实现**：构造时不真启 worker，`submit()` 直接同步调用 fallback
- 引入特性开关：`SERVER_WORKER_POOL_ENABLED=false`（默认关），打开后才真正起 worker
- `WorkerPoolMetricsService` 接到 `RuntimeGmStateService` 的 GM 性能页（与现有 `recordSyncFlushBreakdown` 同处）
- 新增 `packages/server/tsconfig.json` 中 worker 编译路径处理（worker 入口必须能在 `dist/concurrency/workers/*.js` 单独以 `node` 启动；不能依赖 NestJS bootstrap）

**改造点**：
- `packages/server/src/app.module.ts`：注入 `WorkerPoolModule`
- `packages/server/src/runtime/gm/runtime-gm-state.service.ts`：增加 worker pool 指标域
- `packages/server/package.json`：构建脚本确保 worker 入口被 emit 到 dist

**产出**：
- 池化 API 形态稳定（`submit(taskId, payload, deadlineMs) -> Promise<Result>`）
- 启用开关默认关闭，所有 `submit()` 走"主线程同步 fallback"，行为与现状完全一致
- GM 性能页能看到 worker pool 指标（即使全是 0）

**验证**：
- `pnpm verify:quick` 通过（无业务行为变化）
- `pnpm --filter @mud/server smoke:worker-pool-bootstrap`（新增）：开关开/关两种模式启动主进程，确认主进程不卡死、worker 进程能拉起也能优雅退出
- 启动期开关打开/关闭场景下 `pnpm --filter @mud/server proof:server-runtime-boundaries` 通过

**回退**：删除新增 module + 关闭开关即可。

### Phase 1 — AOI envelope 编码外移（最高 ROI）

**目标**：把 `WorldSyncService.flushConnectedPlayers` 中"createDeltaEnvelope + protobuf encode + emit"链路里**最重的两步**（envelope 装配 + protobuf encode）外移到 `EncodingWorkerPool`，主线程只剩"收 bytes → socket.emit"。

**改造点**：

1. **拆分编码契约**：
   - `packages/server/src/network/world-sync-envelope.service.ts:createDeltaEnvelope` 拆成两个阶段：
     - 阶段 A（主线程）：从权威态投影出 plain "envelope spec"（POJO，无 class instance、无 Map/Set）
     - 阶段 B（worker）：吃 envelope spec → 输出 `Uint8Array` envelope bytes
   - 阶段间契约写到 `packages/shared/src/network-envelope-spec.types.ts`（新增），与 `network-protobuf*.ts` 复用 schema

2. **改造 flushConnectedPlayers**：
   - 主线程对每个 binding 投影出 envelope spec，丢进 `EncodingWorkerPool.submit()`，得到 `Promise<Uint8Array>`
   - 用 `Promise.all` 等所有 in-flight 任务返回，按原顺序 `socket.emit`
   - **保留同步 fallback**：worker pool 关闭、worker 异常、deadline 超时 → 同步执行原 `createDeltaEnvelope` 路径

3. **路径独立的次链路同步**：
   - `emitAuxDeltaSync` / `emitQuestSyncIfChanged` / `emitPendingRuntimeEvents` / `emitPendingPlayerStatisticRecords` 仍主线程执行（这些链路涉及主线程权威态读取），不进 Phase 1

4. **特性开关与灰度**：
   - 增加 `SERVER_AOI_ENVELOPE_WORKER_ENABLED=false`，默认关
   - 打开后按 `playerId` 哈希取模决定走 worker 路径还是同步路径，灰度 10% / 50% / 100%

**产出**：
- `flushConnectedPlayers` 主线程 CPU 占用显著降低（基准目标：5000 玩家 / 单帧主线程 envelope 装配+encode 耗时减半以上）
- 同步 fallback 路径与 worker 路径输出 bytes 必须**逐字节相等**（CI 必须验证）

**验证**：
- 新增 `pnpm --filter @mud/server smoke:worker-aoi-envelope-equivalence`：构造 100 个玩家 view，开关关 / 开两种模式各跑一遍，断言 envelope bytes 与日志事件完全相同
- 新增 `pnpm --filter @mud/server bench:worker-aoi-envelope`：构造 5000 玩家虚拟 view，对比开关关 / 开两种模式下 `flushConnectedPlayers` 总耗时与主线程占用时间，要求开模式主线程 CPU 时间下降 ≥ 40%
- `pnpm verify:quick` + `pnpm audit:protocol`（确认协议字段未漂移）

**风险**：
- protobuf encode 在 worker 里复用 `protobufjs.Reader/Writer` 时要避免共享单例引发竞态：worker 内每个任务用 fresh writer
- envelope spec 投影阶段可能比原 `createDeltaEnvelope` 慢（多了一层数据拷贝）：必须在 bench 里证明"投影 + worker encode + emit"总耗时 < 原"主线程 encode + emit"

**回退**：关 `SERVER_AOI_ENVELOPE_WORKER_ENABLED` 即可，主线程同步路径保留。

### Phase 2 — A\* 寻路外移（CPU pool 复用）

**目标**：把服务端 `world-runtime.path-planning.helpers.ts:findPathPointsOnMap` 与客户端 `client/src/pathfinding.ts:findPath` 都接入 `EncodingWorkerPool`（服务端）/ Web Worker（客户端）。

**改造点**：

1. **shared 契约**：
   - `packages/shared/src/pathfinding.ts` 已有 `findBoundedPath(staticGrid, occupied, sx, sy, goals, limits)`，输入已是 `Uint8Array / Uint16Array`，**直接可 Transferable**
   - 新增 `packages/shared/src/pathfinding-task.ts`：`PathfindingTaskInput` / `PathfindingTaskResult`，与 worker 任务 envelope 对齐

2. **服务端**：
   - `world-runtime.path-planning.helpers.ts` 增加 `findPathAsync(input): Promise<PathResult>` 入口：内部 `EncodingWorkerPool.submit('pathfind', input, deadlineMs)`
   - 调用方按需切换：tick 内寻路（不能阻塞 tick）继续走同步路径；玩家挂主动寻路意图（在 `world-runtime-navigation.service.ts:enqueueMoveTo` 解析期）走 async 路径，`enqueueMoveTo` 自身由 socket handler 触发，与 tick 解耦
   - `staticGrid` 用 `transferList` 跨 worker，但**只在首帧或 mapRevision 变更时**重新传输；worker 内按 `(mapId, mapRevision)` 缓存

3. **客户端**：
   - 新增 `packages/client/src/workers/pathfinding.worker.ts`，`Vite` 配置 `import.meta.url + new Worker(..., { type: 'module' })`
   - `packages/client/src/pathfinding.ts:findPath` 改成异步入口；保留同步入口供测试代码或同步预览使用，UI 主线程调用走异步
   - 客户端 worker 同样按 `(mapId, mapRevision)` 缓存 staticGrid，避免每次寻路传一次大数组

4. **Vite 集成**：
   - `packages/client/vite.config.ts` 增加 worker 构建配置（worker format + sourcemap），确保 dev 模式与 build 模式一致

**产出**：
- 服务端长路径（如跨大图玩家寻路意图）不再卡 tick 间空闲
- 客户端手机端弱 CPU 上 64×64 以上地图寻路不再卡 UI 帧

**验证**：
- 新增 `pnpm --filter @mud/server smoke:worker-pathfinding-equivalence`：fixture 用 `benchmarks/pathfinding/fixtures/maps/yunlai_town.json`，断言 worker 路径与同步路径输出完全一致
- `benchmarks/pathfinding/run.sh` 加 worker 模式列：TS 同步 / TS worker / Rust（参考），输出 p50/p95/QPS
- 客户端：`pnpm verify:client` + 手动手机端冒烟（确认寻路点击到地块高亮的延迟未恶化）

**风险**：
- worker 启动 + 首次 staticGrid 传输有一次性开销，必须验证"首次寻路延迟"在可接受范围（< 100ms）
- 路径请求多并发时如果都打到同一个 worker，会回到串行；池化必须按任务粒度负载均衡

**回退**：关 `SERVER_PATHFINDING_WORKER_ENABLED` / 客户端 `?disablePathWorker=1` 调试参数即可。

### Phase 3 — FOV / 怪物派生状态并行化

**目标**：把 `bench-hot-paths.ts` 已识别为热路径的两块——FOV 与怪物派生状态——按"玩家分片 / 怪物分片"切到 `EncodingWorkerPool`。

**改造点**：

1. **FOV**：
   - `packages/server/src/runtime/instance/fov.helpers.ts:collectVisibleTileIndices` 改为同时支持同步与 worker 入口
   - 调用方 `world-runtime-instance-tick-orchestration.service.ts` 在每帧组合"实例下所有玩家"时，把 N 个玩家的 FOV 任务批量 submit，`Promise.all` 等齐后再进入下一阶段
   - FOV 输入：`{ blocksSightMask, width, height, originX, originY, radius }`，全部 plain；输出：`Uint32Array` 可见 tile index 列表（Transferable）

2. **怪物派生状态**：
   - `map-instance-monster-advancer.ts` 中"怪物 AI 决策（追击 / 巡逻 / 仇恨衰减 / 视线计算）"部分按 instance 分片
   - 决策输出"意图列表"（POJO）回主线程，主线程权威应用（移动、施法、攻击）

**产出**：
- tick 内 FOV / 怪物 AI 阶段从"全实例串行"变成"按 worker 数 fan-out / fan-in"

**验证**：
- 新增 `pnpm --filter @mud/server smoke:worker-fov-equivalence`：固定地图 + 固定玩家位置，断言 worker / 同步两种路径返回的可见 tile 集合**完全相等**
- `pnpm --filter @mud/server bench:hot-paths` 加入 worker 模式列
- `pnpm verify:building:perf`（如果 FOV 涉及房间检测）

**风险**：
- 怪物 AI 决策输出"意图"在主线程聚合时，如果两个 worker 同帧产出"互相影响的意图"（例：两只怪物同时决定走同一格），主线程仲裁逻辑必须明确（按 instanceId + monsterId 排序后串行应用）
- FOV 任务量太碎（每个玩家一个）时，postMessage 开销可能超过收益；需 bench 决定"每个 worker 一次处理 K 个玩家"的批量大小

**回退**：关 `SERVER_FOV_WORKER_ENABLED` / `SERVER_MONSTER_AI_WORKER_ENABLED`。

### Phase 4 — 多地图实例 tick 分片（最大架构变更）

**目标**：把 `WorldRuntimeInstanceTickOrchestrationService.advanceFrame` 中的"实例级独立子阶段"按 `instanceId` 哈希分片到 `InstanceWorkerPool`，每个 worker 持有自己分片实例的**镜像状态**，每帧主线程 broadcast tick start，各 worker 跑完返回"权威 mutation 集合"，主线程批量应用。

> 这是本计划最重的一步，建议在 Phase 1-3 全部稳定且取得性能收益**之后**再启动。

**改造点（粗粒度）**：

1. **状态镜像**：
   - 每个 instance worker 维护其分片实例的 read-only 镜像；主线程是唯一 writer，每帧把"上一帧 mutation diff"发给对应 worker
   - 镜像协议复用 `worldDelta` 的 patch 形态，不引入新数据格式

2. **分片策略**：
   - 默认 `instanceId` hash 到 `M = min(N_cpu - 2, 6)` 个 worker
   - GM 可通过 `runtime-gm-state.service.ts` 查询当前分片分布，运维期可触发 rebalance（仅在"无玩家在线 / 维护模式"窗口）

3. **可外移子阶段（仅这些进 worker）**：
   - 资源流动 / 阵法 tick / 建筑 tick（无跨实例依赖，纯实例本地状态机）
   - 怪物 AI 决策（已在 Phase 3 外移）
   - 自动战斗预选目标（候选目标筛选、距离比较、视线判定）

4. **不可外移（主线程保留）**：
   - 玩家 session 进出实例
   - 跨实例传送 / lease 切换 / fence
   - 战斗结算应用（伤害、死亡、掉落、经验、邮件触发）
   - 任何写持久化、写 outbox、写 durable_operation

5. **主线程聚合**：
   - 每帧顺序：`broadcast tick start` → `Promise.all(workers.advancePhase())` → 主线程顺序应用 mutation → `WorldSyncService.flushConnectedPlayers()`（已在 Phase 1 改）→ `flushTick()`

**产出**：
- 10000 实例的"资源流动 / 阵法 / 建筑 / 怪物 AI"由 6 个核分担，主线程只剩"玩家意图应用 / 同步 / 持久化触发"
- `WorldTickService` 跳帧率（`skippedFrameCount`）显著下降

**验证**：
- 新增 `pnpm --filter @mud/server smoke:worker-instance-tick-equivalence`：固定 fixture（10 个实例 + 50 只怪物 + 5 个玩家）跑 1000 帧，断言 worker 模式 / 同步模式产生的 `worldDelta` 序列**完全相等**
- 新增 `pnpm --filter @mud/server bench:instance-tick`：5000 玩家 / 10000 实例 fixture，对比单/多 worker 模式 tick 总耗时
- `pnpm verify:release:full`（持久化 + 协议 + 性能全验证）

**风险**：
- 这是最容易"看上去并行了但实际更慢"的阶段：postMessage 序列化 mutation diff 的开销可能 > 计算开销
- 若 mutation 形态没有压成 `TypedArray + index`，跨 worker 传 plain object 在 1Hz tick 高频下会被结构化克隆吃掉收益
- 需要先在 Phase 1-3 实测中确定 mutation 协议是否高效，再启动 Phase 4

**回退**：关 `SERVER_INSTANCE_WORKER_ENABLED`，回退到现有 `for-of advanceFrame` 路径。

### Phase 5 — 持久化 flush 序列化外移

**目标**：把 14 个 `runtime/world/worker/*.worker.ts` 中"序列化 + bigint 转换 + JSON.stringify raw_payload"部分外移到 `PersistenceWorkerPool`，主线程只保留"调用 `pool.query` 发 SQL + 处理结果"。

**改造点**：

1. **拆分 `flushPlayerDirtyDomains` / `flushInstanceDomains`**：
   - 阶段 A（主线程）：从权威态收集 dirty domains 的 plain snapshot
   - 阶段 B（worker）：把 snapshot 编译成 `{ sql, params }` 数组（含 bigint 转字符串、JSON.stringify raw_payload、column projection）
   - 阶段 C（主线程）：批量 `pool.query` 发出去 + 错误处理 + lease 校验 + `markPersisted`

2. **特性开关**：
   - `SERVER_PERSISTENCE_BUILD_WORKER_ENABLED=false`，默认关
   - 打开后按"player domain"维度灰度（先 ground-item，再 monster-runtime，最后 player-domain）

**产出**：
- 主线程在持久化阶段的 CPU 占用下降，tick 内若发生大批 dirty flush 不再阻塞下一 tick

**验证**：
- 新增 `pnpm --filter @mud/server smoke:worker-persistence-build-equivalence`：构造 100 个玩家 + 50 个实例的 dirty 集合，断言 worker / 主线程两条路径产生的 `{ sql, params }` 序列**完全相等**
- `pnpm verify:release:with-db`（带数据库验证持久化端到端正确性）
- `pnpm --filter @mud/server bench:persistence-flush`：对比开关关 / 开两种模式 5000 玩家全 dirty flush 总耗时

**风险**：
- snapshot 拷贝代价可能大于计算代价；需要 bench 决定哪些 domain 值得外移
- bigint 在 postMessage 结构化克隆中支持，但要避免重复 boxing → unboxing

**回退**：关开关。

### Phase 6 — 客户端 OffscreenCanvas / Web Worker

**目标**：把客户端 Canvas 2D 渲染、tile-sprite-cache 预生成、文字测量、catalog JSON 解析外移到 Web Worker。

**改造点**：

1. **OffscreenCanvas 渲染**：
   - 新增 `packages/client/src/workers/render.worker.ts`，主线程把 main canvas 通过 `transferControlToOffscreen()` 移交 worker
   - `packages/client/src/game-map/renderer/canvas-text-renderer-adapter.ts`、`packages/client/src/renderer/text.ts`、`tile-sprite-cache.ts` 改造为 worker 内执行
   - 主线程只负责：input event、相机指令、当前 patch payload → postMessage 给 worker；不再持有 2D context

2. **协议 decode worker**：
   - `packages/client/src/network/socket.ts` 高频 envelope decode 进 `decode.worker.ts`，输出 plain object → postMessage 主线程，主线程仅负责 store 写入

3. **Catalog 解析**：
   - `editor-catalog.generated.json` (487KB)、`item-sources.generated.json` (139KB)、`i18n.generated.ts` (452KB) 启动期解析 + 索引构建丢到 `bootstrap.worker.ts`

4. **降级**：
   - `OffscreenCanvas` 不支持的浏览器（旧 Safari）回退到主线程渲染
   - URL 调试参数 `?disableRenderWorker=1` 强制回退

**产出**：
- 手机端弱 CPU / 中端 Android 上首屏与稳定帧率显著改善
- 高频 envelope decode 不再让 UI 主线程掉帧

**验证**：
- `pnpm verify:client`
- 手动桌面 / iOS / 中端 Android 三端冒烟：渲染、寻路点击、面板切换体感不退化
- 性能基准：固定场景下 Chrome DevTools Performance 主线程 long task 数量减半以上

**风险**：
- OffscreenCanvas 兼容性 + 字体加载链路；postMessage 频率过高反而拖慢
- DOM UI 与 Canvas 同步显示元素时（例：选中地块的 hover 弹层）必须保持坐标一致

**回退**：URL debug 开关 + 主线程渲染分支保留。

### Phase 7 — 验证编排并行化（运维 / CI 加速）

**目标**：让 `scripts/release-*.js` 和 `tools/smoke-suite.ts` 把无依赖步骤改为并发 spawn。

**改造点**：

1. **release-\*.js**：把当前 `spawnSync` 顺序串行改为：依赖图 + `Promise.all + spawn`；步骤间 stdout/stderr 隔离，失败聚合到末尾报告
2. **smoke-suite**：互不依赖的 smoke 子套件并发跑；碰库的 smoke 走独立 schema 命名空间避免互相踩
3. **bench**：`bench-hot-paths` / `bench-combat` 各 case 并发跑

**产出**：
- 完整 `pnpm verify:release:full` 总耗时显著下降

**验证**：
- 串行 / 并行两种模式跑结果完全一致（output 排序后比对）
- CI 总时长指标记录到 `.runtime/verification-timings/`

**风险**：DB 资源竞争、端口竞争 → 必须每个并发 spawn 用独立 DB schema / 独立端口。

**回退**：保留串行模式作为默认，新增 `--parallel` 开关切换。

## 5. 风险与缓解

| 风险 | 缓解 |
|---|---|
| postMessage 结构化克隆开销 > 计算节省 | 每个 phase 上线前必须有 `bench-*-equivalence` + `bench-*-perf` 双基准；并行模式必须证明端到端总耗时下降 ≥ 30% |
| worker 异常导致玩家请求挂死 | 所有 `submit()` 必须有 `deadlineMs` 与同步 fallback；worker 健康检查由 `WorkerPoolMetricsService` 周期性拉，异常 worker 自动重启 |
| 协议字段漂移 | 每 phase 都有 `*-equivalence` smoke 断言 worker 路径输出与主线程路径**逐字节相等**；CI 强制运行 `pnpm audit:protocol` |
| 内存翻倍（worker 各持镜像状态） | 仅 Phase 4 涉及；启动期记录 RSS，bench 期间监控；如果 16GB / 5000 玩家场景下 RSS 翻倍超 80%，必须先压缩镜像协议 |
| NestJS DI 与 worker 边界混淆 | worker 入口禁止 `import` `@nestjs/core`；ESLint 规则 + `audit-runtime-template-spread` 风格的 `audit-worker-import-boundary` 守护 |
| 启动期 worker 拉起卡 readiness | worker pool init 非阻塞，readiness 仍按现有依赖（DB/Redis）判断；worker 失败自动降级而不是阻塞启动 |
| 灰度回退路径 | 每 phase 都有特性开关；GM 性能页一键关闭并立刻生效（不重启进程） |
| 单测 / smoke 在 worker 模式下不稳定 | smoke 默认走同步路径；专门的 `*-worker-*` smoke 才打开开关，避免测试用例因 worker 启动时间漂移 |

## 8. 不在范围（再次明确，避免范围漂移）

- **不**做 Node cluster / 多 Node 进程 / 多节点 socket.io adapter
- **不**做 Rust/N-API（pathfinding rust 基准只是对照参考）
- **不**改 1Hz tick / AOI / 协议形态 / 持久化 schema
- **不**重写战斗结算公式（Phase 3 只外移"决策"，不外移"结算应用"）
- **不**引入 Redis/Postgres 的 read replica（与并行化是两件事）
- **不**重写 NestJS bootstrap 模型为 worker-first
- **不**做 React UI 的 worker 化（DOM/React render 不是 CPU bound）
- **不**做 GPU / WebGL 渲染
