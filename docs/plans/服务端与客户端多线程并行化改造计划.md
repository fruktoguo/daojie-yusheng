# 服务端与客户端多线程并行化改造计划

> 落点：`packages/server`（主）+ `packages/shared`（协议契约 + Pathfinding 边界）+ `packages/client`（OffscreenCanvas / Web Worker）+ `scripts/`（验证编排并行化）
> 类型：默认多线程执行模型（不是可选灰度功能）
> 优先级：**P1**
> 阶段定位：从"基础设施 + 可选开关"过渡到"**默认多线程 + 故障降级**"
> 配套阅读：`AGENTS.md` §4 商业级 MMO 口径、§8 权威运行时红线、§9 网络同步红线、§12 性能红线
> 关联根修计划：`docs/plans/持久化刷盘并行化根修计划.md`

---

## 1. 背景与现状

### 1.1 已落地的多线程基础设施

`packages/server/src/concurrency/` 已经具备真实的 `worker_threads` 池：

- `EncodingWorkerPoolService` / `InstanceWorkerPoolService` / `PersistenceWorkerPoolService` 三类池
- `worker-pool.module.ts` 启动期 spawn / 退出期 terminate
- `WorkerPoolMetricsService` 指标接到 GM 性能页
- worker 入口 `concurrency/workers/{encoding,instance-advance,persistence-build}.worker.ts` 单独 emit 到 dist，不依赖 NestJS DI
- 客户端 `pathfinding.worker.ts` / `render.worker.ts` 已通过 Vite 构建并接入主流程

阶段化推进的 Phase 0-7 中：

- Phase 0-4、6、7 的"代码可运行性"全部通过
- Phase 5 的 worker 基础设施已就位，但 **worker 输出尚未成为写库输入**

### 1.2 现存问题：默认仍跑在主线程

虽然代码可运行，但生产实际**默认不走 worker**：

- `SERVER_WORKER_POOL_ENABLED` 等 6 个 env 默认 `false`
- `WorkerPoolToggleService` 把 GM runtime flag 当作"产品级开关"，允许在线上关闭整条 worker 路径
- 客户端保留 `?disablePathWorker=1` / `?disableRenderWorker=1` 等禁用入口
- 热路径服务都在做 `isEnabled()` 分支判断，增加无谓开销
- Phase 5 worker 预序列化不接入写库，是无效工作
- `scripts/release-*.js` 默认串行

直接后果：**多线程只在 smoke / bench 验证里触发**，生产仍是单核 CPU 跑全量负载。

### 1.3 路线修正

不再把多线程当作"可灰度的产品功能"，而是当作**默认执行模型**。
保留的"关闭路径"只剩一个：**单个 worker 任务故障时降级到主线程同步执行**——这是容错，不是产品配置。

## 2. 目标与不变量

### 2.1 主目标

1. **默认开启**：所有 worker pool 在进程启动期就 spawn，不依赖任何 env 开关
2. **移除产品级可选项**：删除全部 `SERVER_*_WORKER_ENABLED` env、GM toggle 关闭语义、客户端 `?disablePathWorker` / `?disableRenderWorker` 禁用入口
3. **热路径不再做 `isEnabled()` 分支**：调用方直接走异步 worker 入口，故障降级在 pool 内部透明处理
4. **修正 Phase 5**：让 worker 输出真正成为写库输入（write plan），而不是额外预序列化开销
5. **验证编排默认并行**：`scripts/release-*.js` / `tools/smoke-suite.ts` 默认并行
6. **真实收益证明**：bench 必须证明主线程 CPU 时间和单帧 tick 总耗时下降到目标范围

### 2.2 不变量（必须维持）

- **服务端权威不变**：worker 不持有 `MapInstanceRuntime` 真源、不持有 socket.io 连接、不直连 PG/Redis 池；worker 只做"接收 plain 输入 → 纯计算 → 返回 plain 输出"
- **协议字段不变**：envelope / panel / aux / threat / minimap 输出与现网完全一致
- **持久化真源不变**：PostgreSQL + Redis 仍是唯一持久化与在线态出口
- **1Hz tick 不变**：`WorldTickService` 自调度形态保留；fan-out 必须能在 tick interval 内 join 完成
- **断线重连 / lease / 顶号 / fence / outbox / durable_operation / 同步组包**等语义全部保留，仍由主线程串行处理
- **5000 玩家 / 10000 实例 / 8C16G / 30Mbps 单服口径不变**

### 2.3 故障降级语义（唯一允许的"关闭"）

不是产品开关，是容错：

- **任务级降级**：单个 `submit()` 在 worker 启动失败、postMessage 异常、deadline 超时时，由 pool 内部自动 fallback 到主线程同步路径执行该任务一次，记录指标
- **worker 级重启**：worker 异常退出后由 pool 自动重 spawn，不影响后续任务路由
- **进程级强制单核**：仅在调试 / 故障定位时由 ops 通过 `SERVER_WORKER_POOL_FORCE_SYNC=1` 临时禁用整池（线上默认不设；不允许写入生产配置模板）

## 3. 总体架构

### 3.1 三类 Worker 形态（不变）

```
┌──────────────────────────────────────────────────────────┐
│                Node.js 主线程（NestJS App）                │
│  WorldTickService / WorldRuntimeService / WorldSyncService │
│  socket.io / DatabasePool / RedisClient / Outbox           │
└──────────────────────────────────────────────────────────┘
       │                │                  │
       │ A. CPU pool    │ B. Instance pool │ C. Persistence pool
       ▼                ▼                  ▼
┌────────────────┐  ┌────────────────┐  ┌────────────────┐
│ EncodingWorker │  │ InstanceWorker │  │ FlushWorker    │
│ × N (默认启动) │  │ × M (默认启动) │  │ × K (默认启动) │
│                │  │                │  │                │
│ - protobuf     │  │ - monster AI   │  │ - write plan   │
│   encode       │  │   intent       │  │   构造         │
│ - A* path      │  │   proposals    │  │ - JSON.stringify│
│ - FOV          │  │                │  │ - SQL params   │
│ - fengshui     │  │                │  │                │
└────────────────┘  └────────────────┘  └────────────────┘
     (无状态)         (实例分片态)        (无状态)
```

- **A. CPU pool**：无状态纯函数池。并发度 = `min(N_cpu - 2, 6)`，主线程 round-robin 派发
- **B. Instance pool**：按 `instanceId` 哈希分片；只承担可在 tick 内完整算完且不需写权威态的子阶段；产出"操作意图列表"回主线程
- **C. Persistence pool**：无状态，吃 dirty domain 快照，输出 write plan（SQL + params）；主线程负责 `pool.query` + lease 校验

### 3.2 主线程 → Worker 协议（不变）

```typescript
interface WorkerTaskEnvelope<TPayload> {
  taskId: string;
  kind: 'envelope-encode' | 'pathfind' | 'fov' | 'instance-advance' | 'persistence-build';
  payload: TPayload;
  deadlineMs: number;
}

interface WorkerTaskResult<TResult> {
  taskId: string;
  ok: boolean;
  result?: TResult;
  errorMessage?: string;
  durationMs: number;
}
```

**强约束**：
- 入参出参必须是 plain object / `ArrayBuffer` / `TypedArray`，不传 class 实例、不传 `Map/Set`
- 大 buffer 用 `transferList` 移交，不复制
- 任何 worker 异常 → pool 内部 fallback 到主线程同步执行，对调用方透明
- 任务必须有 `deadlineMs`：超时自动降级并记录指标

## 8. 运维 Runbook

### 8.1 默认行为

服务端启动后自动 spawn 三类 worker pool，无需设置任何 env。GM 性能页 `perf.workerPool` 域展示实时指标。

### 8.2 指标监控

```json
{
  "encoding": { "totalSubmitted": 0, "totalCompleted": 0, "totalTimedOut": 0, "totalFailed": 0, "totalFallback": 0, "p50Ms": 0, "p95Ms": 0, "inFlight": 0, "activeWorkers": 4 },
  "instance": { "..." },
  "persistence": { "..." }
}
```

**关注指标**：
- `totalTimedOut` 持续增长 → worker 负载过高，考虑增加 worker 数
- `totalFailed` 持续增长 → worker 代码异常，检查日志堆栈
- `p95Ms` > 200 → 考虑减少单任务负载或增加 worker 数
- `inFlight` 持续 > poolSize → 任务积压
- `totalFallback` 持续增长 → worker 不健康，检查是否频繁重启

### 8.3 Worker 数量调整

| 环境变量 | 默认值 | 说明 |
|---|---|---|
| `SERVER_ENCODING_WORKER_COUNT` | `min(N_cpu - 2, 6)` | CPU pool worker 数 |
| `SERVER_INSTANCE_WORKER_COUNT` | `min(N_cpu - 2, 6)` | Instance pool worker 数 |
| `SERVER_PERSISTENCE_WORKER_COUNT` | `2` | Persistence pool worker 数 |

### 8.4 故障应急

**症状：worker 崩溃**
1. 该 worker 负责的任务自动 fallback 到主线程同步执行
2. Worker 在 1s 后自动重启
3. 检查日志中的错误堆栈

**症状：主线程 CPU 未下降**
1. 检查 GM 性能页 `activeWorkers > 0`
2. 检查 `totalFallback` 是否异常高（说明 worker 频繁失败）
3. 检查 worker 数量配置是否合理

**症状：需要临时禁用 worker（故障定位）**
1. 设置 `SERVER_WORKER_POOL_FORCE_SYNC=1` 并重启
2. 定位完成后**立即移除**该 env
3. 此 env 不允许写入生产配置模板
