## 4. 嫌疑链路与代码定位

> 以下三条链路都能解释"1.5M 匿名 Object + 高频业务标识字符串重复 N 千倍"现象。是否真泄漏需要进一步用 retainer 路径分析（脚本可扩展）确认；当前文件只列嫌疑、不动手修。

### 4.1 嫌疑 A：projector 每 tick 全量克隆玩家 panel 切片（短命对象高频晋升）

- 入口：`packages/server/src/network/world-projector.service.ts:91 createDeltaEnvelope`
- 链路：
  - 每 tick 1 Hz，对每个在线玩家调用一次
  - `capturePlayerState(player)` → `captureAttrPanelSlice(player)` → `buildAttrBonuses(player)` → `player.bonuses.map((entry) => cloneAttrBonus(entry))`（`packages/server/src/network/world-projector.helpers.ts:244`）
  - `cloneAttrBonus` 为每个 bonus 生成新 plain Object（`packages/server/src/network/projector-clone.ts:299`），含 `source / label / attrs / attrMode / stats / qiProjection / meta` 七个字段
  - 同时 `captureInventoryPanelSlice / captureEquipmentPanelSlice / captureTechniquePanelSlice / captureBuffPanelSlice` 各自全量 spread/clone 一遍
- 嫌疑机制：`cacheByPlayerId.set(playerId, current)` 每 tick 用同 key 覆盖旧 ProjectorState，旧的失去引用本应被 GC，但**短命对象在 1Hz 高频下被晋升到 old_space**，Mark-Compact 周期跟不上 → old_space 持续累积
- 还要注意：`buildAttrDelta` 内即使 `bonusesChanged === false` 也会先 `nextBonuses = buildAttrBonuses(player)` 全量克隆（`world-projector.helpers.ts:683`），克隆后再走比较，**比较结果不变也会留下 N 个临时 Object**
- 数据吻合：`runtime:realm_state` 8618 / `runtime:technique_aggregate` 5671 = 单玩家 1.5 分钟（90 秒）累积，平均 60~95 个/秒，恰好与每 tick 全量克隆一份完整 bonuses 数组的速率级匹配（玩家 bonuses 长度 ~ 50~100）

### 4.2 嫌疑 B：每 tick 全量克隆世界态条目（怪物 + 地块对象）

- 入口：`packages/server/src/network/world-projector.helpers.ts:467`（捕获 worldState）
- 链路：
  - 每 tick 把 `view.localMonsters`（在 `tower:tongtian:layer:32` 这张地图的所有怪物快照）`.map` 出 N 个 `ProjectedMonsterEntry` 字面量
  - `view.localNpcs / localPortals / localGroundPiles / localContainers / localBuildings / localFormations` 同样每 tick 全量 map
- 嫌疑机制：每 tick 临时 Object 数 ≈ 当前地图怪物数 + npc + portals + 建筑 + 地面掉落，按 1430 怪物全在同一张地图算上限 ≈ 1500/tick，90 秒 × 1500 = 13.5 万，**单这一项无法解释 150 万 Object**
- 但若：
  - `cacheByPlayerId` 因为 worldRevision 没变命中复用 `previous` 时跳过 `captureWorldState`（`world-projector.service.ts:111` 三元分支），但**仍然每 tick `combineProjectorState(currentWorld, capturePlayerState(player))` 新建一个外层对象**，旧 ProjectorState 在 GC 前还有引用持有（cacheByPlayerId entry）
  - `worldRevision` 频繁变化（怪物移动、tile damage、aura 变化）会让 captureWorldState 每 tick 跑一次，1430 怪物 × 90 tick ≈ 12.87 万 Object
- 数据吻合：`yunlai_town` 19166 + `public:yunlai_town` 15292 ≈ 3.4 万份 → 与 captureWorldState 每 tick 把所有 portal target 名字塞进 ProjectedPortalEntry 量级匹配（一张大地图里 portal 数量有限，但 portal mid 在 cache 里被 pin 住乘以 tick 数）

### 4.3 嫌疑 C：RuntimeEventBusService playerQueues / instanceQueues 永不清理（**已有计划 H1 提前确认**）

- 上游：`docs/plans/内存问题修复计划.md` H1 条目（`runtime/event-bus/runtime-event-bus.service.ts:75/78/574-582`）
- 与本次现场吻合点：
  - 1430 个 bot 每个都对应一个 `instanceQueues` 条目（bot 属于哪些实例就有哪些 instance queue）
  - 每个 PlayerEventQueue 自带 `panelPatches:Map / activeJobs:Map / techniquePanelDirty:Set / feedback:[]`
  - `discardPlayer(playerId) / discardInstance(instanceId)` 在生产路径未被调用 → bot 退出 / 实例卸载时 entry 永久残留
- 但本次 OOM 是 **单玩家 1.5 分钟**就触发，说明嫌疑 A/B 比 H1 增长更激进。H1 是**累积型慢泄漏**（5000 玩家高频流动场景下显著），本次现场更像是 **每 tick 高频分配 + 没及时回收**

### 4.4 嫌疑 D：dev 模式下 nest 模块源码缓存大头

- stringTop 中前 30 全部是 `"use strict";\nvar __decorate ...`（每条 100~600 KB）
- 单 `MapInstanceRuntime` 模块源码 605 KB，`WorldGateway` 144 KB，`WorldRuntimeFormationService` 137 KB...
- 这些是 dev `tsc --watch` + Nest cjs require cache 持有的模块源码字符串，**生产 docker 镜像里也存在但只 require 一次**，不算泄漏
- 但叠加在 `--max-old-space-size=512MB` 默认限制上，会显著挤占可用堆，导致**真实 GC 余地变小**，把上面 4.1/4.2 的高频分配压力暴露成 OOM
- 性质：dev 环境基线高，不是 bug，但是放大器

### 4.5 嫌疑 E：跨实例 view 拼接 8 次 concat + 8 次全量 map（嫌疑 B 的放大版）

- 入口：`packages/server/src/runtime/world/query/world-runtime-player-view-query.service.ts:225-255 buildOverlayedView`
- 链路：玩家在带 overlay parent 的实例（典型如 `tower:tongtian:layer:32` 这种叠加在某张主图上的塔），每 tick `buildPlayerView` 都会执行：
  ```ts
  return {
      ...view,
      visiblePlayers: view.visiblePlayers.concat(parentInstance.collectVisiblePlayers(...).map(project)),
      localMonsters:  view.localMonsters.concat(parentInstance.collectLocalMonsters(...).map(project)),
      localNpcs:      view.localNpcs.concat(parentInstance.collectLocalNpcs(...).map(project)),
      localPortals:   view.localPortals.concat(parentInstance.collectLocalPortals(...).map(project)),
      localLandmarks: view.localLandmarks.concat(parentInstance.collectLocalLandmarks(...).map(project)),
      localContainers: view.localContainers.concat(parentInstance.collectLocalContainers(...).map(project)),
      localGroundPiles: view.localGroundPiles.concat(parentInstance.collectLocalGroundPiles(...).map(project)),
      localBuildings: view.localBuildings.concat(parentInstance.collectLocalBuildings(...).map(project)),
  };
  ```
- 每 tick 一次玩家投影 = **至少 8 个新数组 + 1 个新外层 view 对象 + 父实例视野内每类对象数 N 个临时 plain Object**
- 嫌疑机制：父实例如果是规模大的主图（比如 `yunlai_town`），其 portals/landmarks/buildings/containers 数量数十至百级，每 tick 都按视野 collect 一遍并构造一份 ProjectedXxxLike 数组
- 数据吻合：`yunlai_town` 在堆里出现 19166 次 + `public:yunlai_town` 15292 次 = 34458 次。1 玩家 90 tick × 父图视野内对象数（含 portal target 标识 / 实例 ID 标识）≈ 与该量级匹配
- 这是嫌疑 B（每 tick 全量 map 怪物/portal/npc）的**塔类玩家叠加场景**版本，量级会被父图规模放大

### 4.6 嫌疑 F：collectLocalMonsters 每 tick 构造怪物视野数组

- 入口：`packages/server/src/runtime/instance/map-instance.runtime.ts:4702 collectLocalMonsters`
- 链路：每次玩家视野查询都 `monsters.push({ runtimeId, monsterId, name, char, color, tier, x, y, hp, maxHp, qi, maxQi })` 构造**新的 13 字段 plain Object**，然后 `monsters.sort(compareLocalMonsters)`
- 在每个 tick 的 `buildPlayerView` 里被调一次（或经 4.5 在 overlay 链路里被调两次：当前实例 + 父实例）
- 嫌疑机制：单玩家视野内若有 50~100 个可见怪物，每 tick 就 50~100 个临时 13 字段 Object；90 tick × 100 = 9000 个。和堆里 `Object` 数量级中"匿名 Object 字面量"那一档（150 万）比起来不算大头，但是**短命对象高频晋升**的典型样本之一

### 4.7 嫌疑 G：怪物 advancer 内部纯解构用临时 Object

- 入口：`packages/server/src/runtime/instance/map-instance-monster-advancer.ts:179`
  ```ts
  const { aggroTargetPlayerId, lastSeenTick, lastSeenX, lastSeenY } = {
      aggroTargetPlayerId: monster.aggroTargetPlayerId,
      lastSeenTick: monster.lastSeenTargetTick,
      lastSeenX: monster.lastSeenTargetX,
      lastSeenY: monster.lastSeenTargetY,
  };
  ```
- 这是用临时字面量包一层再解构，**反而强制创建一个 4 字段 Object**，本来直接读 `monster.xxx` 就行
- 同文件还有多处 `return { x, y }`、`candidates.push({ x, y, facing })` 类的临时小 Object，advanceMonsters 每 tick 对**每只活怪物**跑一次
- 量级估算：1430 怪物 × 90 tick × 单只 1~5 个临时 Object = 12.87 万 ~ 64 万临时 Object
- 性质：**不是泄漏**，是热路径过度分配。短命对象走 young space，本应快速回收，但分配速率高导致频繁晋升

### 4.8 嫌疑 H：MapPersistenceFlushService 高频 flush 触发降级

- 现场日志（同一进程 OOM 前 53 秒）：
  ```
  17:49:22  WARN [MapPersistenceFlushService] 地图最终一致刷盘触发降级退避：
              durationMs=241 thresholdMs=100 backoffMs=5000
              dirtyInstanceCount=6 persistedInstanceCount=6 skippedInstanceCount=0
              domainCounts=fengshui:2,tile_damage:2,tile_resource:5
  ```
- 1 玩家 + 1430 怪物 + 1.5 分钟内 6 个实例同时变脏；flush 走 PG，单次 241 ms 超阈值 100 ms，强制 backoff 5 s
- 嫌疑机制：flush 期间需要构造一份 dirty 域的快照（aura/tile_damage/tile_resource）和 SQL payload。241 ms 期间持有完整 dirty 实例数据 + outbox row 副本，期间触发的下一轮 flush 可能在排队
- 性质：暂未直接证实贡献堆增长，但和 4.1/4.2 的 short-lived 高频分配叠加，会让 V8 在 OOM 临界点更脆弱

### 4.11 嫌疑 K：PlayerCountersPersistenceService 嵌套 Map 按玩家累积

- 入口：`packages/server/src/persistence/player-counters-persistence.service.ts:19-20`
  ```ts
  private readonly cache = new Map<string, Map<string, number>>();
  private readonly pendingWrites = new Map<string, Promise<void>>();
  ```

- 链路：每个玩家一个 inner Map 持有 N 个计数器键值；服务全文搜索只有 `cache.clear()`（整体清空）和 `cache.set(pid, map)`，**没有按玩家 delete 的路径**

- 性质：**真泄漏**（按玩家累积，不清理，重启或 `cache.clear()` 才回收），与 H1 同类

### 4.12 嫌疑 L：runtime/persistence 部分缓存 Map 整体重赋值（短命 Map 高频替换）

- 例：`runtime/player/leaderboard-runtime.service.ts:114 / 180`
  ```ts
  this.cachedLeaderboardSnapshotsByPlayerId = new Map(snapshots.map((snapshot) => [snapshot.playerId, snapshot]));
  ```
- 每次刷新整个 Map 重赋值（不是 `clear()` + 增量写）
- 旧 Map 失去引用后理论上 GC，但**频繁创建大 Map 又丢弃**会贡献短命对象高频晋升
- 性质：**不是泄漏**，但与 4.1/4.2/4.5 一起加剧 V8 GC 跟不上的现象
