# 玩家分域整表清空事故修复

> 落点：`packages/server/src/persistence` + `packages/server/src/network` + `packages/server/src/tools/audit`
> 类型：生产事故根因修复（玩家资产被静默清空）
> 优先级：**P0（紧急）**
> 状态：方案设计完成，待实施

## 1. 事故背景

### 1.1 现象

服务器**非自然重启**后，**多名玩家的背包数据被清空**，PG `player_inventory_item` 表中相关玩家的所有 row 全部消失。受影响玩家的 wallet、equipment、market_storage 等关键资产域同样可能受损（共享同一类 fallback 路径）。

### 1.2 触发模式

非自然重启 = 主进程被 SIGKILL / OOM / 节点宕机 / 容器超过 `stop_grace_period` 被强制 kill。重启完成后大量玩家同时 reconnect，bootstrap 高并发瞬间打到 PG 上，触发临时性读失败窗口（连接池打满 / advisory lock 抢占冲突 / 短期网络抖动）。事故就发生在这个窗口内。

### 1.3 此前修复的相关线

P0 三项（`markPersisted` 精确清理、lease 失效路径、durable buyNow/sellNow）+ P1 一项（flush 间隔 5s→1.5s）已经在另一条线上修过，但**这些修复都是"减少丢失最近 mutation"，不解决"PG 已落库数据被反向清空"**。本次事故是另一个独立的、更严重的 bug。

## 2. 根因分析

事故由两条 fallback 路径**串联**触发：入口路径制造空 starter snapshot，贯穿路径把空 snapshot 翻译成"清空所有 PG 行"。

### 2.1 路径 1：silent-rebirth fallback（入口）

文件：`packages/server/src/network/world-player-snapshot.service.ts:135` `ensureNativeStarterSnapshot`

```ts
async ensureNativeStarterSnapshot(playerId: string): Promise<NativeStarterSnapshotResult> {
  // ...
  const existingSnapshot = await this.playerDomainPersistenceService!.loadProjectedSnapshot(
    normalizedPlayerId,
    (targetPlayerId) => this.playerRuntimeService.buildStarterPersistenceSnapshot(targetPlayerId),
  ).catch((error: unknown) => {
    this.logger.warn(`原生新手分域快照读取失败：playerId=${normalizedPlayerId} error=...`);
    return null;  // ← 致命:把任何 PG 读失败都吞成"该玩家不存在"
  });
  if (existingSnapshot) {
    return { ok: true, seeded: false, snapshot: existingSnapshot, persistedSource: 'native' };
  }

  // ↓ 老玩家在 PG 读失败时会 fall through 到这里
  const starterSnapshot = this.playerRuntimeService.buildStarterPersistenceSnapshot(normalizedPlayerId);
  // ...
  await this.playerDomainPersistenceService!.savePlayerSnapshotProjectionDomains(
    normalizedPlayerId,
    starterSnapshot,                       // ← 全新空玩家快照
    NATIVE_STARTER_PROJECTION_DOMAINS,     // ← 包含 inventory/wallet/equipment 等所有关键资产域
  );
}
```

`NATIVE_STARTER_PROJECTION_DOMAINS`（`world-player-snapshot.service.ts:65-87`）包含：

```
world_anchor, position_checkpoint, vitals, progression, attr,
wallet, market_storage, inventory, map_unlock, equipment,
technique, body_training, buff, quest, combat_pref,
auto_battle_skill, auto_use_item_rule, profession,
alchemy_preset, active_job, enhancement_record, logbook
```

### 2.2 路径 2：replace cleanup DELETE（贯穿）

文件：`packages/server/src/persistence/player-domain-persistence.service.ts`

至少 7 个分域 replace 函数末尾都有相同形态的"基于全快照对账的 cleanup DELETE"，**这段 SQL 不在 `if (rows.length > 0)` 块内**：

| 函数 | 行号 | 写入表 | 影响 |
|---|---|---|---|
| `replacePlayerInventoryItems` | 3326 | `player_inventory_item` | 整玩家背包清空 |
| `replacePlayerWalletRows` | 3447 | `player_wallet` | 整玩家钱包清空（钱归零） |
| `replacePlayerMarketStorageItems` | 3591 | `player_market_storage_item` | 整玩家坊市仓清空 |
| `replacePlayerEquipmentSlots` | 3754 | `player_equipment_slot` | 整玩家装备清空 |
| `replacePlayerTechniqueStates` | 3844 | `player_technique_state` | 整玩家功法清空 |
| `replacePlayerPersistentBuffStates` | 3909 | `player_persistent_buff_state` | 整玩家长效 buff 清空 |
| `replacePlayerQuestProgressRows` | 3986 | `player_quest_progress` | 整玩家任务进度清空 |

典型代码形态：

```ts
async function replacePlayerInventoryItems(client, playerId, items) {
  const rows = [/* ...处理 items... */];

  if (rows.length > 0) {
    // 1) DELETE slot 冲突的旧 row
    // 2) INSERT/UPSERT 新 row
  }

  // ↓ 这段无条件执行,即使 rows=[]
  await client.query(`
    WITH incoming AS (
      SELECT item_instance_id FROM jsonb_to_recordset($2::jsonb) AS entry(item_instance_id varchar(180))
    )
    DELETE FROM player_inventory_item target
    WHERE target.player_id = $1
      AND NOT EXISTS (SELECT 1 FROM incoming WHERE incoming.item_instance_id = target.item_instance_id)
  `, [playerId, JSON.stringify(rows.map(({ item_instance_id }) => ({ item_instance_id })))]);
  // rows=[] → incoming jsonb 为空 [] → NOT EXISTS 永真 → 整玩家所有 row 被 DELETE
}
```

### 2.3 audit 层为什么没拦住

`packages/server/src/tools/audit/production-boundary-audit.ts:429-471` 早就有 forbidden 规则：

```ts
{
  id: "persistence.snapshot_rewrite.player_domain_inventory",
  description: "禁止玩家分域快照回退为 player_inventory_item 整玩家 DELETE 后全量重插",
  pattern: "DELETE FROM ${PLAYER_INVENTORY_ITEM_TABLE} WHERE player_id = $1",
  forbidden: true,
  exclude: [" AND "],
}
```

audit 当时假设的是显式 `DELETE FROM ... WHERE player_id = $1` 这种字面形态，因此 exclude 了 `WHERE player_id = $1 AND ...` 来允许"带条件的安全删除"。

**但当前代码用的形态是 `WHERE player_id = $1 AND NOT EXISTS (SELECT 1 FROM <incoming 空 jsonb>)`**，字符串匹配上有 `AND` 因此被 audit 放行；可空 incoming 让"AND" 后面的条件**等价于永真**，效果就是无条件整玩家 DELETE。audit 的字面 pattern 没法识别这种**语义级整玩家清空**。

### 2.4 触发链完整路径

```
非自然重启
  ↓
玩家批量 reconnect → bootstrap 高并发
  ↓
loadPlayerSnapshotResult / loadProjectedSnapshot 调用 PG
  ↓
PG 短时压力（连接池满 / advisory lock 抢占 / 网络抖动）
  ↓
某些玩家的 query throw（pg "Connection terminated" / lock 超时）
  ↓
上层 loadPlayerSnapshotResult 返回 null（因为各种 catch）
  ↓
fall through 到 ensureNativeStarterSnapshot
  ↓ ← 入口路径开始
内部 loadProjectedSnapshot 第二次尝试，仍然失败
  ↓
.catch(err => null) 静默吞掉
  ↓
existingSnapshot === null → 进入 starter 写入分支
  ↓
buildStarterPersistenceSnapshot → 空 player → snapshot 中 inventory.items=[], wallet.balances=[], equipment.slots=[] 等
  ↓
savePlayerSnapshotProjectionDomains(playerId, 空 snapshot, NATIVE_STARTER_PROJECTION_DOMAINS)
  ↓ ← 贯穿路径开始
分别调 replacePlayerInventoryItems(client, playerId, []), replacePlayerWalletRows(client, playerId, []), ...
  ↓
每个 replace 函数末尾的 cleanup DELETE 在 incoming=[] 时 NOT EXISTS 永真
  ↓
玩家所有 inventory + wallet + equipment + market_storage + technique + buff + quest 被 DELETE
  ↓
事务 COMMIT → PG 落库 → 玩家所有资产数据消失
```

## 3. 修复方案（三层防御）

每一层都能独立挡住事故，三层叠加构成纵深防御。

### 3.1 第一层：入口阻断（根治）

文件：`packages/server/src/network/world-player-snapshot.service.ts`

把 `ensureNativeStarterSnapshot` 中第一次 `loadProjectedSnapshot` 的 `.catch(err => null)` 删掉，改为**让 throw 冒泡到 ensureNativeStarterSnapshot 的整体 try/catch**，PG 读失败时返回明确的失败 result：

```ts
async ensureNativeStarterSnapshot(playerId: string): Promise<NativeStarterSnapshotResult> {
  // ... validate ...

  let existingSnapshot: PersistedPlayerSnapshot | null;
  try {
    existingSnapshot = await this.playerDomainPersistenceService!.loadProjectedSnapshot(
      normalizedPlayerId,
      (targetPlayerId) => this.playerRuntimeService.buildStarterPersistenceSnapshot(targetPlayerId),
    );
  } catch (error: unknown) {
    // 关键：不再 swallow 成 null。读失败时拒绝继续，绝不允许写 starter 覆盖现有玩家数据。
    this.logger.error(
      `原生新手分域快照读取失败，拒绝写入 starter 以避免覆盖现有玩家：playerId=${normalizedPlayerId} error=${error instanceof Error ? error.message : String(error)}`,
    );
    return {
      ok: false,
      failureStage: 'native_snapshot_recovery_load_failed',
    };
  }

  if (existingSnapshot) {
    return { ok: true, seeded: false, snapshot: existingSnapshot, persistedSource: 'native' };
  }

  // 仅在 PG 明确返回"该玩家无任何分域 row"时才进入 starter 写入分支
  // ...
}
```

**副作用**：服务器重启后短暂高负载时，部分玩家登录会失败，需要重试。这远好于"资产被清空"。客户端可以加 backoff 重试 + 友好提示。

### 3.2 第二层：watermark invariant guard（纵深）

同一文件 `ensureNativeStarterSnapshot` 在写 starter 之前，**显式查 `player_recovery_watermark` 表**（这张表只在玩家有过任何分域 save 时才有 row）：

```ts
// existingSnapshot === null 后，写 starter 之前
const hasExistingWatermark = await this.playerDomainPersistenceService!.hasRecoveryWatermark(normalizedPlayerId);
if (hasExistingWatermark) {
  this.logger.error(
    `拒绝用 starter snapshot 覆盖已有 watermark 的老玩家：playerId=${normalizedPlayerId}`,
  );
  return {
    ok: false,
    failureStage: 'native_snapshot_recovery_existing_watermark_block',
  };
}
```

需要在 `PlayerDomainPersistenceService` 上加一个公开方法 `hasRecoveryWatermark(playerId): Promise<boolean>`，单 SELECT 查询，无副作用。

**副作用**：无。纯防御。新玩家不会有 watermark，不影响。

### 3.3 第三层：SQL 层无差别防御（最终兜底）

文件：`packages/server/src/persistence/player-domain-persistence.service.ts`

7 个 replace 函数末尾的 cleanup DELETE 之前加 sanity check：**当传入 incoming 为空数组，但 PG 中该玩家在该域有 N>0 行时，throw 错误让 withTransaction 整体 rollback**。

封装成一个 helper：

```ts
async function refuseEmptyOverwriteIfRowsExist(
  client: PoolClient,
  tableName: string,
  playerId: string,
  incomingCount: number,
  domainTag: string,
): Promise<void> {
  if (incomingCount > 0) {
    return; // 有 incoming，正常 cleanup 路径
  }
  // incoming 为空，先确认 PG 是不是真的也是空（合法零状态）
  const result = await client.query(
    `SELECT 1 FROM ${tableName} WHERE player_id = $1 LIMIT 1`,
    [playerId],
  );
  if ((result.rowCount ?? 0) > 0) {
    throw new Error(
      `replace_${domainTag}_refused_empty_overwrite:playerId=${playerId} table=${tableName}`,
    );
  }
}
```

然后在每个 replace 函数末尾的 cleanup DELETE 之前调一次：

```ts
// replacePlayerInventoryItems
await refuseEmptyOverwriteIfRowsExist(client, PLAYER_INVENTORY_ITEM_TABLE, playerId, rows.length, 'inventory');
await client.query(`DELETE FROM ... NOT EXISTS ...`);
```

应用范围（7 处）：
- `replacePlayerInventoryItems` → tag `inventory`
- `replacePlayerWalletRows` → tag `wallet`
- `replacePlayerMarketStorageItems` 内的 `prunePlayerMarketStorageStaleSlots` 调用点 → tag `market_storage`
- `replacePlayerEquipmentSlots` → tag `equipment`
- `replacePlayerTechniqueStates`（用 `prunePlayerRowsBySnapshotKeys`）→ tag `technique`
- `replacePlayerPersistentBuffStates`（用 `prunePlayerRowsBySnapshotKeys`）→ tag `buff`
- `replacePlayerQuestProgressRows` → tag `quest`

**副作用评估**：
- 玩家正常游戏中 inventory/wallet/equipment/market_storage 不会从有变成全空；至少有起步装备、初始铜钱、新手物品。如果出现"全部都被卖光"的边界场景，需要走显式专门 API，不能通过整快照 replace 触发。
- Technique/Buff/Quest 同样：玩家不会一次清光所有功法/buff/quest。
- 已经搜过 `player-domain-persistence-smoke.ts` 的合法用例，**没有空数组覆盖整玩家的合法测试**；现有的"逐步减少"测试用例是 [A, B] → [A]（B 是 stale）这种，不会触及。

### 3.4 audit 层补强

文件：`packages/server/src/tools/audit/production-boundary-audit.ts`

在现有 7 条 forbidden 规则旁边新增 7 条匹配 `WHERE player_id = $1\n.*AND NOT EXISTS` 形态的 pattern。这种 SQL 形态无论 incoming 是不是空，本质都是"基于整快照对账清空"，应该一概禁止：

```ts
{
  id: "persistence.snapshot_rewrite.player_domain_inventory_not_exists",
  description: "禁止玩家分域用 NOT EXISTS 形态做整快照 cleanup（incoming 为空时等价于整玩家 DELETE）",
  file: "packages/server/src/persistence/player-domain-persistence.service.ts",
  pattern: "WHERE target.player_id = $1\n        AND NOT EXISTS",
  forbidden: false,  // 当前实现就是这个形态，先不打开 forbidden，仅作 audit 标记，后续重构后转 forbidden
  // 等三层防御实施 + 重构验证完后改成 forbidden
}
```

第三层防御本质上是"加 sanity guard"，没有重写 SQL 语句，audit pattern 仍会匹配到这些行；先不开 forbidden，等后续重构（例如把 cleanup 改成基于 dirty-diff 的精确 delete）后再开。

## 5. 玩家数据补偿（独立工作流）

修复只能阻止后续事故。**已经丢失资产的玩家需要补偿**，是单独的运维工作：

1. 通过 `asset_audit_log` 表回查事故时间窗内 `delete` 类型的事件，定位受影响玩家与事件
2. 评估每个玩家在事故前最后一次 backup 的资产快照
3. 通过 GM 工具 `native-gm-player.service` 或邮件附件补回
4. 给受影响玩家发系统通知 + 适当慰问

这个补偿工作不在本计划范围内，由运维人员通过 GM 工具完成。

## 6. 风险与回退

### 6.1 实施风险

- **第一层副作用**：服务器重启后短时间内 PG 高负载，部分玩家登录会失败。比"资产清空"好得多，但需要在客户端加 backoff 重试与友好提示。
- **第二层副作用**：无。
- **第三层副作用**：如果有未发现的合法"replace*(playerId, [])"调用路径，会在 with-db smoke 中暴露出来。修复方案前已经全文搜索过，没有发现合法路径。

### 6.2 回退路径

每一层独立修改、独立 commit。如果某一层在生产环境出问题，可以**单独 revert 那一层**而不影响其他层：
- 第一层 + 第二层在 `world-player-snapshot.service.ts` 一个文件，可独立 revert
- 第三层在 `player-domain-persistence.service.ts`，可独立 revert
- audit 层在 `production-boundary-audit.ts`，可独立 revert

回退第三层最危险（最后兜底层失效），建议**至少保留第三层**。
