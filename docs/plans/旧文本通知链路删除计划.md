# 旧文本通知链路删除计划

## 背景

当前服务端通知消息（`queuePlayerNotice`）采用纯文本拼接方式，由服务端生成完整中文句子后发送给客户端直接显示。战斗系统已部分迁移到结构化数据（`CombatNoticePayload`），但仍保留旧文本 `text` 字段作为 fallback。其余系统（强化、炼丹、掉落、任务、宗门、导航等）全部仍为纯文本。

**问题：**
- 服务端硬编码中文文本，无法支持多语言
- 客户端无法对消息做胶囊/高亮等富文本渲染
- 文本模板散落在各服务中，维护成本高
- 战斗系统同时维护 `text` + `combat` 两套，冗余

## 目标

1. 所有通知消息改为服务端只发结构化数据，客户端负责拼接和渲染
2. 文本模板统一收入客户端中文语言包（`i18n.generated.ts`），支持字符串内插
3. 客户端按消息类型使用胶囊混合渲染（物品名、等级、数值等关键信息用胶囊样式）
4. 删除服务端所有硬编码中文文本拼接
5. 删除战斗消息的旧文本 fallback 链路

## 当前状态盘点

### 战斗系统（部分完成）

| 链路 | 状态 |
|------|------|
| 普攻/技能命中结算 | ✅ 已有 `combat` 结构化字段，客户端优先使用 |
| 旧文本 `text` 字段 | ❌ 仍在生成，未删除 |
| 客户端旧文本正则解析 fallback | ❌ 仍保留（`parseCombatDamageSegment`、`appendCombatLineContent`） |
| 击杀、煞气、神魂等附加消息 | ❌ 纯文本 |
| 吟唱打断/过期/取消 | ❌ 纯文本 |
| 自动战斗提示 | ❌ 纯文本 |

### 强化系统（未迁移）

| 消息 | 当前格式 |
|------|----------|
| 强化成功 | `{物品名} 强化成功，已提升至 +{等级}。` |
| 强化失败（有保护） | `{物品名} 强化失败，保护生效，降为 +{等级}。` |

| 取消/中断/资源不足 | 纯文本长句 |

### 其他系统（未迁移）

- 炼丹：成丹、失败、取消
- 掉落/拾取：获得物品、背包满
- 任务：接取、完成、进度
- 宗门：建宗、大阵、山门
- 导航：无法到达、目标过远
- 装备：穿戴、卸下

## 迁移方案

### 协议结构

```typescript
/** 结构化通知载荷：服务端只发数据，客户端负责渲染。 */
interface StructuredNoticePayload {
  /** 消息模板 key，对应语言包条目。 */
  key: string;
  /** 内插变量，客户端按 key 从语言包取模板后填充。 */
  vars?: Record<string, string | number>;
  /** 需要胶囊渲染的字段名列表（对应 vars 中的 key）。 */
  pills?: string[];
  /** 标签 badge 列表（如"成功"、"失败"、"保护"）。 */
  badges?: string[];
}
```

### 服务端改造

- `queuePlayerNotice` 签名扩展，支持传入 `StructuredNoticePayload`
- 逐系统替换：从纯文本改为 `{ key, vars, pills, badges }`
- 过渡期保留 `text` 字段供旧客户端 fallback，最终删除

### 客户端改造

- 语言包新增所有通知模板条目，支持 `{变量名}` 内插语法
- `buildLineFragment` 扩展：检测结构化载荷后走胶囊渲染路径
- 胶囊渲染复用现有 `chat-target-pill`、`chat-damage-pill`、`chat-combat-badge` 样式
- 删除旧文本正则解析 fallback（`parseCombatDamageSegment` 等）

### 语言包示例

```typescript
// i18n.generated.ts
"notice.enhancement.success": "{itemName} 强化成功，已提升至 +{level}。",
"notice.enhancement.fail-protected": "{itemName} 强化失败，保护生效，降为 +{level}。",
"notice.enhancement.fail-reset": "{itemName} 强化失败，已归零为 +0。",
"notice.enhancement.advance": "{itemName} 已调整为 +{current}，继续冲击 +{target}。",
"notice.enhancement.cancelled": "你停止了 {itemName} 的强化，已投入的本阶材料不会退回。",
"notice.enhancement.resource-exhausted": "{itemName} 当前已到 +{level}，后续资源不足，队列已停止。",
"notice.combat.killed": "{monsterName} 被你斩杀",
"notice.loot.obtained": "获得 {itemLabel}",
"notice.loot.bag-full": "{itemLabel} 掉落在 ({x}, {y}) 的地面上，但你的背包已满。",
```

## 迁移顺序

1. **P0 - 战斗旧文本清理**：删除 `text` 字段生成、删除客户端正则 fallback
2. **P1 - 强化系统**：结构化 + 胶囊渲染
3. **P2 - 炼丹/锻造系统**：同上
4. **P3 - 掉落/拾取**：结构化 + 物品胶囊
5. **P4 - 其余系统**：任务、宗门、导航、装备、修炼、市场等

## 验证

- 每个系统迁移后，确认聊天面板渲染正确（胶囊样式、颜色、tooltip）
- 确认浅色/深色/手机端三模式下胶囊可读
- 确认历史消息缓存兼容（旧缓存无结构化字段时 fallback 到纯文本显示）
- 最终阶段删除所有 fallback 后，确认无回归
