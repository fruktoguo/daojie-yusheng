# 统一对象管理器改造计划

> 配套文档：[对象克隆全面扫描计划](./对象克隆全面扫描计划.md)。
>
> 本计划解决"扫描计划"暴露出的根因——模板表存在但未被严格收口，运行态对象在散落点 spread 模板字段。目标是把"模板源 / 实例工厂 / 运行态字段"三者强约束在统一管理器边界内，并在最热路径上为 SoA / TypedArray 留出可选演进空间。

---

## 1. 目标与边界

### 目标

- 模板数据按域归集到独立 Registry，启动期一次解析、运行期只读共享、`Object.freeze` 兜底
- 运行态实例只能通过 Registry 的 `createInstance` / `hydrate` 工厂出口产出，own 字段限定为"实例可变 + 索引"白名单
- 模板字段（name / desc / range / formula / equipAttrs 等）一律走 prototype 链或按 id 查询，不进运行态 own / 不进持久化 jsonb
- 提供一组 audit / smoke 守护，CI 失败而不是靠人盯 review

### 不在范围

- 不新增玩法、内容、协议字段、UI 入口
- 不改 1Hz tick 频率、AOI 半径、socket 协议形态
- 不引入新的运行进程 / Worker / 数据库
- 不立刻重写 monster runtime 为 ECS（仅留接口位，Step 6 可选）
- 不动 `react-ui` 与 `config-editor` 的内容生产链路

### 不变量

- 服务端权威不变，所有写权威态走原域服务方法
- 协议线上格式（envelope / panel / aux / threat / minimap）字段不变
- 持久化 schema 不变（仅写入内容裁剪：`raw_payload` 内不再带模板字段）
- 5000 玩家 / 10000 实例 / 8C16G / 30Mbps 单服口径不变
- "下次还在"的状态以数据库为真源；Registry 不持有真源

## 2. 现状盘点（基于真实代码）

### 现存 Repository（共 3 个类）

| 文件 | 类 | 内部模板表 | 入口 |
|---|---|---|---|
| `packages/server/src/content/content-template.repository.ts` | `ContentTemplateRepository` | `itemTemplates` / `techniqueTemplates` / `skillTemplatesById` / `sharedTechniqueBuffs` / `formationTemplates` / `monsterRuntimeTemplates` / `monsterDropsByMonsterId` / `monsterRuntimeStatesByMapId` / `starterInventoryEntries` | `createItem` / `normalizeItem` / `createRuntimeMonsterSpawn` / `getSkill` / `getSkillRef` / `hydrateTechniqueState` / `loadSharedTechniqueBuffs` / `rollLootPoolItems` / `rollMonsterDrops` … |
| `packages/server/src/runtime/map/map-template.repository.ts` | `MapTemplateRepository` | `templates` / `mapGroupMembersById` / `npcLocationById` / `questSourceById` | `getOrThrow` / `getNpcLocation` / `getQuestSource` / `registerRuntimeMapTemplate` / `loadAll` |
| `packages/server/src/runtime/building/building-content.repository.ts` | `BuildingContentRepository` | 无（无状态）| `compile(BuildingDef[]) → CompiledBuildingCatalog` |

### 已经做对的部分（来自上一轮 126 项扫描）

- 物品：`createItem` / `normalizeItem` 走 `Object.create(template)`，own keys 仅 `itemId / count / enhanceLevel`
- 妖兽 spawn：`baseAttrs / baseNumericStats / ratioDivisors / statFormula / initialBuffs / skills` 复用模板引用，不再深拷
- 功法：`buildTechniqueRuntimeStateFromTemplate` 让 layers / skills 复用模板引用
- 任务：运行态压成瘦白名单字段，不再携带 `title / desc / rewards`
- buff：`runtime-buff-instance.ts` 用 prototype-share 收口运行态 buff
- tile：`getSharedTileProjection` 在 network 层做 WeakMap 共享

### 仍未收口的真实问题

- **没有独立 BuffTemplateRegistry**：技能内联 buff、PVP buff（`buildPvPSoulInjuryBuffState` / `buildPvPShaInfusionBuffState` / `buildPvPShaBacklashBuffState`）、丹药 buff、妖兽 initial buff（`buildMonsterInitialBuffState`）各自构造，缺统一 `buff_id → BuffDef` 表
- **NPC / Quest / Container / Landmark 模板没有独立 Registry**：被当作 `MapTemplate` 的 plain field 暴露，外部直接解构读 → 不能在运行态做 `Object.freeze`，也不能批量打 audit
- **TileTemplateRegistry 只是一个网络层 helper**：`getSharedTileProjection` 不属于内容仓库，无法作为 hydrate / 持久化的稳定查找入口
- **没有任何 Pool / Slab / TypedArray**：`packages/server/src/**/*Pool.ts` 不存在，monster / 玩家属性热路径仍是 AoS 对象图
- **没有反 spread 守护**：仅有 `server-memory-retention-smoke` 等源码字符串断言，零散覆盖；没有统一的"runtime 路径禁止 spread 模板"扫描器
- **`Object.freeze` 几乎没用**：模板加载完即可冻结，但当前未做，下游"防御性 spread"是真有"会被改穿"的担心

## 3. 红线与不变量（落地阶段必须维持）

- **协议字段不动**：`SyncedItemStack` / `TechniqueUpdateEntryView` / `AttrPanelView` / `BuffStateView` 输出字段集合保持现状；prototype-share 实例必须能完整 JSON 序列化（项目已证明 own keys 写入即可，模板字段在出网时通过显式投影补齐）
- **持久化字段不动**：列字段保持 schema；`raw_payload` 内只允许"无法用列字段表示且非模板"的实例字段
- **运行时不变行为**：玩家可见的物品名 / 技能描述 / buff 文案 / 任务文案，必须在面板 / 通知 / GM 查询时按 id 查 Registry 补出
- **跨服 / 迁移不变**：`buildPersistedPlayerSnapshot` / 域 hydrate 链路对 Registry 失效（模板缺失）必须有显式日志与降级路径，不允许静默生成空对象
- **5000 玩家 1Hz tick**：本计划不允许在热路径增加任何同步 IO、JSON.parse、字符串签名比较

## 4. 目标架构

### 4.1 模板侧：每域一个独立 Registry

```text
packages/server/src/content/registries/
  item-template.registry.ts          ItemTemplateRegistry
  technique-template.registry.ts     TechniqueTemplateRegistry
  skill-template.registry.ts         SkillTemplateRegistry
  buff-template.registry.ts          BuffTemplateRegistry      ← 新建
  formation-template.registry.ts     FormationTemplateRegistry
  monster-template.registry.ts       MonsterTemplateRegistry
  drop-table.registry.ts             DropTableRegistry
packages/server/src/runtime/map/registries/
  map-template.registry.ts           MapTemplateRegistry       ← 由现 MapTemplateRepository 演化
  npc-template.registry.ts           NpcTemplateRegistry       ← 新建（从 MapTemplate 抽出）
  quest-template.registry.ts         QuestTemplateRegistry     ← 新建（接管 questSourceById）
  container-template.registry.ts     ContainerTemplateRegistry ← 新建
  landmark-template.registry.ts      LandmarkTemplateRegistry  ← 新建
  tile-template.registry.ts          TileTemplateRegistry      ← 新建（接管 getSharedTileProjection）
```

每个 Registry 实现统一形状：

```ts
interface TemplateRegistry<TId extends string, TTemplate, TInstance> {
  /** 启动期一次性加载并 freeze。再次调用为 noop（带告警）。 */
  loadAll(): void;
  /** 必有，找不到抛错并带 id；用于内部权威读取。 */
  getRef(id: TId): Readonly<TTemplate>;
  /** 找不到返回 undefined，调用方必须显式处理。 */
  tryGetRef(id: TId): Readonly<TTemplate> | undefined;
  /** 唯一实例化入口；prototype 指向 template，own keys 限定为白名单。 */
  createInstance(id: TId, init: InstanceInit<TInstance>): TInstance;
  /** 从持久化 payload 反实例化；own keys 仅恢复运行态字段。 */
  hydrate(id: TId, payload: PersistedInstance<TInstance>): TInstance;
  /** 调试用，列出全部 id；禁止在 tick 热路径调用。 */
  listIds(): readonly TId[];
}
```

### 4.2 实例侧：own 字段白名单 + prototype 模板

| 域 | own 白名单 | prototype 来源 |
|---|---|---|
| Item | `itemId / count / enhanceLevel`（+ 必要时 `mods` jsonb） | ItemTemplate（含 name/desc/equip*/effects/...） |
| Technique | `techId / level / exp / expToNext / realmLv / skillsEnabled` | TechniqueTemplate（含 name/grade/skills/layers/qiProjection） |
| Skill 运行态 | `skillId / cooldownReadyTick`（仅出现在妖兽 / 玩家 cooldown 表） | SkillTemplate frozen，不进运行态对象 |
| Buff | `buffId / stacks / remainingTicks / duration / sourceRealmLv? / sustainTicksElapsed? / persistOnDeath? / persistOnReturnToSpawn?` | BuffTemplate（含 name/desc/category/attrs/stats/qiProjection/sourceSkill*） |
| Quest | `questId / status / progress / targetCount?` | QuestTemplate（含 title/desc/rewards/路径文本） |
| Monster | `runtimeId / spawnId / monsterId / x / y / hp / qi / dir / aliveTickAt? / respawnAt? / buffs[] / cooldownReadyTickBySkillId / damageContributors` | MonsterTemplate（含 baseAttrs / baseNumericStats / ratioDivisors / statFormula / initialBuffs / skills / name / char / color） |
| Npc | `npcId / instanceId`（位置和 dialog 全在模板） | NpcTemplate（含 name/dialog/shopItems/quests/坐标） |
| Container | `containerId / instanceId / activeSearch? / lootState?` | ContainerTemplate（含 drops / lootPools / 坐标 / hp） |
| Landmark | `landmarkId / instanceId` | LandmarkTemplate |
| Tile | overlay / runtime patch（已存在） | TileTemplate frozen 投影 |

### 4.3 守护层

```text
packages/server/src/tools/audit-runtime-template-spread.ts   ← 新建
packages/server/src/tools/audit-registry-frozen.ts            ← 新建
packages/server/src/tools/server-memory-retention-smoke.ts    ← 扩条目
```

- `audit-runtime-template-spread`：AST 层（ts-morph）扫 `runtime/**` / `network/**` / `persistence/**` 内"对模板对象做 spread / Object.assign"的代码点，白名单需显式列出。CI 集成入 `pnpm audit:boundaries`。
- `audit-registry-frozen`：在所有 Registry 启动后断言模板表元素 `Object.isFrozen` 为真。冷路径运行，归到 `pnpm verify:standard` 链。
- `server-memory-retention-smoke`：继续扩展现有 `cacheLifecycleProof.*` / `itemTemplatePrototypeProof.*` 系列，新增 buff / quest / npc / container 的 own keys 断言。

## 6. Registry 详细设计

### 6.1 BuffTemplateRegistry（最关键的新增）

为何关键：当前唯一一块"既没有独立模板表、又被多处构造"的域，对应扫描计划里 4 份重复 `cloneTemporaryBuff` 的根因。

**模板源**（按优先级）：
1. `data/content/technique-buffs.json` 等共享 buff 文件（已存在 `sharedTechniqueBuffs` 加载）
2. PVP buff：`packages/shared/src/pvp/pvp-buff-defs.ts`（新建）—— `buildPvPSoulInjuryBuffDef(realmLv) → BuffDef` 形态，按境界缓存 frozen 模板
3. 妖兽 initial buff：从 `MonsterDef.initialBuffs` 取 `BuffDef`，统一进入 BuffTemplateRegistry 索引
4. 丹药 / 装备触发 buff：从 ItemTemplate 的 `consumeBuffs / equipTriggerBuffs` 取索引

**Registry 接口**：

```ts
class BuffTemplateRegistry {
  loadAll(): void;
  getRef(buffId: string): Readonly<BuffDef>;
  tryGetRef(buffId: string): Readonly<BuffDef> | undefined;
  createInstance(buffId: string, init: BuffInstanceInit): RuntimeBuff;
  hydrate(buffId: string, payload: PersistedBuff): RuntimeBuff;
  listIds(): readonly string[];
}

interface BuffInstanceInit {
  remainingTicks: number;
  duration: number;
  stacks: number;
  maxStacks?: number;
  sourceRealmLv?: number;
  sustainTicksElapsed?: number;
  persistOnDeath?: boolean;
  persistOnReturnToSpawn?: boolean;
}
```

**实例 own 白名单**：见 §4.2 Buff 行。`name / desc / category / attrs / stats / qiProjection / sourceSkill*` 全部走 prototype。

**协议出口**：`BuffStateView` / `VisibleBuffProjection` 在投影时通过显式字段拷贝补齐文本，不会因 prototype 化丢失。

**风险**：PVP buff 的属性加成依赖境界（`buildPvPSoulInjuryBuffState(realmLv)`）。解决方案：每个境界一份 frozen 模板，按 `(buffId, realmLv)` 复合 key 取，不在运行时构造。

### 6.2 NpcTemplateRegistry / QuestTemplateRegistry

**为何拆出**：当前 `npcLocationById` / `questSourceById` 是 plain Map，外部直接读 `MapTemplate.npcs` 取 `shopItems / quests / dialog`，没有"从 id 拿模板"的稳定入口；对应扫描计划里 NPC 实例化时的 spread 风险与 quest 模板字段污染运行态的问题。

**接口要点**：
- `NpcTemplateRegistry.getShopItems(npcId)` / `getQuests(npcId)` 返回 `ReadonlyArray`
- `QuestTemplateRegistry.getRewards(questId)` / `getNavigation(questId)` 在面板 / 通知时按需补
- 任务运行态依然是 `{ id, status, progress, ... }` 的瘦字段（已落地），文档化为强约束

### 6.3 ContainerTemplateRegistry / LandmarkTemplateRegistry / TileTemplateRegistry

- Container：drops / lootPools / tagGroups 全部是模板字段；运行态只存 `containerId / activeSearch? / lootState?`
- Landmark：纯静态，运行态实质只持引用
- Tile：把 `getSharedTileProjection` 从 network 层迁到 `runtime/map/registries`，提供 `getProjection(templateId, x, y)`；network 层 `world-sync-map-snapshot.service.ts` 改成消费方

### 6.4 MonsterTemplateRegistry

- 模板字段：`monsterId / name / char / color / level / tier / baseAttrs / baseNumericStats / ratioDivisors / statFormula / initialBuffs(string[]) / skills(string[]) / spawn 配置 / drops`
- 实例字段：`runtimeId / spawnId / x / y / hp / qi / dir / aliveTickAt? / respawnAt? / buffs[] / cooldownReadyTickBySkillId / damageContributors`
- `initialBuffs` / `skills` 在模板里改为 `string[]` id 引用，运行时由 BuffTemplateRegistry / SkillTemplateRegistry 解析；当前模板已是引用化但 id 内联在对象里，保留。

## 7. 防回退守护细节

### 7.1 `audit-runtime-template-spread`

ts-morph 实现要点：
- 扫描 `packages/server/src/{runtime,network,persistence}/**/*.ts`
- 对每个 `SpreadElement` / `CallExpression`，通过 `getType` / `getSymbol` 判断 spread 来源是否为 `Readonly<XxxTemplate>` / `XxxDef`
- 命中违规即输出 `{file, line, column, kind, sourceType}`
- `audit-allowlist.json` 例外条目格式：

```json
[
  {
    "file": "packages/server/src/runtime/instance/map-instance.runtime.ts",
    "line": 1234,
    "kind": "spread-template",
    "reason": "GM 替换装备槽必须独立写一份，确认非热路径"
  }
]
```

例外必须含 `reason` 且不超过 30 条，超过即视为重构未完成。

### 7.2 `audit-registry-frozen`

启动期 smoke：
- 对每个 Registry 取 `listIds()` 第一项，断言 `Object.isFrozen(getRef(id))`
- 子结构（attrs / stats / equipAttrs / formula）逐层断言 frozen
- 失败 exit 1，日志含 `registry / id / unfrozenPath`

### 7.3 `server-memory-retention-smoke` 扩展

- `runtimeBuffOwnKeys`：取一玩家 + 一妖兽 buff 实例，断言 own keys ⊆ §4.2 白名单
- `questRuntimeOwnKeys`：玩家任务 own keys ⊆ `{ id, status, progress, targetCount? }`
- `npcInstanceRefShared`：同 mapId 下两次 hydrate，npc 实例的模板 prototype 必须 `===`
- `containerInstanceRefShared`：同上
- `landmarkInstanceRefShared`：同上
- `tileProjectionRefShared`：同 templateId 不同实例的同坐标 tile projection 必须 `===`

## 10. 风险与回滚

| 风险 | 缓解 |
|---|---|
| prototype 实例 `JSON.stringify` 落库时丢字段 | 已通过物品落地证明 own keys 写入 jsonb 的语义；新增 buff / quest / monster 必须扩 retention smoke 同款断言 |
| 模板 frozen 后下游误改 | audit 脚本先扫一轮，违规点改完再开 freeze；保留 `RUNTIME_FREEZE_TEMPLATES=0` 临时关闭开关，仅供线上紧急回退 |
| Registry 拆分破坏 NestJS DI 顺序 | 子 Registry 用 `@Injectable()` + 显式构造注入，`ContentTemplateRepository` facade 在 `onModuleInit` 顺序调用 |
| MapTemplate 字段下游直接解构（`template.npcs`） | Step 2 不立刻删字段，先在 Registry 暴露稳定方法；Step 5 删字段时配 type 错误一次性扫净 |
| Step 6 SoA 失败 | `RUNTIME_USE_SOA_POOL` 默认关闭，仅 bench / 灰度开启 |
| BuffTemplateRegistry 接管 PVP buff 后境界跨变化错算 | 按 `(buffId, realmLv)` 缓存 frozen 模板，玩家境界推进时换 prototype 而非改字段；refresh 路径已落地 |

回滚策略：
- Step 1-3 每个子 Registry 独立 PR，可单独回滚
- Step 4 audit 默认 warn-only 跑两周，再切 fail
- Step 6 整体灰度，回滚即关 env flag
