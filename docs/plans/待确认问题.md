## P0 待确认

### S26 [P0] instance tick 全串行（10000 实例 1Hz 不可能收敛）

- 文件：`runtime/world/world-runtime-instance-tick-orchestration.service.ts:69-133`
- 现象：所有 instance 在单 for 循环内串行 tick，无并行化
- 影响：**致命** — 按目标 10000 实例计算，串行 tick 总耗时远超 1s 间隔，世界推进直接停滞。是扩容到目标并发的硬性阻塞。
- 待确认：并行化方案选择
- 可选方案：Promise.all 分桶 / worker_threads / SharedArrayBuffer
- 关联计划：[服务端与客户端多线程并行化改造计划](./服务端与客户端多线程并行化改造计划.md)

### S65 [P0] creditWallet/debitWallet 内存路径绕过 durable fence

- 文件：`runtime/player/player-runtime.service.ts:1122-1177`
- 现象：钱包变更仅操作内存 + markDirty，依赖异步 flush，崩溃时丢失
- 影响：**中等** — flush 间隔 5s 内崩溃会丢失灵石变更。docker 更新/OOM kill 时必然触发。
- 待确认：是否将钱包变更纳入 DurableOperationService，还是接受当前 flush 间隔风险

### S76 [P0] PvP/Combat 完全不走 DurableOperationService，战斗结算无事务

- 文件：`runtime/combat/` 和 `runtime/world/combat/` 全部文件
- 现象：PvP 杀人奖励、掉落、计数器全走内存 fire-and-forget
- 影响：**高** — PvP 死亡是玩家最敏感场景，崩溃时"杀了人但奖励没到"会引发严重投诉。
- 待确认：DurableOperation 改造范围（仅 PvP 死亡链路 / 全部战斗结算 / 仅资产变更部分）
- 关联计划：[战斗链路商业化重构计划](./战斗链路商业化重构计划.md)

### S64 [P0] 战斗/强化/灵根/掉落全用 Math.random 无 seed PRNG

- 关键文件：craft-panel-runtime.service.ts、player-progression.service.ts、map-instance.runtime.ts、world-runtime-loot-container.service.ts
- 现象：所有随机判定直接 Math.random()，不可重放、不可审计
- 影响：**低（功能正确，运营风险）** — 玩家投诉"强化概率不对"时无法举证回放。
- 待确认：seed 来源（per-tick seed / per-operation seed）、审计格式、是否需要回放能力

### S100 [P0] GM /server/restart 接口可触发自杀，无 audit/二次确认

- 文件：`http/native/native-gm.controller.ts:1108-1129`
- 现象：单次 POST 即触发 SIGTERM，无 audit log、无二次确认、无速率限制
- 影响：**中等** — GM 密码泄漏或 XSS 时攻击者可反复重启服务器造成 DoS。
- 待确认：是否接入 GmAuditLogPersistenceService + 二次确认 token 机制

### S67 [P0] GM 接口缺统一 audit log（部分已有）

- 文件：`http/native/native-gm.controller.ts`、`native-gm-world.service.ts` 等
- 现象：GmAuditLogPersistenceService 已存在且部分接入，但 restart/ban/flush/backup/restore 未接入
- 影响：**低（运维风险）** — 出事后无法追溯"谁在什么时候做了什么"。
- 待确认：哪些 GM 操作必须接入 audit log，是否需要统一 interceptor

### S92 [P0] PlayerDomainPersistence 多 domain 各自独立事务

- 文件：`persistence/player-domain-persistence.service.ts`（32 处 savePlayerXxxDomain）
- 现象：每个域独立事务，同 cycle 部分失败导致状态分裂
- 影响：**中等** — DB 瞬断时触发，后果严重（玩家"花钱没东西"）。改造工作量大。
- 待确认：flushPlayer 单事务 / 按 domain group 分批事务 / 失败重试策略

## P1 待确认

### S77 [P1] protobuf 热路径 13 处 JSON.stringify

- 文件：`packages/shared/src/network-protobuf-update-codecs.ts`、`network-protobuf-tick-codecs.ts`
- 现象：每 tick 协议组包路径用 JSON.stringify 序列化复杂字段塞入 wire string 字段
- 影响：**高（性能）** — 5000 玩家 × 10+ 字段/tick = 5-10 万次 stringify/秒，GC 压力显著。
- 待确认：为每个字段设计真正的 protobuf message schema，还是先做增量 patch 降频

### S81 [P1] map-instance 单实例内存占用未受预算约束

- 文件：`runtime/instance/map-instance.runtime.ts`
- 现象：100×100 地图 × 5 数组 ≈ 200KB/实例，10000 实例 ≈ 2GB 仅基础数组
- 影响：**高（内存）** — 超出 16GB 部署预算。是扩容的硬性阻塞之一。
- 待确认：chunk-based 按需分配 / 不活跃实例完全卸载 / 冷数据 lazy 加载

### S86 [P1] 服务端不连 Redis，与 AGENTS.md §2 描述不符

- 文件：生产代码 0 处 Redis 连接，docker-stack 启动 Redis 容器空跑
- 现象：在线态/session fencing 全用 PG advisory lock 兜底
- 影响：**低（当前单节点）** — 多节点扩展时才成为瓶颈。Redis 容器空跑仅浪费约 50MB 内存。
- 待确认：补上 Redis 还是裁剪（移除 Redis 容器 + 更新文档）

### S99 [P1] A* 寻路无路径缓存，auto-combat 每 tick 重新规划

- 文件：`runtime/world/world-runtime.path-planning.helpers.ts`、`runtime/world/combat/world-runtime-auto-combat.service.ts`
- 现象：1500 玩家 × 1Hz × A*(10000 cells) 单次 1-5ms，可能吃满一个核
- 影响：**中等（性能）** — 实际地图多为小图（30×30），压测后按需优化。
- 待确认：路径缓存策略 / JPS 替代 / worker_threads 算路

### S90 [P1] inventory 全部 Array.find() 线性扫描

- 文件：`runtime/player/player-runtime.service.ts`（10+ 处 O(N) 线性扫）
- 现象：5000 玩家 × 频繁背包操作，每秒数十万次 N=60 线性扫
- 影响：**低（当前规模）** — N=60 在 V8 中约 1-5μs/次，当前阶段不紧急。
- 待确认：Map<signature, ItemEntry> 索引 / immutable structure / dirty bit 索引
