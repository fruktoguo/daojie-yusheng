  一、严重（高优先级，已闭合）

  说明：以下 H/M/L 条目保留原始扫描的问题描述；每项开头的“状态”行是当前修复结论。

  H1. RuntimeEventBusService 玩家/实例队列从不清理

  - 状态：已修复并验证。生产路径已在玩家/实例卸载收口调用 `discardPlayer` / `discardInstance`；`server-memory-retention-smoke` 的 `eventBusProof` 证明 drain/flush 后玩家和实例队列数量归零。

  - 文件：runtime/event-bus/runtime-event-bus.service.ts:75, 78, 574-582
  - 原始现象：
    - playerQueues = new Map<string, PlayerEventQueue>() 在 getOrCreatePlayerQueue(playerId) 懒创建，每个
  PlayerEventQueue 自带 panelPatches:Map、activeJobs:Map、techniquePanelDirty:Set、feedback:[] 等。
    - instanceQueues = new Map<string, InstanceEventQueue>() 同理，自带 combatEffects:[]、aoiEffects:Map。
    - 类提供 discardPlayer(playerId) 和 discardInstance(instanceId)，但全仓库 grep 结果显示这两个方法只在 smoke
  测试里被调用，生产路径（WorldRuntimePlayerSessionService.removePlayer、PlayerRuntimeService.removePlayerRuntime、fence
  InstanceRuntime、destroyManagedInstance 等）一处都没调用。

  - 影响：每个登录过的 playerId、每个曾激活过的 instanceId 都会留下一条 entry。tick 内 flushTick() 会扫遍
  playerQueues.size 和 instanceQueues.size，规模随累计玩家/实例数线性增长，时间和内存都会越走越大。在 5000 玩家高频流动
  + 10000 实例规模下属于真实泄漏。
  - 建议：
    - 在
  WorldRuntimePlayerSessionService.removePlayer、PlayerRuntimeService.removePlayerRuntime、unloadDetachedPlayerRuntime
   任一收口处统一调用 runtimeEventBusService.discardPlayer(playerId)。
    - 在 destroyManagedInstance、fenceInstanceRuntime 卸载实例分支调用
  runtimeEventBusService.discardInstance(instanceId)。

  H2. 持久化层多个 Service 各自 new Pool({ connectionString })，配置缺失

  - 状态：已修复并验证。生产主持久化已收敛到 `DatabasePoolProvider` 共享池；保留的独立池限定在玩家鉴权存储和恢复清理工具，均已配置连接上限、idle timeout、connect timeout 或 statement timeout。

  - 原始发现：共 14 处生产代码绕开 DatabasePoolProvider 直接 new Pool：
    - persistence/player-domain-persistence.service.ts:695
    - persistence/player-persistence.service.ts:255
    - persistence/player-identity-persistence.service.ts:191
    - persistence/market-persistence.service.ts:48
    - persistence/mail-persistence.service.ts:108
    - persistence/durable-operation.service.ts:389
    - persistence/suggestion-persistence.service.ts:45
    - persistence/redeem-code-persistence.service.ts:46
    - http/native/native-gm-secret-store.service.ts:51
    - http/native/native-player-auth-store.service.ts:437
    - http/native/native-gm-admin.service.ts:358
    - runtime/gm/runtime-gm-auth.service.ts:43
    - runtime/world/world-runtime-formation.service.ts:1047
    - runtime/world/world-runtime-sect.service.ts:1171

  - 现象：每个 Pool 都没传 max / idleTimeoutMillis / connectionTimeoutMillis。pg 默认
  max=10、connectionTimeoutMillis=0（无限等待）。
  - 影响：
    - 14 池 × 10 = 140 条预留 PG 连接，超过 PG 默认 max_connections=100，5000 并发玩家场景必然撞墙。
    - connectionTimeoutMillis=0 意味着连接饥饿时调用方挂在 callback 队列里不返回，关联的 closure（包括玩家上下文、SQL
  payload）都无法 GC，是典型的"间接堆增长 + 反压"。
    - 与 DatabasePoolProvider（max=24、timeout 5s、idle 30s）的策略不一致，运行预算无法统一管控。

  - 建议：所有上述 service 改用 databasePoolProvider.getPool('<scope>')，至少明确传 max + connectionTimeoutMillis +
  idleTimeoutMillis。

  二、中（运营累积风险，已闭合明确可修项）

  M1. MarketRuntimeService.storageByPlayerId 全量驻留所有有过非空坊市仓库的玩家

  - 状态：已修复并验证。坊市仓库改为玩家维度 lazy hydrate，并用 `MARKET_STORAGE_RUNTIME_CACHE_LIMIT` 做运行时 LRU 驱逐，不再启动期全量驻留历史玩家仓库。

  - 文件：runtime/market/market-runtime.service.ts:55、清理点 :2714
  - 现象：启动 reloadFromPersistence() 会一次性灌入所有玩家的非空坊市仓库；运行期只在"玩家把仓库取空"时 delete，没有
  LRU、没有按上线/下线驱逐。
  - 影响：长期运营会持续累计"曾经放过坊市仓库且未取空"的全部历史玩家，与并发玩家解耦。每条
  ~0.5–2KB，十万级历史玩家时常驻 50–200MB。
  - 建议：改为按需 lazy-load（玩家上线/打开坊市时拉一次）+ 玩家下线一段时间或 LRU 上限驱逐；至少加 size 监控指标。

  M2. WorldRuntimeFormationService.formationsByInstanceId / restoredFormationInstanceIds 实例销毁时未无条件清理

  - 状态：已修复并验证。实例销毁/fencing 通过 `releaseInstance(instanceId)` 清理 `formationsByInstanceId` 和 `restoredFormationInstanceIds`。

  - 文件：runtime/world/world-runtime-formation.service.ts:35, 36, 349, 616, 815, 817
  - 现象：
    - formationsByInstanceId.delete 只在阵法数组刚好被清空时触发（line 349 formations.length <=
  0）。destroyManagedInstance / fenceInstanceRuntime 路径没有显式

  - 影响：每张曾经被加载阵法的实例都会留 1 条 instanceId 字符串以及（若有 active 阵法时）整组阵法对象。10000 实例 ×

  - 建议：在 destroyManagedInstance 与 fenceInstanceRuntime 卸载分支显式 formationsByInstanceId.delete(instanceId) 和

  M3. player.transferBufferedNotices 数组在 transfer 期间无上限

  - 状态：已修复并验证。transfer notice buffer 统一走有界追加，达到上限时丢弃最旧低优先级缓存。

  - 文件：runtime/player/player-runtime.service.ts:211, 1447, 3633-3650
  - 现象：玩家处于 transferState === 'in_transfer' 时所有 notice 都直接 push 到
  player.transferBufferedNotices，没有上限限制。靠 PLAYER_TRANSFER_TIMEOUT_MS = 120_000 + rollbackExpiredTransfer 兜底。
  - 影响：120s 内若 notice 爆发（战斗、邮件结算、坊市操作），单个玩家可能堆数千条 notice，每条 ~100B-1KB，量级 ≈ 几
  MB。极端场景压力大。
  - 建议：复用 MAX_NOTICES_PER_PLAYER（事件总线已有），对 transferBufferedNotices 加同等上限并丢弃低优先级。

  M4. PlayerRuntimeService notice 回退路径 player.notices.queue 无上限

  - 状态：已修复并验证。fallback 队列统一走有界追加；Nest 启动期对缺失 `RuntimeEventBusService` fail-fast，避免生产静默进入无上限兜底。

  - 文件：runtime/player/player-runtime.service.ts:1438-1454
  - 现象：

    if (this.runtimeEventBusService) {
      this.runtimeEventBusService.queuePlayerNotice(playerId, notice); // 有上限
    } else {
      player.notices.queue.push(notice); // 无任何上限
    }

  - 影响：依赖 NestJS 注入永远成功。一旦在某些 bootstrap、降级或测试桥路径里 runtimeEventBusService 为 null，单玩家
  notice 数组会无限增长。属于隐性兜底缺陷，长期巡检会出问题。
  - 建议：fallback 路径同样复用 MAX_NOTICES_PER_PLAYER 上限，且 NestJS 注入失败时直接 fail-fast 不要默默走 fallback。

  M5. RuntimeEventBusService.flushTick() O(N) 全表扫描 player/instance 队列

  - 状态：已随 H1 修复并验证。玩家/实例队列在 drain/flush 为空后删除，失活玩家和实例也有显式 discard 收口，避免空队列长期堆积放大 `flushTick()` 扫描。

  - 文件：runtime/event-bus/runtime-event-bus.service.ts:447-510
  - 现象：每 tick 都遍历 playerQueues 和 instanceQueues 全部 entry 累计指标。结合 H1，这两个 Map
  会无界增长；遍历开销也随之增长。
  - 影响：tick 1Hz × 5000+ 玩家 + 10000 实例，最坏情况每秒数万次空 entry 扫描，没有泄漏不会爆，但跟 H1
  叠加后是性能放大器。
  - 建议：跟 H1 一起修复（discardPlayer/Instance 收口）；可补一个轻量"脏队列集合"避免 flushTick 全扫描。

  L4. NativeAuthRateLimitService prune 仅在 registerFailure 触发

  - 文件：http/native/native-auth-rate-limit.service.ts:96-128
  - 现象：pruneBuckets(now) 在 registerFailure 入口调用。如果只有 assertAllowed/recordSuccess 调用而 registerFailure
  完全不触发，buckets 不会被清理。但 assertAllowed 走的是 readBucket + 不写回，且过期的 bucket 自动
  delete；recordSuccess 直接 delete。所以默认场景下上限 50000 + 1 分钟 prune 兜底足够，无实际泄漏。
  - 注意：pruneBuckets 内的 sort O(N log N) + slice，N=50000 时 prune 一次大约 10-30ms 阻塞，建议改成单遍堆/部分排序。

  L5. WorldSessionService.expiredBindings 由 reaper 每秒消费
