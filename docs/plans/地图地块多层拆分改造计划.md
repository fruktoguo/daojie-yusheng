# 地图地块多层拆分改造计划

> 本文用于规划当前生产主线的“地图格子从单一 `TileType` 拆分为多层 cell state”的实施方案。
>
> 本文是开发计划，不表示当前主线已经完成。主工作区固定为 `packages/*`，不恢复旧线目录作为新实现入口。

---

## 目标结论

当前 `TileType` 同时表达了草地、泥地、道路、墙、门、树、石头、矿、传送点等多类概念，已经把“地形、地表铺装、地上结构、自然阻挡物、交互物入口”混在同一层。后续建造、房间、风水、地形破坏、矿物采集、掉落物、箱子、阵法如果继续塞进同一个 `TileType`，会导致规则写死、前端投影混乱、持久化迁移困难，并且不利于高性能增量更新。

目标模型是：

```text
一个坐标 cell
-> 底层地形 terrain：草地、泥地、石地、沼泽、水、虚空等
-> 可选地表 surface：道路、地板、地砖、地毯、木板等
-> 地上结构 structure：墙、门、窗、石头、矿、树、建筑部件等
-> 地面物品 ground items：掉落物堆，沿用现有对象层
-> 交互对象 interactables：阵法、箱子、机关、传送点等，沿用或扩展现有对象层
-> 服务端合成 walkable / blocksSight / durability / roomBoundary / renderTile
-> 网络只向客户端发必要合成字段和低频可选分层字段
```

回答用户当前关心点：

- 一个地方可以同时有草地和墙。草地是 `terrain`，墙是 `structure`，最终合成结果不可通行、可能阻挡视线，但底层地形仍然是草地。
- 一个地方可以同时有泥地和道路。泥地是 `terrain`，道路是 `surface`，移动消耗按道路（surface）计算。
- 一个地方可以同时有地板、掉落物和箱子。地板是 `surface`，掉落物是 `groundItems`，箱子是 `interactable` 或容器对象。
- 前端短期没必要整体重做。客户端仍可渲染服务端合成后的 `VisibleTile.type`，只增加可选的分层字段用于 tooltip、GM、后续精细渲染。
- 不建议一次性删除 `TileType`。第一阶段应保留兼容投影，避免地图编辑器、客户端缓存、protobuf、AOI delta、minimap 一次性大面积爆炸。

### 当前问题

- [ ] `TileType` 语义过宽，导致“地形”和“地上物”无法共存。
- [ ] `walkable` / `blocksSight` / `damageable` 等规则只能按单一类型判断，不能表达地形、地表、结构、交互物共同决定。
- [ ] 地板、道路、地毯等铺装和草地、泥地、石地等底层地形混在一起，后续建造会覆盖地形真源。
- [ ] 墙、门、窗、树、石头、矿都作为 tile 类型存在，难以和房间、风水、采集、耐久、建造实例统一。
- [ ] 传送点、机关、阵法、箱子等交互物不应依赖 tile 类型表达，否则协议和编辑器会继续膨胀。
- [ ] 客户端 tooltip 目前倾向展示 `tile.type`，分层后需要显示“地形 / 地表 / 结构 / 交互物 / 掉落物”的合成信息。
- [ ] 持久化当前更偏向保存 tile 结果，未来动态建造和地形变更需要区分基础层、覆盖层和对象层。

## 目标分层模型

### CellLayerState

服务端权威内部目标结构：

```ts
interface CellLayerState {
  terrain: TerrainType;
  surface: SurfaceType | null;
  structure: StructureLayerRef | null;
  groundItems: GroundPileRef | null;
  interactables: InteractableRefList | null;
}
```

运行时不一定真的每格创建对象。热路径实现应拆成 typed array / handle / sparse index：

```ts
interface RuntimeCellLayers {
  terrainByCell: Uint16Array;
  surfaceByCell: Uint16Array;
  structureByCell: Uint32Array;
  compositeFlagsByCell: Uint32Array;
  revisionByCell: Uint32Array;
}
```

约定：

- `0` 表示空 surface / 空 structure / 空 interactable handle。
- `terrain` 必须存在；不存在格子仍由稀疏地图的 `handle = 0` 表达。
- `groundItems` 沿用现有地面物品堆对象，不进入基础 tile 真源。
- `interactables` 以对象索引表达，不强制写进每格对象；cell 只保存必要 handle 或由对象空间索引反查。

### TerrainType

底层地形，决定最基础的存在性、自然移动成本（无 surface 时生效）、自然环境和基础耐久口径。

示例：

- 草地
- 泥地
- 石地
- 沼泽
- 水域
- 熔池
- 寒沼
- 云地
- 虚空

规则：

- 一个存在的 cell 必须有 terrain。
- terrain 不应被建造墙体覆盖；建墙只是新增 structure。
- terrain 可以被地形改造玩法改变，但必须走服务端权威命令和持久化分域。

### SurfaceType

可选地表铺装，表达覆盖在 terrain 上的路面、地板、地毯等。

示例：

- 道路
- 小径
- 石砖地
- 木地板
- 地毯
- 回廊地面

规则：

- surface 可为空。
- surface 一般不单独阻挡移动和视线，但 **决定移动成本**：有 surface 时移动成本按 surface 计算，无 surface 时按 terrain 计算。
- surface 还影响房间舒适度、风水、展示效果。
- surface 不应吞掉 terrain，拆除地板后应恢复原 terrain。

### StructureLayer

地上结构或自然阻挡物，表达墙、门、窗、树、石头、矿、建筑部件等。

示例：

- 墙
- 门
- 窗
- 树
- 竹
- 石头
- 灵矿
- 玄铁矿
- 建筑部件

规则：

- structure 可为空。
- structure 决定主要的 `blocksMove`、`blocksSight`、`damageable`、`durability`、`roomBoundary`、`openingKind`。
- 玩家建造的 structure 应对应 `BuildingInstance` 或后续统一的 `StructureInstance`，不能只改 tile。
- 自然矿石和树木可以先作为静态 structure seed，后续再逐步实例化为可采集/可破坏对象。

### GroundItems

地面掉落物层，沿用现有 `GroundItemPileView`、ground pile patch 和客户端 `groundPiles`。

规则：

- 不阻挡移动和视线，除非后续显式新增“超大物品堆”规则。
- 不参与 terrain/surface/structure 的合成耐久。
- 与 terrain、surface、structure、interactable 可共存。

### Interactables

地上交互对象层，表达箱子、阵法、机关、传送点等。

示例：

- 阵法
- 箱子
- 采集容器
- 机关
- 传送点
- 任务交互物

规则：

- interactable 是对象，不是基础 tile。
- 是否阻挡移动、是否阻挡视线、是否可交互、是否可被攻击由对象配置或运行时状态决定。
- 传送点短期可继续通过兼容 `TileType.Portal` 投影，长期应迁移为 portal interactable。

## 合成规则

服务端应在运行时维护每格合成结果，供寻路、视线、战斗、房间、网络投影读取。

### walkable

```ts
walkable =
  terrain.walkable
  && (surface?.walkableModifierAllows ?? true)
  && !(structure?.blocksMove ?? false)
  && !hasBlockingInteractable
  && !occupiedByActor;
```

注意：

- `occupiedByActor` 不属于静态 cell 合成缓存，应在寻路/移动时叠加判断。
- structure 如果是门，可能根据开关状态或门类型决定 `blocksMove`。
- 阵法边界类对象可通过 interactable 临时阻挡移动，但不能直接改 terrain。

### blocksSight

```ts
blocksSight =
  terrain.blocksSight
  || (structure?.blocksSight ?? false)
  || hasSightBlockingInteractable;
```

注意：

- 地板和道路默认不阻挡视线。
- 窗户可用 `blocksSight = false`、`roomBoundary = true`、`openingKind = 'window'` 表达。
- 未来雾、烟、阵法遮蔽可作为临时 overlay/interactable effect 叠加。

### durability

```ts
durabilityTarget =
  structure
    ? calculateStructureDurability(builderOrMapLevel, structure.hpMultiplier)
    : calculateTerrainDurability(mapRealmLv, terrain.hpMultiplier);
```

原则：

- 自然地形耐久仍按地图等级。
- 玩家建造物的实际生命值按建造者建造等级计算，公式可沿用地块等级耐久公式，只替换等级来源。
- surface 通常不承受战斗破坏；如果地板要可拆除，应作为 buildable surface instance 单独维护耐久。
- 耐久当前不能再只挂在 `TileType` 上，要能挂到 structure 或 buildable surface。

### traversalCost

```ts
traversalCost =
  surface !== null
    ? surface.traversalCost
    : terrain.traversalCost;
```

原则：

- 有 surface 时，移动成本完全由 surface 决定（道路快、地毯慢等）。
- 无 surface 时，移动成本由底层 terrain 决定（草地正常、泥地慢、沼泽很慢等）。
- `movementCostOverride`（地图模板配置的逐格覆盖）优先级最高，存在时直接使用。
- structure 不直接影响移动成本；structure 通过 `blocksMove` 决定是否可通行，不可通行时成本为无穷大。
- 不使用乘法修正模型，避免 surface 和 terrain 成本叠加导致数值膨胀。

### renderTile

短期兼容投影：

```ts
renderTile = composeLegacyTileType(terrain, surface, structure, interactables);
```

优先级建议：

1. 需要占据格子的 structure，例如墙、门、窗、矿、树。
2. 需要兼容旧客户端行为的 portal/stairs 等 interactable。
3. surface，例如道路、地板。
4. terrain，例如草地、泥地、石地。

## 服务端权威设计

进度更新：2026-05-07 已完成第一阶段运行时分层、兼容协议投影、客户端轻量展示、动态地块分层持久化和专项验证；未勾选项仍作为后续编辑器/全量规则迁移/旧 `TileType` 收口任务。

### RuntimeTilePlane 演进

第一阶段不删除现有 `RuntimeTilePlane`，而是扩展为兼容结构：

```ts
class RuntimeTilePlane {
  // 兼容旧逻辑
  getTileType(cellIndex: number): TileType;
  setTileType(cellIndex: number, tileType: TileType): boolean;

  // 新分层逻辑
  getTerrain(cellIndex: number): TerrainType;
  setTerrain(cellIndex: number, terrain: TerrainType): boolean;
  getSurface(cellIndex: number): SurfaceType | null;
  setSurface(cellIndex: number, surface: SurfaceType | null): boolean;
  getStructureHandle(cellIndex: number): number;
  setStructureHandle(cellIndex: number, handle: number): boolean;
  getCompositeFlags(cellIndex: number): number;
  markCompositeDirty(cellIndex: number): void;
}
```

实施要求：

- [x] `getTileType()` 继续返回合成后的 legacy `TileType`，保证旧调用不立即失效。
- [x] 新规则优先读 `getTerrain/getSurface/getStructureHandle/getCompositeFlags`。（寻路/视线/移动成本已切到分层 API 或 composite flags；房间/风水仍按后续阶段收口）
- [ ] 所有 `setTileType()` 调用必须逐步审计归类：到底是在改 terrain、surface 还是 structure。
- [x] 合成 flags 由服务端统一维护，不能让 network/client 自行猜规则。

### 结构实例归属

玩家建造物优先复用现有 `BuildingInstance`，不新增第二套建筑真源。

结构层可分为两类：

| 类型 | 真源 | 说明 |
|---|---|---|
| 静态自然结构 | map seed / runtime structure seed | 树、石头、矿等，短期可由地图模板迁移而来 |
| 玩家建造结构 | `BuildingInstance` | 墙、门、窗、地板、设施等，通过建造命令创建 |

如果后续发现“自然矿石/树木”和“玩家建筑”生命周期差异过大，可新增 `WorldStructureInstance`，但第一版不应为了抽象而扩散系统。

### 操作入口

所有改变分层状态的动作都必须进入服务端受控流程：

- 建造 structure。
- 铺设 surface。
- 拆除 structure。
- 拆除 surface。
- 地形改造 terrain。
- 采集/破坏自然 structure。
- 创建/移除 interactable。

禁止：

- [ ] 确认没有 socket handler 直接改 cell layer。
- [ ] 确认没有客户端决定最终可放置性。
- [ ] 确认没有只发 tile patch 而不更新权威运行时真源。
- [ ] 确认没有只改显示层不改寻路、视线、AOI、房间、持久化。

## 协议与客户端兼容策略

### VisibleTile 兼容扩展

短期协议保持 `VisibleTile.type`，新增可选字段：

```ts
interface VisibleTile {
  type: TileType;
  walkable: boolean;
  blocksSight: boolean;
  terrainType?: TerrainType;
  surfaceType?: SurfaceType | null;
  structureType?: StructureType | null;
  structureInstanceId?: string;
  interactableKinds?: readonly InteractableKind[];
}
```

原则：

- `type` 是兼容渲染用合成结果。
- `walkable` / `blocksSight` 是服务端权威合成结果。
- 新字段低频、可选、按可见范围发，不放长文本详情。
- 高频 delta 只发变化 cell 的 patch，不全量重发地图。
- 客户端不能用分层字段自行裁定移动合法性。

### 客户端短期改动范围

短期只做轻量适配：

- [x] `map-store` 保留现有 tile cache，normalize 时保留可选分层字段。
- [x] tooltip/观察面板可显示“地形 / 地表 / 结构 / 交互物 / 掉落物”。
- [x] minimap 和 canvas 继续使用 `tile.type` 或服务端合成颜色。
- [x] 建造预览继续作为 overlay，不写入真实 tile cache。
- [ ] 手机端不增加额外常驻面板，只在查看详情时展示分层信息。

长期精细渲染可选：

- [ ] terrain 绘制底色。
- [ ] surface 绘制铺装纹理。
- [ ] structure 绘制上层字形或图标。
- [ ] interactable 和 ground item 继续走对象层渲染。

### 网络包体约束

- [x] 首包可携带可见范围内必要分层字段。
- [x] 高频 world delta 只发 `cell + changed layer ids + walkable/blocksSight/type`。
- [x] 静态 label、说明、颜色表走客户端 catalog 或低频 detail，不进高频包。
- [x] 建筑/阵法/箱子详情继续按需请求，不混入 tile patch。
- [x] protobuf/envelope 需要保留兼容字段顺序和缺省值策略。

## 持久化与恢复策略

### 分域真源

建议按生命周期拆分持久化：

| 分域 | 内容 | 是否必须落库 |
|---|---|---|
| terrain cells | 动态新增格子、被地形改造后的底层地形 | 是 |
| surface cells | 玩家铺设或被玩法改变的地表 | 是 |
| structure instances | 玩家建造墙门窗、可破坏自然结构状态 | 是 |
| structure damage | 结构当前耐久、破损状态 | 是 |
| ground item piles | 现有地面掉落物 | 已有，继续保留 |
| interactables | 阵法、箱子、传送点、机关等对象 | 是，按对象已有分域或新增分域 |
| composite cache | walkable/blocksSight/renderTile 等缓存 | 否，可启动重算 |

### 恢复顺序

```text
加载静态地图模板
-> 编译为 terrain/surface/structure seed
-> 读取动态 terrain 覆盖
-> 读取 dynamic surface 覆盖
-> 读取 structure/building/interactable 实例
-> 重建 cell layer handle 和空间索引
-> 重算 composite flags / room topology / AOI projection cache
-> 进入 tick
```

要求：

- [x] composite cache 不作为数据库真源。
- [x] 重启后建造物、地板、地形改造、自然矿破坏状态可恢复。
- [ ] 中途崩溃时，材料扣除和建造目标创建需要有幂等或 durable operation 方案承接。
- [ ] 迁移脚本必须可重复执行，不能重复创建结构实例。
- [x] smoke/proof 创建的持久化对象必须自动清理。

## 房间、风水、建造、寻路影响

> 暂缓范围：房间识别与风水聚合的完整 layer trait/topology 改造本轮暂时不动。后续再处理 `roomBoundary/openingKind`、terrain/surface/structure/interactable trait 聚合、dirty region 增量重算等内容；本轮只允许修复建造、寻路、视线、移动成本、配置/GM/客户端验收这类不要求重写房间/风水模型的事项。

### 房间识别

房间识别应从 `structure.roomBoundary/openingKind/roofCoverage` 读取，不再看具体 `TileType.Wall`、`TileType.Door`。

- [ ] 墙、门、窗来自 structure 层。
- [ ] 地板、地毯来自 surface 层，只影响舒适和风水，不强制决定是否成房。
- [ ] 阵法覆盖来自 interactable/effect 层，只作为房间聚合附加项。
- [ ] 地形可影响房间有效性，例如水域上未铺地板不算有效室内。

### 风水

风水读取 RoomAggregate，不直接遍历 tile 类型：

- [ ] terrain 提供湿度、寒热、地气等基础 trait。
- [ ] surface 提供舒适、洁净、材质、五行修正。
- [ ] structure 提供封闭性、门窗、材质、破损、承重。
- [ ] interactable 提供阵法、箱子、设施等 trait。
- [ ] ground item 默认不参与风水，除非配置显式声明“杂物/污秽/灵物堆积”等效果。

### 建造

建造目标需要明确写入哪一层：

| 建造类型 | 写入层 |
|---|---|
| 地板、路面、地毯 | surface |
| 墙、门、窗 | structure |
| 家具、设施 | interactable 或 building object，按是否占结构层决定 |
| 阵法 | interactable |
| 地形改造 | terrain |

要求：

- [x] 放置校验由服务端根据 terrain/surface/structure/interactable 合成判断。
- [ ] 材料扣除、半成品、剩余工时、建造 buff、动作队列仍由建造系统负责。
- [x] 建造完成时更新对应 layer，并触发 composite dirty、房间 dirty、AOI patch。
- [x] 拆除时只移除目标 layer，不误删 terrain。

### 寻路与视线

寻路和视线只读取服务端合成 flags：

- [x] `isWalkable(cell)` 不再直接等于 `isTileTypeWalkable(原始单层tileType)`，而是读取 `tilePlane.isWalkable(cellIndex)` composite flags。
- [x] `blocksSight(cell)` 不再直接等于 `doesTileTypeBlockSight(原始单层tileType)`，而是读取 `tilePlane.blocksSight(cellIndex)` composite flags。
- [x] `isWalkable` / `blocksSight` 热路径切到直接读 `tilePlane.isWalkable(cellIndex)` composite flags，跳过字符串 TileType 中间层。
- [x] 占位、怪物、玩家、临时屏障等动态阻挡继续在运行时叠加。
- [ ] AOI 视野重算只处理受影响 dirty cell 或 dirty region，避免全图扫描。

## 性能设计

热路径目标：

- [x] cell 查询不分配短命对象。
- [x] 不拼接 `"x,y"` 字符串作为 tick 内主索引。
- [x] 不在 tick 内解析 JSON 或 schema。
- [x] 不在 tick 内按字符串 `defId` 大量分支。
- [ ] layer id、trait id、flags 使用数值 handle。
- [x] composite flags 在 layer 变化时增量重算，不每次寻路临时合成。
- [ ] dirty cell 用队列或 bitset 记录，房间/风水/AOI 只处理脏区域。
- [x] 网络 patch 只发发生变化的 cell 和对象，不发全图。

建议缓存：

```ts
const CELL_FLAG_WALKABLE = 1 << 0;
const CELL_FLAG_BLOCKS_SIGHT = 1 << 1;
const CELL_FLAG_DAMAGEABLE = 1 << 2;
const CELL_FLAG_ROOM_BOUNDARY = 1 << 3;
const CELL_FLAG_HAS_SURFACE = 1 << 4;
const CELL_FLAG_HAS_STRUCTURE = 1 << 5;
const CELL_FLAG_HAS_INTERACTABLE = 1 << 6;
```

## 风险与回滚

### 主要风险

- `TileType` 使用点多，直接替换容易引发地图编辑器、客户端缓存、protobuf、minimap、观察面板联动回归。
- 旧地图字符缺少“底层 terrain”上下文，例如 `Road` 不知道铺在草地还是泥地上，初期只能用默认映射。
- 建筑系统已经有 layer 雏形，若新建独立 structure 真源，可能和 `BuildingInstance` 形成双真源。
- 持久化若只保存合成 tile，会丢失 grass + wall 这类分层真相。
- 前端若过早做多层独立渲染，会扩大 UI/renderer 改动范围，影响移动端稳定性。

### 回滚策略

- [x] 第一阶段保留 `TileType` 合成投影，任何客户端问题可回滚到只读 `type`。
- [x] 分层字段全部设计为可选字段，旧客户端或旧缓存缺字段时按 legacy type 兜底。
- [x] 服务端保留旧 `setTileType/getTileType` 兼容层，迁移期间逐模块切换。
- [x] 动态分层持久化已直接写库（跳过了只读编译阶段）。
- [ ] 写库阶段按地图实例灰度启用，并提供实例级重算 composite 工具。
- [x] 发现规则错误时，可从 terrain/surface/structure 真源重新合成 legacy tile patch。
