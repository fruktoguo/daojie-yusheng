# RuntimeEventBusService 商业级设计计划

## 1. 定位与边界

**RuntimeEventBusService 是什么**：
- 不落库的运行时临时事件门面
- 语义化运行时通知出口，只提供明确业务方法
- 每种方法自带合并、限流、丢弃、flush 规则
- tick 内收集，tick 末尾统一 flush 到网络层

**RuntimeEventBusService 不是什么**：
- 不是通用事件系统（无 `publish/subscribe/topic`）
- 不是持久化通道（不落库、不保证离线送达）
- 不是资产正确性载体（装备/钱包/背包/任务/邮件仍走 DurableOperation + outbox）
- 不是跨进程消息总线（单服内存态，不跨节点）

**与现有系统的关系**：

| 现有机制 | 归属 | RuntimeEventBus 替代？ |
|---------|------|----------------------|
| `player.notices.queue` + `drainNotices` | PlayerRuntimeService | 是，统一收编 |
| `pendingCombatEffectsByPlayerId` + `drainCombatEffects` | PlayerRuntimeService | 是，统一收编 |
| `RuntimeGmStateService.pendingStatePushPlayerIds` + `flushQueuedStatePushes` | RuntimeGmStateService | 是，统一收编 |
| `WorldSyncService.flushConnectedPlayers` | 网络层 | 否，EventBus flush 后仍由 SyncService 组包发送 |
| `OutboxDispatcherService` / `DurableOperationService` | 持久化层 | 否，完全独立；EventBus 只负责"已落库状态的表现推送" |

## 2. 核心架构

```
┌─────────────────────────────────────────────────────────┐
│                    WorldTickService                       │
│  advanceFrame() → RuntimeEventBus.flushTick() → Sync    │
└─────────────────────────────────────────────────────────┘
         ▲                    │
         │ queue*()           │ drain*()
         │                    ▼
┌────────────────┐   ┌──────────────────────┐
│  Runtime 各域   │   │  WorldSyncService    │
│  (combat/craft/ │   │  (组包 + socket.emit)│
│   quest/market) │   └──────────────────────┘
└────────────────┘
```

**数据流**：
1. Runtime 各域在 tick 处理中调用 `RuntimeEventBus.queue*()` 方法
2. 每个 queue 方法按自身规则决定：追加 / 合并 / 丢弃 / 限流
3. tick 末尾 `WorldTickService` 调用 `RuntimeEventBus.flushTick()`
4. flush 按 player/instance 维度 drain 队列，交给 `WorldSyncService` 组包发送
5. flush 完成后队列清空，下一 tick 重新收集

## 3. 公开 API（语义方法集）

```typescript
@Injectable()
export class RuntimeEventBusService {
  // ─── 玩家通知 ───
  queuePlayerNotice(playerId: string, notice: StructuredNoticePayload): void;

  // ─── 面板增量更新 ───
  queuePlayerPanelPatch(playerId: string, panelKind: PanelKind, patch: PanelPatch): void;

  // ─── 活跃任务/制作进度 ───
  queueActiveJobProgress(playerId: string, progress: ActiveJobProgress): void;

  // ─── 功法/技能面板刷新 ───
  queueTechniquePanelRefresh(playerId: string, kind: TechniquePanelKind): void;

  // ─── 战斗表现事件（飘字/特效/击退） ───
  queueCombatPresentation(instanceId: string, event: CombatPresentationEvent): void;

  // ─── GM 面板状态推送 ───
  queueGmStatePush(playerId: string): void;

  // ─── AOI 表现事件（实体出现/消失/动作） ───
  queueAoiPresentation(instanceId: string, event: AoiPresentationEvent): void;

  // ─── 玩家属性/状态变化摘要 ───
  queuePlayerStateDelta(playerId: string, delta: PlayerStateDelta): void;

  // ─── 即时反馈（操作确认/拒绝） ───
  queuePlayerFeedback(playerId: string, feedback: PlayerFeedback): void;

  // ─── flush 入口 ───
  flushPlayer(playerId: string): FlushResult;
  flushInstance(instanceId: string): FlushResult;
  flushTick(): FlushResult;

  // ─── 作用域丢弃（玩家下线/实例销毁） ───
  discardPlayer(playerId: string): void;
  discardInstance(instanceId: string): void;
}
```

**禁止暴露**：
- `publish(topic, payload)` — 无自由 topic
- `subscribe(topic, handler)` — 无运行时订阅
- `on(event, callback)` — 无回调注册
- `emit(event, data)` — 不直接触发网络 IO

## 4. 每种方法的合并/限流/丢弃规则

| 方法 | 模式 | 合并规则 | 上限 | 丢弃策略 | 跨 tick |
|------|------|---------|------|---------|---------|
| `queuePlayerNotice` | 追加 | 同 key+vars 去重 | 32/player/tick | 丢低优先级，保留最新 | 否 |
| `queuePlayerPanelPatch` | 合并 | 同 panelKind 深合并 patch | 1/panelKind/player | 后覆盖前 | 否 |
| `queueActiveJobProgress` | 覆盖 | 同 jobId 只保留最后一次 | 1/jobId/player | 后覆盖前 | 否 |
| `queueTechniquePanelRefresh` | 去重 | 同 kind 只标记一次 | 1/kind/player | 幂等 | 否 |
| `queueCombatPresentation` | 追加 | 不合并 | 64/instance/tick | 丢最早的 | 否 |
| `queueGmStatePush` | 去重 | 同 playerId 只标记一次 | 1/player | 幂等 | 否 |
| `queueAoiPresentation` | 追加 | 同实体同类型合并为最终态 | 128/instance/tick | 丢低优先级 | 否 |
| `queuePlayerStateDelta` | 合并 | 同 player 深合并 delta | 1/player | 后覆盖前 | 否 |
| `queuePlayerFeedback` | 追加 | 不合并 | 8/player/tick | 丢最早的 | 否 |

**规则说明**：

### 4.1 追加模式（Append）
- 保留顺序，客户端按序消费
- 达到上限后按优先级丢弃（notice 有 kind 优先级：error > warn > success > info > system）
- 战斗飘字达上限后丢弃最早条目

### 4.2 覆盖模式（Override）
- 同一 tick 内同 key 只保留最后一次写入
- 适用于进度条、面板状态等"只关心最终值"的场景
- 内存开销恒定，不随写入频率增长

### 4.3 去重模式（Deduplicate）
- 同一 tick 内同 key 只标记一次，flush 时统一处理
- 适用于"需要刷新但不关心刷新几次"的场景
- 用 Set 实现，O(1) 写入

### 4.4 合并模式（Merge）
- 同一 tick 内同 key 的多次写入做浅/深合并
- 适用于面板 patch、属性 delta 等增量场景
- flush 时输出合并后的最终 patch

## 5. 内部数据结构

```typescript
interface EventBusState {
  // 按玩家维度
  playerQueues: Map<string, PlayerEventQueue>;
  // 按实例维度
  instanceQueues: Map<string, InstanceEventQueue>;
}

interface PlayerEventQueue {
  notices: StructuredNoticePayload[];       // 追加，上限 32
  panelPatches: Map<PanelKind, PanelPatch>; // 合并，每种面板 1 条
  activeJobProgress: Map<string, ActiveJobProgress>; // 覆盖，每 job 1 条
  techniquePanelDirty: Set<TechniquePanelKind>;      // 去重
  gmStateDirty: boolean;                    // 去重
  stateDelta: PlayerStateDelta | null;      // 合并
  feedbacks: PlayerFeedback[];              // 追加，上限 8
}

interface InstanceEventQueue {
  combatPresentations: CombatPresentationEvent[]; // 追加，上限 64
  aoiPresentations: AoiPresentationEvent[];       // 追加+同实体合并，上限 128
}
```

**内存预算**（5000 在线玩家 + 10000 地图实例）：
- PlayerEventQueue 平均 ~2KB/player → 5000 × 2KB = ~10MB
- InstanceEventQueue 平均 ~1KB/instance（多数实例无事件） → 活跃实例 ~1000 × 1KB = ~1MB
- 总计峰值 < 15MB，远在 16GB 服务器预算内
- 每 tick flush 后清零，不累积

## 6. 生命周期与 flush 时序

```
tick 开始
  │
  ├─ advanceFrame()
  │   ├─ 各 instance tick（怪物/战斗/掉落/AOI）
  │   │   └─ queue*() 写入 instanceQueues
  │   ├─ 各 player tick（制作/修炼/buff/寻路）
  │   │   └─ queue*() 写入 playerQueues
  │   └─ 系统命令/GM 命令处理
  │       └─ queue*() 写入对应队列
  │
  ├─ RuntimeEventBus.flushTick()
  │   ├─ 遍历 instanceQueues
  │   │   ├─ 按 AOI 范围筛选接收玩家
  │   │   └─ 交给 WorldSyncService 组包
  │   ├─ 遍历 playerQueues
  │   │   ├─ 检查玩家是否在线（有 socket）
  │   │   └─ 交给 WorldSyncService 组包
  │   └─ 清空所有队列
  │
  ├─ WorldSyncService.flushConnectedPlayers()
  │   └─ 发送 delta envelope + EventBus 产出
  │
  └─ tick 结束
```

### 6.1 flush 顺序保证

1. **instance 先于 player**：战斗飘字等 AOI 事件先发，玩家个人通知后发
2. **同一玩家内按写入顺序**：追加模式保持 FIFO
3. **合并/覆盖模式无序保证**：只保证最终态正确

### 6.2 事务成功后再 queue

```typescript
// 正确：durable operation 成功后再 queue UI 变化
const result = await durableOperationService.execute(op);
if (result.success) {
  runtimeEventBus.queuePlayerNotice(playerId, result.notice);
  runtimeEventBus.queuePlayerPanelPatch(playerId, 'inventory', result.patch);
}

// 错误：先 queue 再执行事务
runtimeEventBus.queuePlayerNotice(playerId, successNotice); // 危险！
await durableOperationService.execute(op); // 可能失败
```

## 7. 与现有 tick 流程的集成方案

### 7.1 WorldTickService 改造

```typescript
// 当前流程
private async runTickOnce(): Promise<void> {
  await this.worldRuntimeService.advanceFrame(interval, getMapTickSpeed);
  this.worldSyncService.flushConnectedPlayers();
  this.runtimeGmStateService.flushQueuedStatePushes();
}

// 改造后流程
private async runTickOnce(): Promise<void> {
  await this.worldRuntimeService.advanceFrame(interval, getMapTickSpeed);
  this.runtimeEventBusService.flushTick();  // 统一 flush 入口
  this.worldSyncService.flushConnectedPlayers();
}
```

### 7.2 WorldSyncService 改造

`flushConnectedPlayers` 在组包时从 EventBus 的 drain 结果中读取：
- notices → 附加到 envelope 的 notices 字段
- panelPatches → 附加到 envelope 的 patches 字段
- combatPresentations → 按 AOI 筛选后附加到对应玩家的 envelope
- stateDelta → 合并到 delta envelope 的 player state 部分

### 7.3 渐进式迁移路径

| 阶段 | 范围 | 风险 |
|------|------|------|
| Phase 1 | 新建 RuntimeEventBusService 骨架，实现 queuePlayerNotice + flushTick | 低：旧路径并行运行 |
| Phase 2 | 将 PlayerRuntimeService.notices.queue 迁移到 EventBus | 中：需要同步改 WorldSyncService |
| Phase 3 | 将 pendingCombatEffects 迁移到 EventBus | 中：涉及战斗表现链路 |
| Phase 4 | 将 RuntimeGmStateService.pendingStatePushPlayerIds 迁移到 EventBus | 低：独立子系统 |
| Phase 5 | 新增 panelPatch / activeJobProgress / stateDelta 等新能力 | 低：纯新增 |
| Phase 6 | 删除旧散落的 queue/drain 代码 | 低：Phase 1-4 验证通过后清理 |

每个 Phase 独立可验证、可回滚。Phase 1-2 是核心，后续 Phase 可按需推进。

## 8. 网络包体设计

### 8.1 EventBus flush 产出的协议结构

```typescript
// 附加到现有 delta envelope 中的 eventBus 字段
interface TickEventBusPayload {
  notices?: StructuredNoticePayload[];
  panelPatches?: Record<PanelKind, PanelPatch>;
  jobProgress?: Record<string, ActiveJobProgress>;
  techniqueDirty?: TechniquePanelKind[];
  stateDelta?: PlayerStateDelta;
  feedbacks?: PlayerFeedback[];
  combatEffects?: CombatPresentationEvent[];  // AOI 筛选后
  aoiEffects?: AoiPresentationEvent[];        // AOI 筛选后
}
```

### 8.2 包体优化原则

- 空字段不发送（undefined 不序列化）
- 多数 tick 对多数玩家，eventBus payload 为空（无事件发生）
- 只有产生事件的玩家/实例才有 payload
- 战斗/AOI 事件按视野范围筛选，不全图广播
- 单个 tick 的 eventBus payload 预期 < 2KB/player（峰值战斗场景）

## 9. 可观测性

### 9.1 运行时指标（不落库，暴露给 metrics endpoint）

```typescript
interface EventBusMetrics {
  // 计数器（每 tick 重置）
  tickQueuedTotal: number;        // 本 tick 总入队数
  tickFlushedTotal: number;       // 本 tick 总 flush 数
  tickDroppedTotal: number;       // 本 tick 总丢弃数
  tickMergedTotal: number;        // 本 tick 总合并数

  // 分类计数
  queuedByMethod: Record<string, number>;
  droppedByMethod: Record<string, number>;

  // 队列水位（flush 前快照）
  maxPlayerQueueSize: number;
  maxInstanceQueueSize: number;
  activePlayerQueues: number;     // 有事件的玩家数
  activeInstanceQueues: number;   // 有事件的实例数

  // 耗时
  flushDurationMs: number;        // 本次 flush 耗时
  flushDurationP99Ms: number;     // 滑动窗口 P99
}
```

### 9.2 告警阈值

| 指标 | 警告 | 严重 |
|------|------|------|
| `tickDroppedTotal` | > 100/tick | > 500/tick |
| `flushDurationMs` | > 50ms | > 200ms |
| `maxPlayerQueueSize` | > 24 | > 30（接近上限 32） |
| `maxInstanceQueueSize` | > 96 | > 120（接近上限 128） |

### 9.3 日志策略

- 正常运行：不打日志（热路径）
- 丢弃发生时：每 10 秒聚合一次 warn 日志，报告丢弃数和 top 3 来源
- flush 超时（> 100ms）：打 warn 日志，附带队列水位快照
- 不记录每条事件的内容（避免日志爆炸）

## 10. 边界场景处理

### 10.1 玩家下线

```typescript
// socket 断开时
runtimeEventBus.discardPlayer(playerId);
// 队列直接丢弃，不尝试发送
// 下次登录靠 PostgreSQL 真源重建完整状态
```

### 10.2 地图实例销毁

```typescript
// 实例无人后回收时
runtimeEventBus.discardInstance(instanceId);
// 该实例的 combatPresentations / aoiPresentations 全部丢弃
```

### 10.3 玩家跨图传送

```typescript
// 传送开始
runtimeEventBus.discardPlayer(playerId); // 清空旧图相关事件
// 传送完成，新图的事件从下一 tick 开始收集
```

### 10.4 断线重连

- 重连成功后，EventBus 不补发断线期间的事件
- 客户端靠 initialSync（全量快照）重建状态
- EventBus 只负责重连后新产生的增量事件

### 10.5 tick 超时

- 如果 advanceFrame 耗时过长导致 tick 堆积，EventBus 队列仍然只保留当前 tick 的事件
- 不跨 tick 累积（每次 flushTick 后清空）
- 如果 flush 本身超时，记录指标但不阻塞下一 tick

### 10.6 进程重启

- EventBus 是纯内存态，进程重启后队列为空
- 不需要恢复：玩家重连后靠 initialSync 重建
- 不需要持久化：这是设计约束，不是缺陷

## 11. 与 DurableOperation / Outbox 的协作模式

```
┌─────────────────────────────────────────────────────────────┐
│                     业务操作分类                              │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  纯表现（可丢）          资产变更（必须持久化）               │
│  ─────────────          ──────────────────────              │
│  战斗飘字                背包物品增减                         │
│  进度条更新              钱包余额变化                         │
│  面板刷新                装备穿戴/强化                        │
│  GM 面板推送             邮件收发                             │
│  AOI 实体动画            市场交易                             │
│                         任务完成奖励                          │
│       │                        │                            │
│       ▼                        ▼                            │
│  RuntimeEventBus         DurableOperation                   │
│  (queue → flush)         (execute → outbox → worker)        │
│       │                        │                            │
│       │                        ├─ 落库成功                   │
│       │                        │     │                      │
│       │◄───────────────────────┘     │                      │
│       │  成功后 queue 表现事件        ▼                      │
│       ▼                        outbox worker                │
│  WorldSyncService          (异步后续：邮件投递等)             │
│  (socket.emit)                                              │
└─────────────────────────────────────────────────────────────┘
```

**关键原则**：
1. EventBus 永远不承载资产正确性
2. DurableOperation 成功后才 queue 表现事件
3. 如果 DurableOperation 失败，不 queue 任何"成功"表现
4. 如果 EventBus flush 失败（玩家已下线），资产仍然安全（已落库）
5. 玩家下次登录时，initialSync 从数据库真源重建，不依赖 EventBus 历史

## 14. 热路径性能约束

### 14.1 queue 方法性能要求

| 操作 | 时间复杂度 | 分配 |
|------|-----------|------|
| 追加 | O(1) amortized | 无新对象（直接 push 引用） |
| 覆盖 | O(1) Map.set | 无新对象 |
| 去重 | O(1) Set.add | 无新对象 |
| 合并 | O(fields) 浅合并 | 可能产生新 patch 对象 |

### 14.2 flush 方法性能要求

- 遍历 Map 的 entries，不做 Array.from 转换
- 不在 flush 中做 JSON.stringify
- 不在 flush 中做深拷贝（直接移交引用，flush 后清空 Map）
- flush 总耗时目标 < 10ms（5000 玩家，正常负载）

### 14.3 内存管理

- PlayerEventQueue 使用对象池：玩家上线时从池中取，下线时归还
- InstanceEventQueue 懒创建：只有产生事件时才分配
- 数组使用 `.length = 0` 清空而非重新分配
- Map/Set 使用 `.clear()` 而非重新 `new`

## 15. 客户端消费侧设计

### 15.1 客户端接收入口

```typescript
// packages/client/src/network/event-bus-consumer.ts

export function handleTickEventBusPayload(payload: TickEventBusPayload): void {
  if (payload.notices) {
    noticeStore.appendBatch(payload.notices);
  }
  if (payload.panelPatches) {
    for (const [kind, patch] of Object.entries(payload.panelPatches)) {
      panelStore.applyPatch(kind as PanelKind, patch);
    }
  }
  if (payload.jobProgress) {
    activeJobStore.updateBatch(payload.jobProgress);
  }
  if (payload.techniqueDirty) {
    for (const kind of payload.techniqueDirty) {
      techniquePanelStore.markDirty(kind);
    }
  }
  if (payload.combatEffects) {
    combatRenderer.enqueueBatch(payload.combatEffects);
  }
  if (payload.aoiEffects) {
    aoiRenderer.enqueueBatch(payload.aoiEffects);
  }
  if (payload.stateDelta) {
    playerStateStore.applyDelta(payload.stateDelta);
  }
  if (payload.feedbacks) {
    feedbackRenderer.showBatch(payload.feedbacks);
  }
}
```

### 15.2 客户端消费原则

- 收到 payload 后立即分发到对应 store/renderer，不缓存
- 面板 patch 应用后触发局部 UI 更新，不全量刷新
- 战斗飘字/AOI 动画进入渲染队列，按帧消费
- notice 进入通知队列，按 UI 节奏逐条展示（不一次性弹出 32 条）

## 16. 安全与防御

### 16.1 服务端防御

- queue 方法内部校验 playerId/instanceId 有效性（O(1) Map.has）
- 无效 id 静默丢弃，不抛异常（避免影响 tick 主流程）
- 上限机制防止单个恶意/异常源撑爆内存
- flush 中捕获单个玩家的序列化异常，不影响其他玩家

### 16.2 客户端防御

- 客户端不信任 eventBus payload 中的数值（不直接修改本地资产）
- 资产变化仍然靠 delta envelope 中的权威字段
- eventBus payload 只用于表现层（飘字、动画、通知文本）

## 18. 风险与决策记录

| 决策 | 理由 | 风险 |
|------|------|------|
| 不跨 tick 累积 | 简化内存管理，避免无限增长 | 高频事件可能在单 tick 内被截断 |
| 不落库 | EventBus 只服务在线体验，离线靠真源重建 | 断线瞬间的表现事件丢失（可接受） |
| 固定上限而非动态调整 | 可预测内存开销，避免 GC 压力 | 极端场景（大规模团战）可能丢弃较多 |
| 语义方法而非泛型接口 | 编译期约束，防止滥用 | 新增事件类型需要改代码（有意为之） |
| instance 队列懒创建 | 10000 实例多数无事件，节省内存 | 首次 queue 有一次 Map 分配 |
| flush 后移交引用不拷贝 | 零拷贝性能 | 消费侧不能修改 drain 出的对象 |

## 19. 未来扩展预留

以下能力当前不实现，但架构不阻塞：

1. **跨节点事件转发**：如果未来多进程/多节点，可在 flushTick 中增加 IPC 转发层
2. **事件回放/录制**：可在 flush 出口增加可选的录制 tap，用于调试或回放
3. **客户端预测回滚**：EventBus 可携带 tick 序号，客户端据此判断是否需要回滚预测
4. **优先级队列**：当前用固定上限 + 丢弃策略；未来可升级为优先级堆
5. **批量压缩**：如果包体成为瓶颈，可在 flush 出口增加 delta 编码或 protobuf 序列化
