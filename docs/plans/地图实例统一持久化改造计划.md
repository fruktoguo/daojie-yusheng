# 地图实例统一持久化改造计划

> 本文用于规划当前生产主线的"所有地图实例统一持久化、tickSpeed 实例化、恢复流程收口"的实施方案。
>
> 本文是开发计划，不表示当前主线已经完成。主工作区固定为 `packages/*`。

---

## 目标结论

当前地图实例被分为"持久化的"（模板地图、宗门）和"临时的"（通天塔），但所有实例的运行态（灵气、阵法、时间流速、地面道具、地块覆盖、怪物状态、tick 计数）都应该落盘，重启后从落盘数据恢复，不允许丢失。

核心原则：

- **实例是唯一的运行时单位**，模板只是初始化来源之一。
- **所有实例都持久化**，不区分 persistent / ephemeral。
- **`instance_catalog` 是恢复的唯一入口**，启动时从 catalog 驱动恢复，不依赖"模板列表里有没有这个 ID"。
- **tickSpeed 是实例级属性**，存入实例 checkpoint，不再依赖按模板 ID 索引的独立表。
- **动态模板按需注册**，恢复时根据 catalog 中的 `instance_type` 决定用哪个工厂注册模板。

三种地图的区别仅在于"实例从哪里来"：

| 维度 | 模板地图 | 秘境（通天塔等） | 宗门 |
|------|----------|------------------|------|
| 模板来源 | JSON 配置文件 | 代码动态生成 | 代码动态生成 |
| 实例创建时机 | 启动时自动（现世+虚境） | 玩家触发 | 玩家使用建宗令 |
| 地块恢复规则 | 从模板 tile 定义读取 | 从生成规则/种子重算 | 从宗门等级/扩展逻辑计算 |
| 怪物重生规则 | 模板 monsterSpawns | 波次逻辑 | 守卫刷新逻辑 |
| 销毁条件 | 不销毁 | 超时无人 + 显式退出 | 宗门解散 |

创建之后，所有实例在运行时、持久化、恢复层面完全一致。

## 当前代码事实

### 已存在基础

| 能力 | 当前位置 | 现状 |
|------|----------|------|
| 实例分域持久化 | `persistence/instance-domain-persistence.service.ts` | 支持 tile_cell、tile_resource、tile_damage、temporary_tile、ground_item、container_state、monster_runtime、overlay、building/room/fengshui、time checkpoint |
| 实例目录 | `persistence/instance-catalog.service.ts` | 记录所有实例元数据、状态、租约，启动时用于恢复 |
| 宗门模板动态注册 | `runtime/world/world-runtime-sect.service.ts` | `registerSectTemplate` + `restoreCatalogSectTemplate` |
| 通天塔模板动态注册 | `runtime/world/world-runtime-tongtian-tower.service.ts` | `ensureLayerTemplate` 运行时注册 |
| GM 地图配置持久化 | `persistence/gm-map-config-persistence.service.ts` | `server_gm_map_config` 表，按模板 ID 存储 speed/paused/scale/offsetTicks |
| 运行时地图配置缓存 | `runtime/map/runtime-map-config.service.ts` | 内存 Map 按模板 ID 缓存 tickSpeed/paused/timeConfig |
| 实例 tick 编排 | `runtime/world/world-runtime-instance-tick-orchestration.service.ts` | `advanceFrame` 中按 `instance.template.id` 查询 tickSpeed |
| 实例恢复流程 | `runtime/world/world-runtime-lifecycle.service.ts` | `restorePublicInstancePersistence` + `world-runtime-instance-lease.helpers.ts` 中 `claimRecoverableCatalogInstances` |

### 当前问题

| 问题 | 影响 |
|------|------|
| 通天塔 `persistent: false`，重启后销毁重建 | 玩家打到高层退出，地面掉落、怪物状态、地块伤害全丢 |
| `tickSpeed` 按模板 ID 存储在 `server_gm_map_config` 表 | 同模板的现世/虚境共享流速；动态模板启动时被 prune |
| `restorePersistedMapConfigs` 启动时 prune 不在模板列表中的记录 | 动态模板（通天塔）的 GM 配置被误删 |
| `instance_checkpoint` 只保存 tick 计数 | tickSpeed、paused 等实例级配置重启后丢失 |
| 恢复流程依赖"模板先存在" | 动态模板启动时尚未注册，恢复被跳过或标记 template_missing |
| 通天塔销毁是内存直接删除 | 没有清理数据库中的分域数据，长期运行会积累脏数据 |
| `advanceFrame` 中 tickSpeed 按模板 ID 查询 | 无法为同模板的不同实例设置不同流速 |

## 改造方案

### Phase 1：tickSpeed 迁移到实例级别

**目标**：tickSpeed 成为实例自身属性，存入 checkpoint，重启后自动恢复。

#### 改动清单

1. `MapInstanceRuntime` 新增 `tickSpeed: number` 字段（默认 1），新增 `paused: boolean` 字段（默认 false）
2. `buildTimeCheckpointSnapshot` 中增加 `tickSpeed` 和 `paused` 字段写入
3. `hydrateInstanceFromCheckpoint` 中增加 `tickSpeed` 和 `paused` 恢复
4. `WorldRuntimeInstanceTickOrchestrationService.advanceFrame` 中改为从 `instance.tickSpeed` 读取速度，不再通过回调按模板 ID 查询
5. `WorldTickService` 中移除 `getMapTickSpeed` 回调传递，`advanceFrame` 签名简化
6. GM 修改流速的 API（`PUT /maps/:mapId/tick`）改为按 instanceId 修改，更新实例的 `tickSpeed` 字段并标记 time 域为脏
7. 兼容迁移：启动时若 `server_gm_map_config` 表有数据，一次性迁移到对应实例的 checkpoint 中

#### 验证

- 设置某实例 tickSpeed=10 → 重启 → 确认恢复为 10
- 同模板的现世和虚境设置不同流速 → 重启 → 各自恢复正确值

---

### Phase 2：通天塔实例持久化

**目标**：通天塔实例 `persistent: true`，运行态完整落盘，重启后恢复。

#### 改动清单

1. `ensureLayerInstance` 中 `persistent` 改为 `true`
2. 新增 `restoreCatalogTowerTemplate(entry)` 方法：根据 catalog 中的 templateId（格式 `tongtian_tower_layer_N`）解析层数，动态注册对应模板
3. `claimRecoverableCatalogInstances` 中增加通天塔模板恢复分支（类似宗门的 `restoreCatalogSectTemplate`）
4. `cleanupIdleInstances` 销毁流程改为：标记 catalog destroyed → 清理分域表数据 → 删除内存实例
5. 通天塔波次状态（当前层、当前波、怪物列表）纳入 checkpoint 的 `dungeonState` 字段

#### 验证

- 进入通天塔第 5 层 → 打到一半退出 → 重启 → 确认层数、怪物、地面掉落恢复
- 通天塔空闲超时 → 确认 catalog 标记 destroyed + 分域数据清理 + 内存释放

---

### Phase 3：统一恢复流程

**目标**：启动恢复完全由 `instance_catalog` 驱动，不依赖模板列表是否包含某个 ID。

#### 改动清单

1. 启动流程改为：
   - 从 `instance_catalog` 加载所有 `status='active'` 的条目
   - 按 `instance_type` 分类：`public` → 模板已存在直接创建；`tower` → 动态注册通天塔模板；`sect` → 动态注册宗门模板
   - 创建运行时壳
   - 从分域表恢复完整状态（包括 tickSpeed）
2. `bootstrapPublicInstances` 改为：仅对 catalog 中不存在的模板地图创建新实例（首次启动场景）
3. 移除 `restorePersistedMapConfigs` 中的 prune 逻辑
4. 移除 `applyPersistedMapConfig` 对模板存在性的硬依赖

#### 验证

- 全新数据库启动 → 自动创建模板实例并写入 catalog
- 重启 → 从 catalog 恢复而非重新创建
- 通天塔、宗门实例重启后正确恢复

---

### Phase 4：统一地块恢复接口

**目标**：不同类型地图的地块自动恢复走统一接口，而非硬编码读 template。

#### 接口定义

```typescript
interface TileRecoveryProvider {
  /** 获取指定坐标的原始地块类型（用于自动恢复目标） */
  getOriginalTileType(instanceId: string, x: number, y: number): TileType;
  /** 获取实例的恢复配置 */
  getRecoveryConfig(instanceId: string): { enabled: boolean; intervalTicks: number };
}
```

#### 各类型实现

- **模板地图**：从 `template.legacyTileRows` / `terrainRows` 读取原始地块
- **宗门**：从宗门等级和扩展状态计算当前应有布局（核心区域为空地，外围为石头）
- **秘境**：从生成种子重算基础地形，或标记为不自动恢复（由波次逻辑管理）

#### 改动清单

1. 定义 `TileRecoveryProvider` 接口
2. 实现 `TemplateTileRecoveryProvider`（模板地图）
3. 实现 `SectTileRecoveryProvider`（宗门）
4. 实现 `DungeonTileRecoveryProvider`（秘境，默认不恢复）
5. `advanceTileRecovery` 改为通过 provider 获取恢复目标

---

### Phase 5：秘境框架预留

**目标**：为后续随机地形秘境预留接口，当前通天塔作为第一个实现。

#### 数据模型

`instance_checkpoint` 中增加 `dungeonDescriptor` 字段：

```typescript
interface DungeonDescriptor {
  type: string;           // 'tower' | 'random_cave' | 'trial' 等
  params: unknown;        // 类型特定参数
}
```

各秘境类型的 descriptor 示例：

```typescript
// 通天塔
{ type: 'tower', params: { layer: 5 } }

// 后续随机洞穴
{ type: 'random_cave', params: { seed: 12345, difficulty: 3, biome: 'ice' } }

// 后续试炼场
{ type: 'trial', params: { trialId: 'fire_trial', stage: 2 } }
```

#### 改动清单

1. `instance_checkpoint` snapshot 结构增加 `dungeonDescriptor` 可选字段
2. 恢复时根据 `dungeonDescriptor.type` 选择对应的模板生成器
3. 通天塔改为写入 descriptor：`{ type: 'tower', params: { layer } }`
4. 新增 `DungeonTemplateFactory` 接口，通天塔实现 `TowerTemplateFactory`

---

### Phase 6：废弃 `server_gm_map_config` 表

**目标**：tickSpeed 完全迁移到实例级别后，清理旧表和旧服务。

#### 改动清单

1. 确认所有 tickSpeed/scale/offsetTicks/paused 已迁移到实例 checkpoint
2. 移除 `GmMapConfigPersistenceService`
3. 移除 `RuntimeMapConfigService` 中的 `gmMapTickSpeedByMapId` / `gmMapPausedByMapId` / `gmMapTimeConfigByMapId`
4. GM API 改为直接操作实例的 tickSpeed 字段
5. 删除 `server_gm_map_config` 表（通过迁移脚本）
6. 更新 `gm-map-config-persistence-smoke.ts` 或移除

## 数据迁移策略

| 阶段 | 策略 |
|------|------|
| Phase 1 上线 | 启动时检测 `server_gm_map_config` 表是否有数据，若有则按 templateId 找到对应实例，将 speed/scale/offsetTicks 写入实例 checkpoint，完成后标记迁移完成 |
| 过渡期 | 保留旧表只读，新写入只走实例 checkpoint；读取时优先实例 checkpoint，fallback 旧表 |
| Phase 6 | 确认线上稳定运行一个版本后删除旧表 |

## 风险与注意事项

| 风险 | 应对 |
|------|------|
| 通天塔持久化后数据库存储量增加 | 通天塔地图小（20x20），分域数据量极小；销毁时清理 DB 记录 |
| tickSpeed 迁移期间新旧并存 | 过渡期优先读实例 checkpoint，fallback 读旧表 |
| 恢复流程改动影响面大 | 分 Phase 逐步推进，每个 Phase 独立验证 |
| 秘境实例长期不活跃占用 DB | 保留 TTL 销毁策略，销毁时同步清理所有分域数据 |
| 随机地形种子丢失导致恢复后地形不一致 | Phase 5 中种子必须持久化到 checkpoint |
| 关闭时连接池已销毁但仍有落库请求 | 调整 shutdown 顺序：先完成所有 flush → 再关闭连接池 |

---

## 执行优先级

1. **Phase 1**（tickSpeed 迁移）— 直接解决当前用户问题
2. **Phase 2**（通天塔持久化）— 解决秘境数据丢失
3. **Phase 3**（统一恢复流程）— 架构收口
4. **Phase 4**（地块恢复接口）— 为宗门扩展铺路
5. **Phase 5**（秘境框架）— 预留扩展
6. **Phase 6**（废弃旧表）— 清理技术债
