# 玩家存档拆分方案

## 背景

当前服务端玩家存档采用单表宽行模式，`players` 同时承载：

- 角色核心成长与位置
- 背包、装备、功法、任务等长期真源
- 自动战斗、丹方预设等低频设置
- `online`、`inWorld`、`lastHeartbeatAt`、`offlineSinceAt` 等会话态

同时 `TickService` 每 60 秒会对所有在线玩家执行一次全量落盘，当前实现会把整份玩家 payload 整行 `UPDATE` 回 PostgreSQL。线上日志已经确认瓶颈集中在 `playerService.persistAll()`，其他运行时持久化段只占少量时间。

对照 `参考/jiuzhou` 后，差异不在 ORM 选型，而在持久化颗粒度：

- `jiuzhou`：关系型拆表真源 + 按变更写库 + Redis 运行时投影
- 当前项目：宽角色档 + 定时全量整档刷盘

## 目标

本次拆分目标不是一步把玩家真源完全改成 `jiuzhou` 的多表形态，而是分阶段把最重的热写链路拆开，先止血，再继续结构化拆表。

阶段目标：

1. 先把 `players` 的周期热写从“全量整档”改成“核心快照变化才写”
2. 先把会话态从周期热写里拆出，改成独立小更新
3. 再把低频设置态与大块集合态继续拆表

## 状态分层

### A. 核心长期真源

这些状态要求“下次登录还在”，并且属于角色核心进度，正式真源继续放 PostgreSQL：

- 位置与生存状态：`mapId`、`respawnMapId`、`x`、`y`、`facing`、`viewRange`、`hp`、`maxHp`、`qi`、`dead`
- 成长与数值：`foundation`、`combatExp`、`kill/death counters`、`boneAgeBaseYears`、`lifeElapsedTicks`、`lifespanYears`
- 角色能力与长期配置：`baseAttrs`、`bonuses`、`temporaryBuffs`、`inventory`、`marketStorage`、`equipment`、`techniques`、`bodyTraining`、`quests`
- 长期成长附属：`questCrossMapNavCooldownUntilLifeTicks`、`revealedBreakthroughRequirementIds`、`heavenGate`、`spiritualRoots`
- 长期设置：`unlockedMinimapIds`、`alchemySkill`、`gatherSkill`、`alchemyPresets`、`alchemyJob`、`enhancementSkillLevel`、`enhancementJob`、`enhancementRecords`
- 战斗/挂机设置：`autoBattle`、`autoBattleSkills`、`autoUsePills`、`combatTargetingRules`、`autoBattleTargetingMode`、`combatTargetLocked`、`autoRetaliate`、`autoBattleStationary`、`allowAoePlayerHit`、`autoIdleCultivation`、`autoSwitchCultivation`、`cultivatingTechId`

### B. 会话态 / 在线态

这些状态是运行时热点，仍需要数据库保存以支持重启恢复与 GM 查询，但不应继续混在周期整档热写里：

- `online`
- `inWorld`
- `lastHeartbeatAt`
- `offlineSinceAt`

阶段一先保留在 `players` 表，但改成独立小更新；后续阶段再评估是否单独拆表。

### C. 已有独立即时持久化

这些状态已经有自己的独立持久化链路，不需要再跟随周期整档重复写：

- `pendingLogbookMessages`

## 分阶段实施

### 阶段一：拆写入路径

目标：不改真源归属，只改落盘策略。

实施内容：

- 把玩家持久化 payload 拆成：
  - `core`：长期真源
  - `presence`：在线/会话态
- 周期落盘只比较 `core` 快照，快照未变则跳过数据库写入
- `markPlayerOnline` / `markPlayerOffline` / 世界离场等路径改为独立写入 `presence`
- `pendingLogbookMessages` 保持独立即时持久化，不再混入周期 `core`

预期收益：

- 直接减少 PostgreSQL 宽行 `UPDATE` 次数
- 避免心跳/在线态把整份角色档一起重写
- 不需要先改表结构，风险最低

### 阶段二：拆低频设置态

建议新建角色设置表，例如 `player_settings`，承接：

- 自动战斗与索敌相关设置
- 自动吃药配置
- 丹方预设
- 小地图解锁归档

目标：

- `players` 保留核心成长与世界状态
- 设置态按修改即时写入，不再占用核心存档热写带宽

### 阶段三：拆集合态真源

建议逐步把这些从 `players.jsonb` 拆成独立关系型或明细表：

- 背包 / 坊市仓：`player_inventory_item`
- 装备：`player_equipment_slot`
- 功法：`player_technique`
- 任务：`player_quest_progress`
- 临时 Buff：`player_persistent_buff`

目标：

- 避免整包 `jsonb` 重写
- 让真正变化的单项对象只更新自己的行
- 为后续排行、审计、GM 查询和批量修复提供更好的查询面

## 当前已落地

目前代码已经完成以下拆分：

- 新增 `player_presence`：在线 / 会话态真源
- 新增 `player_settings`：低频设置态真源
- 新增 `player_collections`：背包/装备/功法/任务/Buff/坊市仓真源
- `PlayerService` 已改为从 `players + player_presence + player_settings + player_collections` 组装角色
- 启动时会把旧 `players` 宽表中的历史数据自动回填到三张新表
- 周期落盘已按 `core / settings / collections` 分组比较快照，仅对发生变化的分组写库

当前仍保留对旧 `players` 宽列的兼容镜像写入，用于保持现有 GM 查询、排行榜和部分直接读旧列的链路可用。后续阶段的目标是逐步移除这些兼容镜像与直接读旧列的查询。

## 与 `jiuzhou` 的参考对照

`jiuzhou` 不是“每分钟把所有在线角色整档存一次”。它的落盘模型是：

- 角色基础字段直接写 `characters`
- 背包容量单独在 `inventory`
- 物品实例逐条落 `item_instance`
- 任务、功法、主线、成就、称号等各自独立表
- 资源 / 奖励 / 物品 mutation / 任务软进度先进入 Redis Delta，再由独立 flush worker 按 1 秒批量落库
- 在线战斗快照是 Redis 投影，只在事务提交后刷新，不混进 PG 主存档热路径

### 落盘对照表

| 维度 | 当前项目 | `jiuzhou` | 说明 |
| --- | --- | --- | --- |
| 角色主真源 | `players` 承载 `core`，并保留旧宽列兼容镜像 | `characters` 只保留角色基础字段 | 当前项目已从“一坨宽表”退到“分组真源 + 兼容镜像”，但主表仍比 `jiuzhou` 更宽 |
| 集合态真源 | `player_collections`，仍以 `jsonb` 分组存背包/装备/功法/任务/Buff/坊市仓 | `inventory` + `item_instance` + `character_task_progress` + `character_technique` 等明细表 | 当前项目已完成“按组拆”，`jiuzhou` 则继续拆到“按领域对象/实例” |
| 低频设置真源 | `player_settings` | 大多直接挂在 `characters` 的窄字段，少量独立表 | 当前项目把设置态从热写链路剥离了，但仍主要是 `jsonb`/宽对象 |
| 在线/会话态 | `player_presence`，并保留旧 `players` 镜像 | `last_offline_at` 这类直接写 `characters`；运行时热点更多靠 Redis/内存 | 当前项目显式单独成表，更适合保留“重启后仍要恢复”的会话信息 |
| 周期落盘入口 | `TickService` 每 60 秒触发一轮玩家/地块/掉落/怪物并行落盘 | 没有统一“全角色整档落盘”入口 | 这是两边最大的架构差异 |
| 玩家落盘策略 | `persistAll()` 按批遍历在线玩家，只写变更过的 `core/settings/collections` 分组 | 没有 `saveAllCharacters`；谁变了就写对应领域 | 当前项目已经从“全量整档”进化到“按分组脏写”，但仍保留周期扫描 |
| 高频资源变化 | 仍主要并入玩家分组存档，未独立 Delta 化 | `exp/silver/spirit_stones` 先入 Redis，1 秒批量 `UPDATE characters` | 这是下一步最值得吸收的点之一 |
| 物品变化 | 仍整体落到 `player_collections.inventory/marketStorage/equipment` | 奖励 Delta、实例 mutation 各自 1 秒 flush 到 `item_instance` / `mail` | `jiuzhou` 的库存是真正的实例化明细持久化 |
| 任务变化 | 仍在 `player_collections.quests` 整组快照里 | 任务软进度单独 1 秒 flush，任务真源是独立表 | 当前项目后续应优先把任务从 collections 拆出来 |
| 运行时战斗投影 | 当前项目主要以内存/Redis 缓存配合玩家存档 | 在线战斗快照明确是 Redis 投影，且必须在事务提交后刷新 | `jiuzhou` 的“真源 vs 投影”边界更清晰 |
| 失败补偿模型 | 周期任务失败会在下一轮重试，当前以分组快照比较为主 | Redis Delta 走 `main -> inflight -> finalize/restore` | `jiuzhou` 对异步 flush 的失败恢复机制更完整 |

### `jiuzhou` 的 flush 周期

`jiuzhou` 没有统一角色整档周期，但有多条独立的 1 秒 flush worker：

- 任务软进度 Delta：`1_000ms`
- 物品奖励 Delta：`1_000ms`
- 资源结算 Delta：`1_000ms`
- 物品实例 mutation：`1_000ms`

这意味着它不是“每秒把所有角色存一遍”，而是“每秒把已经积累的某类增量批量结算一次”。

### 对当前项目的直接启发

当前项目已经完成的吸收：

1. 不再把 `online/inWorld/heartbeat` 这类会话态绑在周期整档热写里
2. 不再无条件整档重写，而是按 `core / settings / collections` 分组比较快照
3. 已经建立独立真源表，为后续继续细拆创造了稳定中间层

下一阶段最该继续吸收的，不是照搬 `jiuzhou` 的所有表，而是继续对齐这三条：

1. 把高频结算从“周期扫描玩家”继续推进到“谁变了就推进谁”
2. 把 `player_collections` 从分组 `jsonb` 继续拆成更细的领域真源
3. 把运行时热点和数据库真源之间的投影边界再拉清，避免热路径为“下次还在”承担不必要的整组写库成本

## 风险与回归点

阶段一需要重点验证：

- 断线留在世界、离线超时离场、世界内主动退出时，`online/inWorld/offlineSinceAt` 是否仍然持久化正确
- 登录恢复与重启恢复是否仍按原有规则工作
- `pendingLogbookMessages` 是否仍可跨重连保留
- 周期落盘跳过未变 `core` 后，移动、修炼、战斗、背包变更、任务推进是否仍能在下次登录保留

## 本轮实施范围

本轮已落地：

- 写入路径拆分：`core` / `presence` / `settings` / `collections`
- 新增三张分组真源表：`player_presence`、`player_settings`、`player_collections`
- 周期落盘按分组比较快照后写库
- `pendingLogbookMessages` 从周期分组存档中移除，继续走独立即时持久化
- `market.service`、`gm.service` 相关旁路已接到新的分组保存入口

后续重点：

- 把 GM/排行榜/检索查询逐步改为直接读新表，而不是继续依赖旧 `players` 宽列镜像
- 进一步把集合态从 `jsonb` 继续拆到更细的关系型明细表
