# 反多开方案

## 1. 目标与边界

本文档用于给当前 Web MMO MUD 项目提供一套可落地、低误伤、可持续迭代的反多开方案。

目标：

- 提高批量注册和多号养大号的成本
- 限制多号在经济系统、活动奖励和资源转移上的收益
- 为 GM 提供可追踪、可解释的风险依据
- 在不引入原生客户端的前提下，尽量利用 Web 环境能稳定获得的信号

非目标：

- 不追求“绝对封死”所有多号
- 不把单一 IP、单一设备指纹或单次异常行为直接当作封禁依据
- 不在客户端实现任何决定性的风控判定

结论先行：

- 纯 Web 环境拿不到真正意义上的机器码，不能依赖“硬件唯一标识”做强绑定
- 当前项目已经实现“同账号顶号”，但尚未具备“同人多号识别”能力
- 反多开的重点不应放在“登录入口一刀切”，而应放在“账号主体识别 + 收益限制 + GM 审核”

## 2. 当前现状

结合当前代码，现有账号体系具备以下特征：

- 注册与登录入口位于 `packages/server/src/auth/auth.service.ts`
- 连接接管逻辑位于 `packages/server/src/game/game.gateway.ts`
- 用户表 `packages/server/src/database/entities/user.entity.ts` 目前只保存用户名、显示名、密码哈希、在线时长等基础信息
- 角色表 `packages/server/src/database/entities/player.entity.ts` 目前是一个账号对应一个角色，`userId` 唯一
- 当前连接阶段只按 `userId` 顶号，不会识别“不同账号但疑似同一操作者”
- 当前前端登录注册位于 `packages/client/src/ui/login.ts`，HTTP 请求封装位于 `packages/client/src/ui/auth-api.ts`

因此，当前系统能防：

- 同账号重复在线

当前系统不能防：

- 同一人注册多个账号
- 同一设备短时间批量建号
- 小号向大号转移利益
- 一组关联号刷活动、刷首通、刷兑换码、刷排行榜

## 3. 设计原则

### 3.1 服务端权威

- 风控主判定必须在服务端完成
- 客户端只允许上报辅助信号，例如本地持久化的 `deviceId`
- 所有封禁、限制、冻结、豁免都必须以服务端持久化记录为准

### 3.2 信号分层

将信号按可靠性分为三层：

- 强信号：手机号实名、支付实名、人工认证、可信设备绑定
- 中信号：服务端签发的 `deviceId`、连续登录轨迹、稳定行为模式、资源流向
- 弱信号：IP、UA、浏览器指纹、时区、语言、屏幕尺寸

规则：

- 单个弱信号不能直接封号
- 多个弱信号叠加可触发观察或临时限制
- 中信号可用于限权和人工审核排序
- 强信号才适合做更硬的限制

### 3.3 限收益优先于限登录

多号的真正收益主要来自：

- 资源转移
- 经济系统套利
- 首次奖励重复领取
- 活动奖励重复刷取
- 侦查、占坑、挂机辅助

因此设计顺序应为：

1. 识别风险主体
2. 限制高收益行为
3. 再决定是否需要限制注册或登录

## 4. 为什么不能只靠 IP

IP 只能表示“出口”，不能稳定表示“一个人”。

常见误伤场景：

- 同一家庭、情侣、室友共用家宽
- 学校、公司、宿舍、网吧大量用户共享出口 IP
- 手机网络存在运营商 NAT，同一公网 IP 后可能挂着很多真实用户
- 家宽重拨、移动网络切换、代理变化会导致同一玩家 IP 经常变化

因此 IP 适合作为：

- 注册限流依据
- 登录失败限流依据
- 风险评分的弱信号
- GM 排查时的辅助条件

IP 不适合作为：

- 唯一多号判定依据
- 唯一封禁依据
- 唯一设备归因依据

## 5. 总体方案

采用四层模型：

1. 入口限流
2. 账号主体识别
3. 收益与权限限制
4. GM 审核与申诉

### 5.1 第一层：入口限流

目标是挡住最便宜的脚本批量开号。

建议：

- 对 `/auth/register` 增加验证码
- 对 `/auth/register` 做 Redis 限流
- 对 `/auth/login` 做登录失败次数与冷却限制
- 对同 `deviceId`、同 IP、同 IP 段的注册密度做限制

建议阈值：

- 同 IP 30 分钟内注册不超过 3 个账号
- 同 `deviceId` 24 小时内注册不超过 2 个账号
- 同账号连续登录失败 5 次后进入冷却
- 同 IP 短时间高频失败登录时触发验证码升级

说明：

- 这些阈值用于挡机器人，不应用来直接判定作弊
- 所有限流命中都应记审计日志，供后续调优

### 5.2 第二层：账号主体识别

核心思路不是“找出机器码”，而是给每个账号建立风险画像。

建议采集的字段：

- 注册 IP
- 最近登录 IP
- 首次登录时间
- 最近登录时间
- 登录 UA
- 服务端签发并持久化的 `deviceId`
- 可选的浏览器辅助指纹摘要

推荐做法：

- 首次打开登录页时，客户端请求服务端申请一个 `deviceId`
- `deviceId` 存在 `localStorage` / `IndexedDB`
- 后续注册、登录、Socket 连接都带上该 `deviceId`
- 服务端对 `deviceId` 做签名或使用随机 UUID，避免客户端伪造任意结构

注意：

- `deviceId` 不是可信硬件 ID，只是一个“长期设备会话标识”
- 玩家清缓存、换浏览器、换设备后会变化，因此只能作为中信号

### 5.3 第三层：收益与权限限制

这是最关键的一层。

对于新号或高风险号，不建议直接封登录，而是优先限制高收益行为。

建议限制项：

- 新号前 72 小时禁止或限额使用坊市
- 新号前 72 小时禁止高价值物品转移
- 新号前若干小时内禁用批量兑换码收益
- 活动奖励、首通奖励、邀请奖励按风险主体去重
- 可疑账号组之间的交易、邮寄、丢弃拾取增加额外风控检查

如果未来加入下列系统，也应默认接入反多开：

- 邮件附件
- 玩家间交易
- 拍卖行 / 坊市
- 公会仓库
- 师徒奖励
- 邀请奖励
- 排行榜赛季结算

原则：

- 允许正常玩家登录和游玩
- 重点切断“小号给主号输血”的通道

### 5.4 第四层：GM 审核与申诉

仅靠自动规则容易误伤，因此必须提供人工复核能力。

GM 面板至少应支持查看：

- 一个账号的关联 IP 列表
- 一个账号的关联 `deviceId` 列表
- 同 IP / 同 `deviceId` 下的账号簇
- 账号之间的资源流向
- 注册时间聚集情况
- 是否重复领取过同类奖励
- 当前风险分和命中规则明细

GM 操作建议分级：

- 观察
- 限制交易
- 限制领奖
- 临时冻结高价值行为
- 永久封禁

所有 GM 操作都应记录：

- 操作者
- 时间
- 原因
- 证据摘要
- 恢复条件

## 6. 数据模型建议

建议新增以下表或文档。

### 6.1 `account_devices`

用于维护账号与设备关系。

建议字段：

- `id`
- `userId`
- `deviceId`
- `firstSeenAt`
- `lastSeenAt`
- `firstIp`
- `lastIp`
- `firstUserAgent`
- `lastUserAgent`
- `registerSource`
- `riskFlags`

用途：

- 建立账号与设备的多对多关系
- 统计同设备创建了多少账号
- 给 GM 展示账号簇

### 6.2 `account_risk_events`

用于记录触发的风控事件。

建议字段：

- `id`
- `userId`
- `eventType`
- `scoreDelta`
- `evidence`
- `createdAt`
- `resolvedAt`
- `resolvedBy`

典型事件：

- `register_same_ip_burst`
- `register_same_device_burst`
- `login_fail_burst`
- `new_account_market_attempt`
- `suspicious_value_transfer`
- `duplicate_reward_cluster`

### 6.3 `account_risk_profiles`

用于保存聚合后的风险画像。

建议字段：

- `userId`
- `riskScore`
- `riskLevel`
- `lastEvaluatedAt`
- `manualReviewStatus`
- `restrictionFlags`

### 6.4 在 `users` 表补充的字段

建议新增：

- `registerIp`
- `lastLoginIp`
- `lastLoginAt`
- `registerDeviceId`
- `lastLoginDeviceId`
- `riskLevel`

说明：

- 聚合信息放 `users`，便于常用查询
- 明细关系和事件放独立表，避免把 `users` 变成风控垃圾桶

## 7. 服务端接入点

### 7.1 注册接口

位置：`packages/server/src/auth/auth.service.ts`

需要增加：

- 从请求上下文读取真实客户端 IP
- 读取 `deviceId`
- 注册前执行限流和验证码校验
- 注册成功后写入设备关系与风险事件

### 7.2 登录接口

位置：`packages/server/src/auth/auth.service.ts`

需要增加：

- 登录失败计数
- 失败冷却
- 登录成功后更新 `lastLoginIp`、`lastLoginAt`、`lastLoginDeviceId`
- 对异常账号打风险事件

### 7.3 Socket 连接入口

位置：`packages/server/src/game/game.gateway.ts`

需要增加：

- Socket handshake 带 `deviceId`
- 连接建立时补写在线设备关系
- 对已被限制的账号在连接后下发限制原因或禁止进入高权限玩法

### 7.4 高价值系统入口

优先覆盖：

- 坊市
- 邮件
- 兑换码
- 活动奖励
- 排行榜结算

这些系统在发放收益前应先查询：

- 账号风险等级
- 是否命中冷却期
- 是否属于限制交易主体
- 是否与收款目标形成高风险关联

## 8. 客户端方案

Web 端不应承担主判定，但可以承担辅助采集。

建议改动：

- 登录页初始化时申请或恢复 `deviceId`
- 注册、登录、刷新 token、Socket 连接都带上 `deviceId`
- 当服务端要求验证码时，登录页切换到验证码模式
- 当账号被限制高价值行为时，在 UI 中清晰展示限制类型和结束时间

注意：

- 不建议在前端自行计算复杂指纹并把结果作为强依据
- 如果需要浏览器指纹，只上传服务端定义好的最小摘要字段，不要把大量静态信息混入高频链路

## 9. 风险评分建议

不要设计成“命中即封”，建议使用加分模型。

示例：

- 同 IP 30 分钟内第 2 个注册号：`+10`
- 同 IP 30 分钟内第 3 个注册号：`+20`
- 同 `deviceId` 24 小时内第 2 个注册号：`+30`
- 新号 24 小时内尝试进入坊市：`+20`
- 新号将高价值资源集中流向同一账号：`+40`
- 同一账号簇重复领取同类奖励：`+50`

风险等级建议：

- `0-29`：正常
- `30-59`：观察
- `60-99`：限制高价值行为
- `100+`：进入人工审核或临时冻结

减分建议：

- 持续正常在线若干天无异常，逐步衰减风险分
- 人工核实为家庭共玩或线下朋友共玩，可打白名单标签

## 10. 误伤控制

### 10.1 默认不直接按 IP 封禁

IP 只能做弱信号和限流依据。

### 10.2 默认不直接按单次设备关联封禁

同一台电脑上多人共玩、切号、浏览器共用都可能出现设备重叠。

### 10.3 默认先限收益，再限登录

对正常玩家影响更小，也更容易解释。

### 10.4 提供申诉与豁免

对于家庭、情侣、室友、网吧等场景，应允许 GM 标记豁免。

## 11. 分阶段落地计划

### Phase 1：最低成本上线版

目标：快速获得基础识别能力。

内容：

- 新增服务端签发的 `deviceId`
- 注册与登录记录 `IP + UA + deviceId`
- Redis 注册限流与登录失败限流
- 新号 72 小时限制坊市或高价值流通
- 记录风控事件日志

收益：

- 能有效提高脚本批量开号成本
- 基本不影响正常玩家主线体验

### Phase 2：收益闭环版

目标：切断小号输血。

内容：

- 覆盖坊市、邮件、奖励领取、兑换码
- 加入账号风险分
- 加入账号簇视图
- 对高风险转移链路做二次校验

收益：

- 即使多号注册成功，也很难把收益安全转给主号

### Phase 3：精细运营版

目标：支持长期运营。

内容：

- GM 审核面板
- 白名单与豁免机制
- 风险分自动衰减
- 规则调参与审计报表
- 如果产品需要，再接手机号或实名

## 12. 明确不建议的方案

- 只按 IP 限制多开
- 只靠浏览器指纹直接封号
- 只做登录限制，不做经济收益限制
- 把所有风控逻辑塞进客户端
- 在高频 tick 或 AOI 广播里混入风控详情

## 13. 推荐的第一版落地结论

对当前项目，推荐先做下面四件事：

1. 为注册、登录、Socket 连接补齐 `IP + UA + deviceId` 采集
2. 为 `/auth/register` 和 `/auth/login` 增加 Redis 限流与失败冷却
3. 对新号和高风险号限制坊市、兑换码和高价值流转
4. 增加 GM 可查的“关联账号簇 + 风险事件”视图

这四项能以较低改造成本覆盖最核心的多号收益场景，并且与当前“一账号一角色、同账号顶号、服务端权威”的架构兼容。
