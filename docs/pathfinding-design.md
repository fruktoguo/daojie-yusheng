# 寻路调度与多线程设计

> 更新时间：2026-03-25
>
> 本文档定义《道劫余生》服务端寻路系统的正式收敛方案。目标不是继续堆“局部补丁”，而是明确玩家点击移动、AI 寻路、worker 调度、取消策略和 tick 集成的统一模型。

## 1. 结论先行

当前项目的主方案应当是：

- 通用寻路核心仍然使用 `A* / 有硬预算的加权搜索`
- 玩家手动点击移动优先保证手感：
  - 收到点击后立刻启动寻路任务
  - 默认优先完整路径
  - 只有超预算时才允许部分路径 fallback
- Bot、怪物追击、自动战斗接近位优先保证吞吐：
  - 请求统一排队
  - worker 池并行计算
  - 允许延后一息生效
- 所有世界状态变更仍只允许在 tick 驱动的受控流程内落地

这意味着：

- `A*` 仍是基线能力，兼容任意起点、终点、权重和动态阻挡
- 多线程负责“预计算”和“并发削峰”，不直接写世界状态
- 分段路径不是默认模式，只是超预算时的玩家 fallback

## 2. 当前问题

本轮重构前后，已经暴露出几类明确问题：

- 玩家点击移动如果等到 tick 内统一派发，再走异步 worker，在低倍速世界下会出现明显的多息延迟
- 把“分段执行”误做成“默认分段搜索”会破坏绕路能力和高移速单息位移
- 如果旧请求不能被覆盖或取消，频繁改目标会浪费大量无效算力
- AI、Bot、玩家的优先级混在一起时，玩家手感会被后台任务拖累
- worker 结果即使回来，也只能在 tick 中受控消费，因此必须有“只认最新请求”的一致性模型

## 3. 设计目标

### 3.1 必须满足

- 玩家点击移动，收到请求后立刻开始寻路，不等一息末尾
- 玩家每息最多发起 `4` 次寻路请求
- 同一玩家永远只认“最新目标”
- 队列里尚未开始的旧请求应直接丢弃
- 已在 worker 中运行的旧请求应尽快协作取消
- 玩家默认优先完整路径，不能因为分段优化丢失绕路能力
- 单息移动距离只能受移动点数限制，不能被路径块长度错误截断
- 所有最终位置变更、占位变更仍然只能在 tick 内生效

### 3.2 不作为本阶段目标

- 不把流场寻路作为通用主方案
- 不让客户端参与任何影响正确性的寻路判定
- 不追求“硬中断”已在 worker 中运行的 JS 任务
- 不在本阶段引入跨进程或跨地图分布式寻路

## 4. 总体架构

```
客户端点击
  -> GameGateway 收到 moveTo
  -> 立即创建 PathIntent + PathRequest
  -> PathRequestScheduler 尝试即时分发
     - 有空闲 worker: 立即开始计算
     - 无空闲 worker: 进入等待队列
  -> TickService 在受控流程内消费最新完成结果
  -> NavigationService 应用路径并执行单息移动
```

模块职责：

- `NavigationService`
  - 维护玩家/AI 当前移动意图和路径状态
  - 在 tick 中消费已完成结果
  - 按移动点数推进实际位置
- `PathRequestSchedulerService`
  - 管理等待队列
  - 管理最新请求覆盖
  - 管理取消与结果过滤
  - 管理即时分发和批量分发
- `PathWorkerPoolService`
  - 维护 worker 池
  - 分发任务
  - 回收结果
  - 响应取消
- `MapService`
  - 提供只读寻路快照
  - 提供静态地形代价和动态阻挡快照

## 5. 请求模型

### 5.1 PathIntent

`PathIntent` 是某个实体当前“想去哪”的权威意图，挂在主线程运行时状态上。

建议字段：

```ts
interface PathIntent {
  actorId: string;
  actorType: 'player' | 'monster';
  mapId: string;
  targetX: number;
  targetY: number;
  latestRequestId: string;
  lastIssuedTick: number;
  requestCountThisTick: number;
  path: Array<{ x: number; y: number }>;
  pathComplete: boolean;
  pendingRequestId?: string;
  pendingError?: string;
}
```

### 5.2 PathRequest

`PathRequest` 是一次具体的计算任务。

建议字段：

```ts
interface PathRequest {
  requestId: string;
  actorId: string;
  actorType: 'player' | 'monster';
  kind: 'player_move_to' | 'player_repath' | 'bot_roam' | 'monster_chase';
  mapId: string;
  startX: number;
  startY: number;
  goals: Array<{ x: number; y: number }>;
  moveSpeed: number;
  priority: number;
  enqueueOrder: number;
  limits: {
    maxExpandedNodes: number;
    maxPathLength: number;
    maxGoalDistance?: number;
    allowPartialPath?: boolean;
  };
}
```

## 6. 玩家点击移动流程

### 6.1 输入时机

玩家 `moveTo` 不再等 tick 末尾才启动寻路，而是：

1. `GameGateway` 收到点击请求
2. 读取玩家当前运行时位置和地图
3. 生成新的 `requestId`
4. 更新 `PathIntent.latestRequestId`
5. 立即调用调度器尝试下发

注意：

- 这里启动的是“寻路预计算”，不是状态变更
- 真正移动、占位、进图、AOI 广播仍然发生在 tick 中

### 6.2 每息最多 4 次

玩家每息最多允许发起 `4` 次寻路请求。

建议规则：

- 按地图 tick 计数
- 新的一息开始时重置该玩家的 `requestCountThisTick`
- 第 `5` 次及以后请求直接拒绝创建
- 如果同息内旧请求已被更新目标覆盖，配额依然照常消耗

这样可以限制恶意频繁点击，又保留合理修正目标的空间。

## 7. 最新请求覆盖与取消

### 7.1 队列中旧请求

如果旧请求尚未开始计算：

- 直接从等待队列删除
- 不再占用任何 worker

### 7.2 运行中的旧请求

如果旧请求已经在 worker 中运行：

- 不做强杀线程
- 改为“协作取消”

推荐实现：

- 主线程为每个运行中请求挂一个 `SharedArrayBuffer cancelFlag`
- worker 在扩展固定批次数节点后检查 `cancelFlag`
- 若发现当前请求已被置为取消，则立即返回 `cancelled`

建议检查粒度：

- 每扩展 `32` 或 `64` 个节点检查一次

### 7.3 回收时二次过滤

即使 worker 没来得及中途退出，结果回到主线程时仍必须校验：

- `result.requestId === PathIntent.latestRequestId`

否则直接丢弃。

### 7.4 同时只保留 1 个运行中 + 1 个最新等待中

为了进一步压住浪费，建议每个玩家最多保留：

- `1` 个运行中的请求
- `1` 个等待中的最新请求

也就是说，频繁点击时，等待队列里永远只保留最后一次目标。

## 8. 为什么不默认分段搜索

本轮已经验证，默认分段搜索容易犯两个错误：

- 把“返回路径块长度”误当成“搜索深度上限”，导致不会绕路
- 把“路径块长度”误当成“单息移动上限”，导致高移速玩家被错误截断

所以正式方案应当是：

- 玩家默认先尝试完整路径
- 若完整路径在快速预算内可得，直接使用完整路径
- 只有完整路径超预算时，才允许返回部分路径作为 fallback

也就是：

`完整路径优先，部分路径兜底`

而不是：

`默认部分路径`

## 9. 搜索策略

### 9.1 玩家手动点击

优先级最高，策略如下：

1. 收到点击后立刻启动计算
2. 优先尝试完整路径
3. 若命中快速预算，立即可用
4. 若超预算：
   - 若允许 partial fallback，则返回一段可走前缀
   - 同时继续在后台补完整路径或下一轮重算

建议预算分层：

- 快速完整路径预算：
  - `maxExpandedNodes` 较小
  - 目标是保证手感
- 异步完整路径预算：
  - `maxExpandedNodes` 较大
  - 目标是保证可达性

### 9.2 玩家阻挡重算

策略：

- 先做一次小预算同步重算
- 若仍超预算，再转 worker
- 同时有冷却和退避，避免贴墙反复重算

### 9.3 Bot / 怪物 / 自动战斗

这些默认走统一调度队列，不抢玩家主通道。

排序规则建议：

1. 请求优先级
2. 移速降序
3. 等待时间升序

推荐优先级：

- `player_repath`
- `player_move_to`
- `player_attack_approach`
- `monster_chase`
- `bot_roam`

## 10. Tick 集成规则

tick 内只做三件事：

1. 消费最新完成结果
2. 校验结果是否仍可用
3. 推进本息实际移动

绝不能在 worker 线程内直接做：

- 改玩家坐标
- 改占位
- 切图
- 触发传送
- 触发 AOI 广播

## 11. 多线程模型

### 11.1 即时分发

玩家点击时，调度器立即尝试：

- 若有空闲 worker，立刻 `dispatch`
- 若没有空闲 worker，立刻进入等待队列

不必等待 tick 末尾再统一派发。

### 11.2 tick 批量分发仍然保留

对 AI 和后台任务，tick 内仍可以统一补做一次 `pump`：

- 把等待队列中剩余请求继续往 worker 里塞
- 避免只有玩家点击才能驱动队列流转

所以最终是双通道：

- 玩家即时分发通道
- tick 批量补充分发通道

## 12. 地图快照策略

worker 只能读取快照，不能读取 live 世界状态。

建议：

- 静态地形：
  - `walkable`
  - `traversalCost`
  - 按 `mapRevision` 缓存
- 动态阻挡：
  - 每次请求按当前地图构建阻挡快照
  - 包含玩家、怪物、NPC 阻挡

后续若阻挡同步成为瓶颈，再考虑：

- 增量快照
- 共享内存
- 按地图缓存阻挡 revision

本阶段先不提前复杂化。

## 13. 监控指标

当前寻路专项必须补指标，否则后续无法判断优化是否真实生效。

建议至少记录：

- 每息玩家点击寻路请求数
- 每息 AI 寻路请求数
- 即时分发命中数
- 队列入队数
- 队列丢弃数
- 运行中取消数
- 成功数
- 不可达失败数
- 超节点失败数
- partial fallback 命中数
- worker 平均耗时
- worker 最大耗时
- 每个优先级桶的等待时长

## 14. 任务自动寻路缓存设计

任务自动寻路到固定目标点，是当前最适合做缓存的场景之一。

典型特征：

- 目标点稳定
- 地图稳定
- 同一目标会被大量重复请求
- 起点不同，但终点相同

因此，这类请求不应只复用“完整路径结果”，还应复用“目标反向代价场”。

### 14.1 缓存目标

优先针对这些来源启用缓存：

- 任务追踪导航到 NPC
- 任务追踪导航到交付点
- 任务追踪导航到固定交互点
- 任务追踪导航到固定地图入口或传送点

不优先缓存这些场景：

- 玩家临时乱点地图
- Bot 漫游
- 高频变化的追击目标

### 14.2 缓存分层

#### 第一层：目标反向代价缓存

以目标点为中心缓存整张地图到该目标的最小代价信息。

建议 key：

```ts
interface GoalFieldCacheKey {
  mapId: string;
  targetX: number;
  targetY: number;
  actorType: 'player' | 'monster';
  mapRevision: number;
}
```

建议 value：

```ts
interface GoalFieldCacheValue {
  integrationCost: Uint32Array;
  width: number;
  height: number;
  builtAtTick: number;
}
```

使用方式：

- 玩家自动导航到任务目标时
- 若命中缓存，则从当前位置查看四邻格
- 选择累计代价最小的方向作为下一步或下一段起点

优点：

- 同目标点高复用率
- 支持不同地块移动消耗
- 起点不同也能复用

注意：

- 这层缓存只针对静态地形和静态通行代价
- 不把实时占位永久烘进缓存

#### 第二层：完整路径结果短缓存

以完整起终点为 key，缓存近期计算出的完整路径结果。

建议 key：

```ts
interface PathResultCacheKey {
  mapId: string;
  startX: number;
  startY: number;
  targetX: number;
  targetY: number;
  actorType: 'player' | 'monster';
  mapRevision: number;
}
```

用途：

- 同一玩家短时间重复点同一目标
- 自动寻路重试
- 快速回退

这层缓存命中快，但泛化性较弱，因此优先级低于“目标反向代价缓存”。

#### 第三层：任务导航点解析缓存

任务系统直接维护：

- 当前任务导航目标的地图
- 坐标
- 导航类型

避免每次点击“任务追踪”时临时重新解析目标来源。

### 14.3 缓存失效规则

目标反向代价缓存至少在这些情况下失效：

- `mapRevision` 变化
- 目标点变化
- `actorType` 变化

不建议因为玩家/怪物实时占位变化就全量失效整张缓存。

正确做法：

- 缓存只描述静态最优代价趋势
- 实际走下一步时，再用 live 占位做一次校验
- 若下一步被占：
  - 优先选次优邻格
  - 或触发局部重算 / 小预算 A*

### 14.4 与主方案的关系

任务自动寻路缓存不替代 A*。

正式关系应当是：

- 玩家普通点击：默认走 A*
- 任务自动寻路到固定目标：优先命中目标缓存
- 缓存不可用或无法安全前进：回退到 A*

也就是：

`任务高频固定目标 = 缓存优先`

`通用点击与复杂动态场景 = A* 优先`

## 15. 完整实施计划

### P0 文档与规则收敛

- [x] 明确 `A*` 仍是通用基线
- [x] 明确玩家点击移动必须即时启动寻路
- [x] 明确旧请求覆盖和取消规则
- [x] 明确“完整路径优先，partial 仅兜底”

### P1 玩家请求生命周期重构

- [ ] 在 `GameGateway` 收到 `moveTo` 时立即启动寻路请求
- [ ] 为每个玩家增加“每息最多 4 次”的计数器
- [ ] 为每个玩家增加 `latestRequestId`
- [ ] 点击新目标时直接覆盖等待中的旧请求
- [ ] 已完成但尚未消费的旧结果也直接作废

### P2 调度器取消机制

- [ ] 调度器支持“同玩家只保留最新等待请求”
- [ ] worker 池支持取消正在运行的旧请求
- [ ] worker 内按固定节点批次检查取消标记
- [ ] 主线程回收结果时强制做 `requestId` 过滤

### P3 玩家路径策略收敛

- [ ] 玩家默认先求完整路径
- [ ] 仅当完整路径超预算时才启用 partial fallback
- [ ] 玩家阻挡重算先走小预算同步重算，再退到 worker
- [ ] 保证单息位移只受移动点数限制

### P4 任务自动寻路缓存

- [ ] 为任务系统增加稳定的导航目标解析结果
- [ ] 新增目标反向代价缓存模块
- [ ] 任务固定目标优先命中目标缓存
- [ ] 下一步 live 校验失败时回退到小预算 A*
- [ ] 增加缓存命中率、失效率、回退率指标

### P5 AI 统一接入

- [ ] 怪物追击接入统一调度器
- [ ] 自动战斗接近位接入统一调度器
- [ ] Bot 漫游继续使用低优先级 worker 队列
- [ ] 校正不同 actorType 的阻挡规则

### P6 指标与调优

- [ ] 增加寻路专项 CPU/队列/取消指标
- [ ] GM 面板显示寻路请求数、队列深度、取消数、失败原因
- [ ] 根据实测数据再调：
  - 玩家每息请求上限
  - worker 数
  - 快速完整路径预算
  - partial fallback 启用阈值
  - 任务目标缓存大小
  - 任务目标缓存 TTL

## 16. 里程碑与验收

### 里程碑 M1：玩家手感恢复

完成条件：

- 玩家点击移动收到后立即启动寻路
- 每息最多 4 次请求生效
- 旧请求覆盖正确
- 等待队列不堆积同玩家历史目标

验收口径：

- `0.5x` 下玩家近距离点击不再出现“两三息后才开始走”
- 高频连续改点时，旧结果不会覆盖新目标

### 里程碑 M2：无效计算削减

完成条件：

- worker 支持协作取消
- 运行中的旧请求能尽快退出
- 已完成旧结果会被主线程过滤

验收口径：

- 高频改点时，取消数明显高于成功数
- worker 总耗时不再随着连续改点近似线性暴涨

### 里程碑 M3：任务自动寻路提效

完成条件：

- 固定任务目标启用缓存
- 缓存命中时无需重新完整 A*
- 缓存不可用时能稳定回退

验收口径：

- 同一任务目标反复导航时，平均寻路耗时明显下降
- 缓存命中不影响正确绕路

### 里程碑 M4：AI 并发上限提升

完成条件：

- 怪物追击和自动战斗接近位接入统一调度
- 不同优先级请求互不抢占玩家主通道

验收口径：

- 高并发 AI 场景下，玩家点击寻路延迟仍然稳定

## 17. 最终推荐落地顺序

如果只按最稳的顺序推进，建议严格按下面来：

1. 先做玩家即时启动 + 最新请求覆盖
2. 再做等待队列淘汰 + worker 协作取消
3. 再把玩家默认路径策略收敛成“完整优先”
4. 再做任务自动寻路缓存
5. 最后再扩到怪物和自动战斗

不要再继续在“默认分段路径”的细节上堆补丁。

那条路可以修，但不是最优收敛方向。
